# HBase 写流程详解
## 写流程整体架构图
1. Client（Java/Flink）：发起写入请求
2. 到 Zookeeper：返回 Meta 表位置
    1. 获取 `hbase:meta` 表的位置信息
    2. 首次连接需要，后续缓存到客户端
3. 到`hbase:meta` 表：返回 RegionServer 地址
    1. 查询目标 RowKey 属于哪个 Region
    2. 获取该 Region 所在的 RegionServer 地址
    3. 结果缓存到客户端，后续请求直接使用
4. 到 RegionServer
    1. WAL（Write-Ahead Log）：先写 WAL 日志，持久化到 HDFS，✅ 保证数据不丢失
    2. MemStore：再写 MemStore，跳表结构，✅ 加速后续读取
    3. 达到 Flush 阈值
    4. Flush 操作：MemStore → HFile
    5. 写入 HDFS：StoreFile（HFile 格式），持久化到 HDFS
    6. 返回写入成功
5. Client 收到响应

## 写流程详细步骤
### Step 1：Client 发起写入请求
```java
// 1. 创建连接
Configuration conf = HBaseConfiguration.create();
conf.set("hbase.zookeeper.quorum", "node1:2181,node2:2181,node3:2181");
Connection conn = ConnectionFactory.createConnection(conf);

// 2. 获取表对象
Table table = conn.getTable(TableName.valueOf("user_info"));

// 3. 构造 Put 对象
Put put = new Put(Bytes.toBytes("user_001"));
put.addColumn(Bytes.toBytes("cf1"), Bytes.toBytes("name"),
            Bytes.toBytes("张三"));
put.addColumn(Bytes.toBytes("cf1"), Bytes.toBytes("age"),
            Bytes.toBytes(25));

// 4. 执行写入
table.put(put);  // 触发写流程
```

### Step 2：ZooKeeper 获取 Meta 表位置
ZK 路径：`/hbase/meta-RegionServer-location`



Client 首次连接流程：

1. 连接 ZooKeeper 集群
2. 读取 `/hbase/meta-RegionServer-location` 节点
3. 获取` hbase:meta` 表所在的 RegionServer 地址\
4. 缓存到本地（后续请求直接使用，无需再查 ZK）

⚠️ 注意：只有首次连接或 Meta 表位置变更时才需要查 ZK

### Step 3：查询 `hbase:meta` 表，定位 Region
`hbase:meta` 表存储所有 Region 的位置信息

| RowKey | info:regioninfo       | 列值 |
| --- | --- | --- |
| `user_info,,1645123456789` | `{table:user_info,startKey:空，endKey:user_5000}` | Region A |
| `user_info,user_5000,1645123456790` | `{table:user_info, startKey:user_5000,endKey:user_9999}` | Region B |




Client 定位过程：

1. 计算 `RowKey` 的哈希值
2. 扫描 `hbase:meta` 表，找到包含该 RowKey 的 Region 记录
3. 获取该 Region 所在的 RegionServer 地址
4. 缓存到本地（RegionLocator 缓存）

### Step 4-5：RegionServer 写 WAL 和 MemStore
RegionServer 内部写入流程：RegionServer 接收请求：后续流程

1. WAL（Write-Ahead Log）：✅ 先写 WAL 保证持久化
    1. 构造 Entry：Sequence ID、Table Name、Region Name、Edit Data
    2. 追加到 WAL：顺序写入，HDFS 存储，同步/异步
2.  MemStore（内存缓存）：✅ 后写 MemStore 加速 
    1. 写入跳表：按 RowKey 排序，支持多版本，内存操作快



⚠️ 核心原则：先 WAL 后 MemStore，确保数据不丢失 

### Step 6-7：Flush 生成 StoreFile
Flush 触发条件

1. MemStore大小阈值：单个 MemStore 达到此值触发 Flush
    - 参数配置：`hbase.hregion.memstore.flush.size`
    - 默认值：128MB
2. Region 总 MemStore 阈值：总 Memstore 达到 RegionServer 堆内存40%
    - 参数配置：`hbase.regionserver.global.memstore.size`
    - 默认值：0.4
3. WAL 数量阈值：WAL 文件数量达到阈值
    - 参数配置：`hbase.regionserver.max.logs`
    - 默认值：32
4. 手动触发：通过 API 或命令行手动 Flush，`flush 'table_name'`
5. Region 关闭/RegionServer 宕机：自动触发 Flush



 **Flush 详细过程**

+ Flush 前

```plain
                    MemStore (128MB)
┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐
 user_001     user_002    user_003    ...         user_999
└─────────┘ └─────────┘ └─────────┘ └─────────┘ └─────────┘
```

+ Flush 触发
    - Step 1：标记当前 MemStore 为不可写
    - Step 2：创建新的 MemStore（后续写入进入新 MemStore）
    - Step 3：将旧 MemStore 数据写入临时文件
    - Step 4：临时文件重命名为 StoreFile（HFile 格式）
    - Step 5：更新 Region 元数据，记录新 StoreFile
    - Step 6：清理已 Flush 的 WAL 日志
+ Flush 后：

```plain
                    MemStore (新的，空)
┌─────────────────────────────────────────────────────────┐
  等待新数据写入...
└─────────────────────────────────────────────────────────┘
```

```plain
HDFS - StoreFile
┌─────────────────────────────────────────────────────────┐
    /hbase/data/default/user_info/
    region_abc/ cf1/  store_file_001.hfile
└─────────────────────────────────────────────────────────┘
```

# HBase 读流程详解
## 读流程整体架构图
1. Client（Java/Flink）：发起读取请求
2. 到 Zookeeper
    1. 获取 `hbase:meta` 表的位置信息（首次连接）
    2. 后续请求使用本地缓存
3. 返回 Meta 表位置
4. 到`hbase:meta`表
    1. 查询目标 RowKey 属于哪个 Region
    2. 获取该 Region 所在的 RegionServer 地址
    3. 结果缓存到客户端
5. 返回 RegionServer 地址
6. 到 RegionServer 查，如果命中直接返回，未命中则一次向下查
    1. 先查缓存 BlockCache（读缓存）：L1 堆内缓存，L2 堆外缓存，✅命中则返回
    2. 再查 MemStore（写缓存）：最新写入的数据，跳表结构，快速查找，✅命中则返回
    3. Bloom Filter：快速过滤，判断 RowKey 是否存在，✅不存在则直接返回空
    4. 如果可能存在，扫描磁盘文件（StoreFile/HFile）：通过 Index Block 定位，加载 Data Block 到缓存，✅ 找到则返回
7. 合并结果返回
8. Client 收到数据

## 读流程详细步骤
### Step 1-3：Client 定位 RegionServer（同写流程）
读请求定位流程（同写流程）：Client → ZooKeeper → `hbase:meta` → RegionServer

⚠️ 注意：定位信息会缓存到客户端，后续请求直接使用 

缓存失效场景：

+ Region 分裂
+ Region 迁移
+ RegionServer 故障
+ 缓存 TTL 过期（默认 60 秒）

### Step 4：BlockCache 查询
BlockCache 层级结构：

+ 先查：L1 Cache (堆内)
    - BucketCache 实现
    - 受 JVM 堆大小限制
    - 快速访问（无序列化开销）
    - 默认大小：堆内存的 40%
+ 未命中，查 L2 Cache（堆外）
    - BucketCache 实现
    - 不受 GC 影响
    - 容量更大（可配置）
    - 需要序列化/反序列化



查询逻辑

```java
if (BlockCache.contains(rowKey, family, qualifier)) {
    return BlockCache.get(rowKey, family, qualifier);
    // ✅ 缓存命中，直接返回，无需访问磁盘
} else {
    // ❌ 缓存未命中，继续查询 MemStore
    goToStep5();
}
```

### Step 5：MemStore 查询
MemStore 数据结构：跳表（`SkipList`）

```plain
Level 3:  ──► row_001 ──────────────► row_050 ──► row_100

Level 2:  ──┼─────► row_025 ──────────┼──► row_075

Level 1:  ──┼─────┼─────► row_037 ────┼──┼──► row_088

Level 0:  ──┼─────┼─────┼─────┼───────┼──┼──┼──► row_099

Data:     [Cell] [Cell] [Cell] [Cell] [Cell] [Cell] [Cell]
```



查询特点：

+ 时间复杂度：O(log N)
+ 内存操作，速度极快
+ 存储最新写入的数据（尚未 Flush）
+ 支持多版本查询

### Step 6：Bloom Filter 过滤
什么是 Bloom Filter？

+ 一种概率型数据结构：用于判断一个元素是否存在于集合中
+ 特点：
    - 如果说不存在 → 一定不存在（100% 准确）
    - 如果说存在 → 可能存在（有误判率）



HBase 中的 Bloom Filter

| 类型 | 说明 | 配置参数 |
| --- | --- | --- |
| `NONE` | 不使用 | `hbase.hfile.bloom.enabled` |
| `ROW` | 行级 Bloom Filter | `hbase.hfile.bloom.type` |
| `ROWCOL` | 行+列级 Bloom Filter | `hbase.hfile.bloom.type` |




工作原理：✅ 优势：避免扫描不包含目标 RowKey 的 StoreFile，提升读性能

```plain
StoreFile 1    StoreFile 2    StoreFile 3    StoreFile 4
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
 Bloom        Bloom        Bloom        Bloom
 Filter       Filter       Filter       Filter
 不存在       不存在       存在         不存在
└─────────┘    └─────────┘    └─────────┘    └─────────┘
     ↓               ↓               ↓               ↓
 跳过扫描       跳过扫描       需要扫描       跳过扫描
```

### Step 7：StoreFile 扫描
HFile 结构

+ Data Block：存储实际 KV 数据，默认 64KB
+ Leaf Index Block：存储 Data Block 的偏移量，用于快速定位
+ Bloom Filter Block：快速判断 RowKey 是否存在
+ Trailer：文件尾部，包含所有 Block 的元数据



扫描过程

1. 读取 Trailer（固定位置，快速定位）
2. 加载 Index Block 到内存
3. 通过 Index Block 定位目标 Data Block 的偏移量
4. 从 HDFS 读取目标 Data Block
5. 加载到 BlockCache（供后续查询使用）
6. 在 Data Block 中查找目标 KV 对

## 读流程关键参数
```xml
  <!-- BlockCache 配置 -->
  <property>
      <name>hfile.block.cache.size</name>
      <value>0.4</value>  <!-- BlockCache 占堆内存 40% -->
  </property>

  <property>
      <name>hbase.bucketcache.size</name>
      <value>1024</value>  <!-- L2 缓存大小（MB） -->
  </property>

  <!-- Bloom Filter 配置 -->
  <property>
      <name>hbase.hfile.bloom.enabled</name>
      <value>true</value>  <!-- 启用 Bloom Filter -->
  </property>

  <property>
      <name>hbase.hfile.bloom.type</name>
      <value>ROW</value>  <!-- ROW 或 ROWCOL -->
  </property>

  <!-- 读性能优化 -->
  <property>
      <name>hbase.client.scanner.caching</name>
      <value>100</value>  <!-- Scan 每次从服务端拉取行数 -->
  </property>
```

# BlockCache 详解
## BlockCache 的作用
1. 作用 1：加速读请求
    - 缓存热点 Data Block，避免重复从 HDFS 读取
    - 缓存命中率越高，读性能越好
    - 典型场景：用户画像查询、热点数据访问
2. 作用 2：减少 HDFS IO 压力
    - 减少磁盘读取次数
    - 降低网络传输开销
    - 提升集群整体吞吐能力
3. 作用 3：降低查询延迟
    - 内存访问 vs 磁盘访问：100ns vs 10ms（10 万倍差距）
    - <font style="background-color:#E8F7CF;">缓存命中：毫秒级响应</font>
    - <font style="background-color:#E8F7CF;">缓存未命中：可能秒级响应</font>

## BlockCache 层级架构
传统 BlockCache（单层级）：JVM 堆内缓存

+ 受 GC 影响
+ 容量有限（堆内存 40%）
+ GC 时可能暂停服务



BucketCache（双层架构）- 推荐

+ L1 Cache - BlockCache (堆内)
    - 存储热点 Block 的元数据和索引
    - 快速访问，无序列化开销
    - 大小：堆内存的 10-20%
+ L2 Cache - BucketCache (堆外)
    - 存储实际 Data Block 数据
    - 不受 GC 影响
    - 容量更大（可配置到几百 GB）
    - 需要序列化/反序列化

## BlockCache 淘汰策略
为什么需要淘汰？

+ BlockCache 容量有限（堆内存 40%）
+ 新数据不断写入，缓存会满          
+ 需要淘汰旧数据，为新数据腾出空间   

### 淘汰策略对比
淘汰策略

1. LRU：Least Recently Used（最近最少使用）
    - 淘汰最长时间未访问的 Block 
    - 实现简单，但可能淘汰热点数据
    - HBase 默认策略
2. SLRU：Segmented LRU（分段 LRU）
    - 将缓存分为多个段（热点段、冷数据段）
    - 热点数据优先保留 
    - 更适合读多写少场景
3. TINYCLT：TinyLRU + Clock（改进版）
    - 结合时间戳和访问频率
    - 更智能的淘汰决策   
    - HBase 2.0+ 推荐  

### LRU 淘汰策略详解
LRU 原理

```plain
缓存状态（按访问时间排序）：
┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐
 Block A   Block B   Block C   Block D   Block E
 最新访问                                 最久未访
└─────────┘ └─────────┘ └─────────┘ └─────────┘ └─────────┘
	 ↑
新数据写入，缓存已满时：
	 ↓
淘汰 Block E（最久未访问）
```



LRU 实现

```java
// HBase 中的 LRU 实现（简化版）
public class LruBlockCache implements BlockCache {

    // 使用 LinkedHashMap 实现 LRU
    private LinkedHashMap<String, Block> cache;

    @Override
    public void putBlock(String key, Block block) {
        if (cache.size() >= maxSize) {
            // 淘汰最旧的 entry
            String oldestKey = cache.keySet().iterator().next();
            cache.remove(oldestKey);
        }
        cache.put(key, block);
    }
}
```

### SLRU 淘汰策略详解
SLRU 原理：缓存分为两个段：

1. Probationary Segment：试用段
    - 新数据首先进入此段
    - 容量较小（约 20%）
    - 如果数据被再次访问，晋升到 Protected 段
    - 否则在此段被淘汰
2. 再次访问进入 Protected Segment：保护段
    - 存储热点数据
    - 容量较大（约 80%）、
    - LRU 淘汰策略
    - 优先保留，不易被淘汰



SLRU 优势

+ 避免一次性访问的数据占用缓存空间
+ 热点数据得到更好保护
+ 更适合读多写少、有热点数据的场景

## BlockCache 优化建议
### 场景 1: 读多写少（用户画像查询）
配置建议：

+ hfile.block.cache.size = 0.5  (50% 堆内存给缓存)
+ hbase.bucketcache.size = 2048  (2GB 堆外缓存)
+ 启用 SLRU 淘汰策略
+ 启用 Bloom Filter (ROW 级别)



预期效果：

+ 缓存命中率 > 80%
+ 查询延迟 < 10ms

### 场景 2: 写多读少（日志存储）
配置建议：

+ hfile.block.cache.size = 0.2  (20% 堆内存给缓存)
+ hbase.regionserver.global.memstore.size = 0.5  (50% 给 MemStore)
+ Bloom Filter 可关闭（减少写入开销）



预期效果：

+ 写入吞吐最大化
+ 读性能可接受

### 场景 3: 读写均衡（订单系统）
配置建议：

+ hfile.block.cache.size = 0.4  (40% 堆内存给缓存)
+ hbase.regionserver.global.memstore.size = 0.4  (40% 给 MemStore)
+ 启用 Bloom Filter (ROW 级别)
+ 使用 BucketCache 双层架构



预期效果：

+ 读写性能平衡
+ 缓存命中率 60-80%

    1. L1 快速访问，L2 大容量不受 GC 影响，性能与容量平衡

# 总结速查表
【写流程】

+ Client → ZK → Meta → RegionServer → WAL → MemStore → Flush → StoreFile
+ 核心原则：先 WAL 后 MemStore，确保数据持久化



【读流程】

+ Client → ZK → Meta → RegionServer → BlockCache → MemStore → StoreFile
+ 核心原则：先缓存后磁盘，层层过滤加速查询



【BlockCache】

+ 作用：缓存热点 Data Block，加速读请求
+ 层级：L1（堆内）+ L2（堆外）双层架构
+ 淘汰：LRU（默认）、SLRU（分段）、TINYCLT（改进）
+ 监控：命中率 > 50%，淘汰率 < 30%



【优化方向】

+ 读多写少：增大 BlockCache，启用 SLRU，启用 Bloom Filter
+ 写多读少：增大 MemStore，减小 BlockCache，可关闭 Bloom Filter
+ 读写均衡：BlockCache 40%，MemStore 40%，双层缓存架构
