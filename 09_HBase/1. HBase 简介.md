# HBase 数据模型
HBase 的数据模型

+ 逻辑视图：数据是以表的形式进行组织，表由行、列构成
+ 物理视图：Map，由键值对构成，是稀疏的、分布式的、多维排序的 Map

## 逻辑视图
HBase 中的基本概念

1. **Table：表**，包含多行数据
2. **Row：行**，由 rowkey、column 及对应的值构成
3. **Column：列**，由 Column Family（列簇）和 Qualifier（列名）组成
    - 格式：`column family: qualifier`
    - Column Famliy 建表时指定，不能随意增减
    - 一个 Column Famliy 下可设置任意多个 Qualifier，列簇下的列可动态增加，理论上可扩展上百万列
4. **Timestamp：时间戳**，每个 Cell 写入 HBase 时默认分配一个时间戳作为 Cell 的版本
    - HBase 支持多版本特性，同一个 Rowkey、Column 下可以有多个 Value 存在，使用 Timestamp 作为版本号，版本越大，数据越新
5. **Cell：单元格**，由五元组`(row, column, timestamp, type, value)`组成的结构
    - `type`：表示`Put/Delete`等的操作类型
    - 五元组以 KV 结构存储，`K = (row, column, timestamp, type)`，`V = value`
    - 具体：`K = (rowkey, column family: qualifier, timestamp, type)`

## <font style="color:rgba(0, 0, 0, 0.8);">HBase 的特性</font>
+ **多维：**Key 是复合数据结构，由`rowkey`、`column family`、`qualifier`、`type`、`timestamp`组成
+ **稀疏**：
    - 逻辑视图示例中的一行数据，**只有一列有值，其他列为空值**
    - 不同于其他数据库的空值填充`null`，HBase 的空值不需要填充，避免空间浪费
+ **排序**：按照 KV 中 `key` 排序，而非 `rowkey` 排序
    - 比较顺序： `rowkey` -> `column family : qualifier` -> `timestamp`
+ **分布式**：构成 HBase 的 Map 分布在集群中

## 行式存储、列式存储、列簇式存储
行式存储：

+ 一行数据存储在一起，一行写完再写下一行
+ 获取一行数据时，高效
+ 读取指定列数据，低效：
    - 先取出一行数据
    - 然后在每一行数据中截取待查找目标列
    - 在查找过程中引入大量无用列信息，导致大量内存占用



列式存储

+ 一列数据存储在一起
+ 只查找某些列时，高效
+ 获取一行数据，低效：
    - 多次 IO 读取多个列，最终合并得到一行数据
+ **高压缩特性：同一列的数据通常具有相同的数据结构**



列簇式存储

+ 介于行式存储、列式存储之间，可以通过不同的设计思路在行式存储和列式存储间切换
+ 一张表只设置一个列簇：即行式存储
+ 一个列簇下仅一列：即列式存储
+ HBase 中不建议设置太多列簇

# HBase 体系结构
HBase 体系结构，借鉴 BigTable 论文，是 Master - Slave 模型

+ Master 节点：管理集群
+ RegionServer 节点：实际服务用户读写

其他

+ HDFS：存储 HBase 中的数据
+ Zookeeper：协助 Master 对集群进行管理

1. **HBase 客户端 Client**
    - 提供 Shell 命令行接口、原生 Java API 编程接口、Thrift/REST API 编程接口、MapReduce 编程接口
    - 支持：所有常见的 DML、DDL 操作
    - HBase 客户端访问数据行的过程
        * 通过**元数据表**定位目标数据所在的 RegionServer
        * 发送请求到 RegionServer
        * 元数据被缓存在客户端本地
2. **Zookeeper**
    - 实现 HBase Master 高可用
        * ZK 检测 Master 是否宕机
        * 宕机后，ZK 通过一定机制选举新的 Master
    - 管理核心元数据：包括
        * 正常工作的 RegionServer 集合
        * 保存系统元数据表`hbase:meta`所在 RegionServer 地址
    - 参与 RegionServer 宕机恢复：
        * ZK 通过**心跳机制**感知 RegionServer 是否宕机
        * ZK 在宕机后通知 Master 进行宕机处理
    - 实现分布式表锁
        * HBase 在对一张表进行管理操作前，**先加表锁**，避免同时操作造成表状态不一致
        * ZK 通过特定机制实现分布式表锁
3. **Master：管理工作**
    - 处理用户的各种管理请求：包括建表、修改表、权限操作、切分表、合并数据分片、Compaction 等
    - 管理集群中所有 RegionServer，包括 Region 负载均衡、宕机恢复、Region 迁移等
    - 清理过期日志以及文件
4. **RegionServer**
    - 响应用户的 IO 请求，由 WAL（HLog）、BlockCache、多个 Region 构成
    - WAL（HLog）
        * 实现数据的高可靠：**<font style="background-color:#E8F7CF;">HBase 数据随机写入时，先顺序写入 HLog，然后写入缓存，异步刷新落盘</font>**，如果缓存数据丢失，可通过 HLog 日志恢复
        * 实现 HBase 集群间的**<font style="background-color:#E8F7CF;">主从复制：通过回放主集群推送的 HLog 日志实现</font>**
    - BlockCache：读缓存，客户端从磁盘读取的数据缓存到系统内存中，适用于热点读
