HBase 在数据仓库（Data Warehouse）架构中扮演着【实时服务层】或【明细查询层】的关键角色。

+ 弥补了传统 Hive/Spark 离线数仓查询延迟高（分钟级/小时级）的短板
+ 同时解决了 MySQL 等关系型数据库无法存储海量数据（PB 级）且不支持灵活 Schema 的痛点。

# 实时数仓：存储热点明细数据，支持毫秒级查询
## 场景背景
在传统 Lambda 或 Kappa 架构中，实时计算引擎（如 Flink）处理后的结果需要落地存储，供前端大屏、实时监控或即时决策使用。

这些数据通常具有**写入量大、查询并发高、要求低延迟**的特点。

## HBase 的角色
HBase 定位：实时数仓的 ADS 层或 DWS 层的存储载体。

****

**核心价值**：

+ 将 Flink 计算的实时指标以 Key-Value 形式存储
+ 提供 **< 10ms** 的点查能力。

## 典型设计模式
RowKey 设计：

+ **维度聚合型**：`日期_业务线_维度 ID`
    - 例如：`20231027_Electronics_Shop001`（查询某店铺今日实时销量）
+ **时间反转型**：
    - `反转时间戳_设备 ID`。用于查询设备最近的状态序列

****

**数据更新**：利用 HBase 的 `Put` 覆盖特性，Flink 实时流不断更新同一 RowKey 的最新值（如实时库存扣减、实时计数累加）。

****

**版本控制**：开启多版本（`VERSIONS=5`），既可以看到当前最新值，也可以回溯过去几分钟的历史状态，用于故障排查。

## 案例：电商实时大屏
架构设计

+ 数据流：Kafka (订单流) -> Flink (实时聚合) -> HBase。
+ RowKey：`reverse_timestamp_category_id`。
+ 查询：大屏轮询请求，直接 `Get` 或短范围 `Scan` 获取各类目实时销售额，毫秒级响应，支撑双 11 流量洪峰。

# 用户画像：存储用户标签、行为轨迹
## 场景背景
用户画像系统需要存储

+ 亿级用户的数千个标签（静态属性、统计标签、预测标签）
+ 以及详细的行为轨迹（点击、浏览、收藏）。

数据量通常在 **十亿至百亿级**，且查询模式复杂（单用户查询、人群圈选）。

## HBase 的角色
HBase 定位：画像系统的 **One-ID 主数据存储** 和 **行为明细库**。

****

核心价值：

1. **稀疏性支持**：用户标签极其稀疏（有的用户有“豪车”标签，有的没有），HBase 的列式存储天然适合，不浪费空间。
2. **横向扩展**：轻松存储 PB 级行为日志。
3. **高并发点查**：支持推荐系统、广告系统每秒数万次的 QPS 查询，快速获取用户标签用于实时排序。

## 典型设计模式
表 1：用户标签宽表 (`user_profile`)

+ **RowKey**：`Hash(User_ID)` 或 `Salt_User_ID`（防止热点，支持高并发单点查询）。
+ **列族设计**：按标签类型分列族。
    - `base`: 性别、年龄、地域。
    - `stat`: 近 7 日消费额、近 30 日登录次数。
    - `algo`: 推荐模型打分、流失概率。
+ **优势**：一次 `Get` 操作即可拉取用户所有标签，效率极高。

****

表 2：用户行为轨迹表 (`user_behavior`)

+ **RowKey**：`Salt_User_ID + Reverse_Timestamp`。
+ **价值**：
    - `Salt` 解决写入热点。
    - `Reverse_Timestamp` 保证同一用户的行为按时间倒序排列，查询“最近 100 条行为”只需 Scan 一小段数据，无需全表扫描。
+ **TTL 设置**：行为数据价值随时间衰减，可设置 `TTL=90 天`，自动清理旧数据，降低存储成本。

## 案例：个性化推荐
+ **流程**：
    - 用户打开 APP -> 网关请求画像服务 
    - -> HBase `Get` 用户标签（兴趣偏好、消费能力）+ `Scan` 最近行为 
    - -> 召回/排序模型 -> 返回推荐列表。
+ **性能**：全程耗时 < 20ms，其中 HBase 查询贡献 < 5ms。

# 风控场景：实时反欺诈查询
## 场景背景
在支付、信贷、登录等环节，需要在 **毫秒级** 内判断当前操作是否存在风险。

这需要快速查询用户的历史黑名单记录、设备关联关系、短时高频交易次数等。

## HBase 的角色
**HBase 定位**：风控规则的 **特征存储库** 和 **黑名单库**。

****

**核心价值**：**极低的读取延迟** 和 **高写入吞吐**。

+ 风控规则往往涉及复杂的关联查询（如：该设备 ID 在过去 1 小时内是否关联了 5 个不同用户？），
+ HBase 的多维度索引表设计能完美支撑。

## 典型设计模式
**多维索引设计**（通过多张表实现同一数据的不同查询视角）：

1. **主表**：`Risk_Event` 
    1. RowKey: `Event_ID`
    2. 存储完整事件详情。
2. **索引表 A**：`User_Risk_Index` 
    1. RowKey: `User_ID + Reverse_Time`
    2. 查某用户的历史风险事件。
3. **索引表 B**：`Device_Risk_Index` 
    1. RowKey: `Device_ID + Reverse_Time`
    2. 查某设备的关联风险。
4. **索引表 C**：`IP_Freq_Index` 
    1. RowKey: `IP_Date_Hash`
    2. 查某 IP 在当天的访问频次（配合计数器 Column）。

****

**原子计数器**：利用 HBase 的 `IncrementColumnValue` 实现高性能计数器。

+ 场景：统计某 IP 在 1 分钟内的请求数。
+ 操作：每次请求 `incr`，若返回值 > 阈值，直接拦截。比 Redis 更持久，比 MySQL 更快。

## 案例：信用卡盗刷拦截
**场景**：用户刷卡。

**风控逻辑**：

1. 查黑名单表（`Get` 卡号）：是否在黑名单？(2ms)
2. 查频率表（`Get` 计数器）：该卡过去 5 分钟交易是否 > 3 次？(2ms)
3. 查关联表（`Scan` 设备 ID）：该卡是否在新设备上登录？(5ms)

**结果**：总耗时 < 10ms，决定放行或阻断。

# 订单/日志明细：海量数据存储 + 随机查询
## 场景背景
电商订单、物流轨迹、服务器应用日志、IoT 传感器数据等，数据量增长极快（年增 PB 级）。

业务方既需要

+ 离线分析（走 Hive/Spark）
+ 又经常需要**根据单号/ID 进行在线随机查询**（客服查单、用户查历史订单、运维查报错日志）

MySQL 分库分表维护成本太高，ES 存储成本太贵且写入性能有瓶颈。

## HBase 的角色
HBase 定位：海量明细数据的 **在线归档库** 和 **查询引擎**。

****

核心价值：

1. 无限扩展：理论上存储容量无上限。
2. 低成本：基于 HDFS，存储成本远低于 SSD 关系的数据库或 ES。
3. 随机读写：在 PB 级数据量下，依然能通过 RowKey 实现毫秒级点查。

## 典型设计模式
**RowKey 设计核心**：散列 + 业务 ID

+ 订单表：`Salt_Order_ID` 或 `User_ID_Salt_Order_ID`。
    - 如果主要按订单号查：`MD5(Order_ID)[0-4 位] + Order_ID`。
    - 如果主要按用户查：`User_ID + Reverse_Time + Order_ID`。
+ 日志表：`Reverse_Timestamp_ServiceName_LogLevel_Host`。
    - 支持按时间范围 + 服务名快速定位日志。

****

**冷热分离：**

+ 热数据（最近 3 个月）：存储在 SSD 池或内存缓存比例调大，保证查询速度。
+ 冷数据（3 个月前）：通过 HBase 的 Mob (Medium Object) 特性或底层 HDFS 策略，自动迁移到 HDD 廉价存储，甚至归档到 S3/OSS（通过 Snapshot 导出），降低 TCO。

****

**二级索引方案**：

+ 对于非 RowKey 字段的查询（如按手机号查订单）
+ 通常采用 **Phoenix 二级索引** 或 **Flink 双写索引表** 的方式解决。

## 案例：物流全链路追踪
+ **数据**：数十亿条包裹节点日志。
+ **查询**：用户输入运单号，查询所有流转节点。
+ **实现**：RowKey = `Hash(Waybill_ID) + Timestamp`。
+ **效果**：即使数据量达到百亿级，查询单个运单轨迹依然稳定在 50ms 以内，且存储成本仅为传统方案的 1/5。

# 总结：HBase 在数仓中的定位矩阵
| 场景 | 数据特征 | 核心诉求 | HBase 解决方案关键点 | 替代方案对比 |
| --- | --- | --- | --- | --- |
| 实时数仓 | 实时流、聚合指标 | 低延迟更新与查询 | 时间反转 RowKey、Put 覆盖更新 | 比 Redis 容量大，比 MySQL 写入快 |
| 用户画像 | 稀疏标签、行为序列 | 高并发点查、稀疏存储 | 宽表设计、多版本、TTL | 比 ES 写入成本低，比 Mongo 扩展性好 |
| 风控反欺诈 | 黑名单、频次统计 | 极致低延迟、原子计数 | 多维索引表、Atomic Counter | 比 MySQL 抗并发强，比 Redis 持久 |
| 订单/日志 | PB 级明细、历史归档 | 海量存储、随机检索 | 散列 RowKey、冷热分离、预分区 | 比 MySQL 免分库分表，比 ES 存储便宜 |




最佳实践建议：

+ 在数仓架构中，HBase 通常不作为唯一的存储，而是与 Hive/Iceberg (离线分析)、Redis (极速缓存)、Elasticsearch (复杂全文检索) 组合使用
+ 形成互补的混合存储架构，以最优的成本和性能满足多样化的业务需求。
