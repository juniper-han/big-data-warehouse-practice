Compaction 是 HBase 中合并 StoreFile、清理过期数据的核心机制，直接影响读写性能和存储效率。

# Compaction 基本概念
## 为什么需要 Compaction
| 问题 | 说明 | 影响 |
| --- | --- | --- |
| StoreFile 数量过多 | 每次 Flush 生成新文件 | 读请求需扫描多个文件，性能下降 |
| 删除标记未清理 | Delete 操作只写标记 | 数据逻辑删除但物理仍存在 |
| 过期版本堆积 | 多版本数据累积 | 存储空间浪费，查询效率低 |
| Region 过大 | 文件总大小增长 | 触发 Region 分裂，增加管理开销 |


## Compaction 核心作用
1. 合并文件：多个小 StoreFile 合并为大文件，减少文件数量
2. 清理数据：移除删除标记和过期版本，释放存储空间
3. 提升性能：减少读请求需要扫描的文件数量
4. 优化布局：数据重新组织，提升顺序读取效率

# Compaction 类型对比
## Minor Compaction vs Major Compaction
| 维度 | Minor Compaction | Major Compaction |
| --- | --- | --- |
| **触发条件** | StoreFile 数量达到阈值 | 手动触发，或定时自动触发 |
| **合并范围** | 部分 StoreFile（小的文件） | 所有 StoreFile |
| **数据清理** | 不清理删除标记和过期版本 | 清理删除标记和过期版本 |
| **执行频率** | 频繁（自动触发） | 较少（默认 7 天一次） |
| **资源消耗** | 较小 | 较大（IO、CPU、网络） |
| **对读写影响** | 较小 | 较大（可能阻塞写入） |
| **生成文件数** | 1 个新文件 | 1 个新文件 |
| **默认周期** | 自动触发 | 7 天（`hbase.hregion.majorcompaction`） |


## 两种 Compaction 的示意图
### Minor Compaction 流程
1. Compaction 前

```plain
┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐
│ F1  │  │ F2  │  │ F3  │  │ F4  │  │ F5  │
│1GB  │  │1GB  │  │1GB  │  │1GB  │  │1GB  │
└─────┘  └─────┘  └─────┘  └─────┘  └─────┘
```

2. 合并 F1-F4，F5 保留
3. Compaction 后

```plain
┌───────────┐  ┌─────┐
│   F_new   │  │ F5  │
│   4GB     │  │1GB  │
└───────────┘  └─────┘
```

### Major Compaction 流程
1. Compaction 前

```plain
┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐
│ F1  │  │ F2  │  │ F3  │  │ F4  │  │ F5  │
│有删除│  │有过期│ │正常 │  │正常 │  │正常  │
└─────┘  └─────┘  └─────┘  └─────┘  └─────┘
```

2. 合并所有文件 + 清理
3. Compaction 后：

```plain
┌───────────┐
│  F_final  │  ← 只有一个 StoreFile
│无删除标记  │    无过期版本
└───────────┘
```

# Compaction 详细机制
## Compaction 触发条件
```xml
<!-- hbase-site.xml 关键配置参数 -->

<!-- Minor Compaction 触发阈值 -->
<property>
    <name>hbase.hstore.compaction.threshold</name>
    <value>3</value>
    <description>StoreFile 数量达到 3 个触发 Minor Compaction</description>
</property>

<!-- Major Compaction 周期（毫秒） -->
<property>
    <name>hbase.hregion.majorcompaction</name>
    <value>604800000</value>
    <description>7 天触发一次 Major Compaction</description>
</property>

<!-- 参与 Compaction 的文件大小上限 -->
<property>
    <name>hbase.hstore.compaction.max.size</name>
    <value>2147483648</value>
    <description>超过 2GB 的文件不参与 Minor Compaction</description>
</property>

<!-- 参与 Compaction 的文件大小下限 -->
<property>
    <name>hbase.hstore.compaction.min.size</name>
    <value>1048576</value>
    <description>小于 1MB 的文件优先参与 Compaction</description>
</property>
```

## Compaction 选择策略
Compaction 文件选择算法

探索算法（Exploring Compaction）：

1. 收集候选文件：所有小于 `max.size` 的 StoreFile
2. 计算优先级：根据文件大小、数量、年龄综合评分
3. 选择最优组合：选择能最大化收益的文件组合
4. 执行 Compaction：合并选中文件生成新文件



选择原则：

+ 优先合并小文件                          
+ 避免合并大文件（IO 开销大）               
+ 控制单次 Compaction 文件数量（默认 10 个）

## Compaction 执行流程
**Step 1：Compaction 检查**

+ RegionServer 定期检查每个 Store 的 StoreFile 数量
+ 如果数量 >= `threshold`，触发 Compaction 请求

**Step 2：文件选择**

+ 根据选择算法确定参与 Compaction 的 StoreFile 列表
+ 排除正在被扫描引用的文件

**Step 3：获取 Compaction 锁**

+ 申请 <font style="background-color:#E8F7CF;">Region 级别的 Compaction 锁</font>
+ 防止并发 Compaction 导致数据不一致

**Step 4：执行合并**

1. 顺序读取所有选中文件的数据
2. 按 RowKey+Timestamp 排序合并
3. <font style="background-color:#E8F7CF;">过滤删除标记和过期版本（Major 才清理）</font>
4. 写入新的 StoreFile 到 HDFS

**Step 5：原子切换**

1. 新文件写入完成后，原子更新 Region 元数据
2. 标记旧文件为待删除
3. 等待所有读取完成后删除旧文件

**Step 6：清理旧文件**

+ 异步删除旧的 StoreFile，释放 HDFS 空间

# Compaction 关键参数详解
## 核心配置参数表
| 参数名 | 默认值 | 说明 | 调优方向 |
| --- | --- | --- | --- |
| `hbase.hstore.compaction.threshold` | 3 | StoreFile 数量达到此值触发 Minor Compaction | 增大可减少 Compaction 频率，但文件数会增加 |
| `hbase.hstore.compaction.max.size` | 2GB | 超过此大小的文件不参与 Minor Compaction | 增大可合并更多文件，但 IO 开销增加 |
| `hbase.hstore.compaction.min.size` | 1MB | 小于此大小的文件优先参与 Compaction | 减小可更快合并小文件 |
| `hbase.hregion.majorcompaction` | 7 天 | Major Compaction 自动触发周期 | 缩短可更快清理数据，但资源消耗大 |
| `hbase.hstore.compaction.max` | 10 | 单次 Compaction 最多合并的文件数 | 增大可一次合并更多文件，但耗时增加 |
| `hbase.hstore.compaction.min` | 3 | 单次 Compaction 最少合并的文件数 | 减小可更频繁触发 Compaction |
| `hbase.regionserver.compaction.handler.count` | 3 | Compaction 并发线程数 | 增大可提升 Compaction 并发度 |


## 不同场景的参数配置
### 场景 1：写多读少（日志存储）
方向：尽量减少 Compaction

```xml
<configuration>
    <!-- 增大阈值，减少 Compaction 频率 -->
    <property>
        <name>hbase.hstore.compaction.threshold</name>
        <value>5</value>
    </property>
    
    <!-- 增大文件大小上限，允许合并更多文件 -->
    <property>
        <name>hbase.hstore.compaction.max.size</name>
        <value>4294967296</value>  <!-- 4GB -->
    </property>
    
    <!-- 延长 Major Compaction 周期 -->
    <property>
        <name>hbase.hregion.majorcompaction</name>
        <value>1209600000</value>  <!-- 14 天 -->
    </property>
</configuration>
```

### 场景 2：读多写少（用户画像查询）
方向：增加 Compaction

```xml
<configuration>
    <!-- 减小阈值，更频繁合并文件 -->
    <property>
        <name>hbase.hstore.compaction.threshold</name>
        <value>2</value>
    </property>
    
    <!-- 减小文件大小上限，避免大文件影响读性能 -->
    <property>
        <name>hbase.hstore.compaction.max.size</name>
        <value>1073741824</value>  <!-- 1GB -->
    </property>
    
    <!-- 缩短 Major Compaction 周期，及时清理 -->
    <property>
        <name>hbase.hregion.majorcompaction</name>
        <value>259200000</value>  <!-- 3 天 -->
    </property>
</configuration>
```

### 场景 3：读写均衡（订单系统）
方向：默认，但增加 Compaction 并发

```xml
<configuration>
    <!-- 默认配置，平衡读写性能 -->
    <property>
        <name>hbase.hstore.compaction.threshold</name>
        <value>3</value>
    </property>
    
    <property>
        <name>hbase.hstore.compaction.max.size</name>
        <value>2147483648</value>  <!-- 2GB -->
    </property>
    
    <property>
        <name>hbase.hregion.majorcompaction</name>
        <value>604800000</value>  <!-- 7 天 -->
    </property>
    
    <!-- 增加 Compaction 并发度 -->
    <property>
        <name>hbase.regionserver.compaction.handler.count</name>
        <value>5</value>
    </property>
</configuration>
```

# Compaction 调优策略
## 调优目标与指标
| 调优目标 | 监控指标 | 健康范围 |
| --- | --- | --- |
| 减少 Compaction 对写入的影响 | Compaction 队列长度 | < 5 |
| 提升读性能 | StoreFile 数量/Store | < 10 |
| 控制存储膨胀 | 删除标记比例 | < 20% |
| 避免 Compaction 风暴 | 并发 Compaction 数 | < RegionServer 数 × 2 |
| 保证数据及时清理 | Major Compaction 完成时间 | < 2 小时 |


## 常见问题与解决方案
| 问题 | 现象 | 原因 | 解决方案 |
| --- | --- | --- | --- |
| Compaction 队列堆积 | 待 Compaction 文件越来越多 | Compaction 线程数不足 | 增加 `compaction.handler.count` |
| 读性能下降 | StoreFile 数量过多 | Compaction 触发太慢 | 减小 `compaction.threshold` |
| 写入阻塞 | 写入请求被阻塞 | StoreFile 数量达到上限 | 增大 `blockingStoreFiles` |
| Major Compaction 耗时过长 | 单次 Compaction 超过数小时 | 文件太大或太多 | 调整 `max.size`<br/>，分批次执行 |
| 存储空间增长快 | 删除数据后空间未释放 | Major Compaction 周期太长 | 缩短 `majorcompaction`周期 |


## 调优实战案例
案例：电商订单表 Compaction 调优

问题背景

+ 日增订单 5000 万条
+ StoreFile 数量持续增长，峰值达 50 个/Store
+ 读查询延迟从 10ms 上升到 200ms
+ Compaction 队列长度持续 > 20



问题分析

1. Compaction 阈值默认 3，但文件增长太快
2. Compaction 线程数默认 3，处理不过来
3. 大文件（>2GB）不参与 Minor Compaction，堆积严重
4. Major Compaction 7 天一次，删除标记清理不及时



调优方案

+ Step 1：增加 Compaction 并发度

```xml
<property>
    <name>hbase.regionserver.compaction.handler.count</name>
    <value>8</value>  <!-- 从 3 提升到 8 -->
</property>
```

+ Step 2：调整文件选择策略

```xml
<property>
    <name>hbase.hstore.compaction.max.size</name>
    <value>4294967296</value>  <!-- 2GB → 4GB -->
</property>

<property>
    <name>hbase.hstore.compaction.threshold</name>
    <value>4</value>  <!-- 3 → 4，减少频繁触发 -->
</property>
```

+ Step 3：启用 Exploring Compaction

```xml
<property>
    <name>hbase.hstore.compaction.policy</name>
    <value>org.apache.hadoop.hbase.regionserver.ExploringCompactionPolicy</value>
</property>
```

+ Step 4：手动触发 Major Compaction（紧急）
    - `hbase shell> major_compact 'order_table'`



调优效果

+ StoreFile 数量：50 → 8 个/Store
+ 读查询延迟：200ms → 15ms
+ Compaction 队列：20 → 2
+ 存储空间：释放 30%

# 高级 Compaction 特性
## 分区 Compaction（Compaction Partition）
```plain
┌─────────────────────────────────────────────────────────────────┐
│                    分区 Compaction 机制                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  HBase 2.0+ 支持将大 Compaction 任务拆分为多个小任务并行执行：     │
│                                                                 │
│  传统 Compaction:                                                │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐                       │
│  │ F1  │ │ F2  │ │ F3  │ │ F4  │ │ F5  │  →  串行合并           │
│  └─────┘ └─────┘ └─────┘ └─────┘ └─────┘                       │
│                                                                 │
│  分区 Compaction:                                                │
│  ┌─────┐ ┌─────┐        ┌─────┐ ┌─────┐                        │
│  │ F1  │ │ F2  │  → P1  │ F3  │ │ F4  │  → P2  →  并行合并      │
│  └─────┘ └─────┘        └─────┘ └─────┘                        │
└─────────────────────────────────────────────────────────────────┘
```

优势：                                                         

+ 缩短 Compaction 总耗时                                        
+ 减少对读写请求的影响                                         
+ 更好地利用多核 CPU      

## 节流 Compaction（Throttling）
```xml
<!-- 控制 Compaction 的 IO 带宽，避免影响正常读写 -->

<property>
    <name>hbase.regionserver.compaction.throttle.enabled</name>
    <value>true</value>
    <description>启用 Compaction 节流</description>
</property>

<property>
    <name>hbase.regionserver.compaction.throttle.bytes.per.sec</name>
    <value>104857600</value>  <!-- 100MB/s -->
    <description>Compaction 最大 IO 带宽</description>
</property>

<property>
    <name>hbase.regionserver.compaction.throttle.offpeak.hours</name>
    <value>0-6</value>
    <description>非高峰时段（可全速 Compaction）</description>
</property>
```

# 问题
| 问题 | 参考答案要点 |
| --- | --- |
| 如何优化 Compaction 性能？ | 调整阈值、增加线程数、启用分区 Compaction、设置节流 |
| Compaction 队列堆积如何排查？ | 检查线程数、文件大小、IO 带宽、磁盘空间 |
| 为什么 Major Compaction 不能频繁执行？ | 资源消耗大、可能阻塞读写、IO 压力大 |
| 如何平衡 Compaction 频率和读性能？ | 根据读写比例调整 threshold 和 max.size |
| Compaction 期间数据一致性如何保证？ | 原子切换文件、旧文件等待读取完成后删除 |
