Region 拆分是 HBase 自动扩展的核心机制

# Region 拆分基本概念
## 什么是 Region 拆分
+ 定义：当 Region 大小超过阈值时，自动分裂为两个更小的 Region
+ 目的：实现数据均匀分布，支持水平扩展
+ 单位：Region 是 HBase 负载均衡和故障恢复的基本单位
+ 结果：1 个 Region → 2 个 Region，数据量减半

## Region 拆分示意图
拆分前：

```plain
┌─────────────────────────────────────────────────────────────┐
│                      Region A                               │
│  RowKey 范围：user_0001 ~ user_9999                          │
│  数据大小：12GB（超过阈值 10GB）                              │
│  所在节点：RegionServer-1                                    │
└─────────────────────────────────────────────────────────────┘
```

拆分后：

```plain
┌─────────────────────────────────┐  ┌─────────────────────────────────┐
│         Region A1               │  │         Region A2               │
│  RowKey: user_0001 ~ user_4999  │  │  RowKey: user_5000 ~ user_9999  │
│  大小：6GB                       │  │  大小：6GB                      │
│  节点：RegionServer-1            │  │  节点：RegionServer-1/2         │
└─────────────────────────────────┘  └─────────────────────────────────┘
```

## 拆分与负载均衡的关系
| 操作 | 触发条件 | 执行者 | 目的 |
| --- | --- | --- | --- |
| Region 拆分 | Region 大小超限 | RegionServer | 控制单个 Region 大小 |
| Region 分配 | 新 Region 产生 | HMaster | 将 Region 分配给 RegionServer |
| Region 迁移 | 负载不均衡 | HMaster | 平衡集群负载 |

# 拆分触发条件
## 主要触发条件
| 触发条件 | 配置参数 | 默认值 | 说明 |
| --- | --- | --- | --- |
| Region 文件大小 | `hbase.hregion.max.filesize` | 10GB | StoreFile 总大小超过此值触发 |
| MemStore 大小 | `hbase.hregion.memstore.flush.size` | 128MB | 间接触发（Flush 后文件增多） |
| 手动触发 | 通过 HBase Shell 或 API 手动拆分 | | |
| 预分区创建 | 建表时指定 splits 自动创建多 Region | | |


## 触发条件配置
```xml
<!-- hbase-site.xml 关键配置 -->

<!-- Region 最大文件大小 -->
<property>
    <name>hbase.hregion.max.filesize</name>
    <value>10737418240</value>
    <description>Region 大小超过 10GB 触发拆分</description>
</property>

<!-- MemStore 刷新阈值（间接影响） -->
<property>
    <name>hbase.hregion.memstore.flush.size</name>
    <value>134217728</value>
    <description>单个 MemStore 达到 128MB 触发 Flush</description>
</property>

<!-- RegionServer 全局 MemStore 限制 -->
<property>
    <name>hbase.regionserver.global.memstore.size</name>
    <value>0.4</value>
    <description>RegionServer 堆内存 40% 用于 MemStore</description>
</property>

<!-- 拆分请求处理线程数 -->
<property>
    <name>hbase.regionserver.split.count</name>
    <value>3</value>
    <description>并发处理拆分请求的线程数</description>
</property>
```

# Region 拆分详细过程
## 拆分流程总览
**Step 1：检测拆分条件**

+ RegionServer 定期检查每个 Region 的大小
+ 当 StoreFile 总大小 > `hbase.hregion.max.filesize` 时触发

**Step 2：准备拆分**

1. 暂停对该 Region 的写入请求
2. 等待正在进行的写入完成
3. 刷新所有 MemStore 到磁盘

**Step 3：确定拆分点**

1. 扫描所有 StoreFile 的 Key
2. 找到中间位置的 RowKey 作为拆分点
3. 确保拆分点不在 RowKey 中间（保证完整性）

**Step 4：创建子 Region**

1. 创建 daughter Region A（`startKey ~ splitPoint`）
2. 创建 daughter Region B（`splitPoint ~ endKey`）
3. 继承父 Region 的列族配置

**Step 5：更新元数据**

1. 在 `hbase:meta` 表中注册两个新 Region
2. 标记父 Region 为 `SPLITTING` 状态
3. 通知 HMaster 拆分完成

**Step 6：清理父 Region**

1. 将父 Region 标记为 `SPLIT`
2. 异步删除父 Region 的引用文件
3. 保留文件供子 Region 引用（避免数据复制）

**Step 7：负载均衡（可选）**

+ HMaster 可能将其中一个子 Region 迁移到其他 RegionServer
+ 实现集群负载均衡

## 拆分点选择算法
拆分点选择策略 

+ 原则 1：中间位置
    - 扫描所有 StoreFile 的 Key，找到字典序中间位置的 RowKey
    - 确保两个子 Region 数据量大致相等
+ 原则 2：避免行中间拆分
    - 拆分点必须是完整的 RowKey，不能在某行的中间
    - HBase 会自动调整到最近的完整 RowKey 边界
+ 原则 3：考虑列族分布
    - 如果不同列族数据量差异大，可能需要特殊处理
    - 默认按主列族数据量计算

## 拆分过程状态流转
Region 状态流转图：

```latex
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  OPEN   │───►│SPLITTING│───►│  SPLIT  │───►│ CLOSED  │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
     ▲                                          │
     │                                          │
     └──────────────────────────────────────────┘
                    (新 Region)
```



Region 状态说明：

| 状态  | 说明 |
| --- | --- |
| `OPEN` | 正常服务状态，可读写                             │ |
| `SPLITTING` | 拆分进行中，暂停写入，允许读取 |
| `SPLIT` | 拆分完成，父 Region 标记为已拆分 |
| `CLOSED` | 已关闭，等待清理 |

# Region 拆分策略
## 拆分策略类型
| 策略名称 | 类名 | 说明 | 适用场景 |
| --- | --- | --- | --- |
| 默认策略 | `ConstantSizeRegionSplitPolicy` | 达到固定大小阈值即拆分 | 通用场景 |
| 键前缀策略 | `KeyPrefixRegionSplitPolicy` | 按 RowKey 前缀拆分 | 有明确前缀模式的场景 |
| 递增策略 | `IncreasingToUpperBoundRegionSplitPolicy` | HBase 1.0+ 默认，动态调整阈值 | 大表场景 |
| 自定义策略 | 继承 `RegionSplitPolicy` | 根据业务需求定制 | 特殊业务场景 |


## 策略配置方式
```xml
<!-- 配置 Region 拆分策略 -->
<property>
    <name>hbase.regionserver.region.split.policy</name>
    <value>org.apache.hadoop.hbase.regionserver.ConstantSizeRegionSplitPolicy</value>
    <description>指定 Region 拆分策略类</description>
</property>
```

```java
// 建表时指定拆分策略
HTableDescriptor tableDesc = new HTableDescriptor(TableName.valueOf("my_table"));
tableDesc.setValue(
    HConstants.SPLIT_POLICY, 
    "org.apache.hadoop.hbase.regionserver.KeyPrefixRegionSplitPolicy"
);
```

## 各策略详细对比
| 维度 | 默认策略 | 键前缀策略 | 递增策略 |
| --- | --- | --- | --- |
| 拆分阈值 | 固定 10GB | 固定 10GB | 动态增长 |
| 拆分点选择 | 中间 RowKey | 前缀边界 | 中间 RowKey |
| Region 数量增长 | 线性 | 按前缀数 | 先快后慢 |
| 适用场景 | 通用 | 多租户/多业务线 | 超大表 |
| 优点 | 简单可控 | 业务隔离好 | 减少初期拆分 |
| 缺点 | 可能拆分过频 | 前缀设计要合理 | 阈值难预测 |

# 预分区 vs 自动拆分
## 两种方案对比
| 维度 | 预分区 | 自动拆分 |
| --- | --- | --- |
| 创建时机 | 建表时指定 | 运行时自动触发 |
| Region 数量 | 预先确定 | 动态增长 |
| 拆分开销 | 无运行时拆分开销 | 有拆分开销 |
| 热点风险 | 可预先规避 | 初期可能有热点 |
| 适用场景 | 数据量可预估 | 数据量不确定 |
| 推荐程度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |


## 预分区实现代码
```java
// 方案 1: 简单预分区（按数量）
public static void createTableWithPreSplits(Connection conn) throws Exception {
    HBaseAdmin admin = (HBaseAdmin) conn.getAdmin();
    TableName tableName = TableName.valueOf("user_order");
    
    HTableDescriptor tableDesc = new HTableDescriptor(tableName);
    tableDesc.addFamily(new HColumnDescriptor("cf1"));
    tableDesc.addFamily(new HColumnDescriptor("cf2"));
    
    // 创建 10 个预分区
    byte[][] splits = new byte[9][];
    for (int i = 1; i < 10; i++) {
        splits[i - 1] = Bytes.toBytes(String.format("%02d", i * 1000));
    }
    
    admin.createTable(tableDesc, splits);
    admin.close();
}


// 方案 2: 哈希预分区（均匀分布）
public static void createTableWithHashSplits(Connection conn) throws Exception {
    HBaseAdmin admin = (HBaseAdmin) conn.getAdmin();
    TableName tableName = TableName.valueOf("user_log");
    
    HTableDescriptor tableDesc = new HTableDescriptor(tableName);
    tableDesc.addFamily(new HColumnDescriptor("info"));
    
    // 按哈希值创建 16 个分区
    byte[][] splits = new byte[15][];
    for (int i = 1; i < 16; i++) {
        splits[i - 1] = Bytes.toBytes(String.format("%02x", i * 16));
    }
    
    admin.createTable(tableDesc, splits);
    admin.close();
}


// 方案 3: 业务前缀预分区（按业务场景）
public static void createTableWithBusinessSplits(Connection conn) throws Exception {
    HBaseAdmin admin = (HBaseAdmin) conn.getAdmin();
    TableName tableName = TableName.valueOf("multi_tenant");
    
    HTableDescriptor tableDesc = new HTableDescriptor(tableName);
    tableDesc.addFamily(new HColumnDescriptor("data"));
    
    // 按租户前缀分区
    byte[][] splits = new byte[][] {
        Bytes.toBytes("tenant_a"),
        Bytes.toBytes("tenant_b"),
        Bytes.toBytes("tenant_c"),
        Bytes.toBytes("tenant_d"),
        Bytes.toBytes("tenant_e")
    };
    
    admin.createTable(tableDesc, splits);
    admin.close();
}
```

## 预分区数量估算公式
公式 1：基于数据量

+ Region 数量 = 预估总数据量 / 单 Region 目标大小
+ 示例：500GB / 10GB = 50 个 Region



公式 2：基于 RegionServer 数量

+ Region 数量 = RegionServer 数量 × 2~3
+ 示例：20 个 RegionServer × 2.5 = 50 个 Region



公式 3：基于并发查询量

+ Region 数量 = 峰值 QPS / 单 Region 最大 QPS
+ 示例：10000 QPS / 200 QPS = 50 个 Region



建议：取以上三个公式的最大值，并向上取整到 2 的幂次

# 拆分对系统的影响
## 性能影响分析
| 影响维度 | 影响程度 | 持续时间 | 缓解措施 |
| --- | --- | --- | --- |
| 写入性能 | 高（暂停写入） | 秒级~分钟级 | 避开业务高峰 |
| 读取性能 | 中（元数据更新） | 秒级 | 客户端缓存刷新 |
| HMaster 负载 | 中（元数据更新） | 秒级 | HMaster 独立部署 |
| hbase:meta 表 | 中（记录增加） | 永久 | 定期 Major Compaction |
| RegionServer 内存 | 低 | 短暂 | 预留内存余量 |


## 拆分期间的常见问题
问题 1：写入短暂阻塞

+ 现象：拆分期间写入请求超时或重试
+ 原因：Region 暂停写入等待 Flush 完成
+ 解决：客户端配置合理重试，避开高峰拆分



问题 2：客户端路由失效

+ 现象：客户端缓存的 Region 位置信息过期
+ 原因：拆分后 Region 位置变化
+ 解决：客户端自动刷新缓存，配置合理 TTL



问题 3：拆分风暴

+ 现象：大量 Region 同时拆分，集群负载飙升
+ 原因：建表后数据快速增长，触发批量拆分
+ 解决：预分区、调整拆分阈值、限流

## 监控指标
| 指标名称 | 含义 | 健康范围 | 告警阈值 |
| --- | --- | --- | --- |
| `regionCount` | 集群 Region 总数 | 稳定增长 | 突增 > 50% |
| `splitRequestCount` | 拆分请求数 | < 10/分钟 | > 50/分钟 |
| `splitQueueLength` | 拆分队列长度 | < 5 | > 20 |
| `splitTimeAvg` | 平均拆分耗时 | < 30 秒 | > 2 分钟 |
| `compactingSplitCount` | 拆分中 Region 数 | < Region 总数 5% | > 10% |

# 拆分调优最佳实践
## 调优配置汇总
```xml
<!-- hbase-site.xml 拆分相关配置汇总 -->

<configuration>
    <!-- Region 最大大小（根据场景调整） -->
    <property>
        <name>hbase.hregion.max.filesize</name>
        <value>10737418240</value>
    </property>
    
    <!-- 拆分策略 -->
    <property>
        <name>hbase.regionserver.region.split.policy</name>
        <value>org.apache.hadoop.hbase.regionserver.IncreasingToUpperBoundRegionSplitPolicy</value>
    </property>
    
    <!-- 拆分并发线程数 -->
    <property>
        <name>hbase.regionserver.split.count</name>
        <value>3</value>
    </property>
    
    <!-- Meta 表 Region 大小（单独配置） -->
    <property>
        <name>hbase.hregion.max.filesize.meta</name>
        <value>2147483648</value>
    </property>
    
    <!-- 拆分后是否立即负载均衡 -->
    <property>
        <name>hbase.master.loadbalance.bytable</name>
        <value>false</value>
    </property>
</configuration>
```

## 调优场景建议
| 场景 | 推荐配置 | 理由 |
| --- | --- | --- |
| 写入密集型 | 增大 max.filesize 到 20GB | 减少拆分频率，提升写入吞吐 |
| 读取密集型 | 减小 max.filesize 到 5GB | 增加 Region 数量，提升并发读取 |
| 大表场景 | 使用递增策略 + 预分区 | 避免初期频繁拆分 |
| 多租户场景 | 使用键前缀策略 | 按租户隔离，便于管理 |
| 时序数据 | 预分区 + 时间反转 RowKey | 避免热点，支持范围查询 |


## 拆分问题排查流程
Step 1：确认拆分状态

+ `hbase shell> status 'detailed'`
+ 查看是否有 Region 处于 SPLITTING 状态

Step 2：检查拆分队列

+ 查看 RegionServer UI 或 JMX 指标
+ splitQueueLength > 20 说明拆分积压

Step 3：分析 Region 大小分布

+ `hbase shell> table_size 'table_name'`
+ 查看是否有 Region 远超平均值

Step 4：检查 HMaster 日志

+ `grep "split" hbase-master.log`
+ 查看拆分失败或异常信息

Step 5：检查 RegionServer 日志

+ `grep "Splitting region" hbase-regionserver.log`
+ 查看拆分详细过程和耗时

# 问题


| 问题 | 参考答案要点 |
| --- | --- |
