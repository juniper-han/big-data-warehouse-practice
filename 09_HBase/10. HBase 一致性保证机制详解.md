HBase 是一个典型的 **CP 系统**（Consistency + Partition Tolerance）

在分布式系统 CAP 定理中选择了**牺牲可用性**来保证强一致性。

# CAP 定理基础
## CAP 三要素
+ 一致性：Consistency，所有节点在同一时间看到的数据相同
+ 可用性：Availability，每个请求都能收到响应（不保证最新）
+ 分区容错性：Partition Tolerance，网络分区时系统仍能运行

## CAP 定理核心结论
核心结论：分布式系统最多只能同时满足 CAP 中的两个

+ CA：一致性，可用性
    - ❌ 无分区容错
    - 传统关系数据库（单节点）
+ CP：一致性，分区容错性
    - ❌ 牺牲可用性
    - HBase/ZK，ZooKeeper
+ AP：可用性，分区容错性
    - ❌ 牺牲一致性
    - Cassandra，DynamoDB



⚠️ 注意：

+ 网络分区在分布式系统中不可避免，所以 P 必须选择          
+ 实际选择：CP 或 AP                             

## HBase 的 CAP 选择
| 选择 | 说明 | 体现 |
| --- | --- | --- |
| 保证 C | 强一致性 | 读写都访问最新数据 |
| 保证 P | 分区容错 | 网络分区时集群仍能运行 |
| 牺牲 A | 可用性降低 | 故障时部分数据暂时不可用 |

# HBase 一致性保证机制
## 一致性级别定义
| 级别 | 说明 | HBase 支持 |
| --- | --- | --- |
| 强一致性 | **读操作，总能读到最新写入的数据** | ✅ 支持 |
| 弱一致性 | 读操作可能读到旧数据 | ❌ 不支持 |
| 最终一致性 | 一段时间后所有节点数据一致 | ❌ 不支持 |


## HBase 强一致性的体现
场景 1：**单行读写，一致性** 

+ ✅ 保证：写入成功后，后续读操作一定能读到最新值    

```plain
Client A 写入：Put(row1, value1)             
     ↓                                      
Client B 读取：Get(row1) → 一定返回 value1    
```



场景 2：**<font style="background-color:#E8F7CF;">行级原子性</font>**

+ 对同一行的多列更新：                       

```java
Put(row1, cf1:c1, v1)                  
Put(row1, cf1:c2, v2)                  
Put(row1, cf1:c3, v3)   
```

+ ✅ 保证：同一行，的所有**<font style="background-color:#E8F7CF;">列更新是原子的</font>**         
    - 读者要么看到更新前的值，要么看到更新后的值 
    - 不会看到中间状态                       



场景 3：检查点操作（CheckAndPut）

+ 条件写入：只有当某列等于期望值时才写入
    - `checkAndPut(row, cf, col, expectValue, newPut)`
+ ✅ 保证：检查和新写入是原子操作
    - 不会出现检查通过后被其他客户端修改的情况

## 一致性保证的核心机制
| 机制 | 作用 | 一致性贡献 |
| --- | --- | --- |
| 单 RegionServer 写入 | 同一 RowKey 的写入总是路由到同一 RegionServer | 避免并发写入冲突 |
| 行级锁 | 同一行的操作互斥执行 | 保证行级原子性 |
| WAL 顺序写入 | 写操作先写日志再写内存 | 保证故障后数据可恢复 |
| MemStore 跳表 | 内存中数据有序存储 | 保证读写顺序一致 |
| Region 唯一归属 | 每个 Region 只属于一个 RegionServer | 避免数据多副本不一致 |

# 读写流程中的一致性保证
## 写操作：一致性保证流程
**Step 1：Client 定位 RegionServer**

+ 通过 hbase:meta 表查询 RowKey 所属 Region
+ 获取该 Region 所在的 RegionServer 地址
+ 缓存到客户端，后续请求直接使用
+ ✅ 一致性保证：同一 RowKey 总是路由到同一 RegionServer

**Step 2：RegionServer 获取行锁**

+ 对该 RowKey 加锁（隐性锁，通过同步机制实现）
+ 同一行的并发写入串行执行
+ ✅ 一致性保证：避免并发写入导致数据不一致

**Step 3：写入 WAL（同步）**

+ 先写 WAL 日志到 HDFS
+ 默认每条写入都 hsync 同步到磁盘
+ 可配置为异步（牺牲部分一致性换性能）
+ ✅ 一致性保证：故障后数据可恢复，不丢失

**Step 4：写入 MemStore**

+ WAL 成功后写入内存跳表
+ 跳表按 RowKey+Timestamp 排序
+ 立即对后续读可见
+ ✅ 一致性保证：写入成功后立即可读

**Step 5: 返回成功给 Client**

+ 只有 WAL 和 MemStore 都成功后才返回
+ Client 收到成功后可立即读取最新数据
+ ✅ 一致性保证：写入成功 = 数据持久化 + 可读

## 读操作：一致性保证流程
**Step 1：Client 定位 RegionServer（同写流程）**

+ ✅ 保证：读写请求路由到同一 RegionServer

**Step 2：按顺序查询多个数据源**

+ 查询顺序：BlockCache（缓存）→  MemStore（内存）→ StoreFile
+ ✅ 一致性保证：优先查最新数据源（MemStore 最新）

**Step 3：合并多个版本数据**

+ MemStore 和 StoreFile 中的数据按 Timestamp 排序
+ 返回最新版本的值（或指定版本）
+ 过滤掉删除标记
+ ✅ 一致性保证：总是返回最新有效数据

**Step 4：返回结果给 Client**

+ 返回的数据是合并后的最新状态
+ Client 无需关心数据来源
+ ✅ 一致性保证：读操作总能读到最新写入的数据

## 读写一致性代码示例
```java
// 示例 1: 写后立即可读（强一致性）
public void testWriteReadConsistency(Connection conn) throws Exception {
    Table table = conn.getTable(TableName.valueOf("test_table"));
    
    // 写入数据
    Put put = new Put(Bytes.toBytes("row1"));
    put.addColumn(Bytes.toBytes("cf1"), Bytes.toBytes("name"), 
                  Bytes.toBytes("张三"));
    table.put(put);  // 写入成功
    
    // 立即读取
    Get get = new Get(Bytes.toBytes("row1"));
    Result result = table.get(get);
    
    // ✅ 保证能读到最新值
    String name = Bytes.toString(result.getValue(
        Bytes.toBytes("cf1"), Bytes.toBytes("name")));
    assert "张三".equals(name);  // 一定成立
}

// 示例 2: 检查点操作（原子性）
public void testCheckAndPut(Connection conn) throws Exception {
    Table table = conn.getTable(TableName.valueOf("test_table"));
    
    // 条件写入：只有 age=25 时才更新为 26
    boolean success = table.checkAndPut(
        Bytes.toBytes("row1"),
        Bytes.toBytes("cf1"),
        Bytes.toBytes("age"),
        Bytes.toBytes(25),  // 期望值
        new Put(Bytes.toBytes("row1"))
            .addColumn(Bytes.toBytes("cf1"), Bytes.toBytes("age"), 
                      Bytes.toBytes(26))
    );
    
    // ✅ 保证：检查和写入是原子操作
    // 不会出现检查通过后被其他客户端修改的情况
}

// 示例 3: 行级原子更新
public void testRowAtomicity(Connection conn) throws Exception {
    Table table = conn.getTable(TableName.valueOf("test_table"));
    
    // 同一行的多列更新
    Put put = new Put(Bytes.toBytes("row1"));
    put.addColumn(Bytes.toBytes("cf1"), Bytes.toBytes("name"), 
                  Bytes.toBytes("张三"));
    put.addColumn(Bytes.toBytes("cf1"), Bytes.toBytes("age"), 
                  Bytes.toBytes(25));
    put.addColumn(Bytes.toBytes("cf1"), Bytes.toBytes("city"), 
                  Bytes.toBytes("北京"));
    
    table.put(put);  // ✅ 整行更新是原子的
    
    // 其他客户端要么看到更新前的整行，要么看到更新后的整行
    // 不会看到 name 已更新但 age 未更新的中间状态
}
```

# 故障场景下的一致性处理
## RegionServer 故障场景
RegionServer 故障时的一致性保证

+ 故障前：

```plain
┌─────────────────────────────────────────────────────────┐   
│  RegionServer-1                                          │  
│  ┌─────────────┐  ┌─────────────┐                        │  
│  │  Region A   │  │  Region B   │                        │  
│  │  MemStore   │  │  MemStore   │                        │  
│  │  (部分数据)  │  │  (部分数据)  │                        │   
│  └─────────────┘  └─────────────┘                        │  
│         │                │                                │ 
│         ▼                ▼                                │ 
│  ┌─────────────────────────────────┐                      │ 
│  │              WAL                 │                      │
│  │  (已持久化到 HDFS)                │                      │ 
│  └─────────────────────────────────┘                      │ 
└─────────────────────────────────────────────────────────┘   
```

+ 故障后恢复流程
    - Step 1： ZooKeeper 检测 RegionServer 心跳丢失（30 秒）
    - Step 2： HMaster 标记 RegionServer 为 `Dead`
    - Step 3：HMaster 将 Region A/B 重新分配给其他 RegionServer
    - Step 4：新 RegionServer 加载 WAL 并回放
    - Step 5：恢复 MemStore 中的数据
    - Step 6：Region 恢复服务，对外提供读写
+ ✅ 一致性保证：WAL 回放确保故障前已确认的写入不丢失



⚠️ 注意：故障期间该 Region 的数据暂时不可用（牺牲可用性） 

## 网络分区场景
网络分区时的一致性保证



场景：集群被网络分区为两部分

```plain
                                                         
┌─────────────────────┐         ┌─────────────────────┐  
│     分区 A           │         │     分区 B           │ 
│  ┌───────────────┐  │         │  ┌───────────────┐  │  
│  │   HMaster     │  │         │  │ RegionServer  │  │  
│  │   (Active)    │  │         │  │     -2,3      │  │  
│  └───────────────┘  │         │  └───────────────┘  │  
│  ┌───────────────┐  │         │                     │  
│  │ RegionServer  │  │         │  ⚠️ 无法与 HMaster   │ 
│  │     -1        │  │         │     通信            │  
│  └───────────────┘  │         │                     │  
└─────────────────────┘         └─────────────────────┘  
         │                              │                
         └────────── 网络中断 ──────────┘                 
```



HBase 处理方式：

1. 分区 B 中的 RegionServers 无法与 HMaster 通信
2. ZooKeeper 会话超时，分区 B 中的 RegionServers 被标记为 `Dead`
3. HMaster 将分区 B 的 Region 重新分配给分区 A 中的节点
4. 分区 B 中的 RegionServers 检测到会话过期，主动关闭
5. 分区 A 继续提供服务，分区 B 暂时不可用

✅ 一致性保证：同一 Region 不会同时在两个分区提供服务

❌ 可用性牺牲：分区 B 的 Region 暂时不可用

## 故障场景对比
| 故障类型 | 一致性影响 | 可用性影响 | 恢复方式 |
| --- | --- | --- | --- |
| RegionServer 宕机 | 无影响（WAL 保证） | 部分数据暂时不可用 | WAL 回放恢复 |
| 网络分区 | 无影响（单主保证） | 分区 B 数据不可用 | 分区合并后恢复 |
| HMaster 故障 | 无影响（只影响 DDL） | 无法创建/修改表 | Standby HMaster 接管 |
| ZooKeeper 故障 | 无影响 | 集群不可用 | ZK 集群恢复 |
| HDFS 故障 | 可能数据丢失 | 集群不可用 | HDFS 恢复 + WAL 回放 |

# HBase 一致性的局限性
## 一致性边界
| 范围 | 一致性保证 | 说明 |
| --- | --- | --- |
| **单行操作** | ✅ 强一致 | 单行的读写和更新是原子的 |
| 多行操作 | ❌ 不保证 | 多行之间没有原子性保证 |
| 跨表操作 | ❌ 不保证 | 不同表之间没有事务保证 |
| 跨区域操作 | ❌ 不保证 | 跨 Region 操作没有原子性 |


### 单行一致性（HBase 支持）
+ 定义：对同一行数据的读写操作保证强一致性
+ 范围：同一 RowKey 内的所有列（Column）
+ 保证：写入成功后，后续读操作一定能读到最新值
+ 技术实现
    - 行级锁：同一行的操作互斥执行，隐性锁，通过 RegionServer 单线程处理
    - 单节点路由：同一 RowKey 总是路由到同一 RegionServer
    - 原子写入：整行写入是原子的，Put 操作原子性保证
    - 版本控制：通过 Timestamp 区分版本
+ HBase 支持：通过行级锁 + 单 RegionServer

### 多行一致性（HBase 不支持）
+ 定义：对同一表内多行数据的操作保证一致性
+ 范围：同一表内的多个 RowKey
+ 保证：多行操作要么全部成功，要么全部失败（原子性）
+ 技术实现难点
    - 多行锁：需要同时锁定多行，解决方案：两阶段锁（2PL）
    - 分布式协调：多行可能在不同 Region
    - 故障恢复：部分成功时需要回滚
    - 性能开销：锁竞争和协调开销大
+ HBase：原生不支持，需 Phoenix 或协处理器，对性能的影响高

### 跨表一致性（HBase 不支持）
+ 定义：对不同表之间的操作保证一致性
+ 范围：两个或多个不同的表
+ 保证：跨表操作要么全部成功，要么全部失败
+ 技术实现：较难
+ HBase：原生不支持，需 Phoenix 或外部事务框架

### 跨区域一致性（HBase 不支持）
+ 定义：跨 Region、跨机房、跨地域的操作保证一致性
+ 范围：不同 RegionServer、不同集群、不同地域
+ 保证：跨区域操作的数据一致性
+ HBase：原生不支持，需复制（Replication），只能最终一致

一致性模型对比

| 模型 | 说明 | 延迟 | 适用场景 |
| --- | --- | --- | --- |
| 强一致性 | 所有区域同步后才返回 | 高（受最慢区域影响） | 金融核心数据 |
| 弱一致性 | 本地写入后立即返回 | 低 | 日志、缓存 |
| 最终一致性 | 异步同步，最终一致 | 中 | 用户数据、配置 |
| 读写一致 | 读自己的写 | 中 | 会话数据 |


### 综合对比
| 维度 | 单行一致性 | 多行一致性 | 跨表一致性 | 跨区域一致性 |
| --- | --- | --- | --- | --- |
| 范围 | 同一 RowKey | 同一表多行 | 不同表 | 不同 Region/集群 |
| 实现难度 | ⭐ | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| 性能开销 | 低 | 中 | 高 | 很高 |
| HBase 原生支持 | ✅ | ❌ | ❌ | ❌ |
| 典型方案 | 行级锁 | Phoenix 事务 | 分布式事务 | 复制/共识 |
| 一致性级别 | 强一致 | 强一致 | 强一致/最终一致 | 最终一致/强一致 |
| 故障恢复 | 简单 | 中等 | 复杂 | 很复杂 |


## 多行事务解决方案

// 方案 1: 使用 Phoenix 事务（推荐）
public void testPhoenixTransaction(Connection conn) throws Exception {
    // Phoenix 支持 ACID 事务
    Connection phoenixConn = DriverManager.getConnection(
        "jdbc:phoenix:node1,node2,node3");
    phoenixConn.setAutoCommit(false);
    
    try {
        // 多行更新
        PreparedStatement stmt1 = phoenixConn.prepareStatement(
            "UPDATE table1 SET col1=? WHERE rowkey=?");
        stmt1.setString(1, "value1");
        stmt1.setString(2, "row1");
        stmt1.executeUpdate();
        
        PreparedStatement stmt2 = phoenixConn.prepareStatement(
            "UPDATE table2 SET col1=? WHERE rowkey=?");
        stmt2.setString(1, "value2");
        stmt2.setString(2, "row2");
        stmt2.executeUpdate();
        
        // 提交事务
        phoenixConn.commit();
    } catch (Exception e) {
        // 回滚事务
        phoenixConn.rollback();
    }
}

## 局限性总结
| 局限性 | 影响 | 解决方案 |
| --- | --- | --- |
| 单行原子性 | 多行操作不原子 | Phoenix 事务/应用层补偿 |
| 无跨表事务 | 跨表数据可能不一致 | 分布式事务/最终一致性 |
| 故障时不可用 | 部分数据暂时无法访问 | 接受 CP 特性，设计容错 |
| 无读写分离 | 读写都在同一节点 | 使用二级索引/缓存层 |

# 问题
| 问题 | 参考答案要点 |
| --- | --- |
| HBase 为什么选择 CP 而不是 AP？ | 业务场景需要强一致性（金融、订单），数据准确性优先 |
| 网络分区时 HBase 如何处理？ | 分区 B 的 RegionServer 会话过期，Region 重新分配，保证单主 |
| 如何实现 HBase 的多行事务？ | Phoenix 事务、协处理器、应用层补偿（最终一致性） |
| HBase 一致性的边界是什么？ | 单行强一致，多行/跨表不保证 |
| 与 Cassandra 相比 HBase 的优劣势？ | HBase 强一致但可用性低，Cassandra 高可用但最终一致 |
