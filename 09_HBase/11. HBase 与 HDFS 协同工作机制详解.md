HBase 构建在 HDFS 之上

# HBase 与 HDFS 的关系
## 架构定位
| 维度 | HDFS | HBase |
| --- | --- | --- |
| 定位 | 分布式文件系统 | 分布式 NoSQL 数据库 |
| 数据模型 | 文件（File） | 表（Table） |
| 访问方式 | 顺序读写 | 随机读写 |
| 延迟 | 高（秒级） | 低（毫秒级） |
| 适用场景 | 批处理、离线分析 | 实时查询、在线服务 |


## 依赖关系示意图
<img src="https://cdn.nlark.com/yuque/0/2026/png/54426354/1771910599298-91fbe522-5e61-4885-b84a-fd9194b181bf.png" width="517" title="" crop="0,0,1,1" id="ue824c9f7" class="ne-image">

HBase 依赖 HDFS 提供：

+ 数据持久化存储（HFile、WAL）
+ 数据冗余备份（多副本机制）
+ 故障自动恢复（DataNode 宕机自动复制）

## HBase 存储内容对比
| 存储内容 | 存储位置 | HDFS 路径 | 说明 |
| --- | --- | --- | --- |
| HFile | HDFS | `/hbase/data/default/{table}/{region}/{family}/{hfile}` | 实际数据文件 |
| WAL | HDFS | `/hbase/WALs/{regionserver}/{wal}` | 预写日志 |
| Meta 信息 | HDFS + Memory | `/hbase/.meta.表` | Region 位置信息 |
| BlockCache | 内存 | 读缓存，不存 HDFS | |
| MemStore | 内存 | 写缓存，Flush 后存 HDFS | |

# 写流程中 HBase 与 HDFS 的协同
##  写流程中 HDFS 的参与
<img src="https://cdn.nlark.com/yuque/0/2026/png/54426354/1771911614728-24032110-d953-4787-88cd-5eba42f42946.png" width="2826" title="" crop="0,0,1,1" id="ucf65b8d3" class="ne-image">

详细步骤：

+ Step 1：写入 WAL（HDFS）
    - RegionServer 将写入请求追加到 WAL 文件
    - WAL 文件存储在 `HDFS：/hbase/WALs/{rs-host}/{wal}`
    - 默认每条写入都 hsync 同步到 HDFS
    - HDFS 创建新 Block 或追加到现有 Block
+ Step 2：写入 MemStore（内存）
    - WAL 成功后写入内存跳表
    - 此时数据尚未持久化到 HDFS
    - 故障时可通过 WAL 恢复
+ Step 3：Flush 生成 HFile（HDFS）
    - MemStore 达到阈值触发 Flush
    - 内存数据排序后写入临时 HFile
    - HFile 存储在 `HDFS：/hbase/data/{table}/{region}/{cf}/`
    - 临时文件重命名为正式 HFile
+ Step 4: Compaction 合并（HDFS）
    - 多个小 HFile 合并为大 HFile
    - 读取旧 HFile，写入新 HFile 到 HDFS
    - 旧 HFile 标记删除，等待引用释放后物理删除

## WAL 在 HDFS 上的存储细节
| 属性 | 说明 | 配置参数 |
| --- | --- | --- |
| 存储路径 | `/hbase/WALs/{regionserver-host}:{port}/` | `hbase.wal.dir` |
| 文件大小 | WAL 滚动大小 | `hbase.wal.max.filesize`默认`1GB` |
| 同步策略 | 每条写入是否 hsync | `hbase.wal.sync`（默认 `true`） |
| 副本数 | 继承 HDFS 配置 | `dfs.replication`（默认 `3`） |
| 清理策略 | Flush 后可清理 | `hbase.master.cleaner.interval` |


## Flush 过程与 HDFS 交互
Flush 前：

<img src="https://cdn.nlark.com/yuque/0/2026/png/54426354/1771911818663-51fcaae1-9777-417b-b4a0-85a583425405.png" width="2133" title="" crop="0,0,1,1" id="uc6b651c1" class="ne-image">

Flush 触发



HDFS 交互过程：

1. RegionServer 创建 <font style="background-color:#E8F7CF;">HDFS 输出流</font>
    - `FSDataOutputStream out = fs.create(tempFilePath);`
2. 将 MemStore 数据排序后写入临时文件
    - `HFile.Writer.write(key, value);`
3. 关闭输出流，数据同步到 HDFS
    - `out.close();  // hsync`
4. 临时文件重命名为正式 HFile
    - `fs.rename(tempPath, finalPath);  // 原子操作`
5. 更新 Region 元数据，记录新 HFile
    - 通知 HMaster 和 RegionServer  


Flush 后：`HDFS: /hbase/data/default/table/region/cf/hfile_001`

+ 大小：约 100MB（压缩后）
+ 副本数：3
+ Block 数：1（128MB HDFS Block）

# 读流程中 HBase 与 HDFS 的协同
## 读流程中 HDFS 的参与
<img src="https://cdn.nlark.com/yuque/0/2026/png/54426354/1771911992387-25e7b061-f716-48c8-a0e6-648149f78b11.png" width="2673" title="" crop="0,0,1,1" id="u0d7140c4" class="ne-image">

详细步骤

+ Step 1：查询 `BlockCache`（内存），命中则直接返回，无需访问 HDFS
    - 检查请求的 Data Block 是否在缓存中
    - 缓存命中率直接影响 HDFS 读取次数
+ Step 2：查询 `MemStore`（内存），内存跳表查找，无需访问 HDFS
    - 检查最新写入但未 Flush 的数据
+ Step 3：查询 HFile（HDFS）
    - `BlockCache` 和 `MemStore` 未命中时访问 HDFS
    - 通过 HFile 的 Index Block 定位 Data Block
    - <font style="background-color:#E8F7CF;">从 HDFS 读取目标 Data Block 到 BlockCache</font>
    - <font style="background-color:#E8F7CF;">HDFS 读取涉及网络传输和磁盘 IO</font>
+ Step 4：数据返回
    - 合并 `BlockCache`、`MemStore`、HFile 的结果
    - 按 `Timestamp` 返回最新版本
    - 加载的 Data Block 保留在 BlockCache 供后续使用

## HDFS 读取优化机制
| 优化机制 | 说明 | 效果 |
| --- | --- | --- |
| BlockCache | 缓存热点 Data Block | 减少 HDFS 读取 80%+ |
| Bloom Filter | 快速判断 RowKey 是否存在 | 避免无效 HDFS 读取 |
| HFile Index | 内存中存储 Index Block | 快速定位 Data Block |
| HDFS 短路径读取 | 直接读取本地 DataNode | 减少网络开销 |
| 读缓存预热 | 预加载热点数据 | 降低首次读取延迟 |


## HDFS 读取性能影响因素
+ 因素 1：数据本地性
    - 本地读取（RegionServer 与 DataNode 同节点）：
        * 延迟：~5ms
        * 路径：内存 → 本地磁盘
    - 远程读取（RegionServer 与 DataNode 不同节点）：
        * 延迟：~20ms
        * 路径：内存 → 网络 → 远程磁盘
    - 优化：部署时尽量保证 RegionServer 与 DataNode 同节点
+ 因素 2：HDFS 副本选择
    - HDFS 读取策略：
        * 优先读取本地副本
        * 本地不可用时读取机架内副本
        * 最后读取跨机架副本
    - 配置：`dfs.client.read.shortcircuit=true`（开启短路径读取）
+ 因素 3：HDFS Block 大小
    - Block 大小：128MB（默认）
        * 太大：小文件浪费空间，随机读效率低
        * 太小：元数据过多，NameNode 压力大
    - HBase 场景：128MB 适合大 HFile 顺序读取

# Compaction 与 HDFS 的协同
## Compaction 过程中的 HDFS 操作
| 操作 | HDFS 行为 | 资源消耗 |
| --- | --- | --- |
| 读取旧 HFile | 从 HDFS 读取多个旧文件 | 大量 HDFS 读 IO |
| 合并数据 | 内存中排序合并 | CPU 密集 |
| 写入新 HFile | 写入新文件到 HDFS | 大量 HDFS 写 IO |
| 删除旧文件 | 标记删除，延迟物理删除 | NameNode 元数据操作 |


## Compaction 对 HDFS 的影响
空间放大问题：

+ Compaction 前：
    - `HDFS 占用 = HFile_1 + HFile_2 + HFile_3 + HFile_4 = 400MB`
+ Compaction 中：
    - `HDFS 占用 = 旧文件 400MB + 新文件 100MB = 500MB  ← 临时膨胀`
+ Compaction 后：
    - `HDFS 占用 = HFile_new = 100MB  ← 旧文件延迟删除`
+ ⚠️ 注意：旧文件不会立即删除，需等待引用释放



HDFS 压力分析：

| <font style="color:#000000;">指标</font> | <font style="color:#000000;">Minor Compaction</font> | <font style="color:#000000;">Major Compaction</font> |
| --- | --- | --- |
| <font style="color:#000000;">读取量</font> | <font style="color:#000000;">2 - 4GB</font> | <font style="color:#000000;">10- 50GB</font> |
| <font style="color:#000000;">写入量</font> | <font style="color:#000000;">1 - 2GB</font> | <font style="color:#000000;">5 - 20GB</font> |
| <font style="color:#000000;">持续时间</font> | <font style="color:#000000;">1 - 5 分钟</font> | <font style="color:#000000;">30 分钟 - 2 小时</font> |
| <font style="color:#000000;">对业务影响</font> | <font style="color:#000000;">较小</font> | <font style="color:#000000;">较大</font> |




优化建议：

+ 错峰执行：在业务低峰期执行 Major Compaction
+ 限流控制：配置 Compaction 带宽限制
+ 分区 Compaction：大 Compaction 拆分为多个小任务
+ HDFS 预留空间：保证 Compaction 期间有足够空间

### 5.3 HDFS 文件删除机制
HBase 文件删除流程：

1. Compaction 完成，新 HFile 生效
2. 旧 HFile 标记为待删除（引用计数减 1）
3. 等待所有读取完成（引用计数=0）
4. HBase 调用 HDFS `delete()` 删除文件
5. HDFS 将文件移动到`/.Trash/`目录（如果开启回收站）
6. 回收站到期后物理删除，释放空间



配置参数：

+ `hbase.store.delete.excess.storefile`：超过此数量的 StoreFile 触发清理
+ `hbase.master.cleaner.interval`：清理任务执行间隔
+ `fs.trash.interval`：HDFS 回收站保留时间

# 协同优化关键点
## 部署优化
| 序号 | 优化项 | 建议配置 | 理由 |
| --- | --- | --- | --- |
| 1 | 同节点部署 | RegionServer 与 DataNode 同节点 | 利用 HDFS 短路径读取，减少网络开销 |
| 2 | 磁盘隔离 | HDFS 数据盘与 HBase WAL 盘分离 | 避免 IO 竞争 |
| 3 | SSD 缓存 | HDFS 使用 SSD 作为缓存层 | 加速热点数据读取 |
| 4 | 网络隔离 | HBase 与 HDFS 使用独立网络 | 避免业务流量干扰 |


## 配置优化
### HDFS 配置优化（hdfs-site.xml）
```xml
<configuration>
    <!-- 开启短路径读取 -->
    <property>
        <name>dfs.client.read.shortcircuit</name>
        <value>true</value>
    </property>
    
    <!-- 短路径读取 socket 路径 -->
    <property>
        <name>dfs.domain.socket.path</name>
        <value>/var/run/hadoop-hdfs/dn._PORT</value>
    </property>
    
    <!-- HDFS 块大小 -->
    <property>
        <name>dfs.blocksize</name>
        <value>134217728</value>  <!-- 128MB -->
    </property>
    
    <!-- 副本数 -->
    <property>
        <name>dfs.replication</name>
        <value>3</value>
    </property>
</configuration>
```

### HBase 配置优化（hbase-site.xml）
```xml
<configuration>
    <!-- HFile 压缩 -->
    <property>
        <name>hbase.hstore.compression</name>
        <value>snappy</value>
    </property>
    
    <!-- HFile 数据块大小 -->
    <property>
        <name>hfile.block.size</name>
        <value>65536</value>  <!-- 64KB -->
    </property>
    
    <!-- Bloom Filter -->
    <property>
        <name>hbase.hstore.bloomfilter</name>
        <value>ROW</value>
    </property>
    
    <!-- WAL 同步策略 -->
    <property>
        <name>hbase.wal.sync</name>
        <value>true</value>
    </property>
</configuration>
```

# 常见故障与排查
## 故障场景分析
| 序号 | 故障场景 | 现象 | 原因 | 解决方案 |
| --- | --- | --- | --- | --- |
| 1 | HDFS 空间不足 | HBase 写入失败 | HDFS 使用率 > 90% | 扩容或清理旧数据 |
| 2 | DataNode 宕机 | 部分 Region 不可用 | HDFS 副本不足 | 等待副本恢复或手动平衡 |
| 3 | NameNode 过载 | HBase 响应变慢 | HDFS 文件过多 | 合并小文件，优化Compaction |
| 4 | 网络分区 | HBase 读写超时 | RegionServer 无法访问 HDFS | 检查网络，恢复连接 |
| 5 | HFile 损坏 | 读取报错 | HDFS Block 损坏 | HDFS 自动修复或手动修复 |


## 排查命令
```bash
# 1. 检查 HDFS 空间
hdfs dfs -df -h

# 2. 检查 HBase 目录大小
hdfs dfs -du -h /hbase/data

# 3. 检查 HFile 数量
hdfs dfs -ls -R /hbase/data | grep hfile | wc -l

# 4. 检查 HDFS 损坏 Block
hdfs fsck /hbase -files -blocks -locations

# 5. 检查 RegionServer 与 DataNode 同节点情况
hbase shell> status 'detailed'

# 6. 查看 HBase 日志中的 HDFS 错误
grep "DFSClient" /var/log/hbase/*.log

# 7. 检查 HDFS 副本分布
hdfs fsck /hbase/data -files -blocks -racks
```
