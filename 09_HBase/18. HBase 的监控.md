## HBase 如何监控
从指标的分层体系（RegionServer、Table、HFile）以及异常诊断逻辑出发。

从以下三个层面构建 HBase 的监控体系：

1. 性能指标：反映集群的健康状况。
    1. 请求延迟 (RPC Latency)，指标：Read/Write RPC Latency 的 99 分位数（P99）
        1. 如果 P99 飙升，说明存在热点 Region 或磁盘 IO 瓶颈。
    2. 吞吐量 (Throughput)，指标：Read/Write Request Count
        1. 观察是否有突发流量导致 Flink 侧出现背压
    3. 缓存命中率 (BlockCache Hit Ratio)：
        1. 逻辑：对于 Flink 的维表点查，该指标应维持在 90% 以上
        2. 如果命中率骤降，说明内存不足或查询模式发生了改变（如全表扫描）。
2. 存储与运维指标（Storage & Health）：反映集群的长期稳定性。
    1. Region 状态与分布，指标：Region Count 和 Region Server Load
        1. 监控是否存在 Region 倾斜
        2. 如果某个 RS 的请求数远高于其他节点，说明 Rowkey 设计存在热点
    2. 合并与刷写 (Compaction & Flush），指标：Compaction Queue Size（合并队列长度）和 MemStore Size
        1. 预警逻辑：如果合并队列持续积压，会导致存储文件（HFile）过多，进而大幅拖慢 Flink 的读取速度
    3. 死锁与异常 (Exceptions)：监控 RegionTooBusyException 或 FailedSanityCheckException
        1. 这些通常是 Flink 写入速度过快触发了 HBase 的保护机制。



监控方案的工程实现：在实际生产环境中，通常采用 Prometheus + Grafana + JMX 的组合。

## HBase 核心监控指标的阈值参考表
1. 读写性能指标（直接影响 Flink 侧延迟）：这类指标通常监控 P99 或 P95 分位数，用来捕捉“长尾延迟”。

| 监控项 | 推荐告警阈值 (P99) | 诊断逻辑 |
| --- | --- | --- |
| Read Latency | < 20ms (点查) | 超过 100ms 需检查 BlockCache 命中率或是否存在大范围 Scan。 |
| Write Latency | < 10ms | 超过 50ms 通常意味着 WAL 写入缓慢或 MemStore 达到压力水位 |
| RPC Queue Time | < 10ms | 如果此项很高，说明 RegionServer 处理能力见顶，RPC 请求排队 |
| IPC Call Queue | < 100 | 队列积压严重时，Flink 会抛出 CallQueueTooBigException。 |


2. 内存与缓存指标（稳定性核心）：HBase 的 BlockCache 和 MemStore 是内存消耗大户。

| 监控项 | 推荐阈值 | 影响 |
| --- | --- | --- |
| BlockCache Hit Ratio | > 90% | 低于 80% 说明热点数据被挤出内存 |
| MemStore Size | < 80% (RegionServer总额) | 达到 40%-50% 会触发 Flush，达到高水位会阻塞写入 |
| JVM GC Time | 单次 STW < 500ms | 频繁的 Full GC 是 RegionServer “假死”和 Flink 连接超时的首因。 |
