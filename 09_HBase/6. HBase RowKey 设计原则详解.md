RowKey 设计是 HBase 表设计中**最重要**的环节

直接决定了查询性能、数据分布和集群负载均衡。

# RowKey 基本概念与作用
## RowKey 是什么？
RowKey 是 HBase 表中每行数据的唯一标识

## RowKey 的核心作用
| 作用 | 说明 | 影响 |
| --- | --- | --- |
| **唯一标识** | 每行数据的唯一主键 | 决定数据唯一性 |
| **数据排序** | Region 内数据按 RowKey 字典序排序 | 影响范围查询效率 |
| **数据分布** | RowKey 决定数据落在哪个 Region | 影响负载均衡 |
| **查询路由** | Client 通过 RowKey 定位 Region | 影响查询性能 |


## RowKey 存储特点
RowKey 存储的特性

+ 类型：`byte[]` 字节数组（无强类型）       
+ 长度：建议 10-100 字节（越短越好）
+ 排序：字典序（Byte 比较，非字符串比较）
+ 唯一：同一表中 RowKey 必须唯一
+  不可变：RowKey 一旦写入不能修改（只能删除重写）



⚠️ 注意：

+ RowKey 会存储多次（每个 Column Family 都会存储一份 RowKey）
+ RowKey 过长会浪费存储空间，影响性能

# RowKey 设计四大核心原则
## 原则一：长度原则（越短越好）
为什么 RowKey 要短？

1. **存储空间：RowKey 在每个 Cell 中都会存储**
    - 假设 RowKey 长度为 50 字节，表有 5 个列族，10 亿行数据，额外存储 = 50字节 × 5列族 × 10亿行 = 250GB 浪费空间！
2. **内存占用：RowKey 会加载到 MemStore 和 BlockCache**
    - RowKey 过长 → 内存利用率低 → 缓存命中率下降
3. **网络传输：每次查询都会传输 RowKey**
    - RowKey 过长 → 网络带宽浪费 → 查询延迟增加



建议长度：

| 长度（单位：字节） | 10-32 | 32-64 | 64-100 | >100 |
| --- | --- | --- | --- | --- |
| 评价 | 优秀 | 良好 | 一般 | 较差 |




优化技巧

+ 使用数字/时间戳代替长字符串
+ 使用哈希/编码压缩长 ID 
+ 去除冗余前缀/后缀

## 原则二：散列原则（避免热点）
什么是热点 Region

+ 错误设计：RowKey 按时间顺序递增
+ ❌ 问题：所有新数据都写入最后一个 Region，导致单节点过载



热点危害

+ 单 RegionServer 负载过高，成为性能瓶颈
+ 写入性能下降，无法利用集群并发能力
+ 可能触发 Region 分裂，加剧问题
+ 严重时导致 RegionServer 宕机

## 原则三：唯一原则（保证唯一性）
为什么 RowKey 必须唯一？

+ HBase 没有传统数据库的自增主键
+ 相同 RowKey 的新写入会覆盖旧数据（多版本除外）
+ RowKey 重复会导致数据丢失



保证唯一性的方法

+ 方法 1：业务主键，适用场景：用户信息表、账户表
+ 方法 2：业务主键 + 时间戳，适用场景：用户行为日志、订单流水
+ 方法 3：业务主键 + 序列号，适用场景：订单明细、交易记录
+ 方法 4：UUID/GUID，适用场景：日志数据、追踪 ID
    - `RowKey = UUID.randomUUID()` (全局唯一)
    - ⚠️ 注意：UUID 较长（36 字节），且无序，需配合散列使用

## 原则四：查询场景导向原则
核心思想：RowKey 设计要围绕主要查询场景



HBase 查询特点

+ ✅ 高效查询： 基于 RowKey
    - 通过 RowKey 精确查询（Get）
    - 通过 RowKey 范围查询（Scan with StartRow/StopRow）
+ ❌ 低效查询：
    - 非 RowKey 列查询（需要全表扫描或二级索引）
    - 模糊查询（如 `LIKE '%keyword%'`）
    - 多条件组合查询（需要过滤器，性能较差）



设计方法

+ Step 1：梳理查询场景
    - 查询 1：根据用户 ID 查询用户信息 → RowKey 包含 userId
    - 查询 2：查询某时间段订单 → RowKey 包含时间 + orderId
    - 查询 3：查询某地区用户 → 需要二级索引或冗余表
+ Step 2：确定主要查询场景（80% 的查询）
    - 优先优化高频查询场景
    - 低频查询可通过二级索引/冗余表解决
+ Step 3：设计 RowKey 结构
    - 将查询条件放在 RowKey 前缀（支持范围查询）
    - 高频查询字段优先

# RowKey 设计模式与技巧
## 加盐散列（Salting）
### 原理及优缺点
原理：在 RowKey 前缀添加随机数（盐值），将数据分散到不同 Region

+ RowKey 结构：`salt_businessKey`
+ 示例：`05_user_001, 12_user_002, 03_user_003`
+ 盐值计算：`salt = hashCode(businessKey) % saltBuckets`



优点：

+ 有效避免热点 Region
+ 实现简单，易于理解
+ 写入性能高

缺点：

+ 范围查询需扫描所有盐桶（10 个盐桶需查询 10 次）
+ 数据局部性被破坏

### 参数
| 参数 | 建议值 | 计算方式 | 说明 |
| --- | --- | --- | --- |
| 盐桶数量 | 10-100 | RegionServer 数 × 2~3 | 太少散列不足，太多查询开销大 |
| 盐值位数 | 2 位 | 00-99（100 个桶） | 足够覆盖大多数场景 |
| 分隔符 | `_` | - | 便于解析 RowKey |


### 实现
```java
// 加盐散列实现代码
public class SaltedRowKeyGenerator {
    
    private final int saltBuckets;
    
    public SaltedRowKeyGenerator(int saltBuckets) {
        this.saltBuckets = saltBuckets;
    }
    
    public String generate(String businessKey) {
        // 计算盐值
        int salt = Math.abs(businessKey.hashCode()) % saltBuckets;
        // 格式化盐值（2 位数字）
        String saltStr = String.format("%02d", salt);
        // 组合 RowKey
        return saltStr + "_" + businessKey;
    }
    
    // 查询时需要扫描所有盐桶
    public List<String> getAllSaltedRowKeys(String businessKey) {
        List<String> rowKeys = new ArrayList<>();
        for (int i = 0; i < saltBuckets; i++) {
            rowKeys.add(String.format("%02d_%s", i, businessKey));
        }
        return rowKeys;
    }
}

// 使用示例
SaltedRowKeyGenerator generator = new SaltedRowKeyGenerator(32);  // 32 个盐桶
String rowKey = generator.generate("user_001");  // "05_user_001"
```

## 哈希散列（Hashing）
### 原理及优缺点
+ RowKey 结构：`hash_businessKey`
+ 示例：`a3f2c1d8_user_001, b7e4f9a2_user_002`
+ 原理：对业务 Key 进行哈希运算（MD5/SHA），完全打散数据分布
+ 哈希计算：`hash = MD5(businessKey).substring(0, 8)`



优点：

+ 散列效果最好，数据分布最均匀
+ 完全避免热点
+ 写入性能最高

缺点：

+ 完全失去范围查询能力，只能精确查询（Get），不能 Scan
+ 数据局部性完全被破坏

### 参数
| 参数 | 建议值 | 说明 |
| --- | --- | --- |
| 哈希算法 | MD5/SHA-256 | MD5 性能更好，SHA 更安全 |
| 哈希长度 | 8-16 位 | 8 位足够（16^8=42 亿种组合） |
| 分隔符 | `_` | 便于解析 |


### 实现
```java
// 哈希散列实现代码
public class HashedRowKeyGenerator {
    
    public String generate(String businessKey) {
        try {
            // MD5 哈希
            MessageDigest md = MessageDigest.getInstance("MD5");
            byte[] hashBytes = md.digest(businessKey.getBytes());
            // 取前 8 位十六进制
            String hash = bytesToHex(hashBytes).substring(0, 8);
            // 组合 RowKey
            return hash + "_" + businessKey;
        } catch (Exception e) {
            throw new RuntimeException("Hash generation failed", e);
        }
    }
    
    private String bytesToHex(byte[] bytes) {
        StringBuilder sb = new StringBuilder();
        for (byte b : bytes) {
            sb.append(String.format("%02x", b));
        }
        return sb.toString();
    }
}
```

## 时间反转（Reversing Timestamp）
### 原理及优缺点
+ RowKey 结构：`reversedTime_businessKey`
+ 示例：`7567932876847551615_order_001`
+ 原理：将时间戳反转后放入 RowKey，实现时间降序 + 避免热点
+ 反转计算：`reversedTime = Long.MAX_VALUE - timestamp`



优点

+ 支持时间范围查询
+ 最新数据在前，适合查询最近数据
+ 避免时间顺序递增导致的热点

缺点

+ 只适合时序数据
+ RowKey 较长（20 位时间戳）
+ 非时间字段无法范围查询

### 参数
| 参数 | 建议值 | 说明 |
| --- | --- | --- |
| 时间戳格式 | 20 位定长 | 不足补 0，保证字典序正确 |
| 时间精度 | 毫秒 | 根据业务需求选择 |


### 实现
```java
// 时间反转实现代码
public class ReversedTimeRowKeyGenerator {
    
    private static final long MAX_TIMESTAMP = Long.MAX_VALUE;
    
    public String generate(long timestamp, String businessKey) {
        // 反转时间戳
        long reversedTime = MAX_TIMESTAMP - timestamp;
        // 格式化为 20 位定长
        String timeStr = String.format("%020d", reversedTime);
        // 组合 RowKey
        return timeStr + "_" + businessKey;
    }
    
    // 范围查询示例
    public Scan createTimeRangeScan(long startTime, long endTime, String businessKey) {
        Scan scan = new Scan();
        // 注意：时间范围也要反转
        String startRow = String.format("%020d_%s", MAX_TIMESTAMP - endTime, businessKey);
        String stopRow = String.format("%020d_%s", MAX_TIMESTAMP - startTime, businessKey);
        scan.setStartRow(Bytes.toBytes(startRow));
        scan.setStopRow(Bytes.toBytes(stopRow));
        return scan;
    }
}
```

### 示例：监控时序数据
业务特点：

+ 按时间持续写入
+ 主要查询：最近 N 分钟/小时的数据
+ 时间范围查询频繁



RowKey 设计：`reversedTime_metricId_deviceId`

+ `reversedTime`: Long.MAX_VALUE - timestamp
+ `metricId`: 指标 ID
+ `deviceId`: 设备 ID



选择时间反转原因：

+ ✅ 支持时间范围查询
+ ✅ 最新数据在 RowKey 前面，查询最近数据效率高
+ ✅ 避免时间递增导致热点

## 组合键（Composite Key）
### 原理及优缺点
原理：

+ RowKey 结构：`field1_field2_field3_...`
+ 示例：`beijing_20240101_user_001`
+ 原理：将多个业务字段组合成 RowKey，支持多维度查询
+ 字段顺序：高频查询字段 → 区分度高字段 → 范围查询字段



优点

+ 支持多维度范围查询
+ 数据局部性好，相邻数据在一起
+ 查询灵活，可前缀匹配



缺点

+ 顺序递增字段可能导致热点
+ 需要配合加盐或时间反转使用
+ 设计复杂度较高

### 参数
| 参数 | 建议 | 说明 |
| --- | --- | --- |
| 字段顺序 | 高频→区分度高→范围 | 查询频率高的字段放前面 |
| 分隔符 | `_` | 统一分隔符，便于解析 |
| 字段长度 | 总长<100 字节 | 避免 RowKey 过长 |
| 空值处理 | 默认值或跳过 | 避免空值导致 RowKey 不规范 |


### 实现
```java
// 组合键实现代码
public class CompositeRowKeyGenerator {
    
    public String generate(String... fields) {
        // 过滤空值
        List<String> validFields = new ArrayList<>();
        for (String field : fields) {
            if (field != null && !field.isEmpty()) {
                validFields.add(field);
            }
        }
        // 用分隔符连接
        return String.join("_", validFields);
    }
    
    // 范围查询示例
    public Scan createPrefixScan(String... prefixFields) {
        Scan scan = new Scan();
        String startRow = String.join("_", prefixFields);
        String stopRow = startRow + "_~";  // ~ 是 ASCII 中较大的字符
        scan.setStartRow(Bytes.toBytes(startRow));
        scan.setStopRow(Bytes.toBytes(stopRow));
        return scan;
    }
}
```

## 总结对比
四种 Rowkey 设计模式对比

| 维度 | 加盐散列 | 哈希散列 | 时间反转 | 组合键 |
| --- | --- | --- | --- | --- |
| 散列效果 | ⭐⭐⭐ 较好 | ⭐⭐⭐⭐⭐ 最好 | ⭐⭐ 一般 | ⭐⭐ 一般 |
| 范围查询 | ⚠️ 需扫描多桶 | ❌ 不支持 | ✅ 支持 | ✅ 支持 |
| 写入性能 | ⭐⭐⭐⭐ 高 | ⭐⭐⭐⭐⭐ 最高 | ⭐⭐⭐ 中 | ⭐⭐⭐ 中 |
| 数据局部性 | ❌ 打散 | ❌ 打散 | ⚠️ 部分保留 | ✅ 保留 |
| 实现复杂度 | ⭐⭐ 低 | ⭐⭐ 低 | ⭐⭐ 低 | ⭐⭐⭐ 中 |




各模式适用场景

| 模式 | 适用场景 | 不适用场景 | 选择原因 |
| --- | --- | --- | --- |
| 加盐散列 | 日志存储、行为数据、写入密集型 | 需要范围查询的场景 | 散列效果好，实现简单 |
| 哈希散列 | 用户画像、KV 存储、精确查询 | 需要范围查询、Scan 操作 | 散列最好，但失去范围查询 |
| 时间反转 | 时序数据、监控数据、最新数据优先 | 非时间维度查询 | 支持时间范围 + 避免热点 |
| 组合键 | 多维度查询、业务关联查询 | 单字段查询场景 | 灵活支持多种查询模式 |

# 预分区（Pre-Splitting）
### 原理及方案
原理：建表时预先创建多个 Region，避免运行时自动分裂



为什么需要预分区？问题：默认建表只有 1 个 Region

+ 初始状态： Region_1: [空 ~ 无穷大]
    - ❌ 所有数据都写入这一个 Region，严重热点！
+ 自动分裂过程：Region_1 (10GB) → 分裂 → Region_1 + Region_2 
    - ⚠️ 分裂是同步操作，影响写入性能
    - ⚠️ 分裂过程中可能产生短暂不可用



预分区解决方案

```plain
┌─────────────────────────────────────────────────────────────────┐    
    建表时创建多个 Region：                                              
                                                                        
    ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐       
     Region 1  Region 2  Region 3  Region 4  Region 5       
    [A~E)    [E~J)     [J~O)     [O~T)     [T~Z]           
    └─────────┘ └─────────┘ └─────────┘ └─────────┘ └─────────┘       
                                                                        
    ✅ 数据写入时直接分散到多个 Region                                    
    ✅ 避免运行时分裂，提升性能                                          
└─────────────────────────────────────────────────────────────────┘    
```



实现方式：Java API 创建预分区表

```java
HBaseAdmin admin = connection.getAdmin();                          
TableName tableName = TableName.valueOf("user_order");             
                                                                    
// 定义分区键（16 进制）                                              
byte[][] splits = new byte[][] {                                   
    Bytes.toBytes("1000"),                                         
    Bytes.toBytes("2000"),                                         
    Bytes.toBytes("3000"),                                         
    Bytes.toBytes("4000"),                                         
    Bytes.toBytes("5000"),                                         
    Bytes.toBytes("6000"),                                         
    Bytes.toBytes("7000"),                                         
    Bytes.toBytes("8000"),                                         
    Bytes.toBytes("9000")                                          
};                                                                 
                                                                    
HColumnDescriptor cf1 = new HColumnDescriptor("cf1");              
HTableDescriptor tableDesc = new HTableDescriptor(tableName);      
tableDesc.addFamily(cf1);                                          
                                                                    
admin.createTable(tableDesc, splits);  // 创建预分区表               
```

### 预分区策略
+ 根据数据量估算：每 Region 约 10GB，预估总数据量计算 Region 数 
+ 根据并发估算：`Region 数 ≈ RegionServer 数 × 2~3`           
+ 根据 RowKey 分布：按 RowKey 前缀均匀划分

# 数仓开发实例
## 案例一：用户行为日志表
业务需求

+ 存储用户 APP 行为日志（点击、浏览、购买等）
+ 数据量：日增 10 亿条，月增 300 亿条
+ 查询场景：
    - 查询某用户的行为轨迹（高频）
    - 查询某时间段的行为日志（中频）
    - 实时风控查询（高频）



RowKey 设计

+ `RowKey = salt_userId_reversedTimestamp_eventId`
+ 字段说明
    - `salt`: 2 位数字（00-99），100 个盐桶，打散用户数据
    - `userId`: 用户 ID（8-16 字节）
    - `reversedTimestamp`: 反转时间戳（20 字节）
    - `eventId`: 事件 ID（8-16 字节，保证唯一性）
+ 总长度：约 50 字节

选择加盐原因：

+ ✅ 避免 userId 顺序递增导致热点
+ ✅ 写入性能高
+ ⚠️ 范围查询需扫描所有盐桶（可接受）

预分区设计

+ 按 `salt` 预分区：00, 01, 02, ..., 99（100 个 Region）
+ 每个 RegionServer 管理约 3-5 个 Region（假设 30 个 RegionServer）



查询示例

```java
// 查询用户 user_001 最近 1 小时的行为
long now = System.currentTimeMillis();
long oneHourAgo = now - 3600000;

// 需要扫描 100 个盐桶
for (int salt = 0; salt < 100; salt++) {
    Scan scan = new Scan();
    String startRow = String.format("%02d_user_001_%020d",
                    salt, Long.MAX_VALUE - now);
    String stopRow = String.format("%02d_user_001_%020d",
                    salt, Long.MAX_VALUE - oneHourAgo);
    scan.setStartRow(Bytes.toBytes(startRow));
    scan.setStopRow(Bytes.toBytes(stopRow));
    // 执行查询...
```

## 案例二：订单明细表
业务需求

+ 存储电商订单明细数据
+ 数据量：日增 5000 万单，月增 15 亿单
+ 查询场景：
    - 根据订单 ID 查询详情（高频）
    - 根据用户 ID 查询订单列表（高频）
    - 根据时间范围统计订单（中频）



RowKey 设计方案：双表方案

+ 表 1：`order_main`（主表，按订单 ID 查询）
    - `RowKey = salt_orderId`
        * `salt`: 2 位数字（00-31），32 个盐桶
        * `orderId`: 订单 ID（16 字节）
    - 列族：`order_info`（订单信息）, `user_info`（用户快照）
+ 表 2：`user_order`（**索引表**，按用户 ID 查询）
    - `RowKey = userId_reversedTimestamp_orderId`
        * `userId`: 用户 ID
        * `reversedTimestamp`: 反转时间戳
        * `orderId`: 订单 ID（保证唯一）
    - 列族：`order_ref`（订单引用）



✅ 优势：两个查询场景都能高效支持

⚠️ 代价：数据冗余，写入时需要写两张表

## 案例三：用户画像标签表
业务需求

+ 存储用户标签数据（ `demographic`、`behavior`、`preference` 等）    
+ 数据量：1 亿用户，每个用户约 100 个标签                      
+ 查询场景：
    - 根据用户 ID 查询所有标签（高频）
    - 实时更新标签（高频）          



RowKey 设计

+ `RowKey = userId`
+ 列族设计：
    - `demographic`：人口属性标签（年龄、性别、地区等） 
    - `behavior`：行为标签（活跃度、购买频次等）       
    - `preference`：偏好标签（品类偏好、价格敏感度等） 
    - `algorithm`：算法标签（用户分群、预测标签等）    



示例数据：

```java
RowKey: user_001
```

# RowKey 常见陷阱与规避方法
## 陷阱汇总
| 陷阱 | 问题描述 | 后果 | 解决方案 |
| --- | --- | --- | --- |
| **顺序递增 RowKey** | 时间戳/自增 ID 直接作 RowKey | 热点 Region | 加盐/哈希/时间反转 |
| **RowKey 过长** | 使用长字符串/多字段组合 | 存储浪费、性能下降 | 压缩/编码/精简字段 |
| **分隔符不一致** | 部分 RowKey 有分隔符，部分没有 | 范围查询错误 | 统一格式规范 |
| **字节序问题** | 数字直接用字符串存储 | 排序错误（10<2） | 使用 Bytes.toBytes() |
| **未预分区** | 建表时只有 1 个 Region | 初始热点、分裂开销 | 建表时预分区 |
| **忽略查询场景** | RowKey 不支持主要查询 | 全表扫描、性能差 | 查询导向设计 |
| **盐桶数量不当** | 盐桶过多或过少 | 查询开销大/散列不足 | 根据 RegionServer 数估算 |


## 字节序陷阱详解
 问题：数字用字符串存储会导致排序错误                                   

 错误示例：                                                              

```plain
RowKey 使用字符串存储数字：                                          
"1", "2", "3", ..., "9", "10", "11", ...                           
     ↓ 字典序排序                                                    
"1", "10", "11", ..., "2", "3", ..., "9"  ← 错误！                  
```

正确示例：                                                              

```plain
RowKey 使用 Bytes.toBytes() 存储数字：                                
Bytes.toBytes(1), Bytes.toBytes(2), ..., Bytes.toBytes(10)         
     ↓ 字节序排序                                                    
1, 2, 3, ..., 9, 10, 11, ...  ← 正确！        
```

                                                                         

  解决方案                                                           

+ 数字类型使用` Bytes.toBytes()/Bytes.toLong()` 等转换                   
+ 时间戳使用反转 + 定长格式                                            
+ 统一使用 `byte[]` 进行比较，避免字符串比较

# RowKey 设计速查表
四大原则

| 原则 | 要点 |
| --- | --- |
| 长度 | 10-100 字节，越短越好 |
| 散列 | 避免热点，加盐/哈希/时间反转 |
| 唯一 | 保证数据唯一性，业务 ID+ 时间戳/序列号 |
| 查询 | 围绕主要查询场景设计，高频字段放前缀 |




五种模式

| 模式 | 适用场景 |
| --- | --- |
| 加盐 | 写入密集，精确查询为主 |
| 哈希 | 只需精确查询，写入压力极大 |
| 时间反转 | 时序数据，时间范围查询 |
| 组合键 | 多维度查询，高频字段放前缀 |
| 预分区 | 所有场景，建表时必做 |




避坑指南

+ ❌ 避免顺序递增 RowKey 
+ ❌ 避免 RowKey 过长    
+ ❌ 避免数字用字符串存储  
+ ❌ 避免不预分区        
+ ❌ 忽略查询场景设计

### P7 级别问题
| 问题 | 参考答案要点 |
| --- | --- |
| 如何设计支持多维度查询的 RowKey？ | 组合键 + 冗余表/二级索引 |
| 盐桶数量如何确定？ | RegionServer 数 × 2~3，或根据数据量估算 |
| RowKey 设计对 Compaction 有什么影响？ | 散列好的 RowKey 减少 Region 分裂，降低 Compaction 压力 |
| 如何平衡写入性能和查询性能？ | 写入优先用哈希，查询优先用组合键，折中用加盐 |
| 请设计一个订单系统的 RowKey 方案 | 双表方案：order_main + user_order 索引表 |
