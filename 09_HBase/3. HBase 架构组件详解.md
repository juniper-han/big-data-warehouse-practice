# 核心组件详解
## ZooKeeper —— 协调服务
**职责**

| 功能 | 说明 |
| --- | --- |
| **集群状态监控** | 监控 HMaster 和 RegionServer 的存活状态 |
| **Meta 表地址存储** | 存储`hbase:meta`表的位置信息 |
| **Region 位置缓存** | 缓存 Region 与 RegionServer 的映射关系 |
| **分布式锁** | 保证 HMaster 选举的原子性 |
| **配置信息存储** | 存储集群配置和元数据 |


****

**ZK中的关键节点**

```plain
/hbase
├── /hbase/master          # HMaster地址
├── /hbase/rs              # RegionServer列表
├── /hbase/meta            # Meta表位置
├── /hbase/table           # 表状态信息
└── /hbase/region-in-transition  # Region状态转换
```



数仓开发关注点

1. Client 首次连接时，通过 ZK 获取 Meta 表位置
2. RegionServer 故障时，ZK 触发故障转移
3. 生产环境 ZK 集群，建议 3/5/7 节点（奇数）
4. ZK 故障会导致整个HBase集群不可用

## HMaster —— 管理节点
### 核心职责
| 功能 | 详细说明 |
| --- | --- |
| **表管理** | 创建、删除、修改表结构（Column Family） |
| **Region 分配** | 将 Region 分配给 RegionServer |
| **负载均衡** | 监控 Region 分布，进行 Region 迁移 |
| **故障恢复** | 检测 RegionServer 故障，重新分配 Region |
| **DDL 操作** | 处理所有 schema 变更请求 |


### HMaster工作流程
HMaster 启动流程     

1. 竞争 ZK 锁
2. 成为 Active Master
3. 加载 Meta 表
4. 扫描 Region 分配
5. 处理故障 Region
6. 开始负载均衡

### 高可用设计
```plain
┌──────────────┐         ┌──────────────┐
│  Active      │  ────►  │   Standby    │
│  HMaster     │  心跳   │   HMaster    │
│  (主)        │         │   (备)       │
└──────────────┘         └──────────────┘
       │                        │
       ▼                        ▼
   处理请求                  待命状态
   写入ZK                    监控ZK
```

### 数仓开发关注点
1. <font style="background-color:#E8F7CF;">DDL 操作（建表、修改列族）必须通过 HMaster</font>
    - <font style="background-color:#E8F7CF;">HMaster 故障时无法创建新表或修改表结构</font>
2. <font style="background-color:#E8F7CF;">正常读写不经过 HMaster（直接访问 RegionServer）</font>
    - <font style="background-color:#E8F7CF;">HMaster 故障不影响已有表的读写</font>
3. 生产环境至少部署 2 个 HMaster（1 主 1 备）

## RegionServer —— 数据服务节点
### 核心职责
| 功能 | 详细说明 |
| --- | --- |
| **处理读写请求** | 响应 Client 的`Get`、`Put`、`Scan`等操作 |
| **管理 Region** | 每个 RegionServer 管理多个 Region |
| **MemStore 管理** | 内存中的数据缓存，达到阈值后 Flush |
| **WAL 写入** | 写操作先写 WAL，保证数据不丢失 |
| **Compaction** | 合并 StoreFile，清理删除标记 |


### RegionServer 内部结构
```plain
┌─────────────────────────────────────────────────────────────┐
│                      RegionServer                           │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────┐  │
│  │    Region A     │  │    Region B     │  │   Region C  │  │
│  │                 │  │                 │  │             │  │
│  │  ┌───────────┐  │  │  ┌───────────┐  │  │             │  │
│  │  │ MemStore  │  │  │  │ MemStore  │  │  │             │  │
│  │  │ (内存)    │  │  │  │ (内存)    │  │  │             │  │
│  │  └─────┬─────┘  │  │  └─────┬─────┘  │  │             │  │
│  │        │ Flush  │  │        │ Flush  │  │             │  │
│  │        ▼        │  │        ▼        │  │             │  │
│  │  ┌───────────┐  │  │  ┌───────────┐  │  │             │  │
│  │  │StoreFile  │  │  │  │StoreFile  │  │  │             │  │
│  │  │ (HDFS)    │  │  │  │ (HDFS)    │  │  │             │  │
│  │  └───────────┘  │  │  └───────────┘  │  │             │  │

│  ┌─────────────────────────────────────────────────────┐    │
│  │                    WAL (Write-Ahead Log)            │    │
```

### 关键参数配置
```xml
<!-- hbase-site.xml 关键参数 -->
<property>
    <name>hbase.regionserver.global.memstore.size</name>
    <value>0.4</value>  <!-- MemStore占堆内存比例 -->
</property>

<property>
    <name>hbase.regionserver.global.memstore.size.lower.limit</name>
    <value>0.38</value>  <!-- Flush触发下限 -->
</property>

<property>
    <name>hbase.hstore.blockingStoreFiles</name>
    <value>10</value>  <!-- Compaction触发阈值 -->
</property>

<property>
    <name>hbase.regionserver.handler.count</name>
    <value>200</value>  <!-- RPC处理线程数 -->
</property>
```

### 数仓开发关注点
1. 读写请求直接访问 RegionServer（不经过 HMaster）
2. RegionServer 故障，会导致部分数据暂时不可用
3. MemStore 大小影响 Flush 频率和写入性能
4. WAL影响写入性能和数据安全性
5. 生产环境 RegionServer 建议配置 SSD 提升性能

## Region —— 数据分区单元
### 核心概念
| 特性 | 说明 |
| --- | --- |
| **数据分片** | 表按`RowKey`范围，切分成多个 Region |
| **自动分裂** | Region 达到阈值（默认 10GB）自动分裂 |
| **移动单元** | Region 是负载均衡和故障恢复的基本单位 |
| **有序存储** | Region 内数据按`RowKey`排序存储 |


### Region结构
```plain
┌─────────────────────────────────────────────────────────────┐
│                         Region                              │
│  RowKey范围：user_0001 ~ user_1000                          │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────────────────┐    │
│  │                   Column Family 1                    │    │
│  │  ┌─────────────┐      ┌─────────────────────────┐    │    │
│  │  │  MemStore   │      │      StoreFiles         │    │    │
│  │  │  (内存)     │      │      (HDFS)             │    │    │
│  │  │             │      │  ┌─────┐ ┌─────┐ ┌─────┐│    │    │
│  │  │  • 最新写入 │      │  │ F1  │ │ F2  │ │ F3  ││    │    │
│  │  │  • 读优先   │      │  └─────┘ └─────┘ └─────┘│    │    │
│  │  └─────────────┘      └─────────────────────────┘    │    │
│  └─────────────────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │                   Column Family 2                    │    │
│  │  (结构同CF1，每个CF独立MemStore和StoreFile)          │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

```plain
└─────────┘    └─────────┘    └─────────┘    └─────────┘

     │                                          ▼
     │                                   ┌─────────┐

```

### Region分裂过程
```plain
分裂前：
┌─────────────────────────────────────┐
│  Region A: user_0001 ~ user_9999    │
│  大小：12GB (超过阈值10GB)           │
└─────────────────────────────────────┘

分裂后：
┌─────────────────────────┐  ┌─────────────────────────┐
│  Region A1:             │  │  Region A2:             │
│  user_0001 ~ user_4999  │  │  user_5000 ~ user_9999  │
│  大小：6GB              │  │  大小：6GB              │
└─────────────────────────┘  └─────────────────────────┘
```

### 数仓开发关注点
1. Region 数量影响集群负载均衡
2. RowKey 设计不当会导致 Region 热点
3. 预分区，可以提前创建多个 Region，避免分裂
4. Region 过多会增加 HMaster 管理负担
5. 单表 Region 数建议控制在 1000 以内

# 组件总结对比表
| 组件 | 职责 | 是否参与读写 | 故障影响 | 部署建议 |
| --- | --- | --- | --- | --- |
| **ZooKeeper** | 协调服务、元数据存储 | 是（首次定位） | 集群不可用 | 3/5/7节点 |
| **HMaster** | 管理节点、DDL 操作 | 否 | 无法DDL、无法分配Region | 2节点主备 |
| **RegionServer** | 数据服务、读写处理 | 是 | 部分数据不可用 | N≥3节点 |
| **Region** | 数据分区、存储单元 | - | 触发故障转移 | 自动管理 |

## 组件选型建议
| 场景 | 推荐方案 | 原因 |
| --- | --- | --- |
| 海量明细存储 | HBase | 支持随机读写、扩展性好 |
| 实时查询服务 | HBase + Redis | HBase 存全量，Redis 存热点 |
| 离线分析 | Hive/Spark | 批量处理能力强 |
| 用户画像 | HBase | 支持多版本、列式存储 |
| 时序数据 | HBase + 时间反转 RowKey | 支持时间范围查询 |


## 生产环境部署建议
生产环境集群规模

| 组件 | 节点数 | 配置建议 |
| --- | --- | --- |
| ZooKeeper | 3/5/7 | 4 核 8G，独立部署       |
| HMaster | 2 | 8 核 16G，主备模式 |
| RegionServer | N≥3 | 16 核 64G+，SSD 磁盘 |
| HDFS | N≥3    | 与 RegionServer 同节点部署 |
