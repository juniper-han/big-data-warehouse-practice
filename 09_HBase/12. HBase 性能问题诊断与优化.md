HBase 性能优化是一个系统工程，核心在于

1. 数据分布均匀性
2. IO 效率
3. 内存管理

# 热点 Region 问题诊断与解决
热点 Region（Hotspotting）是 HBase 最常见的性能杀手

表现为：

+ 集群中个别 RegionServer CPU/IO 飙升，而其他节点空闲
+ 导致整体吞吐量上不去，甚至引发宕机。

## 诊断方法：如何发现热点？
### 监控指标观察 (Grafana/Prometheus/HBase UI)
1. RegionServer 负载不均：QPS、IO、Compaction队列
    - 观察 `Requests per second` (QPS)：某个节点 QPS 是其他节点的 10 倍以上。
    - 观察 `Write Bytes` / `Read Bytes`：某个节点的网络 IO 或磁盘 IO 显著高于其他节点。
    - 观察 `Compaction Queue`：热点节点的 Compaction 队列堆积严重。
2. JVM 指标：GC 和 CPU 使用
    - 热点节点的 GC 频率（Young GC/Old GC）显著高于其他节点。
    - CPU Use% 持续高位（>80%）。

### 命令行工具诊断
1. 查看 Region 分布：

```bash
# 查看表的所有 Region 分布情况，关注 startKey 和所在的 Server
hbase shell> status 'detailed'

# 或者使用 balancer 命令前的检查
hbase shell> balance_switch false # 先关闭自动平衡，防止干扰诊断
```

2. 查看慢查询日志：
    - 检查 `hbase-regionserver.log`，搜索 `Slow RPC` 或 `Slow Read/Write`，看是否集中在特定的 Region
    - `grep "Slow.*read" hbase-regionserver.log | head -n 20`
3. 使用 `top` 命令定位进程内线程：
    - 登录到负载高的 RegionServer 节点
    - 执行 `top -H -p <pid>`，查看是哪个线程在忙。
        * 如果是 `IPC` 线程忙，说明请求多；
        * 如果是 `Compaction` 线程忙，说明文件合并压力大。

### 根因分析逻辑
1. **RowKey 设计缺陷**：这是 90% 的原因。
    - 例如使用时间戳、自增 ID 作为 RowKey 前缀，导致所有新写入都落入最后一个 Region。
2. **业务访问模式倾斜**
    - 某些“超级热点”数据被高频访问，即使 RowKey 散列了，但该 Key 本身的访问量过大。
3. **Region 分裂不及时**
    - 数据增长过快，自动分裂跟不上写入速度，导致单个 Region 过大。

## 解决方案
### 根本解决：优化 RowKey 设计 (最推荐)
1. 加盐 (Salting)：在 RowKey 前缀添加随机数（0-99），将写入压力分散到多个 Region。
    - _适用_：写多读少，或精确查询场景。
2. 哈希 (Hashing)：对 RowKey 关键部分进行 MD5/SHA 哈希。
    - _适用_：纯精确查询，完全放弃范围查询。
3. 时间反转 (Reversing Timestamp)：`Long.MAX_VALUE - timestamp`。
    - _适用_：时序数据，既避免热点，又保证最新数据在前（便于 Scan 最近数据）。
4. 组合键：`字段 A_字段 B_时间反转`，利用高基数字段打散。

### 临时急救：手动 Split 与 Balance
如果无法立即修改代码重构 RowKey，可尝试：

1. 手动分裂热点 Region：

```bash
# 找到热点 Region 的名称
hbase shell> split 'table_name', 'split_point_key'
# 如果不指定 split_point，HBase 会尝试自动寻找中点分裂
hbase shell> split 'region_name'
```

2. 触发负载均衡：

```bash
hbase shell> balance_switch true
hbase shell> balancer
```

注意：

+ 这只能缓解读取热点
+ 对于写入热点（因为写入目标由 RowKey 决定），Balance 效果有限，新数据还是会写入新的“尾端”Region。

### 架构层优化：预分区 (Pre-splitting)
在创建表时就规划好足够的 Region，避免初始阶段只有一个 Region 承载所有流量。

### 业务层优化：缓存与异步
1. 本地缓存
    - 对于极热的读请求，在应用层使用 Guava/Caffeine/Redis 缓存，减少直达 HBase 的请求。
2. 写入缓冲
    - 使用 Kafka 削峰填谷，将突发写入平滑化。

# 预分区策略设计
预分区是：防止初期热点、提升写入吞吐的最佳实践。

其核心思想是：在表创建时，根据预估的数据量和增长模式，预先创建多个空的 Region

## 设计原则
### Region 数量估算
Region 数量估算：取以下两者中的最大值

+ 预估总数据量 / 单 Region 目标大小
+ Region Server 数量 × 2 ~ 3



注意：

1. 单 Region 目标大小：通常建议 10GB - 20GB。
    - 太小会导致 Meta 表膨胀和频繁 Split/Merge；
    - 太大会导致 Compaction 压力过大和恢复慢。
2. RegionServer 倍数：、
    - 保证每个 RegionServer 至少管理 2-3 个 Region，以便负载均衡有回旋余地。

### Split 键（Splits）的选择策略
Split 键决定了 Region 的边界 `[StartKey, EndKey)`。



Split 键的选择策略取决于 RowKey 模式：

+ **场景 A：RowKey 包含盐值 (Salting)**
    - 策略：按**盐值个数**预分区。
    - 示例：如果盐值是 00-15 (16 个桶)，则设置 15 个 Split 键：`01`, `02`, ..., `0F`。
    - 优点：数据绝对均匀分布。
+ **场景 B：RowKey 是哈希前缀 (Hashing)**
    - 策略：按哈希空间均匀切分。
    - 示例：使用 MD5 前 2 位十六进制 (`00`-`ff`)。可以每隔 `10` (十六进制) 切一刀，共 16 个区；或更细粒度。
    - 工具：可以使用 HBase 自带的 `SplitKeyCalculator` 或自行编写脚本生成均匀分布的哈希值。
+ **场景 C：RowKey 是时间序列 (TimeSeries)**
    - 策略：按时间范围预分区（需配合时间反转或盐值，否则无意义）。
    - 示例：如果是按天归档的表，可以按 `YYYYMMDD` 的前缀预分区。但更推荐结合盐值。
+ **场景 D：未知分布/均匀递增**
    - 策略：使用 `UniformSplit` 或 `HexStringSplit`。
    - 命令：`create 'table', 'cf', {SPLITS_FILE => '/tmp/splits.txt'}`

## 实操案例：为亿级旅客表设计预分区
假设：

+ 数据量：1 亿条。
+ 单行大小：500B -> 总数据约 50GB。
+ 目标单 Region 大小：10GB -> 理论需要 5 个 Region。
+ 集群规模：10 个 RegionServer。
+ RowKey 策略：`salt (2 位 00-31)` + `flight_id` + ...

**计算**：

+ 按数据量算：5 个 Region (太少，无法利用 10 台机器)。
+ 按机器算：10 × 2 = 20
+ 按 Salt 桶算：设计了 32 个盐桶。
+ **决策**：选择 **32 个 Region** (对应 31 个 Split 键)。
    - 这样既能充分利用 10 台机器（每台约 3 个 Region）
    - 又能完美匹配 Salt 策略，保证写入绝对均匀。

**建表命令 (Shell)**：

```bash
# 生成 01 到 31 的 splits 文件 (伪代码逻辑)
# splits.txt 内容:
# 01
# 02
# ...
# 31

create 'passenger_by_flight', 
  {NAME => 'info', COMPRESSION => 'SNAPPY', VERSIONS => 1}, 
  {NAME => 'status', COMPRESSION => 'SNAPPY', VERSIONS => 5},
  SPLITS_FILE => '/tmp/splits_32.txt'
```

**Java API 预分区示例**：

```java
byte[][] splits = new byte[31][];
for (int i = 1; i < 32; i++) {
    // 假设盐值是 2 位数字字符串 "01", "02"...
    splits[i - 1] = Bytes.toBytes(String.format("%02d", i));
}
admin.createTable(tableDesc, splits);
```

## 常见误区
1. 分区过多：创建了成千上万个 Region。
    - 后果：Zookeeper 压力大，Master 负载高，MemStore 堆内存被大量小 Region 瓜分
    - 导致频繁 Flush
2. 分区过少：只分了 3-5 个区
    - 后果：初期写入集中，后续分裂频繁，影响性能。
3. Split 键分布不均
    - 人为指定的 Split 键没有覆盖数据的真实分布，导致某些 Region 依然很大，某些很小。
    - 务必确保 Split 键在逻辑上是均匀的。

# 读写性能瓶颈分析
当 HBase 变慢时，需要从 

Client -> Network -> RegionServer (Mem/Cache/Disk) -> HDFS 

全链路排查。

## 写性能瓶颈分析
现象：

+ 写入 TPS 低
+ 延迟高
+ RegionServer CPU 高或 GC 频繁。



瓶颈点分析

1. **WAL 同步**
    1. 可能原因：磁盘 IO 慢，WAL 同步阻塞
    2. 诊断
        1. `sync_wal`耗时高；
        2. iostat `%util`100%
    3. 优化方案
        1. 使用 SSD 存储 WAL
        2. 调整 `hbase.wal.sync` (风险：丢数据)
        3. 增加 WAL 滚动策略，避免单文件过大
2. **MemStore 阻塞**
    1. 可能原因：MemStore 满了，触发 Flush 或阻塞写入
    2. 诊断：
        1. `blockedRequests` 计数增加；
        2. 日志出现 `GlobalMemStoreLimit`
    3. 优化方案
        1. 调大 `hbase.regionserver.global.memstore.size`(默认 0.4)
        2. 优化 RowKey 避免热点导致的单 Region 满
        3. 增加 RegionServer 堆内存
3. Compaction 风暴
    1. 可能原因：小文件太多，Compaction 占用大量 IO/CPU
    2. 诊断：
        1. `compactionQueueLength` 高；
        2. CPU 高；
        3. 磁盘 IO 读写双高
    3. 优化方案
        1. 调整 `hbase.hstore.compaction.threshold` (默认 3)
        2. 开启 Compaction 限流 (`hbase.regionserver.compaction.throttle.enabled`)
        3. 错峰执行 Major Compaction
4. GC 停顿
    1. 可能原因：Young GC 频繁或 Old GC 长停顿
    2. 诊断
        1. GC 日志显示 STW 时间长；
        2. 请求超时
    3. 优化方案
        1. 启用 G1 GC
        2. 调整堆大小和 Region 数量 (减少 MemStore 总数)
        3. 开启 BucketCache (堆外缓存) 减轻堆压力
5. 网络带宽
    1. 可能原因：批量写入过大，打满网卡
    2. 诊断：网络流量饱和
    3. 优化方案
        1. 客户端调整 `write.buffer` 大小
        2. 检查是否有大 Value (建议<10KB，大对象存 HDFS 存路径)

## 读性能瓶颈分析
现象：

1. Get/Scan 延迟高
2. QPS 上不去。



瓶颈点分析

1. BlockCache 命中率低
    1. 可能原因：缓存太小或热点分散，导致频繁读磁盘
    2. 诊断：
        1. `blockCacheHitRatio < 50%`；
        2. Disk Read IO 高
    3. 优化方案
        1. 调大 `hfile.block.cache.size` (默认 0.4)。
        2. 启用 BucketCache (L2 缓存)。
        3. 优化 RowKey 让热点数据更集中。
2. StoreFile 过多
    1. 可能原因：未 Compaction，一次读需扫描几十个小文件
    2. 诊断：
        1. `storeFileCount > 20`；
        2. Bloom Filter `misses` 高
    3. 优化方案
        1. 触发 Compaction
        2. 调整 Compaction 策略，更快合并小文件。
        3. 确保 `BloomFilter` 开启 (`ROW` 或 `ROWCOL`)
3. Scan 设计不当
    1. 可能原因：全表扫描，Caching 设置过小，Filter 效率低
    2. 诊断：
        1. 单词 Scan 返回数据量巨大
        2. RPC 次数极多
    3. 优化方案
        1. 严格限制 Scan 的 `Start/Stop Row`。
        2. 调大 `scanner.caching` (减少 RPC 往返，注意 OOM)。
        3. 使用 Filter 在服务端过滤，减少网络传输。
        4. 避免` setAllowPartialResults(false)` 导致的长尾
4. RPC 线程池满
    1. 可能原因：并发请求过多，处理线程不足
    2. 诊断
        1. `handler.count` 满载；
        2. 请求排队等待
    3. 优化方案
        1. 调大 `hbase.regionserver.handler.count` (默认 200)。
        2. 增加 RegionServer 节点
5. HDFS 慢
    1. 可能原因：底层磁盘慢或 DataNode 负载高
    2. 诊断：HDFS 读延迟高
    3. 优化方案
        1. 检查 HDFS 健康度。
        2. 确保数据本地性 (RegionServer 与 DataNode 同节点)。

## 综合排查工具箱
1. RSStatus（RegionServer Status）：
    - 访问 `http://<regionserver-ip>:16030/rs-status` (端口视版本而定)
    - 查看实时的 Load、Requests、Compactions。
2. Ganglia/Prometheus：
    - 查看历史趋势，对比故障时间点前后的 IO、Network、GC 变化。
3. 慢查询日志 (Slow Query Log)：
    - 在 `hbase-site.xml` 中开启：

```xml
<property>
    <name>hbase.rpc.read.warning.threshold</name>
    <value>1000</value> <!-- 超过 1 秒记录日志 -->
</property>
<property>
    <name>hbase.rpc.write.warning.threshold</name>
    <value>1000</value>
</property>
```

4. Coprocessor Observer：
    - 自定义 Coprocessor 埋点，统计特定表的特定操作耗时（高级用法）。

## 性能调优黄金法则
1. **RowKey 是王道：**90% 的性能问题源于 RowKey 设计不当导致的热点。
2. **预分区是基础：**不要依赖自动分裂，建表时必须预分区。
3. 内存换 IO：尽可能利用 `BlockCache` 和 `MemStore`，减少磁盘 IO。
4. 批量优于单条：客户端务必使用 `List<Put>` 或 `List<Get>` 批量提交，减少 RPC 开销。
5. 列族宜少不宜多：尽量控制在 1-2 个列族，避免 Flush/Compaction 时的 IO 放大。
6. 数据本地性：部署时确保 HBase RegionServer 和 HDFS DataNode 在同一台物理机上，利用短路径读取（Short-circuit Read）
