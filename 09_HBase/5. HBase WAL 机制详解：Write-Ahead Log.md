WAL（Write-Ahead Log）是HBase保证**数据持久性**和**故障恢复**的核心机制

# WAL 基本概念
## 什么是 WAL？
WAL：Write-Ahead Log

+ 定义：预写日志，在数据写入内存之前，先写入持久化日志
+ 目的：保证数据不丢失，支持故障恢复
+ 位置：HDFS分布式文件系统
+ 格式：Protobuf 序列化格式（HBase 2.0+） 

## 为什么需要 WAL？
| 场景 | 无 WAL 的风险 | 有 WAL 的保障 |
| --- | --- | --- |
| **RegionServer 宕机** | MemStore 数据全部丢失 | 可通过 WAL 恢复数据 |
| **服务器断电** | 内存数据丢失 | 日志已持久化到 HDFS |
| **进程崩溃** | 未 Flush 数据丢失 | 重启后重放 WAL 恢复 |
| **网络故障** | 写入状态不确定 | **通过 WAL 确认写入状态** |


## WAL 与 MemStore 的协作关系
Client 写入请求

1. 请求到达 RegionServer
2. 先写入 WAL（HDFS 持久化）：保证持久化，顺序追加
3. 后写入 MemStore（内存缓存）：加速读取，达到阈值 Flush
4. Flush 到 StoreFile（HFile）
5. Flush 后可清理 WAL 和 MemStore

<font style="background-color:#E8F7CF;">⚠️</font><font style="background-color:#E8F7CF;"> 核心原则：先写 WAL，再写 MemStore，确保数据不丢失  </font>

# WAL 工作原理详解
## WAL 写入流程
WAL 写入完整流程

+ Step 1：Client 发起写入请求
    - RegionServer 接收请求，获取对应的 WAL 实例   
        * 每个 RegionServer 有多个 WAL（按 RegionGroup 分组）
+ Step 2：构造 WAL Entry（日志条目）
    - WAL Entry 结构：Sequence ID（序列号），Table Name（表名），Region Name（Region 名），Edit（`Put`/`Delete` 操作内容），Timestamp（时间戳） 
+ Step 3：追加写入 WAL 文件（顺序写入 HDFS）
    - 写入 HDFS 的`/hbase/WALs/RegionServer` 目录
    - 顺序追加，性能较高
+ Step 4：WAL 同步（hsync）
    - 确保数据持久化到 HDFS
    - 可配置同步策略（默认每条同步）
+ Step 5：写入 MemStore（内存）
    - WAL 写入成功后才写入 MemStore 
    - 如果 WAL 失败，返回写入失败
+ Step 6：返回成功给 Client

# WAL 故障恢复机制
## 故障场景与恢复流程
RegionServer 故障恢复流程：故障发生

+ Step 1：ZooKeeper 检测 RegionServer 心跳丢失
    - 默认超时时间：30 秒
+ Step 2：HMaster 标记 RegionServer 为死节点
    - 更新 ZK 中的`/hbase/rs` 节点状态
+ Step 3：HMaster 将故障 Region 重新分配给其他 RegionServer
    - 更新 `hbase:meta` 表中的 Region 位置信息
+ Step 4：新 RegionServer 加载故障 RegionServer 的 WAL 文件
    - 从 HDFS 读取 `/hbase/WALs/`故障节点/ 目录
+ Step 5：重放 WAL（Replay）
    - 按 Sequence ID 顺序重放日志条目
    - 恢复 MemStore 中的数据
+ Step 6：WAL 处理完成，清理日志文件
    - 已恢复的 WAL 文件移动到` /hbase/WALs/oldWALs/`
    - 定期清理旧 WAL 文件
+ Step 7：Region 恢复服务，对外提供读写

## WAL 分裂（WAL Splitting）
WAL 分裂的核心原因是一个 RegionServer 上的多个 Region 共享同一个 WAL 文件。

+ 当 RegionServer 故障后，HMaster 会将其上的 Region 重新分配给其他 RegionServer。
+ 如果不分裂 WAL，每个新的 RegionServer 都需要读取完整的 WAL 文件来恢复数据，但实际上每个节点只需要恢复分配给自己的 Region 的日志。

## 恢复时间影响因素
| 因素 | 影响 | 优化方案 |
| --- | --- | --- |
| **WAL 文件大小** | 文件越大，恢复越慢 | 控制 WAL 滚动大小 |
| **WAL 文件数量** | 文件越多，分裂越慢 | 减少 RegionServer 数量 |
| **网络带宽** | 影响 WAL 传输速度 | 优化网络配置 |
| **HDFS 性能** | 影响 WAL 读取速度 | 使用 SSD 存储 HDFS |
| **Region 数量** | Region 越多，恢复越慢 | 合理设计 Region 数量 |




减少 WAL 分裂耗时的方法

| 优化方向 | 具体措施 | 效果 |
| --- | --- | --- |
| 控制 WAL 大小 | 减小 `hbase.wal.max.filesize` | 减少单次分裂数据量 |
| 增加 WAL 组数 | 增大 `hbase.regionserver.wal.groups` | 分散 WAL 文件，并行分裂 |
| 预分区 Region | 提前创建合理数量的 Region | 避免 Region 过度集中 |
| 使用 SSD 存储 | HDFS 使用 SSD 磁盘 | 提升读写速度 |
| 优化网络 | 万兆网络、机架感知 | 加快数据传输 |

# WAL 配置与调优
## 关键配置参数
```xml
<!-- hbase-site.xml 配置 -->

<!-- ========== WAL 基础配置 ========== -->

<!-- WAL 目录 -->
<property>
    <name>hbase.wal.dir</name>
    <value>/hbase/WALs</value>
    <description>WAL 文件在 HDFS 上的存储目录</description>
</property>

<!-- WAL 滚动大小（默认 1GB） -->
<property>
    <name>hbase.wal.max.filesize</name>
    <value>1073741824</value>
    <description>WAL 文件达到此大小后滚动生成新文件</description>
</property>

<!-- ========== WAL 同步策略 ========== -->

<!-- WAL 同步策略 -->
<property>
    <name>hbase.wal.sync</name>
    <value>true</value>
    <description>每条写入是否同步到 HDFS（true=安全，false=性能）</description>
</property>

<!-- 异步 WAL 批次大小 -->
<property>
    <name>hbase.wal.async.batch.size</name>
    <value>4096</value>
    <description>异步 WAL 的批次大小（字节）</description>
</property>

<!-- ========== WAL 数量控制 ========== -->

<!-- 每个 RegionServer 的 WAL 组数 -->
<property>
    <name>hbase.regionserver.wal.groups</name>
    <value>8</value>
    <description>WAL 组数，影响并发写入性能</description>
</property>

<!-- 最大 WAL 文件数量 -->
<property>
    <name>hbase.regionserver.max.logs</name>
    <value>32</value>
    <description>达到此数量后触发 Flush，防止 WAL 过多</description>
</property>

<!-- ========== WAL 恢复配置 ========== -->

<!-- WAL 恢复并行度 -->
<property>
    <name>hbase.master.distributed.log.splitting.max.retries</name>
    <value>10</value>
    <description>WAL 分裂最大重试次数</description>
</property>

<!-- WAL 清理间隔 -->
<property>
    <name>hbase.master.cleaner.interval</name>
    <value>60000</value>
    <description>WAL 清理任务执行间隔（毫秒）</description>
</property>
```

## 同步策略对比
| 策略 | 配置 | 数据安全性 | 写入性能 | 适用场景 |
| --- | --- | --- | --- | --- |
| **同步 WAL** | `hbase.wal.sync=true` | 高 | 较低 | 金融、订单等关键数据 |
| **异步 WAL** | `hbase.wal.sync=false` | 中 | 高 | 日志、行为数据等 |
| **跳过 WAL** | `put.setDurability(SKIP_WAL)` | 低 | 最高 | 临时数据、可丢失数据 |


同步每条都 hsync，异步批量 hsync，性能与安全权衡

# WAL 在数仓开发中的实践
## Java API 中控制 WAL
```java
import org.apache.hadoop.hbase.client.Put;
import org.apache.hadoop.hbase.client.Durability;

// 场景 1: 默认使用 WAL（推荐）
Put put1 = new Put(Bytes.toBytes("row1"));
put1.addColumn(Bytes.toBytes("cf1"), Bytes.toBytes("name"), Bytes.toBytes("张三"));
table.put(put1);

// 场景 2: 跳过 WAL（高性能，数据可丢失）
Put put2 = new Put(Bytes.toBytes("row2"));
put2.setDurability(Durability.SKIP_WAL);  // 不写 WAL
put2.addColumn(Bytes.toBytes("cf1"), Bytes.toBytes("name"), Bytes.toBytes("李四"));
table.put(put2);

// 场景 3: 异步 WAL（性能与安全平衡）
Put put3 = new Put(Bytes.toBytes("row3"));
put3.setDurability(Durability.ASYNC_WAL);  // 异步写 WAL
put3.addColumn(Bytes.toBytes("cf1"), Bytes.toBytes("name"), Bytes.toBytes("王五"));
table.put(put3);

// 场景 4: 同步 WAL（最高安全性）
Put put4 = new Put(Bytes.toBytes("row4"));
put4.setDurability(Durability.SYNC_WAL);  // 同步写 WAL
put4.addColumn(Bytes.toBytes("cf1"), Bytes.toBytes("name"), Bytes.toBytes("赵六"));
table.put(put4);
```

## Flink 写入 HBase 的 WAL 控制
```java
// Flink HBase Sink 配置
HBaseSink<UserInfo> hbaseSink = new HBaseMutationSinkBuilder<UserInfo>()
    .setTableName("user_info")
    .setZookeeperQuorum("node1:2181,node2:2181,node3:2181")
    .setMutator((userInfo, context) -> {
        Put put = new Put(Bytes.toBytes(userInfo.getRowKey()));
        put.addColumn(Bytes.toBytes("cf1"), Bytes.toBytes("name"), 
                     Bytes.toBytes(userInfo.getName()));
        
        // 根据数据重要性选择 WAL 策略
        if (userInfo.isCritical()) {
            put.setDurability(Durability.SYNC_WAL);  // 关键数据
        } else {
            put.setDurability(Durability.ASYNC_WAL);  // 普通数据
        }
        
        return put;
    })
    .build();
```

# 总结对比表
| 特性 | WAL | MemStore | StoreFile |
| --- | --- | --- | --- |
| **存储位置** | HDFS | JVM 堆内存 | HDFS |
| **持久化** | 是 | 否（依赖 WAL） | 是 |
| **写入顺序** | 先写 | 后写 | Flush 后生成 |
| **数据格式** | 日志条目 | 跳表结构 | HFile 格式 |
| **主要作用** | 故障恢复 | 写缓存 | 持久存储 |
| **清理时机** | Flush 后清理 | Flush 后清空 | Compaction 后清理 |
