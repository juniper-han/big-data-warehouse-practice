## 物化视图的使用经验
什么是 Doris 的物化视图？

+ Doris 的物化视图是一种预计算的数据库表。
+ 它本质上是原始明细表（Base Table）的一个“影子副本”，但这个副本是经过特定加工（如聚合、过滤、重列）后的结果。



物理存储：与普通视图（View）不同，物化视图的数据是持久化在磁盘上的，并与主表保持强一致性。

实时同步：当你的 Flink 链路向明细表写入数据时，Doris 会在同一事务中同步更新相关的物化视图。这意味着明细表更新，物化视图即刻更新。



使用物化视图的场景

1. 异步加速与解耦：
    1. 已经在 Flink 做了宽表拼接，如果再把所有复杂的 Group By 聚合逻辑也压在 Flink 状态（State）里，会导致 Flink 任务变得非常重，Checkpoint 时间变长。
    2. 策略：让 Flink 负责“稳”和“全”（输出明细宽表），让 Doris 物化视图负责“快”（预聚合指标）。
2. 查询自动路由（Query Rewrite）：
    1. 这是 Doris 最强大的功能。用户依然查询明细表（Base Table），但 Doris 的优化器会自动判断：如果物化视图中已经包含了查询所需的维度和聚合指标，它会自动将查询透明转发到物化视图上。
    2. 示例：你查 SELECT province, SUM(gmv) FROM base_table GROUP BY province，Doris 发现 MV 里已经预计算好了，就会直接读 MV，性能提升往往在 10 倍以上。



物化视图的分类：Doris 目前支持两种主要的物化视图

+ 强一致性物化视图（同步更新）：
    - 特点：只能基于单表创建，支持简单的聚合（SUM, MIN, MAX, COUNT, BITMAP_UNION, HLL_UNION）。
    - 适用场景：你提到的 Flink 写入宽表后，针对该宽表的常用维度进行实时汇总。
+ 多表异步物化视图（V2.0+ 核心）：
    - 特点：支持多表 Join 和更复杂的计算，数据通过定时刷新或手动刷新。
    - 适用场景：复杂的 T+1 报表或准实时（分钟级延迟）的湖仓一体分析。



落地优化经验

1. Bitmap 聚合（留存/去重分析）：在宽表中，如果你有“去重人数（UV）”的需求，可以在 MV 中使用 BITMAP_UNION。
    - 效果：查询 COUNT(DISTINCT user_id) 时，Doris 会直接从 MV 读取预聚合好的 Bitmap 指令，比扫描千万级明细行再做 Hash 去重要快得多。
2. 控制 MV 的数量：虽然 MV 能加速查询，但每增加一个 MV，Flink 写入时的 写放大（Write Amplification） 就会增加，消耗更多的 BE 节点 CPU 和 IO。
    - 原则：只针对那些“高频、且扫描明细开销巨大”的聚合场景创建 MV。
3. 物化视图与 Rollup 的关系：你会发现 MV 和旧版的 Rollup 很像。你可以解释说：“在 Doris 中，物化视图是 Rollup 的功能增强版，它提供了更多的聚合函数支持和更灵活的查询匹配。”


## 同步、异步物化视图怎么选？
对数据一致性、计算成本与业务场景的权衡能力。

Doris 的同步物化视图（也称单表物化视图）和异步物化视图（也称多表物化视图）有着完全不同的底层逻辑。



从以下三个维度来拆解选型标准：

1. 核心差异对比：

| 特性 | 同步物化视图 (Synchronous MV) | 异步物化视图 (Asynchronous MV) |
| --- | --- | --- |
| 一致性 | 强一致性（与基表原子更新） | 最终一致性（定时或手动触发刷新） |
| 计算范围 | 仅限单表聚合、过滤、重列 | 支持多表 Join、聚合、窗口函数等 |
| 刷新方式 | 写入数据时实时计算（增量） | 全量刷新或分区增量刷新 |
| 查询路由 | 自动路由（透明加速） | 支持自动路由，也可直接查询 MV 表 |
| 写入开销 | 增加写入延迟（写放大严重） | 对写入主表无直接影响，占用后台计算资源 |


2. 选型场景：
    1. 场景 A：选择同步物化视图
        1. 背景： 在 Flink 已经拼好了明细宽表，写入 Doris 后需要对这个宽表做即时的指标汇总。
        2. 理由 1：实时性要求极高。 如果业务方要求看板上的 GMV 必须随着 Flink 的每一条数据更新而变动，同步 MV 是唯一选择。
        3. 理由 2：查询透明化。 分析师不需要知道 MV 的存在，直接查明细表，Doris 优化器会自动命中物理预聚合的数据，降低运维心智。
        4. 注意点： 这种方式会消耗 BE 节点的 CPU 进行实时聚合。如果 Flink 写入吞吐极大，我会限制同步 MV 的数量（通常不超过 3 个）。
    2. 场景 B：选择异步物化视图
        1. 背景： 涉及多个大表关联（如订单表 Join 用户表 Join 活动表），或者需要 T+1/分钟级刷新的报表
        2. 理由 1：解耦多表关联。 Flink 拼宽表很重，如果有些关联逻辑不常变（如基础信息），可以在 Doris 中通过异步 MV 定时刷新来做“二次建模”
        3. 理由 2：支持复杂计算。 如果需要跨分区的全局排序或复杂的子查询，同步 MV 无法支持，必须用异步 MV。
        4. 理由 3：湖仓一体加速。 比如直接加速查询 Hive/Hudi 中的数据，异步 MV 是目前最好的缓存层方案。



高阶视角：性能与稳定性平衡：在实际项目中，不盲目追求同步，从以下两个“指标”出发进行调优：

1. 观察 Compaction Score：如果开启了多个同步 MV，由于每次写入都要更新多份数据，会导致 Version 数激增。如果 BE 节点的 Compaction 压力过大，我会建议将部分非核心指标改为异步 MV（每分钟刷新），以减轻实时写入的负担。
2. 分层思想：
    - 实时层 (Sync MV)： 处理核心、高频、简单的聚合（如当前的 UV/PV/成交额）。
    - 应用层 (Async MV)： 处理复杂的、多维度的分析报告，或者作为数据仓库的 Summary 层。
