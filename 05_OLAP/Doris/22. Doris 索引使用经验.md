## 索引使用经验
罗列类型，说明每种索引解决什么样的数据过滤问题。

Doris 的索引体系分为：内建前缀索引，辅助索引两大类。

实时宽表场景中，索引的设计直接决定了 UPSERT 后的查询响应速度。

内建前缀索引 (Short Key Index)：这是 Doris 自动维护的索引，也是最基础的加速手段。

+ 原理： Doris 将 Key 列（排序列）的前 36 个字节截取出来生成稀疏索引。
+ 特性： 存储在内存中，能够快速定位数据行所在的 Data Block。
+ 优化点： 在建表时，排序列的顺序非常重要。通常将高频出现在 WHERE 条件中、区分度高的维度（如 user_id, order_id）放在前面。



辅助索引 (Additional Indexes)：这是需要手动根据业务查询模式（Query Pattern）去创建的索引

1. 跳数索引 (Zone Map Index)
    1. 机制： 自动为每一列在 Data Page 级别维护 Max / Min / HasNull 等元信息
    2. 场景： 针对范围查询（> < BETWEEN）非常有效。Doris 执行引擎通过比较 Page 的最大最小值，直接跳过不符合条件的整个数据块。
2. 位图索引 (Bitmap Index)
    1. 机制： 利用位图技术加速数据过滤。
    2. 场景： 适用于**低基数（Low Cardinality）**列，即取值重复率很高的列（如：性别、省份、订单状态、是否会员）。
    3. 优势： 通过位运算即可完成过滤，性能极高。
3. 布隆过滤器索引 (Bloom Filter Index)
    1. 机制： 通过概率性数据结构判断一个值“绝对不在”或“可能在”某个数据块中。
    2. 场景： 适用于高基数（High Cardinality）列的等值查询（= 或 IN），例如 user_id 或 trace_id。
    3. 注意： 它不能处理范围查询（> <），且存在极小的误判率，但在海量数据点查时能显著减少磁盘 IO
4. 倒排索引 (Inverted Index) —— V2.0+ 核心特性
    1. 机制： 类似 Elasticsearch 的实现，支持字符串的全文本检索（分词）、等值、范围查询。
    2. 场景： 实时宽表中有大量 Log 日志或 JSON 字符串 搜索需求时，这是 Doris 的“大杀器”。
    3. 优势： 在 V2.0 之后，倒排索引在很多场景下已经可以替代传统的位图索引，成为处理复杂过滤条件的首选。



索引优化经验总结
针对实时宽表的索引策略如下：

1. 分层排布： 优先保证 event_time 或 order_id 位于前缀索引中，解决 80% 的基础查询。
2. 按需添加： 针对高频的业务状态过滤（如 status=1）添加 Bitmap 索引；针对大 ID 的点查添加 Bloom Filter。
3. 避免滥用： 索引虽好，但会增加数据导入时的处理开销（Write Amplification）。对于更新极其频繁的宽表，我们会权衡索引数量，优先保证 Flink 写入吞吐。
4. Runtime Filter： 在多表 Join 时，Doris 会在运行时动态生成过滤条件传给左表，这本质上也是一种“动态索引”技术的体现。
