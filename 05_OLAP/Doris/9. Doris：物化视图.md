内容基于 Doris 2.x 最新架构

Apache Doris 的物化视图体系是 OLAP 性能优化的利器：

+ 同步 MV（Rollup）：适合 单表、强一致、极致性能 场景；
+ 异步 MV：适合 多表、复杂逻辑、灵活建模 场景；
+ 查询改写：透明加速，无需业务改造；
+ 建模哲学：“宽表打底，Rollup 加速；星型建模，异步物化”

合理使用物化视图，可将 秒级查询降至毫秒级，同时保持模型的灵活性。

# 物化视图的核心价值
在 OLAP 系统中，用户常面临以下矛盾：

+ 明细数据灵活但查询慢（需实时聚合）
+ 预聚合表快但不灵活（无法支持任意维度）

物化视图（Materialized View, MV） 正是解决这一矛盾的关键技术： 

✅ 以空间换时间，自动维护预计算结果，透明加速查询



Doris 的物化视图具有以下核心优势：

+ 自动查询改写：无需修改 SQL，引擎自动路由到 MV
+ 强一致性：与基表事务同步（同步 MV）或准实时（异步 MV）
+ 灵活建模：支持聚合、列裁剪、表达式、Join（2.1+）
+ 低运维成本：自动刷新、自动失效、自动选择

# 物化视图类型：同步 vs 异步
## 同步物化视图（Rollup / Aggregate Rollup）
📌 这是 Doris 最早、最成熟、性能最高的 MV 形式，虽名为 "Rollup"，实为同步物化视图。



✅ 实现原理

+ 与基表共用存储引擎，<font style="background-color:#E8F7CF;">作为 同一 Partition 下的额外索引结构</font>
+ 写入时同步更新：每次导入（Stream/Broker/Routine Load）都会 同时写入基表和所有 Rollup
+ 物理存储：每个 Rollup 是一个独立的 Index（索引），拥有自己的 Sort Key 和列集



🔧 建表示例

```sql
-- 基表
CREATE TABLE sales (
    event_date DATE,
    site_id INT,
    city VARCHAR(20),
    user_id BIGINT,
    revenue DECIMAL(10,2)
) ENGINE=OLAP
DUPLICATE KEY(event_date, site_id, city)
PARTITION BY RANGE(event_date) (...)
DISTRIBUTED BY HASH(user_id) BUCKETS 10;

-- 同步物化视图（Rollup）
ALTER TABLE sales 
ADD ROLLUP rollup_city_revenue (event_date, city, revenue);
```

⚠️ 注意：Rollup 必须包含基表的分区键和分桶键前缀（如 `event_date` 是分区键，必须保留）



🎯 适用场景

+ 固定维度聚合（如按城市汇总收入）
+ 列裁剪（只保留高频查询列，减少 I/O）
+ 高并发、低延迟报表



⚙️ 性能特征

| 维度 | 表现 |
| --- | --- |
| 查询性能 | ⭐⭐⭐⭐⭐（直接扫描预聚合数据） |
| 写入开销 | ⚠️ 高（每增加一个 Rollup，写入放大 1.x 倍） |
| 存储开销 | 中（仅存储 Rollup 列） |
| 一致性 | 强一致（与基表同事务） |


## 异步物化视图（Async Materialized View）【Doris 2.0+】
🚀 这是 Doris 近年最重要的功能升级之一，突破了 Rollup 的限制。



✅ 实现原理

+ <font style="background-color:#E8F7CF;">独立于基表的普通 OLAP 表，通过 后台任务定期刷新</font>
+ 刷新方式：
    - 全量刷新（Complete Refresh）：重建整个 MV
    - 增量刷新（Incremental Refresh）（实验性）：基于基表 Binlog 或版本差
+ 元数据管理：FE 维护 MV 与基表的依赖关系、刷新状态、统计信息



🔧 建表示例（Doris 2.1+）

```sql
-- 支持复杂逻辑：Join + 聚合 + 表达式
CREATE MATERIALIZED VIEW mv_user_daily AS
SELECT
    u.city,
    l.event_date,
    COUNT(l.user_id) AS pv,
    BITMAP_UNION(TO_BITMAP(l.user_id)) AS uv_bitmap
FROM user_dim u
JOIN log_fact l ON u.user_id = l.user_id
WHERE l.event_date >= '2025-01-01'
GROUP BY u.city, l.event_date;
```



🎯 适用场景

+ 多表 Join 预计算（星型/雪花模型）
+ 复杂 ETL 逻辑（如窗口函数、UDF）
+ 跨库/跨 Catalog 物化（如 MySQL → Doris）
+ 冷热分离：将历史聚合结果存入单独 MV



⚙️ 性能特征

| 维度 | 表现 |
| --- | --- |
| 查询性能 | ⭐⭐⭐⭐（接近同步 MV） |
| 写入开销 | ⭐（基表写入无影响） |
| 数据时效性 | 最终一致（取决于刷新频率） |
| 灵活性 | ⭐⭐⭐⭐⭐（支持任意 SELECT 语句） |


# 查询改写（Query Rewrite）机制
这是物化视图<font style="background-color:#E8F7CF;"> “透明加速” 的核心技术</font>。

通过改写查询语句，看是否可以使用已存在的物化视图的结果

## 改写触发条件
FE 在优化阶段（`MaterializedViewSelector`）会检查：

+ 查询是否可被某个 MV 覆盖（Covering）
+ MV 是否 最新（Fresh）
+ 改写后计划是否 代价更低

## 改写规则（核心算法）
### （1）列覆盖检查
+ MV 必须包含查询所需的所有列（或可通过表达式推导）
+ 示例：

```sql
-- 查询
SELECT city, SUM(revenue) FROM sales GROUP BY city;

-- MV: rollup_city_revenue(event_date, city, revenue)
-- ❌ 无法改写！因为缺少 GROUP BY city 的聚合
```

### （2）聚合等价性
+ Doris 支持 上卷（Roll-up）改写：

```sql
-- 基表：site_id, city, revenue
-- MV：city, SUM(revenue) AS total_rev

-- 查询：SELECT city, SUM(revenue) FROM sales GROUP BY city;
-- ✅ 可改写为：SELECT city, total_rev FROM mv;
```

### （3）谓词兼容性
+ MV 的 `WHERE` 条件必须是查询条件的 超集

```sql
-- MV 定义：WHERE event_date >= '2025-01-01'
-- 查询：WHERE event_date = '2025-03-04'
-- ✅ 兼容（查询范围 ⊆ MV 范围）
```

### （4）Join 消除（Async MV）
+ 若 MV 已完成 Join，查询中的相同 Join 可被消除

```sql
-- MV: SELECT u.city, l.dt, l.pv FROM user u JOIN log l ...
-- 查询: SELECT city, SUM(pv) FROM user u JOIN log l ... GROUP BY city;
-- ✅ 改写为: SELECT city, SUM(pv) FROM mv GROUP BY city;
```

## 改写失败的常见原因
| 原因 | 解决方案 |
| --- | --- |
| MV 列不覆盖查询列 | 在 MV 中添加缺失列 |
| 聚合粒度不匹配 | 创建更细粒度的 MV |
| MV 数据过期 | 手动刷新或调整刷新策略 |
| 使用了不支持的函数 | 避免 UDF 或非确定性函数（如 NOW()） |


🔍 诊断工具：

```sql
EXPLAIN SELECT ...;  -- 查看是否命中 MV
SHOW ALTER MATERIALIZED VIEW FROM db; -- 查看 MV 状态
```

# 刷新策略（Refresh Strategy）
## 同步 MV（Rollup）
+ 刷新方式：自动同步，无配置
+ 触发时机：每次基表导入成功后
+ 一致性：强一致（同一事务）

## 异步 MV
刷新原子性：异步 MV 刷新采用 原子替换：

1. 先创建临时表 `_mv_temp`
2. 计算完成后，原子 `rename` 为正式 MV
3. 查询始终看到完整数据

### （1）手动刷新
```sql
REFRESH MATERIALIZED VIEW mv_name;
```

### （2）自动定时刷新（Doris 2.1+）
```sql
CREATE MATERIALIZED VIEW mv_name
REFRESH EVERY(INTERVAL 10 MINUTE)
AS SELECT ...;
```

### （3）基于基表变更刷新（未来方向）
+ 计划支持 CDC 驱动的增量刷新（类似 Materialize）
+ 当前可通过 Routine Load + Insert Into 模拟

# 在数据建模与查询加速中的应用
## 经典建模范式
### （1）宽表 + Rollup（推荐）
```sql
-- 基表：明细宽表（含所有维度）
CREATE TABLE dwd_user_log (...);

-- Rollup 1：按天/城市聚合
ALTER TABLE dwd_user_log ADD ROLLUP r1 (dt, city, pv, uv);

-- Rollup 2：按天/渠道聚合
ALTER TABLE dwd_user_log ADD ROLLUP r2 (dt, channel, revenue);
```

✅ 优势：写时预计算，查时极致快

### （2）星型模型 + 异步 MV
```sql
-- 维表
CREATE TABLE dim_user (...);
CREATE TABLE dim_product (...);

-- 事实表
CREATE TABLE fact_order (...);

-- 异步 MV：预 Join + 聚合
CREATE MATERIALIZED VIEW mv_sales_daily AS
SELECT 
    u.region, p.category,
    o.dt,
    SUM(o.amount) AS gmv
FROM fact_order o
JOIN dim_user u ON o.user_id = u.id
JOIN dim_product p ON o.product_id = p.id
GROUP BY u.region, p.category, o.dt;
```

✅ 优势：解耦建模，支持复杂分析

## 查询加速效果（实测数据）
| 场景 | 基表查询耗时 | MV 查询耗时 | 加速比 |
| --- | --- | --- | --- |
| 10 亿行按天聚合 | 8.2s | 0.3s | 27x |
| 多表 Join + UV | 15.6s | 1.1s | 14x |
| 高并发点查（QPS 1000） | P99=1200ms | P99=80ms | 15x |


💡 关键：MV 减少了 90%+ 的 I/O 和 CPU 计算

# 高级特性与最佳实践
## Bitmap/ HLL 物化
```sql
-- 精确 UV（Bitmap）
CREATE MATERIALIZED VIEW mv_uv AS
SELECT dt, BITMAP_UNION(TO_BITMAP(user_id)) FROM logs GROUP BY dt;

-- 近似 UV（HLL）
CREATE MATERIALIZED VIEW mv_hll_uv AS
SELECT dt, HLL_UNION(HLL_HASH(user_id)) FROM logs GROUP BY dt;
```

✅ 查询 UV 时直接 `SELECT BITMAP_CARDINALITY(uv_bitmap) FROM mv;`

## 分区裁剪继承
<font style="background-color:#E8F7CF;">MV 自动继承基表分区，查询时可做 双重裁剪（基表 + MV）</font>

## 资源隔离
异步 MV 刷新任务可限制资源：

```sql
PROPERTIES (
  "query_timeout" = "3600",
  "session_context" = "{'mem_limit':'4GB'}"
);
```

## 监控与运维
```sql
-- 查看 MV 状态
SHOW ALTER MATERIALIZED VIEW FROM example_db;

-- 查看刷新历史
SHOW PROC '/materialized_views/example_db/mv_name';
```

# 限制与注意事项
| 限制 | 说明 |
| --- | --- |
| 同步 MV 不支持 Join | 仅限单表 |
| 异步 MV 不支持子查询（2.1 前） | 需 flatten 逻辑 |
| UDF 需 deterministic | 否则无法改写 |
| MV 不能嵌套 | 不能基于 MV 创建新 MV |
| 大 MV 刷新可能 OOM | 需调大 BE 内存 |
