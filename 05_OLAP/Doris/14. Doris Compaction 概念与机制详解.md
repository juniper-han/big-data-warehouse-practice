## Compaction 的概念
### 为什么需要 Compaction？
Doris 采用 LSM-Tree（Log-Structured Merge-Tree）架构，数据写入流程如下：

1. 写入内存：数据先写入 MemTable（内存结构）
2. 刷盘：MemTable 满后刷成磁盘上的 Segment 文件
3. 版本累积：随着持续写入，会产生大量的小文件（Segment）
4. 查询问题：大量小文件会导致：
    - 元数据膨胀
    - 查询时需要打开大量文件，IO 开销大
    - 数据版本多，需要合并多个版本的数据

Compaction 的作用：将多个小的 Segment 文件合并成大的、有序的文件，清理删除和过期的数据，提升查询性能。

### 数据版本 vs Rowset
1. Rowset：一次数据导入生成的数据集合，包含多个 Segment 文件
2. 版本（Version）：每个 Rowset 对应一个版本号
3. 查询时需要合并所有版本的 Rowset，返回最新数据

## Doris 的 Compaction 类型
Doris 主要有三种 Compaction 机制：

### Base Compaction
触发条件：

1. Cumulative 层数据量达到一定阈值
2. 定期触发（默认 30 分钟检查一次）

特点：

1. 将 Cumulative Rowset 合并到 Base Rowset
2. 合并粒度较大，IO 消耗高
3. 产出单一的 Base Rowset

执行时机：

1. 当 Cumulative Rowset 总大小超过 `base_compaction_num_cumulative_rowsets`（默认 5）时
2. 或者 Cumulative 数据大小超过 Base Rowset 的一定比例

### Cumulative Compaction
触发条件：

1. Cumulative 层 Rowset 数量超过 `cumulative_compaction_num_cumulative_rowsets`（默认 5）
2. 定期触发（默认 10 秒检查一次）

特点：

1. 合并多个小的 Cumulative Rowset
2. 粒度较小，相对轻量
3. 产出新的 Cumulative Rowset

合并策略：

1. Size-tiered：按大小分层合并，适合写多读少场景
2. Num-based：按数量合并，适合持续导入场景

### Full Compaction（手动触发）
特点：

+ 手动触发（`ALTER TABLE table_name COMPACT`）
+ 将表的所有版本合并成一个版本
+ 消耗资源最大，但能最大化查询性能
+ 适合定期维护（如凌晨低峰期）

## Compaction 配置参数详解
### BE 端关键参数（be.conf）
#### Compaction 线程配置
```bash
# Compaction 总线程数，默认 10
# 建议：CPU 核心数的 50%-100%
max_compaction_threads = 10

# 单次 Compaction 任务可用的最大线程数
# 建议：与 max_compaction_threads 保持一致
compaction_task_num_per_disk = 4
```

#### Cumulative Compaction 配置
```bash
# 触发 Cumulative Compaction 的最小 Rowset 数量
# 默认 5，建议：写入频繁时可调大到 10
cumulative_compaction_num_cumulative_rowsets = 5

# Cumulative Compaction 最大产出文件大小
# 默认 10737418240（10GB），建议：根据磁盘 IO 能力调整
cumulative_compaction_max_output_size = 10737418240

# Cumulative Compaction 的最小产出文件大小
# 默认 1073741824（1GB）
cumulative_compaction_min_output_size = 1073741824
```

#### Base Compaction 配置
```bash
# 触发 Base Compaction 的最小 Rowset 数量
# 默认 5
base_compaction_num_cumulative_rowsets = 5

# Base Compaction 最小数据量（字节）
# 默认 5368709120（5GB）
base_compaction_min_data_size = 5368709120

# Base Compaction 间隔（秒）
# 默认 86400（1天），建议：写入量大时可缩短
base_compaction_interval_seconds = 86400
```

### 表级别配置（建表或修改表时指定）
```sql
-- 建表时指定
CREATE TABLE example_table (
    id INT,
    name STRING,
    date DATE
) DUPLICATE KEY(id)
DISTRIBUTED BY HASH(id) BUCKETS 10
PROPERTIES (
    -- 单个 Segment 文件大小，默认 256MB
    "segment_size" = "268435456",
    
    -- 是否开启写时合并（MOW），默认 false
    "enable_unique_key_merge_on_write" = "false",
    
    -- 压缩方式，默认 LZ4
    "compression" = "LZ4"
);

-- 修改表属性
ALTER TABLE example_table SET (
    "segment_size" = "536870912"  -- 调整为 512MB
);
```

## Compaction 监控指标与告警
关键监控指标：

```sql
-- 1. 查看 Compaction 队列长度
SHOW PROC '/compactions';
-- 关注 "pending_num" 字段，如果持续 > 0 表示 Compaction 跟不上

-- 2. 查看各个表的版本数
SHOW TABLET FROM table_name;
-- 正常版本数应 < 50，超过 100 需关注

-- 3. 查看磁盘 IO 使用情况
SHOW PROC '/statistic';
-- 关注 Compaction 相关的 IO 指标
```

Prometheus 监控指标：

1. `doris_be_compaction_total`：Compaction 总次数
2. `doris_be_compaction_duration_ms`：Compaction 耗时
3. `doris_be_compaction_bytes`：Compaction 处理数据量
4. `doris_be_cumulative_compaction_rowsets`：Cumulative 层 Rowset 数量

告警阈值建议：

1. 单个 Tablet 版本数 > 100：告警
2. Compaction 队列 pending > 10：告警
3. Compaction 耗时 > 1 小时：告警

## Doris 版本数限制与写入拒绝机制
### 版本数限制阈值
Doris 不会因为版本数过多而直接拒绝写入，但会触发一系列保护和限制机制，最终可能导致写入失败或性能严重下降。

#### 核心限制参数
| 参数 | 默认值 | 作用 |
| --- | --- | --- |
| `max_tablet_version_num` | 500 | 单个 Tablet 最大版本数，超过后禁止写入 |
| `max_cumulative_compaction_num_singleton_deltas` | 1000 | Cumulative 层最大文件数，超过后强制合并 |
| `max_compaction_concurrency` | -1（自动） | Compaction 并发度限制 |


关键点：

+ 当 Tablet 版本数 ≥ 500 时，Doris 会拒绝该 Tablet 的所有写入操作
+ 这是硬性限制，无法绕过

#### 查看当前限制
```sql
-- 查看全局版本数限制
SHOW FRONTEND CONFIG LIKE '%max_tablet_version_num%';

-- 查看 BE 的版本数限制相关配置
SHOW BACKEND CONFIG LIKE '%version%';
```

### 版本数达到限制时的表现
#### 写入失败的错误信息
当版本数超过 500 时，写入操作会返回类似以下错误：

```plain
ERROR 1105 (HY000): errCode = 2, detailMessage = 
Tablet [tablet_id] has too many versions [501] > max_tablet_version_num [500]
```

#### 实际触发机制
并不是所有版本数达到 500 时立即拒绝，而是：

1. Compaction 努力期（版本数 100-500）：
    - 写入正常
    - Compaction 会加速执行
    - 查询性能开始下降
2. 保护期（版本数 ≥ 500）：
    - 立即拒绝新写入
    - 查询仍然可以进行（但性能极差）
    - Compaction 持续运行，直到版本数降低到 500 以下
3. 恢复期（版本数 < 500）：
    - 自动恢复写入能力
    - 无需人工干预

### 版本数过高的危害阶段
#### 性能影响曲线
| 版本数范围 | 查询性能影响 | 写入影响 | Compaction 状态 |
| --- | --- | --- | --- |
| < 50 | 正常 | 正常 | 正常 |
| 50-100 | 轻微下降（+10-30%） | 正常 | 开始繁忙 |
| 100-200 | 明显下降（+50-100%） | 轻微延迟 | 持续高压 |
| 200-500 | 严重下降（+200-500%） | 明显延迟 | 接近极限 |
| ≥ 500 | **禁止写入**，查询极慢 | **拒绝** | 紧急合并 |


#### 实际案例数据
根据生产环境经验：

```plain
版本数 100：查询耗时增加约 30%
版本数 200：查询耗时增加约 100%
版本数 300：查询耗时增加约 200%
版本数 400：查询耗时增加约 400%
版本数 500：写入拒绝，查询耗时增加 500%+
```

### 监控版本数的方法
#### 查看整体版本分布
```sql
-- 查看所有表的 Tablet 版本分布
SHOW PROC '/statistic';

-- 重点关注 max_version 和 avg_version
-- 输出示例：
+----------+------------------+------------------+-------------+
| DbId     | TableId          | MaxVersion       | AvgVersion  |
+----------+------------------+------------------+-------------+
| 12345    | 67890            | 487              | 156         |
+----------+------------------+------------------+-------------+
```

#### 查看具体表的版本数
```sql
-- 查看某个表的所有 Tablet 版本数
SHOW TABLET FROM table_name;

-- 输出关键字段：
-- VersionCount：当前 Tablet 的版本数
-- 如果 VersionCount 接近 500，需要立即关注

-- 筛选版本数过高的 Tablet
SHOW TABLET FROM table_name 
WHERE VersionCount > 400;
```

#### 查看版本数最高的 Tablet
```sql
-- 通过系统表查询（Doris 2.0+）
SELECT 
    t.table_id,
    t.tablet_id,
    t.version_count
FROM information_schema.tablets t
WHERE t.version_count > 400
ORDER BY t.version_count DESC
LIMIT 10;
```

#### 实时监控 Compaction 状态
```sql
-- 查看 Compaction 队列
SHOW PROC '/compactions';

-- 关注：
-- pending_num：等待合并的 Compaction 任务数
-- running_num：正在执行的任务数
-- 如果 pending_num 持续 > 0，说明版本数在快速增长
```

### 版本数过高的应急处理
#### 紧急处理流程
当发现版本数接近 500 时：

```sql
-- 1. 立即查看版本数最高的 Tablet
SHOW TABLET FROM problem_table 
WHERE VersionCount > 400;

-- 2. 手动触发该表的 Compaction
ALTER TABLE problem_table COMPACT;

-- 3. 监控 Compaction 进度
SHOW PROC '/compactions';

-- 4. 如果仍然增长，临时降低导入并发
-- 在应用层控制导入速度
```

#### 如果已经达到 500 被拒绝写入
**步骤 1：确认问题 Tablet**

```sql
-- 从错误日志中找到 tablet_id
-- 或者查询版本数最高的 Tablet
SHOW TABLET FROM table_name 
ORDER BY VersionCount DESC 
LIMIT 10;
```

**步骤 2：强制 Compaction**

```sql
-- 对该表执行 Full Compaction
ALTER TABLE table_name COMPACT;

-- 如果是分区表，可以只处理问题分区
ALTER TABLE table_name PARTITION(p20240301) COMPACT;
```

**步骤 3：等待恢复**

```sql
-- 持续监控版本数
SHOW TABLET FROM table_name 
WHERE VersionCount > 400;

-- 当版本数降到 400 以下时，写入会自动恢复
```

#### 预防性配置调整
```bash
# be.conf - 提前调整，避免达到 500

# 降低触发 Cumulative Compaction 的阈值
cumulative_compaction_num_cumulative_rowsets = 3  # 默认 5

# 增加 Compaction 线程，加快合并速度
max_compaction_threads = 20

# 减少 Base Compaction 间隔，更频繁合并
base_compaction_interval_seconds = 43200  # 12 小时

# 调整 Cumulative 层最大产出大小
cumulative_compaction_max_output_size = 5368709120  # 5GB
```

```sql
-- FE 配置调整（谨慎，需重启 FE）
-- 修改 fe.conf 或通过 SQL 动态调整
ADMIN SET FRONTEND CONFIG ("max_tablet_version_num" = "1000");

-- 注意：调大此值只是延迟拒绝，不解决根本问题
-- 建议保持默认 500，通过优化 Compaction 来避免达到阈值
```

### 根本原因分析与预防
#### 版本数过高的常见原因
| 原因 | 表现 | 解决方案 |
| --- | --- | --- |
| 高频小批量导入 | 每秒数十次导入，版本数快速增长 | 使用 Group Commit，合并导入批次 |
| Compaction 资源不足 | pending_num 持续 > 0 | 增加 `max_compaction_threads` |
| 磁盘 IO 瓶颈 | Compaction 耗时过长 | 升级 SSD，调整 Compaction 策略 |
| Unique Key 模型 | 更新操作产生大量版本 | 开启 Merge-on-Write |
| 分区设计不当 | 单分区数据量过大 | 细化分区（按天/小时） |


#### 预防性监控告警
**设置告警阈值**：

```plain
版本数 > 300：警告（需要关注）
版本数 > 400：紧急（需要立即处理）
版本数 > 450：严重（即将拒绝写入）
```

**Prometheus 告警规则示例**：

```yaml
groups:
  - name: doris_compaction_alerts
    rules:
      - alert: DorisTabletVersionTooHigh
        expr: doris_be_tablet_version_count > 400
        for: 5m
        annotations:
          summary: "Tablet version count is high"
          
      - alert: DorisCompactionPending
        expr: doris_be_compaction_pending > 10
        for: 10m
        annotations:
          summary: "Compaction queue is backing up"
```

#### 最佳实践建议
1. 控制导入频率：
    - 建议单表导入频率 < 30 次/分钟
    - 使用 Stream Load 批量导入
2. 定期维护：

```sql
-- 每天低峰期执行
ALTER TABLE table_name COMPACT;
```

3. 合理设置 Segment 大小：

```sql
-- 避免产生过多小文件
ALTER TABLE table_name SET (
    "segment_size" = "536870912"  -- 512MB
);
```

4. 使用合适的表模型：
    - Duplicate Key：适合只追加场景
    - Unique Key with MOW：适合更新频繁场景
    - Aggregate Key：适合预聚合场景

### 总结
结论：

1. 硬性限制：版本数 ≥ 500 时，Doris 拒绝写入，返回明确错误
2. 软性影响：版本数 > 100 时，查询性能开始明显下降
3. 恢复机制：Compaction 完成后版本数降到 500 以下，写入自动恢复
4. 预防为主：通过监控告警和优化 Compaction 配置，避免达到硬性限制

核心监控指标：

+ `VersionCount`：每个 Tablet 的版本数
+ `pending_num`：Compaction 等待队列长度
+ `max_compaction_threads`：Compaction 线程使用率

生产环境建议：

+ 设置版本数 > 300 告警
+ 确保 Compaction 线程数足够（CPU 核心数的 50-100%）
+ 避免在高峰期进行大批量导入
+ 定期执行 Full Compaction（如每周一次）

通过以上措施，可以有效避免版本数达到 500 导致写入拒绝的情况。
