
## Doris 用什么进行监控，监控哪些指标？
“从基础设施到核心组件再到业务链路”的全方位监控思路。

Doris 监控的标准方案是 Prometheus + Grafana

Doris 自身通过 HTTP 接口（FE 的 8030 和 BE 的 8040 端口）暴露了符合 Prometheus 标准的 /metrics 指标。



以下是在实际项目中关注的核心监控指标体系：

1. 核心组件健康度（基础设施）
    1. FE 和 BE 存活节点数 (Alive Nodes)：这是集群稳定性的底线。
    2. JVM & 内存： 
        1. FE JVM： 特别关注 Old GC 次数和时间。FE 存储元数据，如果元数据过多导致频繁 Full GC，会导致整个集群不可用。
        2. BE Memory： BE 负责计算和存储，关注其物理内存占用率，防止触发系统的 OOM Killer。
2. 写入链路监控（针对你的 Flink 场景）：“写入延迟”或“数据堆积”
    1. Routine Load / Stream Load 速率： 查看每秒导入的数据量（Rows/sec）和字节数（Bytes/sec）。
    2. Transaction Status： 监控事务的成功率和失败数。
    3. Compaction Score (核心指标)： 这是我最关注的指标。它反映了数据合并的压力。如果 tablet_max_compaction_score 持续处于高位（比如超过 100），说明 Flink 写入太快或小文件太多，BE 合并不过来。这会直接导致查询变慢，甚至报 Too many versions 错误。
3. 查询性能监控
    1. QPS & RPS： 每秒查询数和每秒请求数，反映集群的负载水平。
    2. Query Latency (分位数)： 关注 P95、P99 查询耗时。如果平均耗时正常但 P99 飙升，通常意味着存在查询倾斜或某些 BE 节点负载过高。
    3. Connection Count： FE 的连接数。如果连接数被打满，会导致新的查询请求无法接入。
4. 存储与系统资源
    1. 磁盘 IO & 使用率： 监控数据目录的磁盘空间（Disk Capacity）。
    2. CPU Usage： 观察查询峰值和写入峰值时的 CPU 占用情况。如果 CPU 长期处于 80% 以上，说明需要扩容 BE 节点。



监控看板建设经验

在项目中，通常配置 Grafana 告警规则：

1. P1 级告警： BE 节点宕机、FE Master 切换、磁盘空间低于 10%。
2. P2 级告警： Compaction Score 过高（>300）、查询延迟环比上升 50%、Flink 写入任务异常挂起。
3. 日志审计： 除了 Prometheus，我还会利用 Doris 的 fe.audit.log 监控慢查询 (Slow Query)。通过分析耗时超过 5 秒的 SQL，定位是否需要添加索引或优化宽表结构。


## 监控指标阈值
在 Flink 实时写入 Doris 的场景下，监控阈值通常分为三个等级：普通告警（P2/Notice）、严重告警（P1/Warning）和灾难告警（P0/Critical）

以下是我在生产环境中总结的核心监控指标阈值参考表：

1. 写入链路核心指标（重点关注 Compaction）：由于 Flink 实时写入最容易导致小文件堆积，这一层的阈值最为敏感。

| 监控指标 | 单位 | P2 (阈值) | P1 (阈值) | 说明与处理建议 |
| --- | --- | --- | --- | --- |
| Tablet Max Compaction Score | 分数 | > 100 | > 300 | 核心核心指标。超过 100 说明合并速度慢于写入速度；超过 300 极易触发 Too many versions 报错导致写入停止。 |
| Transaction Success Rate | % | < 99.9% | < 95% | 事务成功率。若持续走低，检查 Flink Sink 端的 Label 冲突或 BE 磁盘 IO 阻塞。 |
| Stream Load Put Latency | ms | > 500ms | > 2000ms | 单次 Push 的响应时间。过高通常反映 BE 压力大或网络抖动。 |


2. BE 存储与资源指标：BE 是执行计算和存储的主力，其资源的枯竭通常是灾难性的。

| 监控指标 | 单位 | P2 (阈值) | P1 (阈值) | 说明与处理建议 |
| --- | --- | --- | --- | --- |
| BE Memory Usage | % | > 75% | > 90% | 防止进程被系统 OOM Killer 杀掉。达到 P1 时，需限制大查询或扩容。 |
| Disk Usage | % | > 70% | > 85% | 磁盘空间。Doris 在磁盘满 90% 时会强制变为 Read-only，必须留足 Buffer。 |
| Disk IO Utilization | % | > 80% | > 95% | 磁盘繁忙度。如果 IO 长期满载，会直接拖慢 Compaction 和查询。 |
| BE CPU Usage | % | > 60% | > 85% | 计算负载。高峰值正常，持续高位需考虑 Workload Group 资源隔离。 |


3. FE 集群管理指标  
FE 负责元数据和查询分发，稳定性决定了集群的“大脑”是否清醒。

| 监控指标 | 单位 | P2 (阈值) | P1 (阈值) | 说明与处理建议 |
| --- | --- | --- | --- | --- |
| FE JVM Old GC Time | ms | > 1000ms | > 5000ms | 如果发生 Full GC，整个集群查询会“掉线”。需调整 JVM 堆内存或检查元数据量。 |
| Active Connections | 个 | > 1000 | > 2000 | MySQL 协议连接数。过高可能存在连接泄露，需配置连接池。 |
| Edit Log Write Latency | ms | > 200ms | > 1000ms | 元数据持久化延迟。如果 FE 部署在机械硬盘上，易出现此瓶颈。 |


4. 查询性能指标  
反映用户端（看板/接口）的直观感受。

| 监控指标 | 单位 | P2 (阈值) | P1 (阈值) | 说明与处理建议 |
| --- | --- | --- | --- | --- |
| Query Latency P99 | ms | > 2000ms | > 10s | 慢查询阈值。需配合 Audit Log 分析是否由于全表扫描或未命索引引起。 |
| Query Error Rate | % | > 1% | > 5% | 排除掉语法错误的查询失败率。 |


根据业务动态调整？

+ 针对“写多读少”场景： 如果宽表更新非常频繁，可以适度放宽 Compaction Score 的 P2 阈值（比如调到 150），但同时必须调优 cumulative_compaction_policy 为 SizeBased，并增加 BE 的合并线程数。
+ 针对“高并发点查”场景： 必须严格监控 Active Connections 和 FE JVM Memory。因为每一个连接都会消耗 FE 的线程资源，建议在前端接入 Doris Proxy (如 Nginx 或 HAProxy) 进行负载均衡。
+ 容量预警： 我通常会设置一个 7 天趋势预警。如果磁盘占用率在过去 7 天内增长了 20%，即使当前只有 50%，也会触发 P2 告警，提醒我们需要提前申请资源扩容。
