# 为什么需要编码与压缩？
在 OLAP 系统中，数据通常具有以下特征：

+ 列式存储：同一列数据类型一致、值相近或重复；
+ 高冗余性：如时间戳连续、状态字段低基数、数值呈趋势；
+ I/O 是瓶颈：磁盘/网络带宽远低于 CPU 计算能力。



因此，Doris 在 列存引擎（Column Store） 中引入 两层优化：

1. 编码（Encoding）：利用数据分布特性，用更紧凑的二进制格式表示原始值（语义无损）；
2. 压缩（Compression）：对编码后的字节流进行通用压缩（如 LZ4、ZSTD），进一步减少体积。



✅ 目标：减少存储空间 + 降低 I/O + 提升扫描吞吐，同时控制 CPU 开销。

# 整体架构：编码与压缩在 Doris 中的位置
Doris 的数据写入流程如下：Row Data 

1. 按列拆分（Column Chunk） 
2. 应用 Encoding（如 RLE、Dictionary） 
3. 分页（Data Page） 
4. 应用 Compression（如 LZ4） 
5. 写入 Segment 文件（.dat）



编码与压缩：

+ 编码是列级别的，不同列可使用不同编码；
+ 压缩是 Page 级别的，每个 Data Page 独立压缩；
+ 查询时：先解压 → 再解码 → 向量化处理。

# 编码（Encoding）详解
Doris 支持多种智能编码策略，自动或手动选择最优方式。

## 自动编码选择（Auto Encoding）
Doris 在建表时若未显式指定编码，会根据 列类型 + 数据特征 自动选择：

| 列类型 | 默认编码 |
| --- | --- |
| `INT` / `BIGINT`（连续或小范围） | `PLAIN` 或 `BIT_SHUFFLE` |
| `STRING` / `VARCHAR`（低基数） | `DICT_ENCODING` |
| `BOOLEAN` | `RLE` |
| `FLOAT` / `DOUBLE` | `PLAIN `或 `BIT_SHUFFLE` |
| `DATE` / `DATETIME` | `PLAIN`（内部为整数） |


💡 从 Doris 2.0 起，`BIT_SHUFFLE` 成为数值型列的默认高性能编码。

## 主要编码类型及原理
### PLAIN（原始编码）
+ 直接按固定长度存储原始值（如 BIGINT 占 8 字节）。
+ 优点：解码快，CPU 开销极低；
+ 缺点：无压缩效果；
+ 适用：高基数、无规律的数值（如 user_id）。

### DICT_ENCODING（字典编码）
+ 构建 值 → ID 映射表，<font style="background-color:#E8F7CF;">数据存储为 ID</font>（通常 1~4 字节）；
+ 同时存储字典（Dictionary）和编码后的 ID 列；
+ 优点：对低基数字符串/枚举值压缩率极高（可达 90%+）；
+ 缺点：字典需常驻内存，高基数时内存爆炸；
+ 适用：`status`, `city`, `category` 等低基数列（distinct < 10,000）。

📌 注意：Doris 的字典是 <font style="background-color:#E8F7CF;">Page 级别</font> 的，避免全局字典膨胀。

### RLE（Run-Length Encoding，游程编码）
+ 将连续相同值压缩为 `(value, count)`；
+ 优点：对大量重复值（如布尔值、分区字段）极高效；
+ 适用：`is_vip`, `partition_date` 等<font style="background-color:#E8F7CF;">高度重复列。</font>

### BIT_SHUFFLE（位混洗编码）【Doris 核心创新】
+ 专为<font style="background-color:#E8F7CF;">数值型列设计的高性能编码</font>；
+ 原理：将多个数值的 相同 bit 位 重新排列（bit transposition），使高位/低位分别聚集；
+ 例如：4 个 INT 值 `[1001, 1002, 1003, 1004]` → 按 bit 列重组；
+ 重组后，高位几乎全 0，低位呈递增，极大提升后续压缩率；
+ 优点：
    - 兼容向量化计算（无需完全解码即可做谓词下推）；
    - 压缩率优于 PLAIN + LZ4；
    - 解码速度接近 PLAIN；
+ 适用：所有数值型列（INT、BIGINT、DECIMAL、DATE 等）——强烈推荐作为默认。

### FOR（Frame of Reference）
+ 记录最小值，其余值存储为 delta（偏移量）；
+ 对有序或小范围数值有效；
+ Doris 中较少使用，已被 BIT_SHUFFLE 取代。

# 压缩（Compression）详解
编码后的字节流会进一步压缩。

Doris 支持以下压缩算法：

| 算法 | 压缩比 | 压缩速度 | 解压速度 | 适用场景 |
| --- | --- | --- | --- | --- |
| LZ4 | 低~中 | ⚡ 极快 | ⚡ 极快 | 默认推荐，平衡性能与压缩 |
| ZSTD | 高 | 中 | 快 | 存储敏感型场景（如历史冷数据） |
| SNAPPY | 低 | 快 | 快 | 兼容旧系统（Doris 已不推荐） |
| GZIP | 很高 | 慢 | 慢 | ❌ 不支持（因解压太慢） |


默认行为

+ Doris 默认使用 LZ4；
+ 可在建表时全局或按列指定压缩算法。

# 建表时如何配置编码与压缩？
## 全局设置（PROPERTIES）
```plsql
CREATE TABLE sales (
  id BIGINT,
  city VARCHAR(64),
  amount DECIMAL(10,2)
) ENGINE=OLAP
DUPLICATE KEY(id)
PROPERTIES(
  "compression" = "lz4",          -- 全局压缩算法
  "enable_dict_encoding" = "true" -- 启用字典编码（默认开启）
);
```

## 按列指定编码（Doris 2.0+）
```plsql
CREATE TABLE logs (
  ts DATETIME,
  level VARCHAR(16),
  msg TEXT
) ENGINE=OLAP
DUPLICATE KEY(ts)
COLUMNS (
  ts DATETIME ENCODING(BIT_SHUFFLE),
  level VARCHAR(16) ENCODING(DICT_ENCODING),
  msg TEXT ENCODING(PLAIN)
)
PROPERTIES("compression" = "zstd");
```

⚠️ 注意：早期版本（<2.0）不支持列级编码显式指定，依赖自动选择。

# 性能权衡与调优建议
## 编码选择原则
| 场景 | 推荐编码 |
| --- | --- |
| 数值型（INT/BIGINT/DECIMAL） | BIT_SHUFFLE（默认且最优） |
| 低基数字符串（<1万 distinct） | DICT_ENCODING |
| 高基数字符串（如 URL、UUID） | PLAIN（避免字典膨胀） |
| 布尔值、高度重复字段 | RLE |
| 时间戳（有序） | BIT_SHUFFLE（内部为整数） |


## 压缩算法选择
| 目标 | 推荐 |
| --- | --- |
| 最大化查询性能 | LZ4（解压快，CPU 友好） |
| 最小化存储成本 | ZSTD（压缩比高 20%~50%） |
| 实时写入高吞吐 | LZ4（压缩快） |


📊 实测数据（TPC-H SF100）：

+ BIT_SHUFFLE + LZ4 比 PLAIN + LZ4 存储减少 30%，扫描快 15%；
+ ZSTD 比 LZ4 存储再减少 25%，但解压 CPU 增加 10%。

## 调优参数（FE/BE 配置）
| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `column_encode_level` | 1 | 编码级别（1=自动，2=强制高级编码） |
| `default_compression` | lz4 | 集群默认压缩算法 |
| `max_dictionary_size` | 10000 | 字典编码最大字典大小（防 OOM） |


# 监控与诊断
## 查看表编码/压缩信息
```sql
SHOW CREATE TABLE your_table;
-- 输出中包含 ENCODING 和 PROPERTIES
```

## 分析 Segment 文件结构（高级）
```bash
# 使用 doris-tools 中的 segment_dump 工具
segment_dump --file=/path/to/segment.dat
```

可查看每列的编码类型、压缩后大小、字典大小等。

## 性能影响判断
+ 若 CPU 使用率高但 I/O 低 → 可能压缩过重（尝试 LZ4）；
+ 若 存储膨胀严重 → 检查是否高基数列误用了 DICT_ENCODING；
+ 若 谓词下推无效 → 检查是否使用了不支持下推的编码（如旧版 FOR）。

# 最新进展（Doris 2.1~2.2）
1. 自适应编码（Adaptive Encoding）：  
写入时采样数据分布，动态选择最优编码（实验性）。
2. 延迟物化（Late Materialization） + 编码协同：  
谓词下推阶段仅解码必要列，进一步减少 CPU 开销。
3. ZSTD 多线程压缩：  
利用多核加速 ZSTD 压缩，缩小与 LZ4 的性能差距。

# 总结：最佳实践清单
✅ 默认策略：

+ 数值列 → BIT_SHUFFLE + LZ4 
+ 低基数字符串 → DICT_ENCODING + LZ4 
+ 高基数文本 → PLAIN + LZ4

✅ 避免：

+ 对 UUID、MD5 等高基数字符串启用字典编码；
+ 在 CPU 敏感型查询中使用 ZSTD（除非存储极度紧张）；
+ 手动指定编码除非有明确收益（自动选择已很智能）。

✅ 进阶优化：

+ 冷数据表使用 ZSTD；
+ 日志类表对 message 列用 PLAIN（避免字典开销）；
+ 结合 Sort Key 设计，提升 BIT_SHUFFLE 和 RLE 效果。
