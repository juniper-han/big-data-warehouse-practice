# 总体设计思想
Doris 的导入系统遵循以下核心原则：

1. 原子性：单次导入要么全部成功，要么全部失败（ACID 中的 A）
2. 一致性：<font style="background-color:#E8F7CF;">导入完成后，所有副本数据一致，查询立即可见</font>
3. 高吞吐 & 低延迟：支持从 KB/s 到 GB/s 的写入场景
4. Exactly-Once 语义：避免重复导入（尤其在 Kafka 场景）
5. 解耦调度与执行：FE 负责协调，BE 负责实际写入



📌 所有导入方式最终都通过 <font style="background-color:#E8F7CF;">两阶段提交（2PC） + 多版本并发控制（MVCC） 实现事务一致性</font>。

# 四大导入方式详解
Apache Doris 的导入系统是一个 高度可靠、语义清晰、工程完备 的数据接入层：

+ Stream Load：简单高效，适合推模式；
+ Broker Load：兼容 Hadoop 生态，适合离线；
+ Routine Load：实时流式王者，唯一原生支持 Exactly-Once；
+ Insert Into：SQL 友好，适合轻量 ETL。

其底层通过 2PC + MVCC + Label/Offset 精确管理，实现了 强一致性、高可用、Exactly-Once 的企业级能力。

## Stream Load（同步 HTTP 推模式）
✅ 适用场景

+ 小批量、低延迟 数据写入（如日志上报、API 写入）
+ 客户端主动推送（Push-based）
+ 单次导入数据量：100MB ~ 1GB



🔧 使用方式

```bash
curl --location-trusted -u user:passwd \
    -H "label:load_20250304_001" \
    -H "column_separator:, " \
    -T data.csv http://fe_host:8030/api/db/_stream_load
```



⚙️ 底层实现原理

1. 客户端 → FE：HTTP 请求携带数据和 Label；
2. FE 路由：根据表分桶规则，将数据 Shuffle 到对应 BE；
3. BE 写入：
    - 数据进入 Delta Writer（内存缓冲区）；
    - 按 Sort Key 排序，生成列式 Segment；
4. 两阶段提交（2PC）：
    - Prepare 阶段：各 BE 将 Rowset 写入磁盘，返回 prepare 成功；
    - Commit 阶段：FE 向所有 BE 发送 publish version，原子更新 Tablet 元数据；
5. 结果返回：FE 汇总状态，返回客户端 success/failure。



🛡️ 事务与 Exactly-Once

+ Label 机制：每个导入任务必须指定唯一 `label`；
+ 幂等性：重复提交相同 label → 返回上次结果（不重复写入）；
+ Exactly-Once 保障：依赖客户端保证 label 唯一（如用 UUID 或 Kafka offset 生成）。



🔄 容错机制

+ BE 故障：FE 检测到 BE 不可用，自动重试其他副本；
+ 网络中断：客户端可重试（使用相同 label 实现幂等）；
+ 部分失败：2PC 保证要么全成功，要么全回滚（Rowset 文件清理）。

## Broker Load（异步拉模式，基于 HDFS/S3）
✅ 适用场景

+ 大批量离线数据导入（TB 级）
+ 数据源在 HDFS、S3、OSS、BOS 等对象存储
+ 允许小时级延迟



🔧 使用方式

```sql
LOAD LABEL db.load_20250304_001
(
    DATA INFILE("hdfs://path/to/data/*")
    INTO TABLE tbl
    COLUMNS TERMINATED BY ","
)
WITH BROKER "my_broker"
PROPERTIES (
    "timeout" = "3600",
    "max_filter_ratio" = "0.1"
);
```



⚙️ 底层实现原理

1. FE 解析任务：切分文件为多个 FileSlice（按 block size）；
2. 调度到 BE：每个 BE 通过 Broker Agent 并行拉取 FileSlice；
3. BE 执行：
    - 从 HDFS/S3 读取原始文件；
    - 解析、转换、排序后写入本地 Delta Writer；
4. 2PC 提交：同 Stream Load；
5. 异步返回：通过 `SHOW LOAD` 查看状态。



🛡️ 事务与 Exactly-Once

+ Label 幂等：同 Stream Load；
+ 无原生 Exactly-Once：因数据源无 offset 语义，需外部保证（如一次性移动文件）；
+ 建议：配合 Airflow/Oozie 实现“文件处理完即删除”避免重复。



🔄 容错机制

+ 文件读取失败：自动重试（可配重试次数）；
+ BE 故障：FE 重新调度 FileSlice 到其他 BE；
+ 数据过滤：支持 `max_filter_ratio` 跳过脏数据。

## Routine Load（持续消费 Kafka）
✅ 适用场景

+ 实时流式导入（Kafka → Doris）
+ 要求 Exactly-Once 语义
+ 高可用、自动故障恢复



🔧 使用方式

```sql
CREATE ROUTINE LOAD db.rl_kafka ON tbl
PROPERTIES(
    "desired_concurrent_number" = "3",
    "max_batch_interval" = "20",
    "max_batch_rows" = "200000"
)
FROM KAFKA(
    "kafka_broker_list" = "host:9092",
    "kafka_topic" = "user_log",
    "property.kafka_default_offsets" = "OFFSET_BEGINNING"
);
```



⚙️ 底层实现原理（核心：**Kafka Consumer Group + 2PC + Offset Commit**）

1. FE 创建 Routine Load Job：
    - 注册为常驻任务；
    - 启动 Job Scheduler；
2. Task 分发：
    - 每个 Partition 分配给一个 RoutineLoadTask；
    - Task 调度到 BE 执行；
3. BE 消费 Kafka：
    - 使用 Kafka Consumer 拉取一批消息（batch）；
    - 解析后写入本地 Delta Writer；
4. 2PC 提交 + Offset 提交：
    - 关键点：Offset 提交与数据提交 绑定在同一事务中；
    - 只有当 2PC 成功后，才将 Kafka offset 更新到 FE 元数据；
5. 心跳与恢复：
    - BE 定期上报消费进度；
    - FE 监控任务健康，自动重启失败 Task。



🛡️ Exactly-Once 语义实现（核心优势！）

| 步骤 | 机制 | 作用 |
| --- | --- | --- |
| 1. Offset 管理 | FE 维护每个 Partition 的 committed offset | 避免重复消费 |
| 2. 事务绑定 | 数据写入 + offset 更新 在同一 2PC 事务中 | 原子性保证 |
| 3. 幂等写入 | 使用 `(job_id, partition, offset_range)`作为内部 label | 防止 Task 重复提交 |
| 4. 故障恢复 | 从 FE 记录的 `last_committed_offset` 继续消费 | 精确恢复 |


✅ 效果：即使 BE/FE 宕机，也不会丢数据、不会重复数据。



🔄 容错机制

+ Kafka 连接失败：自动重连；
+ BE 宕机：FE 将 Task 迁移到其他 BE，从上次 committed offset 继续；
+ 数据格式错误：支持跳过（`max_error_number`）或暂停任务；
+ 背压控制：当 Doris 写入慢时，自动降低 Kafka 拉取速率。

## Insert Into（SQL 写入）
✅ 适用场景

+ ETL 中间结果写入（如 `INSERT INTO tbl2 SELECT ... FROM tbl1`）
+ 小批量测试数据插入
+ 与 BI 工具集成



🔧 使用方式

```sql
-- 方式1：VALUES
INSERT INTO tbl VALUES (1, 'Alice'), (2, 'Bob');

-- 方式2：SELECT
INSERT INTO tbl2 
SELECT id, name FROM tbl1 WHERE dt = '2025-03-04';
```



⚙️ 底层实现原理

1. FE 执行查询计划：
    - 对于 `INSERT ... SELECT`，先执行 SELECT 子查询；
    - 结果集在 FE 或 BE 中暂存；
2. 数据分发：
    - 将结果按目标表分桶规则 Shuffle 到 BE；
3. BE 写入：
    - 走与 Stream Load 相同的 Delta Writer + 2PC 流程；
4. 事务提交：原子生效。



🛡️ 事务特性

+ <font style="background-color:#E8F7CF;">单语句原子性：整个 </font>`<font style="background-color:#E8F7CF;">INSERT</font>`<font style="background-color:#E8F7CF;"> 要么成功，要么失败</font>；
+ 无 Label 机制：不能手动指定 label，但内部自动生成；
+ <font style="background-color:#E8F7CF;">非 Exactly-Once：重复执行 SQL 会重复写入（需应用层控制）</font>。



⚠️ 注意事项

+ 性能较低：不适合大批量写入（建议 >10K 行用 Stream Load）；
+ 资源消耗大：`SELECT` + `INSERT` 占用查询和写入资源；
+ 不支持 Kafka 源：仅适用于 Doris 内部表或 Catalog 表。

# 统一事务机制：2PC + MVCC
所有导入方式共享同一套事务模型：

## 两阶段提交（2PC）
+ Phase 1（Prepare）：
    - BE 将 Rowset 写入磁盘，注册到 Tablet 的 `pending list`；
    - 返回 `prepare ok`；
+ Phase 2（Commit / Abort）：
    - 若所有 BE 成功 → FE 发送 `publish version`，Tablet 原子切换版本；
    - 若任一失败 → FE 发送 abort，BE 清理临时文件。

## 多版本并发控制（MVCC）
+ 每个 Rowset 有 版本区间` [start_version, end_version]`；
+ 查询基于 publish version 快照，只读取 ≤ 该版本的 Rowset；
+ Compaction 后生成新版本，旧版本可安全回收。

✅ 效果：写入不影响查询，查询看到一致快照。

# 作业调度与资源管理
## 调度器角色
| 组件 | 职责 |
| --- | --- |
| FE Master | 全局任务调度、状态管理、元数据更新 |
| BE Executor | 执行具体导入任务（拉数据、写磁盘） |
| Heartbeat | BE 定期上报负载，FE 动态分配任务 |


## 资源隔离
+ 导入内存限制：`write_buffer_size`（默认 128MB/Writer）
+ 并发控制：
    - Stream Load：受 `streaming_load_max_memory` 限制；
    - Routine Load：`desired_concurrent_number` 控制并发 Task 数；
+ 优先级：暂不支持，但可通过 Label 命名规范间接管理。

# 容错与高可用全景图
| 故障类型 | 处理机制 |
| --- | --- |
| FE 宕机 | Follower 自动选举新 Leader，导入任务状态持久化在 BDBJE，可恢复 |
| BE 宕机 | FE 标记 unavailable，未完成任务重调度到其他 BE；副本自动修复 |
| 网络分区 | 2PC 超时触发 abort，客户端可重试 |
| Kafka 故障（Routine Load） | 暂停任务，恢复后从 committed offset 继续 |
| 数据源损坏（Broker Load） | 跳过或失败，记录错误日志 |


✅ 核心保障：元数据强一致（BDBJE） + 2PC 原子提交 + Offset 精确管理

# 选型建议与最佳实践
| 场景 | 推荐方式 | 关键配置 |
| --- | --- | --- |
| 实时日志上报 | Stream Load | 合理设置 label，批量提交（>10MB/批） |
| 离线数仓同步 | Broker Load | 切分大文件，监控 filter ratio |
| Kafka 实时管道 | Routine Load | 开启 Exactly-Once，设置合理 batch size |
| ETL 中间表 | Insert Into | 仅用于 <100K 行，避免长事务 |
| CDC 同步 | Routine Load + Unique Key MoW | 配合 sequence column |
