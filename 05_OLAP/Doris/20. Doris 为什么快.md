## Doris 为什么快？
维度：存储引擎底层设计、执行引擎优化、以及工程实现细节

四个核心维度

1. 存储层：极致的列式存储与索引设计
    1. Doris 的核心是自研的列式存储格式，这决定了它在处理海量数据扫描时的天然优势
        1. 列式存储 (Columnar Storage)：   查询时只读取需要的列，极大减少了磁盘 IO
        2. 压缩比： 同一列数据类型相同，可以使用更高压缩比的算法（如 LZ4、Zstd），进一步减少存储开销和缓存压力。
    2. 多级索引加速：
        1. 前缀索引 (Short Key Index)： 自动为前几列生成稀疏索引，快速定位数据块。
        2. Zone Map： 在每个 Data Page 级别维护 Max/Min 值，实现快速的数据范围过滤（Skipping）
        3. 位图索引 (Bitmap) 与 Bloom Filter： 针对低基数列和等值查询，可以实现不读数据直接过滤。
2. 计算层：MPP 架构与向量化执行
    1. Doris 采用了 MPP (Massive Parallel Processing) 架构，能够充分利用多机、多核的并行能力
    2. 向量化执行 (Vectorized Execution)： 这是 Doris 1.2 版本后的质变。传统的执行引擎是逐行处理，而向量化引擎是按批处理 (Batch Processing)。它利用了现代 CPU 的 SIMD (Single Instruction Multiple Data) 指令集，在一次 CPU 指令中处理一组数据，大幅提升了计算吞吐。
    3. 计算下推 (Predicate Pushdown)： 尽可能将过滤、聚合操作下推到存储节点（BE），减少网络传输的数据量。
3. 分区与分桶的精准裁剪：先分区 (Partition)，后分桶 (Bucket)
    1. 查询裁剪： 在执行 SQL 时，Doris 会根据查询条件中的时间（分区键）和 ID（分桶键）直接锁定特定的数据分片 (Tablets)，剔除掉 90% 以上无关的数据。这在实时宽表场景中，针对特定维度、特定时间的点查或聚合极其高效。
4. 独特的模型优化 (Unique Key & Aggregate Key)：针对你提到的 Upsert 场景，Doris 的快还体现在“处理更新时的效率”：
    1. 写快：采用类似 LSM-Tree 的结构，写入时是追加写 (Append-only)，保证了高频写入的吞吐
    2. 读快： 在 Unique Key 模型下，Doris 引入了 Delete Bitmap（主键模型），在读取时能直接跳过被覆盖的旧数据，避免了昂贵的实时合并 (Merge-on-Read) 开销
    3. 物化视图 (Materialized View)： 支持预聚合。如果你的宽表经常有固定的聚合查询，Doris 会在后台自动维护这些预计算结果，查询时直接命中。
