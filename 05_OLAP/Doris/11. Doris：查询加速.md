Doris 中的查询加速技术体系：覆盖存储层、计算层、优化器层、建模层四大维度

Apache Doris 的查询加速体系是 工程与算法的完美结合：

+ 存储层：通过 分区/索引/编码 最小化 I/O；
+ 计算层：通过 向量化/MPP/缓存 最大化 CPU；
+ 优化器层：通过 RBO/CBO/改写 选择最优路径；
+ 建模层：通过 物化视图/宽表 预计算结果。

🔑 终极心法：  
“<font style="background-color:#E8F7CF;">让数据尽可能少地移动，让计算尽可能靠近数据</font>”

# 存储层加速：I/O 最小化
## 分区（Partition）与分桶（Bucket）裁剪
+ 分区裁剪（Partition Pruning）
    - 基于 `RANGE` 或 `LIST` 分区键（如 `dt = '2025-03-04'`） 
    - 效果：跳过 90%+ 无关数据文件 
    - 最佳实践：时间字段作分区键，单分区 ≤ 100GB
+ 分桶裁剪（Bucket Pruning）
    - 若查询条件包含 分桶键（`DISTRIBUTED BY HASH(key)`），可定位具体 Tablet 
    - 示例：`WHERE user_id = 12345` → 仅扫描 1/N 个 Tablet（N=分桶数）

📌 关键：合理设置分桶数（BE 数 × 10），避免 Tablet 过多或过少

## 多级索引体系（Doris 核心优势）
| 索引类型 | 原理 | 适用场景 | 加速效果 |
| --- | --- | --- | --- |
| Shortkey Index | Sort Key 前缀稀疏索引 | 范围查询（如时间范围） | 跳过无关 Data Page |
| ZoneMap Index | 每 Page 的 Min/Max | 所有列的范围过滤 | 自动启用，零成本 |
| BloomFilter Index | 布隆过滤器（概率型） | 高基数列等值查询：如用户ID | 减少 80%+ Page 读取 |
| Bitmap Index | 位图索引 | 低基数列（<1000 distinct） | AND/OR 快速过滤 |
| Inverted Index | 倒排索引（Lucene） | 全文检索、模糊匹配 | 替代 `LIKE '%xxx%'` |


💡 调优建议：

+ 时间列放 Sort Key 第一位；
+ 高基数字符串启用 BloomFilter（fpp=0.01~0.05）；
+ 维度列（status/city）启用 Bitmap；
+ 日志关键词启用 Inverted Index（Doris 2.0+）

## 编码与压缩：减少物理 I/O
1. BIT_SHUFFLE 编码（数值列默认） 
    - 位混洗 + LZ4 压缩 → 存储减少 30%，扫描提速 15%
2. DICT_ENCODING（低基数字符串） 
    - 字典压缩率可达 90%+
3. RLE（高度重复值） 
    - 如布尔值、分区字段

✅ 效果：同等硬件下，I/O 带宽需求降低 2~5 倍

# 计算层加速：CPU 最大化利用
## 向量化执行引擎（Vectorized Execution）
+ 原理：以 Chunk（1024 行）为单位，按列处理，利用 SIMD 指令（AVX2/AVX-512）
+ 支持算子：Scan、Filter、Project、Agg、Join Probe
+ 性能收益：3~10 倍吞吐提升（TPC-H 测试）

📊 示例：`WHERE clicks > 100`

+ 行式：每行一次分支预测 
+ 向量化：1024 行并行比较，无分支

## MPP + Pipeline 执行模型
+ MPP（Massively Parallel Processing）
    - 多 BE 并行扫描本地 Tablet（数据本地化）
+ Pipeline Engine（Doris 2.0+）
    - 算子链（Operator Chain）异步执行
    - 计算与 I/O 重叠，CPU 利用率提升 30%+

✅ 效果：高并发点查 P99 延迟降低 50%

## 多级缓存体系
| 缓存类型 | 内容 | 默认占比 | 作用 |
| --- | --- | --- | --- |
| Page Cache | Data/Index Page | 30% 内存 | 减少磁盘 I/O |
| Metadata Cache | Segment Footer | 1GB | 加速元数据访问 |
| PK Index Cache | 主键索引（MoW） | 可配 | 加速 Unique Key 更新 |
| Result Cache（实验性） | 查询结果 | - | 相同查询秒级返回 |


🔧 调优：`page_cache_limit = 80%`（热数据场景）

# 优化器层加速：智能计划选择
## RBO（Rule-Based Optimization）
+ Join Reordering：小表在左 → Broadcast Join
+ 谓词/Project 下推：尽早过滤、裁剪列
+ Aggregate 下推：提前聚合减少数据量

## CBO（Cost-Based Optimization）【Doris 2.0+ 强化】
+ 依赖统计信息：

```sql
ANALYZE TABLE sales; -- 收集行数、NDV、Min/Max
```

+ 代价模型：综合 CPU、I/O、Network 成本
+ 动态规划：选择最优 Join Order

💡 最佳实践：对 >10 亿行的大表定期 `ANALYZE`

## 查询改写（Query Rewrite）
+ 物化视图自动匹配：无需改 SQL，透明加速
+ 子查询转 Join：`IN` → Semi Join
+ 派生表下推：FROM 子查询谓词下推

🔍 诊断：`EXPLAIN` 查看是否命中 MV 或下推成功

# 建模层加速：预计算与数据组织
## 物化视图（Materialized View）
### （1）同步 Rollup（强一致）
```sql
ALTER TABLE logs ADD ROLLUP r1 (dt, city, SUM(pv));
```

+ 优势：写时预聚合，查时极致快
+ 适用：固定维度报表

### （2）异步 MV（灵活）
```sql
CREATE MATERIALIZED VIEW mv AS
SELECT city, SUM(revenue) FROM sales GROUP BY city;
```

+ 优势：支持 Join、表达式、跨表
+ 适用：复杂 ETL、星型模型

✅ 加速效果：10 亿行聚合从 8s → 0.3s

## 宽表模型 vs 星型模型
| 模型 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- |
| 宽表 | 无 Join，查询快 | 冗余高，ETL 复杂 | 实时报表、用户行为分析 |
| 星型 | 规范化，易维护 | 需 Join | 传统数仓、多维分析 |


🚀 Doris 推荐：宽表打底 + Rollup 加速；复杂场景用 异步 MV 预 Join

## 数据类型与表达式优化
+ 避免运行时转换： 
    - 存储为 `DATE` 而非 `STRING`
    - 使用 `DECIMAL` 而非 `DOUBLE`（精确计算）
+ 表达式下推： 
    - `WHERE DATE(dt) = '2025-03-04'` → ❌（无法下推） 
    - `WHERE dt = '2025-03-04'` → ✅（直接利用 ZoneMap）

# 高级加速技术
## Colocation Join（协同定位）
+ 原理：相同分桶键的表，数据已在同一 BE
+ 效果：完全避免 Shuffle，Join 速度提升 5~10 倍
+ 启用：

```sql
PROPERTIES("colocate_with" = "group1");
```

## Bucket Shuffle Join
+ 原理：仅 Shuffle 缺失的 Bucket
+ 效果：比普通 Shuffle 减少 50%+ 网络流量

##  延迟物化（Late Materialization）
+ 原理：先计算过滤条件（仅需部分列），再物化结果列
+ 效果：减少内存带宽，尤其适合 宽表 + 少数列查询

# 查询加速调优 Checklist
| 维度 | 检查项 |
| --- | --- |
| 建模 | 是否使用宽表？Rollup/MV 是否覆盖高频查询？ |
| 分区分桶 | 分区键是否用于过滤？分桶数是否合理？ |
| 索引 | 高基数列是否 BloomFilter？低基数列是否 Bitmap？ |
| Sort Key | 高频过滤字段是否在前？ |
| 统计信息 | 大表是否 `ANALYZE`？ |
| Join | 是否 Colocation？小表是否 Broadcast？ |
| 缓存 | Page Cache 是否足够？ |
| SQL | 谓词是否可下推？是否避免 UDF？ |


# 典型场景加速方案
## 场景 1：实时用户行为分析（10 亿行）
+ 建模：宽表（user_id, event_time, event_type, ...）
+ 分区：按天分区
+ Sort Key：`(event_time, user_id)`
+ 索引：`event_type` 建 Bitmap，`user_id` 建 BloomFilter
+ 查询：`SELECT COUNT(*) FROM tbl WHERE event_time BETWEEN ... AND event_type = 'click';`
+ 效果：P99 < 200ms

## 场景 2：多维销售报表（星型模型）
+ 建模：事实表 + 维表
+ 加速：创建异步 MV 预 Join + 聚合
+ 查询：自动改写到 MV
+ 效果：复杂报表从 15s → 1s
