# 索引技术总体设计思想
Doris 的索引体系遵循 “存储计算协同” + “按需启用” 原则：

+ 所有索引<font style="background-color:#E8F7CF;">均内嵌于列存文件（Segment）中，无需额外维护独立索引服务</font>；
+ 索引在 <font style="background-color:#E8F7CF;">数据写入时构建</font>，查询时由执行引擎自动决策是否使用；
+ 目标是 减少 I/O、跳过无关数据块（Data Page），提升谓词下推效率。

📌 注意：Doris 的索引是 轻量级、列级别、面向 OLAP 场景 的，不同于传统数据库的 B+Tree 行级索引。

# 前缀索引（Shortkey Index）
作用：快速定位 Segment 内的目标 Page。

## 实现原理
+ 基于 Sort Key（排序键）构建的稀疏索引。
+ 每 1024 行（默认）采样一行的排序键（Sort Key），构成一个逻辑数据块 (Data Block)；
    - 每个 Data Block 在前缀索引表中存储一个索引项，索引项的长度不超过 36 字节，值为 Data Block 中第一行数据的排序列组成的前缀
    - 注意：排序列，如果遇到 `VARCHAR` 类型时，前缀索引会直接截断，即使没有达到 36 字节，后面的列不再加入索引
+ 索引存储在 Segment 文件头部，内存加载快。
+ 前缀索引比较小，可以全量缓存在内存中，构建 B+Tree-like 索引（内存常驻）；
+ <font style="background-color:#E8F7CF;">前缀索引的粒度是 Block，不是单行数据。</font>
    - 它的作用是快速排除完全不相关的 Block，而不是精确到行
+ 查询时，若 `WHERE` 条件包含 Sort Key 的前缀列，可快速定位到起始/结束 Data Page。

## 创建与使用
创建：表必须定义 `DUPLICATE KEY` 或 `UNIQUE KEY` 的前缀列作为排序键。

+ 前缀索引**无需专门定义**，建表时自动取表的 Key 的前 36 字节作为前缀索引
+ 一个表的 Key 定义是唯一的，所以一个表只有一组前缀索引



使用

能加速时**自动生效**，没有特殊语法。

可以通过 Query Profile 中的下面几个指标分析前缀索引的加速效果。

+ `RowsKeyRangeFiltered`：前缀索引过滤掉的行数，可以与其他几个 Rows 值对比分析索引过滤效果



效果：点查/范围查性能提升 10x+。

⚠️ 注意：若查询条件不含排序键前缀列，则稀疏索引失效！

## 适用场景
+ 高选择性范围查询（如 `WHERE event_time BETWEEN '2025-01-01' AND '2025-01-02'`）
+ 等值或 IN 查询 on Sort Key 前缀列
+ 所有表默认启用，无法关闭

## 使用建议
1. 选择查询中最常用于 `WHERE` 过滤条件的字段作为 `Key`，越常用的字段越放在前面
2. 因为前缀索引遇到字符串会截断，因此字符串尽量放到前缀索引的后面几列
3. 不影响前缀索引的命中前提下，尽量把 `Int` 列放在前缀索引的第一位，`Int` 的过滤速度比字符串块
4. 将高频过滤字段 + 高基数字段 放在 Sort Key 前面；
5. **将过滤性高（区分度高）的列放在前面**。用于快速排除大量无关数据块
6. 避免将低基数列（如 gender='M/F'）放首位，否则索引区分度低；
7. 时间列通常放第一位，便于分区+索引双重裁剪。
8. 最多取前 3 列作为 前缀索引（即使 Sort Key 有 5 列）

# ZoneMap 索引（Min/Max 索引）
作用：在 Segment / Page 级别 过滤无效数据块。

## 实现原理
+ 每个 Data Page 自动记录该页中每列的 min/max 值（数值型、日期型、字符串型均支持）。
+ 存储位置：
    - Segment Footer：整段 Min/Max；
    - 每个 Page Header：该 Page 的 Min/Max
+ 属于元数据索引，开销极小（每页仅 16~32 字节）。
+ 查询时，若谓词为 `col > X` 或 `col < Y`，可直接跳过不满足 min/max 范围的 Page。

例如：某 Page 中 `age` 的 min=18, max=65，若查询 `WHERE age > 70`，则整页跳过。

## 适用场景
+ 所有数值/时间/字符串列的范围查询
+ 默认开启，无需显式创建
+ 对 有序数据效果极佳（因 min/max 范围窄）

## 调优方法
+ 无配置参数，完全自动；
+ 通过 合理设置 Sort Key 提升数据局部有序性，从而缩小 ZoneMap 范围；
+ 避免在乱序导入（如 Kafka 乱序事件）后立即查询，可触发 Compaction 重排序。

✅ 注意：ZoneMap 对 等值查询无效（需配合 BloomFilter 或 Bitmap）。

# BloomFilter 索引
作用：高效判断 某值“不存在”

## 实现原理
BloomFilter：由 Bloom 在 1970 年提出的一种多哈希函数映射的快速查找算法，某元素是否存在的结果有 2 种

+ 可能存在
+ 一定不存在

因此基于 BloomFilter 的索引只能跳过不满足条件的数据，不能精确定位满足条件的数据。



BloomFilter 索引

+ 基于布隆过滤器（Bloom Filter）的概率型索引。
+ 为指定列的每个 Data Page 构建一个 BloomFilter 结构（位数组 + 多个哈希函数）。
+ 支持快速判断 某值“一定不存在”或“可能存在”。
+ 可有效加速 高基数列的等值查询（=, IN）
+ 不支持对 Tinyint、Float、Double 类型的列建 BloomFilter 索引

## 适用场景
+ 高基数字符串列（如 user_id、device_id、url）
+ 等值或 IN 查询（如 `WHERE user_id = 'U12345'`）
+ 不适用于范围查询、低基数列（易误判）

## 创建与管理
1. 建表时创建 BloomFilter 索引：由于历史原因，BloomFilter 索引定义的语法与倒排索引等通用 INDEX 语法不一样，在 PROPERTIES 中定义

```plsql
CREATE TABLE IF NOT EXISTS sale_detail_bloom  (
)
Duplicate  KEY(...)
PROPERTIES (
...
"bloom_filter_columns" = "column_name1,column_name2"
);
```

2. 查看 BloomFilter 索引：`SHOW CREATE TABLE table_name;`
3. 已有表增加 BloomFilter 索引：为 column_name3 增加 BloomFilter 索引

```plsql
ALTER TABLE table_name SET ("bloom_filter_columns" = "column_name1,column_name2,column_name3");
```

4. 已有表删除 BloomFilter 索引：删除 column_name1 的 BloomFilter 索引：重新为 col2 和 col3 创建索引

```plsql
ALTER TABLE table_name SET ("bloom_filter_columns" = "column_name2,column_name3");
```

5. 能加速时自动生效，可以通过 Query Profile 中的指标分析索引加速效果。
    - `RowsBloomFilterFiltered`：过滤掉的行数
    - `BlockConditionsFilteredBloomFilterTime`：索引消耗的时间

## 调优参数
| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `bloom_filter_fpp` | 0.05（5%） | 误判率（False Positive Probability），越低越准但空间越大 |
| 单列 BloomFilter 大小 | ~1KB/Page | 受 fpp 和 distinct count 影响 |


## 最佳实践：
+ 仅对 真正需要加速的高基数列 启用（每列增加 ~10% 存储）；
+ 避免对低基数列（如` status in ('success','fail')`）使用（BloomFilter 优势不明显）；
+ 误判不会导致结果错误，只会多读一些 Page，安全可靠。
+ 代价：额外存储空间（约 1~5%），构建/查询 CPU 开销。
+ 对 `IN` 和 `=` 之外的查询没有效果，比如` !=, NOT IN, >, < `等

# Bitmap 索引
作用：<font style="background-color:#E8F7CF;">加速低基数列的等值/</font>`<font style="background-color:#E8F7CF;">IN</font>`<font style="background-color:#E8F7CF;"> 查询。</font>

## 实现原理
+ 为低基数列构建位图索引（Bitmap Index）。
+ 每个唯一值对应一个 Bitmap，标记哪些行包含该值。
+ 支持高效的 `AND/OR/NOT` 位运算，适合多条件组合过滤。

例如：`gender` 列有 `'M' `和 `'F'`，则：

+ `Bitmap_M = [1,0,1,1,0,...]`
+ `Bitmap_F = [0,1,0,0,1,...]`

## 适用场景
+ 低基数列（distinct values < 1000）
+ 多维交叉过滤（如 `WHERE city='Beijing' AND category='Electronics'`）
+ `COUNT DISTINCT` 加速（配合 BITMAP 类型列）

## 创建方式
```sql
ALTER TABLE sales ADD INDEX idx_city (city) USING BITMAP;
-- 或建表时指定（Doris 2.0+ 支持）
```

## 限制与注意事项
+ 仅支持以下类型：`TINYINT`、`SMALLINT`、`INT`、`BIGINT`、`CHAR`、`VARCHAR`（长度 ≤ 50）
+ 不适用于高基数列（Bitmap 会爆炸式增长）
+ 占用额外存储（可能达原始数据 20%~50%）

## 最佳实践：
+ 用于 维度表关联字段（如 `region`、`product_type`、`status`）
+ 与 Aggregate 模型 + BITMAP_UNION 结合，实现秒级 UV 计算；
+ 避免在频繁更新的 Unique 表上使用（Bitmap 维护成本高）。

# 倒排索引（Inverted Index）
【Doris 2.0+ 新特性】

作用：支持 全文检索、模糊匹配、数组/JSON 查询。

## 实现原理
+ 借鉴搜索引擎技术，为文本列构建 词项 → 行号列表 的映射。
+ 支持 全文检索、模糊匹配、多词组合。
+ 基于 Apache Lucene 或自研轻量级倒排结构（Doris 2.1+ 逐步优化）。
+ 代价：写入性能下降 20%~50%，存储膨胀 1.5~3x



Doris 倒排索引实现

+ Table 的一行对应一个文档，一列对应文档中的一个字段
+ 根据关键词可以快速定位包含它的行
+ 跟其他索引不同，倒排索引使用独立文件存储，与数据文件一一对应，在创建、删除索引时不用重写数据文件

## 适用场景
+ 日志分析中的关键词搜索（如 `message LIKE '%error%'`）
+ 用户评论、商品描述的模糊查询
+ 替代低效的 `LIKE '%xxx%'` 全表扫描

## 创建方式（Doris 2.1+）
```sql
CREATE TABLE app_log (
    id BIGINT,
    msg TEXT,
    INDEX idx_msg (msg) USING INVERTED
) ENGINE=OLAP
DUPLICATE KEY(id);
```

## 支持的查询语法
```sql
-- 全文匹配
SELECT * FROM app_log WHERE msg MATCH 'error timeout';

-- 模糊前缀（需分词器支持）
SELECT * FROM app_log WHERE msg LIKE 'err%';

-- 布尔组合
SELECT * FROM app_log WHERE msg MATCH '+error -warning';
```

## 调优参数
| 参数 | 说明 |
| --- | --- |
| `inverted_index_parser` | 分词器（standard、simple、ngram） |
| `ngram_size` | N-gram 分词长度（用于中文/模糊匹配） |


## 最佳实践：
+ 仅对 真正需要全文检索的 `TEXT`/`VARCHAR` 列 启用；
+ 中文场景建议使用 `ngram` 分词（如 `ngram_size=2`）；
+ 倒排索引会显著增加 写入延迟和存储开销（约 1.5~2 倍），需权衡；
+ 适合 日志、监控、客服对话 等非结构化文本场景。

## 综合调优建议
1. 优先利用免费索引：Shortkey + ZoneMap 是“零成本”加速器，务必设计好 Sort Key。
2. 索引不是越多越好：每种索引都增加写入开销和存储成本，按需启用。
3. 监控索引效果：
    - 通过 `EXPLAIN` 查看是否命中索引；
    - 通过 `SHOW PROC '/backends'` 观察 I/O 和 Page 跳过率。
4. 结合物化视图：对复杂过滤+聚合场景，可建带索引的物化视图。
5. 版本升级收益：Doris 2.0+ 的倒排索引、2.1+ 的 Partial Update + MoW，极大扩展了索引应用场景。
