# Join：分布式 Join 的实现与优化
Doris 支持多种 Join 算法，其选择直接影响查询性能。

核心思想是：<font style="background-color:#E8F7CF;">减少网络 Shuffle、利用数据分布特性</font>。

## Join 类型支持
| SQL 语法 | Doris 支持 | 说明 |
| --- | --- | --- |
| `INNER JOIN` | ✅ | 默认 |
| `LEFT/RIGHT [OUTER] JOIN` | ✅ | 支持 |
| `FULL OUTER JOIN` | ❌（截至 2.2） | 不支持 |
| `CROSS JOIN` | ✅（需显式） | 笛卡尔积，慎用 |
| `SEMI JOIN` / `ANTI JOIN` | ✅（通过子查询改写） | 如 `WHERE a IN (SELECT b FROM t2)` |


## Join 执行算法（物理实现）
Doris 根据<font style="background-color:#E8F7CF;">表大小、分桶策略、统计信息</font>自动选择最优算法：

### （1）Broadcast Join（广播 Join）
原理：将<font style="background-color:#E8F7CF;">右表全量广播到左表所在 BE 节点</font>

触发条件：

+ 右表预估大小 < `broadcast_join_threshold`（默认 100MB）
+ 或手动 Hint：`/*+ SET_VAR(broadcast_join_threshold=1073741824) */`

优点：无 Shuffle，适合 大表 Join 小维表

缺点：右表过大 → 内存溢出（OOM）

📌 最佳实践：维度表（如 `user_dim`, `product_dim`）通常 < 1GB，天然适合 Broadcast

### （2）Shuffle Join（重分区 Join）
原理：对<font style="background-color:#E8F7CF;">两表按 Join Key 做 Hash Shuffle，相同 Key 落到同一 BE</font>

触发条件：两表均大于广播阈值

优点：可处理任意大小表

缺点：网络 I/O 高，需额外磁盘 Spill（若内存不足）

⚠️ <font style="background-color:#E8F7CF;">性能瓶颈：Shuffle 阶段常占查询 60%+ 时间</font>

### （3）Colocation Join（协同定位 Join）【Doris 特色】
原理：若两表<font style="background-color:#E8F7CF;"> 分桶键相同 + 分区策略一致，则相同 Bucket 数据已在同一 BE</font>

效果：完全避免 Shuffle，本地 Join

启用条件：

```sql
-- 建表时指定 Colocation Group
CREATE TABLE t1 (...) 
DISTRIBUTED BY HASH(k1) BUCKETS 10
PROPERTIES("colocate_with" = "group1");

CREATE TABLE t2 (...) 
DISTRIBUTED BY HASH(k1) BUCKETS 10
PROPERTIES("colocate_with" = "group1");
```

验证：`EXPLAIN` 中出现 `COLOCATE` 字样

✅ 性能提升：相比 Shuffle Join，快 3~10 倍

### （4）Bucket Shuffle Join（桶级 Shuffle）【Doris 2.0+】
原理：仅 Shuffle 缺失的 Bucket，而非全表

适用：<font style="background-color:#E8F7CF;">Colocation 不满足，但分桶数一致</font>

效果：比普通 Shuffle 减少 50%+ 网络流量

## Join 优化建议
| 场景 | 建议 |
| --- | --- |
| 维表 Join 事实表 | 确保维表 < 100MB，走 Broadcast |
| 大事实表 Join | 设计相同分桶键，启用 Colocation |
| Join Key 倾斜 | 使用 `SKEW JOIN` Hint（实验性）或预处理打散 Key |
| 多表 Join | 按表大小升序排列（小表在左），利于 RBO 优化 |


# 子查询（Subquery）：从语法支持到执行优化
## 支持的子查询类型
| 类型 | 示例 | 支持情况 |
| --- | --- | --- |
| 标量子查询 | `SELECT (SELECT max(id) FROM t2) FROM t1;` | ✅ |
| IN 子查询 | `SELECT * FROM t1 WHERE id IN (SELECT id FROM t2);` | ✅（自动转 Semi Join） |
| EXISTS 子查询 | `SELECT * FROM t1 WHERE EXISTS (SELECT 1 FROM t2 WHERE t2.id = t1.id);` | ✅（转 Anti/Semi Join） |
| 关联子查询 | `SELECT * FROM t1 WHERE id > (SELECT avg(id) FROM t2 WHERE t2.dt = t1.dt);` | ✅（需 Decorrelation） |
| FROM 子查询（派生表） | `SELECT * FROM (SELECT id FROM t1) t;` | ✅（自动下推谓词） |


⚠️ 不支持：`NOT IN` 含 `NULL` 值（语义歧义），建议用 `NOT EXISTS`

## 子查询优化技术
### （1）去关联化（Decorrelation）
+ 将相关子查询 转为 Join + Group By
+ 示例：

```sql
-- 原始
SELECT * FROM orders o 
WHERE o.amount > (SELECT AVG(amount) FROM orders o2 WHERE o2.user_id = o.user_id);

-- 优化后
SELECT o.* 
FROM orders o
JOIN (SELECT user_id, AVG(amount) AS avg_amt FROM orders GROUP BY user_id) tmp
  ON o.user_id = tmp.user_id
WHERE o.amount > tmp.avg_amt;
```

### （2）Semi/Anti Join 转换
+ `IN` → `LEFT SEMI JOIN`
+ `NOT EXISTS` → `LEFT ANTI JOIN`
+ 优势：可利用 Join 优化器（如 Broadcast）

### （3）谓词下推（Predicate Pushdown）
+ 对派生表（FROM 子查询），将外层 WHERE 条件下推

```sql
SELECT * FROM (SELECT id, name FROM users) u WHERE u.id > 100;
-- 下推后：SELECT id, name FROM users WHERE id > 100;
```

# 多维聚合分析：Grouping Sets / Cube / Rollup
Doris 支持标准 SQL 的多维聚合扩展，用于 BI 报表、下钻分析。

## 语法支持
```sql
-- Grouping Sets：指定多个分组组合
SELECT city, category, SUM(sales)
FROM sales
GROUP BY GROUPING SETS ((city), (category), ());

-- CUBE：所有可能组合（2^N）
SELECT city, category, SUM(sales)
FROM sales
GROUP BY CUBE(city, category);

-- ROLLUP：层级聚合（(A,B,C), (A,B), (A), ()）
SELECT year, month, day, SUM(sales)
FROM sales
GROUP BY ROLLUP(year, month, day);
```

## 执行原理
+ 逻辑计划阶段：展开为 UNION ALL 多个 GROUP BY
+ 物理计划阶段：共享 Scan 和部分 Agg 计算（避免重复扫描）
+ 结果标识：使用 `GROUPING()` 函数区分 NULL 是真实值还是汇总标记

```sql
SELECT 
    city,
    category,
    GROUPING(city) AS grp_city,
    SUM(sales)
FROM sales
GROUP BY CUBE(city, category);
```

## 性能考量
| 方式 | 扫描次数 | 适用场景 |
| --- | --- | --- |
| 单次 CUBE | 1 次 | 维度 ≤ 3，结果集可控 |
| 多次独立查询 | N 次 | 维度多、结果大，避免 OOM |


💡<font style="background-color:#E8F7CF;"> 替代方案：对高频 CUBE 查询，预先创建物化视图</font>

# 分析函数（窗口函数，Window Function）
Doris 全面支持 ANSI SQL 窗口函数，用于 排名、移动平均、累计计算 等场景。

## 支持的函数类别
| 类别 | 函数示例 |
| --- | --- |
| 排名函数 | `ROW_NUMBER()`, `RANK()`, `DENSE_RANK()` |
| 聚合函数 | `SUM()`, `AVG()`, `COUNT()`, `MIN()`, `MAX()` |
| 偏移函数 | `LAG()`, `LEAD()`, `FIRST_VALUE()`, `LAST_VALUE()` |
| 分布函数 | `NTILE()`, `PERCENT_RANK()`（部分支持） |


## 窗口定义语法
```sql
function(...) OVER (
    [PARTITION BY partition_expr]
    [ORDER BY order_expr [ASC|DESC]]
    [window_frame]
)
```

窗口帧（Window Frame）类型：

+ `ROWS`：物理行（如 `ROWS BETWEEN 2 PRECEDING AND CURRENT ROW`）
+ `RANGE`：逻辑值范围（如 `RANGE BETWEEN INTERVAL 7 DAY PRECEDING AND CURRENT ROW`）
+ 默认：`RANGE UNBOUNDED PRECEDING`（累计窗口）

## 执行原理与优化
### （1）执行模型
+ 排序阶段：按 `PARTITION BY + ORDER BY` 排序
+ 滑动计算：维护窗口内状态（如 Sum、Count）
+ 向量化支持：Doris 2.0+ 对常见窗口函数（如 SUM、ROW_NUMBER）实现向量化

### （2）性能关键点
| 问题 | 优化方案 |
| --- | --- |
| 大 Partition 排序慢 | 确保 `PARTITION BY`列高基数，避免单 Partition 过大 |
| 无界窗口（UNBOUNDED） | 内存占用高，慎用于大表 |
| RANGE 窗口含时间 | 使用 `DATETIME`列，避免字符串转换 |


### （3）典型应用场景
```sql
-- 用户行为序列：上一次访问时间
SELECT 
    user_id,
    event_time,
    LAG(event_time, 1) OVER (PARTITION BY user_id ORDER BY event_time) AS last_time
FROM user_log;

-- 销售排行榜
SELECT 
    product_id,
    sales,
    RANK() OVER (ORDER BY sales DESC) AS rk
FROM daily_sales;

-- 7 日移动平均
SELECT 
    dt,
    revenue,
    AVG(revenue) OVER (
        ORDER BY dt 
        RANGE BETWEEN INTERVAL 6 DAY PRECEDING AND CURRENT ROW
    ) AS ma7
FROM sales_daily;
```

# 高级功能对比与选型指南
| 功能 | 适用场景 | 性能提示 |
| --- | --- | --- |
| Broadcast Join | 大表 + 小维表 | 控制维表大小 < 100MB |
| Colocation Join | 同分桶事实表关联 | 建表时规划分桶键 |
| 物化视图 | 固定聚合/Join | 替代运行时 CUBE/Join |
| 窗口函数 | 序列分析、排名 | 避免无界窗口 |
| Grouping Sets | 多维 BI 报表 | 维度不宜过多（≤3） |

# 总结
Apache Doris 的高级 SQL 能力已达到 企业级 OLAP 数据库水平：

+ Join：通过 Broadcast / Colocation / Shuffle 三剑客，覆盖 99% 场景；
+ 子查询：自动优化为高效 Join，降低用户心智负担；
+ 多维聚合：标准 SQL 支持，无缝对接 BI 工具；
+ 窗口函数：完整支持分析场景，向量化加速关键路径。

建模与查询建议：

🔸 优先用物化视图预计算 复杂 Join/聚合；

🔸 窗口函数避免无界帧；  
🔸 多维分析控制维度爆炸。
