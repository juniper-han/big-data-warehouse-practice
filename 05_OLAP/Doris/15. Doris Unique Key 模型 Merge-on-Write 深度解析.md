## MOW 概念
### 什么是 Merge-on-Write？
Merge-on-Write (MOW) 是 Doris 2.0+ 版本为 Unique Key 模型引入的重大优化，彻底改变了数据更新和查询的处理方式。

核心思想：

+ 写入时合并：在数据写入时（而非查询时）完成版本合并
+ 实时可见：更新后的数据立即可见，无需查询时合并
+ 消除读放大：查询时直接读取最终数据，无需合并多个版本

### 对比：MOR vs MOW
| 特性 | Merge-on-Read (MOR) 传统方式 | Merge-on-Write (MOW) 新方式 |
| --- | --- | --- |
| 写入方式 | 写入增量数据，保留多个版本 | 写入时完成合并，只保留最新版本 |
| 查询性能 | 需要合并多个版本，性能差 | 直接读取最终数据，性能接近 Duplicate Key |
| 写入性能 | 快（只追加） | 稍慢（需要合并） |
| 存储空间 | 大（保留多个版本） | 小（只保留最新数据） |
| 版本数 | 容易膨胀 | 稳定可控 |
| 适用场景 | 写入密集，查询较少 | 查询密集，需要实时更新 |


## MOW 存储结构
### 文件组织
```plain
Tablet (MOW)
├── Rowset 1 (Base)
│   ├── segment_1.dat
│   └── segment_1.idx
├── Rowset 2 (Delta - 更新)
│   ├── segment_2.dat
│   ├── segment_2.idx
│   └── _delete_bitmap (删除位图)
└── Rowset 3 (Delta - 更新)
    ├── segment_3.dat
    ├── segment_3.idx
    └── _delete_bitmap
```

关键组件：

1. 数据文件：存储实际数据
2. 索引文件：主键索引
3. 删除位图：标记被覆盖的数据版本（MOW 核心）

### Delete Bitmap 机制
作用：记录哪些旧版本的数据被新数据覆盖

```plain
版本 1: key=1, value=100
版本 2: key=1, value=200

Delete Bitmap 记录：
- key=1 在版本 1 中被版本 2 覆盖
- 查询时跳过版本 1 的 key=1
```

数据结构：

+ 基于 Roaring Bitmap 实现，高效存储
+ 每个 Rowset 维护自己的 Delete Bitmap
+ Compaction 时清理被标记的数据

## 写入流程详解
### 数据写入完整流程
```sql
-- 示例表
CREATE TABLE user_profile (
    user_id BIGINT,
    name STRING,
    age INT,
    city STRING,
    last_active TIMESTAMP
) UNIQUE KEY(user_id)
DISTRIBUTED BY HASH(user_id) BUCKETS 10
PROPERTIES (
    "enable_unique_key_merge_on_write" = "true",
    "compression" = "LZ4"
);
```

写入步骤：

1. 写入请求到达
2. 检查主键是否存在
3. 如果存在 → 标记旧数据为删除（Delete Bitmap）
4. 写入新数据到 MemTable
5. MemTable 刷盘 → 生成新的 Segment
6. 更新 Delete Bitmap 索引
7. 查询时通过 Delete Bitmap 过滤旧数据

#### INSERT 操作
`INSERT INTO user_profile VALUES (1, 'Alice', 25, 'Beijing', '2024-01-01');`

内部流程：

1. 检查 key=1 是否存在 → 不存在
2. 直接写入新数据
3. 更新主键索引

#### UPDATE 操作
`UPDATE user_profile SET age=26 WHERE user_id=1;`

内部流程：

1. 查询 key=1 当前所在的 Rowset
2. 在 Delete Bitmap 中标记旧数据为删除
3. 写入新的数据（age=26）
4. 更新主键索引指向新数据

Delete Bitmap 变化：

```plain
写入前：
Rowset 1: [key=1, age=25]

写入后：
Rowset 1: [key=1, age=25] + Delete Bitmap: 标记删除
Rowset 2: [key=1, age=26]
```

#### DELETE 操作
`DELETE FROM user_profile WHERE user_id=1;`

内部流程：

1. 查询 key=1 所在位置
2. 在 Delete Bitmap 中标记该数据为删除
3. 不写入新数据
4. 更新主键索引（标记已删除）

Delete Bitmap 变化：

```plain
Rowset 1: [key=1, age=25] + Delete Bitmap: 标记删除
Rowset 2: [key=1, age=26] + Delete Bitmap: 标记删除
```

### 批量写入优化
MOW 对批量写入做了特殊优化：内部优化：

1. 批量检查主键
2. 批量生成 Delete Bitmap
3. 批量写入数据
4. 一次性更新索引

性能特征：

+ 批量更新比单条更新快 10-100 倍
+ 建议单批次 1000-10000 条

```sql
-- 批量更新（Stream Load / Broker Load）
-- 数据格式：
user_id,name,age,city
1,Alice,26,Beijing
2,Bob,30,Shanghai
3,Charlie,28,Guangzhou
```

## 查询机制详解
### 查询时如何过滤数据
核心原理：通过 Delete Bitmap 快速过滤被覆盖的数据

```sql
-- 查询示例
SELECT * FROM user_profile WHERE user_id=1;
```

查询流程：

1. 定位 Tablet 和 Rowset 列表
2. 查询主键索引，找到 key=1 的所有位置
3. 读取 Delete Bitmap
4. 过滤掉被标记删除的数据
5. 返回最新的有效数据

示例场景：

```plain
数据分布：
Rowset 1: user_id=1, age=25 (version 1)
Rowset 2: user_id=1, age=26 (version 2) - 最新
Rowset 2 Delete Bitmap: 标记 Rowset 1 的 user_id=1 为删除

查询处理：
1. 读取所有 Rowset 中 user_id=1 的数据
2. 应用 Delete Bitmap 过滤
3. 只返回 Rowset 2 的数据（age=26）
```

### 查询性能优势
MOR vs MOW 查询对比：

```sql
-- 场景：表有 1000 万行，更新了 10 万次
-- 查询：SELECT COUNT(*) FROM table WHERE date='2024-01-01'
```

| 指标 | MOR | MOW |
| --- | --- | --- |
| 需要合并的版本数 | 1000+ | 1 |
| 扫描数据量 | 10GB + 元数据 | 1GB |
| 查询耗时 | 30-60 秒 | 1-3 秒 |
| CPU 使用率 | 高（合并开销） | 低（只读取） |


### 谓词下推优化
MOW 支持完整的谓词下推：

```sql
-- 查询条件可以直接下推到存储层
SELECT * FROM user_profile 
WHERE age BETWEEN 20 AND 30 
  AND city = 'Beijing';

-- 存储层优化：
-- 1. 通过 Zone Map 跳过不符合条件的 Segment
-- 2. 通过 Bloom Filter 快速过滤
-- 3. 只读取有效数据（Delete Bitmap 已过滤）
```

## Compaction 在 MOW 中的特殊处理
### MOW Compaction 流程
MOW 的 Compaction 与普通表不同，需要处理 Delete Bitmap：

```plain
Compaction 前：
Rowset 1: 1000 行有效数据 + 500 行删除标记
Rowset 2: 800 行有效数据 + 300 行删除标记
Rowset 3: 1200 行有效数据 + 0 行删除标记

Compaction 过程：
1. 读取所有 Rowset 的有效数据
2. 跳过 Delete Bitmap 标记的数据
3. 合并有效数据到新 Segment
4. 生成新的 Delete Bitmap（清空旧标记）
5. 删除旧 Rowset

Compaction 后：
New Rowset: 3000 行有效数据 + 0 行删除标记
```

### MOW 特定配置
```bash
# be.conf

# MOW Compaction 线程数（建议单独配置）
mow_compaction_threads = 8

# 是否启用 MOW 增量 Compaction
enable_mow_compaction = true

# MOW Compaction 最小触发行数
mow_compaction_min_rows = 100000

# MOW Compaction 间隔（秒）
mow_compaction_interval_seconds = 3600
```

```sql
-- 表级别配置
ALTER TABLE user_profile SET (
    -- MOW 特有：是否启用增量 Compaction
    "enable_mow_compaction" = "true",
    
    -- 合并阈值
    "mow_compaction_threshold" = "3"
);
```

### 场景二：批量导入 + 增量更新
特点：

+ 每天一次全量导入（几亿行）
+ 全天持续小批量更新（几千次）

优化策略：

```sql
-- 1. 使用分区表，按天分区
CREATE TABLE order_mow (
    order_id BIGINT,
    user_id BIGINT,

PROPERTIES (
    "enable_unique_key_merge_on_write" = "true",
    "dynamic_partition.enable" = "true",
    "dynamic_partition.time_unit" = "DAY",
    "dynamic_partition.start" = "-7",

-- 2. 全量导入时使用临时分区
-- 创建临时分区

-- 导入完成后替换
ALTER TABLE order_mow REPLACE PARTITION (p20240301)

```

### 场景三：高并发点查
特点：

+ 大量点查询（SELECT * FROM table WHERE key=?）

## 监控与诊断
### MOW 特定监控指标
```sql
-- 1. 查看 MOW 表的状态
SHOW TABLET FROM user_profile_mow;

-- 重点关注：
-- DeleteBitmapSize：删除位图大小（过大说明需要 Compaction）
-- RowsetCount：Rowset 数量
-- VersionCount：版本数

-- 2. 查看 MOW Compaction 状态
SHOW PROC '/compactions';
-- 查看 mow_compaction 相关字段
```

### 关键指标告警
| 指标 | 正常范围 | 告警阈值 | 说明 |
| --- | --- | --- | --- |
| DeleteBitmapSize | < 100MB | > 1GB | 删除位图过大影响性能 |
| VersionCount | < 50 | > 100 | 版本数过多 |
| MOWCompactionQueue | 0 | > 10 | MOW Compaction 积压 |
| UpdateRate | < 1000/s | > 5000/s | 更新频率过高 |


### 问题诊断 SQL
```sql
-- 1. 找出 Delete Bitmap 过大的表
SELECT 
    t.table_name,
    SUM(t.delete_bitmap_size) / 1024 / 1024 AS total_delete_bitmap_mb
FROM information_schema.tablets t
WHERE t.enable_mow = true
GROUP BY t.table_name
HAVING total_delete_bitmap_mb > 1024  -- 超过 1GB
ORDER BY total_delete_bitmap_mb DESC;

-- 2. 查看更新频率最高的表
SELECT 
    table_name,
    update_count,
    delete_count
FROM information_schema.table_statistics
WHERE enable_mow = true
ORDER BY update_count DESC
LIMIT 10;

-- 3. 检查 Compaction 效率
SHOW PROC '/compactions';
-- 计算 Compaction 平均耗时
-- 如果持续 > 30 分钟，需要优化
```

## 常见问题与解决方案
### Delete Bitmap 膨胀
问题表现：

+ 存储空间持续增长
+ 查询变慢
+ Compaction 耗时过长

原因：

+ 高频更新导致大量删除标记
+ Compaction 不及时

解决方案：

```sql
-- 1. 手动触发 Full Compaction
ALTER TABLE user_profile_mow COMPACT;

-- 2. 调整 Compaction 策略
ALTER TABLE user_profile_mow SET (
    "enable_mow_compaction" = "true",
    "mow_compaction_threshold" = "2"  -- 降低触发阈值
);
```

### 写入性能下降
问题表现：

+ 导入速度变慢
+ 延迟增加

原因分析：

+ 主键索引过大
+ 分桶不合理

解决方案：

```sql
-- 1. 增加分桶数，分散写入压力
ALTER TABLE user_profile_mow 
DISTRIBUTED BY HASH(user_id) BUCKETS 32;

-- 2. 调整写入批次大小
-- 推荐：单批次 1000-5000 行
-- 避免：单条更新或超大批次（> 10 万行）

-- 3. 使用 SSD 存储 Delete Bitmap
-- 在 be.conf 中配置
delete_bitmap_storage_path = /ssd_path/delete_bitmap
```

### 查询偶尔超时
问题表现：

+ 大部分查询正常
+ 偶尔出现慢查询或超时

原因：

+ Compaction 执行时资源争抢
+ 热点数据更新频繁

解决方案：

```sql
-- 1. 调整 Compaction 执行时间
-- 将 Compaction 调度到低峰期

-- 2. 使用查询限流
SET SESSION query_timeout = 30;  -- 30 秒超时

-- 3. 增加查询缓存
ALTER TABLE user_profile_mow SET (
    "enable_tablet_cache" = "true",
    "cache_ttl_seconds" = "300"
);
```

## 总结
### 适用场景判断
MOW 适合：

+ ✅ 查询密集场景（读多写少）
+ ✅ 需要强一致性（更新后立即可见）
+ ✅ 点查询和范围查询都很重要
+ ✅ 更新频率适中（< 10,000/秒）

MOW 不适合：

+ ❌ 纯追加场景（使用 Duplicate Key）
+ ❌ 超高频率更新（> 50,000/秒）
+ ❌ 不需要更新和删除的场景

### 配置 Checklist
```sql
-- 建表必选项
PROPERTIES (

    "bloom_filter_columns" = "user_id",            -- 主键 Bloom Filter
    "light_schema_change" = "true"                 -- 开启轻量 schema 变更

-- 可选优化
PROPERTIES (

```

### 日常运维建议
1. 监控关键指标：
    - Delete Bitmap 大小（< 1GB）
    - 版本数（< 50）
    - 更新 QPS（< 10,000）
2. 定期维护：

```sql
-- 每周低峰期执行
ALTER TABLE user_profile_mow COMPACT;
```

3. 容量规划：
    - MOW 存储空间 ≈ 原始数据 × (1 + 更新频率 × 保留时间)
    - 建议预留 50% 额外空间
4. 版本升级注意：
    - 从 MOR 迁移到 MOW 需要重建表
    - 建议在测试环境验证后再升级
