# 湖仓一体背景与 Doris 的战略定位
## 湖仓一体的核心诉求
传统架构痛点：

+ 数据孤岛：OLAP 系统（如 Doris）与数据湖（Hudi/Iceberg）分离；
+ ETL 复杂：需频繁导出导入，延迟高、成本高；
+ 语义不一致：同一份数据在不同系统中 schema 不统一。

湖仓一体目标：

✅ 一份存储，多引擎访问  
✅ 实时 + 批处理统一  
✅ ACID 事务跨系统一致

## Doris 的角色演进
+ 过去：独立 OLAP 引擎，依赖 Broker Load 导入湖数据；
+ 现在（Doris 2.0+）：湖仓联邦查询引擎，通过 Multi-Catalog 直读湖表；
+ 未来：Lakehouse 查询加速层，支持写入湖表（CDC 同步、物化视图回写）。

📌 核心价值：无需数据搬迁，直接分析 Hudi/Hive 表，实现“湖上建仓”

## 整体架构

+ FE（Frontend）：负责 Catalog 元数据管理、查询解析、计划生成
+ BE（Backend）：负责 实际读取外部数据文件（Parquet/ORC）
+ Catalog：逻辑命名空间，隔离不同数据源

## 数据目录（Catalog）
数据目录：用于描述一个数据源的属性

+ 在 Doris 中，可以创建多个数据目录指向不同的数据源（如 Hive、Iceberg、MySQL）
+ Doris 会通过数据目录，自动获取对应数据源的库、表、Schema、分区、数据位置等。
+ 用户可以通过标准的 SQL 语句访问这些数据目录进行数据分析，并且可以对多个数据目录中的数据进行关联查询

## 支持的 Catalog 类型（截至 Doris 2.2）
| Catalog 类型 | 数据源 | 文件格式 | 事务支持 |
| --- | --- | --- | --- |
| Hive Catalog | Hive Metastore | Parquet, ORC, Text | 快照读（无 ACID） |
| Hudi Catalog | Hudi 表路径 | Parquet (Copy-On-Write) | MOR/COO 快照读 |
| Iceberg Catalog | Iceberg REST/HMS | Parquet | Snapshot Isolation |
| JDBC Catalog | MySQL/PostgreSQL | - | 实时读 |
| Es Catalog | Elasticsearch | - | 近实时 |


✅ Hudi 是 Doris 湖仓集成的重点方向（社区深度合作）

# 以 Hive Catalog 为例：连接与查询
## 创建 Hive Catalog
```sql
CREATE CATALOG hive_catalog
PROPERTIES (
    'type' = 'hms',
    'hive.metastore.uris' = 'thrift://hive-metastore:9083',
    'hive.version' = '3.1.2'
);
```

+ 元数据同步：FE 启动 HiveMetaStoreClient，按需拉取表 schema、分区信息；
+ 缓存机制：元数据缓存在 FE 内存，可通过 `REFRESH CATALOG` 手动刷新。

## 查询 Hive 表
```sql
-- 切换 Catalog
USE CATALOG hive_catalog;

-- 直接查询
SELECT user_id, SUM(clicks) 
FROM default.user_log 
WHERE dt = '2025-03-04'
GROUP BY user_id;
```

## 底层执行流程
1. FE 解析：识别 `hive_catalog.default.user_log`；
2. 元数据获取：从 Hive Metastore 获取：
    - 表结构（列名、类型）
    - 分区列表（如 `dt=2025-03-04`）
    - 文件位置（HDFS/S3 路径）
3. 计划生成：
    - 分区裁剪：仅扫描 `dt=2025-03-04` 分区
    - 列裁剪：只读取 `user_id`, `clicks`
4. BE 执行：
    - 通过 HDFS Client 读取 Parquet 文件
    - 向量化解析 Parquet 列存
    - 返回结果给 FE

⚠️ <font style="background-color:#E8F7CF;">限制：Hive 表必须是 Parquet/ORC 格式（TextFile 性能极差）</font>

# 以 Hudi Catalog 为例：实时湖表直读
Hudi 是 Doris 湖仓集成的 明星场景，因其支持 近实时更新。

## Hudi 表类型支持
| Hudi 表类型 | Doris 支持 | 说明 |
| --- | --- | --- |
| Copy-On-Write (COW) | ✅ 完全支持 | 读优化，每次更新重写文件 |
| Merge-On-Read (MOR) | ✅（Doris 2.1+） | 读时合并 log + base 文件 |


✅ Doris 可读取 Hudi 的任意时间点快照（Time Travel）

## 创建 Hudi Catalog
```sql
CREATE CATALOG hudi_catalog
PROPERTIES (
    'type' = 'hudi',
    'warehouse' = 'hdfs://namenode:8020/path/to/hudi'
);
```

+ 自动发现：FE 扫描 `warehouse` 下所有 Hudi 表（识别 `.hoodie` 目录）

## 查询 Hudi 表（含 Time Travel）
```sql
USE CATALOG hudi_catalog;

-- 查询最新快照
SELECT * FROM orders;

-- 查询指定时间点（精确到秒）
SELECT * FROM orders /*+ OPTIONS('hoodie.query.as.of.timestamp' = '20250304000000') */;
```

## 底层实现原理（核心技术突破）
### （1）Hudi 元数据解析
+ FE 读取 `.hoodie` 目录下的：
    - `commits` / `deltacommits`：事务提交记录
    - `schema`：Avro Schema
    - `partition metadata`：分区结构

### （2）MOR 表读取（关键难点）
+ BE 同时读取：
    - Base File（Parquet）
    - Log Files（Avro，含更新/删除）
+ 向量化 Merge：
    - 按 record key 合并 log 与 base
    - 应用 preCombineField（如 `ts` 字段）去重
+ 性能优化：
    - Log 文件索引（BloomFilter）加速 record lookup
    - 异步预读 log 文件

📊 性能实测：10 亿行 Hudi MOR 表，Doris 查询 P99 < 2s（同等资源下比 Spark 快 3x）

# 查询下推（Predicate Pushdown）与性能优化
Doris 对外部表实现 深度谓词下推，减少数据扫描量。

## 下推能力矩阵
| 下推类型 | Hive | Hudi COW | Hudi MOR |
| --- | --- | --- | --- |
| 分区裁剪 | ✅ | ✅ | ✅ |
| 列裁剪 | ✅ | ✅ | ✅ |
| Parquet 谓词下推 | ✅（Min/Max） | ✅ | ✅（Base File） |
| Hudi Record Key 下推 | ❌ | ✅ | ✅（通过 BloomFilter） |
| Time Travel | ❌ | ✅ | ✅ |


## 性能调优建议
| 方向 | 建议 |
| --- | --- |
| 文件大小 | Hudi/Hive 小文件合并（<128MB → OOM 风险） |
| 列式格式 | 强制使用 Parquet（ORC 支持有限） |
| 统计信息 | 在 Hive/Hudi 中收集列统计（提升 CBO 效果） |
| BE 资源 | 增大 `external_sort_enable`，避免 Spill |
| 缓存 | 开启 HDFS 缓存（Alluxio）或对象存储 CDN |


# 事务一致性与数据可见性
## 一致性模型
+ Hive：最终一致（依赖 Hive Metastore commit）
+ Hudi：快照隔离（Snapshot Isolation）
    - Doris 读取 Hudi 的 committed timeline
    - 不会读到未提交或中间状态数据

## Time Travel 语义
+ Hudi：通过 `hoodie.query.as.of.timestamp` 指定 精确到秒的时间点
+ 实现：FE 查找 ≤ 该时间的最大 commit instant

✅ 效果：可实现 “回滚查询” 或 “历史对比分析”

# 限制与注意事项
| 限制 | 说明 |
| --- | --- |
| 不支持 Hudi 索引表 | 如 BloomIndex、HBaseIndex |
| MOR 性能依赖 log 大小 | 频繁小更新导致 log 膨胀 |
| 无 ACID 写入 | 当前仅支持读，写入需通过 Flink/Spark |
| Schema Evolution | Hudi 新增列需手动 `REFRESH CATALOG` |
