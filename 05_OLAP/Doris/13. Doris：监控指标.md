FE（Frontend）、BE（Backend）、查询性能、集群健康、资源消耗五大维度，系统梳理 Doris 生产环境常用监控指标

# 生产环境监控 Checklist（告警项）
| 类别 | 告警项 | 阈值 | 严重性 |
| --- | --- | --- | --- |
| 健康 | BE 失联 | >0 | 🔴 紧急 |
| 数据 | 不一致 Tablet | >0 | 🔴 紧急 |
| 资源 | BE 内存使用率 | >90% | 🟠 严重 |
| 性能 | 慢查询数/分钟 | >10 | 🟠 严重 |
| 导入 | 导入失败率 | >1% | 🟠 严重 |
| 容量 | 磁盘使用率 | >85% | 🟡 警告 |


# FE（Frontend）监控指标
FE 负责元数据管理、查询解析、调度，是集群的“大脑”。

## **JVM 相关**
| 指标 | 说明 | 告警建议 |
| --- | --- | --- |
| `jvm.heap.used.percent` | JVM 堆内存使用率 | >85% 持续 5 分钟 → 告警 |
| `jvm.gc.time.ms` | GC 耗时（尤其是 Full GC） | 单次 Full GC > 5s 或频繁 Minor GC → 排查内存泄漏 |
| `jvm.thread.count` | 线程数 | 异常飙升可能死锁 |


💡 FE 默认堆内存 8~16GB，若元数据量大（>10万表/分区），需调优。

## **元数据与状态**
| 指标 | 说明 |
| --- | --- |
| `fe.is_master` | 是否为 Master FE（高可用依赖） |
| `catalog.trash.expired.partition.count` | 回收站中待删除分区数（反映 DROP 操作积压） |
| `edit_log.roll.interval.ms` | Edit Log 滚动间隔（影响元数据持久化延迟） |


## **连接与请求**
| 指标 | 说明 |
| --- | --- |
| `qps` / `query.total.count` | 查询 QPS |
| `connection.total.count` | 当前连接数 |
| `http.request.latency.ms` | HTTP 请求（如 Stream Load）延迟 |


⚠️ 连接数突增可能被恶意扫描或连接池泄漏。

# BE（Backend）监控指标
BE 负责数据存储、计算，是性能瓶颈主要发生地。

## 系统资源
| 指标 | 说明 | 告警建议 |
| --- | --- | --- |
| `cpu.utilization.percent` | CPU 使用率 | >90% 持续 → 可能查询复杂或并发过高 |
| `memory.used.percent` | 内存使用率（非 JVM） | >85% → 可能 OOM kill |
| `disk_used_percent` | 磁盘使用率 | >85% → 触发 Tablet 均衡；>90% → 写入拒绝 |
| `io.util.percent` | 磁盘 IO 利用率 | >90% → 存储瓶颈 |


## Tablet 与副本健康
| 指标 | 说明 | 重要性 |
| --- | --- | --- |
| `tablet.total.count` | Tablet 总数 | 反映数据分片规模 |
| `tablet.inconsistent.count` | 副本不一致数，必须为 0！ | 非 0 表示数据丢失风险 |
| `tablet.clone.job.running` | 正在进行的副本修复任务数 | 持续 >0 可能磁盘故障 |
| `tablet.version.count` | 版本数量（反映 Compaction 压力） | 过高 → 查询变慢 |


🔥 核心告警项：`inconsistent_tablet_num > 0` → 立即处理！

## Compaction（合并）指标
Doris 通过 Compaction 合并小版本，提升查询性能。

| 指标 | 说明 |
| --- | --- |
| `compaction.pending.tasks` | 待执行 Compaction 任务数 |
| `compaction.used.time.avg.ms` | 平均 Compaction 耗时 |
| `base_compaction.pending.tasks` | Base Compaction（大合并）积压 |


⚠️ Compaction 积压会导致：

+ 查询需 merge 多个版本 → 性能下降；
+ 磁盘空间无法释放。

# 查询性能监控（核心！）
## 查询延迟与吞吐
| 指标 | 说明 | 告警建议 |
| --- | --- | --- |
| `query.duration.avg.ms` | 平均查询耗时 | 对比基线突增 2x → 告警 |
| `query.scan.rows.per.sec` | 扫描行速率 | 下降可能 I/O 或 CPU 瓶颈 |
| `query.memory.cost.avg.bytes` | 单查询平均内存 | 异常高可能未走索引或笛卡尔积 |


## 慢查询识别
+ 开启慢查询日志（`enable_profile` + `log_slow_query`）；
+ 监控：`slow_query_count`（单位时间慢查询数）；
+ 慢查询定义：`query_time > 5s`（按业务调整）。

🛠️ 工具：通过 `SHOW PROC '/current_queries';` 实时查看运行中查询。

## Query Profile 关键指标
对单条查询，关注：

+ `ScanTime`：数据扫描时间；
+ `JoinTime` / `AggTime`：计算耗时；
+ `ReturnRows` vs `ScanRows`：过滤率 = ReturnRows / ScanRows，越低越好（理想 < 1%）。

# 数据导入（Load）监控
Doris 支持多种导入方式（Stream Load、Broker Load、Routine Load），需监控：

| 指标 | 说明 |
| --- | --- |
| `load.job.success.rate` | 导入成功率（应 >99.9%） |
| `load.job.pending.count` | 待调度导入任务数 |
| `stream_load.push.duration.avg.ms` | Stream Load 推送延迟 |
| `routine_load.error.rows` | Routine Load 错误行数（数据格式问题） |


⚠️ 导入失败常见原因：

1. 数据格式错误（JSON/CSV 解析失败）；
2. 内存不足（`mem_limit` 超限）；
3. Tablet 副本不可用。

# 集群健康与高可用
## 节点状态
| 指标 | 说明 |
| --- | --- |
| `fe.alive.num` / `be.alive.num` | 存活 FE/BE 数量 |
| `be.last_heartbeat` | BE 最后心跳时间 |


## 副本均衡
| 指标 | 说明 |
| --- | --- |
| `tablet.balance.job.running` | 均衡任务是否运行 |
| `be.disk.path.used.percent.stddev` | 各磁盘路径使用率标准差 |


## 高可用（HA）
+ FE 必须部署 奇数个（3/5），确保 Master 选举；
+ 监控：`fe.role`（应有 1 Master + N Follower/Observer）。

# 如何采集这些指标？
## 方式 1：Doris 自带 HTTP 接口（推荐）
+ FE 指标：`http://fe_host:8030/metrics`
+ BE 指标：`http://be_host:8040/metrics`
+ 返回 Prometheus 格式，可直接对接 Prometheus + Grafana

## 方式 2：Doris Manager（商业版）或社区工具
+ 如 [Doris Manager](https://github.com/apache/doris-manager)（官方运维平台）
+ 或自研 Agent 采集 `/proc` 和 Doris 日志

## 方式 3：日志监控
关键日志关键词：

1. `FATAL`、`ERROR`（FE/BE 日志）
2. `tablet inconsistent`
3. `memory limit exceeded`
4. `load failed`

# 总结：Doris 监控的核心目标
保障“三高”：高可用、高性能、高可靠

+ 高可用：通过 FE/BE 健康监控 + 副本一致性保障；
+ 高性能：通过查询延迟、Compaction、资源使用率优化；
+ 高可靠：通过导入成功率、数据一致性、慢查询治理。

✅ 最佳实践：

1. 必开：慢查询日志 + Prometheus 指标暴露；
2. 必监：`inconsistent_tablet_num`、`be_alive`、`disk_used_percent`；
3. 必优：定期分析 Query Profile，优化大查询。
