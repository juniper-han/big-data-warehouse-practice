内容基于 Doris 2.x 最新架构

Apache Doris 查询引擎是一个 高度优化、工程卓越 的 OLAP 执行系统：

+ 前端：RBO/CBO 混合优化器 + 智能改写
+ 执行：MPP + Pipeline + 向量化 三级并行
+ 存储协同：谓词下推、列裁剪、延迟物化

其设计哲学是：让简单查询极快，让复杂查询可行。

# 整体架构：查询引擎在 Doris 中的位置
Doris 采用 FE（Frontend） + BE（Backend）分离架构：

+ FE：负责 SQL 解析、优化、计划生成
+ BE：负责 物理计划执行、数据扫描、计算

查询流程如下：

Client SQL → <font style="background-color:#E8F7CF;">FE: Parser → FE: Rewriter → FE: Optimizer - RBO/CBO → FE: Plan Fragmenter</font> → 

<font style="background-color:#E6DCF9;">BE: Execute Fragments in Parallel </font>→ Result to Client

✅ 关键思想：计算下推 + 数据本地化 + 向量化执行

# 查询处理全流程详解
## 语法解析（Parsing）
输入：SQL 字符串（兼容 MySQL 协议）

工具：基于 ANTLR4 构建的自定义语法解析器

输出：抽象语法树（AST），如 `SelectStmt`、`InsertStmt` 等



示例：`SELECT user_id, SUM(clicks) FROM logs WHERE dt = '2025-03-04' GROUP BY user_id;`

→ 生成 `SelectStmt` 对象，包含 `SelectList`, `FromClause`, `WherePredicate`, `GroupByClause`



特点：

+ 支持 Doris 特有语法（如 `ROUTINE LOAD`, `ROLLUP`）
+ 严格校验关键字、函数、表名合法性

## 语义分析（Analysis）
职责：将 AST 转换为 逻辑执行计划（Logical Plan）

核心任务：

+ Catalog 解析：查元数据（表是否存在、列类型、分区信息）
+ 表达式绑定：将 `user_id` 绑定到具体表的列
+ 类型推导：确定 `SUM(clicks)` 返回 `BIGINT`
+ 权限校验：检查用户是否有 `SELECT` 权限
+ <font style="background-color:#E8F7CF;">分区裁剪（Partition Pruning）</font>：根据 `dt = '2025-03-04'` 过滤无关分区
+ <font style="background-color:#E8F7CF;">桶裁剪（Bucket Pruning）</font>：若 `user_id` 是分桶键，可定位具体 Tablet

输出：Analyzed Logical Plan（带元数据信息的树）

📌 此阶段已消除 90% 的无效 I/O（通过分区/桶裁剪）

## 查询改写（Query Rewriting）
目的：等价变换，提升后续优化空间

典型规则（由 `RuleExecutor` 驱动）：

| 改写规则 | 说明 |
| --- | --- |
| 谓词下推（Predicate Pushdown） | 将 `WHERE` 条件尽可能下推到 Scan 节点 |
| Project 下推 | 只读取查询涉及的列（列裁剪） |
| 子查询去关联（Decorrelation） | 将相关子查询转为 Join |
| 物化视图匹配 | 自动重写查询到预聚合物化视图 |
| Aggregate 合并 | 合并连续的 GROUP BY |


实现：基于 Volcano/Cascades 风格的规则引擎，支持自定义 Rule



示例：`SELECT user_id FROM (SELECT user_id, clicks FROM logs) t WHERE clicks > 10;`

→ 改写为：`SELECT user_id FROM logs WHERE clicks > 10;`

## 查询优化器：RBO 与 CBO 协同工作
Doris 采用 混合优化器（Hybrid Optimizer）：RBO（Rule-Based）为主，CBO（Cost-Based）为辅

### RBO（Rule-Based Optimization）
触发时机：所有查询必经 RBO

优势：确定性高、开销低、适合 OLAP 场景

核心规则：

+ Join Reordering：基于 Left-Deep Tree，按表大小排序（小表在左，用于 Broadcast Join）
+ Join 算法选择：
    - Broadcast Join：右表 < 100MB（默认阈值）
    - Shuffle Join：大表 Join
    - Colocation Join：同分桶表，避免 Shuffle
+ Aggregate 下推：提前聚合减少数据量
+ TopN 下推：限制中间结果行数

### CBO（Cost-Based Optimization）【Doris 2.0+ 强化】
启用条件：需开启 `enable_cost_based_join_reorder = true`

核心组件：

+ 统计信息收集：`ANALYZE TABLE logs;`收集：行数、列 NDV、Min/Max、直方图（实验性）
+ 代价模型：
        * CPU Cost：函数计算复杂度
        * I/O Cost：预计扫描行数 × 列数
        * Network Cost：Shuffle 数据量
    - 动态规划（DP）：枚举 Join Order，选择最小代价

当前局限：

+ 统计信息需手动收集（自动收集在规划中）
+ 直方图支持有限，对倾斜数据估计不准

✅ 最佳实践：简单查询用 RBO，<font style="background-color:#E8F7CF;">复杂多表 Join 开启 CBO</font>

# MPP 执行引擎：分布式物理计划生成
## Plan Fragmentation（计划分片）
FE 将优化后的逻辑计划切分为多个 PlanFragment，<font style="background-color:#E8F7CF;">每个 Fragment 可独立在 BE 上执行</font>。

+ ExchangeNode：表示数据 Shuffle 点
+ Fragment 树：根 Fragment 负责汇总结果

示例（两表 Join）：

```plain
Fragment 0 (Root):
  └─ MERGE SORT
Fragment 1:
  └─ AGGREGATE
     └─ EXCHANGE (from Fragment 2 & 3)
Fragment 2:
  └─ SCAN table A → SHUFFLE by join key
Fragment 3:
  └─ SCAN table B → SHUFFLE by join key
```

## 并行调度
+ FE 通过 Thrift + BRPC 将 Fragment 发送给 BE
+ 每个 Fragment 实例绑定到 特定 Tablet 所在 BE（数据本地化）
+ BE 启动 Pipeline Engine（Doris 2.0+）执行任务

# 多机多核并行执行模型
Doris 执行引擎支持 三级并行：

| 并行层级 | 说明 | 实现 |
| --- | --- | --- |
| 节点间并行（Inter-node） | 多 BE 并行处理不同 Tablet | MPP Shuffle / Colocation |
| 节点内并行（Intra-node） | 单 BE 多线程处理一个 Fragment | Pipeline 并行 |
| 向量化并行（SIMD） | 单指令多数据（CPU 指令级） | Vectorized Execution |




**Pipeline 执行引擎（Doris 2.0 重大升级）**

传统 Volcano 模型问题：Pull 模式，函数调用开销大

Pipeline 模型：

+ 将算子拆分为 Operator Chain
+ 数据以 Batch（Chunk） 形式在 Chain 中流动
+ 支持 异步 I/O + 计算重叠

✅ 效果：CPU 利用率提升 30%~50%，尤其适合高并发点查

# 向量化执行引擎（Vectorized Execution）
这是 Doris 高性能的核心支柱。

## 基本原理
+ 传统行式处理：

```cpp
for (row : rows) {
    if (row.a > 10) output(row);
}
// 每行一次分支预测、函数调用
```

+ 向量化处理：

```cpp
// 一次处理 1024 行（一个 Chunk）
Column a = batch.column("a");
NullAwareBoolColumn filter = VectorizedFunction::gt(a, 10);
Batch result = batch.filter(filter); // SIMD 加速
```

## 关键技术
| 技术 | 说明 |
| --- | --- |
| Columnar Batch | 数据按列组织，每列是一个 `IColumn`对象（如 `ColumnInt64`） |
| SIMD 指令 | 利用 AVX2/AVX-512 并行比较、计算（如 `vpcmpgtd`） |
| 延迟物化（Late Materialization） | 先计算过滤条件，再物化所需列，减少内存带宽 |
| 字典编码向量化 | 低基数字符串直接在 ID 上运算，避免解码 |


## 支持的算子（部分）
+ `Scan`：谓词下推向量化
+ `Filter`：布尔表达式向量化
+ Project：表达式计算（`+`、`-`、`SUBSTR`、`DATE_ADD` 等）
+ Aggregate：`SUM`、`COUNT`、`MIN/MAX` 向量化
+ Join：Hash Join Build/Probe 向量化

📊 性能收益：相比非向量化，吞吐提升 3~10 倍（TPC-H 测试）

## 内存管理
+ Arena 内存池：Chunk 内存统一申请，避免频繁 malloc/free
+ 零拷贝传递：Batch 在算子间通过引用传递，减少复制

# 查询执行全流程示例
以 `SELECT user_id, SUM(clicks) FROM logs WHERE dt='2025-03-04' GROUP BY user_id` 为例：

1. FE：
    - 解析 → 分析（绑定 logs 表，裁剪分区 p20250304）
    - 改写（下推 `WHERE`，裁剪 `clicks`/`user_id` 列）
    - 优化（RBO：Agg 下推；CBO：估算行数）
    - 生成 Fragment：Scan → Agg → Exchange → Final Agg
2. BE：
    - 接收 Fragment，启动 Pipeline
    - Scan 算子：
        * 读取 Tablet Segment
        * 向量化过滤 `dt='2025-03-04'`
        * 输出 `(user_id, clicks)` Chunk
    - Local Agg 算子：
        * Hash Table 向量化聚合
        * 输出部分结果
    - Exchange：Shuffle 到 Root BE
    - Final Agg：合并全局结果
3. 返回：FE 汇总结果，通过 MySQL 协议返回客户端

# 性能调优关键点
| 方向 | 建议 |
| --- | --- |
| 统计信息 | 对大表定期 `ANALYZE TABLE`，提升 CBO 准确性 |
| Join 优化 | 小表用 Broadcast，大表确保分桶键一致（Colocation） |
| 向量化 | 确保表达式被向量化（避免 UDF 退化为行式） |
| 并行度 | 调整 `parallel_fragment_exec_instance_num`（默认 1） |
| 内存 | 增大 `vector_chunk_size`（默认 4096 行）提升 SIMD 效率 |


# 未来演进方向

2. GPU 加速：利用 CUDA 加速聚合/Join
3. 编译执行（Compilation）：将表达式编译为 LLVM IR，进一步减少解释开销
4. 智能索引推荐：结合查询模式自动建议 BloomFilter/Bitmap
