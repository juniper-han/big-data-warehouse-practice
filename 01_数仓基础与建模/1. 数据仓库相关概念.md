### 总结：数仓核心概念解释
1. 数据域（Data Domain）：数据域是从数据标准化和治理角度划分的数据集合，用于整合具有相似特性或用途的数据元素。例如，时间数据域包含日期、季度等字段，财务数据域包含收入、成本等指标。
2. 业务域（Business Domain）：业务域是从企业业务视角划分的功能模块或业务范围，代表独立的业务单元或流程。例如，零售行业的“销售管理”“库存管理”“客户管理”均属于不同的业务域
3. 主题域（Subject Area）：按分析视角划分的数据主题集合，聚焦于特定分析目标，通常跨数据域整合数据，用于支撑具体分析需求，如用户画像
4. 业务过程（Business Process）：企业运营中可被量化或记录的具体活动，通常对应一个或多个事实表。指导事实表设计。
5. 原子指标（Atomic Metric）：不可再拆分的业务度量，直接反映某一业务过程的量化结果。需明确口径（业务过程 + 度量）
6. 派生指标（Derived Metric）：基于原子指标通过计算规则衍生出的复合指标，通常包含统计周期、维度、聚合方式。`派生指标名称 = 统计周期 + 维度 + 原子指标 + 聚合规则 `
7. 粒度（Granularity）：数据仓库中数据的详细程度，由事实表中每行记录的业务含义决定。
8. 总线矩阵（Bus Matrix）：- 用于规划数仓架构的工具，横轴为业务过程（事实表），纵轴为维度（维度表），交叉点表示关联关系。
9. 数据集市（Data Mart）：面向特定部门或业务线的数据子集（如财务数据集市、营销数据集市）。
10. 数据湖（Data Lake）：存储原始多格式（结构化/半结构化/非结构化）数据的存储库。
11. 元数据（Metadata）：描述数据的数据，包括表结构、字段含义、数据血缘等。
12. 数据血缘（Data Lineage）：追踪数据从源系统到最终应用的完整流转路径（Apache Atlas、Alation）
13. 数据质量（Data Quality）：准确性，完整性，一致性
14. 物化视图（Materialized View）：预先计算并存储查询结果的视图，支持自动刷新。
15. 多维分析（OLAP）：基于维度模型的多角度数据探索，支持上卷（Roll-up）、下钻（Drill-down）、切片（Slice）等操作。

**数仓知识体系全景**

| **类别** | **核心概念** |
| --- | --- |
| **数据建模** | 事实表、维度表、粒度、总线矩阵、缓慢变化维（SCD）、代理键 |
| **架构设计** | ODS、DWD、DWS、数据集市、数据湖、实时数仓 |
| **数据处理** | ETL/ELT、CDC、数据管道（Data Pipeline）、批处理/流处理 |
| **数据治理** | 元数据、数据血缘、数据质量、数据目录（Data Catalog）、主数据管理（MDM） |
| **分析应用** | 原子指标、派生指标、聚合表、物化视图、OLAP、ABI（分析型商业智能） |

### 数仓最重要的是什么？
**数据质量**

一个数仓无论技术多先进、模型多复杂，如果产出的数据不准确、不可信，那么就完全失去了存在的价值。

重要性顺序

+ **数据质量**：准确性、一致性、完整性、可信赖
+ **数据模型**：规范性、稳定性、可扩展性
+ **价值导向**：业务赋能、易用性、性能
+ **管理规范**：流程、标准、治理

### 什么是 SCD？SCD 的常用处理方式
Slowly Changing Dimension, SCD 维度属性，随时间缓慢变化，有历史追溯需求

+ **类型一：重写**：覆盖历史、仅保留最新
+ **类型二：新增行**：保留历史、版本标记（生效时间、是否当前）
+ **类型三：新增列**：保留部分历史、当前值与上一值

选择哪种方式取决于业务上对历史数据的需求强度。

### 对元数据的理解？
+ **定义**：是数据的数据，描述数据本身特征的信息
+ **分类**：
    - 技术元数据：数据的技术细节，库名、表名、字段名、数据类型、存储位置
    - 业务元数据：业务角度解释数据，业务定义，指标口径，负责人
    - 操作元数据：数据的过程信息，作业运行时间、数据量、质量
    - 管理元数据：数据的管理属性，数据权限、访问策略、安全等级、生命周期策略、版本信息
+ **价值**：让数据可理解、可管理、可信任，是建立数据血缘、数据治理的基础
+ **应用场景**：数据发现、血缘分析、影响分析、数据治理

+ **技术组成**：采集器、存储库、服务层

### 元数据管理标准？
+ **国际标准**：ISO/IEC 11179, CWM
+ **行业实践**：Apache Atlas
+ **内容**：覆盖数据的全生命周期
    - 技术元数据：数据结构、存储位置、血缘、代码和调度依赖
    - 业务元数据：业务含义，指标定义、计算口径，数据质量规则，数据业务分类，负责人
    - 操作元数据：数据处理过程的动态信息，数据作业的运行历史、失败成功状态、消耗的资源、数据量等
    - 管理元数据：数据的管理属性，数据权限、访问策略、安全等级、生命周期策略、版本信息

### 什么是数据模型
数据模型是对现实世界中的【数据特征】进行抽象和描述的工具，它定义了数据的结构、属性、关系以及约束条件。

+ **数据模型定义**：现实中数据特征的抽象和描述，定义了数据的结构、关系、约束
+ **目的**：业务人员和技术人员的沟通工具，是数据库设计的蓝图，数据资产管理的基础
+ **三个抽象层次**：概念模型、逻辑模型、物理模型
+ **价值作用**：消除数据的歧义，保证数据的质量和一致性，提高数据存储、访问效率

---

### 概念数据模型、逻辑数据模型、物理数据模型分别是什么？
代表了数据模型设计的三个不同抽象层次，从业务概念逐步落实到技术实现。

+ **概念模型（有什么）**：高层抽象，业务视角，核心业务实体以及之间的关系，业务人员和技术人员就业务领域达成共识
+ **逻辑模型（是什么样）**：大幅细化，详细设计，技术无关，明确实体的属性、类型、主外键关系，遵从设计范式
+ **物理模型（怎么建）**：具体实现，对应特定的技术，定义技术细节（分区、分桶、索引）

---

### 模型设计的思路，业务驱动？技术驱动？
正确思路：业务驱动为主，技术驱动为辅  
以业务目标和需求为导向进行模型规划与设计，同时充分考虑技术平台的特性与限制  
**业务决定“做什么”，技术决定“怎么做”**。

**业务驱动内涵**：解决业务问题、支撑分析需求、快速实现价值  
**技术驱动角色**：技术可行性、性能约束、长远扩展

---

### 为什么需要数仓建模？
数仓建模的根本目的是为了将原始数据有序地组织起来，从而高效、可靠地支撑分析决策。  
通过数仓建模，可以：

+ **性能**：提高查询速度，快速响应
+ **成本**：减少扫描，节约计算资源
+ **易用**：模型易于理解、简化查询
+ **质量**：数据一致、口径统一，消除数据歧义，提高可信度

---

### OLAP 和 OLTP 的建模方法的选择？
建模方法基于核心目标

+ OLTP：满足 3NF 的实体关系模型
    - 目标：面向事务，随机读写，追求极致写入速度、事务一致性，模型需要高度规范化、避免数据冗余
+ OLAP：维度建模
    - 目标：面向分析，批量读写，追求极致查询性能，增加数据冗余

---

### 三范式是什么？
+ **第一范式**：原子性、列不可分
+ **第二范式**：前提：满足第一范式，非主键完全依赖主键，不能只依赖主键的一部分
+ **第三范式**：前提：满足前两范式，非主键直接依赖主键、消除传递依赖
+ **核心目标**：减少冗余、避免异常

---

### ER模型是什么
ER模型，也称为实体-关系模型，是一种用于描述现实世界中数据概念及其相互关系的数据建模方法。

+ **核心组件**：实体、属性、关系
+ **图形化表示**：ER图
+ **设计目标**：概念设计、规范化、减少冗余
+ **应用场景**：OLTP系统数据库设计

---

### 维度建模是什么？
维度建模是一种专门为数据仓库和分析型数据库设计的数据库结构技术

+ **组件**：事实表、维度表
+ **设计目标**：提升分析查询性能、业务可理解性
+ **建模过程**：选择业务过程、声明粒度、确认维度、确认事实
+ **典型结构**：星型模型、雪花模型

---

### 事实表是什么？
反映具体业务事件，存储业务过程中可量化的数值型度量数据。

+ **内容**：存储的是度量，事实，通过外键连接到维度表
+ **特征**：数值型、可加性、业务过程
+ **设计基础**：粒度
+ **类型**：事务事实表、周期快照事实表、累计快照事实表

---

### 维度表是什么
维度表是维度建模中用于描述业务环境的表。

+ **内容**：描述性属性，一般是文本信息
+ **作用**：对事实数据进行筛选、分组、分类和标记
+ **设计特征**：宽表，反规范化设计，层次结构放在一张表里
+ **与事实表关系**：主键与外键关联

---

### 事实表的类型？
3 种类型

+ **事务事实表**：记录的原子事件，事件发生才记录，数据只插入不更新
+ **周期快照事实表**：不以事件为中心，定期地记录状态，间隔周期内的度量统计，事务事实表之上的聚集表
+ **累计快照事实表**：记录具有明确工作流和多个里程碑的短期过程，记录关键时间点，流程推进->数据更新
+ **核心区别**：粒度、事实类型、更新行为

---

### 维度建模的三种模式
+ **星型模式**：事实表在中心，周围扁平化维度表，反规范化，简单、查询高效
+ **雪花模式**：维度表规范化，减少冗余，节省空间，但查询复杂影响性能
+ **星座模式**：多个事实表共享维度

---

### 维度建模步骤 / 事实表设计过程
+ **选择业务过程**：分析目标、数据源
+ **声明粒度**：精确定义事实表一行代表什么
+ **确认维度**：基于粒度，确定需要哪些描述性属性
+ **确认事实**：确认业务过程中的量化度量，数值型可加性

---

### 维度表设计过程
+ **选择维度**：基于事实表的业务过程，选择相关维度
+ **确定主维度和属性**：主键、层次结构、文本描述
+ **反规范化处理**：宽表、减少连接
+ **缓慢变化维处理**：历史追踪、类型选择

---

### 维度表设计中有哪些值得注意的地方
+ **属性丰富性**：提供详细、易懂的文本描述，避免直接使用代码值
+ **数据质量**：处理空值或默认值，确保数据完整性
+ **查询性能**：反规范化，常用筛选字段建立索引
+ **可维护性**：SCD策略，代理键而非自然键，屏蔽业务主键变化的影响

---

### 事实表设计中有哪些值得注意的地方
+ **粒度声明**：原子性，定义清晰无歧义
+ **仔细考虑事实可加性**：可加、半可加、不可加
+ **保证维度键的一致性**：使用跨业务流程的一致性维度，使用代理键而非自然键，以应对源系统变化和 SCD
+ **空值处理**：事实可以为空，但维度外键不为空，应关联到一个特殊的未知维度行，以确保查询完整性

---

### 数仓架构体系是什么样的？
是分层的、模块化的整体。分成以下几个逻辑层次，数据自下而上流动

+ **<font style="color:rgb(0, 0, 0);">数据采集与集成层</font>**<font style="color:rgb(0, 0, 0);">：数据采集与集成</font>
+ **<font style="color:rgb(0, 0, 0);">存储计算层</font>**<font style="color:rgb(0, 0, 0);">：技术核心，大数据平台，数仓分层在这一层体现</font>
+ **<font style="color:rgb(0, 0, 0);">数据服务层</font>**<font style="color:rgb(0, 0, 0);">：统一数据服务，实现数据与应用的解耦</font>
+ **<font style="color:rgb(0, 0, 0);">应用层</font>**<font style="color:rgb(0, 0, 0);">：面向最终用户，BI报表平台，即席查询工具，各类业务系统</font>
+ **<font style="color:rgb(0, 0, 0);">数据治理与管理体系</font>**<font style="color:rgb(0, 0, 0);">：贯穿所有层次，元数据、质量、安全</font>

---

### 为什么要对数仓分层
+ **清晰化管理**：数据分层管理，每层职责明确，易于维护
+ **数据复用**：公共数据被上层多次复用，减少重复计算，一次开发多次使用，节省资源
+ **保障数据质量**：屏蔽原始数据异常，逐层清洗，统一口径，方便追踪数据血缘
+ **效率与成本**：简化复杂计算、降低资源消耗

---

### 数仓各层
+ **ODS层**：操作数据层、原始数据、贴源存储
+ **DWD层**：数据仓库明细层、清洗整合、维度建模
+ **DWS层**：数据仓库汇总层、轻度汇总、主题宽表
+ **ADS层**：应用数据层、高度聚合、个性化指标
+ **DIM层**：汇总维度表

---

### 模型设计的基本原则都有哪些？
+ **高内聚低耦合**：数据相关性、模块独立性
+ **可扩展性**：结构稳定、易于扩展
+ **可读性**：命名规范、结构清晰
+ **高性能与低成本**：查询效率、存储成本

---

### 数仓底层是如何抽取数据的
抽取方式，基于数据量、源系统类型、实时性要求

+ **<font style="color:rgb(0, 0, 0);">抽取方式</font>**<font style="color:rgb(0, 0, 0);">：全量抽取、增量抽取</font>
+ **<font style="color:rgb(0, 0, 0);">技术工具</font>**<font style="color:rgb(0, 0, 0);">：基于查询（Sqoop），CDC（OGG），Kafka</font>
+ **<font style="color:rgb(0, 0, 0);">实现模式</font>**<font style="color:rgb(0, 0, 0);">：直连抽取、基于文件、日志解析</font>

### 为什么选择 Sqoop
<font style="color:rgb(0, 0, 0);">选择Sqoop主要基于其在特定场景下的核心优势。</font>

+ **<font style="color:rgb(0, 0, 0);">场景定位</font>**<font style="color:rgb(0, 0, 0);">：在 Hadoop 生态中，关系型数据库与 HDFS 批量数据同步</font>
+ **<font style="color:rgb(0, 0, 0);">核心优势</font>**<font style="color:rgb(0, 0, 0);">：稳定可靠高效，MapReduce并行 -> 高效利用集群资源，自动完成到Hive类型映射，支持全量、增量导入，功能成熟</font>
+ **<font style="color:rgb(0, 0, 0);">适用场景</font>**<font style="color:rgb(0, 0, 0);">：结构化数据、全量/增量、数据库到HDFS</font>

### Lambda 架构
用于处理大规模数据的系统设计范式

+ **三层架构**：
    - 批处理层：历史数据，批处理引擎
    - 速度层：增量数据，流处理引擎
    - 服务层：合并处理批/流结果，对外提供统一接口
+ **核心思想**：冗余计算，最终一致
+ **批处理层**：全量数据、高延迟、准确无误
+ **速度层**：增量数据、低延迟、近似结果
+ 劣势：两套逻辑，资源占用多，维护复杂

---

### Kappa 架构
Lambda 架构的改进，使用一套流处理系统来统一处理实时数据和历史数据，简化架构

+ **核心组件**：支持数据重播的不可变日志系统，流处理引擎
+ **处理流程**：重播历史、统一逻辑
+ **优势与挑战**：简化维护、历史重算开销
