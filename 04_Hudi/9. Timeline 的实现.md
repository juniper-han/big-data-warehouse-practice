Hudi 的 Timeline 并不是简单的 log 文件，而是一个精心设计的**元数据管理系统**。它由多个`Instant`文件组成，每个文件都是一个**独立的元数据记录**，这样设计有利于并发写入和高效查询。

# Timeline 的物理存储结构
## 目录结构
以下为 Hudi 表的元数据区

```plain
/user/hudi/table/.hoodie/        # 元数据区
├── hoodie.properties           # 表配置文件
├── timeline/
│   ├── 20240101000000.commit  # 提交记录
│   ├── 20240101000000.deltacommit
│   ├── 20240101000001.commit
│   ├── 20240101000002.clean   # 清理记录
│   ├── 20240101000003.compaction  # 压缩记录
│   └── .aux/                   # 辅助文件（索引等）
```

## Instant（即时点）文件格式
每个 Instant 是一个独立的文件，包含：

```json
{
  "state": "COMPLETED",      // 状态：REQUESTED, INFLIGHT, COMPLETED
  "action": "commit",        // 操作类型
  "timestamp": "20240101000000", // 时间戳
  "fileSlice": {             // 影响的文件切片
    "partition": "2024/01/01",
    "fileId": "abc123"
  },
  "metrics": {               // 统计信息
    "bytesWritten": 134217728,
    "recordsInserted": 1000000
  }
}
```

## Timeline vs Log File
Timeline（时间轴）：表的"操作日志"（元数据层面）

+ 存储位置：`/.hoodie/timeline/`
+ 内容：操作记录（commit、deltacommit、compaction等），什么时间、做了什么操作、影响了哪些文件
+ 格式：每个 instant 一个文件（JSON 或 Avro）
+ 作用：协调、版本控制、事务管理

Log File（日志文件）：表的"数据变更日志"（数据层面）

+ 存储位置：`/partition/path/.fileId.log`
+ 内容：具体数据记录的增删改
+ 格式：Avro 序列化记录
+ 作用：存储增量更新，支持近实时查询

# Timeline 的实现原理
1. Timeline 不是单一日志文件，而是**事件文件集合**
    1. 传统日志文件：顺序追加，需要顺序读取
    2. Hudi Timeline：每个 Instant 是独立文件，可以并行读取
2. Timeline 有三个关键目录：
    1. `timeline/ `：已完成的 Instant
    2. `.inflight/ `：进行中的 Instant  
    3. `.requested/ `：已请求但未开始的 Instant

### 策略 1：时间范围过滤
```plain

2. 只读取这些commit涉及的文件

```

# Timeline 的读写流程
## 写入流程（创建一个Instant）：
```java
// 1. 创建.requested文件
String instantTime = HoodieActiveTimeline.createNewInstantTime();
Path requestedFile = new Path(timelinePath, instantTime + ".commit.requested");
fs.create(requestedFile);  // 原子创建，作为锁

// 2. 写入数据文件
writeDataFiles(instantTime);

// 3. 转换为.inflight
renameTo(requestedFile, instantTime + ".commit.inflight");

// 4. 写入完成，转换为.completed
renameTo(inflightFile, instantTime + ".commit");
```

## 读取流程（查询某个时间点）：
```java
public List<FileSlice> getFileSlicesAtTime(String queryTime) {
    // 1. 找到queryTime之前最后一个commit
    HoodieInstant lastInstant = timeline.getInstants()
        .filter(instant -> instant.getTimestamp() <= queryTime)
        .maxBy(HoodieInstant::getTimestamp);
    
    // 2. 获取这个commit影响的文件
    List<FileSlice> slices = getAffectedFileSlices(lastInstant);
    
    // 3. 获取这些文件的元数据（从metadata表）
    Map<String, FileMeta> fileMetas = metadataTable.getFileMetas(
        slices.stream().map(FileSlice::getFileId)
    );
    
    return slices;
}
```

## Timeline 与 Log File 的关联机制
### 写入时的关联
数据写入时

1. 在 Timeline 上创建`DELTA_COMMIT instant`
2. 写入数据到 Log File
3. 收集写入的 Log File 信息
4. 创建`DELTA_COMMIT_Metadata`：创建这次写入的元数据，包含写入的 Log File 的信息
5. 在 Timeline 中记录`DELTA_COMMIT`，指向这个 Log File：完成`DELTA_COMMIT`，将元数据写入 Timeline，存储在 `instant`中

### 查询时的协作流程
MOR 表查询，读时合并

1. 从 Timeline 获取最新的完整的`COMMIT instant`（Base File）
2. 从 Timeline 获取该`COMMIT instant`之后的所有`DELTA_COMMIT instant`（Log Files）
    1. 读取`DELTA_COMMIT instant`文件内容
    2. 反序列化为`DELTA_COMMIT_Metadata`
    3. 提取 Log Files 信息
3. 合并 Base File 和 Log Files，得到 FileSlice

### 压缩（Compaction）时的交互
压缩（Compaction）工作流程

1. 在 Timeline 上创建 `Compaction Instant`
2. 读取 Timeline，收集需要压缩的 FileSlices（Base File + Log Files）
3. 读取 Log Files 和 Base File，合并写入到新的 Base File
    1. 注意：旧的 Log Files 和旧的 Base File不会被立即删除，而是在`Clean`操作时删除
4. 提交元数据，完成 Compaction

```latex
压缩前Timeline：
  20240101000000.commit      # 初始base file
  20240101010000.deltacommit # 添加log1.log
  20240101020000.deltacommit # 添加log2.log
  20240101030000.deltacommit # 添加log3.log

压缩后Timeline：
  20240101000000.commit      # 初始base file
  20240101010000.deltacommit # 添加log1.log
  20240101020000.deltacommit # 添加log2.log  
  20240101030000.deltacommit # 添加log3.log
  20240101040000.commit      # 新的commit，合并了log1-3到新base file
```

### 故障恢复时的协同
1. 在 Timeline 上读取`DELTA_COMMIT`的元数据
2. 删除该`DELTA_COMMIT`创建的所有 Log Files
3. 从 Timeline 中删除该`DELTA_COMMIT Instant`

## COW 表与 MOR 表的 Timeline 对比
COW 表的 Timeline 特点

+ COW 表的 Instant 只包含 parquet 文件信息
+ Instant 记录了 parquet 文件的变化

MOR 表的 Timeline 特点

# 性能优化技术
1. 分层存储
    1. 最新的Instant：在内存缓存
    2. 近期Instant：在本地磁盘缓存
    3. 历史Instant：在对象存储，按需加载
2. Compaction：定期将多个小 Instant 文件合并
    - 输入：1000个小的Instant文件
    - 输出：1个大的parquet文件 + 1个索引文件
3. 预取和缓存
    1. 按时间顺序预取下一个 Instant
    2. 按访问模式预取（如经常查询的时间范围）

## 优化效果：
一个典型的大表（1PB，10亿文件）：

+ 传统`listFiles()`: 无法完成（超时）
+ Hudi Timeline查询: < 1秒  
原因：Timeline只记录变更，不记录所有文件

# 总结
1. **事件驱动存储**：每个Instant独立，避免大日志文件
2. **元数据表加速**：专门的统计信息表
3. **分层缓存**：热数据在内存，冷数据在存储
4. **索引辅助**：布隆过滤器、列统计等
