# 事务写（ACID能力）
Hudi 的 ACID 事务能力，是数据湖从"数据存储"升级为"Lakehouse（湖仓一体）"的关键特性。

## 传统数据湖的痛点
没有 ACID 的数据湖写入

1. Flink作业写入 100 个文件，写到第 50 个时作业失败
2. 结果：50 个文件留在存储中，50 个没写
3. 查询引擎看到的是"一半新一半旧"的数据
4. 数据一致性被破坏



业务关键场景的要求：

+ 原子发布：一批数据要么全写成功，要么全不写
+ 部分失败回滚：写入中途失败要能自动清理
+ 查询一致性：查询不能看到"部分写入"的数据
+ 数据可靠性：提交的数据必须持久化，不丢失

## Hudi 的 ACID 保证及体现
1. Atomicity（原子性）：要么全成功，要么全失败
    1. Flink 作业写入流程：开始事务 → 写入数据文件 → 写入提交记录 → 提交成功
    2. 如果中途失败 → 回滚所有写入 → 清理临时文件
2. Consistency（一致性）：数据始终保持有效状态
    1. 保证1：模式一致性：写入的数据必须符合表结构定义
    2. 保证2：约束一致性：主键唯一性约束（Hudi独有特性）
    3. 保证3：索引一致性：记录索引与数据文件保持一致
3. Isolation（隔离性）：并发写入互不干扰
    1. 通过 Timeline 协调、Marker 机制、早期冲突检测，确保并发写入像串行执行一样
4. Durability（持久性）：写入成功后数据不会丢失
    1. Hudi 依赖：Timeline 持久记录，多副本机制（由底层存储保证），云存储的持久性

## Hudi 高效的回滚机制
需要回滚的原因：Flink 作业写入失败，可能的原因

+ TaskManager故障
+ 网络中断
+ 数据格式错误
+ 并发冲突



Hudi 的回滚流程

1. **检测失败的事务**
    - 通过 Timeline 上状态为`inflight`的 `instant`
    - 心跳超时检测
2. **清理孤儿文件**
    - 使用 Marker 机制找到所有部分写入的文件
    - 删除这些文件
3. **恢复表状态**
    - 将 Timeline 状态更新为`rollback`
    - 确保表回到一致状态



避免的常见问题：

+ 孤儿文件：无人引用的数据文件
+ 中间状态残留：写入过程中的临时文件
+ 部分可见数据：查询看到不完整的数据

## FlinkSQL + Hudi 的事务写入流程
一个 Flink 作业的完整事务生命周期：

```sql
-- Flink SQL 作业
INSERT INTO hudi_table 
SELECT * FROM source_table;
```

Hudi 在后台的执行：

1. 开始事务（创建`inflight instant`）
2. 分配写入 ID（`write token`）
3. 写入数据文件（配合 Markers）
4. 写入索引更新
5. 提交事务（完成`instant`）
6. 如果失败 → 触发回滚

关键时间点

+ t0: 作业开始，创建`inflight instant`
+ t1: 数据文件开始写入
+ t2: 所有文件写入完成
+ t3: 提交元数据（原子操作）
+ t4: 查询看到新数据

关键：t0-t3期间，查询看不到新数据，t3之后，查询立即看到完整的新数据



场景：多流拼接，三个流同时写入同一张表

```sql
-- 流A：实时点击流（高频）
INSERT INTO user_table SELECT *, 'click' FROM click_stream;

-- 流B：订单流（中频）
INSERT INTO user_table SELECT *, 'order' FROM order_stream;

-- 流C：批处理修正（低频）
INSERT INTO user_table SELECT *, 'correction' FROM correction_job;
```

没有ACID的问题：

```latex
流A写入成功
流B写入失败，但留下了部分文件
流C写入成功

查询结果：
  - 看到流A的完整数据
  - 看到流B的部分数据（损坏！）
  - 看到流C的完整数据
  数据不一致！
```

有ACID的保障：

```latex
流A：写入成功，完整提交
流B：写入失败，自动回滚，不留痕迹
流C：写入成功，完整提交

查询结果：
  - 只看到流A和流C的完整数据
  - 流B就像从来没运行过一样
  数据一致！
```

## Hudi 两阶段提交
阶段1：准备

+ 写入所有数据文件
+ 创建必要的索引
+ 但不让查询可见

阶段2：提交

+ 原子操作更新元数据
+ 使新数据对查询可见
+ 如果失败，清理阶段1的所有内容
