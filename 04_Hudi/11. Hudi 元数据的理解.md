易混淆的几个概念关系

+ Hudi 元数据
+ Hive Metastore
+ Flink 等计算引擎侧的 Catalog

# Hudi 元数据
## **Hudi 数据与元数据的存储位置**
**Hudi 数据文件：**

Hudi 表的数据文件存在在 HDFS（或 OSS / S3）上

```plain
/warehouse/db/table/
  ├── partition=2025-01-01/
  │     ├── file1.parquet
  │     ├── file2.parquet
  ├── partition=2025-01-02/
```

****

**Hudi 元数据**

每个 Hudi 表的元数据，都存在自己的表路径下的 `.hoodie/` 目录中：

+ 注意：不在 Hive Metastore 中

```plain
/warehouse/db/table/.hoodie/
  ├── hoodie.properties
  ├── hoodie.commit
  ├── hoodie.deltacommit
  ├── hoodie.rollback
  ├── hoodie.clean
  ├── metadata/
```

这部分元数据记录了：

+ commit 时间线（timeline）
+ 哪些文件属于哪个 commit
+ 文件级别的增量信息
+ compaction / clean / rollback 状态
+ 表类型（COW / MOR）
+ 主键、preCombine 字段等

⚠️** 这些内容从来不会写进 Hive Metastore**

---

## Hudi 元数据管理的层级：单表自治
### 概述
Hudi 本身**「没有」一个跨表、跨库的“全局元数据汇总层”**。  
Hudi**<font style="background-color:#FBDE28;"> 是“以表为边界”的元数据系统</font>**。

也就是说：

+ ✅ **单表级元数据**：有（`.hoodie/`）
+ ❌ **多表级全局元数据目录**：没有
+ ❌ **自动发现所有 Hudi 表的机制**：没有

### 为什么 Hudi 不做“全局元数据层”？
原因：基于设计目标的**刻意架构取舍**。



**Hudi 的设计目标**：保证“一张表”的事务一致性、增量语义和写入性能

所以它的元数据边界是：**一张表 = 一个事务域（Transaction Boundary）**



如果做一个“跨表元数据中心”，就必须解决：

+ 跨表事务一致性？
+ 跨表 Schema 演进？
+ 表创建 / 删除的并发？
+ 多引擎并发注册？

👉 这些问题**直接把系统复杂度拉爆，**所以 Hudi **刻意不碰这件事**。

### 针对 Hudi“单表自治”的解决方案
Hudi 的**“单表自治”是有代价的**：没有自动发现所有 Hudi 表的机制

业界常见的补充方案：

1. Hive Metastore / Glue / Unity Catalog（最常见）
    - 解决**表发现、跨引擎共享问题，且**不参与事务
2. 自研 Catalog / 表管理服务
    - 用一个**元数据服务，存储：**表名、path、负责人、数据量、更新频率

注意：这些系统**不是 Hudi 的一部分**

---

## Hudi 表如何被“发现”
Hudi 表通过**表的存储路径**被“发现”

每一张 Hudi 表都有一个**确定的根路径，**在该路径下有 `.hoodie/`

```plain
/warehouse/db1/table1/.hoodie/
/warehouse/db2/table2/.hoodie/
```

任何计算引擎：只要能获取到 Hudi 表的`path`，它就能**直接加载并读写**这张 Hudi 表

# Catalog 和 Hive Metastore
## Catalog
Catalog 是计算引擎侧的概念

+ 是计算引擎中的元数据抽象
+ 用来管理表的命名空间、schema 以及表与物理存储路径之间的映射关系

## Hive Metastore 存储的是 Hudi 表的哪些“元数据”？
Hive Metastore 只是【多引擎共享表定义】的“注册中心”，不是 Hudi 的元数据存储中心。

+ Hudi 的元数据以表级别自治的方式存储在 HDFS 上
+ Hive Metastore 只是表名到路径的注册中心，负责对外暴露表结构与访问入口。



Hive Metastore 只是一个：**“把 Hudi 表的 path 组织成 namespace 的外部系统”**

+ 它做的是：`db.table  →  hdfs://xxx/warehouse/db/table`
+ 它不做：管理 Hudi commit / 事务 / 文件版本



Hive Metastore 中存的 Hudi 表的元数据是：“**把 Hudi 当成一张表来用时，需要的那一小部分元数据**”

+ 包括：database 名、table 名、schema（字段名、类型）、partition 字段、表类型（EXTERNAL）、表路径（LOCATION）、表的 input/output format（Hudi）

以上这些是 “SQL 引擎层能认表”的信息

## 如何通过 Catalog 和 Hive Metastore 发现 Hudi 表
目的：发现一张 Hudi 表

直接方案：Flink 等计算引擎直接使用这张 Hudi 表的存储路径“发现”这张表。



间接方案（生产中常用）：将该 Hudi 表在 Hive Metastore 中注册

Hive Metastore 存储了 Hudi 表的表名和存储路径的映射关系；

因此 Flink 等计算引擎可以**通过 Catalog **对接 Hive Metastore 等元数据服务，将**表名映射到具体路径**，从而支持通过 SQL 以**表名的方式**访问 Hudi 表。



因此：Flink 等计算引擎需要使用 Catalog 才能在 Hive Metastore 中通过 Hudi 表名获取该表的存储路径

## 如果不用 Hive Metastore
以下场景，不使用 Hive Metastore 管理 Hudi 表的元数据

+ Flink 可以通过 Hudi 表路径，实时写 Hudi
+ 内部作业系统可以统一管理 Hudi 表路径

这种场景下

+ Hive 查不了
+ Hudi 表完全可用
+ 事务 / 增量 / 回溯全都正常



如果使用 Hive Metastore 管理 Hudi 表的元数据，但是 HMS 挂了

+ Hive / Trino 查不了
+ Hudi 表 **不会损坏**
+ Flink / Spark 仍然可以通过 path 读

# Hudi 表 vs Hive Metastore vs Catalog
对一张 Hudi 表

Hudi 表

+ 知道：自己有哪些 commit；哪些文件有效
+ 不知道：自己叫啥名字；属于哪个 database

Hive Metastore

+ 知道：`db.table → hdfs path`；schema
+ 不知道：Hudi 的事务细节
+ 提供“使用便利性”

Catalog（Flink / Spark）

+ 知道：去哪找表（Metastore）
+ 不知道：表内部是怎么存的



Hudi 表的存储

| 内容 | 在哪里 |
| --- | --- |
| 数据文件（Parquet / Log 文件） | HDFS |
| commit / delta commit | `.hoodie/` |
| 文件版本、增量语义 | `.hoodie/` |
| 表 schema | Hive Metastore |
| 表路径 | Hive Metastore |
| 分区字段 | Hive Metastore |


# Hudi 表一致性的边界与 Hive Metastore
## 结论
+ Hudi 的事务一致性边界在`“表路径 + .hoodie”`
+ Hive Metastore 与 Hudi 表之间只保证“最终一致性”：这是一个“被设计出来、且可接受”的最终一致性

****

Hive Metastore 和 Hudi 会存在短暂的不一致，但这是设计使然。

+ Hudi 的事务一致性以表路径下的 `.hoodie` 为准
+ Hive Metastore 只负责表的发现和 SQL 访问便利性，不影响正确性

因此两者之间是最终一致的关系，不影响数据的正确性和增量语义。

## 只能保证最终一致性的原因
### Hudi 和 Hive Metastore 解决的是完全不同的问题
| 系统 | 关注点 |
| --- | --- |
| Hudi | 单表事务、一致性、增量 |
| Hive Metastore | 表发现、命名空间、schema |


如果 Hive Metastore 也要参与事务：

+ 就必须感知 commit / rollback
+ 就必须参与文件级一致性
+ 就会变成事务协调者

👉 这会直接把系统复杂度拉到不可控

所以 Hudi 的设计是：事务只在表路径内闭环，不外溢

### **Hive Metastore 为什么“必然可能不一致”？**
典型流程是：

1. Hudi 在 HDFS 上完成一次 commit
2. `.hoodie` timeline 更新
3. （异步）向 Hive Metastore 同步表 / 分区信息

在 2 → 3 之间这个时间窗内：

+ Hudi 表：✅ 已提交
+ Hive Metastore：❌ 还没更新

即：短暂不一致

### 这个“不一致”不影响数据正确性
这个不一致，不影响数据的正确性，只是会影响正确数据的“可发现性”

+ 新数据已经正确写入
+ 但是：Hive 可能不能马上看到新数据



原因：

+ 数据是否正确 → 由 `.hoodie` 决定
+ Hive 是否能查到 → 取决于 Metastore 是否更新

所以：

+ Flink / Spark 通过 path 读：永远是强一致
+ Hive 通过 table 查：可能短暂看不到最新分区
    - Hive Metastore 只是一个 Index Registry Directory

### 这是“工程上可接受的最终一致性”
原因

1. 因为这影响的是“发现能力”，而非“数据正确性”
2. 不影响事务的正确性：Hudi Commit 已完成就不会被破坏
3. 不影响增量语义：所有 reader 都基于 timeline 读
4. 不影响可恢复性：重试/同步失败可以再次执行

## 总结图
```latex
┌──────────────────────────────┐
        │        计算 / 查询层          │
        │  Flink / Spark / Hive / SQL   │
        └───────────────▲──────────────┘
                        │
               （Catalog 解析表名）
                        │
        ┌───────────────┴──────────────┐
        │     Hive Metastore（可选）    │
        │  表名 / Schema / Location     │
        │  ⚠ 仅最终一致                  │
        └───────────────▲──────────────┘
                        │
               （表路径 / location）
                        │
        ┌───────────────┴──────────────┐
        │        Hudi 表路径             │
        │  data files + .hoodie/        │
        │  ✅ 强一致 / 事务权威          │
        └───────────────▲──────────────┘
                        │
                   HDFS / OSS
```

# Hudi 和 Hive 的元数据管理差异
+ Hive：元数据与数据分开存储
+ Hudi：元数据与数据存储在同一目录下



把元数据和数据放在一起，是为了让“一张表”成为一个自包含、可迁移、可恢复的事务单元。

这就是 Hudi 和传统 Hive 最大、最底层的设计差异。

## **Hive 元数据和数据分开存储**
Hive 时代的假设是：

+ 数据是 append-only
+ 分区是粗粒度
+ 表结构变化少
+ 没有强事务语义

所以 Hive 选择：

```plain
Hive Metastore：schema / partition / location
HDFS：数据文件
```

这个模型在离线批处理时代是完美的。



但是，针对“更新 + 增量 + 实时”的场景，Hive 这种存储方式，就无法保证元数据和数据的“一致性边界”了

## Hudi 的设计：把边界放在表里
Hudi 的设计

+ 外部系统不参加事务
+ 一致性边界收敛到单表内部
+ 不跨系统协调



Hudi 把所有的表相关的元数据都放到了`/table_path/.hoodie`目录下：这张表的存储目录下，包含了这张表的全部。

因此：

1. Hudi 表是自描述的：只要拷走 Hudi 表目录，只要有 Hudi Reader，表就能读
2. Hudi 表是自恢复的：自己存储事务日志，不依赖外部状态
3. Hudi 表是自一致的：commit 成功即一致，不需要跨系统协调
4. Hudi 表是可迁移的：把表目录拷走就行



Hudi 把 Metastore 从“权威”降级成“索引”，从“真理”降级成“工具”

# 从元数据边界视角看 **Hive / Hudi / Iceberg / Delta**
| 表格式 | 元数据是否与数据同址 | 事务一致性边界 | 是否依赖外部 Metastore |
| --- | --- | --- | --- |
| Hive | ❌ 分离 | ❌ 无事务 | ✅ 必须 |
| Hudi | ✅ 同址（`.hoodie`） | ✅ 单表强一致 | ❌ 非必需 |
| Iceberg | ⚠ 半同址（metadata dir） | ✅ 单表强一致 | ❌ 非必需 |
| Delta | ✅ 同址（`_delta_log`） | ✅ 单表强一致 | ❌ 非必需 |


一句话总结：

从 Hive 到 Hudi / Iceberg / Delta，是从“外部元数据权威”到“表内事务自治”的转变。



Hive 的元数据和数据分离，适合离线批处理；  
Hudi、Iceberg 和 Delta 都将事务和元数据内聚到表目录中，实现单表级一致性，只是在元数据组织方式上有所差异。

---

## **用“元数据边界”看清本质**
### **Hive ——「元数据在外，表本身是哑的」**
**结构**

```plain
Hive Metastore  ←—— 真正的权威
│
└── 表名 / schema / partition
      ↓
HDFS：/warehouse/db/table/part=xxx/
```

**本质**

+ 表 无法自解释
+ 没有事务
+ Metastore 一挂，表就“失忆”

👉 Hive 表不是一个“系统”，只是文件集合

### Hudi ——「一张表就是一个事务系统」
**结构**

```plain
/table_path/
├── data files
└── .hoodie/
    ├── timeline
    ├── commits
    ├── schema
```

本质

+ 元数据 = 表内资产
+ commit 即一致性边界
+ 不依赖外部系统

👉 Hudi 把“数据库的一半”搬进了表目录

### Iceberg ——「元数据集中管理，但仍然表内自治」
**结构**

```plain
/table_path/
├── data/
└── metadata/
    ├── v1.metadata.json
    ├── v2.metadata.json
```

本质

+ 元数据 逻辑集中
+ 但仍然属于该表
+ snapshot 切换即事务

👉 Iceberg 是“显式元数据文件”的表自治

### Delta Lake ——「日志驱动的事务表」
**结构**

```plain
/table_path/
├── data/
└── _delta_log/
    ├── 00001.json
    ├── 00002.json
```

本质

+ 元数据即事务日志
+ append-only log
+ 快照由 log 重放生成

👉 Delta 是 WAL 思想的极致实现

## **为什么这三种新格式都“放弃”Hive Metastore？**
你可以用这一句话理解：跨系统事务是分布式系统的地狱。

所以它们都选择：

+ 把事务边界收敛到“表目录”
+ 把 Metastore 降级成索引

## 但它们“放弃 Metastore”了吗？——没有
非常重要的一点：

它们放弃的是“事务依赖”，不是“集成能力”。

| 能力 | Hive | Hudi | Iceberg | Delta |
| --- | --- | --- | --- | --- |
| 表发现 | Metastore | 可选 | 可选 | 可选 |
| SQL 使用 | 必须 | 常用 | 常用 | 常用 |
| 一致性 | 无 | 表内 | 表内 | 表内 |


****

**真正强的系统，把“真理”放在自己能完全控制的地方。**

+ **Hive 把真理交给外部 → 不稳**
+ **Hudi / Iceberg / Delta 把真理收回来 → 稳**





Catalog 这个概念是计算引擎侧的抽象（Flink / Spark / Hive）

Hudi 本身并不“定义 Catalog”，而是依赖计算引擎或 Metastore 来管理元数据。

### Flink 中的 Catalog
在 Flink 中：

> **Catalog = 表和数据库的命名空间 + 元数据访问入口**
>

+ Flink SQL 所有 `CREATE TABLE / SELECT`
+ 都必须通过 Catalog 去找表的定义

Flink 的 Catalog 决定了：

+ 表的 schema
+ 表的 location
+ 表的类型（Hive / Hudi / 普通表）

---

### Hive Metastore（不是 Catalog，但常被当成）
Hive Metastore 是：

> **一个元数据服务**
>

它负责存：database、table、partition、location

但它本身不关心：

+ 计算引擎是谁
+ 表是 Hudi / Iceberg / Parquet

---

### Hudi 在这里的位置（关键）
Hudi 的定位是：

> **一种表格式（table format）+ 存储管理层**
>

Hudi 负责：

+ 数据文件组织
+ 表版本
+ 增量语义
+ 更新 / 删除

❌ Hudi 不提供 Catalog 服务  
❌ Hudi 不管理 database / namespace

它只是：

> **把“这是一个 Hudi 表”这件事，登记到 Hive Metastore 中**
>

---

### “通过 Catalog 管理 Hudi 表”的含义
真正含义是：

> **通过 Flink 的 Catalog，对接 Hive Metastore，来访问和操作 Hudi 表的元数据。**
>

而不是：

+ Hudi 自己有 Catalog
+ 或 Hudi 管理 Catalog

---

### **“面试级标准总结”（建议背）**
> **Catalog 是计算引擎侧的元数据抽象，存在于 Flink、Spark、Hive 等引擎中；**
>
> **Hudi 作为表格式，本身不提供 Catalog，而是依赖计算引擎通过 Catalog 访问 Hive Metastore 中登记的表元数据。**
>
