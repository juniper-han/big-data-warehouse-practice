# Marker 机制
## 概述
Marker 机制（标记机制）：确保数据写入一致性和故障恢复的核心机制

Apache Hudi 在写入时，引入【标记机制】，来有效跟踪写入存储的<font style="background-color:#E8F7CF;">数据文件</font>。

## Marker 机制引入原因
分布式写入的“烂摊子”

+ 多个 Spark Executor 同时写入文件
+ 可能中途失败，留下“半成品”
+ 需要一种机制来清理这些“烂摊子”

具体挑战

+ 部分写入：Executor 写到一半挂了
+ 重复写入：Spark 重试导致相同数据写多次
+ 提交失败：整个作业在最后一步失败



Marker =“正在施工”的牌子，类比：建筑工地

+ 开工前：先立个牌子"正在施工"
+ 施工中：正常作业
+ 完工后：拆掉牌子
+ 如果中途停工：牌子还在，别人知道这里没完工

## Marker 机制介绍
Marker 文件结构

+ 文件名：`数据文件名 + .marker + 操作类型`
    - 示例：`91245ce3-bb82-0_20210820173605.parquet.marker.CREATE`
+ 组成：
    - 数据文件名：`91245ce3-bb82-0_20210820173605.parquet`
    - 扩展名：`.marker`（表明这是【标记文件】）
    - 操作类型：`CREATE`（创建新文件）
        * `CREATE`：插入新数据
        * `MERGE`：更新/删除数据（需要合并）
        * `APPEND`：追加数据



Marker 工作流程：

1. 正常流程（写入成功）
    1. 创建所有待写入文件的 Marker
    2. 写入数据文件
    3. 所有数据写入完成，提交元数据成功
    4. 删除 Marker
2. 异常流程（写入失败）
    1. 创建所有待写入文件的 Marker
    2. 写入数据文件
    3. 写入失败：在写入过程中发生故障（比如 Driver 崩溃）
    4. Marker 保留：因为提交没有成功，所以 Marker 不会被删除

关键点：

+ Marker 的创建时机：在数据文件写入之前
+ Marker 的删除时机：所有数据写入成功，<font style="background-color:#FBDE28;">提交元数据成功后，批量删除</font>
    - 注意：不是在数据文件写入完成后立即删除
+ 如果提交失败，Marker 会保留



## Marker 的作用
Marker 机制的价值

+ 原子性保证：要么全写，要么全不写
+ 一致性保证：查询不会看到部分写入的数据
+ 故障恢复：快速清理失败作业的残留
+ 资源节约：及时回收存储空间

### 删除重复/不完整文件
场景：多个进程做同样的事情，较慢的写不完，通过 Marker 发现不完整的文件并删除

### 处理回滚失败的提交
场景：

1. 作业开始写入：创建了 1000 个 Marker，开始写 1000 个数据文件，写完第 500 个时，进程崩溃
2. 写入结果：
    - 已写完的 500 个数据文件存在，对应的 Marker 也存在；
    - 还没写的 500 个数据文件不存在，只有对应的 Marker 存在



下次作业启动：

1. 发现`inflight`状态的提交，读取该提交的所有 Markers（1000个），对于每个 Marker：
    1. 如果对应的数据文件存在 → 删除数据文件（前500个）
    2. 删除 Marker 文件
2. 清理提交元数据



清理结果：

+ 清理了 500 个"孤儿"数据文件
+ 清理了 1000 个 Marker 文件
+ 表恢复到一致状态

## 基于 Timeline 服务器的标记机制提高写入性能
Marker 直接标记文件，会导致大量小文件：每个数据文件都需要一个 Marker 文件

解决方案：集中化管理

+ 分布式标记 -> 集中式标记管理，TimeServer 内存存储，减少文件系统操作

类比理解

```latex
旧方式：每家每户在自家门口贴告示（分散）
新方式：大家都把告示贴到小区的集中公告栏（集中）
好处：
- 管理员（Timeline Server）统一管理
- 告示（markers）整理得井井有条
- 更新公告栏时批量操作，效率高
```



基于 Timeline Server 的 Marker 文件存储：

```latex
/user/hudi/table/.hoodie/.temp/
├── markers/
│   ├── markers_20240101000000_000.meta  # 包含 Job1 所有的markers
│   ├── markers_20240101000000_001.meta  # 包含 Job2 所有的markers
│   └── current_markers.meta -> markers_20240101000000_001.meta  # 符号链接
```



TimeServer 类型的 Marker 的**生命周期时间线**

1. Timeline Server 启动：加载已存在的 Markers
2. 写入作业启动，开始新 Instant（`inflight instant`）：连接到 Timeline Server
3. 创建第一个 Marker（先检查 Marker 是否已存在）：Marker 被放入内存队列，客户端异步等待结果
4. 批量刷新线程，将 Markers 写入存储：每 20ms 或 1000 个 Markers 刷新一次
5. 数据文件开始写入：检查冲突（定期执行）
6. 所有数据写入完成：提交元数据到 Timeline
7. 提交成功，删除Markers：发送删除请求到Timeline Server，从内存和存储中移除



Timeline Server 故障恢复

1. 扫描存储中的 Marker 文件
2. 读取所有的 Marker 文件
3. 重建内存状态
4. 检查哪些`instant`需要回滚：获取所有有 Marker 但是提交未完成的`instant`

## FlinkSQL 中配置 Marker
Marker 的清理策略

```sql
CREATE TABLE hudi_table (...) WITH (
  -- Marker相关配置
  'write.marker.type' = 'direct',  -- 或 'timeline_server'
  
  -- 失败提交的清理策略
  'cleaner.policy.failed.writes' = 'LAZY',  -- 延迟清理
  'clean.async.enabled' = 'true',           -- 异步清理
  
  -- 对于重要作业，可以保留失败的markers用于调试
  'hoodie.keep.failed.commit.markers' = 'false'
);
```



各流独立的 Marker 命名空间

```sql
-- 每个作业使用不同的marker后缀
'hoodie.write.marker.suffix' = 'job1'  -- 作业1
'hoodie.write.marker.suffix' = 'job2'  -- 作业2

-- Marker文件名示例：
-- file1.parquet.marker_job1
-- file1.parquet.marker_job2
```



监控 Marker 的状态

```sql
-- 查询未清理的markers
SELECT instant_time, count(*) as marker_count
FROM hudi_markers
WHERE state = 'pending'
GROUP BY instant_time
HAVING count(*) > 0;

-- 监控marker与数据文件的不匹配
SELECT 
  instant_time,
  SUM(CASE WHEN data_file_exists THEN 1 ELSE 0 END) as files_with_data,
  SUM(CASE WHEN NOT data_file_exists THEN 1 ELSE 0 END) as markers_without_data
FROM hudi_marker_status
WHERE instant_state = 'inflight'
GROUP BY instant_time;
```

# 早期冲突检查
Hudi 的【早期冲突检测】机制（Early Conflict Detection），<font style="background-color:#E8F7CF;">是解决多写入者场景下资源浪费问题的关键技术</font>

## 概述
传统 OCC 的“事后诸葛亮”问题：提交时才检测冲突，如果失败，浪费大量资源

<font style="background-color:#E8F7CF;">早期冲突检测的核心思想：尽早发现冲突，尽早终止</font>

类比：

+ 传统方式：两个人同时装修同一间房子，都买好材料、开始施工了，最后才发现冲突
+ 早期检测：一个人先去房子门口挂个"正在装修"的牌子，另一个人看到牌子就不进去了

## Hudi 的两种冲突检测机制
### 基于直接标记的早期冲突检测：
适用于：直接标记机制，小规模并发

工作原理：

1. Flink Task 要写文件 A
2. 先检查文件 A 的 Marker 是否已存在
3. 如果存在 → 冲突！立即停止
4. 如果不存在 → 创建 Marker，开始写入

### 基于 Timeline Server 的早期冲突检测
适用于：Timeline Server 标记机制，大规模并发

工作原理：

1. Flink Task 向 Timeline Server 请求创建 Marker
2. Timeline Server 异步定期检查冲突
3. 发现冲突 → 通知相应 Task 中止
4. 注意：可能有短暂的时间窗口两个 Task 同时写

## Hudi 的支撑机制
1. **Marker 机制（标记机制）**
    1. 作用：跟踪每个写入作业正在写哪些文件
    2. 实现：每个文件对应一个 Marker，表示"我正在写这个文件"
2. **心跳机制**
    1. 作用：跟踪所有活跃的写入者
    2. 实现：每个写入作业定期向 Timeline 发送心跳
    3. 超时判断：如果心跳停止，认为作业已失败，清理相关 Markers

## 工作流程
Flink Task 开始写数据：

1. 检查是否启用 OCC 和早期冲突检测
2. 如果是 → 执行早期冲突检测
3. 检测到冲突 → 抛出异常，Task 失败
4. 未检测到冲突 → 创建marker，开始写入
5. 写入完成，提交元数据

## FlinkSQL 早期冲突检测配置
1. 启用

```sql
-- 在Flink SQL中创建Hudi表时配置
CREATE TABLE hudi_table (
  ...
) WITH (
  'connector' = 'hudi',
  -- 启用乐观并发控制
  'write.concurrency.mode' = 'optimistic_concurrency_control',
  -- 启用早期冲突检测
  'early.conflict.detection.enable' = 'true',
  -- 心跳间隔（秒）
  'heartbeat.interval' = '30',
  -- 心跳超时（秒）
  'heartbeat.timeout' = '120',
);
```

2. 根据标记类型选择配置：
    1. 直接标记机制

```sql
-- 适合小规模并发
'marker.type' = 'direct',
-- 冲突检查立即执行
'early.conflict.detection.strategy' = 'sync'
```

    2. Timeline Server 标记机制

```sql
-- 适合大规模并发
'marker.type' = 'timeline_server',
-- Timeline Server地址
'timeline.server.url' = 'host:port',
-- 异步冲突检查间隔
'conflict.check.interval' = '5000' -- 5秒
```
