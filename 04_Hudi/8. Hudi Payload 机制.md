# Payload 简介
Apache Hudi 的 Payload：是一种可扩展的【数据处理机制】

+ 可以在写入时自定义数据的<font style="background-color:#E8F7CF;">合并行为</font>
+ 通过不同的 Payload，可以实现复杂场景的【定制化数据写入方式】，大大增加了数据处理的灵活性。



PayLoad：在 Hudi 中实现为一个工具类

+ 用于：写入和读取 Hudi 表时，对数据进行去重、过滤、合并等操作
+ 通过参数`hoodie.datasource.write.payload.class`指定需要使用的 Payload Class

# 为何需要 Payload
传统的数据写入方式（整行插入，整行覆盖），无法满足全部的数据处理需求。

比如：

+ 场景：两个流更新同一张表
    - 流A：完整用户信息更新，包含：`user_id`, `name`, `age`, `address`, `update_time`
    - 流B：只更新用户年龄，包含：`user_id`, `age`, `update_time`
+ 问题：
    - 整行插入：如果用户不存在，没问题
    - 整行覆盖：如果用户已存在，流B会覆盖整行，导致`name`、`address`字段被置为`null`



Payload 解决的痛点

1. **部分字段更新**：只更新提供的字段，保留其他字段不变
2. **精确控制**：可以控制到字段级别的更新
3. **条件更新**：根据业务规则决定是否更新（如只保留更晚的数据）
4. **自定义去重**：根据多个字段或复杂逻辑去重
5. **数据清洗**：在写入时进行数据转换

# Payload 作用机制
## `HoodieRecordPayload` 接口
Hudi 所有 Payload 都实现`HoodieRecordPayload`接口，该接口需要实现以下两个重要方法

1. `preCombine`
2. `combineAndGetUpdateValue`



Payload 的工作方式

+ 将每条数据封装为一个`Payload`对象
+ 数据比较 → `Payload`对象比较
+ <font style="background-color:#E8F7CF;">通过实现</font>`<font style="background-color:#E8F7CF;">Payload</font>`<font style="background-color:#E8F7CF;">接口，自定义比较逻辑</font>

## Precombine Field（预合并字段）
写入 Hudi 表时，需要指定一个参数：`hoodie.datasource.write.precombine.field`

+ 这个字段也称为`Precombine Key`，预合并字段
+ 用于确定相同主键记录的去重顺序

Hudi Payload 就是根据这个指定的字段来处理数据：

+ 它将每条数据，都构建成一个 Payload
+ 因此数据间的比较就变成了 Payload 之间的比较。

只需要根据业务需求实现 Payload 的比较方法，即可实现对数据的处理。



预合并字段：必须是单调递增的字段，如时间戳、序列号

作用：当两条记录主键相同时，保留`update_time`更大的那条。

```plsql
-- 在FlinkSQL中创建表时指定
CREATE TABLE hudi_table (
  ...
  update_time TIMESTAMP(3),
  PRIMARY KEY (user_id) NOT ENFORCED
) WITH (
  'connector' = 'hudi',
  'precombine.field' = 'update_time'  -- 关键配置
);
```

## `**preCombine**`** 方法**
作用：数据写入时的去重逻辑。

机制：该方法比较当前数据和`oldValue`，然后返回一条记录。



场景举例：同一批数据中，有两条相同主键的记录

+ 示例：
    - 记录 1：`user_001, age=20, update_time=10:00`
    - 记录 2：`user_001, age=21, update_time=10:05`
+ `preCombine`会：
    - 比较两条记录的`update_time`
    - 保留`update_time`更大的记录（记录 2）



Flink Task 处理数据时调用`preCombine`方法

1. 接收到多条相同主键的数据
2. 调用`preCombine`进行去重
3. 只保留一条记录继续处理



对于 MOR 表的额外作用：读取 MOR 表时

+ Log 文件中可能有对同一记录的多次更新
+ 读取时调用`preCombine`合并这些更新
+ 返回最终的结果

## `combineAndGetUpdateValue`方法
该方法将`currentValue`（即现有`parquet`文件中的数据）与新数据进行对比，判断是否需要持久化新数据。

<font style="background-color:#E8F7CF;">作用：新数据与旧数据的合并逻辑</font>

由于 COW 表和 MOR 表的读写原理差异，该方法的调用在 COW 和 MOR 中也有所不同

+ 针对 <font style="background-color:#E8F7CF;">COW 表的写入阶段</font>：在 COW 写入时，会将新写入的数据与表中原有的`currentValue`进行比较，返回需要持久化的数据
+ 针对 <font style="background-color:#E8F7CF;">MOR 表的读取阶段</font>：在 MOR 读取时，会将经过`preCombine`处理的 Log 中的数据与 Parquet 文件中的数据进行比较，返回需要持久化的数据



<font style="background-color:#E8F7CF;">COW 表的</font>**<font style="background-color:#E8F7CF;">写入</font>**<font style="background-color:#E8F7CF;">流程：</font>

1. 读取现有`parquet`文件中的旧数据
2. 调用`combineAndGetUpdateValue(新数据, 旧数据)`
3. 返回合并后的数据
4. 写入新的`parquet`文件



<font style="background-color:#E8F7CF;">MOR 表的读取流程</font>

1. 读取 Base Parquet 文件中的旧数据
2. 读取 Log 文件中的增量更新
3. 调用`combineAndGetUpdateValue(Log 数据, Base 数据)`
4. 返回合并后的数据给查询

# Hudi 预置 Payload 实现
## `**OverwriteWithLatestAvroPayload**`**（默认）**
行为：总是用新数据覆盖整个记录

+ 保留`precombine.field`最大的记录，完全覆盖旧记录
+ `'hoodie.datasource.write.payload.class' = 'org.apache.hudi.common.model.OverwriteWithLatestAvroPayload'`

## `**DefaultHoodieRecordPayload**`
行为：比较`precombine.field`，只有新数据更新时才覆盖。

+ 检查新记录是否比旧记录新，只有更新时才覆盖
+ `'payload.class' = 'org.apache.hudi.common.model.DefaultHoodieRecordPayload'`

## `**PartialUpdateAvroPayload**`**（重要！）**
行为：部分字段更新，只更新非空字段

1. 新数据中为`null`的字段 → 保留旧值；
2. 新数据中不为`null`的字段 → 更新为新值

`'payload.class' = 'org.apache.hudi.common.model.PartialUpdateAvroPayload'`

## `**ExpressionPayload**`
  行为：通过表达式控制合并逻辑

+ 使用 SQL 表达式定义合并行为

```sql
'payload.class' = 'org.apache.hudi.common.model.ExpressionPayload'
'hoodie.datasource.write.payload.expression' = 'age > old.age ? current : old'
```

## `**EmptyHoodieRecordPayload**`
行为：标记记录为删除。

+ 用于删除记录
+ `'payload.class' = 'org.apache.hudi.common.model.EmptyHoodieRecordPayload'`

# FlinkSQL 中使用 Payload
```sql
CREATE TABLE user_profile (
  ...
  update_time TIMESTAMP(3),
  PRIMARY KEY (user_id) NOT ENFORCED
) WITH (
  'connector' = 'hudi',
  
  -- 基础配置
  'path' = 'hdfs:///user/hudi/user_profile',
  'table.type' = 'MERGE_ON_READ',  -- 或 COPY_ON_WRITE
  
  -- Payload相关配置
  'precombine.field' = 'update_time',
  'hoodie.datasource.write.payload.class' = 'org.apache.hudi.common.model.PartialUpdateAvroPayload',
  
  -- 其他配置
  'write.operation' = 'upsert',
  'write.insert.drop.duplicates' = 'true'
);
```

# 使用建议
1. 选择合适的预合并字段
    1. 预合并字段必须是：单调递增的字段，如时间戳、序列号
2. 避免复杂的 Payload 逻辑
    1. 简单的 Payload 逻辑，性能更好
    2. 复杂的业务逻辑尽量在 Flink 作业中处理
3. 统一所有流的配置：确保所有写入者使用相同的 Payload 逻辑
4. 合理设计数据结构：使用嵌套类型组织不同来源的数据
5. 监控数据质量：定期检查合并后的数据完整性
