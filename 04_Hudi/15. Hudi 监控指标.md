Hudi 通过<font style="background-color:#E8F7CF;"> Timeline（时间线）元数据、Metrics Reporter（指标上报） 和 日志 </font>暴露大量可观测性信息。

涵盖写入、读取、Compaction、Clean、索引、资源消耗等维度。

# 生产环境监控 Checklist
| 序号 | 监控项 | 告警阈值建议 |
| --- | --- | --- |
| 1 | Commit 耗时 | > 10 分钟（根据 SLA 调整） |
| 2 | Compaction backlog | > 50 个 pending |
| 3 | 写入失败记录数 | > 0 |
| 4 | 小文件比例（<64MB） | > 20% |
| 5 | 存储日增长率 | 异常突增（如 > 3x 平均） |
| 6 | Clean 失败 | 立即告警 |
| 7 | Index lookup 超时 | > 5s |


# 核心概念：Hudi Timeline 与 Action
Hudi 的所有操作都记录在 `.hoodie` 目录下的 Timeline 中，每种操作称为一个 Action：监控指标大多围绕这些 Action 展开。

| Action 类型 | 说明 |
| --- | --- |
| `commit`/ `deltacommit` | 数据写入（COW 表用 commit，MOR 表用 deltacommit） |
| `compaction` | MOR 表的合并操作（log + base file → 新 base file） |
| `clean` | 清理过期文件版本 |
| `rollback` | 回滚失败的写入 |
| `savepoint` | 手动创建快照 |


# 关键监控指标分类详解
## 写入（Write）相关指标
适用于 `commit`（COW）或 `deltacommit`（MOR）

| 指标名称（常见命名） | 含义 | 生产关注点 |
| --- | --- | --- |
| `totalRecordsWritten` | 本次写入的总记录数（含 insert + update） | 写入吞吐量是否正常 |
| `totalUpdateRecordsWritten` | 被更新的记录数 | 更新比例过高可能引发高写放大 |
| `totalInsertRecordsWritten` | 新插入的记录数 | 判断数据增长趋势 |
| `totalWriteBytes` | 实际写入存储的字节数（含重写文件） | 计算 Write Amplification |
| `numFilesUpdated`<br/>/`numFilesInserted` | 被更新/新增的数据文件数量 | 文件级操作粒度 |
| `commitTime`/ `durationMs` | 提交耗时（端到端） | 延迟是否超标（如 >5min） |
| `numWriteStatusFailures` | 写入失败的分区/文件数 | 非零即告警！ |


✅ 告警建议：

+ <font style="background-color:#E8F7CF;">单次 commit 耗时突增（可能因小文件合并、资源不足）</font>
+ `totalRecordsWritten = 0` 但任务成功（空跑？数据源问题？）
+ 写入失败率 > 0

## **Compaction（压缩）相关指标（MOR 表专属）**
| 指标 | 含义 | 关注点 |
| --- | --- | --- |
| `compactionRequested` | 待压缩的文件组数量 | 积压过多 → 读性能下降 |
| `compactionCompleted` | 已完成的 compaction 数量 | 是否及时消费 |
| `compactionDurationMs` | 单次 compaction 耗时 | 过长可能阻塞后续写入 |
| `bytesWrittenCompaction` | compaction 写入的字节数 | 反映写放大成本 |
| `compactionMemoryUsed` | 内存使用（若启用内存合并） | OOM 风险 |


✅ 最佳实践：

+ <font style="background-color:#E8F7CF;">监控 compaction backlog（待处理任务数），设置阈值告警（如 > 100）</font>
+ Compaction 应与 ingestion 并行，避免成为瓶颈

## Clean（清理）相关指标
| 指标 | 含义 |
| --- | --- |
| `filesDeleted` | 本次清理删除的文件数 |
| `bytesDeleted` | 删除的总字节数 |
| `cleanDurationMs` | 清理耗时 |
| `earliestCommitToRetain` | 保留的最早 commit（由 `hoodie.cleaner.commits.retained`控制） |


⚠️ 注意：

+ 若 `filesDeleted = 0` 持续多轮，可能配置错误（如保留 commit 数过大）
+ 清理失败会导致存储成本飙升

## **索引（Index）性能指标**
Hudi 使用索引来定位记录所属文件（用于 upsert）。

| 指标 | 含义 | 优化方向 |
| --- | --- | --- |
| `indexLookupTimeMs` | 索引查询耗时 | 越低越好（Bloom/Record Index 应 < 1s） |
| `indexUpdateTimeMs` | 索引更新耗时 | 影响写入延迟 |
| `numKeysLookedUp` | 查询的 key 数量 | 与输入记录数对比，判断效率 |
| `indexHitRatio` | 索引命中率 | 低命中率 → 扫描过多文件 → 性能差 |


💡 建议：

+ 对高频更新场景，优先使用 Record-level Index（Hudi 0.13+） 或 LSM Index
+ 避免使用 Simple Index（全表扫描）

## **文件系统与存储指标**
| 指标 | 说明 |
| --- | --- |
| 小文件数量 | 分区下 < 64MB 的文件数（可通过 Hudi CLI 或 Spark SQL 统计） |
| 平均文件大小 | 应接近 `hoodie.parquet.max.file.size`（如 128MB） |
| HDFS/S3 请求次数 | 高频小文件导致元数据压力（尤其 S3 的 LIST/GET 成本） |
| 存储增长率 | 每日新增存储量 vs 原始数据量（评估写放大） |


## 写放大 Write Amplification
用于衡量：【实际写入存储系统的数据量】与用户【逻辑上写入的数据量】之间的比率



定义：Write Amplification（WA） = 实际写入存储的总数据量 / 用户逻辑写入的数据量

+ 用户逻辑写入的数据量：指的是通过 Hudi 写入的新记录或更新记录的原始大小。
+ 实际写入存储的总数据量：包括：
    - 新写入的数据文件（Data Files）
    - 因为更新/合并而重写的旧数据文件
    - 写入的日志文件（Log Files，在 MOR 表中）
    - 元数据文件（占比小，可忽略）
+ WA > 5 在 COW 表中较常见，MOR 通常 < 2（不含 compaction）



为什么 Hudi 会有写放大？

Hudi 支持高效的 upsert（插入/更新） 操作，这需要在底层进行文件级别的合并（compaction）或重写（rewrite），从而带来额外的 I/O 开销：

+ COW 表 <font style="background-color:#E8F7CF;">→ 写放大可能很高，尤其在小批量高频更新场景。</font>
    - 每次更新会重写整个包含该记录的数据文件。
    - 如果一个 100MB 的文件中有 1 条记录被更新，整个 100MB 文件都要重写。
+ MOR 表：整体写放大通常低于 COW，但引入了读放大或 compaction 开销
    - 更新先写入日志文件（log file），读时或后台 compaction 时再与 base file 合并
    - 写入阶段写放大较低（只写 log），但 compaction 阶段会产生写放大

## 资源与运行时指标（通过 Spark/YARN/K8s 监控）
虽然非 Hudi 原生，但密切相关：

| 指标 | 重要性 |
| --- | --- |
| Executor 内存使用率 | Hudi 写入（尤其索引、合并）内存敏感 |
| GC 时间 | 长 GC 会导致 commit 超时 |
| Shuffle 量 | 大量 shuffle 可能因分区不合理 |
| Task 失败率 | 可能因文件系统权限、超时等 |


# 如何采集指标
## 方式 1：**Hudi 内置 Metrics（推荐）**
启用 Prometheus 或 JMX 上报：

```properties
# 启用 metrics
hoodie.metrics.on=true
hoodie.metrics.reporter.type=PROMETHEUS

# 或 JMX
hoodie.metrics.reporter.type=JMX
```

暴露的指标前缀如：

+ `HUDI_METRICS_COMMIT_DURATION`
+ `HUDI_METRICS_COMPACTION_PENDING_COUNT`
+ `HUDI_METRICS_WRITE_BYTES`

## 方式 2：**解析 **`**.hoodie**`** 元数据**
每个 commit 的 JSON 文件包含详细统计信息，可定期解析：

```bash
# 示例：查看最近 commit
cat .hoodie/202603191400.commit | jq '.fileIdToWriteStats'
```

## 方式 3：**Hudi CLI 或 Spark SQL 查询**
```sql
-- 查看 commits
SELECT * FROM hudi_table_name_ro LIMIT 10;

-- 统计小文件
SELECT partition_path, count(*) as small_files 
FROM (
  SELECT partition_path, file_size_in_bytes 
  FROM hudi_table_name_ro 
  WHERE file_size_in_bytes < 67108864
) 
GROUP BY partition_path;
```

## 方式 4：**日志监控**
+ 关键日志关键词：`ERROR`, `Rollback`, `Failed`, `Timeout`
+ 使用 ELK / Loki 收集 driver/executor 日志
