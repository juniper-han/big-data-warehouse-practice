# 时间轴（TimeLine）
## Timeline：Hudi 的"操作日志"
<img src="https://cdn.nlark.com/yuque/0/2026/png/54426354/1768053453663-01966169-6003-4cb3-ba87-72bef8d61a85.png" width="1080" title="" crop="0,0,1,1" id="ucd19c06f" class="ne-image">

Hudi 的核心是：维护【表】上【在不同的即时时间】（Instants，时刻）执行的所有【操作】的时间轴（TimeLine）。



将 Timeline 类比为：Hudi 表的操作历史记录本：

+ 每次对表做任何修改（新增、更新、删除等），Hudi 都会在这个记录本上新写一行记录
+ 每条记录就是一个Instant（即时点）
+ 每条记录包含：什么时间、做了什么操作、操作状态如何



在每一次对 Hudi 表的数据集操作时，都会在<font style="background-color:#E8F7CF;">该表的 Timeline</font> 上生成一个 Instant

从而可以实现：

+ 仅查询某个时间点之后成功提交的数据
+ 或是仅查询某个时间点之前的数据，有效避免了扫描更大时间范围的数据
+ 同时，可以高效地只查询更改前的文件（如在某个 Instant 提交了更改操作后，仅 Query 某个时间点之前的数据，则仍可以 Query 修改前的数据）

## Instant：每次操作的"快照"
一个 Instant 由以下三部分组成：

1. **Instant Action：在表上执行的操作类型**
    1. `COMMITS`：普通数据提交，一次`COMMIT`表示将一批数据<font style="background-color:#E8F7CF;">原子性地写入一个表</font>
    2. `CLEANS`：清理过期数据（清理旧版本）
        1. 删除旧的 Base File（只保留配置的版本数），不删除最新的 Base File
        2. 删除已被压缩的 Log Files，不删除最新 Base File 之后的 Log Files
    3. `DELTA_COMMIT`：增量提交（写增量日志，不生成新文件），将一批数据<font style="background-color:#E8F7CF;">原子性</font>地写入一个 MOR 表，其中部分或所有数据可以写入增量日志
    4. `COMPACTION`：仅针对 MOR 表，合并小文件（优化存储），在内部`COMPACTION`体现为 TimeLine 上的特殊提交。
        1. 合并 Base File 和 Log Files，生成新的 Base File
        2. 不删除旧的 Base File 以及 Log Files，但是在<font style="background-color:#E8F7CF;">元数据中记录哪些旧文件可以被清理</font>
        3. 该操作<font style="background-color:#E8F7CF;">不删除文件</font>，避免因为 Compaction 失败导致数据丢失
    5. `ROLLBACK`：回滚失败的操作
    6. `SAVEPOINT`：创建恢复点
2. **Instant Time**
    1. 通常是一个时间戳，按照动作开始时间的顺序，单调递增
    2. 时间戳格式：`<font style="background-color:#E8F7CF;">20190117010349</font>`<font style="background-color:#E8F7CF;">，年月日时分秒的数字</font>，非 UNIX 时间戳
3. **State**
    1. `REQUESTED`：表示当前 Action 已经调度，但尚未执行
    2. `INFLIGHT`：表示当前 Action 正在执行
    3. `COMPLETED`：表示当前 Action 已经完成



Timeline 是 Hudi <font style="background-color:#E8F7CF;">用来管理提交（Commit）的抽象</font>

+ 每个 Commit 都绑定一个固定时间戳，分散到时间线上
+ 在 Timeline 上，每个 Commit 被抽象为一个 HoodieInstant
+ 一个 Instant 记录了一次提交（Commit）的行为、时间戳和状态。

## 关键能力：时间旅行（Time Travel）
这是 Timeline 带来的好处

场景示例：假设现在是10:30，你的Timeline上有：

+ 10:00：新增一批数据 ✓
+ 10:05：更新部分数据 ✓
+ 10:10：删除某些数据 ✓
+ 10:15：又有新增 ✓

用户可以：

+ 查询 10:07 的状态：看到 10:00 和 10:05 的操作结果，但看不到 10:10 及之后的
+ 查询 10:12 的状态：看到 10:00、10:05、10:10 的操作结果
+ 查询最新状态：看到所有操作的结果

这样避免了全表扫描，只需要读取相关时间点的文件即可。

## 两个时间概念
这是流处理中的核心概念：

+ 到达时间（Arrival Time）：数据到达 Hudi 系统的时间
+ 事件时间（Event Time）：Record 中记录的时间，即数据实际发生的时间

示例：一条用户点击记录：

+ 实际点击时间：9:00（事件时间）
+ 由于网络延迟，10:20 才到 Hudi 系统（到达时间）
+ **Hudi 会根据 9:00（事件事件） 把它放入 9:00 对应的分区**
+ **但在 Timeline 上记录的是 10:20 这个到达时间**

<img src="https://cdn.nlark.com/yuque/0/2026/png/54426354/1768053437019-1a375e47-03d7-44bc-ac54-3b26760cacaf.png" width="1280" title="" crop="0,0,1,1" id="oBcDx" class="ne-image">

## 表服务
数据湖表通常需要运行一些公共服务来维护表的性能和管理存储，这些在 Hudi 中被称为表服务（Table Services），表服务是幂等的（可重复执行，结果一致）

+ `CLEANS`：清理旧版本数据（类似垃圾回收）
+ `COMPACTION`：合并增量日志到基础文件
+ `CLUSTERING`：重组文件布局优化查询性能

## 总结
Hudi 的 Timeline 就像是一个带时间戳的版本控制系统（类似 Git 的 commit 历史），用户可以：

+ 查看任意时间点的数据快照，追溯任何时间点的数据状态
+ 只获取某个时间点后的变化
+ 根据事件时间，正确处理迟到数据

这样设计就是为了解决流处理中常见的**<font style="background-color:#E8F7CF;">数据延迟</font>**和**<font style="background-color:#E8F7CF;">增量处理</font>**问题。

# 文件布局（File Layout）
## Hudi 表的映射结构
Hudi 将一个表，映射为如下文件结构

<img src="https://cdn.nlark.com/yuque/0/2026/png/54426354/1768053428875-06b83b79-fe71-4939-97aa-82f368c89e98.png" width="864" title="" crop="0,0,1,1" id="ua9ea36aa" class="ne-image">

Hudi 存储分为两个部分

1. **元数据：**`.hoodie目录`
    1. 包括表的版本管理（Timeline）、归档目录（存放过期的 Instant，也就是版本）
    2. 一个 Instant 记录了一次提交（Commit）的行为、时间戳和状态（文件名也由这三者拼接而成）
    3. <font style="background-color:#E8F7CF;">Hudi 以时间轴的形式维护了在数据集上执行的所有操作的元数据</font>；
2. **数据：**
    1. 和 Hive一样，以分区方式存放数据；
    2. 分区里面存放着<font style="background-color:#E8F7CF;"> Base File（</font>`<font style="background-color:#E8F7CF;">.parquet</font>`<font style="background-color:#E8F7CF;">）和 Log File（</font>`<font style="background-color:#E8F7CF;">.log.*</font>`<font style="background-color:#E8F7CF;">）</font>；
    3. 一个分区是一个目录

## 数据文件组织
<img src="https://cdn.nlark.com/yuque/0/2026/png/54426354/1768053418537-fcce5122-7ca7-4551-8bb3-f8586d68844d.png" width="1080" title="" crop="0,0,1,1" id="u69bf2252" class="ne-image">

数据部分的文件组织方式：

1. Hudi 将数据表，组织成，分布式文件系统基本路径（basepath）下的，目录结构
    1. 即：<font style="background-color:#E8F7CF;">表体现为目录</font>
2. 表被划分为多个分区（Partition），这些分区目录下是文件组（FileGroups）
    1. 即：<font style="background-color:#E8F7CF;">分区也是目录</font>
3. 在每个分区中，数据文件被组织成文件组（FileGroup），由文件 ID 唯一标识
4. 每个文件组，包含多个文件片（FileSlice），<font style="background-color:#E8F7CF;">每个文件片代表一个版本，</font>每个文件片包含 1个基础文件（Base File，`.parquet`）和若干个增量日志文件（Log File，`.log.*`）

## 文件片与版本控制
<img src="https://cdn.nlark.com/yuque/0/2026/png/54426354/1768053849770-4d4aa922-a4c4-4d2f-a946-c1203289fd4e.png" width="556" title="" crop="0,0,1,1" id="u320da79d" class="ne-image">

如上图所示，每个文件片（FileSlice）包含：

1. 一个基本文件：Base File，`.parquet`
    1. 在某个`commit/compaction`时间点，生成的文件；
    2. 注意：MOR 可能没有
2. 多个日志文件：Log File，`.log.*`
    1. 这些日志文件包含自生成基本文件以来，对基本文件的插入/更新；
    2. 注意：COW 没有



基本文件（Base File，`.parquet` 文件）

+ 在 Footer 的 Meta 记录了 <font style="background-color:#E8F7CF;">Record Key （主键，Hudi 特色主键名称）</font>组成的 BloomFilter
+ 用于在 File Based Index 的实现中，实现高效率的 Key Contains 检测
+ 只有不在 BloomFilter 的 Key 才需要扫描整个文件消灭假阳

日志文件（Log，Avro 文件）

+ 是自己编码的，通过积攒数据 buffer，以 LogBlock 为单位写出
+ 每个 LogBlock 包含 magic number、size、content、footer 等信息，用于数据读、校验和过滤

Hudi 采用了<font style="background-color:#E8F7CF;">多版本并发控制（Multiversion Concurrency Control, MVCC）</font>

1. `Compaction`操作：合并日志和基本文件，以产生新的文件片
2. `Clean`操作：清除不使用的/旧的文件片，以回收文件系统上的空间

# 索引（Index）
索引的需求，来源于 Hudi 的主键更新（Upsert）。

当一条新数据到来时（`INSERT`或`UPDATE`），Hudi 需要快速判断：

+ 这条数据的主键，是否存在？如果存在，具体在哪个文件、哪个位置？

索引：即“<font style="background-color:#E8F7CF;">主键 -> 文件位置”的映射关系</font>



对比 Hive

+ Hudi 表建表时，建议使用索引
+ Hive 表：索引比较鸡肋，后来不支持索引了

## 原理
索引机制：通过【主键】快速定位【数据文件的位置】。

Hudi 通过`Hoodie Key`实现索引机制：

+ `<font style="background-color:#E8F7CF;">Hoodie Key = record key + partition path</font>`
+ 即：<font style="background-color:#E8F7CF;">Hoodie Key = 主键 + 分区路径</font>
+ 分区路径下包含多个文件组
+ 这种映射关系，数据第一次写入文件后保持不变，固定且唯一

<font style="background-color:#E8F7CF;">一个 FileGroup 包含了</font>**<font style="background-color:#E8F7CF;">这个分区</font>**<font style="background-color:#E8F7CF;">的 Record Key 的</font>**<font style="background-color:#E8F7CF;">所有版本记录</font>**

+ 即：具有相同主键的数据，一定会被分配到同一个 文件组里（FileGroup）
+ 有了索引之后，更新的数据，可以快速被定位到对应的 File Group

Index 用于区分消息是 `INSERT` 还是 `UPDATE`

<img src="https://cdn.nlark.com/yuque/0/2026/png/54426354/1768054085480-ba8b5c3f-2525-47bd-a108-e909b9577a47.png" width="556" title="" crop="0,0,1,1" id="u0f22a989" class="ne-image">

上图为例：

+ 白色是基本文件，黄色是更新数据
+ 有了索引机制，可以做到：避免读取不需要的文件、避免更新不必要的文件、无需将更新数据与历史数据做分布式关联，只需要在 File Group 内做合并

## 索引类型和原理
### Spark 支持的索引
1. Bloom Index：布隆过滤器
    1. 原理：默认配置，使用布隆过滤器来判断记录存在与否，也可使用 record key，范围裁剪需要的文件
    2. 优点：效率高，不依赖外部系统，数据和索引保持一致性，可以做范围裁剪
    3. 缺点：因假阳性问题，通过过滤器检验的 key 还需回溯原文件再查找一遍
2. Simple Index
    1. 原理：把`update`/`delete`操作的新数据和老数据进行`join`
    2. 优点：实现最简单，无需额外的资源
    3. 缺点：性能比较差
3. HBase Index
    1. 原理：把 Index 存放在 HBase。在插入 File Group 的定位阶段，所有 Task 向 HBase 发送 Batch Get 请求，获取 Record Key 的 Mapping 信息
    2. 优点：对于小批次的 keys，查询效率高，如果索引占据空间大，则适合放在外部系统
    3. 缺点：需要外部的系统，增加了运维压力

### Flink 可配置的索引
1. <font style="background-color:#FBDE28;">Flink State-based Index：一种全局、内存（状态）后端的索引</font>
    1. 原理：利用 Flink State 机制，将整个表的“主键 → 分区路径 + 文件ID + 记录位置”映射关系完整地保存在 TaskManager 的内存中（或如果配置了 RocksDB 状态后端，则大部分在磁盘）
    2. 工作流程
        1. 初始化：任务启动时，如果开启了 `'index.bootstrap.enabled' = 'true'`，它会扫描 Hudi 表的所有历史数据文件，将每一条记录的主键和位置信息加载到 Flink State 中。这个过程非常耗时，但是一次性的
        2. 查询：当新数据进入时，直接通过内存/状态快速查询主键，立刻知道目标记录在哪（或不存在）。
        3. 更新：随着数据写入，State会同步更新。
    3. 优点：索引是全局的，可以跨分区高效定位记录。内存/状态查询速度极快。
    4. 缺点：Flink 大状态问题，内存消耗巨大，初始化慢，状态管理复杂
    5. 使用场景：主键更新频繁且范围不确定（跨分区），同时数据量不大的场景；
        1. 对于超大表，资源成本很高。
2. <font style="background-color:#FBDE28;">Bucket Index：一种基于哈希分桶、无外部状态、存储绑定的索引。</font>
    1. 原理：预先定义好一个固定的桶（文件）数量 N (如 `hoodie.bucket.index.num.buckets = 256)`
        1. 定位：对于任意一条记录，根据其主键通过一个确定的哈希函数计算出它属于哪个桶。
        2. 文件绑定：<font style="background-color:#E8F7CF;">在同一个分区内</font>，每个 `bucket_id` 会对应到一个（或少数几个）固定的物理文件。
            1. 例如，`<font style="background-color:rgb(235, 238, 242);">partition='2024-05-01'</font>`下的`<font style="background-color:rgb(235, 238, 242);">bucket_001</font>`永远写入文件`<font style="background-color:rgb(235, 238, 242);">file_001.parquet</font>`。
    2. 工作流程：
        1. 写入：对于新数据，直接通过主键哈希算出 `bucket_id`，然后只向该分区下对应的那个文件进行查询和写入（插入或更新）。它不需要在内存中维护全局映射。
        2. 读取：查找时，同样算法定位到具体文件，然后在该文件内搜索（可能还需配合 Bloom Filter 等文件内索引）。
    3. 优点：零内存开销，写入路径稳定，初始化快（无需加载历史状态）
    4. 缺点：无法跨分区（Bucket Index 是分区内的），数据倾斜，CPU 开销（每次都要计算 Hash），不灵活（桶的数量一旦设定，几乎不能更改）
        1. 如果一条记录的主键对应的数据从一个分区移动到另一个分区（比如更新的日期字段变了），BUCKET 索引无法处理这种跨分区更新，会将其视为一条新记录插入，导致重复。
    5. 适用场景：数据量巨大、主键稳定（无跨分区更新）、追求作业稳定性和资源可控性的生产环境。

<font style="background-color:#FBDE28;">生产环境建议使用 BUCKET 索引。</font>



Flink State 索引和 Bucket 索引对比

| 特性 | FLINK_STATE 索引 | BUCKET 索引 |
| --- | --- | --- |
| 存储位置 | Flink State（内存/磁盘） | 文件系统（与数据共存） |
| 资源占用 | 高内存/状态消耗 | **几乎零内存**，占用磁盘IO和CPU |
| 索引范围 | **全局，可跨分区** | **分区内** |
| 初始化速度 | 慢（需加载全量历史） | 快（无需加载） |
| 小文件问题 | 可能产生（依据写入策略） | **天然避免**（文件数固定） |
| 扩展性 | 受限于 State 大小 | 好，资源消耗恒定 |
| 适用数据量 | 中小型 | **大型、超大型** |


## 全局索引与非全局索引
+ **全局索引**：Global Index
    - 在全表的所有分区范围下，强制要求键的唯一性
    - 全局索引提供了更强的保证
    - 但是随着表增大，`update/delete` 操作损失的性能越高，索引的维护难度也更大，因此更适用于小表。
+ **非全局索引**：**默认**的索引实现，Non-Global Index
    - 只能保证数据在分区的唯一性
    - 非全局索引依靠写入器，为同一个记录的`update/delete`提供一致的分区路径
    - 大幅提高了写入效率，降低索引的维护难度，更适用于大表。



HBase 索引本质上是一个全局索引，bloom 和 simple index 都有全局选项：

+ `hoodie.index.type=GLOBAL_BLOOM`
+ `hoodie.index.type=GLOBAL_SIMPLE`

## 索引的选择策略
<img src="https://cdn.nlark.com/yuque/0/2026/png/54426354/1768054987222-e9d83a43-1273-4dae-a414-045a87518307.png" width="1280" title="" crop="0,0,1,1" id="u8a2d78e5" class="ne-image">

**场景 1**：如上图

+ 描述：表一直在增长，大部分的更新随机发生在较新的几个分区上，小部分会发生在旧分区
+ 选择：Bloom Index
+ 原因：
    - 查询索引可以靠设置得当的布隆过滤器，来裁剪很多数据文件
    - 如果生成的键可以以某种顺序排列，参与比较的文件数会进一步通过范围裁剪而减少
    - Hudi 用所有文件的键域来构造区间树，这样能来高效地依据输入的更删记录的键域来排除不匹配的文件
    - 为了高效地把记录键和布隆过滤器进行比对，即尽量减少过滤器的读取和均衡执行器间的工作量，Hudi缓存了输入记录并使用了自定义分区器和统计规律来解决数据的偏斜
    - 如果布隆过滤器的假阳性率过高，查询会增加数据的打乱操作
    - Hudi 支持动态布隆过滤器（设置`hoodie.bloom.index.filter.type=DYNAMIC_V0`）。它可以根据文件里存放的记录数量来调整大小从而达到设定的假阳性率。

场景 2：如上图

+ 描述：事件表去重
+ 选择：Bloom Index，可范围裁剪
+ 原因：
    - 利用作为首类处理对象的时间，来构造由事件时间戳和事件 ID（`event_ts+event_id`）组成的键，这样插入的记录就有了单调增长的键。这会在最新的几个分区里大幅提高裁剪文件的效益
+ 不能使用 HBase 索引：虽然可以用一个键值存储来实现去重，但索引存储的消耗会随着事件数增长而线性增长以至于变得不可行

+ 如果范围比较不能裁剪许多文件的话，那么布隆索引并不能带来很好的效益。
+ 在这样一个随机写入的作业场景下，更新操作通常会触及表里大多数文件，从而导致布隆过滤器依据输入的更新对所有文件标明阳性。
+ 最终会导致，即使采用了范围比较，也还是检查了所有文件。
+ 使用简单索引对此场景更合适，因为它不采用提前的裁剪操作，而是直接和所有文件的所需字段连接。
+ 如果额外的运维成本可以接受的话，也可以采用 HBase 索引，其对这些表能提供更加优越的查询效率。
+ 当使用全局索引时，也可以考虑通过设置`hoodie.bloom.index.update.partition.path=true`或`hoodie.simple.index.update.partition.path=true`来处理的情况；
+ 例如对于以所在城市分区的用户表，会有用户迁至另一座城市的情况。这些表也非常适合采用Merge-On-Read表型

# 表类型（Table Types）
可以平衡读和写

## Copy On Write（COW 表）
COW 表

+ 对每一个新批次写入，都将创建相应数据文件的新版本（新的 FileSlice），新版本文件（全量最新）包括：
    - 旧版本文件的记录
    - 和，来自传入批次的记录
+ 在 COW 表中，只有数据文件/基本文件（`.parquet`）,没有增量日志文件（`.log.*`）
+ Copy 的含义：Copy 老文件，将写入的新数据与老文件合并，生成最新的文件

我们进行一批新的写入

通过索引，这些记录与 File group 1、File group 2 匹配，然后有新的插入，我们将为其创建一个新的文件组（File group 4）

因此 data_file1 和 data_file2 都将创建更新的版本，data_file1 V2 是 data_file1 V1 的内容与 data_file1 中传入批次匹配记录的记录合并。



注意：

+ 由于在写入期间进行合并，COW 会产生一些<font style="background-color:#FBDE28;">写入延迟</font>。
+ 但是 COW  的优势在于它的简单性，不需要其他表服务（如 Compaction），也相对容易调试

## Merge On Read（MOR 表）
MOR 表中，包含

+ <font style="background-color:#FBDE28;">列存</font>的基本文件（`.parquet`）
+ <font style="background-color:#FBDE28;">行存</font>的增量日志文件（基于行的 avro 格式，`.log.*`）。



MOR 表的<font style="background-color:#FBDE28;">合并成本在读取端</font>

+ 在写入期间，MOR 表不会合并或创建较新的数据文件版本
+ 标记/索引完成后，对于有更新记录的数据文件，Hudi 会创建`.log.*`增量日志文件，并根据规则命名，以便【基本文件 + 增量日志文件】都属于一个文件组



<img src="https://cdn.nlark.com/yuque/0/2026/png/54426354/1768094723685-b3e1ca51-4f44-4c07-84a2-869638d8edcd.png" width="439" title="" crop="0,0,1,1" id="u8610ff9c" class="ne-image">

+ 读取端：实时合并【基本文件】及其各自的【增量日志文件】
+ 每次的读取延迟都比较高



所以 Hudi 使用压缩机制（Compaction），将数据文件和日志文件合并在一起，并创建新版本的数据文件。

+ <font style="background-color:#E8F7CF;">Compaction 完成后，读取端只需要读取最新的数据文件，而不必关心旧版本文件。</font>

<img src="https://cdn.nlark.com/yuque/0/2026/png/54426354/1768094791139-e6acccc2-640b-4a4e-b94e-e7ecfea1dd00.png" width="1080" title="" crop="0,0,1,1" id="u2fb14746" class="ne-image">

用户可以选择<font style="background-color:#E8F7CF;">内联或异步模式</font>进行 Compaction。

Hudi 提供了不同的 Compaction 策略供用户选择，最常用的策略：<font style="background-color:#E8F7CF;">基于提交的数量</font>

+ 例如：将 Compaction 最大增量日志配置为 4，这意味着在进行 4 次增量写入后，将对数据文件进行 Compaction 并创建更新版本的数据文件。



MOR 表的写入行为，依据 Index 类型的不同会有细微的差别：

+ 无法对 Log File 生成 Index 的索引类型：如，Bloom Index： 
    - `INSERT` 消息写 Base file
    - `UPDATE` 消息会 Append Log File
        * 因为 Base File 已经记录了该 `UPDATE` 消息的 FileGroup ID
+ 可以对 Log File 生成 Index 的索引类型：如 Flink Writer 中基于 State 的索引
    - 每次写入都是 Log Format，并且会不断追加和 Roll Over

## COW 与 MOR 的对比
|  | **COW** | **MOR** |
| --- | --- | --- |
| 写入延迟 | 高 | 低 |
| 查询延迟 | 低 | 高 |
| Update(I/O) 更新成本 | 高（重写整个 Parquet 文件） | 低（追加到增量日志） |
| Parquet 文件大小 | 低（更新成本 I/O 高） | 较大（低更新成本） |
| 写放大 | 大 | 低（取决于 Compaction 策略） |
| 适用场景 | 批 | 流，批也可以 |


# 查询类型（Query Types）
Hudi 支持以下三种查询类型

+ Snapshot Queries：快照查询
+ Incremental Queries：增量查询
+ Read Optimized Queries：读优化查询

## 快照查询 Snapshot Queris
快照查询：查询指定的`commit/delta commit`操作后，表的最新快照

+ MOR 表：通过<font style="background-color:#E8F7CF;">即时合并最新 FileSlice 中的 Base File 和 Log File，提供</font>**<font style="background-color:#E8F7CF;">近实时表</font>**<font style="background-color:#E8F7CF;">（几分钟）</font>
+ COM 表：替代现有的 Parquet 表（或相同基本文件类型的表），同时提供`upsert/delete`和其他写入方面的功能
    - 可以理解为：查询最新版本的 Base File（即 Parquet 数据文件）

## 增量查询 Incremental Queries
增量查询：查询给定`commit/delta commit`操作以来的【新写入的数据】

+ 有效的提供变更流来启用增量数据管道。



假设用户上次查询是 10:00，现在想获取 10:00 之后的新数据

Hudi 不扫描全表，而是：

+ 查看 Timeline 上 10:00 之后的`commit`记录
+ 只读取这些`commit`对应的新增/修改文件
+ 返回给用户

## 读优化查询 Read Optimized Queries
读优化查询：查询给定的`commit/<font style="background-color:#FBDE28;">compact</font>`操作的表的最新快照，仅针对 MOR 表

+ 仅<font style="background-color:#E8F7CF;">将最新文件片的基本文件（Base File，列存）暴露给查询</font>，并保证与非 Hudi 表相同的列查询性能。
+ 注意：快照查询查的是`commit/delta commit`操作后
    - Read Optimized Queries 是<font style="background-color:#E8F7CF;">对 MOR 表 Snapshot Queries 的优化</font>



MOR 表，快照查询和读优化查询的对比

|  | **Snapshot** | **Read Optimized** |
| --- | --- | --- |
| 写入延迟 | 低 | 高 |
| 查询延迟 | 高（合并 Base File + Log File） | 低（仅最新 Base File） |
