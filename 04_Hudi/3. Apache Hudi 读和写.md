# 数据写
## 写操作
写操作包含以下 3 个类型

1. `UPSERT`：默认行为
    1. 数据先通过 Index 打标，确认是`INSERT`还是`UPDATE`
    2. 通过启发式算法，决定消息的组织方式，以优化文件的大小
    3. 然后  CDC 导入
2. `INSERT`：跳过 Index，写入效率更高 => Log Deduplication
3. `BULK_INSERT`：写排序，对大数据量的 Hudi 表初始化友好，对文件大小的限制 best effort（写 HFile）

## `UPSERT` 写流程
### COW 表
1. 去重：根据`Key`，对 Records 去重
2. 创建索引：
    1. 对这批数据创建索引：`HoodieKey => HoodieRecordLocation`
    2. 通过索引区分哪些 Records 是 `Update`，哪些 Records 是 `Insert`
3. 对`Update`的 Records：直接找到对应`Key`所在的最新 FileSlice 中的 Base File，做 Merge 后，写新的 Base FIle（即：新的 FileSlice）
4. 对`Insert`的 Records：扫描当前 Partition 所有的 SmallFile（小于一定大小的 Base File），然后 Merge，写新的 FileSlice；如果没有 SmallFile，直接写新的 FileGroup + FileSlice

### MOR 表
1. 去重（可选）：根据`Key`，对 Records 去重
2. 创建索引：
    1. 对这批数据创建索引：`HoodieKey => HoodieRecordLocation`
    2. 通过索引区分哪些 Records 是 `Update`，哪些 Records 是 `Insert`
3. 对`Update`的 Records：，写对应的 File Group + FileSlice，直接 Append 最新的 Log File
    1. 如果碰巧是当前最小的小文件，会 Merge Base File，生成新的 FileSlice
4. 对`Insert`的 Records：
    1. 如果 Log File 不可建索引（默认）：
        1. 尝试 Merge 分区内最小的 Base File （即：不包含 Log File 的 FileSlice），生成新的 FileSlice；
        2. 如果没有 Base File，就新写一个 FileGroup + FileSlice + Base File；
    2. 如果  Log File 可建索引：
        1. 尝试 Append 小的 Log File
        2. 如果没有就新写一个  FileGroup + FileSlice + Base file
5. Log File 大小达到阈值，会 Roll Over 一个新的

## `INSERT` 写流程
### COW 表
1. 去重（可选）：根据`Key`，对 Records 去重
2. 不会创建 Index
3. 如果有小的 Base File，Merge Base File，生成新的 FileSlice + Base File
4. 如果没有小的 Base File：直接写新的 FileSlice + Base File

### MOR 表
1. 去重（可选）：根据`Key`，对 Records 去重
2. 不会创建 Index
3. 如果 Log File 可索引，且有小的 FileSlice：尝试追加或写最新的 Log File；
4. 如果 Log File 不可索引，写一个新的 FileSlice + Base File

## `INSERT OVERWRITE` 写流程
全量覆盖

适用于首次导入全量数据场景，会覆盖历史数据，相对 Insert Into 写入性能更高（BulkInsert）。

注：对于主键表，引擎不会做去重，需要由业务侧增加去重逻辑。



`INSERT OVERWRITE`：

1. 在同一分区中，创建新的 File Group
2. <font style="background-color:#E8F7CF;">现有的 File Group，被标记为 "删除"。</font>
3. 根据新记录的数量，创建新的 File Group



注意：操作的对象是 File Group（文件组）



**COW 表**`**INSERT OVERWRITE**`** 写流程**

1. 在插入分区之前：分区包含：
    - `file1-t0.parquet`
    - `file2-t0.parquet`
2. 插入相同数量的记录覆盖
    - 分区将添加：`file3-t1.parquet`，`file4-t1.parquet`
    - `file1`, `file2`，在 t1 后的元数据中被标记为无效
3. 插入覆盖更多的记录
    - 分区将添加：`file3-t1.parquet``file4-t1.parquet``file5-t1.parquet`..`fileN-t1.parquet`
    - `file1`, `file2`在 t1 后的元数据中被标记为无效
4. 插入重写 1 条记录
    - 分区将添加`file3-t1.parquet`
    - `file1`, `file2`在 t1 后的元数据中被标记为无效



**MOR 表**`**INSERT OVERWRITE**`** 写流程**

1. 在插入分区之前：分区包含：
    - `file1-t0.parquet`
    - `file2-t0.parquet`
    - `.file1-t00.log`
2. 插入相同数量的记录覆盖
    - 分区将添加：`file3-t1.parquet`，`file4-t1.parquet`
    - `file1`, `file2`，在 t1 后的元数据中被标记为无效
3. 插入覆盖更多的记录
    - 分区将添加：`file3-t1.parquet``file4-t1.parquet``file5-t1.parquet`..`fileN-t1.parquet`
    - `file1`, `file2`在 t1 后的元数据中被标记为无效
4. 插入重写 1 条记录
    - 分区将添加`file3-t1.parquet`
    - `file1`, `file2`在 t1 后的元数据中被标记为无效



`**INSERT OVERWRITE**`**优点**

1. COW 和 MOR 在执行方面非常相似，不干扰 MOR 的 Compaction。
2. 减少`.parquet`文件大小，因为重新规划
3. 不需要更新关键路径中的外部索引
    - 索引可以检查：文件组是否无效（类似于在 HBase Index 中检查 Commit 是否无效的方式）。
4. 可以扩展清理策略，在一定的时间窗口后删除旧文件组



`**INSERT OVERWRITE**`**缺点**

1. 需要转发旧的元数据
    - 在 t1，比如`file1`被标记为无效，我们在`t1.commit`中存储 `invalidFiles=file1`，或者在 MOR 中存储`deltacommit`
    - 在 t2，比如`file2`也被标记为无效。我们转发之前的文件，并在`t2.commit`中标记`invalidFiles=file1, file2`（或 MOR 的`deltacommit`）
2. 忽略老数据：忽略磁盘中存在的`.parquet`文件

## HoodieKey 生成策略
用来生成 HoodieKey（record key + partition path）的生成策略

+ Record Key：记录键（即主键，Hudi 对主键的独特称呼）怎么选择
+ Partition Path：分区路径怎么写

目前支持以下策略：

+ 支持多个字段组合 Record Keys
+ 支持多个字段组合的 Parition Path （可定制时间格式，Hive Style Path Name）
+ 非分区表

## 删除策略
### 逻辑删
将 Value 字段全部标记为 `null`

### 物理删
1. 通过 `OPERATION_OPT_KEY ` 删除所有的输入记录
2. 配置 `PAYLOAD_CLASS_OPT_KEY = org.apache.hudi.EmptyHoodieRecordPayload` 删除所有的输入记录
3. 在输入记录添加字段：`_hoodie_is_deleted`

## 总结
通过对写流程的梳理可以了解到 Apache Hudi 相对于其他数据湖方案的核心优势：

+ <font style="background-color:#E8F7CF;">写入过程充分优化了文件存储的小文件问题</font>
    - COW 写，会一直将一个 bucket （FileGroup）的 Base 文件<font style="background-color:#E8F7CF;">写到设定的阈值大小，才会划分新的</font> Bucket；
    - MOR 写在同一个 Bucket 中，Log File 也是一直 Append 直到大小超过设定的阈值，才会 Roll Over。
+ 核心优势：对 `UPDATE` 和 `DELETE` 的支持非常高效
    - 一条 Record 的整个生命周期操作都发生在同一个 Bucket，不仅减少小文件数量，也提升了数据读取的效率（不必要的 `join` 和 `merge`）。

# 数据读
## Snapshot 读
读取所有 Partiiton 下，每个 FileGroup 中最新的 FileSlice 中的文件

+ COW 表读 parquet 文件
+ MOR 表读 parquet + log 文件

## Incremental 读
当前的 Spark data source 可以指定消费的起始和结束 commit 时间，读取 commit 增量的数据集。

但是内部的实现不够高效：拉取每个 commit 的全部目标文件再按照系统字段 `_hoodie_commit_time_apply` 过滤条件。

## Streaming 读
0.8.0 版本的 HUDI Flink Writer 支持实时的增量订阅，可用于同步 CDC 数据，日常的数据同步 ETL Pipeline。

Flink 的 Streaming 读做到了真正的流式读取，Source 定期监控新增的改动文件，将读取任务下派给读 Task。

# Compaction：仅针对 MOR 表
1. 没有 Base File：走 COW `Insert` 流程，直接 Merge 所有的 Log file 并写 Base file
2. 有 Base File：走 COW `Upsert` 流程，先读 Log File 建 Index，再读 Base File，最后读 Log File 写新的 Base File



Flink 和 Spark Streaming 的 Writer 都可以 Apply 异步的 Compaction 策略，按照间隔 Commits 数或者时间来触发 Compaction 任务，在独立的 Pipeline 中执行。

# Hudi 与 Flink
## FlinkSQL 结合 Hudi 的写入流程
<font style="background-color:#E8F7CF;">Flink-Hudi 的写入流程是：</font>

1. <font style="background-color:#E8F7CF;">数据</font>**<font style="background-color:#E8F7CF;">先写入内存缓冲区</font>**<font style="background-color:#E8F7CF;">（对应 Flink 的 Checkpoint 状态）</font>
2. <font style="background-color:#E8F7CF;">在 </font>**<font style="background-color:#E8F7CF;">Checkpoint 触发时，内存中的数据才会刷写到文件系统</font>**<font style="background-color:#E8F7CF;">（生成 Hudi 文件）</font>
3. <font style="background-color:#E8F7CF;">只有</font>**<font style="background-color:#E8F7CF;"> Checkpoint 完成后，对应的数据才对下游查询可见</font>**



数据的可见性延迟

+ 对于COW（Copy-On-Write）表：
    - 数据可见性 ≥ Checkpoint 间隔
    - 例如 3 分钟 Checkpoint，数据延迟通常为 3-5 分钟（包含 Checkpoint 执行时间）
+ 对于MOR（Merge-On-Read）表：
    - 增量数据：通常也需要等 Checkpoint 完成
    - 读取 Log 文件：通过增量查询可能提前看到部分数据，但完整一致性仍依赖 Checkpoint



影响数据可见性的因素

1. **<font style="color:rgb(15, 17, 21);">Checkpoint 执行时间</font>**
    - <font style="color:rgb(15, 17, 21);">实际延迟 = Checkpoint 间隔 + Checkpoint 执行时间</font>
    - <font style="color:rgb(15, 17, 21);">大数据量时，Checkpoint 执行可能需要 1-2 分钟</font>
2. **<font style="color:rgb(15, 17, 21);">Hudi配置</font>**

```properties
# 可能影响可见性的配置
write.commit.async.enabled=true  # 默认true，异步提交
write.bucket_assign.tasks=4      # 并行度
```

3. **<font style="color:rgb(15, 17, 21);">读取模式</font>**
    - **<font style="color:rgb(15, 17, 21);">快照读取</font>**<font style="color:rgb(15, 17, 21);">：必须等 Checkpoint 完成</font>
    - **<font style="color:rgb(15, 17, 21);">增量读取</font>**<font style="color:rgb(15, 17, 21);">：可能有一定灵活性</font>

<font style="color:rgb(15, 17, 21);"></font>

<font style="color:rgb(15, 17, 21);">在默认配置下，Hudi 数据的可见性确实会超过 Flink 的 Checkpoint 间隔时间，通常为：</font>

+ <font style="color:rgb(15, 17, 21);">最小延迟 = Checkpoint 间隔</font>
+ <font style="color:rgb(15, 17, 21);">实际延迟 = Checkpoint 间隔 + Checkpoint 执行时间 + 少量提交开销</font>
