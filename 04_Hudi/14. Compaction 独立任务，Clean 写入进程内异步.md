另起一个进程执行 Compaction 任务；Clean 进程跟随入湖的进程，异步执行

# 这种混合模式的合理性分析
## 架构特点
+ Compaction 分离：另起任务执行，避免资源抢占
+ Clean 写入内异步：简单轻量，无需额外运维

## 为什么这种组合很合理？
| 维度 | Compaction | Clean | 分析 |
| --- | --- | --- | --- |
| 资源消耗 | 高（CPU+IO密集，需大量内存） | 低（主要是元数据操作） | Clean较轻，不影响写入主流程 |
| 执行频率 | 高（每 3-10 次提交） | 低（每10-30次提交） | Clean频率低，异步开销小 |
| 失败影响 | 可能导致写入积压 | 仅存储增长，无数据风险 | Clean可容忍短暂失败 |
| 复杂度 | 需独立作业运维 | 一行配置即可 | Clean无需额外运维 |


# 具体配置方案
## **Flink SQL 写入作业配置：Clean**
```sql
CREATE TABLE hudi_sink (
  order_id STRING,
  user_id STRING,
  amount DOUBLE,
  ts TIMESTAMP(3),
  dt STRING,
  PRIMARY KEY(order_id) NOT ENFORCED
) PARTITIONED BY (dt)
WITH (
  'connector' = 'hudi',
  'path' = 'hdfs:///user/hudi/orders',
  'table.type' = 'MERGE_ON_READ',
  'hoodie.datasource.write.recordkey.field' = 'order_id',
  'hoodie.datasource.write.precombine.field' = 'ts',
  
  -- 索引配置
  'hoodie.index.type' = 'BUCKET',
  'hoodie.bucket.index.num.buckets' = '256',
  
  -- ===== Compaction 配置：只生成计划，不执行 =====
  'compaction.async.enabled' = 'false',        -- 不执行compaction
  'compaction.schedule.enabled' = 'true',       -- 但生成compaction计划
  'compaction.delta_commits' = '5',              -- 每5次提交生成一个计划
  
  -- ===== Clean 配置：在写入作业中异步执行 =====
  'clean.async.enabled' = 'true',                -- 开启异步clean
  'clean.delta_commits' = '15',                   -- 每15次提交触发一次clean
  'hoodie.keep.max.commits' = '30',               -- 最多保留30个提交版本
  'hoodie.keep.min.commits' = '20',                -- 最少保留20个
  'hoodie.cleaner.parallelism' = '2',              -- clean并发度（不要太高）
  
  -- 可选：基于时间保留
  'hoodie.cleaner.retainer.hours' = '72',          -- 保留72小时内数据
  
  -- 文件大小控制
  'hoodie.parquet.max.file.size' = '536870912'
);

-- 写入数据
INSERT INTO hudi_sink SELECT ... FROM kafka_source;
```

```bash
# 提交独立的compaction作业
```

# 这种模式的优势
## **资源利用最优化**
+ Clean：轻量操作，写入作业后台线程执行，对写入影响极小（<5% 资源消耗）
+ Compaction：重型操作，独立作业可灵活分配资源，不影响写入

## **运维复杂度平衡**
+ 减少一个独立作业（不用单独维护 Clean 作业）
+ 但保留了 Compaction 的资源隔离

## **数据一致性保障**
+ Clean 操作是无状态的，即使失败也不会影响数据完整性
+ Compaction 独立作业可通过调度平台控制执行窗口

# 注意事项和最佳实践
## **Clean 并发度控制**
```sql
-- 不要设置太高的clean并行度，避免NameNode压力
'hoodie.cleaner.parallelism' = '2'  -- 2-5为宜
```

## **监控关键指标**
```sql
-- 在Flink Web UI观察
-- 1. Clean操作耗时（通常应在秒级）
-- 2. Checkpoint是否受影响
-- 3. .hoodie目录下.inflight文件数量
```

## **版本保留策略的联动**
```sql
-- 确保这些值协调一致
'hoodie.keep.max.commits' = '30',     -- 总保留版本
'clean.delta_commits' = '15',          -- 每15次提交清理一次
'compaction.delta_commits' = '5'       -- 每5次提交生成compaction计划

-- 这样设计保证：
-- 一个版本在被clean删除前，至少经历过3次compaction机会
-- 即：30/5 = 6次compaction机会，15/5 = 3次compaction机会后才会被clean
```

## **高峰期调整**
```sql
-- 如果发现写入高峰期clean影响性能，可以临时调低频率
-- 通过StreamPark动态配置
'clean.delta_commits' = '20'  -- 高峰期从15改为20
```

# 六、总结
这种"Compaction分离 + Clean异步"的混合模式是生产环境的黄金组合：

+ ✅ Compaction：独立作业，重型操作资源隔离
+ ✅ Clean：写入内异步，轻量操作无需额外运维
+ ✅ 配置简单：只需在写入端调整几个参数
+ ✅ 性能平衡：既保证了写入稳定性，又减少了运维负担

适用场景：

+ 中等规模表（100GB-1TB/天）
+ 对写入稳定性要求高
+ 希望减少独立作业数量
+ 有StreamPark等平台方便配置管理

这种方案在实践中经过验证，既能保证写入作业的稳定，又能有效管理存储空间，是性价比很高的选择。
