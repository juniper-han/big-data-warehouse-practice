# <font style="color:rgb(0, 0, 0);">Compaction 参数</font>
1. 异步 Compaction 开关
    - `true` 表示 JobManager 中会另外起一个线程定期调度和执行 Compaction。

```sql
-- 是否开启异步 Compaction
'compaction.async.enabled' = 'true'     -- 推荐 true，生产环境通常开启
'compaction.schedule.enabled' = 'true'  -- 推荐 true，与异步配合使用
```

2. Compaction 触发间隔
    - 生产建议：优先使用 `compaction.delta_commits`，例如设置为 5，表示每完成 5 次提交就触发一次 Compaction 调度。
    - 原因：基于时间间隔容易在数据量少时做无效调度，或在大流量时来不及调度；基于 Commit 次数的与数据量挂钩，更贴合实际负载。

```sql
'compaction.delta_commits' = '5'        -- 基于 commits 次数触发（推荐生产使用）
-- 或
'compaction.schedule.interval' = '60000' -- 基于时间间隔触发（单位毫秒）
```

3. Compaction 资源控制，生产建议
    - `compaction.tasks`：建议设置为 CPU 核心数 / 2 左右。如果集群资源紧张，可以适当调小（如 2~4），避免 Compaction 占用太多 slot
    - `compaction.memory`：如果单个 Parquet 文件较大（例如 512MB 以上），建议上调至 2048~4096，防止 OOM

```sql
-- 同时进行的 Compaction 任务数
'compaction.tasks' = '10'                -- 默认 10
-- 每个 Compaction 任务的内存（MB）
'compaction.memory' = '1024'             -- 默认 1024 MB
```

4. 目标文件大小
    - ODS / DWD 层（大表）：建议 512MB ~ 1GB，避免 HDFS 小文件，同时 NameNode 压力适中
    - DWS / ADS 层（小表或高更新率表）：建议 256MB ~ 512MB，便于后续查询快速扫描

```sql
-- Compaction 后希望达到的文件大小（单位字节）
'compaction.target.file.max.size' = '536870912'  -- 512MB，默认 1GB
```

# Clean 参数
1. Clean 策略与保留版本数
    - `KEEP_LATEST_COMMITS`：基于提交版本保留
    - `KEEP_LATEST_FILE_VERSIONS`：基于每个文件的最新版本保留，推荐，因为它是全局视角，清理逻辑更清晰，便于管理

```sql
-- 保留的提交版本数（默认 20）
'hoodie.keep.max.commits' = '30'

-- 最少保留的提交版本数（默认 10）
'hoodie.keep.min.commits' = '20'

-- Clean 策略（默认 KEEP_LATEST_COMMITS）
'hoodie.cleaner.policy' = 'KEEP_LATEST_COMMITS'  -- 或 KEEP_LATEST_FILE_VERSIONS
```

2. 保留小时数（基于时间）
    - 如果业务有数据回滚或时间旅行查询需求，建议设置较大的值（如 72~168 小时）。如果存储空间紧张，可以适当调低（如 12~24 小时）

```sql
-- 保留最近多少小时的数据版本（默认 24 小时）
'hoodie.cleaner.retainer.hours' = '48'

-- 保留最近多少分钟的提交（优先级高于小时）
'hoodie.cleaner.retainer.minutes' = '120'
```

3. Clean 触发频率
    - 优先使用 `clean.delta_commits`，例如设置为 5 或 10。基于 commits 次数与数据写入量挂钩，避免在数据量少时做无效 Clean

```sql
-- 异步 Clean 开关（默认 true）
'clean.async.enabled' = 'true'

-- 基于 commits 次数触发 Clean（推荐）
'clean.delta_commits' = '5'  -- 每 5 次提交触发一次 Clean

-- 或基于时间间隔触发（毫秒）
'clean.trigger.strategy' = 'TIME_BASED'
'clean.trigger.max.delay' = '3600000'  -- 1 小时
```

4. Clean 并行度与资源控制
    - 并行度建议设置为 CPU 核数/2 或稍低，如果 NameNode 压力大，建议调低

```sql
-- Clean 操作的并行度（默认 20）
'hoodie.cleaner.parallelism' = '10'

-- 是否在 Clean 时进行文件系统校验（默认 false）
'hoodie.cleaner.commits.retained' = '30'  -- 与 keep.max.commits 配合
```

5. 清理策略细化

```sql
-- 是否清理孤儿文件（默认 true）
'hoodie.cleaner.policy.failed.writes' = 'LAZY'

-- 清理孤儿文件的最早时间
'hoodie.cleaner.truncate.retain.hours' = '24'
```

# 生产环境推荐配置
```sql
CREATE TABLE hudi_table (...) WITH (
    'connector' = 'hudi',
    'table.type' = 'MERGE_ON_READ',
    
    -- 压缩配置
    'compaction.async.enabled' = 'true',
    'compaction.delta_commits' = '4',
    
    -- 清理配置
    'clean.async.enabled' = 'true',
    'cleaner.policy' = 'KEEP_LATEST_FILE_VERSIONS',
    'cleaner.fileversions.retained' = '3',  -- 保留3个版本
    
    -- 归档配置（管理Timeline）
    'archive.min_commits' = '20',
    'archive.max_commits' = '30',
    
    -- 小文件处理
    'hoodie.parquet.small.file.limit' = '104857600'  -- 100MB
);
```
