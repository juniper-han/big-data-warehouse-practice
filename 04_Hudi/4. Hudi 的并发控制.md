# 并发概述
不同数据库系统在并发控制上存在一定的差异，且 <font style="background-color:#E8F7CF;">Hudi 需要自己的并发控制机制</font>

+ 核心观点：没有万能的并发控制，**不同工作负载需要不同的并发控制机制**

## 四种系统的并发控制比较
1. RDBMS
    - 特点：最严格、最全面
    - 原因：要处理最复杂的情况
    - 机制：多个隔离级别（读已提交、可重复读、串行化等）、细粒度锁（行锁、表锁、意向锁等）、死锁检测和解决、强制主键、外键约束、维护索引的一致性
    - 适用场景：银行交易、订单系统等需要严格一致性的场景
2. NoSQL
    - 特点：宽松，简单
    - 原因：为了更高的扩展和性能
    - 机制：最终一致性（数据最终会一致，但不保证立即）、简单的行级原子性（每行操作是原子的）、放弃复杂的事务特性
    - 适用场景：社交网络、推荐系统等可以容忍短暂不一致的场景
3. 传统数据仓库
    - 特点：类似 RDBMS，但针对分析优化
    - 原因：既要保证一致性，又要处理海量数据分析
    - 机制：基于列存储、类似 RDBMS 的锁机制、强制键约束
    - 适用场景：企业级数据仓库、BI 分析
4. 云数据仓库
    - 特点：存算分离架构
    - 原因：为了云原生的弹性扩展
    - 机制：存算分离、更少的隔离级别、**不强制键约束（重要差异）**
    - 适用场景：Snowflake、BigQuery 等云上数据分析

## 数据湖并发控制的特殊性
数据湖的特点

1. 大规模数据：PB 级数据，不能像 RDBMS 那样细粒度锁
2. 多引擎访问：Spark、Flink、Presto、Hive 等同时读写
3. 存储成本低但延迟高：对象存储（S3/HDFS）不支持原子覆盖
4. 批流混合：既有批量 ETL，又有实时流处理



已有的并发控制不适合数据湖

1. 传统数据库的锁机制，在数据湖规模下会成为瓶颈
2. 但 NoSQL 的"最终一致性"又不够用（数据湖需要保证数据处理的一致性）
3. 云数据仓库的"不强制键约束"可能不适合某些场景

## Hudi 对并发控制的需求
Hudi 要设计自己的并发控制机制

+ 要有事务性：保证数据写入的一致性
+ 但不能太重：不能像 RDBMS 那样影响性能
+ 要支持多并发：多个作业同时读写
+ 要处理延迟数据：流处理中的迟到数据

# 数据湖并发控制中的陷阱
## 乐观并发控制（OCC）
当前数据湖并发控制的现状：乐观并发控制（OCC）

+ 原理：先假设不会冲突，写完再检查
+ 实现方式：
    - 获取表级锁（往往只是JVM级别的轻量锁），写入文件
    - 检查是否有其他作业写了重叠的文件
    - 如果有冲突→回滚重试；如果没有→提交成功
+ OCC 的适用场景（已过时）
    - 仅追加（append-only）的数据模型
    - 轻量级批处理作业
    - 低冲突率的场景



OCC 在现代数据湖中的严重问题

+ 现代数据湖需要更新、删除、合并：传统 OCC 为不可变/仅追加设计，这导致冲突率大大增加
+ <font style="background-color:#E8F7CF;">长事务场景下的“饥饿”问题</font>：运行时间越长，冲突概率越高，几乎注定失败，长时间运行的作业（几小时）可能最后才发现冲突，全部重做，资源浪费巨大
+ 资源浪费严重：长时间作业失败，浪费的不仅是时间，还有计算资源、存储I/O等



痛点场景：如下图所示

## 悲观锁
传统替代方案：悲观锁也不好，悲观锁存在的问题

+ 限制并发性：锁会阻塞其他操作
+ 性能瓶颈：在大规模数据湖中，锁会成为扩展的瓶颈
+ 死锁风险：需要复杂的死锁检测机制

# Hudi 并发方案：流式思维 + 日志协议
## 简介
Hudi 创新并发方案：流式思维 + 日志协议

+ 洞察：数据湖的写入更像流处理，而非数据库事务，流处理通过**有序的事件日志**（如 Kafka）解决并发
+ Hudi 解决方案：基于 Timeline 的日志协议
+ 技术实现要点
    - 文件级别的并发控制（不是行级，也不是表级）
    - 基于 Timeline 的协调：利用之前讲的 Timeline 作为"协调日志"
    - 最低限度的原子写入：依赖云存储（S3/HDFS）的基础原子操作



## Hudi 协议的工作原理
简化版

+ 步骤1：作业想要修改某个文件
+ 步骤2：在 Timeline 上"预订"一个位置（生成一个 Instant）
+ 步骤3：实际修改文件
+ 步骤4：在 Timeline 上标记完成
+ 步骤5：其他作业通过读取 Timeline 知道文件状态

## 与传统 OCC 的区别
| <font style="color:rgb(15, 17, 21);">方面</font> | <font style="color:rgb(15, 17, 21);">传统OCC</font> | <font style="color:rgb(15, 17, 21);">Hudi的日志协议</font> |
| --- | --- | --- |
| <font style="color:rgb(15, 17, 21);">冲突检测</font> | <font style="color:rgb(15, 17, 21);">写完后检查</font> | <font style="color:rgb(15, 17, 21);">写前通过 Timeline 协调</font> |
| <font style="color:rgb(15, 17, 21);">锁机制</font> | <font style="color:rgb(15, 17, 21);">表级锁</font> | <font style="color:rgb(15, 17, 21);">无锁，基于版本</font> |
| <font style="color:rgb(15, 17, 21);">失败代价</font> | <font style="color:rgb(15, 17, 21);">整个作业重跑</font> | <font style="color:rgb(15, 17, 21);">仅冲突部分重试</font> |
| <font style="color:rgb(15, 17, 21);">长事务支持</font> | <font style="color:rgb(15, 17, 21);">差</font> | <font style="color:rgb(15, 17, 21);">好（通过版本隔离）</font> |


## Hudi 并发机制应对多流拼接的挑战
| 挑战 | Hudi 解决方案 |
| --- | --- |
| 多个流同时写入同一张表 | 每个流作为独立的生产者，通过 Timeline 协调 |
| 数据到达时间不同，但要保证一致性 | 基于事件的排序，而不是基于锁的阻塞 |
| 可能互相覆盖或冲突 | 即使有冲突，也能通过版本控制合并 |


# Hudi 并发模型
## Hudi 并发模型 1：单写入，表服务同步运行
Hudi 并发模型 1 描述：单写入，内联表服务，无并发控制

+ 场景：只有一个**<font style="background-color:#FBDE28;">写入作业（单写入）</font>**，写完后立即运行表服务
+ 写入作业执行流程
    - 写入数据
    - 立即执行表服务（清理、压缩等），表服务在写入后，<font style="background-color:#FBDE28;">同步执行</font>
    - 完成提交
+ 适用场景：简单 ETL 管道、离线批处理、测试环境
+ 优点：简单易用，表状态总是最新的，无需额外协调

## Hudi 并发模型 2：单写入，表服务异步执行
场景描述

+ **单写入**：仍然只有一个写入作业
+ **多种操作**：但写入包含多种操作（插入、更新、删除）
+ **删除跨分区**：删除可能跨越很多分区（如 GDPR 合规删除）

关键问题：将它们混合在同一个工作负载中，可能会大大影响摄取延迟

+ 如果删除操作，需要扫描整个表（2 小时）
+ 但写入需要：每 30 分钟快速完成
+ 同步执行会导致：写入延迟暴增



Hudi 解决方案：Hudi 并发模型 2，基于文件锁的并发控制：

应用场景：单个 Flink 作业内的多个 task 并发写入



工作原理：

1. Flink Task1 要写文件 A：
    1. 尝试获取文件 A 的锁
    2. 如果获取成功 → 写入文件 A
    3. 如果获取失败 → 等待重试
2. Flink Task2 也要写文件A：
    1. 尝试获取文件 A 的锁
    2. 发现已被 Task1 占用 → 等待
    3. 等待超时或 Task1 释放锁

技术实现

+ 使用分布式锁服务（如 HDFS 文件锁）
+ 或基于存储的原子操作实现轻量级锁

**<font style="background-color:#FBDE28;"></font>**

**<font style="background-color:#FBDE28;">写入和【表服务】解耦</font>**

+ 写入流程：
    - 写入作业：快速写入增量日志（Avro 格式）
    - 记录所有操作：插入、更新、删除
    - 写入便宜（相比 Parquet 便宜10倍）
    - 低延迟
+ 表服务流程：异步在后台运行
    - 压缩服务（Compaction）将日志合并到 Parquet
    - 清理服务回收空间
    - 不阻塞写入



技术创新

+ 记录级索引：知道每条记录在哪个文件里
+ 增量日志（Avro）：便宜高效的记录变更
+ 时间线（Timeline）：协调写入和异步服务

优势

+ 写入延迟低：**只写日志**，不立即合并
+ 可回溯性好：通过日志可以追溯所有变更
+ 扩展性强：Uber 验证了 100PB 规模



Uber 的实践（100PB 规模）

+ 将所有操作（插入、更新、删除）都发送到同一个 Kafka 主题
+ 单个写入作业从这个主题消费，并写入 Hudi
+ 实现了无锁的高并发控制

## Hudi 并发模型 3：多写入（乐观并发控制）
场景：**多个写入作业**同时写同一张表

+ 例如：一个 SQL 删除作业 + 一个流式摄取作业

挑战：删除作业和摄取作业可以相互竞争和饿死



<font style="background-color:#E8F7CF;">Hudi的折中方案：写操作用 OCC 写入</font>**<font style="background-color:#E8F7CF;">日志</font>**<font style="background-color:#E8F7CF;">，表服务无锁异步</font>

+ 应用场景：多个独立 Flink 作业并发写入

工作原理：作业A 和 作业B 同时写入

1. 两个作业都读取当前表状态
2. 各自计算要写入的数据
3. 提交时检查是否有冲突
4. 无冲突 → 提交成功
5. 有冲突 → 按策略处理（失败重试/合并）



多写入者工作流：每个流用一个写入作业

+ 写入者 1（删除作业）：
    1. 获取锁（OCC 机制）
    2. 写入删除操作到日志
    3. 释放锁
+ 写入者 2（摄取作业）：
    1. 获取锁（OCC 机制） 
    2. 写入更新操作到日志
    3. 释放锁
+ 表服务（异步）：
    1. 无锁运行
    2. 合并所有写入者的日志
    3. 生成最终的 Parquet 文件



设计原因：

+ 写操作冲突少：因为只写日志，速度快，冲突概率低
+ 表服务不竞争：后台异步执行，不阻塞写入
+ 失败代价小：即使冲突回滚，也只损失写日志的时间



与传统 OCC 的关键改进：

| <font style="color:rgb(15, 17, 21);">传统 OCC</font> | <font style="color:rgb(15, 17, 21);">Hudi的多写入模型</font> |
| --- | --- |
| <font style="color:rgb(15, 17, 21);">整个作业冲突回滚</font> | <font style="color:rgb(15, 17, 21);">只回滚日志写入部分</font> |
| <font style="color:rgb(15, 17, 21);">表级锁阻塞时间长</font> | <font style="color:rgb(15, 17, 21);">短暂的乐观锁</font> |
| <font style="color:rgb(15, 17, 21);">表服务同步执行阻塞</font> | <font style="color:rgb(15, 17, 21);">表服务完全异步无锁</font> |


## 三种模型对比总结
| <font style="color:rgb(15, 17, 21);">模型</font> | <font style="color:rgb(15, 17, 21);">并发写入者</font> | <font style="color:rgb(15, 17, 21);">表服务执行</font> | <font style="color:rgb(15, 17, 21);">适用场景</font> | <font style="color:rgb(15, 17, 21);">优点</font> |
| --- | --- | --- | --- | --- |
| <font style="color:rgb(15, 17, 21);">模型1</font> | <font style="color:rgb(15, 17, 21);">单写入者</font> | <font style="color:rgb(15, 17, 21);">内联同步</font> | <font style="color:rgb(15, 17, 21);">简单批处理</font> | <font style="color:rgb(15, 17, 21);">简单、表状态最新</font> |
| <font style="color:rgb(15, 17, 21);">模型2</font> | <font style="color:rgb(15, 17, 21);">单写入者</font> | <font style="color:rgb(15, 17, 21);">异步</font> | <font style="color:rgb(15, 17, 21);">混合工作负载（更新 + 删除）</font> | <font style="color:rgb(15, 17, 21);">低延迟、高吞吐</font> |
| <font style="color:rgb(15, 17, 21);">模型3</font> | <font style="color:rgb(15, 17, 21);">多写入者</font> | <font style="color:rgb(15, 17, 21);">异步</font> | <font style="color:rgb(15, 17, 21);">多来源写入（SQL + 流）</font> | <font style="color:rgb(15, 17, 21);">一定程度的并发控制</font> |


# 跨任务并发写支持
## 常见的并发写入场景 & Flink Job 概念解析
Flink Job：一个完整的 Flink 应用程序执行单元

+ 通过 `flink run` 提交到集群
+ 有自己的 JobID
+ 在一个 Flink Web UI 中可以看到
+ 可以独立启停、扩缩容



### **场景一：一个 Flink Job**
一个 Flink Job：一个 SQL 文件，多个`INSERT`

+ 提交到 StreamPark：创建一个 Application，上传这个 SQL 文件，执行，**产生一个 Job**
+ Hudi 视角：
    - 只有一个写入者（这个 Job）
    - 使用并发模型2（单写入者，异步表服务）
    - 不需要乐观并发控制

```sql
-- 文件：multi_stream_job.sql
CREATE TABLE sourceA (...);
CREATE TABLE sourceB (...);
CREATE TABLE sourceC (...);
CREATE TABLE hudi_table (...);

-- 三个流在同一个作业中
INSERT INTO hudi_table SELECT * FROM sourceA;
INSERT INTO hudi_table SELECT * FROM sourceB;
INSERT INTO hudi_table SELECT * FROM sourceC;
```

### 场景二：多个 Flink Job
多个 Flink Job：多个 SQL 文件，每个一个`INSERT`

+ 提交到 StreamPark：创建三个独立的 Application，分别上传三个 SQL 文件，分别执行，产生三个独立的 Flink Job
+ Hudi 视角
    - 有三个独立的写入者
    - 使用并发模型3：多写入者
    - 需要乐观并发控制

```sql
-- 文件1：stream_a_job.sql
CREATE TABLE sourceA (...);
CREATE TABLE hudi_table (...);
INSERT INTO hudi_table SELECT * FROM sourceA;

-- 文件2：stream_b_job.sql  
CREATE TABLE sourceB (...);
CREATE TABLE hudi_table (...);
INSERT INTO hudi_table SELECT * FROM sourceB;

-- 文件3：stream_c_job.sql
CREATE TABLE sourceC (...);
CREATE TABLE hudi_table (...);
INSERT INTO hudi_table SELECT * FROM sourceC;
```

## 如何判断是单个还是多个 Flink Job
方法一：查看 Flink Web UI

+ 如果多个流在同一个 Flink Web UI 界面中 ——> 同一个 Job
+ 如果多个流在不同的 Flink Web UI 界面中 ——> 多个 Job



方法二：查看 JobID

+ FlinkSQL：`SELECT CURRENT_JOB_ID;`

## FlinkSQL 中的配置
### **场景一：一个 Job**
一个 Job，多`INSERT`写入，并发控制模型2（单写入，异步表服务）

+ 优点：简化并发控制，统一故障恢复，资源利用率高
+ 推荐用于多流拼接
+ 要求：流的数据量不大，可以合并处理

```sql
-- 在 StreamPark 的一个 Application 中
CREATE TABLE hudi_table (...) WITH (
  'connector' = 'hudi',
  
  -- 单写入者模式，不需要并发控制
  'hoodie.write.concurrency.mode' = 'single_writer',
  
  -- 使用 PartialUpdateAvroPayload
  'write.payload.class' = 'org.apache.hudi.common.model.PartialUpdateAvroPayload',
  
  -- 多字段预合并
  'precombine.field' = 'stream_a_ts:col_a;stream_b_ts:col_b;stream_c_ts:col_c'
);

-- 三个 INSERT 在同一个作业中
INSERT INTO hudi_table SELECT ..., 'stream_a' as stream_id FROM sourceA;
INSERT INTO hudi_table SELECT ..., 'stream_b' as stream_id FROM sourceB;  
INSERT INTO hudi_table SELECT ..., 'stream_c' as stream_id FROM sourceC;
```

### 场景二：多个 Job
多个 Job，并发控制模型3（多写入者，OCC）

+ 缺点：需要复杂的并发控制，可能发生冲突，有资源浪费风险
+ 需要特定理由选择该方法，如
    - 数据量巨大，需要独立扩缩容
    - 故障隔离要求高（一个流失败不影响另一个）
    - 不同团队负责不同流
    - 需要不同的 SLA（如流 A 要求低延迟，流 B 要求高吞吐）

```sql
-- 每个 Application 的配置需要不同
-- Application 1（流A）：
CREATE TABLE hudi_table (...) WITH (
  'connector' = 'hudi',
  -- 启用乐观并发控制
  'hoodie.write.concurrency.mode' = 'optimistic_concurrency_control',
  'hoodie.write.log.suffix' = 'job1',  -- 唯一标识
  -- 配置锁提供者（可选）
  'hoodie.write.lock.provider' = 'org.apache.hudi.client.transaction.lock.FileSystemBasedLockProvider',
  -- 锁配置
  'write.lock.hdfs.timeout_ms' = '120000',      -- 锁超时时间
  'write.lock.hdfs.lease_duration_ms' = '60000' -- 锁租约时长
  'write.payload.class' = 'org.apache.hudi.common.model.PartialUpdateAvroPayload'
);

-- Application 2（流B）：
CREATE TABLE hudi_table (...) WITH (
  'connector' = 'hudi',
  -- 启用乐观并发控制
  'hoodie.write.concurrency.mode' = 'optimistic_concurrency_control',
  'hoodie.write.log.suffix' = 'job2',  -- 不同标识
  -- 配置锁提供者（可选）
  'hoodie.write.lock.provider' = 'org.apache.hudi.client.transaction.lock.FileSystemBasedLockProvider',
  -- 锁配置
  'write.lock.hdfs.timeout_ms' = '120000',      -- 锁超时时间
  'write.lock.hdfs.lease_duration_ms' = '60000' -- 锁租约时长
  'write.payload.class' = 'org.apache.hudi.common.model.PartialUpdateAvroPayload'
);
```
