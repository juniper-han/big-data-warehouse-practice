# Watermark 介绍
## Watermark 简介
水印（Watermark）：本质上是一种单调递增的时间戳

+ **<font style="background-color:#FBDE28;">针对的是 EventTime</font>**<font style="background-color:#FBDE28;">：Flink 为了处理 EventTime 窗口计算提出的一种机制</font>
+ 由 **Flink Source **或者自定义的 Watermark 生成器按照需求，以 Punctuated 或 Periodic 两种方式生成的一种系统 Event
+ 它表示“所有时间戳 ≤ Watermark 的数据都已经到达系统”。
+ 算子接收到一个 Watermark 时，框架知道不会再有任何小于该 Watermark 的时间戳的数据元素到来了，所以 Watermark 可以看做是告诉 Flink 数据流已经处理到什么位置（时间维度）的方式

## 引入 Watermark 的必要性
引入水印的原因，解决两大难题

+ 数据乱序：数据到达顺序与发生顺序不一致（常见于网络传输）
    - 解决方案：设置一个允许延迟的阈值（如5秒），告诉系统可以多等一会儿
+ 延迟计算：不知道何时可以安全计算窗口结果（如“9点的数据是否到齐？”）
    - 解决方案：Watermark 提供了计算触发信号：“时间戳≤Watermark的数据已到齐，可以计算了！”

如果没有 Watermark

+ 窗口会无限期等待，无法输出结果（因为不知道还有没有数据迟到）
+ 或者窗口直接关闭，导致迟到数据被丢弃（计算结果不准确）



Watermark 的作用

+ **控制事件时间进展**：推动 Flink 内部的事件时间时钟。
+ **判断迟到数据的标准**
+ **触发窗口计算**：当 Watermark ≥ 窗口结束时间时，触发窗口计算。
+ **平衡延迟与准确性**：通过设置延迟容忍度，在“等待更多数据”和“及时输出结果”之间取得平衡

## Watermark 必须单调递增
Watermark 的本质是**事件时间的进展标记**，它必须满足**“Watermark 的时间戳只能前进（或保持不变），绝不能后退”**。  
公式：`Watermark = max(历史最大EventTime, 新数据EventTime) - 延迟`

举例：

```plain
事件时间轴：9:00   9:05   9:10   9:15                           9:20
数据到达：  [A]    [C]    [B]   (B的时间戳是9:08，但9:15才到)

Watermark生成：
- 收到A(9:00) → Watermark = 9:00 - 5s = 8:55
- 收到C(9:10) → Watermark = 9:10 - 5s = 9:05
- 收到B(9:08) → Watermark = max(9:10, 9:08) - 5s = 9:05

窗口[9:00-9:05]触发时机：当Watermark ≥ 9:05时
```

关注点：B  
在 B 到达时

+ 已收到的数据时间戳：A(9:00)、C(9:10) → 当前最大时间戳 = 9:10
+ 新到达的数据时间戳：B(9:08)

如果直接计算：`Watermark = 9:08（B的时间戳） - 5s = 9:03`  
但之前的 Watermark 已经是 `9:10 - 5s = 9:05`，如果此时将 Watermark 降到 9:03，就违反了**单调性**原则！

因此，Flink 的策略是：  
**永远取 **`max(历史最大EventTime, 当前数据EventTime)`**作为基准**，再减去延迟阈值。  
这样能保证 Watermark 不会后退。

```plain
已知：
- 历史最大时间戳 = max(9:00, 9:10) = 9:10
- 新数据时间戳 = 9:08
- 允许延迟 = 5秒

计算：
Watermark = max(历史最大时间戳, 新数据时间戳) - 延迟
          = max(9:10, 9:08) - 5s
          = 9:05
```

# 使用 Watermark
## Watermark 的生成策略
<img src="https://cdn.nlark.com/yuque/0/2026/jpeg/54426354/1767618763517-a894d1f3-9b74-492c-85c0-cca949588ae7.jpeg" width="600" title="" crop="0,0,1,1" id="HWTDC" class="ne-image">

Flink 有两种方式用于生产 Watermark：

+ Punctuated：数据流中每一个**递增**的 EventTime 都会产生一个Watermark
+ Periodic：周期性的（一定时间间隔、或达到一定的记录条数）产生一个Watermark。在实际的生产中Periodic的方式



实际中，两种方式的优缺点，根据业务场景选择

+ Punctuated：在 TPS 很高的场景下会产生大量的 Watermark，在一定程度上对下游算子造成压力，所以只有在实时性要求非常高的场景才会选择
+ Periodic：必须结合时间和积累条数两个维度继续周期性产生Watermark，否则在极端情况下会有很大的延时



当我们构建了一个 DataStream 之后，使用`assignTimestampsAndWatermarks()`方法来构造水印，需要传入一个`WatermarkStrategy`对象

`DataStream.assignTimestampsAndWatermarks(WatermarkStrategy<T>)`



`WatermarkStrategy`接口的作用

+ 提供了很多静态的方法和带有缺省实现的方法
+ 只有一个方法是非`default`和没有缺省实现的，就是下面的这个方法。

```java
/**
  * Instantiates a WatermarkGenerator that generates watermarks according to this strategy.
  */
@Override
WatermarkGenerator<T> createWatermarkGenerator(WatermarkGeneratorSupplier.Context context);
```

所以默认情况下，我们只需要实现这个方法就行了，这个方法主要是返回一个 `WatermarkGenerator`，我们在进入这里边看看

```java
@Public
public interface WatermarkGenerator<T> {
    /**
  * Called for every event, allows the watermark generator to examine and remember the
  * event timestamps, or to emit a watermark based on the event itself.
  */
    void onEvent(T event, long eventTimestamp, WatermarkOutput output);
    /**
  * Called periodically, and might emit a new watermark, or not.
  *
  * <p>The interval in which this method is called and Watermarks are generated
  * depends on {@link ExecutionConfig#getAutoWatermarkInterval()}.
  */
    void onPeriodicEmit(WatermarkOutput output);
}
```

这个接口主要有两个方法：

+ `onEvent` ：每个元素都会调用这个方法，如果我们想依赖每个元素生成一个水印，然后发射到下游(可选，就是看是否用`output`来收集水印)，我们可以实现这个方法。
+ `onPeriodicEmit` : 如果数据量比较大的时候，我们每条数据都生成一个水印的话，会影响性能，所以这里还有一个周期性生成水印的方法。这个水印的生成周期可以这样设置：`env.getConfig().setAutoWatermarkInterval(5000L);`

## 内置水印生成策略
为了方便开发，Flink 提供了一些内置的水印生成方法供我们使用

### 单调递增的水印生成器
+ **特点**：数据时间戳严格递增，没有乱序。
+ `WatermarkStrategy.forMonotonousTimestamps();`这个也就是相当于上述的延迟策略去掉了延迟时间，以`eventTime`当了水印。
+ **生成方式**：`Watermark = 当前最大时间戳 - 0ms`，没有延迟
+ **代码**：

```java
DataStream dataStream = ...... ;
dataStream.assignTimestampsAndWatermarks(WatermarkStrategy.forMonotonousTimestamps());
```

它的底层实现是`AscendingTimestampsWatermarks`，其实它就是`BoundedOutOfOrdernessWatermarks`类的一个子类，没有了延迟时间，具体源码为

```java
@Public
public class AscendingTimestampsWatermarks<T> extends BoundedOutOfOrdernessWatermarks<T> {
    /**
  * Creates a new watermark generator with for ascending timestamps.
  */
    public AscendingTimestampsWatermarks() {
        super(Duration.ofMillis(0));
    }
}
```

### 固定的延迟时间
+ **特点**：数据乱序到达，需要设置最大允许延迟（如5秒）。
+ **生成方式**：`Watermark = 当前最大时间戳 - 最大允许延迟`
+ `WatermarkStrategy.forBoundedOutOfOrderness(Duration.ofSeconds(10));`

当水印落后于流中看到的最大（事件时间）时间戳指定的时间量时。Flink 提供了`BoundedAutofordernessWatermarks`生成器，该生成器将`maxOutOfOrderness`作为参数，即触发窗口计算时，允许元素延迟的最长时间。

+ 例如：`maxOutOfOrderness = 5`，如果当前的最大时间戳是`2020-10-10 10:30:00`，那么允许带有`2020-10-10 10:25:00 - 2020-10-10 10:30:00`范围内的事件时间的事件进入窗口，如果一旦出现的事件不在这个区间中，那么就会立即出发窗口计算。

```java
DataStream<Event> stream = input
    .assignTimestampsAndWatermarks(
        WatermarkStrategy
            .<Event>forBoundedOutOfOrderness(Duration.ofSeconds(5)) // 允许5秒延迟
            .withTimestampAssigner((event, timestamp) -> event.timestamp)
    );
```

+ 关键参数：`forBoundedOutOfOrderness(Duration)`设置最大允许延迟

## Watermark 单位
Watermark 是一种时间戳

**时间戳单位**：Flink 默认时间戳单位是**毫秒**（Unix 时间戳）。如果数据是秒或微秒，需转换

```java
// 秒→毫秒
.withTimestampAssigner((event, recordTimestamp) -> event.timestamp * 1000) 
```

+ 时间戳字段不一定是 Long 类型，但需确保最终提取的值是 **毫秒时间戳**

## 提取 EventTime
从数据中抽取`EventTime`，需要`TimestampAssigner`

```java
@Public
@FunctionalInterface
public interface TimestampAssigner<T> {
    ............

    long extractTimestamp(T element, long recordTimestamp);
}
```

使用的时候我们主要就是从我们自己的元素`element`中提取我们想要的`EventTime`。



使用 Flink 自带的水印策略和 `EventTime` 抽取类，可以这样用：

```java
DataStream dataStream = ...... ;
dataStream.assignTimestampsAndWatermarks(
    WatermarkStrategy
    .<Tuple2<String,Long>>forBoundedOutOfOrderness(Duration.ofSeconds(5))
    .withTimestampAssigner((event, timestamp)->event.f1));
```

## 使用建议
EventTime 和 Watermark 可以由`SourceFunction`产生，或者由用户自定义的时间戳分配器和水位线产生器来生成。

任何分区操作都会将元素的顺序打乱，例如：并行度改变，keyBy()操作等等。

所以最佳实践是：在尽量接近 Source 的地方分配 EventTime 和产生 Watermark，甚至最好在`SourceFunction`中分配EventTime 和产生 Watermark

当然在分配 EventTime 和产生 Watermark之前可以对流进行`map`和`filter`操作是没问题的，也就是说必须是窄依赖

示例：首先`filter`流，然后再分配时间戳和水位线。

```java
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment;

// 从调用时刻开始给env创建的每一个stream追加时间特征
env.setStreamTimeCharacteristic(TimeCharacteristic.EventTime);

DataStream<SensorReading> readings = env
  .addSource(new SensorSource)
  .filter(r -> r.temperature > 25)
  .assignTimestampsAndWatermarks(new MyAssigner());
```

# Watermark 如何解决乱序问题
## 问题描述

+ 例如：在实际时间的第 1 秒产生的数据有可能在第 5 秒中产生的数据之后到来。
+ 选具体某个 delay 的元素来说，假设在一个 5 秒的Tumble窗口，有一个EventTime是 11秒的数据，在第16秒时候到来了。

## EventTime Window 触发条件
EventTime Window 的触发条件是：

+ 当 Window 计算的 Timer <= 当前系统的 Watermak 时间戳时
+ 进行计算

对应的DDL

```sql
  ...,
  Event_time TimeStamp,   -- 事件事件
  WATERMARK wk1 FOR Event_time as withOffset(Event_time, 0)
  -- 水印
) with (
  ...
);
```

# 空闲数据源
在某些情况下，由于数据产生的比较少，导致一段时间内没有数据产生，进而就没有水印的生成，导致下游依赖水印的一些操作就会出现问题

+ 比如某一个算子的上游有多个算子，这种情况下，水印是取其上游两个算子的较小值，如果上游某一个算子因为缺少数据迟迟没有生成水印，就会出现 EventTime 倾斜，导致下游没法触发计算。
+ EventTime 倾斜：即有的 Source 的 EventTime 明显比其他要快或慢，导致取最小值的总体 Watermark 被慢 Watermark 拖慢的问题

## 空闲数据源简介和典型场景
在 Flink 中，**空闲数据源**指的是：

+ 在 **Keyed 数据流**中，某个 **Key 的分区**（partition）
+ 在 **一段时间内**（如5分钟）**没有任何数据到达**
+ 但其他 Key 的分区仍然有数据持续到达

**典型场景**：

+ 多租户系统中，某个用户突然停止产生数据
+ 多地区数据中，某个地区的数据源暂时中断
+ 多设备监控中，某个设备离线

## 不良影响
+ **核心问题：Watermark 停滞（全局进度阻塞）**
    - Flink 的 Watermark 是**全局进度标志**，遵循"木桶原理"：
    - `全局Watermark = min(所有并行分区的Watermark)`
+ 窗口无法触发
+ 状态无限增长
    - 窗口无法关闭 → 窗口状态无法清理 → 内存泄漏
    - 计时器（Timer）无法触发 → 状态持续累积
+ 实时性丧失
    - 数据处理延迟从**秒级**退化为**小时级**
    - 监控告警、实时仪表盘全部失效

## 处理方案
Flink 提供了 ****`withIdleness()`**** 方法专门处理这种情况：

允许用户在配置的时间内（即超时时间内）没有记录到达时将一个流标记为空闲。这样就意味着下游的数据不需要等待水印的到来。当下次有水印生成并发射到下游的时候，这个数据流重新变成活跃状态。

```java
WatermarkStrategy<Event> strategy = WatermarkStrategy
    .<Event>forBoundedOutOfOrderness(Duration.ofSeconds(5))
    .withTimestampAssigner((event, ts) -> event.timestamp)
    .withIdleness(Duration.ofMinutes(5)); // 标记5分钟无数据的分区为空闲
```

**工作原理**

1. **检测空闲**：某个分区超过指定时间（如5分钟）无数据到达
2. **标记空闲**：将该分区从 Watermark 计算中**排除**
3. **全局推进**：Watermark 基于**剩余活跃分区**计算，继续推进
4. **恢复参与**：当该分区再次有数据时，重新参与 Watermark 计算

****

**空闲时间设置**：根据业务场景调整

```java
.withIdleness(Duration.ofMinutes(5));    // 实时业务（较短）
.withIdleness(Duration.ofHours(1));      // 批量业务（较长）
```

**监控与告警**

```java
// 在ProcessFunction中监控Watermark进展
long currentWatermark = ctx.timerService().currentWatermark();
if (currentWatermark < System.currentTimeMillis() - 60000) {
    // Watermark延迟超过1分钟，触发告警
    alert("Watermark停滞可能！");
}
```

**多级超时策略**：对于重要业务，可以结合多种机制：

```java
WatermarkStrategy
    .forBoundedOutOfOrderness(Duration.ofSeconds(30))
    .withIdleness(Duration.ofMinutes(5))  // 第一级：标记空闲
    .withTimeout(Duration.ofMinutes(30)); // 第二级：完全超时（如有需要）
```
