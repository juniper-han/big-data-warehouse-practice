# Flink Join 简介
Flink 主要支持以下三种 Join 方式

+ 动态表（流） Join 动态表（流）
+ 动态表（流）Join 外部维表（如 Redis）
+ 动态表字段的列转行（特殊 Join）



细分 Flink 支持的 Join

+ 流与流的 Join
    - Regular Join
    - Interval Join：两条流一段时间区间内的 Join
    - Temporal Join：包括事件时间，处理时间的 Temporal Join，类似于离线中的快照 Join
+ 流与外部维表的 Join
    - Lookup Join：流与外部维表的 Join
+ 动态表字段的列转行
    - Array Expansion：表字段的列转行，类似于 Hive 的 explode 数据炸开的列转行
    - Table Function：自定义函数的表字段的列转行，支持 Inner Join 和 Left Outer Join

# Regular Join
Regular Join：通过等式条件关联两条流（左流 L，右流 R）数据输出，包括

+ `INNER JOIN`（Inner Equal Join）：流任务中，只有两条流 Join 到才输出，输出 `+[L, R]`
+ `LEFT JOIN`（Outer Equal Join）：流任务中
    - 左流数据到达，无论有没有 Join 到右流的数据，都会输出
        * Join 到输出 `+[L, R]`
        * 没 Join 到输出 `+[L, null]`
    - 如果之后右流数据到达，发现左流之前输出过没有 Join 到的数据，则会发起回撤流，先输出 `-[L, null]`，然后输出 `+[L, R]`
+ `RIGHT JOIN`（Outer Equal Join）：和 Left Join 一样，左表和右表的执行逻辑完全相反
+ `FULL JOIN`（Outer Equal Join）：流任务中，
    - 左流或者右流的数据到达之后，无论有没有 Join 到另外一条流的数据，都会输出
        * 对右流来说：Join 到输出 `+[L, R]`，没 Join 到输出 `+[null, R]`
        * 对左流来说：Join 到输出 `+[L, R]`，没 Join 到输出 `+[L, null]`
    - 如果一条流的数据到达之后，发现之前另一条流之前输出过没有 Join 到的数据，则会发起回撤流
        * 左流数据到达：回撤 `-[null, R]`，输出 `+[L, R]`
        * 右流数据到达：回撤 `-[L, null]`，输出 `+[L, R]`



关于 Regular Join 的注意事项：

+ 会产生撤回流
+ 实时 Regular Join 可以不是 等值 Join。等值 Join 和 非等值 Join 区别在于
    - 等值 Join 数据 Shuffle 策略是 Hash：会按照关联中的等值条件作为 id 发往对应的下游；
    - 非等值 Join 数据 Shuffle 策略是 Global：所有数据发往一个并发，按照非等值条件进行关联
+ Join 的流程：左流新来一条数据之后，会和右流中符合条件的所有数据做 Join，然后输出
+ 流的上游是无限的数据，所以要做到关联的话，**Flink 会将两条流的所有数据都存储在 State 中，所以 Flink 任务的 State 会无限增大**，因此需要为 State 配置合适的 TTL，以防止 State 过大

# Interval Join（时间区间 Join）
Interval Join：一条流 Join 另一条流中前后一段时间内的数据

应用场景：避免撤回流



Interval Join （左流 L，右流 R）包括

+ Inner Interval Join：流任务中，只有两条流 Join 到，才输出，输出 `+[L, R]`
    - 满足 Join on 中的条件：两条流的数据在时间区间 + 满足其他等值条件
+ Left Interval Join：流任务中
    - 左流数据到达之后，如果没有 Join 到右流的数据，就会等待（放在 State 中等）
    - 如果之后右流之后数据到达之后，发现能和刚刚那条左流数据 Join 到，则会输出 `+[L, R]`
    - 事件时间中随着 Watermark 的推进（也支持处理时间）。如果发现发现左流 State 中的数据过期了，就把左流中过期的数据从 State 中删除，然后输出 `+[L, null]`
    - 如果右流 State 中的数据过期了，就直接从 State 中删除
+ Right Interval Join：和 Left Interval Join 执行逻辑一样，只不过左表和右表的执行逻辑完全相反
+ Full Interval Join：流任务中
    - 左流或者右流的数据到达之后，如果没有 Join 到另外一条流的数据，就会等待（左流放在左流对应的 State 中等，右流放在右流对应的 State 中等）
    - 如果之后另一条流数据到达之后，发现能和刚刚那条数据 Join 到，则会输出 `+[L, R]`
    - 事件时间中随着 Watermark 的推进（也支持处理时间），发现 State 中的数据能够过期了，就将这些数据从 State 中删除并且输出
        * 左流过期输出 `+[L, null]`
        * 右流过期输出 `+[null, R]`

示例

```sql
CREATE TABLE show_log_table (
    log_id BIGINT,
    show_params STRING,
    row_time AS cast(CURRENT_TIMESTAMP as timestamp(3)),
    WATERMARK FOR row_time AS row_time
) WITH ();

CREATE TABLE click_log_table (
    log_id BIGINT,
    click_params STRING,
    row_time AS cast(CURRENT_TIMESTAMP as timestamp(3)),
    WATERMARK FOR row_time AS row_time
)
WITH ();

CREATE TABLE sink_table (
    s_id BIGINT,
    s_params STRING,
    c_id BIGINT,
    c_params STRING
) WITH (
  'connector' = 'print'
);

INSERT INTO sink_table
SELECT
    show_log_table.log_id as s_id,
    show_log_table.show_params as s_params,
    click_log_table.log_id as c_id,
    click_log_table.click_params as c_params
FROM show_log_table INNER 
JOIN click_log_table 
  ON show_log_table.log_id = click_log_table.log_id
 -- Interval 关联条件
 AND show_log_table.row_time 
     BETWEEN click_log_table.row_time - INTERVAL '4' HOUR 
         AND click_log_table.row_time;
```

# Temporal Join (时态表关联)
Temporal Join（时态表关联），用于

+ 流与维表实时关联
+ 流与变化历史表关联

解决了传统流处理中，“如何用最新/历史维度数据打宽事实流”的难题

## Temporal Join 简介
Temporal Join 是一种

+ 基于**时间属性**的关联操作
+ 它允许一条流记录根据其**时间戳**，去查询另一张“随时间变化的表”**在该时刻**的状态



在流处理中，常遇到两类需求

| 场景 | 问题 | Temporal Join 解决方案 |
| --- | --- | --- |
| 流 + 静态维表 | 维度数据会更新（如用户地址变更） | 关联当前最新维表（Lookup Join） |
| 流 + 变化历史表 | 需要关联事件发生时的维度状态（如订单创建时的汇率） | 关联历史版本维表（Event-Time Temporal Join） |


+ Lookup Join：查“现在”的维表（Processing Time）
+ Temporal Join：查“过去某个时刻”的维表（Event Time）



因此，Temporal Join 包含两种类型

+ Processing-Time Temporal Join（处理时间时态关联）：别称 Lookup Join，见第 5 节
+ Event-Time Temporal Join（事件时间时态关联）

## Versioned Table
### 简介
Versioned Table：版本表

+ 是一张随时间演化的表
+ 其中每条主键记录可能有多个版本
+ 每个版本有一个有效时间区间` [valid_from, valid_to)`，用于表示该版本在哪个时间段内生效



在实时业务中，有些维度数据是不断变化的，而事实事件发生在特定历史时刻，关联的维度数据需要是【发生时刻有效的版本】

### 数据模型
**物理输入：Changelog 流（CDC）**

Versioned Table 并非直接创建，而是由 Changelog Stream（变更日志流） 构建而来：来源：Debezium、Canal、Flink CDC 等

| op | product_id | price | update_time |
| --- | --- | --- | --- |
| +I | P001 | 100 | 2024-06-01 09:00 |
| +U | P001 | 120 | 2024-06-01 11:00 |
| +U | P001 | 90 | 2024-06-01 15:00 |




**逻辑视图：Versioned Table（Flink 内部构建）**

Flink 将 changelog 流物化为带时间版本的表：

| product_id | price | valid_from | valid_to |
| --- | --- | --- | --- |
| P001 | 100 | 2024-06-01 09:00:00 | 2024-06-01 11:00:00 |
| P001 | 120 | 2024-06-01 11:00:00 | 2024-06-01 15:00:00 |
| P001 | 90 | 2024-06-01 15:00:00 | +∞ (或下一个更新时间) |


关键特性：

+ 同一主键 (`product_id`) 多版本共存；
+ 时间区间 左闭右开` [valid_from, valid_to)`；
+ 最新版本的 `valid_to = +∞`。

### 如何构建 Versioned Table
Flink 不提供显式 DDL 创建 Versioned Table，而是通过以下方式隐式构建：

前提条件

+ 输入必须是 Changelog 流（有完整 `INSERT/UPDATE/DELETE`）；
+ 定义主键（`PRIMARY KEY`）；
+ 定义事件时间属性（Event Time）和 `WATERMARK`

只要满足上述三个条件（changelog + PK + event time），Flink 就会自动将该表视为 Versioned Table，无需额外声明。

例如：从 Kafka CDC 构建

```sql
-- Step 1: 定义 changelog 流（即 Versioned Table 的来源）
CREATE TABLE product_prices (
  product_id STRING,
  price DECIMAL(10, 2),
  update_time TIMESTAMP(3),
  -- 必须定义事件时间 + watermark
  WATERMARK FOR update_time AS update_time - INTERVAL '5' SECOND,
  -- 必须声明主键
  PRIMARY KEY (product_id) NOT ENFORCED
) WITH (
  'connector' = 'kafka',
  'format' = 'debezium-json',  -- 或 canal-json, maxwell, etc.
  'topic' = 'mysql.products.price_updates'
);

-- Step 2: 在 Temporal Join 中自动视为 Versioned Table
SELECT 
  o.order_id,
  o.product_id,
  o.qty,
  p.price,
  o.qty * p.price AS amount
FROM order_stream AS o
JOIN product_prices FOR SYSTEM_TIME AS OF o.order_time AS p
  ON o.product_id = p.product_id;
```



Flink 在 State Backend（如 RocksDB） 中维护 Versioned Table 的全量历史；

+ 每个主键的所有版本按时间排序存储；
+ 状态大小 = 主键数量 × 平均版本数，可能非常大！

⚠️ 重要：必须配置 State TTL 防止 OOM

### Hive 拉链表与 Versioned Table
Flink 无法直接将 Hive 拉链表识别为 Versioned Table，因为

+ Hive 拉链表是静态快照表，不是流
+ 无操作类型
+ 无事件时间字段，均为业务时间，非变更时间
+ Flink 读 Hive 表默认是 Batch 模式，无法产生 Changelog



如何将 Hive 拉链表转变为 Versioned Table？

解决方案：完成 “拉链表 → Changelog 流” 的转换

有 2 条主流路径

1. （推荐）通过 FlinkCDC 同步源系统  → 构建实时 Changelog
    - 不依赖 Hive 拉链表，而是直接监听源数据库（如 MySQL）的 Binlog
    - 用 Flink CDC 将用户表变更实时捕获为 Changelog 流
2. 定期全量读 Hive 拉链表 + 手动构建 Changelog，过于复杂，不建议

## Event-Time Temporal Join（事件时间时态关联）
+ 用途：关联事件发生时的历史维度状态，用于精确回溯历史状态，是高级实时数仓的核心。
+ 维表要求：Changelog 流（如 CDC 数据），带事件时间
+ 时间属性：使用事件时间（`WATERMARK`）

```plsql
-- 示例：订单流关联汇率变化流（查下单时的汇率）
SELECT o.order_id, o.amount, r.rate
FROM order_stream AS o
JOIN currency_rates FOR SYSTEM_TIME AS OF o.row_time AS r
  ON o.currency = r.currency;
```

核心语法：`JOIN ... FOR SYSTEM_TIME AS OF <time_attr>`

前提：维表必须是 Changelog 流

+ 维表需来自 CDC（Change Data Capture），如 Debezium + Kafka；
+ 数据包含 完整变更日志`（INSERT/UPDATE/DELETE）`；
+ 必须定义【事件时间】和【`WATERMARK`】



维表示例：汇率变化流

```plsql
CREATE TABLE currency_rates (
  currency STRING,
  rate DECIMAL(10, 4),
  update_time TIMESTAMP(3),
  WATERMARK FOR update_time AS update_time - INTERVAL '5' SECOND,
  PRIMARY KEY (currency) NOT ENFORCED
) WITH (
  'connector' = 'kafka',
  'value.format' = 'debezium-json'
);
```



关联逻辑：

+ 订单时间：`2024-06-01 10:30:00`
+ 汇率变更：
    - `2024-06-01 10:00:00` → `USD/CNY = 7.20`
    - `2024-06-01 11:00:00` → `USD/CNY = 7.25`
+ 结果：订单关联到 `7.20`（因为下单时汇率尚未更新）

关联过程

Flink 内部将 Changelog 流物化为按时间版本化的表：

| currency | rate | valid_from | valid_to |
| --- | --- | --- | --- |
| USD | 7.20 | 10:00:00 | 11:00:00 |
| USD | 7.25 | 11:00:00 | +∞ |


1. 提取 `order_time = 10:30:00`；
2. 自 Versioned Table 中查找满足条件的版本

```plsql
WHERE currency = 'USD'
  AND valid_from <= '10:30:00'
  AND '10:30:00' < valid_to
```

3. 匹配到`rate = 7.20`因为 `valid_from ≤ 10:30 < valid_to` 
4. 输出结果

# Lookup Join（维表 Join）
## Lookup 简介
Lookup 指的是：从一条流数据（主表）出发，根据某个字段（如 `user_id`），实时去外部系统（维表）查询对应的维度信息（如 `user_name`, `city`）的过程。

这本质上是一种 “流 + 维表”的实时关联（Real-time Enrichment）。每条流记录触发一次对外部系统的点查（Lookup）



并非所有数据源都能作为维表用于实时 Lookup Join！只有那些支持“**按主键快速点查**（Point Query）”的外部系统，才被称为“可 lookup 的”

+ 可 Lookup 的系统（支持点查）：MySQL/PostgreSQL、HBase、Redis、Elasticsearch
    - 这些系统都支持 O(1) 或 O(log n) 的低延迟点查，适合高并发实时查询
+ 不可 Lookup 的系统（不支持高效点查）
    - Hive 表：存储在 HDFS 上，查询需启动 MapReduce/Tez/Spark 作业，延迟高（秒~分钟级）
    - Kafka Topic：本质是日志流，无法按 key 随机读取历史数据
    - 普通文件（CSV/Parquet）：无索引，需全表扫描
    - 未建索引的数据库大表：全表扫描满
+ 判断标准：“能否在 < 100ms 内，根据主键返回单条记录？”如果不能，就不适合作为维表进行 Lookup Join

## Flink 实现 Lookup 的技术原理
Flink 通过 Lookup Join 算子 实现维表关联，其内部机制如下：

1. 同步 Lookup（不推荐）
    - 每条流记录到来 → 阻塞等待维表查询返回；
    - 吞吐低，易背压
2. 异步 Lookup（Async Lookup，推荐）
    - 使用 Flink Async I/O API；
    - 多条记录并发查询，结果通过回调处理；
    - 高吞吐、低延迟
    - 维表 DDL 需开启异步查询：`'lookup.async' = 'true'`
3. 缓存优化（LRU/全量）
    - 减少对外部系统的重复查询；
    - 通过`'lookup.cache.max-rows' `配置

## Processing-Time Temporal Join
基于官方文档：“Processing-Time Temporal Join” 等价于 “Lookup Join”

所有基于外部维表（JDBC/HBase/Redis）的 Join，在 Flink 中都属于 Processing-Time Temporal Join，无论是否显式写出 `proc_time`

### Flink 如何隐式处理时间
场景：普通 Join，无 `PROC_TIME`

```plsql
-- 看似没有 proc_time
SELECT o.*, u.name
FROM orders AS o
JOIN mysql_users AS u
  ON o.user_id = u.user_id;
```

Flink 内部完成

1. 检测到 mysql_users 是 可 lookup 的维表（通过 connector 类型判断）；
2. 自动将此 Join 重写为 Temporal Join；
3. 隐式注入一个处理时间属性（即使 DDL 中没定义）；
4. 等价于

```plsql
SELECT o.*, u.name
FROM orders AS o
JOIN mysql_users FOR SYSTEM_TIME AS OF PROCTIME() AS u
  ON o.user_id = u.user_id;
```

### 以下场景必须显式写`PROCTIME()`
1. 场景 1：需要多次关联不同维表
    - 若不显式定义，Flink 会为每个 Join 生成独立的 `PROCTIME()`，可能导致两次查询时间不一致（虽然相差毫秒，但理论上存在窗口错位）

```plsql
-- 显式定义 proc_time，避免歧义
CREATE TABLE orders (
  order_id STRING,
  user_id STRING,
  product_id STRING,
  ptime AS PROCTIME()  -- 显式声明
);

SELECT 
  o.*,
  u.name,
  p.category
FROM orders AS o
JOIN users FOR SYSTEM_TIME AS OF o.ptime AS u ON o.user_id = u.user_id
JOIN products FOR SYSTEM_TIME AS OF o.ptime AS p ON o.product_id = p.id;
```

2. 场景 2：与 Event-Time 流混合使用时，需要显式给出处理时间字段，否则 Flink 不清楚用哪个时间做 Lookup
3. 场景 3：调试与可读性。显式写出`FOR SYSTEM_TIME AS OF o.proc_time`能让代码意图更清晰

### `PROCTIME()`：时间锚点
`PROCTIME()` 在 Flink SQL 的 Lookup Join（即 Processing-Time Temporal Join） 中扮演着时间锚点（Time Anchor） 的核心角色。虽然 Lookup Join 本质上是“查当前维表最新值”，但 `PROCTIME()` 并非可有可无的装饰，而是 Flink 时态语义模型的必要组成部分。



`PROCTIME()` 

核心作用：为每条流记录打上一个“查询发起时刻”的时间戳，作为 Lookup Join 的时间锚点，使关联操作符合 Flink 统一的 Temporal Join 语义模型。

作用

1. 提供 Temporal Join 的语法合法性：Flink 要求所有 Temporal Join 必须指定一个时间属性（Time Attribute），即使 Flink 可以隐式推导，底层仍需要该属性
    - `<time_attr>` 必须是 Processing Time Attribute 或 Event Time Attribute；
    - 对于维表（非 Changelog 流），只能使用 Processing Time；
    - `PROCTIME() `是生成 Processing Time Attribute 的唯一方式。

```sql
JOIN dim_table FOR SYSTEM_TIME AS OF <time_attr> ...
```

2. 确定“查询维表的时刻”：虽然维表是“当前最新”，但 “当前”是相对的——它指的是**这条记录被处理的那一刻**
    - 例：如果维表在 10:00:03 更新了 `user_id=100` 的地址，那么：
        * Record A 查到旧地址；Record B 查到新地址。
        * `PROCTIME() `精确捕获了这一查询发起时刻，保证结果与处理顺序一致。

| 记录 | 到达时间 | `PROCTIME()` 值 | 查询维表的时刻 |
| --- | --- | --- | --- |
| Record A | 10:00:00.123 | 10:00:00.123 | 10:00:00.123 |
| Record B | 10:00:05.456 | 10:00:05.456 | 10:00:05.456 |


3. 支持多维表关联的一致性（最重要！）：同 5.3.2 场景 1



`PROCTIME()` 的工作机制

1. 生产 Processing Time
2. 触发 Lookup 操作：当记录到达 Join 算子
    1. 提取 `proc_time` 字段值（如 `1717020000123L`）；忽略该值的具体数值（因为维表无历史版本）
    2. 立即发起异步查询（Async I/O）到维表
    3. 返回结果后输出

注意：在 Lookup Join 中，`proc_time` 的具体值其实不重要，重要的是它作为一个 “触发信号”，告诉 Flink：“现在该去查维表了”

## 如何判断一个 Connector 是否支持 Lookup？
查看 Flink 官方文档中该 Connector 是否支持以下特性：

1. 支持` PRIMARY KEY` 声明；
2. 支持 `lookup` 相关参数，如：
    - `'lookup.async'`
    - `'lookup.cache.max-rows'`
    - `'lookup.batch-size'`
3. 文档明确说明可用于 “Temporal Table Join” 或 “Lookup Join”

例如，Flink JDBC Connector 文档 明确写道：

> “The JDBC table can be used for lookup joins with temporal tables.”
>

Hive Connector 文档 则从未提及 Lookup Join 支持



生产建议：维表必须来自 OLTP 或 KV 存储，绝不能直接用 Hive 做维表

## 为什么 Hive 表不能做维表
原因

1. 存储位置：Hive 表数据在 HDFS，无随机读能力
2. 查询引擎：需启动批处理引擎（如 Tez），无法做到毫秒级响应
3. Flink 设计：Flink 的 `HiveCatalog` 仅用于元数据读取和流式写入，不支持 Lookup Join



若需用 Hive 中的维度数据，应先将其同步到可 lookup 的系统中：

+ Hive → HBase（通过 Sqoop / Spark）
+ Hive → MySQL（通过 DataX）
+ Hive → Redis（通过自定义程序）

## 示例
示例：使用曝光用户日志流（show_log）关联用户画像维表（user_profile）关联到用户的维度之后，提供给下游计算分性别，年龄段的曝光用户数使用

```sql
CREATE TABLE show_log (
    log_id BIGINT,
    `timestamp` as cast(CURRENT_TIMESTAMP as timestamp(3)),
    user_id STRING,
    proctime AS PROCTIME()
)
WITH (
  'connector' = 'datagen',
  'rows-per-second' = '10',
  'fields.user_id.length' = '1',
  'fields.log_id.min' = '1',
  'fields.log_id.max' = '10'
);

CREATE TABLE user_profile (
    user_id STRING,
    age STRING,
    sex STRING
    ) WITH (
  'connector' = 'redis',
  'hostname' = '127.0.0.1',
  'port' = '6379',
  'format' = 'json',
  'lookup.cache.max-rows' = '500',
  'lookup.cache.ttl' = '3600',
  'lookup.max-retries' = '1'
);

CREATE TABLE sink_table (
    log_id BIGINT,
    `timestamp` TIMESTAMP(3),
    user_id STRING,
    proctime TIMESTAMP(3),
    age STRING,
    sex STRING
) WITH (
  'connector' = 'print'
);

-- lookup join 的 query 逻辑
INSERT INTO sink_table
SELECT 
    s.log_id as log_id
    , s.`timestamp` as `timestamp`
    , s.user_id as user_id
    , s.proctime as proctime
    , u.sex as sex
    , u.age as age
FROM show_log AS s
LEFT JOIN user_profile FOR SYSTEM_TIME AS OF s.proctime AS u
ON s.user_id = u.user_id
```

## 维表 Join 注意事项
+ 维表必须支持高效点差
    - 禁止使用 Hive、Kafka、普通文件作为维表；
    - 必须使用 OLTP 或 KV 存储：MySQL、HBase、Redis、Elasticsearch 等；
    - 验证标准：单次查询 P99 延迟 < 50ms
+ 主键必须匹配
    - 维表 DDL 中必须声明`PRIMARY KEY`；
    - Join 条件必须使用该主键字段；
    - 否则 Flink 无法生成高效查询
+ Checkpoint 超时必须 > Lookup 超时
    - Flink 要求在一次 Checkpoint 触发后，所有正在处理的异步请求（包括维表 Lookup）必须在 Checkpoint 超时前完成或取消；如果 Lookup 的超时时间比 Checkpoint 超时还长，那么当 Lookup 卡住时，Checkpoint 就会因“等待异步操作完成”而超时失败
    - 推荐：Checkout 超时时间是 Lookup 超时时间的至少 2-4 倍
+ 会发生实时新建/更新的维表，应建立数据延迟监控机制，防止出现流表数据先于维表数据到达，导致关联不到维表数据
+ 同一条数据关联到的维度数据可能不同：
    - 实时数仓中常用的实时维表都是在不断的变化中的，当前流表数据关联完维表数据后，如果同一个 key 的维表的数据发生了变化，**已关联到的维表的结果数据不会再同步更新**。
    - 这是有可能会影响数据质量的。所以在评估实时任务时要考虑到这一点。

## 维表 Join 优化
作业配置

```sql
-- Checkpoint 必须 > lookup timeout
SET 'execution.checkpointing.interval' = '2 min';
SET 'execution.checkpointing.timeout' = '5 min';

-- 异步缓冲区防背压
SET 'table.exec.async-lookup.buffer-capacity' = '500';
```

维表配置

```sql
-- 启用异步，设置超时和重试时长
'lookup.async' = 'true',
'lookup.async.timeout' = '20 s',
'lookup.max-retries' = '2',

-- 启用缓存
'lookup.cache.max-rows' = '300000',
'lookup.cache.ttl' = '20 min'

-- 开启批量查询
'lookup.batch-size' = '400',
'lookup.buffer-timeout' = '150 ms',
```
