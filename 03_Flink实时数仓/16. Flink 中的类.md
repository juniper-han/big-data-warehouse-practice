# POJO
## POJO 是什么
**POJO**（Plain Old Java Object）中文常译为 **“普通 Java 对象”**。顾名思义，它是指一个

+ 不依赖于特定框架：比如不使用框架的注解
+ 不继承特定父类、不实现特殊接口的

最普通的 Java 对象。

它的核心思想是：**追求简单和纯粹**，让对象只关注承载数据和行为，而不被框架所绑架。

典型 POJO

```java
// 一个简单的POJO，代表用户
public class User {
    // 1. 私有字段（属性）
    private String name;
    
    // 2. 公有默认构造方法（无参构造）
    public User() {}
    
    // 3. 公有getter和setter方法
    public String getName() {
        return name;
    }
    public void setName(String name) {
        this.name = name;
    }
    
    // 4. 可选的：重写toString(), equals(), hashCode() 等方法
    @Override
    public String toString() {
        return "User{name='" + name + "', age=" + age + "}";
    }
}
```

## POJO 在 Flink 中的优势
在 Flink 中，POJO 是**非常重要的数据类型**。

相比难以管理的 `Tuple`类型和性能较低的 `Row`类型，POJO 提供了**清晰的字段命名和强大的类型安全**。

Flink 通过**反射**自动识别 POJO 的结构，这带来了两大好处：

1. **字段名引用**：在 DataStream API 或 Table API 中，可以直接使用字段名，代码可读性极高。

```java
// DataStream API 按键分组
dataStream.keyBy(user -> user.getName()); // 直接引用字段名

// Table API 字段引用
tableEnv.fromDataStream(dataStream, $("name")); // 使用字段名"name"
```

2. **类型安全**：编译器能在编译时检查类型错误，避免运行时异常。

```java
String name = user.getName(); // 编译时已知是String类型，安全
```

## Flink 认可一个类为 POJO 的规则
要让 Flink 的反射机制能够正确识别你的类，它必须满足以下条件：

1. **公有类（Public Class）**。
2. **包含一个公有的无参构造方法**（默认构造方法）。
3. **所有字段都是公有的，或者有标准的 getter 和 setter 方法**。
    - Getter 方法命名：`getField()`（布尔字段可以是 `isField()`）。
    - Setter 方法命名：`setField(value)`。
4. **类中的字段类型必须是 Flink 支持的类型**。
    - 基本类型（如 `int`, `long`, `double`）及其包装类。
    - 其他 POJO。
    - 集合类型（如 `List`, `Map`）。
    - 数组类型。
    - 其他（如 `String`, `Date`, `BigDecimal`）

## 在 Flink 中使用 POJO
### 在 DataStream API 中使用
```java
// 创建一个SensorReading类型的DataStream
DataStream<SensorReading> dataStream = env
    .fromElements(
        new SensorReading("sensor_1", 1677847200000L, 25.6),
        new SensorReading("sensor_2", 1677847200500L, 28.1)
    );

// 使用字段名进行keyBy，非常清晰！
dataStream.keyBy(SensorReading::getSensorId);
```

### 在 Table&SQL API 中使用
```java
// 将DataStream转换为Table
Table table = tableEnv.fromDataStream(dataStream);

// 在SQL中直接使用POJO的字段名
tableEnv.executeSql(
    "SELECT sensorId, AVG(temperature) " +
    "FROM " + table +
    " GROUP BY sensorId"
);
```

# Row 类
## 简介
Row：**通用的、动态类型的数据结构**

+ Flink 中表示**一行数据**的核心数据结构
    - 类似于关系型数据库中的一行记录，常用于 Table&SQL API
    - DataStream 也可使用，类似于一个**可以动态扩展的元组**
+ `Row`对象可以包含任意数量、任意类型的字段
    - 但通常需要在表的模式中定义清楚
    - 支持嵌套结构（如 `Row`中包含另一个 `Row`）。
+ 通过索引来设置和获取 Row 对象的字段值，但没有字段名的概念
    - 字段通过**位置索引**（从 0 开始）或**名称**（如果指定了字段名）访问。
+ 在Flink Table API 中，`Row`通常用于用户自定义函数（UDF）和表函数（UDTF）的输出。
    - 在UDTF中，可以通过`collect`方法输出`Row`，从而产生多行数据、
+ 与 Table API 深度绑定
    - 是 Table API 中数据的**默认内部表示形式**
    - SQL 查询结果的返回类型通常是 `Row`。
+ 序列化优化
    - 针对 Flink 的序列化框架（如 `TypeInformation`）进行了优化，效率高于普通 Java 对象。

## Row 类的构造与使用
### 创建 Row 类对象
创建 Row 的方式

```java
// 创建Row的几种方式
Row row1 = new Row(3);          // 指定字段数量
Row row2 = Row.of("Tom", 25);   // 直接传入值（推荐方式），工厂方法（推荐）
Row row3 = Row.withNames();     // 创建带有字段名的Row（1.14+）
```

### 访问/设置字段
Row 的使用：索引从 0 开始

+ 设置字段值：`row.setField(索引, 值); `
+ 获取字段值：`row.getField(索引);`

```java
Row row = Row.of("Alice", 30, "New York");

// 获取字段值
String name = (String) row.getField(0);      // "Alice" - 索引0
int age = (int) row.getField(1);            // 30 - 索引1
String city = (String) row.getField(2);     // "New York" - 索引2

// 设置字段值
row.setField(0, "Bob");
row.setField(1, 35);

// 获取字段数
int arity = row.getArity();  // 3

// Row的比较
Row rowA = Row.of("A", 1);
Row rowB = Row.of("A", 1);
boolean equal = rowA.equals(rowB);  // true

// 转换为字符串
String str = rowA.toString();  // "A,1"
```



Row 的字段类型：由`getResultType`方法返回的`DataType`决定

+ 在`getResultType`中，使用`DataTypes.createRowType`来定义`Row`的字段类型
+ 例如：`DataTypes.createRowType(DataTypes.STRING, DataTypes.STRING, DataTypes.LONG)`，这表示Row有三个字段，依次是`STRING`、`STRING`、`LONG`
+ `Row`的字段索引必须与`getResultType`中定义的顺序一致

### 与 Table API 交互
```java
// 将DataStream<Row>转换为Table
DataStream<Row> dataStream = env.fromElements(
    Row.of("Alice", 25),
    Row.of("Bob", 30)
);

Table table = tableEnv.fromDataStream(dataStream, $("name"), $("age"));

// 执行SQL查询（返回的仍是Row）
Table result = tableEnv.sqlQuery("SELECT name, age FROM table WHERE age > 26");
DataStream<Row> resultStream = tableEnv.toDataStream(result);
```

### Row 在 FlinkSQL 中使用
作为`TableFunction`，输出的`Row`会作为Flink SQL表的一行：

```sql
-- 在Flink SQL中使用这个UDTF
SELECT store_code, alias, id
FROM MyTable, 
LATERAL TABLE(AlienUDTF(json_str)) AS T(store_code, alias, id);

-- 输出结果：
-- store_code    | alias              | id
-- --------------+--------------------+------
-- "BJ_SF_001"   | "北京顺丰分拨中心" | 1001
-- "SH_SF_002"   | "上海顺丰分拨中心" | 1002
```

## Row 的运行时类型信息
Row 的运行时类型信息：RowTypeInfo  
为了确保 Flink 能正确序列化和处理 `Row`，需要为其指定 `RowTypeInfo`（类型信息）。

1. **定义类型信息**

```java
import org.apache.flink.api.common.typeinfo.Types;
import org.apache.flink.api.java.typeutils.RowTypeInfo;

// 定义字段类型
RowTypeInfo typeInfo = new RowTypeInfo(
    Types.STRING,   // 第0个字段：name
    Types.INT,      // 第1个字段：age
    Types.BOOLEAN   // 第2个字段：isStudent
);

// 应用类型信息到DataStream
DataStream<Row> dataStream = env
    .fromElements(Row.of("Alice", 25, true))
    .returns(typeInfo);
```

2. **带字段名的类型信息**

```java
RowTypeInfo namedTypeInfo = new RowTypeInfo(
    new String[]{"name", "age", "isStudent"},  // 字段名
    new TypeInformation[]{Types.STRING, Types.INT, Types.BOOLEAN}
);

// 按名称访问字段
Row row = Row.of("Alice", 25, true);
String name = (String) row.getField("name");  // 需要RowTypeInfo支持
```

## Row 的内存表示
```java
// 假设HBase查询得到的结果：
// store_code = "BJ_SF_001"
// alias = "北京顺丰分拨中心"
// id = "1001"

Row row = new Row(3);
row.setField(0, "BJ_SF_001");
row.setField(1, "北京顺丰分拨中心");
row.setField(2, 1001L);

// 内存中的Row结构：
// +------------------+---------------------+-------+
// | 索引0            | 索引1               | 索引2 |
// +------------------+---------------------+-------+
// | "BJ_SF_001"      | "北京顺丰分拨中心"  | 1001L |
// +------------------+---------------------+-------+
//   类型: String        类型: String         类型: Long
```

## Row vs POJO
| **特性** | **Row** | **POJO** |
| --- | --- | --- |
| **类型安全** | ❌ 运行时类型检查（易出错） | ✅ 编译时类型检查 |
| **字段访问** | 按位置或名称（动态） | 通过`getter`/`setter`（静态） |
| **性能** | ⚠️ 较高序列化开销 | ✅ 优化后的序列化（高效） |
| **可读性** | ❌ 低（需维护字段顺序） | ✅ 高（字段名即代码） |
| **适用场景** | Table API 中间结果、动态结构 | DataStream API、固定结构 |
| **Schema 演化** | ✅ 灵活（适应字段增减） | ❌ 需重新编译 |


```java
// 使用Row的方式
Row row = Row.of("BJ_SF_001", "北京顺丰分拨中心", 1001L);

// 使用POJO的方式（可读性更好）
public class StoreInfo {
    private String storeCode;
    private String alias;
    private Long id;
    // getters/setters
}

StoreInfo info = new StoreInfo("BJ_SF_001", "北京顺丰分拨中心", 1001L);
```

## 总结
+ **Row 优势**：灵活适应动态 Schema，完美适配 Table API 的查询结果。
+ **Row 劣势**：
    - 类型安全性较低，需要手动转换类型（`(String) row.getField(0)`）
    - 性能略逊于 POJO
    - 索引易错，可读性差：通过数字索引访问，容易出错
+ **最佳实践**：
    - 在 **Table API/SQL 结果处理** 和 **动态数据解析** 时优先使用。
        * **预定义 **`RowTypeInfo`****：显式指定类型信息可减少运行时反射开销。
        * **优先使用 **`Row.of()`****：比 `new Row()`+ `setField()`更高效。
    - 在 **DataStream API 中处理固定结构数据** 时，仍推荐使用 POJO。
+ 使用注意事项
    - 复用 Row 对象：在高频调用时，可以考虑复用 Row 对象减少 GC
    - 字段类型匹配：确保`setField`的类型与`getResultType`定义的一致
    - 避免类型转换：尽量直接使用正确的类型，避免频繁转换

# Tuple（元组）
## Tuple 简介
**Tuple** 是一个****

+ **固定长度**
+ **存储多个不同类型元素**

**的容器**。Tuple **不是集合。**

****

在 Flink 中，**Tuple 是一个泛型类**：

+ 从 `Tuple0`（空元组）到 `Tuple25`（最多25个元素）有具体的实现类。
+ 最常用的是 `Tuple2`（两个元素，可表示 Key-Value 对）。
    - `Tuple2<K, V>`**表示一个独立的二元组**，其结构是 `(单个K, 单个V)`
    - `Tuple2<String, Integer> a = Tuple2.of("key", 42)`

Tuple 的特性

| 性质 | 描述 | 示例 |
| --- | --- | --- |
| **固定长度** | 创建时即确定元素个数，不可变。 | `Tuple2`永远有2个元素。 |
| **类型安全** | 通过泛型参数指定每个位置元素的类型。 | `Tuple2<String, Integer>` |
| **按位置访问** | 通过公有字段 `f0`, `f1`, ..., `f24`访问元素。 | `myTuple.f0`获取第一个元素。 |
| **效率高** | 非常轻量级，序列化/反序列化开销低。 | 适用于高性能场景。 |


## Tuple 不是集合！！！
Tuple2 的类定义

```java
public class Tuple2<K, V> extends Tuple {
    public K f0;  // 第一个元素（类型为K）
    public V f1;  // 第二个元素（类型为V）
}
```

+ `f0`**和 **`f1`**各自只能存储一个值**。
+ 如果需要存储多个键值对，应该使用 `List<Tuple2<K, V>>`或 `DataStream<Tuple2<K, V>>`。

```java
// 单个二元组
Tuple2<String, Integer> singlePair = Tuple2.of("age", 25);
String key = singlePair.f0;   // "age"
Integer value = singlePair.f1; // 25

// 多个二元组需要容器存储
List<Tuple2<String, Integer>> pairs = Arrays.asList(
    Tuple2.of("age", 25),
    Tuple2.of("height", 180)
);
```

## 使用 Tuple
### 创建与赋值
```java
import org.apache.flink.api.java.tuple.Tuple2; // 导入二元组
import org.apache.flink.api.java.tuple.Tuple3; // 导入三元组

// 方式1：直接创建并赋值
Tuple2<String, Integer> person = new Tuple2<>("Alice", 25);

// 方式2：使用静态工具方法.of()（更简洁，推荐）
Tuple3<String, Integer, Double> product = Tuple3.of("Laptop", 1, 999.99);

// 方式3：先创建，后设置字段
Tuple2<Long, String> logEntry = new Tuple2<>();
logEntry.f0 = System.currentTimeMillis(); // 设置第一个字段
logEntry.f1 = "Error: File not found";    // 设置第二个字段
```

### 访问元素
```java
// 通过公有字段名直接访问
String name = person.f0;   // "Alice"
Integer age = person.f1;   // 25

// 三元组访问
String itemName = product.f0;
Integer quantity = product.f1;
Double price = product.f2;
```

### 在 DataStream 中使用 Tuple
Tuple 最常见的使用场景是在 `keyBy`操作中。

```java
DataStream<Tuple2<String, Integer>> dataStream = env.fromElements(
    Tuple2.of("Category_A", 100),
    Tuple2.of("Category_B", 200),
    Tuple2.of("Category_A", 50)
);

// 1. 按整个Tuple作为Key（很少用）
dataStream.keyBy(0).sum(1).print(); 
// 输出：("Category_A", 100) -> ("Category_A", 150) -> ("Category_B", 200)

// 2. 更常见：按Tuple的某个字段进行分组（Lambda表达式）
dataStream.keyBy(value -> value.f0).sum(1).print(); // 按f0（品类名称）分组，对f1（数值）求和

// 3. 使用字段表达式（已逐渐被Lambda取代）
dataStream.keyBy("f0").sum("f1").print();
```

## Tuple 优缺点
 **优点**

1. **轻量高效**：底层实现简单，序列化和网络传输开销小，性能好。
2. **快速开发**：无需定义POJO类，适合快速原型开发和小型作业。
3. **泛型安全**：编译时提供类型检查，比 `Row`类型更安全。

**缺点**

1. **可读性差**：代码中充斥 `f0`, `f1`, `f2`等魔法数字，难以直观理解字段含义，降低代码可维护性。
2. **重构困难**：字段顺序是硬编码的，若要增加或调整字段顺序，需要修改所有访问该字段的代码。
3. **长度限制**：最多只能有25个字段（`Tuple25`），虽然不常用到，但也是一个限制。

## Tuple vs POJO vs Row
| 特性 | **Tuple** | **POJO** | **Row** |
| --- | --- | --- | --- |
| **可读性** | ❌ 差（`f0`, `f1`） | ✅ **优**（有字段名） | ⚠️ 中（按位置或名称） |
| **类型安全** | ✅ **优**（编译时检查） | ✅ **优**（编译时检查） | ❌ 差（运行时检查） |
| **性能** | ✅ **优**（极高） | ✅ 优（高） | ⚠️ 中（有反射开销） |
| **灵活性** | ❌ 差（固定长度） | ✅ 优 | ✅ **优**（动态结构） |
| **开发效率** | ✅ 优（无需定义类） | ❌ 差（需定义类） | ✅ 优（无需定义类） |
| **适用场景** | 简单KV对、临时数据 | **生产环境首选** | Table API 中间结果 |


1. **避免使用**：在新的 Flink 项目中，**尽量避免使用 Tuple** 作为主要的数据类型。
2. **临时场景**：仅在快速测试、原型开发或处理简单的 Key-Value 对时临时使用。
3. **生产推荐**：在**生产环境**中，始终优先使用 **POJO**。
    1. POJO 通过有意义的字段名提供了无与伦比的可读性和可维护性，这是大型、复杂项目成功的关键。
4. **API 限制**：某些 Flink 内置算子（如 `Window`操作的某些方法）可能要求返回 `Tuple`类型，这是少数不得不使用 Tuple 的场景。

**总结一句话**：**Tuple 是 Flink 中的“快捷方式”，虽然方便，但为了代码的长远健康，请养成使用 POJO 的好习惯。**
