Flink Metrics（指标系统） 是理解、监控和调优 Flink 应用程序运行状态的关键工具。

# Flink Metrics 简介
## Flink Metrics 目标
Flink Metrics 系统旨在：

+ **实时反映作业运行状态（如吞吐量、延迟、背压等）**
+ **暴露系统资源使用情况（CPU、内存、网络、磁盘 I/O）**
+ 支持故障诊断与性能调优
+ 与外部监控系统无缝集成（如 Prometheus、Datadog、JMX、StatsD 等）

## 指标类型（Metric Types）
Flink 支持以下四种基本指标类型：

| 类型 | 说明 | 典型用途 |
| --- | --- | --- |
| Counter，计数器 | 递增的整数值，可增可减（但通常只增） | 记录处理记录数、失败次数等 |
| Gauge（仪表） | 返回任意类型的**当前值**（可动态变化） | 内存使用量、队列长度、当前时间戳等 |
| Histogram（直方图） | 统计数值分布（如 P50/P95/P99、均值、最大/最小值） | 处理延迟、事件时间差等 |
| Meter（速率计） | 测量事件发生频率（单位时间内的平均速率） | 每秒处理记录数（records/sec） |


注意：Flink 的 Histogram 默认使用 Exponentially Decaying Reservoir 算法，适合流式场景。

## 指标作用域（Metric Scope）
Flink 通过**<font style="background-color:#E8F7CF;">层级化命名空间</font>**组织指标

+ 每个指标都有一个唯一的逻辑标识（<font style="background-color:#E8F7CF;">scope + name</font>）。
+ 作用域（Scope）：决定了指标的“归属上下文”。

指标命名：

+ `<scope_prefix>.job.<job_name>.operator.<operator_name>.task.<subtask_index>.<metric_name>`
+ 作用域 → Flink 任务名（Job）→ 算子名（Operator）→ 子任务名（subtask）→ 指标名称
+ 示例：`job.my-job.operator.MyMap.source-id.numRecordsInPerSecond`

常见作用域层级（从粗到细），具体：

+ System Scope：JVM、CPU、内存等系统级指标
+ JobManager Scope：JobManager 相关指标（如 checkpoint 协调信息）
+ TaskManager Scope：TaskManager 资源使用（如网络缓冲池）
+ Job Scope：整个作业级别的指标（如总处理记录数）
+ Task/Operator Scope：算子或子任务级别的细粒度指标（最常用）



也可以通过 `metrics.scope.*` 配置项自定义命名格式（如适配 Prometheus 的 label 结构）。

# 核心指标分类汇总
## 状态（State）相关指标
状态相关的指标包括：Gauge（主要）、Counter、Histogram 指标（仅针对 RocksDB）

### 状态：Gauge 指标
作用域：Task/Operator

1. `totalStateSize`：当前子任务的总状态大小（bytes）
    1. 计算公式：所有状态 backend 序列化后估算大小
    2. 监控用途：状态增长趋势、Checkpoint 大小预估
    3. 判断：正常应稳定或缓慢增长
        1. 突增 → 数据倾斜或状态未清理
        2. 接近磁盘/内存上限 → 风险
    4. Web UI 位置：Job → Checkpoints → Details（显示为“State Size”）
2. `managedKeyedState`：Keyed State 大小（bytes）
    1. 计算公式：Keyed state backend 报告
    2. 监控用途：Keyed 状态膨胀监控
    3. 判断：同指标`totalStateSize`
3. `managedOperatorState`：Operator State 大小（bytes）
    1. 计算公式：Operator state backend 报告
    2. 监控用途：Broadcast/Union state 监控
    3. 判断：通常较小；突增需警惕
4. `numStateEntries`：状态条目总数（key-value 对数）
    1. 计算公式：
        1. RocksDB：`rocksdb.estimate-num-keys`
        2. Heap：`Map.size()`
    2. 监控用途：状态规模、内存压力
    3. 判断：与业务量成比例；
        1. 异常增长 → 未设 TTL



💡注意： `totalStateSize` 是 Checkpoint 大小的主要决定因素，直接影响：

+ Checkpoint 耗时
+ 故障恢复时间
+ 存储成本（HDFS/S3）

### 状态 Counter 指标
1. `numberOfRestoredStateEntries`：从状态恢复的条目数
    1. 计算公式：<font style="color:rgb(6, 10, 38);">恢复时统计</font>
    2. <font style="color:rgb(6, 10, 38);">作用域：Task/Operator</font>
    3. <font style="color:rgb(6, 10, 38);">监控用途：</font><font style="color:rgb(6, 10, 38);">验证状态恢复完整性</font>
    4. <font style="color:rgb(6, 10, 38);">正常判断：应 ≈ 上次</font>`<font style="color:rgb(6, 10, 38);"> numStateEntries</font>`
    5. <font style="color:rgb(6, 10, 38);">Web UI 位置：Job → Overview（启动时可见）</font>

### <font style="color:rgb(6, 10, 38);">状态 Histogram 指标（</font>**<font style="color:rgb(6, 10, 38);">RocksDB 特有）</font>**
仅当 `state.backend: rocksdb` 时生效，作用域均为 Task/Operator

1. `rocksdb.block.cache.hit.count`：Block Cache 命中次数
    1. 统计内容：Counter（但部分版本暴露为 Histogram）
    2. 监控用途：缓存效率
    3. 判断：Hit rate = hit / (hit + miss) > 90%
2. `rocksdb.block.cache.miss.count`：Block Cache 未命中次数
    1. 统计内容：Counter（但部分版本暴露为 Histogram）
    2. 监控用途：缓存效率
    3. 判断：越低越好
3. `rocksdb.bytes.read`：从磁盘读取的字节数
    1. 监控用途：I/O 压力
    2. 判断：突增 → 缓存失效或大范围 scan
4. `rocksdb.bytes.written`：写入磁盘的字节数
    1. 监控用途：Checkpoint/Flush 压力
    2. 判断：与 `totalStateSize` 正相关
5. `rocksdb.compaction.time`：Compaction 耗时（ms）
    1. 监控用途：后台压缩对性能影响
    2. 判断：P99 < 1s；
        1. 持续高 → 磁盘慢或写放大严重



注意：RocksDB 指标默认不全开启，需在 `flink-conf.yaml `中显式启用：

```yaml
state.backend.rocksdb.metrics.block-cache: true
state.backend.rocksdb.metrics.compaction: true
# 其他详见官方文档
```

## Checkpoint 指标
作用域：JobManager / Job

### Checkpoint Gauge 指标
1. `lastCheckpointDuration`：最近一次 checkpoint 耗时（ms）
    1. 监控用途：Checkpoint 性能
    2. 判断：
        1. 正常：小于 checkpoint interval（如 1min 内）
        2. 异常：突增 → 状态过大或 HDFS 慢
    3. Web UI 位置：Job → Checkpoints
2. `lastCheckpointSize`：最近一次 checkpoint 总大小（bytes）
    1. 监控用途：存储成本、恢复时间
    2. 判断：应 ≈ sum(`totalStateSize`)
3. `lastCheckpointExternalPath`：最近 checkpoint 的外部路径
    1. 监控用途：用于自动化运维
    2. 判断：正常应为非空且可访问
4. `checkpointProgress`：当前 checkpoint 进度（0~1）
    1. 用途：实时跟踪 checkpoint
    2. 判断：应快速完成；卡住 → 问题
5. `numberOfInProgressCheckpoints`：
    1. 用途：并发控制
    2. ≤1（除非配置并发 checkpoint）

### Checkpoint Counter 指标
1. `**numberOfCompletedCheckpoints**`：成功触发并完成 checkpoint 的次数
    1. 用途：监测 Checkpoint 健康度
    2. 作用域：JobManager / Job
    3. 判断
        1. 正常：稳定增长
        2. 异常：若长时间不变 → Checkpoint 阻塞或失败
    4. Web UI 位置：Job → Checkpoints
2. `**numberOfFailedCheckpoints**`：失败的 checkpoint 累计次数
    1. 用途：监测 Checkpoint 故障频率
    2. 作用域：JobManager / Job
    3. 判断
        1. 正常：应为 0 或极低；
        2. 异常：>0 需立即排查
    4. Web UI 位置：Job → Checkpoints
3. `**restoredCheckpoints**`：作业重启时，从 checkpoint 恢复的次数
    1. 用途：Flink 作业故障从 Checkpoint 恢复的频率
    2. 作用域：JobManager / Job
    3. 判断
        1. 正常：生产中应 ≤1（首次启动不算）
        2. 异常：频繁恢复 → 不稳定
    4. Web UI 位置：Overview / Job

### Checkpoint Histogram 指标
Web UI 位置均为：Job → Checkpoints → Details

指标介绍

1. `checkpointAlignmentTime`：Barrier 对齐耗时（ms）
    1. 统计内容：从第一个 barrier 到最后一个 barrier 到达的时间
    2. 监控用途：背压对 checkpoint 影响
    3. 判断：应接近 0：理想
        1. >1s → 存在背压
        2. 持续高 → 需优化算子或资源
2. `checkpointStartDelayNanos`：从 checkpoint 触发到实际开始的时间（纳秒）
    1. 统计内容：调度延迟
    2. Checkpoint 调度效率
    3. 判断：应极小（微秒级）；
        1. 大值 → TM 过载

## 背压（Backpressure）相关指标
⚠️ 注意：Flink 没有名为 backpressure 的直接指标，而是通过 网络缓冲区、算子繁忙度、队列长度等间接指标推断背压。

### 背压 Gauge 指标
1. `busyTimeMsPerSecond`：每秒内算子处于“忙碌”状态的毫秒数
    1. 作用域：Task / Operator
    2. 计算公式：基于处理线程的采样（每秒统计实际执行时间）
    3. 监控用途：最核心的背压代理指标
    4. 判断：0–300ms：空闲；700–1000ms：高负载；
        1. 持续 >900ms 且下游更低 → 背压发生在本算子
    5. Web UI 位置：Job → Task → Metrics
2. `idleTimeMsPerSecond`：每秒空闲时间（= 1000 - `busyTime`）
    1. 作用域：Task / Operator
    2. 用途：反向验证负载
    3. 判断：越高越健康（非源头算子）
    4. Web UI 位置：Job → Task → Metrics
3. `buffers.inputQueueLength`：输入缓冲队列中等待处理的 Buffer 数量，实时计数
    1. 作用域：Task（每个 Input Gate）
    2. 用途：直接反映上游发送快于本 task 处理速度
    3. 判断： 持续 = 0：无背压；持续 > 0：存在背压
        1. 接近 `network.memory.buffers-per-channel` → 严重阻塞
    4. Web UI 位置：Job → Task → Metrics（需展开 input gate）
4. `buffers.outputQueueLength`：输出缓冲队列中未被下游拉取的 Buffer 数，实时计数
    1. 作用域：Task（每个 Result Partition）
    2. 用途：下游消费慢导致本 task 积压
    3. 判断：同`buffers.inputQueueLength`
    4. Web UI 位置：同`buffers.inputQueueLength`
5. `availableMemorySegments`：全局可用网络内存段数量
    1. 作用域：TaskManager
    2. 计算公式：`totalMemorySegments - usedMemorySegments`
    3. 用途：集群级背压信号
    4. 判断：> 20% of total：健康
        1. < 5%：严重背压，可能丢数据或 checkpoint 失败
    5. Web UI 位置：Task Managers → Metrics
6. `exclusiveBuffersUsage`：每个 channel 专用 buffer 使用率（Flink 1.16+）
    1. 作用域：Task
    2. 计算公式：已用 / 配置值
    3. 用途：细粒度通道背压
    4. 判断：接近 1.0 → 单流瓶颈

### 背压 Counter 指标
1. `numRecordsDropped`：因背压或资源不足丢弃的记录数
    1. 作用域：Task（仅部分 Source/Sink）
    2. 计算公式：条件计数（如 Kafka source 限速丢弃）
    3. 用途：监控数据完整性风险
    4. 判断：应为 0；
        1. >0 需立即告警

💡 注：Flink 默认不丢数据（背压会反压到 source），但某些 connector（如 UDP source）或自定义逻辑可能丢弃。

### 如何通过指标定位背压源头
1. 找到 `busyTimeMsPerSecond` 最高的算子（通常是瓶颈）
2. 检查其 下游算子的 `inputQueueLength` 是否高 → 若是，则瓶颈在下游
3. 检查其 上游算子的 `outputQueueLength` 是否高 → 若是，则本算子是瓶颈

## I/O（输入/输出）相关指标
I/O 指标主要反映【数据吞吐】、【网络带宽】、【序列化开销】，是容量规划和性能分析的基础。

### Meter：核心吞吐指标
1. `numRecordsInPerSecond`：每秒接收记录数
    1. 监控用途：吞吐量监控，输入吞吐能力
    2. 作用域：Task / Operator
    3. 计算公式：默认使用 1 分钟 EWMA（类似 Linux load average），反映近期平均速率，避免瞬时抖动
    4. 判断
        1. 正常：数值稳定，符合业务周期
        2. 值突降：源问题或背压
        3. 值突升：流量高峰
    5. Web UI 位置：Job → Task → Metrics
2. `numRecordsOutPerSecond`：每秒输出记录数
    1. 监控用途：输出吞吐、端到端吞吐
    2. 判断：通常 ≈ `numRecordsInPerSecond`（除非聚合/过滤）
    3. 作用域、计算公式、Web UI 位置均同上
3. `numBytesInPerSecond`：每秒输入字节数
    1. 监控用途：网络带宽使用率（入）
    2. 判断：结合网络限速判断是否瓶颈（如 1Gbps ≈ 120MB/s）
    3. 作用域、计算公式、Web UI 位置均同上
4. `numBytesOutPerSecond`：每秒输出字节数
    1. 监控用途：网络带宽使用率（出）
    2. 判断：结合网络限速判断是否瓶颈（如 1Gbps ≈ 120MB/s）
    3. 作用域、计算公式、Web UI 位置均同上



📌 关键洞察：

+ 若 `numRecordsInPerSecond` 高但 `numBytesInPerSecond` 低 → 小事件高频
+ 若 `numBytesInPerSecond` 高但 `numRecordsInPerSecond` 低 → 大事件低频（如视频元数据）

### Counter：累计 I/O 量
1. `**numRecordsIn**`：算子接收到的**记录总数**
    1. 用途：数据流入完整性校验
    2. 判断：应单调递增；重启后重置
        1. 正常：持续增长
        2. 异常：停滞 → 源或上游故障
    3. 作用域：Task/Operator
    4. Web UI 的位置：Job → Task → Metrics
2. `**numRecordsOut**`：算子输出的记录总数
    1. 用途：数据流出完整性、ETL 正确性验证
    2. 判断：与下游 `numRecordsIn` 一致（除分区边界）
        1. 正常：通常 ≥ `numRecordsIn`（除非过滤）
        2. 异常：若远小于 → 逻辑错误或背压丢弃（一般不会丢）
    3. 作用域、Web UI 的位置同上
3. `**numBytesIn**`：接收的网络字节数，是序列化后字节累加的值
    1. 用途：总带宽成本估算
    2. 判断：结合吞吐看数据密度
    3. 作用域、Web UI 的位置同上
4. `**numBytesOut**`：发送的网络字节数，是序列化后字节累加的值
    1. 用途：网络带宽消耗
    2. 判断：结合吞吐看数据密度
    3. 作用域、Web UI 的位置同上



<font style="color:rgba(6, 10, 38, 0.7);">⚠️</font><font style="color:rgba(6, 10, 38, 0.7);"> </font>注意：`numRecordsIn`/`Out` 在 Source/Sink 处有特殊语义：

+ Source 的 `numRecordsOut` = 总产出
+ Sink 的 `numRecordsIn` = 总消费

### Gauge：I/O 延迟与队列
1. `currentInputWatermark`：当前输入水位线（ms）
    1. 作用域：Task / Operator
    2. 计算公式：所有 input channel watermark 的 min()
    3. 监控用途：事件时间进度
    4. 判断：应随时间推进；卡住 → 某 source 无数据或事件时间异常
2. `currentOutputWatermark`：当前输出水位线（ms）
    1. 作用域：Task / Operator
    2. 计算公式：本算子发出的 watermark
    3. 监控用途：下游事件时间基准
    4. 判断：应 ≥ `currentInputWatermark`
3. `buffers.inputExclusiveBuffersUsed`：每个 input channel 专用 buffer 使用数
    1. 作用域：Task
    2. 计算公式：实时计数
    3. 监控用途：单流拥塞检测
    4. 判断：接近配置上限 → 需扩容或调优

## 网络与缓冲区指标
均为 Gauge 指标，作用域为 TaskManager 或 Task

1. `availableMemorySegments`：可用网络内存段数量
    1. 计算公式：`total - used`
    2. 用途：背压核心指标，比 `busyTime` 更底层的背压信号，反映网络层是否阻塞
    3. 判断：正常应 > 20% of total；
        1. 接近 0 → 严重反压
2. `totalMemorySegments`：网络内存段总量
    1. 由 `taskmanager.network.memory.*` 决定，容量基线，固定值
3. `inputQueueLength`：输入队列长度（每个 input gate）
    1. 计算公式：当前 buffer 数
    2. 监控用途：下游处理慢
    3. 判断：持续 > 0 → 背压
    4. Web UI 位置：Job → Task → Metrics
4. `outputQueueLength`：输出队列长度
    1. 计算公式：当前 buffer 数
    2. 监控用途：上游发送快于下游接收
    3. 判断：持续 > 0 → 背压
    4. Web UI 位置：Job → Task → Metrics

## **<font style="color:rgb(6, 10, 38);">JVM 与系统资源指标（Gauge）</font>**
作用域：JobManager / TaskManager

Web UI 位置：Task Managers → Metrics



系统资源指标介绍

1. `Status.JVM.Memory.Heap.Used`：JVM 堆内存已用（bytes）
    1. 计算公式：`MemoryMXBean.getHeapMemoryUsage().getUsed()`
    2. 监控用途：OOM 风险
    3. 判断：< 80% of Max；
        1. 持续上升 → 泄漏
2. `Status.JVM.Memory.Heap.Max`：堆内存上限
    1. `-Xmx` 设置值
    2. 用途：容量规划，固定值
3. `Status.JVM.GarbageCollector.PS_Scavenge.Count`：Young GC 次数
    1. 计算公式：GC MXBean
    2. 监控用途：GC 频率
    3. 判断：正常应为每分钟 < 10 次（视堆大小）
4. `Status.JVM.GarbageCollector.PS_Scavenge.Time`：Young GC 总耗时（ms）
    1. 监控用途：GC 停顿影响
    2. 判断：正常应为：单次 < 50ms；占比 < 5%
5. `Status.JVM.GarbageCollector.PS_MarkSweep.Count/Time`：Full GC 次数/耗时
    1. 监控用途：严重问题信号
    2. 判断：正常应为 0；
        1. >0 → 内存不足或泄漏
6. `Status.JVM.Threads.Count`：线程总数
    1. 计算公式：ThreadMXBean
    2. 监控用途：线程泄漏
    3. 判断：正常应是稳定的
        1. 突增 → 异步操作未管理
7. `Status.JVM.CPU.Load`：进程 CPU 使用率（0~1）
    1. 计算公式：`OperatingSystemMXBean.getProcessCpuLoad()`
    2. 监控用途：CPU 瓶颈
    3. 判断：正常应 < 0.8
        1. 持续 1.0 → 计算密集或死循环

# 生产环境监控体系
Flink 生产环境中最核心、可直接用于监控与告警的完整内置指标清单

+ 指标全名（Prometheus 格式）
+ 监控类型（状态 / IO / 背压 / Checkpoint / JVM / Watermark / 资源）

## 使用建议
1. 指标命名一致性：确保 Flink 的 `metrics.scope` 配置统一（推荐默认格式，Prometheus 可自动解析 label）。
2. 采样频率：Prometheus scrape interval 建议 ≤ 15s，避免漏掉短时背压。
3. 资源隔离：为 Flink Metrics 单独分配 Prometheus 实例或 namespace，避免影响主业务监控。
