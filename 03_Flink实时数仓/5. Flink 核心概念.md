# 分布式缓存
## 分布式缓存简介
分布式缓存最初的思想诞生于 Hadoop 框架：

+ 将一些数据或者文件缓存在 HDFS 上
+ 在分布式环境中让所有的计算节点调用同一个配置文件。

在 Flink 中，分布式缓存类似 Hadoop

+ 目的：在分布式环境中，让每一个 TaskManager 节点保存一份相同的数据或者文件
+ 当前 TaskManager 节点上的 Task 就像读取本地文件一样拉取这些配置。

分布式缓存在实际中的应用

+ 大表 Join 小表：把较小的表进行缓存，在每个 TaskManager 都保存一份，然后进行 Join 操作。

## 使用分布式缓存
使用分布式缓存的步骤

+ 第一步：首先需要在 env 环境中注册一个文件，该文件可以来源于本地，也可以来源于 HDFS ，并且为该文件取一个名字。
+ 第二步：在使用分布式缓存时，可根据注册的名字直接获取



举例：把一个本地的 `distributedcache.txt `文件注册为 `distributedCache`，在下面的 `map` 算子中直接通过这个名字将缓存文件进行读取并且进行了处理。

```java
public static void main(String[] args) throws Exception {

final ExecutionEnvironment env = ExecutionEnvironment.getExecutionEnvironment();
   env.registerCachedFile("/Users/wangzhiwu/WorkSpace/quickstart/distributedcache.txt", "distributedCache");
       //1：注册一个文件,可以使用hdfs上的文件 也可以是本地文件进行测试
       DataSource<String> data = env.fromElements("Linea", "Lineb", "Linec", "Lined");

       DataSet<String> result = data.map(new RichMapFunction<String, String>() {
           private ArrayList<String> dataList = new ArrayList<String>();

           @Override
           public void open(Configuration parameters) throws Exception {
               super.open(parameters);
               //2：使用该缓存文件
               File myFile = getRuntimeContext().getDistributedCache().getFile("distributedCache");
               List<String> lines = FileUtils.readLines(myFile);
               for (String line : lines) {
                   this.dataList.add(line);
                   System.err.println("分布式缓存为:" + line);
               }
           }

           @Override
           public String map(String value) throws Exception {
               //在这里就可以使用dataList
               System.err.println("使用datalist：" + dataList + "-------" +value);
               //业务逻辑
               return dataList +"：" +  value;
           }
       });

       result.printToErr();
   }
```

## 注意事项
+ 缓存的文件在任务运行期间最好是**只读状态**，否则会造成数据的一致性问题。
+ 缓存的文件和数据不宜过大，否则会影响 Task 的执行速度，在极端情况下会造成 OOM。

# 故障恢复和重启策略
自动故障恢复是 Flink 提供的一个强大的功能

在实际运行环境中，我们会遇到各种各样的问题从而导致应用挂掉，比如我们经常遇到的非法数据、网络抖动等。

Flink 提供了强大的可配置故障恢复和重启策略来进行自动恢复。

## 故障恢复
Flink 支持不同级别的故障恢复策略，配置文件中的配置项：  
`jobmanager.execution.failover-strategy`   
有两种取值

+ `full`：Task 发生故障，那么该任务的所有 Task 都会发生重启
    - 大作业包含几百个 Task，整个任务重启会导致数据延迟
+ `region`：局部重启策略，只重启有问题的部分 Task

局部重启策略：

+ Flink 把任务分成不同的 Region，当某一个 Task 发生故障时，Flink 会计算需要故障恢复的最小 Region
+ Flink 在判断需要重启的 Region 时，采用了以下的判断逻辑：
    - 发生错误的 Task 所在的 Region 需要重启；
    - 如果当前 Region 的依赖数据出现损坏或者部分丢失，那么生产数据的 Region 也需要重启；
    - 为了保证数据一致性，当前 Region 的下游 Region 也需要重启。

## 重启策略
Flink 提供了多种类型和级别的重启策略，常用的重启策略包括：

+ 固定延迟重启策略
+ 失败率重启策略
+ 无重启策略

Flink 在判断使用的哪种重启策略时做了默认约定

+ 如果用户配置了 checkpoint，但没有设置重启策略，那么会使用**固定延迟重启策略**
+ 如果用户没有配置 checkpoint，那么默认不会重启。

可以通过两种方式进行重启策略配置

+ `flink-conf.yaml`文件中的`restart-strategy`
+ 在程序中使用代码指定

在实际生产环境中由于每个任务的负载和资源消耗不一样，推荐在代码中指定每个任务的重试机制和重启策略。

### 无重启策略
在这种情况下，如果作业发生错误，任务直接退出。

方式一：`flink-conf.yaml`中配置：

```yaml
restart-strategy:none
```

方式二：程序中使用代码指定：

```java
env.setRestartStrategy(restartStrategies.noRestart())
```

### 固定延迟重启策略
固定延迟重启策略，需要指定两个参数

+ Flink 会根据用户配置的重试次数进行重试
+ 每次重试之间根据配置的时间间隔进行重试

参数如下表

| 配置 | 说明 |
| --- | --- |
| `restart-strategy.fixed-delay.attempts` | 需要重试的次数 |
| `restart-strategy.fixed-delay.delay` | 每次重试的时间间隔 |


配置：任务重试 3 次，每次重试间隔 5 秒

方式一：`flink-conf.yaml`中配置：

```yaml
restart-strategy:fixed-delay
restart-strategy.fixed-delay.attempts: 3
restart-strategy.fixed-delay.delay: 5 s
```

方式二：程序中使用代码指定：

```java
env.setRestartStrategy(restartStrategies.fixedDelayRestart(
        3, // 重启次数
        Time.of(5, TimeUnit.SECONDS)  // 时间间隔
))
```

### 失败率重启策略模式
失败率重启策略在 Job 失败后会重启，但是超过失败率后，Job 会最终被认定失败。在两个连续的重启尝试之间，重启策略会等待一个固定的时间。

这种重启模式需要指定三个参数，如下表所示。

| 配置 | 说明 |
| --- | --- |
| `restart-strategy.failure-rate.delay` | 每次重试的时间间隔 |
| `restart-strategy.failure-rate.failure-rate-interval` | 计算失败率的时间间隔 |
| `restart-strategy.failure-rate.max-failures-per-interval` | 在指定的时间间隔内最大的失败次数 |


假如 5 分钟内若失败了 3 次，则认为该任务失败，每次失败的重试间隔为 5 秒。

方式一：`flink-conf.yaml`中配置：

```yaml
restart-strategy:failure-rate
restart-strategy.failure-rate.max-failures-per-interval: 3
restart-strategy.failure-rate.failure-rate-interval: 5 min
restart-strategy.failure-rate.delay: 5 s
```

方式二：程序中使用代码指定：

```java
env.setRestartStrategy(restartStrategies.failuerRateRestart(
        3, // 每个时间间隔的最大故障次数
        Time.of(5, TimeUnit.MINUTES),  // 测量故障率的时间间隔
        Time.of(3, TimeUnit.SECONDS)  // 每次任务失败时间间隔
))
```

# 并行度
并行度的定义：在分布式运行环境中，一个算子任务被切分成了多少个子任务并行执行。

提高任务的并行度（Parallelism）在很大程度上可以大大提高任务运行速度。

一般情况下，可以通过四种级别来设置任务的并行度。

## 算子级别
在代码中可以调用`setParallelism`方法来设置每一个算子的并行度

Flink 的每个算子都可以单独设置并行度。这也是最推荐的一种方式，可以针对每个算子进行任务的调优。

```java
.sum(1).setParallelism(1)
```

## 执行环境级别
在创建 Flink 的上下文时可以显示的调用`env.setParallelism()`方法，来设置当前执行环境的并行度。  
这个配置会对当前任务的所有算子、Source、Sink 生效。  
当然还可以在算子级别设置并行度来覆盖这个设置。

```java
env.setParallelism(5)
```

## 提交任务级别
用户在提交任务时，可以显示的指定 `-p`参数来设置任务的并行度

```shell
./bin/flink run -p 10 WordCount.jar
```

## 系统配置级别
 `flink-conf.yaml` 中的`parallelism.default`：该配置即是在系统层面设置所有执行环境的并行度配置。

## 注意事项
整体上讲，这四种级别的配置生效优先级如下：  
算子级别 > 执行环境级别 > 提交任务级别 > 系统配置级别。

要注意 Flink 中的 Slot 概念：

+ Flink 中的 TaskManager 是执行任务的节点，在每一个 TaskManager 里，还会有“槽位”，也就是 Slot。
+ Slot 个数代表的是每一个 TaskManager 的并发执行能力。
+ 假如指定`taskmanager.numberOfTaskSlots: 3`，即每个 taskManager 有 3 个 Slot ，那么整个集群就有`3 * taskManager 的个数`多的槽位。这些槽位就是我们整个集群所拥有的所有执行任务的资源。
