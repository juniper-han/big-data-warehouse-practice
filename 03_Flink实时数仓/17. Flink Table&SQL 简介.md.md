# Flink Table&SQL API 简介
## 简介
Apache Flink 提供了两种**关系型** API 用于统一流和批处理，Table 和 SQL API

+ Table API：用 Java、Scala、Python 按照 SQL 的查询接口封装了一层 lambda 表达式的查询 API
+ SQL API：基于 SQL 标准的 Apache Calcite 框架实现



FlinkSQL 和 Table API 可以做到在用户接口层面的**流批统一**

+ 无论输入是连续（流处理）还是有界（批处理）
+ 在 Table 和 SQL 任一 API 中同一条查询语句
+ 是具有相同的语义，并且会产出相同的结果

## 运行环境依赖
后续均基于 Flink1.17.2

### 依赖模块
1. `**flink-table-api-java-bridge**`** 或**`**flink-table-api-scala-bridge**`

功能：

    - API 入口 + 基础运行时
    - 包含 SQL 解析和基础执行逻辑（但不含优化器）。
    - 支持与 DataStream API 互操作（如 `tableEnv.toDataStream()`）。

注意事项：

    - 必须与 Planner 配合使用
        * 本地用 `flink-table-planner-blink`
        * 集群用内置 Planner
    - 生产环境需**随作业 Jar 包提交**。
    - 即使纯 SQL 作业，底层仍依赖该模块提供运行时。
2. `**flink-table-planner-blink**`

功能：SQL 优化器 + 完整执行能力

    - 提供 SQL 优化器，基于 Apache Calcite
    - 生成查询计划，算子转换

注意：

    - **本地开发必须引入，但生产环境集群已内置 Planner**，需设 `<scope>provided</scope>`。
    - `flink-table-planner-blink`相当于后端
    - `flink-table-api-java-bridge`相当于前端
3. `**flink-table-common**`**（非必需）**

功能：

    - 定义 Table 模块的抽象接口（如 `Table`、`Catalog`）。
    - 提供扩展点 SPI，用于开发自定义 Connector、Format、UDF

注意事项：

    - 普通用户无需直接依赖，开发自定义扩展时才需要。
    - **生产环境随作业 Jar 包提交**。
    - 版本必须与其他 Flink 模块严格一致。

## SQL/Table 的代码结构和 API
代码结构：

1. 使用 `EnvironmentSettings` + `TableEnvironment.create()`创建执行环境
2. 创建输入表
3. 计算逻辑
4. 创建输出表
5. 计算结果写入输出表

```java
import org.apache.flink.table.api.EnvironmentSettings;
import org.apache.flink.table.api.TableEnvironment;

// 1. 创建一个 TableEnvironment，为后续使用 SQL 或者 Table API 提供运行环境
EnvironmentSettings settings = EnvironmentSettings.newInstance()
    .inStreamingMode() // 声明为流任务 //.inBatchMode() // 声明为批任务
    .build();

TableEnvironment tEnv = TableEnvironment.create(settings);

// 2. 创建一个输入表
tableEnv.executeSql("CREATE TEMPORARY TABLE table1 ... WITH ( 'connector' = ... )");
// 3. 创建一个输出表
tableEnv.executeSql("CREATE TEMPORARY TABLE outputTable ... WITH ( 'connector' = ... )");

// 4. 使用 Table API 做一个查询并返回 Table
Table table2 = tableEnv.from("table1").select(...);
// 4. 使用 SQl API 做一个查询并返回 Table
Table table3 = tableEnv.sqlQuery("SELECT ... FROM table1 ... ");

// 5. 将 table2 的结果使用 Table API 写入 outputTable 中，并返回结果
TableResult tableResult = table2.executeInsert("outputTable");
tableResult...
```

## 常用 API
### `EnvironmentSettings` 类
`EnvironmentSettings`：用于配置 Flink Table/SQL 执行环境的**设置类**，用于配置：

+ 使用哪种 执行模式（Batch / Streaming）
+ ~~使用哪个 Planner（查询优化器）（在 Flink 1.13+ 后统一为 Blink Planner）~~
    - ⚠️ 从 Flink 1.13 开始，旧的 flink-planner 已被废弃，Flink 默认且唯一支持的是 Blink Planner

### SQL/Table 上下文：`TableEnvironment`类
`TableEnvironment`：是 SQL/Table API 使用的入口（上下文），包含了一个 Flink 任务运行时的环境信息，包括：

+ Catalog 管理
+ SQL 查询
+ UDF 管理：注册用户定义函数
+ UDF 扩展：加载可插拔 Module（Module 可以理解为 Flink 管理 UDF 的模块，是可插拔的）



注意：DataStream 和 Table（Table\SQL API 的查询结果）之间可以进行转换，但实际生产中使用较少



方法：

+ 表管理
    - `void createTemporaryView(String path, Table table)`：注册表
    - `Table from(String path)`：获取已注册的表
    - `void dropTemporaryTable(String path)`：删除临时表
+ SQL 执行
    - `TableResult executeSql(String sql)`：执行DDL/DML
    - `Table sqlQuery(String query)`：执行查询并返回 `Table`
+ UDF 管理
    - `void createTemporaryFunction)`：注册UDF
    - `void createTemporarySystemFunction()`：注册系统级UDF（跨Catalog可见）|

### `<font style="color:rgb(6, 10, 38);">Configuration</font>`<font style="color:rgb(6, 10, 38);"> 类</font>
`Configuration` 类是 Flink 的核心配置管理工具，用于以键值对（key-value） 的形式统一管理运行时的各种参数，贯穿于 Flink 的整个执行栈

+ 类名：`org.apache.flink.configuration.Configuration`
+ 作用：Flink 的配置中心，所有参数都通过它传递、解析
+ 线程安全

### 生产环境综合使用
Streaming 模式：

+ 适用于 Kafka、Pulsar 等实时数据源，需开启 Checkpoint、设置状态 TTL、Watermark 等。

```java
import org.apache.flink.configuration.Configuration;
import org.apache.flink.configuration.CoreOptions;
import org.apache.flink.configuration.RestOptions;
import org.apache.flink.streaming.api.environment.StreamExecutionEnvironment;
import org.apache.flink.table.api.EnvironmentSettings;
import org.apache.flink.table.api.TableEnvironment;
import org.apache.flink.table.api.bridge.java.StreamTableEnvironment;
import org.apache.flink.table.api.config.ExecutionConfigOptions;
import org.apache.flink.table.api.config.TableConfigOptions;

import java.time.Duration;

public class StreamingTableEnvTemplate {

    public static void main(String[] args) {
        // 1. 创建 Configuration，集中管理所有参数
        Configuration config = new Configuration();

        // --- 通用配置 ---
        config.set(CoreOptions.DEFAULT_PARALLELISM, 4);               // 默认并行度
        config.set(RestOptions.PORT, 8081);                          // Web UI 端口（本地测试用）
        config.set(CoreOptions.CHECK_LEAKED_CLASSLOADER, false);     // 生产可关闭类加载器检查

        // --- Checkpoint 配置（关键！）---
        config.set(ExecutionConfigOptions.TABLE_EXEC_CHECKPOINTING_INTERVAL, Duration.ofSeconds(30));
        config.set(ExecutionConfigOptions.TABLE_EXEC_CHECKPOINTING_TIMEOUT, Duration.ofMinutes(10));
        config.set(ExecutionConfigOptions.TABLE_EXEC_CHECKPOINTING_MIN_PAUSE, Duration.ofSeconds(5));
        config.set(ExecutionConfigOptions.TABLE_EXEC_CHECKPOINTING_MAX_CONCURRENT, 1);
        config.set(ExecutionConfigOptions.TABLE_EXEC_CHECKPOINTING_EXTERNALIZED_CHECKPOINT, 
                   ExecutionConfigOptions.ExternalizedCheckpointCleanup.RETAIN_ON_CANCELLATION);

        // --- State & TTL ---
        // 注意：状态后端通常在 flink-conf.yaml 或启动脚本中配置，但也可在此设置路径
        // config.set(StateBackendOptions.STATE_BACKEND, "rocksdb");
        // config.set(CheckpointingOptions.CHECKPOINTS_DIRECTORY, "hdfs:///flink/checkpoints");

        // --- Table 执行优化 ---
        config.set(TableConfigOptions.TABLE_EXEC_RESOURCE_DEFAULT_PARALLELISM, 4);
        config.set(TableConfigOptions.TABLE_EXEC_STATE_TTL, Duration.ofHours(24)); // 状态 TTL
        config.set(TableConfigOptions.TABLE_EXEC_SOURCE_IDLE_TIMEOUT, Duration.ofMinutes(5)); // 空闲源超时

        // 2. 创建 StreamExecutionEnvironment（用于精细控制流行为）
        StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment(config);

        // ⚠️ 注意：上面已通过 Configuration 传入 Checkpoint 设置，
        // 也可以显式调用（效果等价）：
        // env.enableCheckpointing(30_000);
        // env.getCheckpointConfig().setMinPauseBetweenCheckpoints(5_000);
        // env.getCheckpointConfig().setCheckpointTimeout(600_000);

        // 3. 创建 EnvironmentSettings（流模式）
        EnvironmentSettings settings = EnvironmentSettings.newInstance()
            .inStreamingMode()
            .withConfiguration(config)
            .build();

        // 4. 创建 TableEnvironment（推荐使用 bridge 方式，与 DataStream 环境绑定）
        StreamTableEnvironment tableEnv = StreamTableEnvironment.create(env, settings);

        // ✅ 此时 tableEnv 已具备：
        // - 并行度 4
        // - 30s Checkpoint
        // - 状态 TTL 24 小时
        // - 与底层 DataStream 环境一致的容错机制
    }
}
```

使用 `StreamTableEnvironment.create(env, settings)`创建 Table 环境：确保 Table 层与 DataStream 层共享同一个执行环境（包括 Checkpoint、Watermark、时间语义等）



Batch 模式

+ 适用于 Hive、HDFS、JDBC 等离线数据源，自动启用批处理优化（如 shuffle-free join、broadcast）。

```java
import org.apache.flink.configuration.Configuration;
import org.apache.flink.configuration.CoreOptions;
import org.apache.flink.table.api.EnvironmentSettings;
import org.apache.flink.table.api.TableEnvironment;
import org.apache.flink.table.api.config.TableConfigOptions;

public class BatchTableEnvTemplate {

    public static void main(String[] args) {
        // 1. 创建 Configuration
        Configuration config = new Configuration();

        // --- 批处理核心配置 ---
        config.set(CoreOptions.DEFAULT_PARALLELISM, 8);  // 批作业通常更高并行度
        config.set(TableConfigOptions.TABLE_EXEC_RESOURCE_DEFAULT_PARALLELISM, 8);

        // --- 批处理优化（Flink 自动启用，但可显式确认）---
        // Flink 1.17 批模式默认开启以下优化，无需额外设置：
        // - Operator chaining
        // - Broadcast hash join
        // - Sort-based blocking shuffle

        // 可选：设置结果输出模式（如只输出最终结果）
        config.set(TableConfigOptions.TABLE_EXEC_EMIT_EARLY_FIRE_ENABLED, false);

        // 2. 创建 EnvironmentSettings（批模式）
        EnvironmentSettings settings = EnvironmentSettings.newInstance()
            .inBatchMode()
            .withConfiguration(config)
            .build();

        // 3. 创建 TableEnvironment
        TableEnvironment tableEnv = TableEnvironment.create(settings);

        // ✅ 此时 tableEnv 已运行在纯批处理模式，享受所有批优化

        // 示例：读取 Hive 表进行 ETL
        // tableEnv.useCatalog("myhive");
        // tableEnv.executeSql("INSERT INTO result_table SELECT ... FROM hive_table");
    }
}
```

Batch 模式无需使用`StreamExecutionEnvironment`，无 Checkpoint、无状态 TTL

# 动态表
FlinkSQL 和 Table API 中的表，是动态表。

+ 是 Flink 对**无限流数据的表抽象**，表示一个不断变化的表，在任意时间点，都可以像静态表一样可查询
+ **动态表是逻辑上的概念**

## 表的命名规则
表名遵循**三层命名空间**，由三个部分组成：`Catalog 名称.数据库名称.表名称`。

```sql
[catalog_name.][database_name.]table_name
```

**默认值**：

+ `catalog_name`：`default_catalog`
+ `database_name`：`default_database`



SQL API 创建表

```java
tableEnv.executeSql("CREATE TEMPORARY TABLE table1 ... WITH ( 'connector' = ... )");
```

## 动态表的类型和特点
动态表基于不同的**数据来源**，可分成两类：

1. 物理表（Physical Table）：与**外部系统**绑定，是数据真正的来源或目的地
    - **创建方式**： `tEnv.executeSql(CREATE TABLE ... WITH (...))`
    - 数据存储在外部系统（如 Kafka、MySQL、文件系统）
    - 拥有**固定的 Schema**（列名、类型）
    - 通过 Connector 的格式定义（如 JSON、Avro）进行序列化/反序列化
2. 视图（View/Logical Table 逻辑表）：从已经存在的表中创建，一般是一个 SQL 逻辑的查询结果
    - 创建方式：使用 `CREATE VIEW ... AS SELECT ...`
    - 无论是否 TEMPORARY，都**不存储数据**，只是一个查询定义的别名，每次被引用时都会重新执行底层查询逻辑。
    - 用于简化复杂查询、权限控制或逻辑分层
    - 推荐使用`CREATE TEMPORARY VIEW`，语义清晰



动态表基于不同的**生命周期**，可划分为

1. 临时表/视图（Temporary Table/View）：当前 Session 有效，通常保存于内存中
    - 名称允许与持久化表重复，临时表优先级更高
    - 分为：临时外部表、临时视图
    - 创建方式：`CREATE TEMPORARY TABLE/VIEW ...`
2. 持久化表（Persistent Table，永久表）：Catalog 管理，跨会话，多个 Session 有效
    - 生命周期：显式创建（`CREATE TABLE`，没有 `PERSIST` 关键字），显式删除（`DROP TABLE`）
    - 元数据被持久化，表结构在作业重启后依然存在，通常与物理表关联，指向外部数据源
    - 注意：一般不使用持久化视图，因为持久化视图也不会缓存中间结果

## 易混淆点：Flink 中的“物化视图”
Flink 中没有物化视图（Materialized View）的概念（如传统数据库中的`CREATE MATERIALIZED VIEW`）

+ Flink 中的视图，无论是否持久化，都不会缓存中间结果，而是每次使用时都重新计算

但是：Flink 通过持续查询和状态管理机制，在流处理场景下天然实现了“动态物化视图”的语义。



为实现“物化视图”的功能，需要满足

1. 持续计算：基于流式数据源，实时更新计算结果
2. 状态管理：Flink 自动维护聚合、连接等操作的状态
3. 持久化（Sink to External Storage）：将计算结果写入外部存储，以便后续直接读取，无需都重新计算。
    - Sink 是必须的：只有通过 Sink 将计算结果持久化到外部存储，才能实现真正意义上的“物化视图”。

## SQL 与 DataStream API 转换
注意：生产环境很少使用

场景：发放补贴券，在发放的补贴券总金额超过 1w 元时，及时报警，控制预算

+ 使用 SQL 计算补贴券的发放结果，但 SQL 无法报警
+ 将 SQL 的查询的结果（即 Table 对象）转为 DataStream，在 DataStream 后自定义报警逻辑的算子

```java
public static void main(String[] args) throws Exception {

    FlinkEnv flinkEnv = FlinkEnvUtils.getStreamTableEnv(args);

    // 1. pdd 发补贴券流水数据
    String createTableSql = "CREATE TABLE source_table (\n"
            + "    id BIGINT,\n" -- 补贴券的流水 id
            + "    money BIGINT,\n" -- 补贴券的金额
            + "    row_time AS cast(CURRENT_TIMESTAMP as timestamp_LTZ(3)),\n"
            + "    WATERMARK FOR row_time AS row_time - INTERVAL '5' SECOND\n"
            + ") WITH (\n"
            + "  'connector' = 'datagen',\n"
            + "  'rows-per-second' = '1',\n"
            + "  'fields.id.min' = '1',\n"
            + "  'fields.id.max' = '100000',\n"
            + "  'fields.money.min' = '1',\n"
            + "  'fields.money.max' = '100000'\n"
            + ")\n";

    // 2. 计算总计发放补贴券的金额
    String querySql = "SELECT UNIX_TIMESTAMP(CAST(window_end AS STRING)) * 1000 as window_end, \n"
            + "      window_start, \n"
            + "      sum(money) as sum_money,\n" -- 补贴券的发放总金额
            + "      count(distinct id) as count_distinct_id\n"
            + "FROM TABLE(CUMULATE(\n"
            + "         TABLE source_table\n"
            + "         , DESCRIPTOR(row_time)\n"
            + "         , INTERVAL '5' SECOND\n"
            + "         , INTERVAL '1' DAY))\n"
            + "GROUP BY window_start, \n"
            + "        window_end";

    flinkEnv.streamTEnv().executeSql(createTableSql);

    Table resultTable = flinkEnv.streamTEnv().sqlQuery(querySql);

    // 3. 将金额结果转为 DataStream，然后自定义超过 1w 的报警逻辑
    flinkEnv.streamTEnv()
            .toDataStream(resultTable, Row.class)
            .flatMap(new FlatMapFunction<Row, Object>() {
                @Override
                public void flatMap(Row value, Collector<Object> out) throws Exception {
                    long l = Long.parseLong(String.valueOf(value.getField("sum_money")));

                    if (l > 10000L) {
                        log.info("报警，超过 1w");
                    }
                }
            });

    flinkEnv.env().execute();
}
```

# Catalog
## Catalog 简介
Catalog：FlinkSQL 的**元数据管理中心**，负责管理所有可用组件的元数据（包括数据库、表、视图、函数、分区等）



Catalog 的作用

+ 元数据统一管理
    - 解耦元数据与计算逻辑：将表名、列信息、连接器属性等元数据与作业代码分离，使 SQL 查询（如 `SELECT * FROM my_kafka_table`）保持简洁。
    - 唯一真相源：为 Flink 作业提供统一的元数据视图，确保所有 TaskManager 对表的定义是一致的。
+ 跨会话持久化
    - 临时 Catalog：基于内存的 `GenericInMemoryCatalog`，作业停止后元数据丢失。
    - 持久化 Catalog：如 `HiveCatalog`，将元数据存储在外部系统（如 Hive Metastore），作业重启、甚至新建作业，依然可以识别已创建的元数据。这是生产环境的必备特性。
+ 多租户与资源隔离
    - 通过创建多个 Catalog 和 Database，可以实现逻辑上的资源隔离。例如：
        * `finance_catalog`下的表供财务团队使用。
        * `logistics_catalog`下的表供物流团队使用。
+ Catalog 允许用户引用其数据存储系统中现有的元数据，并自动将其映射到 Flink 的相应元数据。
    - 例如，Flink 可以直接使用 Hive MetaStore 中的表的元数据，也可以将 Flink SQL 中的元数据存储到 Hive MetaStore 中

## Flink 内置 Catalog
目前 Flink 包含以下四种 Catalog：

1. `GenericInMemoryCatalog`：默认 Catalog，基于内存实现，元数据存储在内存中
    - 所有元数据只在 Session 的生命周期内可用
    - 针对元数据对象名称，`GenericInMemoryCatalog` 区分大小写。
    - 适用场景：开发测试、临时查询
2. `JdbcCatalog`
    - Flink 通过 JDBC 协议连接到关系数据库，将关系型数据库的表映射为 Flink 表
    - PostgresCatalog 是当前实现的唯一一种 JDBC Catalog，即可以将 Flink SQL 的元数据存储在 Postgres 中
    - 适用场景：查询已有关系型数据库中的表
3. `**HiveCatalog**`**：生产环境标配，元数据持久化到 Hive Metastore**
    - HiveCatalog 还可作为读写现有 Hive 元数据的接口
    - 注意：Hive MetaStore 以小写形式存储所有元数据对象名称

```java
TableEnvironment tableEnv = TableEnvironment.create(settings);

String name            = "myhive";
String defaultDatabase = "mydatabase";
String hiveConfDir     = "/opt/hive-conf";

HiveCatalog hive = new HiveCatalog(name, defaultDatabase, hiveConfDir);
tableEnv.registerCatalog("myhive", hive);

// set the HiveCatalog as the current catalog of the session
tableEnv.useCatalog("myhive");
```

4. 用户自定义 Catalog：用户可以实现 Catalog 接口实现自定义 Catalog

## FlinkSQL：Catalog API
### 创建 Catalog
```sql
-- 创建 Hive Catalog
CREATE CATALOG hive_catalog WITH (
    'type' = 'hive',
    'hive-conf-dir' = '/path/to/hive/conf',
    'hadoop-conf-dir' = '/path/to/hadoop/conf',
    'default-database' = 'default'
);
-- 创建 JDBC Catalog（MySQL/PostgreSQL）

    'password' = 'flink_password',
    'base-url' = 'jdbc:mysql://localhost:3306'
);

-- 创建 Generic InMemory Catalog
CREATE CATALOG memory_catalog WITH (
    'type' = 'generic_in_memory'
);
```

# Flink SQL 中的数据类型
Flink SQL 数据类型系统

1. 原子数据类型：数值、字符、时间、布尔、二进制
2. 复合数据类型：数组、映射、行类型
3. 用户自定义类型

## 原子数据类型
### 数值类型（Numeric Types）
| 类型 | 说明 | 示例 | 对应Java类型 |
| --- | --- | --- | --- |
| `TINYINT` | 1字节整数 | `age TINYINT` | `Byte` |
| `SMALLINT` | 2字节整数 | `id SMALLINT` | `Short` |
| `INT` | 4字节整数（最常用） | `count INT` | `Integer` |
| `BIGINT` | 8字节整数 | `userId BIGINT` | `Long` |
| `FLOAT` | 4字节单精度浮点，   有损精度 | `temperature FLOAT` | `Float` |
| `DOUBLE` | 8字节双精度浮点，   有损精度 | `price DOUBLE` | `Double` |
| `DECIMAL(p, s)`   `NUMERIC(p, s)` | 高精度小数，   <font style="color:rgb(25, 27, 31);">固定长度和精度的数值类型</font> | `amount DECIMAL(10, 2)` | `BigDecimal` |




`DECIMAL(p, s)`或`NUMERIC(p, s)`：精确数值类型

+ `p`：数值位数（长度），取值范围`[1, 38]`，默认 10
+ `s`：小数点后的位数（精度），取值范围`[0, p]`，默认 0
+ `DECIMAL`、`DECIMAL(p)`、`DECIMAL(p, s)`、`DEC`、`DEC(p)`、`DEC(p, s)`、`NUMERIC`、`NUMERIC(p)`、`NUMERIC(p, s)`

### 字符类型（Character Types）
| 类型 | 说明 | 示例 | 对应Java类型 |
| --- | --- | --- | --- |
| `VARCHAR(n)`/`STRING` | 可变长度字符串 | `name STRING` | `String` |
| `CHAR(n)` | 定长字符串（用空格填充） | `code CHAR(5)` | `String` |


两种类型的`n`

+ 取值范围均为：1 - 2147483647
+ 如不指定，则默认取 1

`STRING`等同于`VARCHAR(2147483647)`

### 时间类型（Temporal Types）
| 类型 | 说明 | 对应Java类型 |
| --- | --- | --- |
| `DATE` | 日期（年月日），不带时区 | `java.sql.Date` |
| `TIME(p)` | 时间（时分秒毫秒），不带时区 | `java.sql.Time` |
| `TIMESTAMP(p)` | **时间戳**（日期+时间），不带时区 | `java.sql.Timestamp` |
| `<font style="color:rgb(25, 27, 31);">TIMESTAMP(p) WITH TIME ZONE</font>` | 时间戳，带时区 |  |
| `TIMESTAMP_LTZ(p)` | 带本地时区的时间戳，本地时区由<font style="color:rgb(25, 27, 31);"> </font>`<font style="color:rgb(25, 27, 31);background-color:rgb(248, 248, 250);">table.local-time-zone</font>`<font style="color:rgb(25, 27, 31);"> 参数设置</font> | `java.time.Instant` |
| `INTERVAL` | 时间间隔，用于给`TIMESTAMP`和`TIMESTAMP_LTZ`添加偏移量 |  |


+ **时间精度 **`(p)`：指定秒的小数部分精度（0到9），默认 6
+ `TIMESTAMP(3)`表示毫秒精度（3位小数），对应的就是 Java 和 Scala 中的`java.sql.Timestamp`
+ `INTERVAL`类型使用方式较多，` INTERVAL YEAR TO MONTH`、`INTERVAL DAY TO SECOND`
    - `INTERVAL '10' YEAR/MONTH/DAY/HOUR/MINUTE/SECOND`
    - `INTERVAL '10-03' YEAR(3) TO MONTH`
    - `INTERVAL '10 03' DAY(3) TO HOUR`
    - `INTERVAL '10 03:12' DAY(3) TO MINUTE`
    - `INTERVAL '10 00:00:00.004' DAY TO SECOND(3)`

### 其他基本类型
| 类型 | 说明 | 示例 |
| --- | --- | --- |
| `BOOLEAN` | 布尔值 | `isValid BOOLEAN` |
| `BYTES`/`VARBINARY(n)` | 可变长二进制字符串 | `image BYTES` |
| `BINARY(n)` | 定长二进制字符串 |  |
| `NULL`类型 |  | `NULL` |
| `RAW`类型 | <font style="color:rgb(25, 27, 31);">只会在数据发生网络传输时进行序列化，反序列化操作，   </font><font style="color:rgb(25, 27, 31);">可以保留其原始数据</font> | `<font style="color:rgb(25, 27, 31);">RAW('class', 'snapshot')</font>`<font style="color:rgb(25, 27, 31);">：   </font>`<font style="color:rgb(25, 27, 31);">class</font>`<font style="color:rgb(25, 27, 31);">：Java 类型   </font>`<font style="color:rgb(25, 27, 31);">snapshot</font>`<font style="color:rgb(25, 27, 31);">：序列化器</font> |


注意：两种二进制字符串类型中的`n`

+ 取值范围均为：1 - 2147483647

`BYTES`等同于`VARBINARY(2147483647)`

## 复合数据类型（Composite/Collection Types）
**注意事项**：

+ `ARRAY`、`MAP`和 `ROW`的序列化/反序列化开销较大，在性能关键路径上谨慎使用。

### 数组类型（ARRAY）
存储同类型元素的集合：数组最大长度为 2,147,483,647

+ 对应 Java：`E[]`或`List<E>`
+ 对应 Scala：`Array[E]`或`Seq[E]`

```sql
-- 定义和使用数组
ARRAY<INT>         -- 整数数组
ARRAY<STRING>      -- 字符串数组
ARRAY<TIMESTAMP(3)> -- 时间戳数组

-- 示例：查询数组
SELECT tags[1] FROM events; -- 访问数组元素（索引从1开始）
SELECT CARDINALITY(tags) FROM events; -- 获取数组长度
```

### 映射类型（MAP）
存储键值对集合，键类型相同，值类型相同，键和值可以是不同类型

+ 对应 Java：`Map<K, V>`
+ 对应 Scala：`Map[K, V]`

```sql
-- 定义和使用Map
MAP<STRING, INT>    -- {key: value} 映射

-- 示例：查询Map
SELECT config['timeout'] FROM settings; -- 通过键访问值
SELECT MAP_KEYS(config) FROM settings;  -- 获取所有键
```

### 集合类型（MULTISET）
集合类型：`MULTISET` 可存储重复数据

+ 对应 Java：`List`

### 对象类型（ROW）
类似于一个微型的表结构，可以包含多个命名字段，每个字段有自己的类型。

+ 对用 Java 和 Scala：`Row`

```sql
-- 定义和使用ROW
ROW<name STRING, age INT, score DOUBLE>

-- 示例：访问字段
SELECT address.city FROM users; -- 访问嵌套字段
```

## 自定义数据类型
在 Flink SQL 中自定义数据类型通常涉及两种场景：

+ 自定义结构化类型：如POJO（Plain Ordinary Java Object，简单 Java 对象）
+ 扩展原生类型系统（如UDT）

目前数据类型不支持使用 `CREATE TABLE` 的 DDL 进行定义，只支持作为函数的输入输出参数。

### 自定义结构化类型（POJO类）
第一步：定义POJO类

```java
public class User {

    // 1. 基础类型，Flink 可以通过反射类型信息自动把数据类型获取到
    // 关于 SQL 类型和 Java 类型之间的映射见：https://nightlies.apache.org/flink/flink-docs-release-1.13/docs/dev/table/types/#data-type-extraction
    public int age;
    public String name;

    // 2. 复杂类型，用户可以通过 @DataTypeHint("DECIMAL(10, 2)") 注解标注此字段的数据类型
    public @DataTypeHint("DECIMAL(10, 2)") BigDecimal totalBalance;
}
```

第二步：在 UDF 中使用此数据类型

```java
public class UserScalarFunction extends ScalarFunction {

    // 1. 自定义数据类型作为输出参数
    public User eval(long i) {
        if (i > 0 && i <= 5) {
            User u = new User();
            u.age = (int) i;
            u.name = "name1";
            u.totalBalance = new BigDecimal(1.1d);
            return u;
        } else {
            User u = new User();
            u.age = (int) i;
            u.name = "name2";
            u.totalBalance = new BigDecimal(2.2d);
            return u;
        }
    }
    
    // 2. 自定义数据类型作为输入参数
    public String eval(User i) {
        if (i.age > 0 && i.age <= 5) {
            User u = new User();
            u.age = 1;
            u.name = "name1";
            u.totalBalance = new BigDecimal(1.1d);
            return u.name;
        } else {
            User u = new User();
            u.age = 2;
            u.name = "name2";
            u.totalBalance = new BigDecimal(2.2d);
            return u.name;
        }
    }
}
```

第三步：<font style="color:rgb(25, 27, 31);">在 Flink SQL 中使用</font>

```sql
-- 1. 创建 UDF
CREATE FUNCTION user_scalar_func AS 'flink.examples.sql._12_data_type._02_user_defined.UserScalarFunction';

-- 2. 创建数据源表
CREATE TABLE source_table (
    user_id BIGINT NOT NULL COMMENT '用户 id'
) WITH (
  'connector' = 'datagen',
  'rows-per-second' = '1',
  'fields.user_id.min' = '1',
  'fields.user_id.max' = '10'
);

-- 3. 创建数据汇表
CREATE TABLE sink_table (
    result_row_1 ROW<age INT, name STRING, totalBalance DECIMAL(10, 2)>,
    result_row_2 STRING
) WITH (
  'connector' = 'print'
);

-- 4. SQL 查询语句
INSERT INTO sink_table
select
    -- 4.a. 用户自定义类型作为输出
    user_scalar_func(user_id) as result_row_1,
    -- 4.b. 用户自定义类型作为输出及输入
    user_scalar_func(user_scalar_func(user_id)) as result_row_2
from source_table;

-- 5. 查询结果
+I[+I[9, name2, 2.20], name2]
+I[+I[1, name1, 1.10], name1]
+I[+I[5, name1, 1.10], name1]
```

### 自定义UDT（User-Defined Type）
当需要完全自定义一个原生类型系统不支持的逻辑类型时（如地理坐标、IP 地址等），可以通过实现`UserDefinedType`扩展类型系统。



实现抽象类`UserDefinedType`

```java
import org.apache.flink.table.types.logical.LogicalType;
import org.apache.flink.table.types.logical.UserDefinedType;

public class PointType extends UserDefinedType {
    public PointType() {
        super(true, LogicalTypeRoot.RAW); // 是否可为null, 基础逻辑类型
    }

    @Override
    public TypeInformation<?> getTypeInformation() {
        return Types.POJO(Point.class); // 关联到自定义类
    }

    @Override
    public String asSerializableString() {
        return "POINT"; // 在SQL中显示的类型名称
    }
}
```

自定义类型需确保序列化/反序列化的正确性  
复杂类型的序列化可能成为瓶颈，需测试优化



定义数据类

```java
public class Point {
    public double x;
    public double y;
    
    public Point() {}
    public Point(double x, double y) {
        this.x = x;
        this.y = y;
    }
}
```



注册自定义类型

```java
// 注册到TableEnvironment
tableEnv.createTemporaryFunction("POINT", new PointType());

// 或在Catalog中全局注册
tableEnv.getCatalog("my_catalog")
    .ifPresent(catalog -> catalog.registerType("POINT", new PointType()));
```



在 SQL 中使用

```sql
CREATE TABLE locations (
    id BIGINT,
    position POINT  -- 使用自定义类型
);

-- 插入数据（需通过UDF转换）
INSERT INTO locations VALUES (1, CAST('(1.2,3.4)' AS POINT));
```
