# DDL：创建/修改/删除语句
`CREATE`语句用于向当前或指定的 Catalog 中注册库、表、视图或函数

包括

+ `CREATE TABLE`
+ `CREATE DATABASE`
+ `CREATE VIEW`
+ `CREATE FUNCTION`

## 创建表：`CREATE TABLE`
### 建表语法
建表语句：用于连接外部系统并定义表结构

如果同名表已经在 Catalog 中存在，则无法注册。

```sql
CREATE TABLE [IF NOT EXISTS] [catalog_name.][db_name.]table_name
  (
    { <physical_column_definition> | -- 物理列
      <metadata_column_definition> | -- 元数据列
      <computed_column_definition>   -- 计算列
    }[ , ...n]
    [ <watermark_definition> ]       -- Watermark
    [ <table_constraint> ][ , ...n]  -- 约束
  )
  [COMMENT table_comment]
  [PARTITIONED BY (partition_column_name1, partition_column_name2, ...)]  -- 分区
  WITH (key1=val1, key2=val2, ...)  -- 连接外部系统参数
  [ LIKE source_table [( <like_options> )] ]
```

说明：

1. `IF NOT EXISTS`：可选，表不存在时才创建
2. 三种列的定义

```sql
<physical_column_definition>:   -- 物理列
  column_name column_type [ <column_constraint> ] [COMMENT column_comment]

<computed_column_definition>:   -- 计算列
  column_name AS computed_column_expression [COMMENT column_comment]

<metadata_column_definition>:  -- 元数据列
  column_name column_type METADATA [ FROM metadata_key ] [ VIRTUAL ]
```

3. 约束：主要是主键

```sql
<column_constraint>:
  [CONSTRAINT constraint_name] PRIMARY KEY NOT ENFORCED

<table_constraint>:
  [CONSTRAINT constraint_name] PRIMARY KEY (column_name, ...) NOT ENFORCED
```

4. Watermark 定义

```sql
<watermark_definition>:
  WATERMARK FOR rowtime_column_name AS watermark_strategy_expression
```

5. `LIKE`：基于现有表定义创建新表

```sql
<like_options>:
{
   { INCLUDING | EXCLUDING } { ALL | CONSTRAINTS | PARTITIONS }
 | { INCLUDING | EXCLUDING | OVERWRITING } { GENERATED | OPTIONS | WATERMARKS } 
}[, ...]
```

### 表中的列
表中的列包含以下类型

1. 常规列（也称物理列或基本列）
    - 定义了物理介质中存储的数据中字段的名称、类型和顺序
2. 元数据列：允许访问数据源本身具有的一些元数据
    - 元数据列由 `METADATA` 关键字标识
    - 元数据：从连接器元数据中提取信息
3. 计算列：基于已有列，经过自定义的运算生成的新列
    - 虚拟列，本身没有以物理形式存储在数据源中
    - 可以包含其他列、常量或者函数，但是**不能写子查询**
    - 常用于定义时间属性

### 元数据列
元数据列参数说明

+ `metadata_key`：取决于连接器类型
    - 如果自定义的列名和 Connector 的`metadata_key`相同，`FROM metadata_key`可省略
    - 每种 Connector 提供的 `metadata_key`，可参考官网文档 Connectors 部分
+ `VIRTUAL`关键字：表示不写入 Sink
    - `VIRTUAL`元数据列具有以下特征
        * 仅作为逻辑字段存在，不会物化到数据中，不写入 Sink，不参与序列化
        * 仅用于条件过滤、时间提取
    - **非**`VIRTUAL`元数据列具有以下特征
        * 被当作普通列处理，可以写入 Sink，参与序列化
        * 不带`VIRTUAL`的元数据列默认会写入Sink，推荐显式声明`NOT VIRTUAL`以增强可读性
        * 需确认 Sink 连接器支持该元数据字段的写入



注意：

+ 默认情况下，Flink 认为元数据列是可读可写的，但某些外部存储系统的元数据是只能读，不能写入
    - 例：Kafka 的 `Offset` 不能被写入

****

举例：读取 Kafka 数据源的元数据

使用元数据列从 Kafka 数据中读取 Kafka 数据的**偏移量**、**自带时间戳**（数据写入 Kafka 时，Kafka 引擎给这条数据打上的时间戳）

```sql
CREATE TABLE kafka_source (
  user_id BIGINT,
  message STRING,
  offset BIGINT METADATA FROM 'offset' VIRTUAL,  -- Kafka消息偏移量，是只读的，offset为自定义字段名
  ts TIMESTAMP(3) METADATA FROM 'timestamp'     -- Kafka消息时间戳，ts为自定义字段名
) WITH (
  'connector' = 'kafka',
  ...
)

-- 注意
   -- 自定义的列名称和 Connector 中定义 metadata_key 相同，省略FROM xxx 
   -- `timestamp` TIMESTAMP_LTZ(3) METADATA
```

如果 DDL 中的数据类型和 Connector 中 Metadata 字段的数据类型不一致，则运行时自动 Cast 强转，但要求这两种数据类型是可以强转的

### 定义 Watermark
Watermark 需要在 `Create Table` 中定义

如果使用事件时间语义，则必须设置事件时间属性和 WATERMARK 生成策略。



语法

```sql
 WATERMARK FOR rowtime_column_name AS watermark_strategy_expression
```

其中：

+ `rowtime_column_name`：表的事件时间属性字段
    - 类型：`TIMESTAMP(3)`或`TIMESTAMP_LTZ(3)`
    - 该列为计算列。
+ `watermark_strategy_expression`：定义 Watermark 的生成策略
    - 生成策略：一般是由`rowtime_column_name`列减掉一段固定时间间隔（最大延误时间）



Watermark 发出一般是**间隔**一定时间的，Watermark 的发出间隔时间可以由`pipeline.auto-watermark-interval`进行配置



FlinkSQL 提供了 3 种 Watermark 生成策略

1. **有界无序**：一般采用此策略
    - 设置方式：`WATERMARK FOR rowtime_column AS rowtime_column - INTERVAL 'string' timeUnit`
    - 此类策略用于设置最大乱序时间，通常用于有数据乱序的场景
2. 严格升序：一般不用
    - 设置方式：`WATERMARK FOR rowtime_column AS rowtime_column`
    - 适用于：数据源的时间戳是严格升序的，严格升序代表 Flink 任务认为时间戳只会越来越大，也不存在相等的情况，只要相等或者小于之前的，就认为是迟到的数据
3. 递增：一般不用
    - 设置方式：`WATERMARK FOR rowtime_column AS rowtime_column - INTERVAL '0.001' SECOND`
    - 如果设置此类，则允许有相同的时间戳出现

### 主键约束
```sql
PRIMARY KEY (column_name[, ...]) NOT ENFORCED
```

特点：

+ Upsert 连接器（如JDBC、ES）需要定义主键
+ `NOT ENFORCED`表示 Flink 不验证数据唯一性
+ 主键列不能为 `NULL`

### `WITH`子句参数详解
`WITH`子句定义表的【连接器】和【配置属性】，格式为键值对：

```sql
WITH (
  'key1' = 'value1',
  'key2' = 'value2',
  ...
)
```

通用参数

| 参数 | 说明 | 示例 |
| --- | --- | --- |
| `connector` | 连接器类型 | `'kafka', 'jdbc', 'filesystem'` |
| `format` | 数据格式 | `'json', 'csv', 'avro'` |
| `scan.startup.mode` | 源表消费模式 | `'earliest-offset', 'latest-offset'` |
| `sink.parallelism` | Sink并行度 | `'2'` |
| `update-mode` | 更新模式 | `'append', 'upsert'` |


其余各连接器参数参考官方文档

+ 主要用于批处理场景
+ 支持动态分区插入
+ 文件系统/ Hive 连接器支持较好

```sql
CREATE TABLE partitioned_table (
) PARTITIONED BY (dt, hr)
WITH (
  'connector' = 'filesystem',
  'path' = '/data/partitioned',
  'format' = 'parquet'
)
```

### `CREATE TABLE LIKE`子句
`CREATE TABLE LIKE`：用于基于现有表定义创建新表：

```sql
CREATE TABLE new_table LIKE original_table [ ( <like_options> ) ]

<like_options>:
{
   { INCLUDING | EXCLUDING } { ALL | CONSTRAINTS | PARTITIONS }
 | { INCLUDING | EXCLUDING | OVERWRITING } { GENERATED | OPTIONS | WATERMARKS } 
}[, ...]
```

参数：

+ `INCLUDING`：包含指定内容
+ `EXCLUDING`：排除指定内容
+ `OVERWRITING`：覆盖指定内容

示例

```sql
-- 复制表结构但使用不同的 Kafka 主题
CREATE TABLE kafka_backup LIKE kafka_source (
  EXCLUDING OPTIONS
) WITH (
  'connector' = 'kafka',
  'topic' = 'backup_topic',
  ...
)
```

# DML：单表查询语句
## `WITH`语句：建立临时表
`WITH`语句（也称为【公用表】表达式 CTE，Common Table Expression）：允许在查询中定义临时结果集，提高复杂查询的可读性和可维护性。

### `WITH`语句基础语法
```plsql
WITH [RECURSIVE] cte_name [ (column_name [, ...]) ] AS (
  subquery
)
[, ...]
main_query
```

其中：

+ `cte_name`：临时结果集的名称
+ ****`(column_name [, ...])`****：可选，为CTE的列指定名称
+ ****`subquery`****：定义CTE的查询表达式
+ `main_query`：引用 CTE 的主查询

### 多 CTE 串联
```sql
WITH 
-- 第一个CTE：高价值用户
high_value_users AS (
    SELECT user_id 
    FROM users 
    WHERE credit_score > 800
),
-- 第二个CTE：这些用户的订单
hv_user_orders AS (
    SELECT o.*
    FROM orders o
    JOIN high_value_users hvu ON o.user_id = hvu.user_id
)
-- 主查询
SELECT 
    COUNT(*) AS high_value_orders,
    SUM(amount) AS total_amount
FROM hv_user_orders;
```

### 递归 CTE（RECURSIVE）
Flink 1.16+ 支持递归 CTE，用于处理层次结构数据：

```sql
WITH RECURSIVE employee_hierarchy AS (
    -- 基础查询：获取顶级管理者
    SELECT id, name, manager_id, 1 AS level
    FROM employees
    WHERE manager_id IS NULL
    
    UNION ALL
    
    -- 递归部分：获取下属
    SELECT e.id, e.name, e.manager_id, eh.level + 1
    FROM employees e
    JOIN employee_hierarchy eh ON e.manager_id = eh.id
)
SELECT * FROM employee_hierarchy;
```

### 注意事项
1. **作用域**：CTE 仅在当前查询中有效
2. **嵌套深度**：过深的嵌套可能影响可读性
3. **递归限制**：递归 CTE 必须有终止条件
4. **版本兼容**：递归 CTE 需要Flink 1.16+

## `SELECT`语句
### 标准`SELECT`查询骨架
```sql
SELECT [DISTINCT] select_list
FROM table_expression
[ WHERE boolean_expression ]
[ GROUP BY grouping_expression ]
[ HAVING boolean_expression ]
[ ORDER BY order_expression ]
```

**注意**：在无界流上，`ORDER BY`需要后跟 `LIMIT`子句。

### `SELECT DISTINCT`语义分析
```sql
INSERT into target_table   // Kafka
SELECT 
    DISTINCT id 
FROM Orders  // Kafka
```

SQL 语义分析

+ 数据源算子（`FROM Orders`）
    - 连接到 Kafka Topic
    - 数据源算子一直运行，实时的从 Order Kafka 中一条一条的读取数据，然后一条一条发送给下游的**去重算子**
+ 去重算子（`DISTINCT id`）
    - 接收到上游算子发的一条一条的数据，然后判断这个 `id` 之前是否已经来过
    - 判断方式：使用 Flink 中的 State 状态
        * 如果状态中已有该 `id`，说明已来过，则不往下游算子发
        * 如果状态中没有该 `id`，说明未来过，则往下游算子发，也是一条一条发给下游**数据汇算子**
+ 数据汇算子（`INSERT INTO target_table`）
    - 接收到上游发的一条一条的数据，写入到 `target_table` Kafka 中



注意：实时任务**状态的无限增长**

+ 状态大小取决于不同 Key（上述案例为 `id` 字段）的数量
+ 为防止状态无限变大，可以设置状态的 TTL
+ 但是这可能会影响查询结果的正确性，比如某个 Key 的数据过期从状态中删除了，如果后续又来这个 Key，由于在状态中找不到，又会输出一遍

## 时间窗口
### 4 种时间窗口
FlinkSQL 支持四种时间窗口

+ 滚动窗口（Tumble）
    - 将每个元素分配给指定窗口大小的窗口。
    - 滚动窗口具有固定大小，且不重叠。
+ 滑动窗口（Hop）
    - 将元素指定给固定长度的窗口
    - 滑动窗口有两个参数
        * 窗口大小：Window Size
        * 滑动步长：Window Slide
    - 如果滑动的步长小于窗口大小，则滑动窗口之间每个窗口是可以重叠的。在这种情况下，一条数据就会分配到多个窗口当中。
+ 会话窗口（Session）
    - 没有固定的持续时间
    - 如果在定义的间隔期（Session Gap）内没有新的数据出现，则 Session 就会窗口关闭
+ 累积窗口（Cumulate，也称渐进式窗口）
    - 首先开一个最大窗口大小的滚动窗口，然后根据用户设置的触发的窗口间隔，将这个滚动窗口拆分为多个窗口
    - 这些窗口具有相同的窗口起点和不同的窗口终点
    - 需要两个参数
        * 最大窗口大小：Max Window Size
        * 窗口间隔：Window Step

### 窗口表值函数（Window TVF）
TVF：Table-Valued Function（表值函数）  
**定义**：窗口 TVF 是 Flink 1.13 引入的窗口定义方式，将窗口作为表函数使用，通过 `TABLE()`语法调用，返回一个包含窗口元数据的关系表



注：有关 Group Window Aggregation

+ 窗口声明写在`GROUP BY`子句中，Flink 1.13 以前只能使用该方案 
+ 此方案在 1.13 及之后版本已经标记为**废弃**
+ ~~例：~~`~~GROUP BY tumble(row_time, interval '1' MINUTE)~~`



**TVF 基础语法结构**

+ 窗口声明写在`FROM`数据源的 Table 子句中

```sql
SELECT col1,
       col2,
       agg_col
FROM TABLE(窗口函数名(
                      TABLE 数据源表,  -- 声明数据源表
                      DESCRIPTOR(时间列),    -- 声明数据源的时间戳
                      窗口大小参数,   -- 声明窗口大小
                      [其他参数]
                    )
          )
[GROUP BY 窗口字段 + 分组键]
```



窗口元数据字段（`GROUP BY`子句的窗口字段）：所有窗口 TVF 都会自动生成以下元数据列：

1. `window_start`：`TIMESTAMP(3)`，窗口起始时间（含）
2. `window_end`：`TIMESTAMP(3)`，窗口结束时间（不含）
3. `window_time`：`TIMESTAMP_LTZ(3)`，窗口内最大有效时间戳（等于**窗口结束时间 - 1 毫秒**）
    - **核心用途**：用于**时态表关联（Temporal Join）**，确保能关联到窗口关闭前的维表版本
    - **设计意图**：解决 `window_end`不包含在窗口内的问题，`window_end`不属于当前窗口时间范围，用 `window_end`关联可能会得到下一个窗口的维表版本

窗口TVF必须配合`GROUP BY`使用，分组需包含：

+ 至少一个窗口元数据字段（`window_start`/`window_end`）
+ 业务分组键

### 滚动窗口（TUMBLE）
窗口函数名取`TUMBLE`

```sql
-- 数据源表
CREATE TABLE source_table (  
    dim STRING,  -- 维度数据
    user_id BIGINT, -- 用户 id
    price BIGINT,  -- 金额
    -- 事件时间戳
    row_time AS cast(CURRENT_TIMESTAMP as timestamp(3)),
    -- watermark 设置
    WATERMARK FOR row_time AS row_time - INTERVAL '5' SECOND
) WITH (
  ...
)

-- 数据汇表
CREATE TABLE sink_table (
    dim STRING,
    pv BIGINT,
    sum_price BIGINT,
    max_price BIGINT,
    min_price BIGINT,
    uv BIGINT,
    window_start bigint
) WITH (
  'connector' = 'print'
)

-- 数据处理逻辑
INSERT INTO sink_table
SELECT 
      dim,
      UNIX_TIMESTAMP(CAST(window_start AS STRING)) * 1000 as window_start,
      count(*) as pv,
      sum(price) as sum_price,
      max(price) as max_price,
      min(price) as min_price,
      count(distinct user_id) as uv
FROM TABLE(TUMBLE(   -- 声明滚动窗口
        TABLE source_table    -- 数据源表
        , DESCRIPTOR(row_time)   -- 数据源时间戳
        , INTERVAL '60' SECOND
        )
      )   -- 窗口大小为 60 秒
GROUP BY window_start, 
      window_end,
      dim
```



SQL 语义分析

+ 数据源算子：算子一直运行，从数据源一条一条读取数据，一条一条发送给下游窗口聚合算子
+ 窗口聚合算子（`TUMBLE` 算子）
    - 接收到上游算子发的一条一条的数据，将每一条数据按照时间戳划分到对应的窗口中（根据事件时间、处理时间的不同语义进行划分）
    - 上述案例为事件时间，滚动窗口算子接收到**上游的 Watermark 大于窗口的结束时间时**，则说明当前这一分钟的滚动窗口已经结束了，将窗口计算完的结果发往下游算子（一条一条发给下游数据汇算子）
+ 数据汇算子：算子一直运行，接收上游发送的一条一条数据后，写入外部存储



**注意：事件时间中滚动窗口的窗口计算触发是由 Watermark 推动的。**

### 滑动窗口（HOP）
窗口函数名取`HOP`，需要同时给出滑动步长和窗口大小

```sql
-- 数据源表
CREATE TABLE source_table (  
    dim STRING,  -- 维度数据
    user_id BIGINT, -- 用户 id
    price BIGINT,  -- 金额
    -- 事件时间戳
    row_time AS cast(CURRENT_TIMESTAMP as timestamp(3)),
    -- watermark 设置
    WATERMARK FOR row_time AS row_time - INTERVAL '5' SECOND
) WITH (
  ...
)

-- 数据汇表
CREATE TABLE sink_table (
    dim STRING,
    uv BIGINT,
    window_start bigint
) WITH (
  'connector' = 'print'
);

-- 数据处理逻辑
INSERT INTO sink_table
SELECT 
    dim,
    UNIX_TIMESTAMP(CAST(window_start AS STRING)) * 1000 as window_start, 
    count(distinct user_id) as bucket_uv
FROM TABLE(HOP(   -- 声明滑动敞口
        TABLE source_table
        , DESCRIPTOR(row_time)
        , INTERVAL '1' MINUTES -- 第一个参数：滑动步长
        , INTERVAL '5' MINUTES  -- 第二个参数：窗口大小
          )
      )
GROUP BY window_start, 
      window_end,
      dim
```

### 会话窗口（SESSION）
窗口函数名为`SESSION`，需给出会话的超时时长

```sql
-- 计算用户会话内的行为统计
SELECT
  window_start,
  window_end,
  user_id,
  COUNT(*) AS events_count
FROM TABLE(
  SESSION(
    TABLE user_activity,
    DESCRIPTOR(activity_time),
    INTERVAL '30' MINUTES   -- 会话超时时长
  )
)
GROUP BY window_start, window_end, user_id;
```

### 累积窗口（CUMULATE）
将数据按**固定步长**累积计算，直到达到**最大窗口大小**

+ 窗口函数`CUMULATE`
+ 窗口间隔 Window Step
+ 最大窗口大小 Max Window Size

```sql
-- 数据源表
CREATE TABLE source_table (
    user_id BIGINT,-- 用户 id
    money BIGINT,   -- 金额
    -- 事件时间戳
    row_time AS cast(CURRENT_TIMESTAMP as timestamp(3)),
    -- watermark 设置
    WATERMARK FOR row_time AS row_time - INTERVAL '5' SECOND
) WITH (
  ...
);

-- 数据汇表
CREATE TABLE sink_table (
    window_end bigint,
    window_start bigint,
    sum_money BIGINT,
    count_distinct_id bigint
) WITH (
  'connector' = 'print'
);

-- 数据处理逻辑
INSERT INTO sink_table
SELECT 
    UNIX_TIMESTAMP(CAST(window_end AS STRING)) * 1000 as window_end, 
    window_start, 
    sum(money) as sum_money,
    count(distinct id) as count_distinct_id
FROM TABLE(CUMULATE(
       TABLE source_table
       , DESCRIPTOR(row_time)
       , INTERVAL '60' SECOND  -- 累积步长，窗口每次扩展的时间间隔
       , INTERVAL '1' DAY)) -- 最大窗口大小，窗口停止增长的最大时间范围
GROUP BY
    window_start, 
    window_end
```

步长：5 分钟，最大窗口 15 分钟；窗口生成序列（假设从 00: 00 开始）  
`[00:00,00:05), [00:00,00:10), [00:00,00:15)`

### TVF 支持 Grouping Sets、Rollup、Cube
Grouping Sets 只在 Window TVF 中支持

示例：计算每日零点累计到当前这一分钟的分汇总、age、sex、age+sex 维度的用户数

```sql
-- 用户访问明细表
CREATE TABLE source_table (
    age STRING,
    sex STRING,
    user_id BIGINT,
    row_time AS cast(CURRENT_TIMESTAMP as timestamp(3)),
    WATERMARK FOR row_time AS row_time - INTERVAL '5' SECOND
) WITH (
  ...
);

CREATE TABLE sink_table (
    age STRING,
    sex STRING,
    uv BIGINT,
    window_end bigint
) WITH (
  'connector' = 'print'
);

INSERT INTO sink_table
SELECT 
    UNIX_TIMESTAMP(CAST(window_end AS STRING)) * 1000 as window_end, 
    if (age is null, 'ALL', age) as age,
    if (sex is null, 'ALL', sex) as sex,
    count(distinct user_id) as bucket_uv
FROM TABLE(CUMULATE(
       TABLE source_table
       , DESCRIPTOR(row_time)
       , INTERVAL '5' SECOND
       , INTERVAL '1' DAY))
GROUP BY
```

## Group By 聚合
### Group By 聚合与时间窗口聚合对比
示例：滚动窗口聚合，转换为`GROUP BY`聚合

```sql
-- 数据源表
CREATE TABLE source_table (
    dim STRING,  -- 维度数据
    user_id BIGINT,  -- 用户 id
    price BIGINT,  -- 用户
    -- 事件时间戳
    row_time AS cast(CURRENT_TIMESTAMP as timestamp(3)),
    -- watermark 设置
    WATERMARK FOR row_time AS row_time - INTERVAL '5' SECOND
) WITH (
 ...
)

-- 数据汇表
CREATE TABLE sink_table (
    dim STRING,
    pv BIGINT,
    sum_price BIGINT,
    max_price BIGINT,
    min_price BIGINT,
    uv BIGINT,
    window_start bigint
) WITH (
  'connector' = 'print'
)

-- 数据处理逻辑
-- 滚动窗口聚合
insert into sink_table
select dim,
    count(*) as pv,
    sum(price) as sum_price,
    max(price) as max_price,
    min(price) as min_price,
    -- 计算 uv 数
    count(distinct user_id) as uv,
    UNIX_TIMESTAMP(CAST(tumble_start(row_time, interval '1' minute) AS STRING)) * 1000  as window_start
from source_table
group by
    dim,
    -- 按照 Flink SQL tumble 窗口写法划分窗口
    tumble(row_time, interval '1' minute)

-- group by 聚合
insert into sink_table
select dim,
    count(*) as pv,
    sum(price) as sum_price,
    max(price) as max_price,
    min(price) as min_price,
    -- 计算 uv 数
    count(distinct user_id) as uv,
    cast((UNIX_TIMESTAMP(CAST(row_time AS STRING))) / 60 as bigint) as window_start
from source_table
group by
    dim,
    -- 将秒级别时间戳 / 60 转化为 1min
    cast((UNIX_TIMESTAMP(CAST(row_time AS STRING))) / 60 as bigint)
```



如果是离线批处理，滚动窗口聚合和 Group By 聚合可以等同。

但是针对流处理，窗口聚合和 Group By 聚合的差异在于：窗口聚合具有**时间语义**而 Group By 没有

+ 窗口聚合：
    - 计算触发：由时间（Watermark） 推动
    - 计算结果输出后，输出结果值是确定不变的，迟到数据不会改变原有结果
+ Group By 聚合：
    - 计算触发：数据推动，新数据来了就会触发计算
    - 产生撤回流：新数据来了，撤回上次输出结果，发出新的计算结果

### Group By 聚合 SQL 语义分析
+ 数据源算子（`FROM source_table`）：
    - 数据源算子一直运行，实时从数据源中读取数据，然后逐条发送给下游的 Group By 算子
    - 向下游发送数据的 Shuffle 策略：根据 Group By 中的 key 进行发送，相同 key 的数据发到同一个 SubTask（并发）
+ Group By 聚合算子（`GROUP BY key + SUM\COUNT\MAX\MIN`）：
    - 接收到上游算子发来的数据，去状态 State 中找这个 key 之前的 `SUM\COUNT\MAX\MIN` 结果
        * 如果有结果 `oldResult`：
            + 拿出来和当前的数据进行 `SUM\COUNT\MAX\MIN`，计算出这个 key 的新结果 `newResult`，并将新结果 `[key, newResult]`更新到 State 中
            + 在向下游发送新计算的结果之前，先发一条撤回上次结果的消息 `-[key, oldResult]`，然后再将新结果发往下游`+[key, newResult]`
        * 如果 State 中没有当前 key 的结果
            + 则直接使用当前这条数据计算 `SUM\COUNT\MAX\MIN` 结果 `newResult`，并将新结果` [key, newResult]`更新到 State 中
            + 当前是第一次往下游发，则不需要先发回撤消息，直接发送 `+[key, newResult]`
+ 数据汇算子：（`INSERT INTO target_table`）
    - 接收到上游发的一条一条的数据，写入到外部存储中



注意：

+ **Group by 聚合涉及到了撤回流**（也叫 Retract 流）
+ **Group by 聚合涉及到了状态**：状态大小也取决于不同 key 的数量。为了防止状态无限变大，我们可以设置状态的 TTL

### Group By 聚合支持 Grouping sets、Rollup、Cube
Group 聚合也支持 Grouping sets、Rollup、Cube

```sql
SELECT 
    supplier_id
    , rating
    , product_id
    , COUNT(*)
FROM (VALUES
    ('supplier1', 'product1', 4),
    ('supplier1', 'product2', 3),
    ('supplier2', 'product3', 3),
    ('supplier2', 'product4', 4))
AS Products(supplier_id, product_id, rating)
GROUP BY GROUPING SET (
    ( supplier_id, product_id, rating ),
    ( supplier_id, product_id         ),
    ( supplier_id,             rating ),
    ( supplier_id                     ),
    (              product_id, rating ),
    (              product_id         ),
    (                          rating ),
    (                                 )
)
```

## 聚合函数
+ 所有聚合函数自动忽略NULL值
+ `COUNT(*)`计算所有行，`COUNT(col)`只计算非`NULL`值
+ `SUM/AVG`只支持数值类型
+ `MIN/MAX`支持所有可比较类型（包括`STRING`、`TIMESTAMP`等）

### 基础聚合函数
| 函数名 | 返回值类型 | 功能描述 |
| --- | --- | --- |
| `COUNT(*)` | `BIGINT` | 统计行数 |
| `COUNT([DISTINCT] col)` | `BIGINT` | 统计非`NULL`值（可去重） |
| `SUM([DISTINCT] col)` | 与输入相同 | 求和（数值类型） |
| `AVG([DISTINCT] col)` | `DOUBLE` | 平均值（数值类型） |
| `MIN(col)` | 与输入相同 | 最小值 |
| `MAX(col)` | 与输入相同 | 最大值 |


### 高级聚合函数
| 函数名 | 返回值类型 | 功能描述 | 示例 |
| --- | --- | --- | --- |
| `COLLECT(col)` | `MULTISET` | 收集所有值到多重集 | `SELECT COLLECT(product_id) FROM orders GROUP BY user_id` |
| `LISTAGG(col [, delimiter])` | `STRING` | 连接字符串（可指定分隔符） | `SELECT LISTAGG(name, ',') FROM products GROUP BY category` |


### 统计聚合函数
| 函数名 | 返回值类型 | 功能描述 | 示例 |
| --- | --- | --- | --- |
| `STDDEV_POP(col)` | `DOUBLE` | 总体标准差 | `SELECT STDDEV_POP(score) FROM exams` |
| `STDDEV_SAMP(col)` | `DOUBLE` | 样本标准差 | `SELECT STDDEV_SAMP(score) FROM exams` |
| `VAR_POP(col)` | `DOUBLE` | 总体方差 | `SELECT VAR_POP(price) FROM stocks` |
| `VAR_SAMP(col)` | `DOUBLE` | 样本方差 | `SELECT VAR_SAMP(price) FROM stocks` |
| `CORR(col1, col2)` | `DOUBLE` | 皮尔逊相关系数 | `SELECT CORR(price, sales) FROM products` |
| `COVAR_POP(col1, col2)` | `DOUBLE` | 总体协方差 | `SELECT COVAR_POP(temp, humidity) FROM sensors` |
| `COVAR_SAMP(col1, col2)` | `DOUBLE` | 样本协方差 | `SELECT COVAR_SAMP(temp, humidity) FROM sensors` |

## 开窗函数（`Over`聚合）
```sql
SELECT agg_func(agg_col) OVER (
        [PARTITION BY col1[, col2, ...]]
        ORDER BY time_col
        range_definition),
  ...
FROM ...
```

其中

+ `ORDER BY`：必须是时间戳列（事件时间、处理时间）
+ `PARTITION BY`：标识了聚合窗口的聚合粒度
+ `range_definition`：标识聚合窗口的聚合数据范围，在 Flink 中有两种指定数据范围的方式
    - 按照行数聚合：`ROWS BETWEEN 5 PRECEDING AND CURRENT ROW`
    - 按照时间区间聚合：`RANGE BETWEEN INTERVAL '1' HOUR PRECEDING AND CURRENT ROW`

## 集合操作
1. `UNION`：将集合合并，且去重
2. `UNION ALL`：将集合合并，不做去重
3. `INSERSECT`：交集，并且去重
4. `INSERSECT ALL`：交集不做去重
5. `EXCEPT`：差集，并且去重
6. `EXCEPT ALL`：差集，不做去重
7. `IN`子查询：只支持一列，也会涉及大状态问题，需要设置 TTL



集合操作为撤回流

+ 流式任务中，如果一条左流数据先来了，没有从右流集合数据中找到对应的数据时会直接输出
+ 当右流对应数据后续来了之后，会下发回撤流将之前的数据給撤回

## `ORDER BY`、`LIMIT` 子句
`Order By` 和 `Limit` 实时任务中一般用的非常少



实时任务中，`Order By` 子句中必须要**有时间属性字段**，并且时间属性必须为**升序时间属性**

+ 即 `WATERMARK FOR rowtime_column AS rowtime_column - INTERVAL '0.001' SECOND`
+ 或者 `WATERMARK FOR rowtime_column AS rowtime_column`

# 列转行
## Array Expansion（数组列转行）
+ 应用场景（支持 Batch\Streaming）：将表中 ARRAY 类型字段（列）拍平，转为多行
+ 实际案例：比如某些场景下，日志是合并、攒批上报的，就可以使用这种方式将一个 Array 转为多行

```sql
CREATE TABLE show_log_table (
    log_id BIGINT,
    show_params ARRAY<STRING>
) WITH ();
-- 原始数据：+I[7, [a, b, c]]

CREATE TABLE sink_table (
    log_id BIGINT,
    show_param STRING
) WITH (
  'connector' = 'print'
);

INSERT INTO sink_table
SELECT
    log_id,
    t.show_param as show_param
FROM show_log_table
-- array 炸开语法
CROSS JOIN UNNEST(show_params) AS t (show_param)

```

## Table Function（自定义列转行）
应用场景（支持 Batch\Streaming）：和 Array Expansion 功能类似，但是 Table Function 本质上是个 UDTF 函数，和离线 Hive SQL 一样，我们可以自定义 UDTF 去决定列转行的逻辑



Table Function 使用分类

+ Inner Join Table Function：如果 UDTF 返回结果为空，则相当于 1 行转为 0 行，这行数据直接被丢弃
+ Left Join Table Function：如果 UDTF 返回结果为空，折行数据不会被丢弃，只会在结果中填充 null 值



示例

定义 UDTF

```java
public static class UserProfileTableFunction extends TableFunction<Integer> {
        public void eval(long userId) {// 自定义输出逻辑
            if (userId <= 5) {
                collect(1);   // 一行转 1 行
            } else {// 一行转 3 行
                collect(1);
                collect(2);
                collect(3);
            }
        }
    }
```

注册后在 FlinkSQL 中使用：`LATERAL TABLE`

```sql
SELECT user_id,
       name, 
       age, 
       row_time 
FROM source_table,
LATERAL TABLE(user_profile_table_func(user_id)) t(age);
```

## SQL Hints
生产环境中不建议使用

+ 在查询中动态提供优化信息，覆盖表配置。
+ 应用场景：临时的参数修改
+ 示例：有一个 kafka 数据源表 `kafka_table1`，用户想直接从 `latest-offset `Select 一些数据出来预览，其元数据已经存储在 Hive MetaStore 中，但是 Hive MetaStore 中存储的配置中的 `scan.startup.mode` 是 `earliest-offset`，通过 SQL Hints，用户可以在 DML 语句中将 `scan.startup.mode` 改为 `latest-offset`查询

语法

```sql
SELECT *
FROM table_path /*+ OPTIONS(key=val [, key=val]*) */
```
