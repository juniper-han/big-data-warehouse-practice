侧输出（Side Output） 是 Flink 提供的一种灵活的**数据分流机制**

+ 允许在**单个算子中，**将数据流按业务逻辑拆分为多个独立的输出流
+ 适用于处理“主路径 + 异常路径”、“正常数据 + 告警数据”等场景

# 分流方法
分流：把输入源按照需要进行拆分



通常来说针对不同的场景，有以下 2 种办法进行流的拆分。

+ Filter 分流：需要重复遍历原始流，浪费资源，不建议使用
    - 根据用户输入的条件进行过滤，每个元素都会被` filter()` 函数处理，如果 `filter() `函数返回 `true` 则保留，否则丢弃。
+ Side OutPut 分流：侧输出，推荐

# 侧输出的概念与原理
## 侧输出定义
+ 主输出（Main Output）：算子默认的输出流
    - 通过 `collect()` 发送
+ 侧输出（Side Output）：通过 `OutputTag` 标识的额外输出流
    - 通过 `context.output(tag, value)` 发送）
+ 特点：
    - 一个算子可定义多个侧输出；
    - 侧输出流与主输出流完全独立，可分别进行后续处理；
    - 侧输出不会影响主流程的语义和性能

## 工作原理

+ 关键组件：`OutputTag<T>`（泛型标签，用于标识侧输出流）；
+ 底层实现：Flink 在算子内部维护一个`Map<OutputTag, Collector>`，将不同标签的数据路由到不同下游

✅ 优势：避免为简单分流逻辑拆分多个算子，减少 DAG 复杂度。

## 典型应用场景
| 场景 | 说明 |
| --- | --- |
| 异常数据分离 | 将格式错误、空值、超限数据分流到侧输出，主流程只处理干净数据 |
| 多级告警 | 正常数据走主输出，Warning/ Error 级别事件走不同侧输出 |
| A/B 测试分流 | 按用户 ID 尾号分流到不同实验组 |
| 延迟数据处理 | 在窗口中，将迟到数据发送到侧输出（Flink 窗口原生支持） |
| 数据打标 | 为主数据流附加元信息（如来源、优先级） |


核心价值：单一职责 + 逻辑解耦

+ 主流程专注核心业务，异常/辅助逻辑独立处理。

# 使用侧输出（DataStream API）
## 步骤 1：定义 `OutputTag`
```java
// 定义侧输出标签（必须是静态 final）
private static final OutputTag<String> LATE_DATA_TAG = 
    new OutputTag<String>("late-data") {};

private static final OutputTag<String> INVALID_DATA_TAG = 
    new OutputTag<String>("invalid-data") {};
```

## 步骤 2：在 `ProcessFunction` 中发送数据
```java
SingleOutputStreamOperator<String> mainStream = inputStream
    .process(new ProcessFunction<String, String>() {
        @Override
        public void processElement(String value, Context ctx, Collector<String> out) {
            try {
                // 主流程：解析并验证数据
                MyEvent event = parse(value);
                if (isValid(event)) {
                    out.collect(format(event)); // 主输出
                } else {
                    ctx.output(INVALID_DATA_TAG, value); // 侧输出：无效数据
                }
            } catch (Exception e) {
                ctx.output(INVALID_DATA_TAG, value); // 侧输出：解析失败
            }
        }
    });
```

## 步骤 3：获取侧输出流并处理
```java
// 从主算子结果中提取侧输出
DataStream<String> invalidStream = mainStream.getSideOutput(INVALID_DATA_TAG);

// 对侧输出流进行独立处理
invalidStream
    .map(error -> "ERROR: " + error)
    .addSink(new AlertSink()); // 发送到告警系统

// 主输出继续正常处理
mainStream.addSink(new MainSink());
```

⚠️ 注意：

+ `OutputTag` 必须是 `static final`（因涉及序列化）；
+ 侧输出流必须在定义它的算子之后立即获取（不能跨算子链）

# FlinkSQL 不支持侧输出
Flink SQL 本身不支持显式定义侧输出。

因为：

+ Flink SQL 的设计哲学是 “声明式”，强调逻辑表转换，而非过程式分流；
+ 但可通过 “隐式侧输出” 或 “SQL + DataStream 混合” 实现类似效果。

## 替代方案 1：使用 `CASE WHEN` + 多 Sink（最常用）
```plsql
-- 将数据打标
CREATE VIEW tagged_data AS
SELECT 
  data,
  CASE 
    WHEN data IS NULL OR data = '' THEN 'invalid'
    WHEN CAST(ts AS BIGINT) < 0 THEN 'late'
    ELSE 'valid'
  END AS data_type
FROM input_table;

-- 主输出：有效数据
INSERT INTO main_sink 
SELECT data FROM tagged_data WHERE data_type = 'valid';

-- 侧输出：无效数据
INSERT INTO alert_sink 
SELECT data FROM tagged_data WHERE data_type = 'invalid';
```

✅ 优点：纯 SQL 实现，无需编码；

❌ 缺点：数据被多次扫描（非物理分流）。

## 替代方案 2：SQL + DataStream 混合编程
```java
// 1. 用 SQL 处理主逻辑
Table resultTable = tEnv.sqlQuery("SELECT ... FROM input");

// 2. 转换为 DataStream
DataStream<Row> stream = tEnv.toRetractStream(resultTable, Row.class)
    .filter(t -> t.f0) // retract stream 需过滤 insert
    .map(t -> t.f1);

// 3. 在 DataStream 中使用侧输出
SingleOutputStreamOperator<Row> main = stream.process(...);

// 4. 获取侧输出并转回 Table（可选）
DataStream<String> side = main.getSideOutput(INVALID_TAG);
Table sideTable = tEnv.fromDataStream(side, $("data"));
tEnv.createTemporaryView("side_output", sideTable);
```

 优点：结合 SQL 简洁性 + DataStream 灵活性；

❌ 缺点：需混合编程，复杂度增加。

## 替代方案 3：利用 内置侧输出（特定场景）
Flink SQL 的某些算子原生支持侧输出，例如：

窗口的迟到数据（Watermark 机制）

```sql
-- Flink 1.17+ 支持
CREATE TABLE windowed (
  ...
) WITH (
  'sink.late-data.side-output' = 'true' -- 启用迟到数据侧输出
);

-- 但目前仍需 DataStream API 获取侧输出流
```

截至 Flink 1.18，SQL 无法直接消费内置侧输出，仍需转 DataStream

# 注意事项
✅ 最佳实践

+ 标签命名规范：`new OutputTag<>("alert-invalid-user") {}`；
+ 避免过度分流：侧输出不宜超过 3~5 个，否则逻辑混乱；
+ 监控侧输出量：异常数据突增可能是上游故障信号；
+ 与主流程解耦：侧输出处理失败不应影响主流程（如用独立 Sink）。

⚠️ 注意事项

+ 序列化要求：`OutputTag` 的泛型类型必须可序列化；
+ 作用域限制：侧输出流只能在定义它的算子下游获取；
+ Checkpoint 一致性：侧输出与主输出共享同一 Checkpoint，保证 exactly-once。
