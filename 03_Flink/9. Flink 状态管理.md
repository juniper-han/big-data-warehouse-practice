# 状态的概念
## 什么是状态
状态（State）：计算过程中需要记住的中间结果或上下文。

State 使得 Flink 能在处理无界数据流时，记住历史信息，从而实现复杂事件处理、窗口聚合、Top-N、去重等高级功能。



状态的必要性

+ 无状态计算：输出仅依赖当前输入，如`map, filter`
+ 有状态计算：输出依赖历史输入，如`sum, count, window, keyBy + process`

## 状态的分类体系
可以从两个维度对状态进行分类

### 按作用域分类
| 类型 | 作用域 | 隔离性 | 典型场景 |
| --- | --- | --- | --- |
| Keyed State | `keyBy()` 后的每个 key | 每个 key 独立 | 用户行为分析、实时聚合 |
| Operator State | 整个算子实例 | 每个并行子任务独立 | Source offset、全局计数器 |


+ 90%+ 的业务逻辑使用 Keyed State

### 按管理方式分类
| 类型 | 说明 | 推荐度 |
| --- | --- | --- |
| Managed State | Flink Runtime 托管，自动序列化/分区/恢复 | ✅ 强烈推荐 |
| Raw State | 用户自管理序列化与分区 | ❌ 仅限 Flink 内部组件 |


Keyed State 和 Operator State 均为 Managed State

# Keyed State 键控状态
## 简介
Keyed State 使得 Flink 能够在`keyBy()`分区后的流上，为每个`key`维护独立的状态

从而实现如“每个用户最近登录时间”、“每个商品实时销量”等按维度精细化有状态计算。

+ 作用域：仅在`keyBy()` 之后的算子中可用；
+ 隔离性：每个 `key` 的状态完全隔离，互不影响；
+ 并行性：Flink 自动将不同 `key` 的状态分配到不同 Task 实例；
+ 语义：相当于 “每个 `key` 一个独立的状态机”



Keyed State 必须在 `RichFunction` 中通过 `RuntimeContext` 获取：

```java
ValueState<Long> countState = getRuntimeContext()
    .getState(new ValueStateDescriptor<>("count", Long.class));
```

## 类型
Flink 提供 5 种 Managed Keyed State，均由 Runtime 托管，自动序列化、分区、恢复

1. `ValueState<T>`
    - 语义：单值状态，类似 `Map<KEY, T>` 中的一个 `value`；
    - API：`value()`, `update(T)`, `clear()`
    - 存储：直接存 `T` 的序列化字节。
    - 典型场景：保存最新值、计数器、标志位，例如：记录每个用户的最后登录时间
2. `ListState<T>`
    - 语义：列表状态，保存多个值；
    - API：`add(T), addAll(List<T>), get() → Iterable<T>`
    - 典型场景：缓存最近 N 条记录、窗口数据暂存
    - 注意：不保证顺序；避免存储过大列表
3. `MapState<K, V>`高性能之选
    - 语义：Map 结构状态，支持 key-value 存储；
    - API：`put(K, V), get(K), contains(K), entries()`
    - 典型场景：用户画像（用户 → 行为类型 → 次数）、多维指标
    - 优势
        * RocksDB 下，每个 entry 独立存储；
        * 查询/更新单个 key 无需反序列化整个 Map；
        * 内存和 I/O 效率远高于 `ValueState<Map<K,V>>`。
4. `ReducingState<T>`
    - 语义：自动聚合的状态，每次更新都触发 reduce 函数；
    - API：`add(T)`（自动触发 reduce）
    - 优势：只存聚合结果，内存占用小
    - 场景：`sum`、`max`、`min` 等可结合操作
5. `AggregatingState<IN, OUT>`
    1. 语义：带转换的聚合状态，输入类型 ≠ 输出类型；
    2. 核心：需实现 `AggregateFunction<IN, ACC, OUT>`
    3. 场景：`avg`、`variance` 等复杂聚合
    4. 存储：实际存的是 `accumulator (ACC)`，非最终结果 `OUT`

## Keyed State 的生命周期
1. 创建
    - 在`RichFunction.open()` 中通过 `RuntimeContext` 获取；
    - 首次访问时自动初始化为空（`null` 或 `empty`）
2. 更新
    - `ValueState.update()`
    - `ListState.add(), MapState.put()`
    - `ReducingState.add(), AggregatingState.add()`
3. 清除
    - 手动清除：`state.clear()`
    - 自动清除：
        * 作业取消（除非保留 Savepoint）；
        * State TTL（推荐方式）



注意：`clear()` 只清除当前 key 的状态；没有“清除所有 key 状态”的 API（需遍历所有 key，不现实）

## 访问模式
+ 读取：`state.value(), mapState.get(key)`
+ 写入：`state.update(val), mapState.put(key, val)`
+ 清除：`state.clear()`（清除当前 key 的所有状态）
+ ⚠️ 重要：`clear()` 不会立即释放物理内存（尤其 RocksDB），需配合 TTL。

## Keyed State 的底层存储
Keyed State 底层存储机制：高度依赖 State Backend 的实现

+ 不同的状态后端，有不同的存储方式

### 核心抽象：Keyed State 的逻辑视图
从用户角度看，Keyed State 是一个 嵌套 Map：

```java
Map<Key, Map<StateName, StateValue>>
```

+ 外层 Key：来自 `keyBy()` 的分区键（如 `user_id`）；
+ 内层 
    - `StateName`：状态描述符的名称（如 `"last-login"`）；
    - `StateValue`：实际状态值，物理内容和结构，由 Keyed State 类型（`ValueState`、`ListState`、`MapState`、`ReducingState`、`AggregatingState`）决定



关键点：Flink 必须将这个逻辑结构高效映射到物理存储

### 物理存储的核心：State Backend 实现
Flink 提供三种 State Backend，在 Keyed State 的底层存储方面，差异巨大：

| Backend | 运行时存储 | Checkpoint 存储 | 适用场景 |
| --- | --- | --- | --- |
| MemoryStateBackend | JVM Heap (HashMap) | JobManager Heap | 本地测试 |
| FsStateBackend | JVM Heap (HashMap) | 分布式文件系统（HDFS/S3） | 中小状态 |
| <font style="background-color:#E8F7CF;">RocksDBStateBackend</font> | <font style="background-color:#E8F7CF;">本地磁盘（RocksDB）</font> | <font style="background-color:#E8F7CF;">分布式文件系统</font> | <font style="background-color:#E8F7CF;">✅</font><font style="background-color:#E8F7CF;"> 生产大状态</font> |


### RocksDBStateBackend 下的 Keyed State 存储结构
整体架构

```plain
TaskManager
└── Task (subtask)
    └── Keyed State Operator
        └── RocksDB Instance (per subtask)
            ├── ColumnFamily: "state-namespace-1"  ← 每个 StateDescriptor 一个 CF
            ├── ColumnFamily: "state-namespace-2"
            └── ...
```

+ 每个算子子任务（subtask）拥有独立的 RocksDB 实例；
+ 每个 StateDescriptor（即每种状态）对应一个 ColumnFamily（CF）；
+ 所有 key 的状态都存入同一个 RocksDB，但通过 复合 Key 编码 隔离。



Key 的编码格式

+ RocksDB 是 KV 存储，Flink 必须将`(key, stateName)`编码为单一`byte[]`作为 RocksDB 的`key`
+ 完整 Key 结构：`[KeyGroupPrefix (2 bytes)] + [KeySerializer] + [NamespaceSerializer]`



Value 的存储

+ 直接存储状态值的序列化结果；
+ 不同状态类型存储方式一致（RocksDB 不感知 `ValueState`/`MapState` 语义）；
+ `MapState` 的特殊处理：
    - 逻辑上是一个 Map，但物理上 每个 entry 单独存为一条 KV；
        * Key: `[KeyGroup][UserKey][MapKey]`
        * Value: `MapValue`
    - 优势：可单独更新/查询某个 map entry，无需反序列化整个 Map



Key Group：状态分片与扩缩容基石

+ 逻辑分片单元，数量 = Max Parallelism
+ 每个 subtask 负责连续的 Key Group 范围
+ 扩缩容时，Key Group 是最小迁移单元
    - 如果没有 Key Group，直接按 `key.hashCode() % parallelism` 分区 → 扩容时所有 key 重分布；

# Operator State 算子状态
## 简介
Operator State 算子状态是：

+ 作用域：绑定到整个算子的并行子任务（Subtask）上
+ 隔离性：每个**并行子任务**拥有自己独立的一份 Operator State；
+ 语义：用于存储，与数据流中的具体 key 无关的，全局元数据或上下文信息
+ 理解：Operator State 是 Flink 连接“流处理”与“外部世界”的桥梁，它让 Flink 能记住自己从哪里读数据（Source），也能实时响应外部指令（Broadcast）



与 Keyed State 的区别

+ Keyed State 是 “数据驱动” 的（每个 key 有状态）；
+ Operator State 是 “算子驱动” 的（每个 Task 有状态）。

| 维度 | Keyed State | Operator State |
| --- | --- | --- |
| 作用域 | 每个 key | 每个算子子任务 |
| 触发条件 | 必须在 `keyBy()` 之后 | 任何算子均可 |
| 数据关联 | 与数据 key 强绑定 | 与数据内容无关 |
| 典型场景 | 用户行为聚合、窗口计算 | Source offset、全局计数器 |


+ 如果状态与 数据中的某个字段（key） 强相关 → Keyed State；
+ 如果状态描述 算子自身的工作进度或全局上下文 → Operator State。

## 类型
Operator State 有三种 Managed State 类型，均由 Runtime 托管：

### `ListState<T>` 【最常用】
+ 语义：一个列表，存储多个同类型元素；
+ 扩缩容行为：支持 状态重分布（Redistribution）；
+ 典型场景：Kafka Source 的分区偏移量（Partition Offsets）
+ 示例场景：全局错误计数器



工作原理（以 Kafka Source 为例）：

+ 假设 Kafka Topic 有 4 个 Partition：`[P0, P1, P2, P3]`；
+ 作业并行度 = 2：
    - Subtask 0 负责 `[P0, P1]` → `ListState` 存` [Offset0, Offset1]`
    - Subtask 1 负责` [P2, P3]` → `ListState` 存` [Offset2, Offset3]`
+ 扩容到并行度=4：
    - Flink 将所有 offset 合并成一个大列表 `[O0,O1,O2,O3]`；
    - 均匀分配给 4 个新 Subtask（如每个负责 1 个 partition）。

✅ 优势：天然支持动态扩缩容，保证数据不丢不重。

### `UnionListState<T>`
+ 语义：一个列表，但扩缩容时全量广播给所有新 Subtask；
+ 扩缩容行为：每个新 Subtask 都会收到完整的状态列表；
+ 典型场景：需要全局一致视图的初始化数据（较少用）。

示例场景：

+ 初始化规则列表，所有 Task 需要相同的完整规则集；
+ 扩容后，新 Task 也能获得全部历史规则。

### `BroadcastState<K, V>` 【特殊且重要】
+ 语义：Map 结构的广播状态；
+ 关键特性：
    - 只能通过 `BroadcastStream` 更新；
    - 全量复制到每个算子子任务；
    - 只写不读（从 `BroadcastStream` 写入，主数据流可读）。
+ 典型场景：动态维表、实时规则引擎。
+ 场景示例：事实更新的风控规则
+ 注意：`BroadcastState` 是 Operator State 的一种，但它不能通过 `RuntimeContext` 获取，只能在 `BroadcastProcessFunction` 中访问。

## Operator State 的生命周期
1. 创建与获取
    - 必须在 `RichFunction` 中（如 `RichSourceFunction`, `RichFlatMapFunction`）；
    - 通过 `CheckpointedFunction` 接口（而非 `RuntimeContext`）
2. 关键接口：`CheckpointedFunction`
    - `initializeState()`: 作业启动/恢复时调用；
    - `snapshotState():` Checkpoint 时调用，保存当前状态



与 Keyed State 的区别：

+ Keyed State 通过 `RuntimeContext.getState()` 获取；
+ Operator State 通过 `FunctionInitializationContext.getOperatorStateStore()` 获取

## Operator State 的底层存储
+ 逻辑视图：`Map<SubtaskIndex, StateValue>`
+ 物理存储：
    - MemoryStateBackend：JVM Heap HashMap；
    - FsStateBackend：JVM Heap + Checkpoint 到 HDFS；
    - RocksDBStateBackend：JVM Heap + Checkpoint 到 HDFS；



注意：针对算子状态，三种状态后端

+ 在运行时，算子状态都存储在 TM 的堆内存中
+ Checkpoint 时，FsStateBackend 和 RocksDBStateBackend 会将算子状态写入外部文件系统
    - 二者行为一致
    - 不会将算子状态放入 RocksDB 中

## 缩扩容行为（Rescaling）
| State 类型 | 扩容行为 | 缩容行为 | 适用场景 |
| --- | --- | --- | --- |
| `ListState` | 拆分列表（Round-Robin 分配） | 合并列表 | Kafka offset、分片任务 |
| `UnionListState` | 全量广播（每个新 Task 得到全部） | 全量广播 | 全局配置、初始化数据 |
| `BroadcastState` | 自动全量复制 | 自动全量复制 | 动态规则、维表 |


## 注意
1. 优先使用 `ListState`：它是唯一支持高效重分布的类型；
2. 避免大对象：Operator State 也会受内存限制；
3. 设置 TTL？：**Operator State 不支持 TTL！需自行管理生命周期。**

# 状态存储与状态后端
Flink 状态存储

+ 默认情况：所有状态存储在 JVM 的堆内存中
+ 状态数据过多，为避免 OOM，Flink 该提供了其它存储方式，这些存储方式统一称为状态后端

状态后端主要有 3 种

1. MemoryStaeBackend
    - 适用场景：本地开发测试，仅用于验证逻辑。
2. FsStateBackend
    - 适用场景：中小状态生产环境，或对 Checkpoint 速度要求不高时。
3. RockDBStateBackend
    - 适用场景：所有生产级有状态作业。

## 整体架构：三层模型
<img src="https://cdn.nlark.com/yuque/0/2026/png/54426354/1771404916307-16df722d-0803-4ef4-a4cb-e372c0e8c7aa.png" width="957" title="" crop="0,0,1,1" id="ALELK" class="ne-image">

Flink 通过分层设计，将抽象的状态概念落地为高效、可扩展、容错的物理存储系统

职责分离、层层分装

+ 顶层 (`StateBackend`)：决定 Checkpoint 的持久化策略；
+ 中层 (`OperatorStateBackend`)：负责 算子状态（Operator State） 的具体实现；
+ 底层 (`KeyedStateBackend`)：负责 键控状态（Keyed State） 的具体实现。

理解：

+ `StateBackend` 是“总开关”，它决定了整个状态系统的“根基”；
+ `XXStateBackend` 是“执行者”，负责具体的存储逻辑。

## 顶层：StateBackend —— Checkpoint 策略的总入口
这是最上层的抽象，它不直接处理数据

+ 作为工厂和策略中心
+ **根据配置，**创建下层的具体实现。



选择哪种 StateBackend，从根本上决定了你的状态数据如何被持久化和恢复。

+ 不同的状态后端主要影响的是 Keyed State 的存储方式

| 状态后端 | 状态类型 | 运行时 (Runtime) 存储位置 | Checkpoint 时存储位置 |
| --- | --- | --- | --- |
| Memory   StateBackend | Keyed State | TaskManager JVM 堆内存 (Heap) | JobManager 内存（通过 RPC 直接发送） |
| | Operator State | TaskManager JVM 堆内存 (Heap) | JobManager 内存 |
| Fs   StateBackend | Keyed State | TaskManager JVM 堆内存 (Heap) | 分布式文件系统 |
| | Operator State | TaskManager JVM 堆内存 (Heap) | 分布式文件系统 |
| RocksDB   StateBackend | <font style="background-color:#E8F7CF;">Keyed State</font> | <font style="background-color:#E8F7CF;">TaskManager 本地磁盘   </font><font style="background-color:#E8F7CF;">（RocksDB 实例）</font> | 分布式文件系统（支持增量 Checkpoint） |
| | Operator State | TaskManager JVM 堆内存 (Heap) | 分布式文件系统 |


注：从 Flink 1.13+ 起，可通过 `RocksDBStateBackend.enableFullyManagedState()` 将 Operator State 也存入 RocksDB，但默认仍为堆内存。

说明

+ 运行时 vs Checkpoint
    - 运行时：指作业正常处理数据时，状态实际存放的位置，直接影响 内存消耗、GC 压力、访问延迟。
    - Checkpoint 时：指做快照（Snapshot）时，状态被持久化到哪里，用于 容错恢复。
+ RocksDBStateBackend 的特殊性
    - 唯一将 Keyed State 从堆内存卸载到磁盘的后端，因此能支持 TB 级状态。
    - Operator State 默认仍在堆内存，所以即使使用 RocksDB，也要为 Operator State 预留足够堆内存。
    - Checkpoint 支持增量模式：只上传自上次 Checkpoint 以来变更的 SST 文件，极大提升效率。
+ MemoryStateBackend 的局限性
    - 所有状态都在内存中，极易 OOM；
    - Checkpoint 存在 JobManager 内存，JobManager 成为单点瓶颈；
    - 仅适用于本地测试或极小状态场景。
+ FsStateBackend 的定位
    - 是 Memory 和 RocksDB 之间的折中方案；
    - 状态仍在堆内存，无法解决大状态问题；
    - 适合 中小状态 + 需要外部持久化 的场景。

## 中层：Operator State Backend —— 全局状态的管理者
这一层由 `DefaultOperatorStateBackend` 实现，专门**负责管理算子状态** (Operator State)



它是一个 代理层，会根据顶层 `StateBackend` 的选择，决定 Operator State 的**实际存放位置**：

+ 若顶层是 MemoryStateBackend → Operator State 存在 TaskManager 堆内存；
+ 若顶层是 FsStateBackend → Operator State 存在 TaskManager 堆内存 + Checkpoint 到文件；
+ 若顶层是 RocksDBStateBackend → 同 FsStateBackend

关键点：

+ `DefaultOperatorStateBackend` 是统一接口，屏蔽了底层存储细节
+ 不同的状态后端，**存储算子状态的数据结构和位置都是一样的，都在内存中**

## 底层：Keyed State Backend —— 键控状态的存储引擎
这是 Flink 状态管理的核心，负责管理 键控状态 (Keyed State)

不同于 Operator State，不同的状态后端用于存储 Keyed State 的数据结构有所不同，有两种实现

+ HeapKeyedStateBackend 实现：状态后端 MemoryStateBackend，FsStateBackend
+ RocksDBKeyedStateBackend 实现：状态后端 RocksDBStateBackend

### HeapKeyedStateBackend
+ 存储位置：JVM 堆内存；
+ **数据结构：内部使用 Java 的 HashMap 等集合类**；
+ 特点：
    - 访问速度极快；
    - 状态大小受限于 TaskManager 堆内存；
    - 不支持 TTL；
    - 不适合大状态。
+ 适用场景：小规模状态、开发测试
+ 适用的状态后端：MemoryStateBackend，FsStateBackend

### RocksDBKeyedStateBackend
+ 存储位置：本地磁盘（RocksDB）；
+ 数据结构：基于 LSM-Tree 的键值存储；
+ 特点：
    - 支持 TB 级状态；
    - 内存占用低（热数据缓存，冷数据在磁盘）；
    - 支持增量 Checkpoint 和 TTL；
    - 是生产环境的绝对首选。
+ 适用场景：所有需要大状态或高可靠性的生产作业。
+ 适用的状态后端：RocksDBStateBackend

## 工作流程示例（以 RocksDBStateBackend 为例）
1. 用户配置 `env.setStateBackend(new RocksDBStateBackend("hdfs://..."))`
2. Flink 创建 `RocksDBStateBackend` 实例；
3. 当用户注册 `ValueState` 时：
    - `RocksDBStateBackend` 会创建一个 `RocksDBKeyedStateBackend` 实例；
    - `RocksDBKeyedStateBackend` 在本地启动一个 `RocksDB` 实例；
    - 所有 Keyed State 操作`(get, put)`都通过 RocksDB 接口完成；
4. Checkpoint 时：
    - `RocksDBKeyedStateBackend` 触发 RocksDB 的 Snapshot；
    - Snapshot 文件被上传到 HDFS/S3；
    - 文件路径发送给 JobManager
