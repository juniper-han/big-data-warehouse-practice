# Catalog 简介
在 Flink SQL 中，Catalog（元数据目录） 

+ 是管理和组织数据库、表、函数、类型（Type）等元数据的**核心抽象**。
+ 是 Flink 中的元数据管理中心，负责管理表、视图、函数、数据库等**元数据的生命周期**。
+ 它类似于传统数据库中的"数据字典"，是 Flink Table API 和 SQL 的基础设施。

# Catalog 的作用
1. **元数据容器：存储 Database、Table、View、Function 等对象的定义**
2. 元数据发现与解析：当执行 SQL 查询时，Catalog 负责解析表名和函数名
3. 权限与命名空间管理：提供 `catalog.database.table` 三级命名空间
4. **连接外部系统：通过 Catalog 集成 Hive、Kafka、JDBC、Paimon 等系统**
5. 持久化元数据：支持将临时表提升为持久化表（需持久化 Catalog）
6. 统一访问接口：屏蔽底层存储差异，提供标准 SQL 访问



在 Flink 的 Table/SQL API 中

+ 当用户创建一个表时，可以指定将表的元数据存储到某个 Catalog 中
+ 在查询时，可以从 Catalog 中获取表的元数据，包括表的 schema、分区信息、数据位置等。
+ Catalog 还支持数据库的概念，允许用户在不同的数据库中创建相同名称的表，从而实现了元数据的隔离。



总结：Catalog 是 Flink SQL 的“元数据中枢”，让 Flink 能像操作本地表一样操作 Kafka、Hive、MySQL 等外部系统。

# Catalog 接口（源码）
Flink 定义了 `org.apache.flink.table.catalog.Catalog `接口，关键方法包括：

```java
public interface Catalog {
  // 数据库操作
  List<String> listDatabases();
  void createDatabase(String name, ...);
  
  // 表操作
  List<String> listTables(String databaseName);
  CatalogBaseTable getTable(ObjectPath tablePath);
  void createTable(ObjectPath tablePath, CatalogBaseTable table, ...);
  
  // 函数操作
  void createFunction(UnresolvedIdentifier identifier, CatalogFunction function);
}
```



Flink 中的 Catalog 是一个元数据存储和管理的**<font style="background-color:#FBDE28;">接口</font>**

+ 定义了元数据的操作方式，主要接口是`Catalog`，并提供了常用的实现类，位于`flink-table`模块下的`org.apache.flink.table.catalog`包中
+ 它允许用户存储和管理数据库、表、分区、视图、函数等元数据。
+ 常用方法
    - 打开/关闭 Catalog；数据库/表/函数的操作（创建、删除、统计），分区操作，统计信息
+ 常用实现类
    - `HiveCatalog`：用于连接 Hive Metastore
        * 类型：`hive`
        * 特点
            + 元数据存储在 Hive Metastore；
            + 支持与 Hive 生态无缝互操作；
            + 表数据可存于 HDFS、S3、OSS 等。
        * 适用场景：湖仓一体、批流统一
    - `GenericInMemoryCatalog`：基于内存的 Catalog
        * 类型：`generic_in_memory`
        * 特点：元数据仅存在于 JobManager 内存中，作业重启后丢失；
        * 适用于临时表、测试场景。
        * 默认行为：`CREATE TABLE t (...)` → 存入 `default_catalog`
    - `JdbcCatalog`：连接关系型数据库获取元数据
        * 类型：`jdbc`
        * 特点：
            + 将关系型数据库（MySQL、PostgreSQL 等）的 schema 映射为 Database；
            + 表结构自动同步；
            + 只读（不能通过 Flink DDL 修改源库表结构）。
        * 示例：`jdbc_catalog.mysql_db.user_table`

# 理解 Catalog
Catalog 这个概念是计算引擎侧的抽象（Flink / Spark / Hive）

+ 备注：Hudi 本身并不“定义 Catalog”，而是依赖计算引擎或 Metastore 来管理元数据。

## Flink 中的 Catalog
在 Flink 中：

**Catalog = 表和数据库的命名空间 + 元数据访问入口**

+ Flink SQL 所有 `CREATE TABLE / SELECT`
+ 都必须通过 Catalog 去找表的定义

Flink 的 Catalog 决定了：

+ 表的 schema
+ 表的 location
+ 表的类型（Hive / Hudi / 普通表）

## Hive Metastore
Hive Metastore（不是 Catalog，但常被当成）

Hive Metastore 是：一个元数据服务

它负责存：database、table、partition、location

但它本身不关心：

+ 计算引擎是谁
+ 表是 Hudi / Iceberg / Parquet
