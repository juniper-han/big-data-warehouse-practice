# FlinkSQL 参数配置简介
## FlinkSQL 配置参数分类
FlinkSQL 任务的配置参数分为以下几个方面

1. 环境参数（Environment Configuration）
2. 运行时参数：优化 Flink SQL 任务在执行时的任务性能
3. 优化器参数：Flink SQL 任务在生成执行计划时，经过优化器优化生成更优的执行计划

详细内容参考：[Configuration](https://nightlies.apache.org/flink/flink-docs-release-1.13/docs/dev/table/config/)

## 参数配置方式
FlinkSQL 相关参数

通常在`SET`语句、`sql-client-defaults.yaml`、StreamPark 的“Flink 配置”中配置 

```java
// instantiate table environment
TableEnvironment tEnv = ...

// access flink configuration
Configuration configuration = tEnv.getConfig().getConfiguration();
// set low-level key-value options
configuration.setString("table.exec.mini-batch.enabled", "true");
configuration.setString("table.exec.mini-batch.allow-latency", "5 s");
configuration.setString("table.exec.mini-batch.size", "5000");
```

# 参数配置
## 环境参数
环境参数（Environment Configuration）：用于初始化 `TableEnvironment` 或作业全局行为

### Checkpoint / Savepoint 参数配置
功能：配置状态快照和故障恢复  
目的：保证作业的 Exactly-Once 语义和故障恢复能力

1. `SET 'execution.checkpointing.interval' = '60s';`
    - 作用：Checkpoint 触发间隔
    - 默认值：未设置（即不开启 Checkpoint）
    - 生产建议：必须显式设置，否则无容错能力
        * 实时场景：`1min ~ 5min`；间隔越短，恢复越快，但性能开销越大，从 3 分钟开始调整
        * 准实时/批流一体：可设为 `10min+`；
2. `SET 'execution.checkpointing.timeout' = '20min';`
    - 作用：单次 Checkpoint 允许的最大完成时间，超时则失败
    - 默认值：`10min`
    - 生产建议：如果频繁超时，优先调大此项而非调大`interval`
        * 通常保持默认或设为 `10~30min`；
        * 若状态极大（TB 级），可适当调高；
        * 不能小于 `interval`，一般是`interval`的 1-2 倍
3. `SET 'execution.checkpointing.tolerable-failed-checkpoints' = '3';`
    - 作用：允许连续失败的 Checkpoint 次数，超过则作业失败
    - 默认值：`0`，即一次 Checkpoint 不成功，Flink 作业就失败
    - 生产建议
        * 设为`3 ~ 5`，容忍网络抖动、HDFS 瞬时不可用等；
        * 避免因偶发故障导致作业中断
4. `SET 'execution.checkpointing.externalized-checkpoint-retention' = 'RETAIN_ON_CANCELLATION';`
    - 作用：是否保留外部化 Checkpoint（用于手动恢复）
    - 可选值
        * `NO_EXTERNALIZED_CHECKPOINTS`（默认）：作业取消后自动删除；
        * `RETAIN_ON_CANCELLATION`：作业取消后保留（推荐）
    - 生产建议：必须设为 `RETAIN_ON_CANCELLATION`，便于：
        * 调试状态问题，作业升级回滚；
        * 手动从指定状态恢复（`-s <path>`）；
5. `SET 'execution.checkpointing.alignment-timeout' = '2min';`
    - 作用：Barrier 对齐超时时间（仅适用于 Exactly-Once 语义）
    - 默认值：`0`（即无限等待）
    - 生产建议：设为 `'1min'` 或 `'2min'`，避免因某条流停滞导致整个 Checkpoint 卡住
        * 超时后自动降级为 At-Least-Once（Flink 1.11+ 支持）；
        * 强烈建议设置，提升作业鲁棒性
6. `SET 'state.savepoints.dir' = 'hdfs://namenode:9000/flink/savepoints';`
    - 作用：Savepoint 存储路径（用于手动触发快照）
    - 生产建议：同样必须配置，与 Checkpoint 路径分开

### 状态后端配置
配置状态存储方式和位置，优化状态访问性能，支持大状态存储

1. `SET 'state.backend' = 'rocksdb';`
    - 作用：指定 Flink 作业使用的 状态后端（State Backend） 类型
    - `HashMap`：状态全在 JVM 堆内存中，仅用于本地测试，生产禁用（OOM 风险极高）。
    - `RocksDB`：**生产推荐**，嵌入式 LSM-Tree 数据库，状态存于 TaskManager 本地磁盘（非 HDFS）；JVM 堆外，支持超大状态（TB 级），性能略低于内存，但可扩展性强；
2. `SET 'state.checkpoints.dir' = 'hdfs://mycluster/flink/checkpoints/prod-job-001';`
    - 作用：指定 Checkpoint 元数据和状态快照的持久化存储路径
    - 生产建议：
        * 路径必须是 所有 JobManager/TaskManager 可访问的共享存储
        * 必须配置为高可用分布式存储（HDFS/S3/OSS）
3. `SET 'state.backend.incremental' = 'true';`
    - 作用：启用 RocksDB 增量 Checkpoint
    - 全量 Checkpoint（默认）：每次 Checkpoint 都上传完整的 RocksDB 状态快照 → 网络/IO 开销大。
    - 增量 Checkpoint（开启后）：只上传自上次 Checkpoint 以来变更的 SST 文件 → 大幅减少 Checkpoint 时间和存储占用
        * 增量 Checkpoint 依赖 RocksDB 的 SST 文件版本机制；
        * 恢复时需从最新完整 Checkpoint + 所有增量片段重建，因此**不能删除历史 Checkpoint**；
        * Flink 1.15+ 对增量 Checkpoint 稳定性大幅提升。
    - 强烈建议开启：状态较大（>10GB）、Checkpoint 频繁（如 30s-1min）、使用云存储（按量计费）
4. `SET 'state.backend.rocksdb.ttl.compaction.filter.enabled' = 'true';`
    - 作用：启用 RocksDB 的 TTL 压缩过滤器（Compaction Filter），实现**状态过期自动清理**。
    - 当在 Flink 中为状态设置 TTL，同时开启此参数，RocksDB 在 后台 Compaction（合并 SST 文件）过程中，会自动丢弃已过期的状态条目；这是最高效的状态清理方式，避免状态无限增长
    - 清理时机：在 RocksDB Compaction 时触发，需配合`StateTtlConfig`使用

### 重启策略配置
功能：配置作业失败后的重启策略，用于在作业失败时自动重试  
目的：提高作业的健壮性和可用性

以下三个参数共同配置了 Flink 作业的 指数退避重启策略（Exponential Delay Restart Strategy），用于在作业失败时自动重试，并通过 逐渐增加等待时间 的方式避免“雪崩式重试”对系统造成冲击。

1. `SET 'restart-strategy' = 'exponential-delay';`
    - 作用：指定 Flink 作业的全局故障重启策略
    - 可选值，Flink 支持的重启策略
        * `fixed-delay`：固定延迟，每次失败后等待固定时间重试
        * `exponential-delay`：指数延迟，**等待时间随失败次数指数增长（推荐）**
        * `failure-rate`：失败率，在时间窗口内允许一定失败次数
        * `none`：失败即终止（测试用）
    - 生产推荐：`exponential-delay`，原因
        * 初次失败可能是瞬时故障（如网络抖动），快速重试有效
        * 若持续失败，说明是系统性问题（如依赖服务宕机），应拉长重试间隔，避免无意义消耗资源或加剧下游压力
2. `SET 'restart-strategy.exponential-delay.initial-backoff' = '10s';`
    - 作用：设置首次失败后的初始等待时间（Initial Backoff）
        * 第 1 次失败 → 等待 10 秒 后重启；
        * 第 2 次失败 → 等待 10s * multiplier（默认乘数为 2）→ 20 秒；
        * 第 3 次失败 → 40 秒；
        * 注意：实际等待时间 =` initial-backoff × (multiplier)^(attempt-1)`
    - 生产建议：
        * 瞬时故障多（如 Kafka 临时不可用）：设为 `5s~30s`；
        * 依赖外部系统恢复慢（如数据库维护）：可设为 `1min`；
3. `SET 'restart-strategy.exponential-delay.max-backoff' = '5min';`
    - 作用：设置最大等待时间上限（Max Backoff），防止等待时间无限增长
    - 即使按指数增长，单次等待不会超过 `5min`；
    - 生产建议：根据 业务容忍度 和 依赖系统 MTTR（平均恢复时间） 设置；
        * 常见值：`5min`、`10min`、`30min`；
        * 不要设为过小（如 `1min`），否则失去“退避”意义；
        * 不要设为过大（如 `1h`），可能导致故障长时间未恢复。

### TaskManger 内存 / 网络管理配置
配置 TaskManager 内存/网络分配和使用，优化内存使用，防止 OOM 错误

1. `SET 'taskmanager.memory.managed.size' = '2gb';`
    - 作用：显式指定 TaskManager 中用于托管内存（Managed Memory） 的绝对大小（如 `'512mb'`, `'2gb'`）
    - 托管内存：堆外内存，专供排序、Hash Join/ Sort Merge Join、窗口聚合的中间缓存、RocksDB 状态后端的 Block Cache 等操作使用，此内存由 Flink 统一管理，避免直接使用 JVM 堆内存导致 GC 压力
    - 生产建议：不推荐使用此参数，推荐使用`fraction`（更弹性，适配不同规格机器）
        * `managed.size` 和 `managed.fraction` 互斥，只能设置其中一个；
        * 仅在需要精确控制时使用 `size`（如容器化环境内存固定）；
2. `SET 'taskmanager.memory.managed.fraction' = '0.5';`
    - 作用：指定 托管内存占 TaskManager 总内存（Total Flink Memory）的比例。
    - 默认值：`0.4`（即 40%）
    - 此参数很重要：若作业包含大量 `JOIN`、`GROUP BY`、`OVER WINDOW`、`RANK() `等操作，托管内存不足会导致频繁 spill 到磁盘，性能急剧下降
    - 生产建议
        * 简单 ETL（无复杂算子）：`0.2 ~ 0.3`
        * 含多表 `JOIN` / 聚合 / 排序：`0.5 ~ 0.6`
        * 大规模窗口计算：`0.6+`
3. `SET 'taskmanager.network.memory.buffers-per-channel' = '4';`
    - 作用：设置 每个网络输出通道（Result Partition Channel）预分配的网络缓冲区（Buffer）数量
    - 数据从上游算子发送到下游时，会写入本地 Buffer，再通过网络传输；每个 (上游 task → 下游 task) 连接 = 1 个 channel；此参数控制每个 channel 初始分配多少 Buffer（每个 Buffer 默认 32KB）
    - 默认值：`2`
    - 生产建议：通常无需调整，除非观察到 `Network: Output pool usage` 指标持续 > 80%。
        * 数据倾斜或高吞吐场景：可适当调高（如`4~8`），减少反压；
        * 但不要过高，否则浪费内存
4. `SET 'taskmanager.network.memory.floating-buffers-per-gate' = '16';`
    - 作用：为每个输入 Gate（Input Gate） 额外分配的浮动缓冲区（Floating Buffers）数量，用于应对突发流量或数据倾斜
        * Input Gate：下游 task 接收数据的入口；
        * Floating Buffers：全局共享的“应急”缓冲区，当某个 channel 的固定 buffer 不足时，可临时借用；
    - 此参数的必要性：在数据倾斜场景下，某些 channel 流量远高于平均，固定 buffer 很快耗尽，导致反压。浮动 buffer 可缓解此问题。
    - 默认值：`8`
    - 生产建议：
        * 高吞吐或存在倾斜：可调至 `16~32`；
        * 低延迟敏感场景：可略调低（如 `4`），减少内存占用；
        * 与 `buffers-per-channel` 配合使用



监控指标

| 指标 | 说明 | 健康阈值 |
| --- | --- | --- |
| `network.availableMemorySegments` | 可用网络 buffer 数 | > 0 |
| `network.totalMemorySegments` | 总网络 buffer 数 | — |
| `managedMemoryUsed` | 托管内存使用率 | < 90% |
| `numBuffersInUse` | Buffer 使用量 | 无持续堆积 |


注意：网络参数一般保持默认，仅在出现反压或网络内存不足时调整

## 运行时参数配置
### 微批处理配置
作用：控制 Flink SQL 中的 Mini-Batch（微批）优化机制，是提升流式作业吞吐、降低状态访问压力的关键手段，尤其适用于高吞吐聚合、维表 Join、去重等场景。

+ 在纯流模式下，Flink 默认 每条记录触发一次计算（如更新一次聚合状态），这会导致：
    - 高频访问状态后端（RocksDB）→ I/O 压力大；网络/算子开销高 → 吞吐受限；Checkpoint 负担重
+ Mini-Batch 通过“攒批”方式：
    - 缓存输入数据；达到 时间阈值（allow-latency）或 数量阈值（size）后；
    - 批量处理整批数据（如一次更新聚合状态）；输出结果
    - 本质：用可控的延迟换取吞吐和资源效率提升



参数

1. `SET 'table.exec.mini-batch.enabled' = 'true';`
    - 作用：启用 Mini-Batch 优化
    - 默认值：`false`，纯流处理
    - 开启后，以下操作收益：`GROUP BY`聚合（包括 UDAF）、`OVER`窗口聚合、维表 Join（需配合 `lookup.cache`）、`ROW_NUMBER()`去重
    - 不适用于：纯事件时间窗口（它们有自己的触发机制），需要【毫秒级】实时性的场景
    - 生产建议：绝大多数流式聚合作业都应开启，除非对延迟极度敏感（<1s）。
2. `SET 'table.exec.mini-batch.allow-latency' = '5s';`
    - 作用：设置 Mini-Batch 的最大等待时间，即使未达到 `size`，也会在该时间后触发计算
    - 实际触发条件：满足任一即触发
        * 缓存记录数 ≥ `mini-batch.size`；或
        * 自上次触发后等待时间 ≥ `allow-latency`
    - 生产建议
        * 实时大屏/告警：`'1s' ~ '3s'`
        * 日志分析/指标聚合：`'5s' ~ '30s'`
        * 准实时数仓：`'30s' ~ '2min'`
    - 注意：不能设为 `0`，否则退化为纯流
3. `SET 'table.exec.mini-batch.size' = '5000';`
    - 作用：设置 Mini-Batch 的最大记录数阈值
    - 类型：整数（无单位）
    - 过小 → 失去批处理优势；过大 → 内存占用高，可能 OOM
    - 生产建议：基于数据吞吐判断
        * < 1k records/s：`1000 ~ 5000`
        * 1k ~ 10k records/s：`5000 ~ 20000`
        * > 10k records/s：`20000 ~ 50000`
    - 经验公式：`size ≈ 吞吐量 (records/s) × allow-latency (s) × 0.5~1.0`



注意事项

+ Mini-Batch 基于系统时间，仅支持 Processing Time，不支持 Event Time（窗口聚合仍用原生窗口）
+ 结果非严格 Exactly-Once：在故障恢复时，Mini-Batch 可能重发整批数据（但 Sink 需幂等）
+ 内存占用增加：缓存区会占用 TaskManager 堆内存，需监控 `usedHeap`
+ **不适用于低吞吐场景：若数据稀疏，实际延迟 ≈ **`**allow-latency**`**，可能劣于纯流**

### 并行度配置
1. `SET 'parallelism.default' = '4';`
    - 作用：设置 Flink 作业中所有算子的默认并行度，即未显式指定并行度时的全局默认值
    - 与 TaskManager Slot 的关系：
        * 总并行度 ≤ 所有 TaskManager 的总 Slot 数；
        * 若 `default=4`，但集群只有 2 个 Slot → 作业提交失败（除非开启 slot sharing）
    - 生产建议
        * 小规模测试：`1~4`
        * 中等吞吐生产作业：`32~128`（根据 Kafka 分区数、数据量调整）
        * 高吞吐作业：与 Source 分区数对齐（如 Kafka topic 有 64 分区 → 并行度 ≥64）
        * Source 并行度 ≈ 输入分区数（避免数据倾斜或资源浪费）；
        * Sink 并行度 ≤ 目标系统写入能力（如 MySQL 主库写入瓶颈）；
        * 全局 `default` 可作为“兜底”，关键算子应显式设置
2. `SET 'pipeline.max-parallelism' = '128';`
    - 作用：设置作业的最大并行度，不影响运行时并行度，但限制未来可扩容的上限
    - 核心思想：Flink 的 Keyed State（键控状态） 在内部按 Key Group 分区存储，而
        * `Key Group 数量 = max-parallelism`
        * 每个并行子任务（Subtask）负责连续的一段 Key Group；当并行度从 4 扩容到 8 时，Flink 会重新分配 Key Group，但 Key Group 总数（= max-parallelism）不能变
    - 举例：`max-parallelism = 128`
        * 并行度 = 4 → 每个 Subtask 负责 32 个 Key Group（128 / 4）
        * 扩容到 8 → 每个 Subtask 负责 16 个 Key Group（128 / 8）
        * 但无法扩容到 256，因为 Key Group 只有 128 个
    - 该参数的重要性
        * Savepoint/Checkpoint 恢复依赖 `max-parallelism`；
        * `max-parallelism` 一旦设定，不能更改（否则状态无法恢复）
    - 默认值：`min(128, nextPowerOfTwo(parallelism))`，可能不够
    - 生产建议：必须 ≥ 当前并行度
        * 预留足够扩容空间：通常设为 当前并行度的 2~10 倍，取 2 的幂更高效
        * 避免过大：过大会增加状态元数据开销（默认上限 32768）



注意：

+ 单个并行度处理**数据量不应低于 100 条/秒**，避免产生大量小文件
+ `max-parallelism` 是 面向未来的架构决策，应在作业首次上线前慎重设定；

### State TTL 状态过期
`table.exec.state.ttl`：全局配置参数，用于统一设置所有状态的生存时间（Time-To-Live, TTL），防止状态无限增长、避免内存/磁盘爆炸

+ 作用：自动清理过期的状态数据，无需修改业务逻辑代码
+ **有状态流作业不上 TTL，迟早出事：务必合理配置该参数**
+ 适用于以下场景中的状态：`GROUP BY` 聚合、`OVER` 窗口、维表 `Join` 的缓存（若启用 lookup cache）、去重（`ROW_NUMBER() + WHERE row_num = 1`）、`ProcessFunction` 中的 Keyed State（通过 Table API 生成的）



参数：`SET 'table.exec.state.ttl' = 'duration';`

+ 取值：`duration`，如`'24 h'`、`'86400000 ms'`、`'1 d'`
    - 必须包含空格和单位（如 `'24 h'`，不能写 `'24h'`）；
    - 单位支持：`ms, s, m, h, d`（毫秒、秒、分、小时、天）
+ 工作原理：
    - TTL 计时起点：状态最后一次被**访问或更新**的时间（非创建时间）
    - 清理时机（取决于状态后端）
        * RockDB：在 Compaction 过程中自动丢弃过期数据（需开启 `ttl.compaction.filter.enabled`）
        * Heap（HashMap）：在状态访问时惰性检查，读/写时发现过期则清理
+ 生产建议
    - 用户行为实时聚合（如 PV/UV）：`1~7 d`
    - 设备指标监控：`24~72 h`
    - 会话（Session）分析：`30 m ~ 2 h`
    - 维表缓存（若未单独设` lookup.cache.ttl`）：`10 m ~ 1 h`
+ 不适用场景
    - 事件时间窗口（Event Time Window）：窗口状态由 watermark 自动清理，无需额外 TTL；
    - 有明确生命周期的状态：如订单状态只保留 30 天，应通过业务逻辑控制。
+ 注意事项
    - 不要设为 0 或负数 → 等效于禁用 TTL；
    - 避免过短 TTL（如 < 1m）→ 可能导致状态频繁重建，性能下降；
+ 该参数一些说明
    - 仅对 Keyed State 有效：非 keyed 算子（如`GLOBAL GROUP BY`）的状态不受此参数影响
    - 不保证精确清理时间：TTL 是“至少保留这么久”，实际清理可能延迟（取决于访问频率或 Compaction 触发）
    - 与 Checkpoint 无关：TTL 是状态生命周期管理，Checkpoint 是容错机制，二者正交



多级 TTL 体系：优先级从高到低

+ 算子级 TTL（Java/Scala API）：`StateTtlConfig.newBuilder(Time.days(1)).build();`
+ 表 DDL 中指定（部分 Connector 支持，如维表缓存）

```plsql
CREATE TABLE dim_table (...)
WITH (
  'lookup.cache.ttl' = '10min'
);
```

+ 全局 SQL 配置 ←`table.exec.state.ttl`（本参数），兜底策略

### 其他
1. `SET 'table.exec.source.idle-timeout' = 'duration';`
    - 作用：当 Source 在指定时间内未收到任何新数据时，Flink 会将其标记为“空闲（idle）”，并停止为其生成 Watermark
    - 解决了流处理中的经典问题：如果某个分区/Source 永远不再产生数据，它的 Watermark 会停滞，导致下游全局 Watermark 无法前进，进而使窗口无法触发、状态无法清理
    - 默认值：`0 ms`，禁用空闲检测
    - 取值：必须包含空格和单位，如`'5 min'`，不能写`'5min'`
    - 生产调优建议：
        * 单 Source 且持续有数据：不需要
        * Kafka 多 Partition、文件流、CDC 流：需要
    - 如何选择 timeout 值：应大于 业务最大静默间隔，但小于窗口大小或状态 TTL
        * 示例：业务数据 10 min 一批，偶尔故障延迟 30 min，滚动窗口 1h，参数值设置为 45 min
    - 注意：
        * 仅影响 Event Time
        * 一旦恢复数据，自动推出空闲状态
        * 与 Checkpoint 无关：空闲 Source 仍会参与 Checkpoint（保存 offset 等元数据）；但其状态可能因 Watermark 不推进而无法清理
2. `SET 'table.exec.async-lookup.buffer-capacity' = '500';`
    - 作用：设置每个 Async Lookup 算子的请求缓冲区最大容量（即最多缓存多少个未完成的查询请求）
    - 原理：Async Lookup Join：当使用外部维表进行关联查询时，为避免同步查询阻塞流处理，通常采用异步 I/O（Async I/O） 模式，Flink 会将待查询的 key 缓存在内部缓冲区（Async Lookup Buffer） 中，批量/异步发送给外部系统，查询结果返回后，与主数据流进行关联输出
    - 默认值：`100`
    - 工作机制：
        * 主数据流每来一条记录，生成一个 lookup 请求，请求被放入异步缓冲区（Async Lookup Buffer）
        * 当满足以下任一条件时，触发实际查询：
            + 缓冲区满（达到 `buffer-capacity`）
            + 或定时 flush（由底层 Async I/O 驱动，非此参数控制）
        * 外部系统返回结果后，从缓冲区移除对应请求
        * 如果缓冲区已满，新请求将被阻塞（Backpressured），直到有空间释放
    - 参数影响分析：
        * 参数值过小（10）：吞吐低（频繁小批量查询），延迟低，内存低，背压风险高（易阻塞）
        * 参数值适中（100 - 1k）：吞吐高，延迟可控，内存合理，背压风险平衡
        * 参数值过大（1w+）：吞吐极高，延迟高，内存高（OOM 风险），背压风险低
    - 选择合适值，参考公式：`buffer-capacity ≈ 目标 QPS × 平均查询延迟（秒）`
        * 示例：维表查询 P99 延迟 = 0.05 s，主流吞吐 = 1w QPS，则参数设置为 500
    - 典型场景推荐值
        * 低吞吐（< 1k/s），低延迟要求：50 - 100
        * 中等吞吐（1k - 10k/s）：200 - 1000
        * 高吞吐（> 10k/s），维表响应快（< 20ms）：1000 - 5000
    - 维表必须配合设置，DDL 中启用异步
        * 必须开启：`'lookup.async' = 'true',`
        * 异步模式下通常禁用本地缓存：`'lookup.cache.max-rows' = '0'`
    - 注意事项：
        * 该参数控制的是 未完成请求的缓冲上限，不是每次发多少条；实际批量大小由 Async I/O 驱动（如 JDBC connector 的 batch size）决定。
        * 内存安全：每个请求对象会占用堆内存；需监控 TaskManager Heap 使用率，防止 OOM
        * 背压：若维表响应慢 + buffer-capacity 小 → 上游快速反压；若 buffer-capacity 过大 → 掩盖背压问题，但可能堆积大量请求，导致 GC 压力或超时。
        * 不适用于同步 Lookup：若 `'lookup.async' = 'false'`，此参数完全无效。
3. `SET 'table.exec.sink.not-null-enforcer' = 'ERROR';`
    - 作用：当写入 Sink 的记录中包含`NULL`值，但目标表字段定义为`NOT NULL`时，Flink 应该如何处理
        * 严格模式（`ERROR`）：默认值，遇到 `NULL` 直接抛出异常，作业失败；
            + 适用场景：数据质量要求高、需及时发现问题
            + 核心业务链路（支付、订单等），下游系统无法容忍 `NULL`（如主键、Kafka 消息 key）。
        * 宽松模式（`DROP`）：自动丢弃包含非法 `NULL` 的整条记录，作业继续运行
            + 适用场景：容忍部分脏数据、保障作业持续运行
    - 注意：
        * Flink 仅检查 DDL 中的`NOT NULL`，不检查数据库实际约束，若 DDL 未写`NOT NULL`，即使下游 DB 有约束，Flink 也不会拦截
        * 对 Upert Sink 影响：若主键（PK）为 `NULL`：
            + `ERROR` → 作业失败；
            + `DROP` → 丢弃该更新，可能导致状态不一致（如本应删除的记录未删除）
4. `SET 'table.exec.source.cdc-events-duplicate' = 'true';`
    - 作用：用于控制 CDC源表是否可能存在重复变更事件的优化参数，根据该参数的设置，Flink 会决定是否启用 “去重优化”（即避免为每个事件都更新状态）
        * `true`：默认值，CDC 事件可能存在重复，Flink 会为每个事件强制更新状态（安全但低效）
        * `false`：CDC 事件严格无重复（Exactly-Once 保证），Flink 可跳过冗余状态更新，启用优化
    - 在真实 CDC 场景中，以下情况会导致相同主键的变更事件重复出现：
        * Kafka Topic 重放：作业从较早 offset 重启 → 重放历史变更
        * Debezium 快照 + Binlog 重叠：初始快照包含某条记录，启动后 binlog 又发送了该记录的旧变更
        * Source Connector 重试机制：网络抖动导致消息重复投递
    - 如果设置为`false`，Flink 将进行如下优化
        * Flink 假设“每个主键的最新事件一定是最终一致的，无需担心历史重复事件覆盖新状态”
        * 跳过状态版本检查：直接覆盖状态，省去比较开销
        * 减少 RocksDB 写入：避免为重复事件写入中间状态
        * 简化 Changelog 处理逻辑：更激进地 retract 旧值、emit 新值
    - 生产实践：仅在经过充分验证的 CDC 链路上，才考虑开启此优化

# FlinkSQL 聚合算子优化
## 优化手段
针对聚合算子的优化，主要包含以下四种优化：

1. MiniBatch 聚合：常用
2. 两阶段聚合：常用
3. Split 分桶：不常用
4. 去重 Filter 子句：常用

## 两阶段优化
### 参数介绍
Flink 的聚合默认采用两阶段聚合（Two-Phase Aggregation） 以减少网络传输和状态压力：

+ Local Agg（本地预聚合）：每个 TaskManager 对本地数据先聚合；
+ Global Agg（全局聚合）：将 Local 结果按 Group Key Shuffle 到下游统一合并。



`table.optimizer.agg-phase-strategy`：查询优化器，控制聚合执行阶段划分策略

+ 作用：决定 Flink 如何将一个逻辑聚合拆分为多个物理执行阶段（通常是两阶段：Local + Global）
+ 参数可选值
    - `AUTO`：默认，优化器自动决定是否分阶段，通用推荐
    - `TWO_PHASE`：强制启用两阶段聚合，适用于高基数、大吞吐场景
    - `ONE_PHASE`：禁用分阶段，全量 Shuffle 后聚合，适用于低基数或调试

**生产建议：显式启用两阶段**`**SET 'table.optimizer.agg-phase-strategy' = 'TWO_PHASE';**`

可选值说明

+ `ONE_PHASE`（单阶段）
    - 执行计划：`Source → [Shuffle by GROUP KEY] → Global Agg → Sink`
    - 特点：
        * 所有原始记录都通过网络 Shuffle；
        * 全局聚合算子状态 = 所有唯一 Group Key；
    - 缺点：
        * 网络 I/O 极高（尤其当输入数据远大于聚合后数据时）；
        * 全局状态巨大 → Checkpoint 慢、RocksDB 压力大；
+ `TWO_PHASE`（两阶段）
    - 执行计划：`Source → Local Agg → [Shuffle by GROUP KEY] → Global Agg → Sink`
    - 特点：
        * Local Agg：在每个并行子任务内预聚合，大幅减少输出记录数；
        * Shuffle 数据量 = 预聚合后的中间结果；
        * 全局状态 = 所有唯一 Group Key（与 `ONE_PHASE` 相同），但网络和 CPU 开销显著降低；
    - 优势：
        * 吞吐提升 2~10 倍（实测常见）；
        * 减轻网络背压；
        * 更适合 Mini-Batch 优化。
+ `AUTO`（自动）
    - 优化器根据以下因素决策：
        * 聚合函数是否可分解（decomposable）；
        * 是否启用了 Mini-Batch；
        * 输入数据统计信息（若可用）；
    - 通常等价于 TWO_PHASE，除非检测到不支持分阶段。

### 两阶段优化适用场景
Flink 的两阶段聚合优化，能解决【由“高基数 + 均匀分布”】引起的普通数据倾斜场景。

它能解决的倾斜场景如下：

+ “伪倾斜”：由于数据在多个 Source 分区中分布不均，导致某些 Subtask 处理更多原始记录；
    - 例如：Kafka Partition 0 有 100 万条 `userA`，Partition 1 有 100 万条 `userB`；
    - Local Agg 后，每个 TM 输出`(userA, 100万)` 和`(userB, 100万)`，Shuffle 均匀。
    - 这种情况下，两阶段聚合天然消除了因分区不均导致的负载不均

它无法解决的倾斜场景：

+ 真正的热点 Key 倾斜（Hot Key Skew）：某个或少数几个 key 占据绝大部分流量；
    - 例如：`user_id = 'guest'` 占 60% 请求；或 `product_id = '热门商品'` 占 80% 订单。
    - 此时：
        * Local Agg 无法减少 key 的数量（所有 guest 仍在同一个 Global Agg Subtask）；
        * Shuffle 后，所有 guest 仍发往同一个下游 Task → 该 Task 成为瓶颈。
    - 结果：Global Agg 阶段依然严重倾斜，CPU/内存/网络打满，作业背压。
+ 此时的解决方案：手动加随机前缀（Salting）

### 手动加随机前缀
何时需要手动加随机前缀（Salting）

| 场景 | 是否需要 Salting | 原因 |
| --- | --- | --- |
| 多个 Key 均匀分布，但总量大 | ❌ 不需要 | 两阶段已足够 |
| 少数 Key 占比 > 20%~30% | ✅ 需要 | Global Agg 会成为单点瓶颈 |
| 聚合函数不支持分阶段（如旧版 `COUNT(DISTINCT)`） | ✅ 需要 | 两阶段未生效 |


如何判断是否需要手动打散

+ 监控指标

| 指标 | 倾斜表现 |
| --- | --- |
| Subtask 处理速率差异 | 最快 vs 最慢 > 5 倍 |
| Global Agg 算子 CPU | 某个 Subtask 持续 100%，其他空闲 |
| Network Output Queue | 某个 Subtask 的输出队列堆积 |


+ 快速诊断 SQL：若 Top 1 占比 > 30%，建议 Salting。

```plsql
-- 查看 Top N 热点 Key（需开启 mini-batch）
SELECT user_id, COUNT(*) AS cnt
FROM source_table
GROUP BY user_id
ORDER BY cnt DESC
LIMIT 10;
```

+ 仅当检测到热点 Key 时，才在 SQL 中加随机前缀，不要盲目加，会增加计算和状态复杂度
+ 策略：先开 `TWO_PHASE`，再根据监控决定是否 Salting

## Split 分桶
### 引入原因
```properties
table.optimizer.distinct-agg.split.enabled
table.optimizer.distinct-agg.split.bucket-num
```

以上两参数用于：优化 `COUNT(DISTINCT)` 等**去重聚合**性能的关键配置，尤其在高基数场景下能显著提升吞吐、降低状态压力。



为什么需要优化`COUNT(DISTINTCT)`

+ 标准 SQL：`SELECT category, COUNT(DISTINCT user_id) FROM events GROUP BY category;`
+ `COUNT(DISTINCT)`需要为每个`category`维护一个去重集合（如`HashSet`）存储所有`user_id`；
+ 后果
    - 状态大小 = 所有唯一` (category, user_id) `对 → 可能达 TB 级；
    - Checkpoint 慢、RocksDB 压力大；
    - GC 频繁（若用 Heap State Backend）



核心机制：Split Distinct Agg（拆分去重聚合）

Flink 引入 “两阶段 + 分桶” 策略，将一个大去重集合拆分为多个小集合，从而：

+ 减少单点状态压力；
+ 提升并行度利用率；
+ 支持更高效的合并

### 参数介绍
1. `SET 'table.optimizer.distinct-agg.split.enabled' = 'true';`
    - 作用：启用/禁用去重聚合的“分桶拆分”优化
    - 默认值：`false`，出于兼容性考虑，需显式开启
    - 工作原理：开启后，Flink 将原始聚合逻辑重写为 两阶段 + 分桶：
        * Local Stage：把数据随机分桶，在每个桶内进行去重聚合，状态远小于全局聚合
        * Global Stage：基于分桶聚合的结果，进行全局聚合
    - 注意：仅对 `COUNT(DISTINCT)` 和 自定义支持 `split` 的 UDAF 生效。
2. `SET 'table.optimizer.distinct-agg.split.bucket-num' = '2048';`
    - 作用：设置分桶数量（Bucket Number）
    - 默认值：若上个参数开启，但未设置此参数，默认未`1024`
    - 如何选择合适的值
        * 值越大：状态越分散，并行度利用越好，但元数据开销增加
        * 值越小：状态合并开销小，但可能仍存在倾斜
        * 经验公式：`bucket-num = max(1024, parallelism * 4)`
            + `parallelism`：下游 Global Agg 并行度
        * 若`key`基数极高，可适当调大该值

### 注意事项
1. 必须配合 `TWO_PHASE` 聚合：确保 `table.optimizer.agg-phase-strategy` 未强制为 `ONE_PHASE`
2. 不适用于所有 `DISTINCT` 场景
    - 仅优化 单列 `COUNT(DISTINCT col)`；
    - 多列去重（如 `COUNT(DISTINCT a, b)`）可能不生效（Flink 1.17 有限支持）；
    - 其他 `DISTINCT` 聚合（如 `SUM(DISTINCT)`）不支持
3. 内存 vs 网络权衡
    - 分桶会增加 Shuffle 数据量（每个桶都要传输）；
    - 但在高基数下，状态和 CPU 收益远大于网络开销。

## 去重 Filter 子句
### 简介
在 Flink SQL 的聚合优化中，“去重 filter 子句”（Distinct with Filter Clause）用于在单次`GROUP BY`中高效计算多个带不同过滤条件的去重指标（如“男性 UV”、“女性 UV”等），避免重复扫描数据或多次聚合。

### 问题背景：传统写法的痛点
假设要统计：

+ 总 UV（去重用户数）
+ 男性 UV
+ 女性 UV

传统写法：低效，三个独立子查询（3次全表扫描 + 3次去重状态）

```plsql
SELECT 
  (SELECT COUNT(DISTINCT user_id) FROM events) AS total_uv,
  (SELECT COUNT(DISTINCT user_id) FROM events WHERE gender = 'M') AS male_uv,
  (SELECT COUNT(DISTINCT user_id) FROM events WHERE gender = 'F') AS female_uv;
```

存在问题

+ 多次遍历输入流；
+ 为每个指标维护独立的去重集合（状态冗余）；
+ 网络 Shuffle 重复

### 正确写法：`FILTER (WHERE ...) `+ 共享去重状态
优化写法：推荐

```plsql
SELECT 
  COUNT(DISTINCT user_id) AS total_uv,
  COUNT(DISTINCT user_id) FILTER (WHERE gender = 'M') AS male_uv,
  COUNT(DISTINCT user_id) FILTER (WHERE gender = 'F') AS female_uv
FROM events
GROUP BY window(...); -- 或其他分组
```

特性

+ 单次扫描：输入流只读一次；
+ 共享去重集合：Flink 为 `user_id` 维护一个全局去重集合；
+ 按条件计数：在 emit 阶段，根据 `FILTER` 条件从同一集合中计算不同子集大小。
+ 这不是简单的语法糖，而是 Flink 执行引擎的深度优化！

### Flink 执行计划的优化
1. 状态共享
    - 无论有多少个`COUNT(DISTINCT) FILTER (...)`，Flink 只为` (group_key, user_id) `维护一个去重集合； 避免存储多份相同 `user_id`
2. 延迟计算：在窗口触发或 Checkpoint 时：
    - 遍历去重集合；
    - 对每个 `user_id` 判断其是否满足各 `FILTER` 条件，分别累加计数
3. 与 Split Distinct Agg 协同
    - 若同时开启` table.optimizer.distinct-agg.split.enabled = true`
    - 去重集合会被分桶（bucketed），但所有 `FILTER` 共享同一套分桶结构，进一步提升并行度

### 使用限制
1. 仅支持 `COUNT(DISTINCT)`
    - `SUM(DISTINCT ...) FILTER (...)` ❌ 不支持；
    - 其他聚合函数（如 `AVG`）的 `FILTER` 子句不涉及去重优化
2. `FILTER` 条件必须是确定性的
    - 不能包含 `RAND()`、`PROCTIME()` 等非确定性函数
    - 否则无法保证结果一致性
