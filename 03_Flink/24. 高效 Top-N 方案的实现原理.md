# 背景
Top-N 是统计报表和大屏非常常见的功能，主要用来实时计算排行榜。

流式的 Top-N 不同于批处理的 Top-N，它的特点是：

+ 持续计算：持续的在内存中按照某个统计指标计算 Top-N 排行榜
+ 增量更新：当排行榜发生变化时，发出更新后的排行榜。

FlinkSQL 官方文档给出的实现 Top-N 的方案：通过`ROW_NUMBER()`

```sql
SELECT [column_list]
FROM (
   SELECT [column_list],
     ROW_NUMBER() OVER ([PARTITION BY col1[, col2...]]
       ORDER BY col1 [asc|desc][, col2 [asc|desc]...]) AS rownum
   FROM table_name)
WHERE rownum <= N [AND conditions]
```

# Top-N SQL 语义分析
## Global Top-N
示例：取某个搜索关键词下的搜索热度前 10 名的词条数据

```sql
-- 输入数据为搜索词条数据的搜索热度数据
-- 当搜索热度发生变化时，会将变化后的数据写入到数据源的 Kafka 中
CREATE TABLE source_table (
    name BIGINT NOT NULL,  -- 搜索热度名称
    search_cnt BIGINT NOT NULL,  -- 热搜消费热度（比如 3000）
    key BIGINT NOT NULL,  -- 搜索关键词
    row_time AS cast(CURRENT_TIMESTAMP as timestamp(3)),
    WATERMARK FOR row_time AS row_time
) WITH ();
   
CREATE TABLE sink_table (
    key BIGINT,  -- 搜索关键词
    name BIGINT,  -- 搜索热度名称
    search_cnt BIGINT,   --  热搜消费热度（比如 3000）
    `timestamp` TIMESTAMP(3)   -- 消费词条时间戳
) WITH ();

-- DML 逻辑
INSERT INTO sink_table
SELECT key, name, search_cnt, row_time as `timestamp`
FROM (
   SELECT key, name, search_cnt, row_time, 
  -- 根据热搜关键词 key 作为 partition key
  -- 然后按照 search_cnt 倒排取前 100 名
     ROW_NUMBER() OVER (PARTITION BY key ORDER BY search_cnt desc) AS rownum
   FROM source_table)
WHERE rownum <= 100
```

SQL 语义分析：上面的 SQL 会翻译成以下三个算子：

+ 数据源：
    - 数据源即最新的词条下面的搜索词的搜索热度数据
    - 消费到 Kafka 中数据后，按照`partition key`将数据进行 Hash 分发到下游排序算子，相同的`key`数据将会发送到一个并发中
+ 排序算子
    - 为每个`key`维护了一个 Top-N 的榜单数据
    - 接受到上游的一条数据后，如果 Top-N 榜单还没有到达 N 条，则将这条数据加入 Top-N 榜单后，直接下发数据
    - 如果到达 N 条之后，经过 Top-N 计算，发现这条数据比原有的数据排序靠前，那么新的 Top-N 排名就会有变化，将变化了的这部分数据之前下发的排名**<font style="background-color:#FBDE28;">数据撤回（即撤回流）</font>**，然后下发新的排名数据
+ 数据汇
    - 接收到上游的数据之后，然后输出到外部存储引擎中

## Window Top-N
Window Top-N 是一种特殊的 TopN，它的返回结果是每一个窗口内的 N 个最小值或者最大值。

+ Window Top-N 不会出现撤回数据
    - 因为 Window Top-N 是在窗口结束时，输出最终结果，不会产生中间结果
+ Window Top-N 在窗口结束时，会自动清除 State

```sql
SELECT [column_list]
FROM (
   SELECT [column_list],
     ROW_NUMBER() OVER (PARTITION BY window_start, window_end [, col_key1...]
       ORDER BY col1 [asc|desc][, col2 [asc|desc]...]) AS rownum
   FROM table_name)  -- windowing TVF
WHERE rownum <= N [AND conditions]

```



示例：取当前这一分钟的搜索关键词下的搜索热度前 10 名的词条数据

```sql
CREATE TABLE source_table (
    name BIGINT NOT NULL,  -- 搜索热度名称
    search_cnt BIGINT NOT NULL,   -- 热搜消费热度（比如 3000）
    key BIGINT NOT NULL,   -- 搜索关键词
    row_time AS cast(CURRENT_TIMESTAMP as timestamp(3)),  
    WATERMARK FOR row_time AS row_time
) WITH ();

CREATE TABLE sink_table (
    key BIGINT,   -- 搜索关键词
    name BIGINT,   -- 搜索热度名称
    search_cnt BIGINT,  -- 热搜消费热度（比如 3000）
    window_start TIMESTAMP(3),  -- 窗口开始时间戳
    window_end TIMESTAMP(3)   -- 窗口结束时间戳
) WITH ();

-- 处理 sql：
INSERT INTO sink_table
SELECT key, name, search_cnt, window_start, window_end
FROM (
   SELECT key, name, search_cnt, window_start, window_end, 
     ROW_NUMBER() OVER (PARTITION BY window_start, window_end, key
       ORDER BY search_cnt desc) AS rownum
   FROM (
      SELECT window_start, window_end, key, name, max(search_cnt) as search_cnt
      -- window tvf 写法
      FROM TABLE(TUMBLE(TABLE source_table, DESCRIPTOR(row_time), INTERVAL '1' MINUTES))
      GROUP BY window_start, window_end, key, name
   )
)
WHERE rownum <= 100
```

SQL 语义

+ 数据源
    - 数据源即最新的词条下面的搜索词的搜索热度数据，消费到 Kafka 中数据后，将数据按照窗口聚合的 key 通过 Hash 分发策略发送到下游窗口聚合算子
+ 窗口聚合算子
    - 进行窗口聚合计算，随着时间的推进，将窗口聚合结果计算完成发往下游窗口排序算子
+ 窗口排序算子
    - 这个算子其实也是一个窗口算子，只不过这个窗口算子为每个 Key 维护了一个 Top-N 的榜单数据，接受到上游发送的窗口结果数据进行排序，随着时间的推进，窗口的结束，将排序的结果输出到下游数据汇算子。
+ 数据汇
    - 接收到上游的数据之后，然后输出到外部存储引擎中

# Flink 优化 Top-N 执行方案
基于 Flink 1.17.2 探究，`ROW_NUMBER()` 实现高效 Top-N 的原理：

+ Flink 并不会对所有数据进行全局排序，而是通过 “Ranking 算子 + 堆优化” 实现高效计算

## Flink 识别 Top-N：优化器规则
Flink 通过 `WindowStartEndPropertiesRule` 和 `RankingRewriteRule`（内部名称）识别 Top-N 模式



识别条件：`org.apache.flink.table.planner.plan.rules.logical.RankingRewriteRule`

将普通 Window 算子替换为更为高效的 Rank 算子



Flink 将以下结构重写为 Ranking 算子：必须满足（即高效 Top-N 的条件）

+ `ROW_NUMBER()` 是唯一窗口函数；
+ 过滤条件为 `row_num <= N`（N 为常量）；
+ 无其他复杂表达式。

```java
// 伪代码：匹配条件
if (query contains:
    - ROW_NUMBER() in SELECT
    - OVER(PARTITION BY ... ORDER BY ...)
    - WHERE row_number <= N
    - N is constant (e.g., 3, not column)
) {
    rewrite to LogicalRank;
}
```



执行计划优化：

+ 普通窗口：未优化

```latex
Calc
+- Window
   +- TableSourceScan
```

+ Top-N 优化后：Window 算子 → Rank 算子，这是高效执行的核心！

```latex
Rank(
  partitionBy=[category],
  orderBy=[sales DESC],
  rankType=ROW_NUMBER,
  rankRange=[1, 3]
)
+- TableSourceScan
```

## 核心算子：`Rank`
`Rank`算子：`org.apache.flink.table.runtime.operators.rank`

算子类型

+ `RetractableTopNFunction`：用于 `ROW_NUMBER()`（支持撤回）
+ `DeduplicationProcessFunction`：用于去重（非 Top-N）

### 核心数据结构：双堆
针对`RetractableTopNFunction`

为高效维护 Top-N，Flink 使用 两个优先队列（堆）：

+ Top-N 小顶堆
+ 候选大顶堆

| 堆 | 作用 | 排序规则 |
| --- | --- | --- |
| `topHeap` | 存储当前 Top-N 记录 | 按排序字段 **升序**（小顶堆） |
| `bottomHeap` | 存储 N 名之后的记录 | 按排序字段 **降序**（大顶堆） |


术语堆

+ 堆顶（Heap Top / Root）：树的根节点，有序（最大/最小）
+ 堆底（Heap Bottom）：所有叶子节点（最后一层），无顺序保证



小顶堆（Min-Heap）：

+ 堆顶（root） = 当前第 N 名（最小）
+ 堆底（leaf） ≠ 第 1 名，第 1 名可能在任意叶子节点

大顶堆（Max-Heap）：

+ 堆顶 = 当前缓存中最好的非 Top-N 记录，最优候补（第 N+1 名）
+ 堆底（叶子节点） = 若干较小值（无序）
+ `bottomHeap` 不是为了精确维护“第 N+1 到 N+K 名”，而是为了在内存受限时，尽可能保留高价值候选



工作机制

+ `topHeap` 的堆顶 = 当前第 N 名（最小值）；
+ 新记录若 > `topHeap`堆顶，则挤掉第 N 名，进入 Top-N；
+ 新纪录若 < `topHeap`堆顶，且 >`bottomHeap`堆顶，则挤掉`bottomHeap`堆顶
+ 新纪录若 < `bottomHeap`堆顶，如果此时
    - `bottomHeap`已满，则丢弃新纪录
    - `bottomHeap`未满，进入堆
+ 支持高效插入/删除/更新。

### 状态管理（Keyed State）
每个 (`partition_key`) 对应一个 `RankingState`：

```java
// 源码：RankingState.java
public class RankingState {
  // 双堆存储 (sortKey, record) 映射
  private Map<SortKey, RowData> topHeapMap;
  private Map<SortKey, RowData> bottomHeapMap;
  
  // 堆本身只存 SortKey（节省内存）
  private PriorityQueue<SortKey> topHeap;
  private PriorityQueue<SortKey> bottomHeap;
}
```

内存优化：

+ 堆中只存 排序键（`SortKey`），实际数据存在 `Map` 中；
+ `SortKey` = 排序字段的序列化值

### 处理流程（`processingElement`）
当一条新纪录到达

```java
// 伪代码：RetractableTopNFunction.processElement()
void processElement(RowData input) {
  SortKey sortKey = extractSortKey(input);
  
  if (isInTopN(sortKey)) {
    // 情况1: 新记录能进 Top-N
    if (topHeap.size() < N) {
      // 直接加入
      addToTopHeap(input, sortKey);
    } else {
      // 挤掉原第N名
      SortKey evicted = topHeap.poll();
      emitRetract(evicted);          // 撤回旧记录
      addToBottomHeap(evicted);
      
      addToTopHeap(input, sortKey);
      emitInsert(input);             // 输出新记录
    }
  } else {
    // 情况2: 不能进 Top-N，放入 bottomHeap
    addToBottomHeap(input, sortKey);
  }
  
  // 更新状态
  state.update(rankingState);
}
```

关键点：

+ 只输出 Top-N 内的记录；
+ 支持撤回（Retract）：当排名变动时，先撤回旧记录，再插入新记录；
+ 状态仅存 Top-N + 部分候选，而非全量数据

## 性能
| 方案 | 时间复杂度 | 空间复杂度 | 是否支持更新 |
| --- | --- | --- | --- |
| 全排序（Naive） | O(M log M) per group | O(M) | ❌ |
| Flink Top-N（堆） | O(log N) per record | O(N + K) | ✅ |


Flink Top-N 的复杂度与数据总量无关，只与 N 相关

+ 状态：仅存 Top-N + 有限候选，O(N) 内存
+ 复杂度：O(log N) per record，与数据总量无关

排序字段要求：

+ 必须是 确定性字段；
+ 建议加 WATERMARK 防乱序



因此：高性能 Top-N 的不支持场景

| 场景 | 原因 |
| --- | --- |
| `RANK()` / `DENSE_RANK()` | 需要全排序 |
| 动态 N（如 `row_num <= some_column`） | 无法预知 N |
| 多个窗口函数 | 优化器无法识别 |

# 嵌套 Top-N 解决热点问题
TopN 的计算与 GroupBy 的计算类似，**如果数据存在倾斜，则会有计算热点的现象**。

+ 比如全局 TopN，那么所有的数据只能汇集到一个节点进行 TopN 的计算，那么计算能力就会受限于单台机器，无法做到水平扩展。

解决思路：与 GroupBy 是类似的，就是使用嵌套 TopN，或者说两层 TopN。

+ 在原先的 TopN 前面，再加一层 TopN，用于分散热点。



例如，计算全网排名前十的商铺，会导致单点的数据热点

+ 那么可以先加一层分组 TopN，组的划分规则是根据店铺 ID 哈希取模后分成128组（并发的倍数）
+ 第二层 TopN 与原先的写法一样，没有 PARTITION BY

第一层会计算出每一组的 TopN，而后在第二层中进行合并汇总，得到最终的全网前十。

第二层虽然仍是单点，但是大量的计算量由第一层分担了，而第一层是可以水平扩展的。

使用嵌套 TopN 的优化写法如下所示：

```sql
CREATE VIEW tmp_topn AS
SELECT *
FROM (
  SELECT *,
    ROW_NUMBER() OVER (PARTITION BY HASH_CODE(shop_id)%128 ORDER BY sales DESC) AS rownum
  FROM shop_sales)
WHERE rownum <= 10

SELECT *
FROM (
  SELECT shop_id, shop_name, sales,
    ROW_NUMBER() OVER (ORDER BY sales DESC) AS rownum
  FROM tmp_topn)
WHERE rownum <= 10

```

# 常用优化方法
## 调整 Top-N 的 Cache 大小
### 参数简介
参数：`table.exec.topn.cache-size`，优化 Top-N 查询性能的配置参数

+ 限制了每个分组中“Top-N 之外”的候选记录缓存数量
+ 防止状态无限增长，同时为未来可能进入 Top-N 的记录保留缓冲空间



Top-N 的状态问题：Flink 维护两类状态数据

| 状态 | 数据结构 | 作用 | 是否受 `cache-size` 限制 |
| --- | --- | --- | --- |
| Top-N 记录 | 小顶堆（Min-Heap） | 存储当前排名前 N 的记录 | ❌ 不受限制（始终保留） |
| 候选记录 | 大顶堆（Max-Heap） | 存储 Top-N 之外、但可能未来进入 Top-N 的记录 | ✅ 受 `cache-size` 限制 |


+ `cache-size` 只限制候选记录（bottomHeap）的数量；
+ Top-N 本身（topHeap）永远完整保留，不受此参数影响。



`table.exec.topn.cache-size`

+ 定义：每个分区（partition）中，Top-N 算子缓存的非 Top-N 记录的最大数量。
+ 默认值：无限制（-1），风险：
    - 缓存全部排名 > N 的数据
+ 作用范围：每个 (`partition_key`) 独立计数
+ 单位：记录条数（per partition）



该参数的作用机制：当 bottomHeap 大小达到 `cache-size`时：

1. 新来的非 Top-N 记录会与 bottomHeap 的堆顶（最大值） 比较；
2. 如果新记录更优，则替换堆顶；
3. 否则直接丢弃。

效果：bottomHeap 始终保存当前最有希望进入 Top-N 的 K 条记录（K = cache-size）

源码位置：`flink-table/flink-table-runtime/src/main/java/org/apache/flink/table/runtime/operators/rank/RetractableTopNFunction.java`

### 参数值过大的后果
如果该参数使用默认值（-1）或不设置或设置过大，则后果：

| 问题 | 说明 |
| --- | --- |
| State 无限增长 | 每个分组缓存所有历史记录 |
| 内存 OOM | 尤其高基数 key（如 user_id） |
| Checkpoint 失败 | State 太大无法持久化 |
| 背压（Backpressure） | 状态操作变慢，上游阻塞 |


### 参数值过小的后果
如果该参数设置过小，则导致：Top-N 结果不准确

`table.exec.topn.cache-size` 设置过小，会导致部分本应有机会进入 Top-N 的候选记录被提前丢弃；

当 Top-N 内部记录发生**撤回**时，系统无法从被丢弃的记录中选取更优者补位，从而输出次优甚至错误的 Top-N 结果



`cache-size`过小，说明`bottomHeap`过小，即大顶堆过小

+ 大顶堆的堆顶是：当前缓存中最好的非 Top-N 记录（最大值）
+ 大顶堆的堆底是：若干较小值（无序，可能包含次优项）

如果在`bottomHeap`已满后，有新数据进来，比堆顶小，就会被丢弃，实际上新数据不一定比非堆顶的数据要差

+ 当多个 Top-N 记录同时撤回，且被丢弃的记录实际优于剩余候选时

就会导致 Top-N 不准确



参数值过小的后果：

| 环节 | 问题 |
| --- | --- |
| 1. 候选缓存不足 | `cache-size` 太小，无法容纳所有潜在有效候选 |
| 2. 淘汰策略激进 | 新候选若不如当前 `bottomHeap` 堆顶（最大值），即被丢弃 |
| 3. 信息永久丢失 | 被丢弃的记录不会进入 State，后续无法恢复 |
| 4. 撤回暴露缺陷 | 当 Top-N 内部变动时，系统只能从有限候选中补位，可能遗漏更优解 |


### 合理设置该参数值
初始值不低于 3N

根据业务波动性调整

| 业务类型 | 推荐 cache-size |
| --- | --- |
| 稳定排行榜（如月榜） | `2N ~ 3N` |
| 日榜/小时榜 | `5N ~ 10N` |
| 实时竞价/股票 | `10N ~ 20N` |


原则：

永远优先保证结果正确性，在内存允许范围内尽量设大 `cache-size`。只有在资源极度受限时，才谨慎调小，并配合严格的数据校验机制。

## **无排名优化（解决数据膨胀问题）**
Top- N语法：

```plsql
SELECT *
FROM (
  SELECT *,
  ROW_NUMBER() OVER ([PARTITION BY col1[, col2..]]
                    ORDER BY col1 [asc|desc][, col2 [asc|desc]...]) AS rownum
  FROM table_name)
WHERE rownum <= N [AND conditions]
```



数据膨胀问题：

+ `rownum` 字段：写入结果表，这可能导致数据膨胀的问题。
    - 例如，原排名 9 的数据，更新后排名上升到 1，则从 1 到 9 的所有的排名字段都发生了变化
    - 需要将这些排名字段，都作为更新都写入结果表
    - 则会产生数据膨胀，结果表会因收到太多的数据而降低更新速度。



优化方式：

+ <font style="background-color:#FBDE28;">使用</font>`<font style="background-color:#FBDE28;">ROW_NUMBER()</font>`<font style="background-color:#FBDE28;">时，绝对不能使用 </font>`<font style="background-color:#FBDE28;">SELECT *</font>`
+ 如果结果表不需要排序字段，则在外层查询中，将 `rownum` 字段裁剪掉

```plsql
-- 最外层的字段，不写 rownum
SELECT col1, col2, col3
FROM (
  SELECT col1, col2, col3
  ROW_NUMBER() OVER ([PARTITION BY col1[, col2..]]
                    ORDER BY col1 [asc|desc][, col2 [asc|desc]...]) AS rownum
  FROM table_name
  )
WHERE rownum <= N [AND conditions]
```

## `PARTITION BY` 字段的选择注意事项
在实现按时间周期（如每天、每小时）统计的 Top-N 时，`PARTITION BY` 字段中应包含该时间周期字段

这是因为

+ 每个时间周期内的 Top-N 是独立的计算单元，将其纳入分区键可确保状态按周期隔离；
+ 同时，配合 State TTL 能够安全地清理过期周期的状态，避免不同周期的数据相互干扰或状态无限累积



Flink 的 Top-N 算子基于 Keyed State 进行计算，状态的生命周期和隔离性由`PARTITION BY`的字段组合决定：

+ 若业务需求是“每天的 Top-N”，则 2024-06-01 的排名与 2024-06-02 完全无关，二者应视为两个独立的计算任务。
+ 将`dt`（日期）加入`PARTITION BY`后，状态`key`变为`(category, dt)`，使得每天的数据在各自的状态分区内处理，互不污染。
+ 同时，**State TTL 会以**`** (category, dt) **`**为单位进行老化：**
    - 当某天的数据不再更新（如 3 天后），其对应的状态可被自动清除，既防止内存泄漏，又不影响其他日期的计算。

因此，时间周期字段作为分区键的一部分，是实现周期性 Top-N 正确性和资源可控性的必要设计。
