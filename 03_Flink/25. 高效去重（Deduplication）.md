# 简介
在 Flink SQL 中，高效去重（Deduplication） 

+ 通常通过 Top-N 语法中 `N = 1` 的特例实现。
+ Flink 对此类场景做了深度优化，使其远比普通 `ROW_NUMBER() ... WHERE row_num = 1` 更高效、更轻量。



场景：

+ 保留每个用户最近一次登录记录；
+ 保留每个订单最新状态；
+ 每个设备最新心跳。



**<font style="background-color:#FBDE28;">Flink SQL使用</font>**`**<font style="background-color:#FBDE28;">ROW_NUMBER</font>**`**<font style="background-color:#FBDE28;">去重时，状态中只会保留最新的 1 或者 n 条数据，单个主键的状态不会无限增长。</font>**

# 语法
```sql
SELECT [column_list]
FROM (
   SELECT [column_list],
     ROW_NUMBER() OVER ([PARTITION BY col1[, col2...]]
       ORDER BY time_attr [asc|desc]) AS rownum
   FROM table_name)
WHERE rownum = 1
```

当满足特定条件时，Flink 不会使用通用 Top-N 算子，而是将其优化为专用的 Deduplication 算子，内存和 CPU 开销极低。



Deduplication 支持**事件时间**、**处理时间**两种时间语义

+ Order by 事件时间 DESC：会出现回撤流，因为当前 key 下 可能会有比当前事件时间还大的数据
+ Order by 事件时间 ASC：会出现回撤流，因为当前 key 下 可能会有比当前事件时间还小的数据
+ Order by 处理时间 DESC：会出现回撤流，因为当前 key 下 可能会有比当前处理时间还大的数据
+ Order by 处理时间 ASC：不会出现回撤流，因为当前 key 下 不可能会有比当前处理时间还小的数据

# Flink 如何识别“去重”模式？（优化器规则）
Flink 通过 `RankingRewriteRule` 识别以下模式并重写为 `LogicalDeduplication`：

## 触发条件（必须同时满足）
| 条件 | 说明 |
| --- | --- |
| `ROW_NUMBER()` 是唯一窗口函数 | 不能有其他聚合 |
| `PARTITION BY` 非空 | 指定去重维度 |
| `ORDER BY` 字段明确 | 用于确定“保留哪一条” |
| `WHERE row_num = 1` | **必须是 **`**= 1**`**，不能是 **`**<= 1**`** 或其他** |
| 无 `RANGE/ROWS` 显式窗口 | 默认 `UNBOUNDED PRECEDING AND CURRENT ROW` |


## 注意：非时间字段也能触发 Dedup
Flink 的 Deduplication（去重）优化并不要求 ORDER BY 字段必须是时间属性（Time Attribute）。

只要满足 Top-N 且 `N=1` 的语法模式，Flink 就会将其优化为高效的 Deduplication 算子，无论 `ORDER BY` 的字段是什么类型。



使用非时间字段排序的场景：也可触发 Dedup

| 场景 | 可用 `ORDER BY` |
| --- | --- |
| 保留 ID 最大的记录 | `id DESC` |
| 保留字典序最大的名称 | `name DESC` |
| 基于优先级去重 | `priority DESC` |




源码：在 Flink 的优化规则 `RankingRewriteRule` 中，判断是否转为 Deduplication 的核心逻辑是：

```java
// org.apache.flink.table.planner.plan.rules.logical.RankingRewriteRule
if (rankRange.getStart() == 1 && rankRange.getEnd() == 1) {
  // 转为 Deduplication
}
```

没有检查` ORDER BY` 字段是否为时间类型



+ “能不能优化” → 只看 SQL 结构（`N=1`）；
+ “结果对不对” → 取决于 `ORDER BY` 字段是否合理表达业务顺序。

# 执行计划
执行计划对比：通用 Top-N vs 专用 Dedup

## 普通 Top-N（N > 1）
```latex
Rank(
  rankType=ROW_NUMBER,
  rankRange=[1, N],
  ...
)
```

## 高效去重（N = 1）
```latex
Deduplication(
  partitionBy=[user_id],
  orderBy=[event_time DESC],
  keepFirst=false  // false = 保留最后（最新）
)
```

`Deduplication` 算子专为 `N=1` 设计，内部不使用堆，只维护单条记录

# 原理：源码解析
`DeduplicationProcessFunction` 源码

专用去重算子实现：`org.apache.flink.table.runtime.operators.deduplication.DeduplicationProcessFunction`

## 状态结构（极简）
```java
// 每个 key 只存一条记录！
private final MapState<Void, RowData> state;
// 实际等价于 ValueState<RowData>
```

 内存占用：`O(1) per key`（仅存当前保留的那条记录）

## 处理逻辑（伪代码）
```java
void processElement(RowData input) {
  RowData current = state.value(); // 获取当前保留的记录
  
  if (current == null) {
    // 首次到达，直接保留
    state.update(input);
    output(input);
  } else {
    // 比较排序字段（如 event_time）
    if (shouldKeep(input, current)) { // input 更新？
      // 撤回旧记录
      output(retract(current));
      // 保留新记录
      state.update(input);
      output(input);
    }
    // 否则丢弃 input（无需输出）
  }
}
```

## 关键策略：`keepFirst` vs `keepLast`
| 写法 | `keepFirst` | 行为 |
| --- | --- | --- |
| `ORDER BY time ASC` | true | 保留最早（第一条） |
| `ORDER BY time DESC` | false | 保留最新（最后一条）✅ |


绝大多数去重场景用 `ORDER BY event_time DESC`（保留最新）

# 与其他去重方式对比
## 与通用 Top-N（N=1）的本质区别
| 维度 | 通用 Top-N（N=1） | 专用 Deduplication |
| --- | --- | --- |
| 算子类型 | `Rank` | `Deduplication` |
| 状态结构 | 小顶堆 + 候选堆 | 单条记录（`ValueState`） |
| 内存开销 | `O(1 + cache-size)` | `O(1)` |
| 是否支持 `retract` | 是 | 是 |
| 是否检查 `cache-size` | 是 | 否（无需） |
| CPU 开销 | 堆操作（log K） | 直接比较（O(1)） |


当 N=1 时，Flink 自动选择最轻量的实现路径，性能接近手写 `KeyedProcessFunction`。

## 与其他去重方式对比
| 方法 | 语法 | 是否高效 | 支持更新 | 适用场景 |
| --- | --- | --- | --- | --- |
| Top-N (N=1) | `ROW_NUMBER() ..WHERE =1` | ✅ 是 | ✅ 是 | 推荐！实时去重 |
| `DISTINCT` | `SELECT DISTINCT user_id` | ❌ 否（全缓存） | ❌ 否 | 静态去重Batch |
| `GROUP BY` | `GROUP BY user_id` | ⚠️ 部分 | ❌ 否 | 需配合 `MAX_BY` |
| `MATCH_RECOGNIZE` | 复杂模式 | ❌ 重 | ✅ 是 | 复杂序列去重 |


# 示例
## 事件时间
场景：腾讯 QQ 用户等级的场景

+ 每一个 QQ 用户都有一个 QQ 用户等级
+ 需要求出当前用户等级：在 星星，月亮，太阳 的用户数分别有多少

```plsql
-- 数据源：当每一个用户的等级初始化及后续变化的时候的数据，
-- 即用户等级变化明细数据。
CREATE TABLE source_table (
  user_id BIGINT COMMENT '用户 id',
  level STRING COMMENT '用户等级',
  row_time AS CAST(CURRENT_TIMESTAMP AS TIMESTAMP(3)) COMMENT '事件时间戳',
  WATERMARK FOR row_time AS row_time
) WITH ();

-- 数据汇：输出即每一个等级的用户数
CREATE TABLE sink_table (
  level STRING COMMENT '等级',
  uv BIGINT COMMENT '当前等级用户数',
  row_time TIMESTAMP(3) COMMENT '时间戳'
) WITH (
  'connector' = 'print'
);

-- 处理逻辑：
INSERT INTO sink_table
SELECT  level
      , COUNT(1) AS uv
      , MAX(row_time) AS row_time
FROM (
      SELECT
          user_id,
          level,
          row_time,
          ROW_NUMBER() OVER(PARTITION BY user_id ORDER BY row_time DESC) AS rn
      FROM source_table
)
WHERE rn = 1
GROUP BY level;
```

SQL 语义分析如下

+ 数据源
    - 消费 Kafka 的数据，将数据按照`Partition By`的`key`通过 Hash 分发策略发送到下游去重算子
+ `Deduplication` 去重算子
    - 接受到上游数据之后，根据`Order By`中的条件，对比当前数据和之前数据时间戳的大小
        * 如果当前数据时间戳大：则撤回之前下发的中间结果，将最新的结果下发
        * 如果当前数据时间戳小：则不做操作。
    - 此算子产出的结果：每一个用户的对应的最新等级信息。
+ `Group By` 聚合算子
    - 接受到上游数据之后，对数据进行聚合计算（每一个等级的用户数），结果发往下游数据汇算子
+ 数据汇
    - 接收到上游的数据后，输出到外部存储引擎中

## 处理时间
场景：

+ 基于最原始的日志明细数据
+ 根据用户`id`，筛选出这个用户当天的第一条数据
+ 发往下游，下游可以据此计算分各种维度的 DAU

```plsql
-- 数据源：原始日志明细数据
CREATE TABLE source_table (
  user_id BIGINT COMMENT '用户 id',
  name STRING COMMENT '用户姓名',
  server_timestamp BIGINT COMMENT '用户访问时间戳',
  proctime AS PROCTIME()
) WITH ();

-- 数据汇：根据 user_id 去重的第一条数据
CREATE TABLE sink_table (
  user_id BIGINT,
  name STRING,
  server_timestamp BIGINT
) WITH (
  'connector' = 'print'
);

-- 处理逻辑：
INSERT INTO sink_table
SELECT user_id,
       name,
       server_timestamp
FROM (
      SELECT
          user_id,
          name,
          server_timestamp,
          ROW_NUMBER() OVER(PARTITION BY user_id ORDER BY proctime ASC) AS rn
      FROM source_table
  )
WHERE rn = 1
```

SQL 语义

+ 数据源
+ `Deduplication` 去重算子
    - 根据数据中的处理时间，进行判断
        * 如果 State 中没有存储过当前`key`，说明是当前`key`的第一条数据，则直接发往下游
        * 如果 State 中存储过当前`key`，说明不是第一条数据，则直接丢弃
+ 数据汇

# 高级用法：保留第一条（去重取首）
```plsql
-- 保留每个用户首次登录
SELECT col_list
FROM (
  SELECT col_list,
    ROW_NUMBER() OVER (
      PARTITION BY user_id
      ORDER BY event_time ASC   -- 注意：ASC
    ) AS row_num
  FROM user_log
)
WHERE row_num = 1;
```

+ Flink 仍会优化为 `Deduplication(keepFirst=true)；`
+ 状态只保留最早的一条，后续全部丢弃

# 注意事项与限制
1. 必须用`row_num = 1`，不能用 `<= 1`

```sql
-- ✅ 正确：触发 Dedup 优化
WHERE row_num = 1

-- ❌ 错误：退化为通用 Top-N（即使 N=1）
WHERE row_num <= 1
```

2. 不支持多字段复杂排序（但实际支持）
    - 支持 `ORDER BY time DESC, id ASC`
    - 只要能比较大小，Flink 就能处理
3. 时间属性建议
    - `ORDER BY` 字段强烈建议使用事件时间（Event Time）；
    - 配合 `WATERMARK` 防止乱序导致去重错误。
4. 监控状态大小
    - 指标：`numKeyGroups`, `stateSize`
    - 警惕高基数 `key`（如 `user_id` 无过滤）
