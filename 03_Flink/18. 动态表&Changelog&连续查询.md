# 概述
动态表（Dynamic Table）和连续查询（Continuous Query） 是

+ Table API / SQL 实现【流批统一】与【关系代数语义】的两大核心理论基础
+ 它们将无限、无界的流数据映射为“不断变化的表”，并允许用户用标准 SQL 对其进行操作



具体的：

+ FlinkSQL【动态输入表技术】：将【输入流】映射到 SQL 中的【输入表】
+ FlinkSQL【连续查询技术】：将【计算逻辑】映射到 SQL 中的【运算语义】
+ FlinkSQL【动态输出表技术】：将输出【数据流】映射到 SQL 中的【输出表】

# 动态表（Dynamic Table）
## 产生背景
传统观念

+ 批处理：操作静态表（有限数据集）
+ 流处理：处理无界事件流（记录逐条到达）



Flink 认为：流 = 表的 Changelog

+ 流是动态表的 Changelog
+ 动态表是流的逻辑抽象

| 视角 | 描述 |
| --- | --- |
| 流视角 | `(Alice, +1)`, `(Bob, +1)`, `(Alice, +1)`… |
| 表视角 | 一张不断更新的表：`user_id` |


这样，同一个 SQL 查询既可用于批（静态表），也可用于流（动态表），实现【流批一体】。

## 动态表详解
动态表：是一个随时间不断变化的表，其行可以被插入、更新或删除

+ 对无限流数据的【表抽象】，是逻辑上的概念，在任意时间点，都可以像静态表一样可查询
+ 初始为空，每当有新事件到达，表就可能发生变化
+ 变化：以【Changelog 流】的形式输出

# 连续查询（Continuous Query）
## 连续查询定义
连续查询：是对【动态表】执行的 SQL 查询，它会**持续监听输入表的变化**，**并实时更新输出表**。

+ 不是一次性执行，而是永不停止的计算，除非显式停止
+ **输入：动态表**
+ **输出：也是一个动态表**
+ **输出变化：以【Changelog 流】形式发出**

## 连续查询特性
1. 增量计算：不重算全量数据，只根据**新输**入【增量更新】状态和输出。
    - **<font style="color:rgb(51, 51, 51);">每次计算的结果，都是一个中间结果</font>**
2. 状态驱动：如 `GROUP BY` 会为每个 `key` 维护聚合状态。
3. 时间语义支持：支持基于 事件时间的窗口（`TUMBLE, HOP, SESSION`）

## 举例：窗口聚合
场景：实时统计每小时用户点击次数

```plsql
SELECT 	user_id,
        COUNT(*) AS click_cnt,
        TUMBLE_START(ts, INTERVAL '1' HOUR) AS w_start
FROM clicks
GROUP BY user_id, TUMBLE(ts, INTERVAL '1' HOUR);
```



执行过程

1. 输入动态表 `clicks`：不断追加新点击事件
2. 连续查询执行：
    - 按 `user_id` 和 1 小时滚动窗口分组
    - 为每个` (user_id, window) `**维护 **`**count**`** 状态**
3. 输出动态表
+ 当窗口触发（`watermark` 超过窗口结束时间），输出最终结果（`+I`）
+ 若使用 `CUMULATE` 或早期触发，可能先输出`-U/+U`



输出示例

```latex
+I (Alice, 5, 2024-06-01 10:00)  -- 窗口 [10:00, 11:00) 完成
+I (Bob, 3, 2024-06-01 10:00)
```

 此输出可直接写入 Kafka、Paimon 等，供下游消费。

## 动态表的两种更新模式
Flink 根据连续查询是否产生更新，将动态表分为两类

| 类型 | 名称 | 特点 | 触发条件 |
| --- | --- | --- | --- |
| Append-only Stream | 仅追加流 | 只有 `+I` 消息 | 查询无 `GROUP BY`、无 `JOIN`、无 `DISTINCT`、无窗口 |
| Changelog Stream | 更新流 | 包含 `+I`, `-U`, `+U`, `-D` | 查询包含聚合、连接、去重等 |


# Changelog 
在 Flink 的 Table API / SQL 层（特别是流处理模式下）

**数据在各算子之间的流转：本质上是【通过 Changelog（变更日志）模型完成的】**

这是 Flink 实现动态表（Dynamic Table）和连续查询（Continuous Query） 的核心机制。

## Changelog 定义
Changelog： 类似 <font style="color:rgb(25, 27, 31);">MySQL Binlog</font>

+ 一种【描述动态表变化】的流式数据模型，每条记录代表对结果表的一次变更操作，包含以下类型

| Changelog 类型 | 含义 | 说明 |
| --- | --- | --- |
| `+I` (Insert) | 插入新行 | 首次出现某条记录 |
| `-U` (Update Before) | 更新前的旧值 | 仅在更新时出现（可选） |
| `+U` (Update After) | 更新后的新值 | 更新后的最新状态 |
| `-D` (Delete) | 删除一行 | 显式删除某条记录 |


📌 注意：Flink 内部使用 `RowKind` 枚举表示这四种类型。

## 引入 Changelog 的必要性
传统数据库中的表是静态的，而 Flink 处理的是动态表

为了在流上模拟表的行为，Flink 引入了 Changelog Stream 模型：

+ 输入流 → 动态表 → Changelog 流 → 下游算子
+ 每个算子都消费 Changelog，并产生新的 Changelog
+ 下游算子接收到的就是一系列的 Changelog 消息，而不是“完整快照”。

## Changelog 如何在算子间流转
Flink 的执行计划中，每个 Table 算子（如 Aggregate、Join）都会生成一个 Changelog Stream，并通过 DataStream API 的 `Row + RowKind` 传递给下一个算子。



Row 对象 + RowKind（行类型）

+ 代码层面：`Row row = Row.withKind(RowKind.INSERT, 1001, "Jack");`
    - 控制台输出：`+I[1001, "Jack"]`
+ `RowKind`枚举值：`INSERT, UPDATE_BEFORE, UPDATE_AFTER, DELETE`
+ 存储或传输：序列化为 JSON、Avro、Protobuff 等格式
+ Row 对象更高效，内存操作时不需要序列化/反序列化



在 Flink Web UI 的 DAG 图中，算子之间的连接就是 Changelog 流。

## Changelog 的编码方式
### 概念
Changelog：数据流

+ 是语义模型
+ 描述“表如何变化”，比如是插入？更新？还是删除？



编码方式：Changelog 的实现策略，Changelog 如何被具体表达（展现）

+ Flink 如何用消息类型（RowKind）表达“表的变化”，Flink 具体用哪些消息组合来表达这种变化
+ Flink 不会总是用全部 4 种消息类型，而是根据场景选择最高效的子集组合
+ “编码方式”决定了在 `print()` 中看到的 Changelog 长什么样子。



举例：

+ 语义（Changelog）：“把旧值改成新值”
+ 编码方式 1（Retract）：“先发一条‘删掉旧值’，再发一条‘插入新值’”
+ 编码方式 2（Upsert）：“直接发一条‘新值’，系统自己覆盖旧值”

“编码方式”的本质：是 Changelog 语义的物理实现协议。

### 三种编码方式
Changelog 有 3 种主要编码方式

1. Append-only
2. Retract
3. Upsert



| 编码方式 | 编码规则 | 特点 | 是否需要主键 | 状态存储 |
| --- | --- | --- | --- | --- |
| Append-only | 只使用 `+I`   每条输入 → 一条 `+I` 输出 | 最简单、最高效、无状态开销 | 否 | 无 |
| Retract（撤回流） | 用 `-D` + `+I` 模拟更新   不使用 `-U/+U` | 每次更新发 2 条消息，   网络和存储开销大   通用性强，不要求主键 | 否 | 全量缓存 |
| Upsert（更新插入流） | 首次出现 → `+I`   更新 → `+U`（直接发新值）   删除 → `-D`   通常省略 `-U`（因 Sink 可通过主键定位旧值） | 更新只需 1 条消息，Sink（如 Kafka、Hudi）可通过主键自动覆盖。 | 是 | 主键索引 |


建议

+ 如果确认有主键：使用 Upsert（高效）
+ 如果没有或不确认有主键：使用 Retract（通用但低效）



注意：

+ Flink 默认优化掉 `-U`，除非强制保留（如审计场景）。
    - `tableEnv.toChangelogStream(table, ChangelogMode.all()).print();`

### 查看编码方式
+ 方法 1：打印 Changelog
+ 方法 2：检查执行计划
    - 在 Flink WebUI 的 JobGraph 中，算子名含`Retract`或`Upsert`

### 如何控制编码方式
Flink 自动选择编码方式，但可以通过以下方式影响它

1. 定义主键：触发 Upsert
2. 避免全局聚合：避免 Retract
    - 例如：`SELECT COUNT(*) FROM tb;`

### 特殊场景下的 Changelog 变体
底层仍是以上三种，但某些操作中会表现出混合或优化行为

1. **Full Changelog：完整变更日志**
    - 特点：显式输出 全部四种消息：`+I, -U, +U, -D`
    - 出现场景：
        * 当 Flink 无法优化掉`-U`时
        * 常见于复杂多层查询或自定义函数
        * 使用 `toChangelogStream(table, ChangelogMode.all())` 强制启用
    - 用途：调试、需要精确旧值的场景（如审计日志）
2. **Windowed Changelog：窗口化变更日志**
    - 特点：每个窗口独立维护状态，窗口结束前可能多次更新（early firing）
    - 消息类型：`+I`（最终触发）或`-U/+U`（早期触发）
    - 行为
        * 若只在窗口结束时输出 → `+I`
        * 若配置 `emit.early.fire` → 先` -U/+U`，最后 `+I`
    - 注意：这不是新类型，而是 Upsert 或 Retract 在窗口语义下的表现。
3. **Temporal Join Changelog：时态连接变更日志**
    - 特点：维表更新会触发撤回旧 `join` 结果 + 插入新结果
    - 消息类型：`-D` + `+I`（即 Retract 模式）
    - 原因：即使事实表有主键，维表变化会导致 `join` 结果整体变化，Flink 无法用 Upsert 表达
    - 这属于 Retract 流的典型应用场景。

### 对比 Retract 和 Upsert
<font style="color:rgb(15, 17, 21);">Upsert 与 Retract 流的主要区别在于：</font>`<font style="color:rgb(15, 17, 21);">UPDATE</font>`<font style="color:rgb(15, 17, 21);"> 操作是用单个 Message 编码的，因此效率更高。</font>

# Sink
## Sink 与 Changelog
Sink 必须能够正确处理上游算子产生的 Changelog 流，否则会导致数据不一致、重复或丢失。

不同的 Sink 对 Changelog 支持的差异主要在：是否可以处理`-U`/`+U`/`-D`语义

| Sink 类型 | 是否支持完整 Changelog | 适用场景 | Exactly-Once 支持 |
| --- | --- | --- | --- |
| Upsert-Kafka | ✅ 完全支持 | 实时维表、聚合结果 | ✅（需开启事务） |
| 普通 Kafka | 原样输出 | 调试、中间层传输 | ✅ |
| Hudi | ✅（`-D`需配置） | 数据湖、CDC 入湖 | ✅ |
| JDBC / File / Hive | ❌ 不支持 | 初始化、日志归档 | ⚠️ 需自定义 |
| Print / Blackhole | ✅（调试用） | 开发测试 | ❌ |


## 常用 Sink 解析
1. **Upsert-Kafka（推荐用于聚合结果）**
+ 特点：
    - 要求定义 主键（`PRIMARY KEY`）
    - 自动将相同 `key` 的新值覆盖旧值
    - 忽略`-U` 消息（因 Kafka 本身无 `delete` 语义）
    - 利用 Kafka key 自动 `upsert`，不显式存 `rowkind`
    - JSON 格式
+ 适合：实时大屏，维表缓存，指标服务
+ DDL

```plsql
CREATE TABLE user_clicks_sink (
  user_id STRING,
  total_clicks BIGINT,
  PRIMARY KEY (user_id) NOT ENFORCED  -- ⭐ 必须定义主键
) WITH (
  'connector' = 'upsert-kafka',
  'topic' = 'user-clicks-result',
  'properties.bootstrap.servers' = 'kafka:9092',
  'key.format' = 'json',
  'value.format' = 'json'
);

-- 写入（自动处理更新）
INSERT INTO user_clicks_sink
SELECT user_id, COUNT(*) FROM clicks GROUP BY user_id;
```

2. **普通 Kafka（谨慎使用）**
+ 特点
    - 原样输出 Changelog，包括 `-U`
    - Kafka 的下游必须能解析 Flink 的 Changelog 格式  
会保留 `rowkind` 字段
    - JSON 格式
+ 用途：仅用于调试或中间层传输（下游是另一个 Flink 作业）
+ DDL

```plsql
CREATE TABLE debug_sink (
  user_id STRING,
  cnt BIGINT
) WITH (
  'connector' = 'kafka',
  'topic' = 'debug-changelog',
  'format' = 'json'  -- 会输出 {"rowkind":"+U","fields":[...]}
);

INSERT INTO debug_sink
SELECT user_id, COUNT(*) FROM clicks GROUP BY user_id;
```

+ 输出示例（JSON）

```json
{"rowkind":"+I","fields":["Alice",1]}
{"rowkind":"-U","fields":["Alice",1]}
{"rowkind":"+U","fields":["Alice",2]}
```

## 生产环境使用建议
1. 聚合结果 → 用 upsert-kafka 或 Hudi

避免下游处理复杂 changelog。

2. 避免将 Changelog 写入不支持更新的系统

如普通 Kafka、HDFS、JDBC（除非确认只有 +I）。

3. 始终定义主键（PRIMARY KEY）

对 upsert-kafka、Hudi 等是必需的。

4. 调试时用`toChangelogStream().print()`

观察实际输出的 changelog 类型

5. Exactly-Once 配置：`<font style="color:rgb(0, 0, 0);">'sink.semantic' = 'EXACTLY_ONCE' -- 仅 upsert-kafka / Paimon 支持</font>`

# FlinkSQL 处理流程
动态表 & 连续查询两项技术在一条流 SQL 中的执行流程，总共包含了三个步骤

+ 第一步：将数据输入**流**转换为**动态表**。这里的转化其实就是指将【输入流】映射（绑定）为一个【动态输入表】
+ 第二步：在动态表上执行一个连续查询，生成一个新的动态表（Changelog）
+ 第三步：生成的【动态表】被转换回【数据输出流】（通过解析 Changelog）

## 输出：动态表转换为输出流
+ 连续查询（Continuous Query）的输出：Changelog
+ 在将 Changelog 转换为流或将其写入外部系统时，需要对`INSERT/DELETE/UPDATE`（这些描述更改的数据）进行编码
+ Flink 的 Table API 和 SQL API 支持三种方式来编码一个动态表的变化
    - Append-Only
    - Retract
    - Upsert
