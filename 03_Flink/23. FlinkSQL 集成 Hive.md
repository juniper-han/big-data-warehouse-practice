Flink SQL 提供了对 Hive 元数据、Hive 表读写、Hive UDF 等的深度支持，使得用户可以用熟悉的 SQL 语法操作 Hive 数据

# 为什么需要 Flink 集成 Hive
| 场景 | 需求 | Flink + Hive 解决方案 |
| --- | --- | --- |
| 实时数仓 | 流数据写入 Hive 分区表 | Flink 流式 Sink 到 Hive（Streaming File Sink） |
| 元数据统一 | 避免重复建表 | 直接读写 Hive Catalog 中的表 |
| UDF 复用 | 使用已有 Hive UDF | 注册 Hive 函数，Flink SQL 直接调用 |
| 流批一体 | 同一套 SQL 处理流/批 | Hive 表既可作为批源（Batch Source），也可作为流汇 |


# Flink 集成 Hive 的理论基础
## HiveCatalog：元数据桥梁
1. Flink 通过 `HiveCatalog` 连接 Hive Metastore；
2. 支持**读取/写入** Hive 表结构、分区信息、存储格式（ORC/Parquet/TextFile）；
3. 不依赖 Hive 执行引擎，仅使用 Hive 元数据

## 集成模式
1. Flink Dialect Mode（默认）：使用标准 Flink SQL
2. ~~Hive Dialect Mode：使用 HiveQL 语法（如 ~~`~~INSERT OVERWRITE~~`~~）~~
    - 备注：Flink 1.15+ 已弃用

## 流式写入 Hive：Streaming File Sink
1. Flink 将流数据写入 HDFS 文件（in-progress）；
2. 通过 Partition Commit Trigger 控制何时提交分区（对 Hive 可见）；
3. 支持 Metastore 更新（自动 add partition）。

# FlinkSQL 读取 Hive 表（Batch / Streaming）
## Hive 表作为批处理数据源（Batch Source）
```plsql
-- 创建 Hive Catalog
CREATE CATALOG myhive WITH (
  'type' = 'hive',
  'default-database' = 'default',
  'hive-conf-dir' = '/opt/hive/conf'
);

USE CATALOG myhive;

-- 直接查询 Hive 表（批模式）
SELECT * FROM hive_table WHERE dt = '2024-01-01';
```

## Hive 表作为维表（Lookup Join）
维表必须是可 Lookup 的（如 JDBC/HBase），**Hive 表不能直接作为维表！**

但可通过以下方式间接使用：

+ 方案1：将 Hive 表同步到 HBase/Redis
+ 方案2：使用 Temporal Join（需 CDC）

**<font style="background-color:#FBDE28;">Hive 表是只读的批表，不能用于 Lookup Join（维表关联）</font>**

# 写入 Hive 表（Streaming Sink）
这是 Flink + Hive 最核心的生产场景

## 前提条件
+ Hive 表必须**预先存在**（Flink 不创建 Hive 表）；
+ Hive 表需为**分区表**（非分区表不支持流式写入）；
+ **Flink 任务需开启 Checkpoint**（分区提交依赖 Checkpoint）

## Hive 表 DDL 示例（在 Hive CLI 中执行）
```plsql
CREATE EXTERNAL TABLE ods.user_behavior (
  user_id STRING,
  item_id STRING,
  behavior STRING
)
PARTITIONED BY (dt STRING, h STRING)
STORED AS ORC
TBLPROPERTIES (
  'partition.time-extractor.timestamp-pattern' = '$dt $h:00:00',
  'sink.partition-commit.delay' = '1h',
  'sink.partition-commit.trigger' = 'partition-time',
  'sink.partition-commit.policy.kind' = 'metastore,success-file'
);
```

## FlinkSQL 写入
```plsql
-- 1. 创建 Hive Catalog
CREATE CATALOG myhive WITH (
  'type' = 'hive',
  'default-database' = 'ods',
  'hive-conf-dir' = '/opt/hive/conf',
  'version' = '3.1.2'
);

USE CATALOG myhive;
USE ods;

-- 2. 创建源表（如 Kafka）
CREATE TABLE kafka_source (
  user_id STRING,
  item_id STRING,
  behavior STRING,
  ts AS PROCTIME()
) WITH (
  'connector' = 'kafka',
  ...
);

-- 3. 插入 Hive 表（自动提取分区）
INSERT INTO user_behavior
SELECT 
  user_id,
  item_id,
  behavior,
  DATE_FORMAT(ts, 'yyyy-MM-dd') AS dt,
  DATE_FORMAT(ts, 'HH') AS h
FROM kafka_source;
```

## `CREATE CATALOG`说明
即使生产环境已存在 Hive Metastore，Flink 作业中仍然必须执行 `CREATE CATALOG ... WITH (...)`。

+ 这个 `CREATE CATALOG` 是“声明式连接”，不是“创建物理 Catalog”；
+ 它只对当前 Flink 会话（Session）有效，不影响 Hive 或其他作业



Hive 的 Catalog VS Flink 的 Catalog

| 系统 | Catalog 含义 | 是否需要“创建” |
| --- | --- | --- |
| Hive | Hive Metastore 本身就是一个 Catalog（包含所有 DB/表元数据） | 已存在，由 Hive 管理 |
| Flink | Flink 内部的一个“元数据访问接口”，用于连接外部系统（如 Hive） | 每次 Flink 会话需声明 |




`CREATE CATALOG` 的作用范围

+ 仅限当前 Flink SQL Session（即当前作业）；
+ 不会修改 Hive Metastore；
+ 不会影响其他 Flink 作业；
+ 作业结束后，该 Catalog 自动销毁



为什么必须显式声明 Flink Catalog？

Flink 默认不知道 Hive 的存在。即使集群已部署 Hive，Flink 也需要你明确告诉它：

+ Hive Metastore 地址在哪？（通过 `hive-conf-dir 中的 hive-site.xml`）
    - 配置路径必须是 Flink 集群可见的
+ Hive 版本是多少？（兼容性）
+ 默认数据库是什么？



若需访问多个 Hive 集群，可声明多个 Catalog

# 使用 Hive UDF/ UADF
Flink 支持加载 Hive 自定义函数：

```plsql
-- 注册 Hive 函数（需在 HiveCatalog 下）
LOAD MODULE hive;

-- 直接调用（假设 Hive 中有函数 my_udf）
SELECT my_udf(user_id) FROM hive_table;
```

要求：

+ UDF JAR 包需在 Flink classpath 中；
+ 函数已在 Hive Metastore 中注册

# 配置与调优
## 流式写入 Hive 的 TBLPROPERTIES
是 Flink 流式写入 Hive 分区表 的核心配置，共同决定了 “何时将写入的分区对 Hive 可见”。属于 Hive 表的 `TBLPROPERTIES`（表属性）

1. `partition.time-extractor.timestamp-pattern`
    - 从分区字段中提取一个逻辑时间戳（Logical Timestamp），用于后续的“分区提交触发”判断
    - 工作机制
        * Flink 将分区字段值按指定模式解析为 `TIMESTAMP`；
        * 该时间戳用于判断：“当前系统时间是否已超过（分区时间 + `delay`）”；
        * 只有满足条件，分区才会被提交。
    - 生产值：`'$dt $h:$m:00'`
        * 字段名必须和`pattern`匹配，如分区列为`day`，则`pattern`应为`$day`
        * 时区问题：默认使用 JVM 时区，若 Hive 使用 UTC，需确保 Flink 时区一致
        * 示例：分区字段`dt='2024-06-01', h='14', m='30'`，逻辑时间戳 = `'2024-06-01 14:30:00'`
    - 生产建议：
        * 命名规范：分区字段统一用 `dt`（日期）、`h`（小时）、`m`（分钟）；

```plsql
-- 按天分区
'partition.time-extractor.timestamp-pattern' = '$dt 00:00:00'
-- 按小时分区
'partition.time-extractor.timestamp-pattern' = '$dt $h:00:00'
-- 按分钟分区（高频场景）
'partition.time-extractor.timestamp-pattern' = '$dt $h:$m:00'
```

2. `sink.partition-commit.delay`
    - 在逻辑时间戳基础上，延迟一段时间再提交分区，确保数据完整性（避免因延迟数据导致分区过早提交）
    - 工作机制：分区提交条件（当`trigger = 'partition-time'`时）：`当前系统时间 ≥ (分区逻辑时间戳 + commit.delay)`
    - 为什么需要 Delay：流数据可能存在乱序或延迟
    - 生产值：`commit.delay ≥ 数据最大允许延迟`
3. `sink.partition-commit.trigger`
    - 定义分区提交的触发策略：基于事件时间（精准）还是处理时间（简单）
    - 生产值：`'partition-time'`（推荐）或 `'process-time'`
        * `partition-time`：基于事件时间，精准，支持乱序数据；
            + 触发条件：系统时间 ≥ (事件时间解析的分区字段 + `delay`)
        * `process-time`：基于系统时间，简单但可能不准，生产环境禁止使用
            + 当前系统时间 ≥ 首次写入时间 + `commit.delay`
    - `partition-time`工作流程
        * Flink 写入分区 `/dt=2024-06-01/h=14/`
        * 提取逻辑时间：`2024-06-01 14:00:00`
        * 等待至系统时间 ≥ `2024-06-01 15:00:00`（若 `delay=1h`）
        * 提交分区（更新 Metastore + 写 `_SUCCESS`）
    - `process-time`缺陷
        * 若作业在 14:30 停机，15:00 重启：14:00 分区会被重新创建；
        * 但 `process-time` 以 15:00 为起点，导致 14:00 分区永远不提交！
4. `sink.partition-commit.policy.kind`
    - 定义分区提交时执行的具体操作（策略组合）
    - 取值说明
        * `metastore`：调用 Hive Metastore API，执行 `ALTER TABLE ... ADD PARTITION`，必需。
            + 功能：将分区注册到 Hive Metastore；Hive/Spark/Trino 等引擎可直接查询该分区
        * `success-file`：在 HDFS 分区目录下创建空文件 `_SUCCESS`，可选（部分工具依赖此文件）
            + 标识分区已完成写入；被某些 ETL 工具（如 Oozie）用作依赖信号；
            + 注意：Hive 本身不依赖此文件，但生态工具可能需要
        * `custom`：自定义提交逻辑（需实现 `PartitionCommitPolicy`），特殊场景
    - 生产值：`'metastore,success-file'`



四参数协同工作流程

```plsql
-- Hive 表 DDL（在 Hive CLI 中执行）
CREATE EXTERNAL TABLE ods.user_log (
  user_id STRING,
  event STRING
)
PARTITIONED BY (dt STRING, h STRING)
STORED AS ORC
TBLPROPERTIES (
  'partition.time-extractor.timestamp-pattern' = '$dt $h:00:00',
  'sink.partition-commit.delay' = '1h',
  'sink.partition-commit.trigger' = 'partition-time',
  'sink.partition-commit.policy.kind' = 'metastore,success-file'
);
```

分区提交时间线

| 时间（系统时间） | 事件 |
| --- | --- |
| 14:05 | Flink 开始写入分区 `dt=2024-06-01, h=14` |
| 15:00 | 系统时间 ≥ `14:00 + 1h` → 触发提交 |
| 15:00:01 | 1. 调用 Hive Metastore 添加分区   2. 在 HDFS 目录创建 `_SUCCESS` |
| 15:00:02 | Hive 可查询 `WHERE dt='2024-06-01' AND h='14'` |


## Checkpoint 配置（必须）
流式写入 Hive 依赖 Checkpoint：在 FlinkSQL 中配置 Checkpoint

```plsql
-- 在 SQL 脚本开头设置
SET 'execution.checkpointing.interval' = '5 min';
SET 'execution.checkpointing.timeout' = '10 min';
SET 'execution.checkpointing.max-concurrent-checkpoints' = '1';
```

若未开启 Checkpoint，分区永远不会提交（文件 remain in `_tmp`）！

## 小文件合并（生产必备）
高频写入易产生小文件，需配置：

```plsql
-- 在 Hive 表 TBLPROPERTIES 中添加
'sink.rolling-policy.file-size' = '128MB',
'sink.rolling-policy.rollover-interval' = '30 min'
```

或使用 Compaction Job 定期合并
