# 时间
## 时间类型
Flink 中的时间与现实世界中的时间是不一致的，在 Flink 中被划分为三种

+ 事件时间（Event Time）：即事件实际发生的时间；
+ 处理时间（Processing Time）：事件被处理的时间。
+ ~~摄取时间（Ingestion Time）：事件进入流处理框架的时间；Flink1.13版本废弃；~~

### 事件时间：EventTime
事件时间 EventTime 

+ 是事件发生的时间，是数据本身携带的时间。
+ 这个时间通常是在事件到达 Flink 之前就嵌入到数据记录中，并且 `EventTime` 也可以从记录中提取出来
+ <font style="background-color:#FBDE28;">EventTime 程序必须指定如何生成 EventTime 水印，这是表示 EventTime 进度的机制</font>。
+ 最常用的时间

优势

+ EventTime 一旦产生，就不会改变
+ 假设所有数据都已到达， Event Time 操作将按照预期运行，即使在处理无序事件、延迟事件、重新处理历史数据时也会产生正确且一致的结果。

不足：

+ 需要处理无序事件
+ 由于只能等待一段有限的时间，因此就难以保证处理 EventTime 将产生完全一致和确定的结果。

### 处理时间：Processing Time
处理时间（Processing Time）：

+ 指的是数据被 Flink 框架处理时，机器的系统时间（即 `System.currentTimeMillis()`）
+ Processing Time 是 Flink 的时间系统中最简单的概念，它不关心数据真实发生的时间（EventTime），只关注 **数据被处理时的当前时间**

优势：

+ 简单高效：无需提取时间戳或处理乱序数据，性能高

存在的问题：

+ 结果非确定性：受系统负载、数据延迟等因素影响，同一数据在不同时刻处理结果可能不同

## 时间属性
时间属性是`StreamExecutionEnvironment`的一个属性，有以下值：

+ `ProcessingTime`
+ `EventTime`



设置时间属性：`EventTime`

```java
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment;
env.setStreamTimeCharacteristic(TimeCharacteristic.EventTime);
DataStream<SensorReading> sensorData = env.addSource(...);
```

如果要使用`ProcessingTime`，将`TimeCharacteristic.EventTime`替换为`TimeCharacteristic.ProcessingTime`即可。

# Timer
## Timer 简介
`Timer`：Flink 内部的定时器，类似一个”闹钟”，把 **Timer 想象成一个可以设置的、在特定时间点响起的“闹钟”**

+ **注册 Timer**：设置闹钟，你在处理某一条数据时，根据你的业务逻辑（比如“5分钟后提醒我检查这个订单的状态”），调用代码注册一个 Timer（`registerEventTimeTimer()`）
+ **触发检查**：当 Flink 的内部时间（EventTime 或 ProcessTime）来到设定的时间点，这个“闹钟”就会“响”
+ **Timer 触发回调**：这个“闹钟”一响，Flink 就会自动调用预先写好的一个方法（`onTimer`），可以在这个方法里执行任何操作，比如更新状态、发送告警、触发计算等。

****`**Timer**`** 的本质就是一个基于时间的回调机制**。它允许你在未来的某个时间点，执行一段特定的代码。

****`**Timer**`** 的核心机制**：当 Flink 的内部时间（EventTime 或 ProcessingTime）到达 Timer 预设的时间点时，自动触发回调函数 `onTimer`的执行。

## Timer 应用场景
Timer 主要用于那些 **“基于时间触发”** 而不是 **“基于事件触发”** 的业务逻辑

应用场景：**延迟触发/超时处理**

+ **场景**：监控订单状态。用户下单后，如果1小时内未支付，系统应自动关闭订单。
+ **如何用 Timer**：当“下单”事件到来时，你为一个 `orderId`注册一个 **1小时后（事件时间或处理时间）** 响起的 Timer。1小时后，Timer 触发，在 `onTimer`方法里，你去检查该订单的支付状态。如果还是“未支付”，就执行关单逻辑。

**核心思想**：Timer 让你有能力在**时间维度**上对数据流进行控制和管理

## Timer 的工作原理和代码流程
Timer 必须和 Flink 的 **状态（State）** 和 **KeyedProcessFunction** 这类低级 API 配合使用。

以 **“监控订单超时”** 为例，看看它是如何工作的：

**步骤 1：设置闹钟（注册 Timer）**

```java
public class OrderTimeoutFunction extends KeyedProcessFunction<String, OrderEvent, String> {

    // 状态描述符，用于存储订单状态（这里存的是订单创建时间）
    private ValueState<Long> orderState;

    @Override
    public void open(Configuration parameters) {
        // 初始化状态
        orderState = getRuntimeContext().getState(new ValueStateDescriptor<>("orderState", Long.class));
    }

    @Override
    public void processElement(OrderEvent event, Context ctx, Collector<String> out) throws Exception {
        // 当一条新的订单事件到来时...
        if (event.getType().equals("CREATE")) {
            // 1. 将订单创建时间保存到状态中
            orderState.update(event.getTimestamp());
            
            // 2. 注册一个Timer（设置闹钟）：
            // 计算闹钟响起的时间：当前事件时间 + 1小时（3600000毫秒）
            long triggerTime = event.getTimestamp() + 3600000;
            // ctx.timerService() 获取计时服务
            // registerEventTimeTimer 注册一个基于事件时间的Timer
            ctx.timerService().registerEventTimeTimer(triggerTime);
        }
        
        // 如果事件是“支付”事件，可以清理状态和Timer，这里省略
    }
```

**步骤 2：闹钟响了（处理 Timer 回调）**

```java
@Override
    public void onTimer(long timestamp, OnTimerContext ctx, Collector<String> out) throws Exception {
        // 当Timer触发时（闹钟响了），自动调用此方法
        // timestamp 就是你当初设置的 triggerTime
        
        // 1. 从状态中取出之前保存的订单创建时间
        Long savedTime = orderState.value();
        
        // 2. 做一些判断（例如，检查订单是否还是未支付状态，这里简化了）
        // 3. 执行超时逻辑，比如输出超时信息
        out.collect("订单超时！订单创建时间：" + savedTime + "，当前时间：" + timestamp);
        
        // 4. 可选：清理状态
        orderState.clear();
    }
}
```

工作流程总结

1. **准备工作：初始化状态与 TimeService**：`open`方法，调用 `getRuntimeContext().getState()`时，Flink 会创建或恢复状态（从 Checkpoint）。
2. **数据到来**：`processElement`方法处理每一条数据，把需要的信息保存到状态
3. **注册定时**：在处理数据时，根据业务逻辑，通过 `TimerService`向系统注册一个未来时间点的 Timer。`registerEventTimeTimer()`会将 Timer 存入 **优先级队列**（`HeapInternalTimerService`）
4. **时间推进**：Flink 的时间（由 Watermark 推动）不断向前走
5. **Timer 触发检查**：Flink 会不断检查当前时间是否已经达到了某个已注册 Timer 的时间点。当 `Watermark >= Timer 时间`时，Flink 从队列中取出 Timer
6. **Timer 触发回调**：一旦达到，Flink 就会调用 `onTimer`方法，执行预设的逻辑
7. **取消 Timer（可选）**：如果业务逻辑中途完成（比如订单已支付），需调用 `deleteEventTimeTimer`取消 Timer，否则会无效触发，`deleteEventTimeTimer()`会从优先级队列中移除对应 Timer

## 注意事项
+ **Timer 是和 Key 绑定的**：Timer 是 **Keyed** 级别的。每个 Timer 都关联在一个当前的 Key（例如 `orderId`）上，不同 Key 的 Timer 相互隔离。当 Timer 触发时，`onTimer`方法中的上下文可以访问到这个 Key。这确保了为不同订单设置的闹钟不会互相干扰。
+ **时间类型**：Timer 分为 **事件时间定时器** (`registerEventTimeTimer`) 和 **处理时间定时器** (`registerProcessingTimeTimer`)。前者由 Watermark 触发（用于处理乱序事件），后者由系统挂钟时间触发（类似于简单的延时任务）。
+ **状态是核心，必须提前保存**：因为 `onTimer`触发时可能已经没有新的数据了，所以必须初始化状态（在`open`方法中），把需要的信息（比如订单创建时间）在 `processElement`中保存到**状态（State）** 里，才能在 `onTimer`中取出并使用。
+ `onTimer`提供了基于时间的上下文，让你可以访问状态，但它不能直接访问当前正在处理的数据流。
+ **性能**：Flink 底层使用 **时间轮** 等高效数据结构来管理海量的 Timer，所以性能很高。
+ **Timer 的持久化**：Timer 会被 Flink 的 **Checkpoint 机制**持久化，任务失败恢复时能重新触发。避免为每个 Key 注册大量 Timer（可通过 `State TTL`自动清理）。
+ `onTimer`的调用是**同步的**，如果其中包含耗时操作，会阻塞后续 Timer 的处理！耗时操作不要在 `onTimer`中同步执行（可异步输出）。
+ 如果不再需要 Timer，务必调用 `deleteProcessingTimeTimer()`避免内存泄漏。

# EventTimer/ProcessTimer
## 事件时间定时器
使用事件时间定时器

事件事件定时器：EventTime Timer  
需继承：`KeyedProcessFunction<String, OrderEvent, String>`类

### 注册 EventTimer
+ ****`InternalTimerService`****：管理 Timer 的核心接口。
+ ****`HeapInternalTimerService`****：默认的基于堆的 Timer 存储实现（使用 **优先级队列** 按时间排序）。

```java
// flink-streaming-java 源码中的关键注册逻辑 
public void registerEventTimeTimer(N namespace, long time) {     
    // 将Timer放入优先级队列     
    eventTimeTimersQueue.add(new TimerHeapInternalTimer<>(time, keyContext.getCurrentKey(), namespace)); 
}
```

`registerEventTimeTimer`方法：

+ 注册的是基于事件时间的 Timer
+ 例如：`registerEventTimeTimer(event.timestamp + 3600000L)`，即 1 小时后触发，基于 EventTime

### 触发计算
使用 Timer 触发计算的时机源码

+ `InternalTimerServiceImpl` 类
+ `advanceWatermark`方法
+ 主要逻辑在`tryAdvanceWatermark`方法中

Watermark 推进：

+ Flink 需要 Watermark 来判断“当前 EventTime 是否已经推进到了 Timer 的设置时间”。（Watermark = “当前已处理的事件时间 - 最大允许延迟”）
+ 后续的数据，携带的 EventTime 逐渐增大
+ Watermark 更新为 `当前最大EventTime - 延迟阈值`（例如 `12:45 - 5min = 12:40`）。

Timer 触发

+ 当 Watermark ≥ Timer 时间（即 `Watermark >= 13:00`），Flink 触发 `onTimer`。
+ **注意**：如果数据延迟导致 Watermark 迟迟不推进，Timer 不会触发！
+ ****`Watermark`**推进时**，检查是否有 Timer 需要触发：

```java
// 在 StreamInputProcessor 中处理 Watermark 
if (watermark.getTimestamp() > currentWatermark) {     
    // 推进时间，触发Timer     
    timerService.advanceWatermark(watermark); 
}
```

+ ****`onTimer`**调用栈**：

```java
HeapInternalTimerService#advanceWatermark    
    → Triggerable#onTimer      
        → KeyedProcessFunction#onTimer
```

## 处理时间定时器
需继承`KeyedProcessFunction<String, OrderEvent, String>`类

**场景：每5秒输出一次当前处理的订单数**

1. 初始化状态，重写`open`方法：初始化订单计数
2. 处理数据，更新状态，重写`processElement`方法，方法内包含
    - 添加新的状态，`update`方法：更新订单计数
    - 注册 ProcessingTime Timer：
        * `long triggerTime = ctx.timerService().registerProcessingTimeTimer(triggerTime) + 5000L`
        * `ctx`为上下文
        * `triggerTime`：基于当前系统时间计算触发时间，5 秒后触发
3. 触发 Timer：当系统时间到达 `triggerTime`时，Flink 自动调用 `onTimer`
    - 重写`onTimer`方法

核心方法

| 方法名 | 作用 |
| --- | --- |
| `timerService().currentProcessingTime()` | 获取当前系统时间（毫秒）。 |
| `registerProcessingTimeTimer(long)` | 注册一个 ProcessingTime Timer。 |
| `deleteProcessingTimeTimer(long)` | 取消已注册的 Timer（需传入相同的触发时间）。 |
| `onTimer()` | Timer 触发时的回调逻辑。 |
| `registerProcessingTimeTimer()` 方法 |  |


+ 注册一个`ProcessingTime`定时器
+ 每当一个新的定时器被加入到`processingTimeTimersQueue`这个优先级队列中时，如果新来的 Timer 时间戳更小，那么更小的这个 Timer 会被重新注册`ScheduledThreadPoolExecutor`定时执行器上。
