# <font style="color:rgb(15, 17, 21);">Flink 中的时间</font>
Flink 支持 **处理时间**、**事件时间**、~~摄入时间~~（已废弃） 三种时间语义。

在生产环境的使用频次：事件时间（SQL 常用） > 处理时间（SQL 几乎不用，DataStream 少用） > 摄入时间（不用）

## Flink 三种时间属性
1. 事件时间：生产环境中用的最多
    - 数据本身携带的时间，事件产生的时间，且 Flink SQL 触发计算时，也使用数据本身携带的时间，这才是事件时间
    - 如果只是数据携带了时间，Flink 也消费了这个数据，但是在 Flink 中没有使用数据的这个时间作为计算的触发条件，也不能把这个 Flink 任务叫做事件时间的任务。
2. 处理时间：具体算子计算数据执行时的机器时间（例如在算子中 Java 取 `System.currentTimeMillis()`) ）
3. ~~摄入时间：指的是数据从数据源进入 Flink 的时间。摄入时间已废弃。~~

## Flink 时间属性的应用场景
+ 场景一（最常见）：包含时间窗口的计算：
    - 用于标识任务的时间进度，来判断是否需要触发窗口的计算
    - 如：滚动窗口、滑动窗口等
+ 场景二：自定义时间语义的计算
    - 例：用户自定义每隔 10s 的本地时间，或者消费到的数据的时间戳每增大 10s，就把计算结果输出一次，时间在此类应用中也是一种标识任务进度的作用

## FlinkSQL 指定时间属性的两种方式
如果需要使用时间属性进行计算，FlinkSQL 需要

1. 先声明：在数据源表上声明时间属性
2. 后使用：支持时间相关的操作

## <font style="color:rgb(15, 17, 21);">示例：事件时间</font>
```sql
CREATE TABLE user_actions (
  user_name STRING,
  data STRING,
  user_action_time TIMESTAMP(3),
  -- 使用下面这句来将 user_action_time 声明为事件时间，
  -- 并且声明 watermark 的生成规则，即 user_action_time 减 5 秒
  WATERMARK FOR user_action_time AS user_action_time - INTERVAL '5' SECOND
) WITH (
  ...
);

SELECT TUMBLE_START(user_action_time, INTERVAL '10' MINUTE), COUNT(DISTINCT user_name)
FROM user_actions
-- 然后就可以在窗口算子中使用 user_action_time
GROUP BY TUMBLE(user_action_time, INTERVAL '10' MINUTE);

-- 检查时间属性是否生效
SELECT 
    user_name,
    user_action_time,
    -- 检查时间属性是否生效
    CURRENT_WATERMARK(user_action_time) AS watermark 
FROM user_actions;
```

事件时间需要通过显式定义 Watermark 来声明，包括 2 条定义语句：

1. 声明事件事件字段：
    - 从数据中提取的时间戳
    - 字段类型必须是 `TIMESTAMP(3)`或 `TIMESTAMP_LTZ(3)`（毫秒精度）
2. 定义 Watermark：
    - 使用声明的事件时间和最大延迟时间，定义 Watermark 生成规则
    - `WATERMARK FOR 事件时间字段名 AS ...`



注意：如果事件事件类型是**字符串或长整型（如 Unix 毫秒时间戳**），需先转换

```sql
 CREATE TABLE user_actions (
  user_name STRING,
  data STRING,
  -- 1. 这个 ts 就是常见的毫秒级别时间戳
  ts BIGINT,
  -- 2. 将毫秒时间戳转换成 TIMESTAMP_LTZ 类型
  time_ltz AS TO_TIMESTAMP_LTZ(ts, 3),
  -- 3. 然后声明
  WATERMARK FOR time_ltz AS time_ltz - INTERVAL '5' SECOND
) WITH (
  ...
);
```

## <font style="color:rgb(15, 17, 21);">示例：处理时间</font>
```sql
CREATE TABLE user_actions (
  user_name STRING,
  data STRING,
  -- 使用下面这句来将 user_action_time 声明为处理时间
  user_action_time AS PROCTIME()
) WITH (
  ...
);

SELECT TUMBLE_START(user_action_time, INTERVAL '10' MINUTE), COUNT(DISTINCT user_name)
FROM user_actions
-- 然后就可以在窗口算子中使用 user_action_time
GROUP BY TUMBLE(user_action_time, INTERVAL '10' MINUTE);
```

处理时间无需数据中的时间戳，由`PROCTIME()`系统函数自动生成，类型`TIMESTAMP_LTZ(3)`

+ 不会存储在数据源中，每次计算时实时生成。
+ 不能作为表的`PRIMARY KEY`，不能用在`WHERE`、`JOIN`条件（因为每次查询可能不同）。
+ 同一个算子的所有并行实例会基于各自的系统时钟生成时间戳，因此**不同并行实例的时间可能有微小偏差**
+ 在写入外部系统时，处理时间列**不会被输出**（因为它是运行时生成的元数据）
+ 注意：**不可用于有状态操作**
    - `PROCTIME()`**不能**用于需要状态回溯的操作（如 `INTERVAL JOIN`、`MATCH_RECOGNIZE`），因为这些操作依赖事件时间语义。
    - 尝试使用会报错：`Processing time is not supported in [操作名]`

# Flink SQL 中的时区
## 时区问题概述
核心原则：Flink 内部统一使用 UTC（0 时区）

+ Flink 所有时间相关的内部计算（包括 Watermark、窗口触发、状态 TTL）都基于 UTC 毫秒时间戳（long 类型，0 时区）。
+ 但为了满足业务需求（如“每天 0 点统计”），Flink 提供了【上层时区抽象】，让用户可以按照本地时区定义逻辑，而底层仍保持 UTC 一致性。



Hive 中的时区：底层已经转换为当前时区，无需特殊处理

## 两种时间类型：`TIMESTAMP` vs `TIMESTAMP_LTZ`
| 特性 | `TIMESTAMP(p)` | `TIMESTAMP_LTZ(p)` |
| --- | --- | --- |
| 全称 | Timestamp without time zone | Timestamp with local time zone |
| 存储内容 | “本地时间字面值”   （无时区信息） | “带时区的时间”   （隐含时区上下文） |
| 内部表示 | 直接转为 UTC 毫秒 | 根据 `table.local-time-zone` 转为 UTC 毫秒 |
| 是否受 `table.local-time-zone` 影响 | ❌ 否 | ✅ 是 |
| 推荐使用场景 | 已知是 UTC 时间的数据 | 带本地时区语义的数据（如日志时间） |


## 时区问题 bug 根源
假设输入字符串：`"2024-06-15 10:00:00"`

| 配置 | `TIMESTAMP` 解释 | `TIMESTAMP_LTZ` 解释 |
| --- | --- | --- |
| `table.local-time-zone = UTC` | UTC 10:00    → 毫秒 A | UTC 10:00    → 毫秒 A |
| `table.local-time-zone = Asia/Shanghai` | 仍当作 UTC 10:00    → 毫秒 A | 当作 Shanghai 10:00 (UTC+8)    → 毫秒 B = A - 8h |


**绝大多数时区 bug 的根源：**

**用 **`**TIMESTAMP**`** 存储本地时间，但 Flink 当作 UTC 处理，导致窗口偏移！**

## 核心配置：`table.local-time-zone`
这是 Flink SQL 的时区配置参数（Flink 1.13+ 引入）

****

配置方式：3 种

+ 方式 1：`flink-conf.yaml`（全局）
    - `table.local-time-zone: Asia/Shanghai`
+ 方式 2：SQL 客户端（会话级）
    - `SET 'table.local-time-zone' = 'Asia/Shanghai';`
+ 方式 3：`TableEnvironment（Java/Scala）`
    - `tableEnv.getConfig().setLocalTimeZone(ZoneId.of("Asia/Shanghai"));`



`table.local-time-zone` 不改变底层 UTC 计算，只影响时间值的【解释和格式化】。

影响范围：以下函数受时区设置影响

+ `CURRENT_TIMESTAMP`：返回 `TIMESTAMP_LTZ`
+ `PROCTIME()`：返回带时区的时间
+ `DATE_FORMAT(ts, 'yyyy-MM-dd')`：若 `ts` 是 `TIMESTAMP_LTZ`，按本地时区格式化
+ `TIMESTAMP_LTZ` 字段解析
+ `EXTRACT(HOUR FROM ts)`：若 `ts` 是 `TIMESTAMP_LTZ`，返回本地小时
+ 状态 TTL（基于 Processing Time）



以下不受时区设置影响

+ `TIMESTAMP` 字段解析
+ `LOCALTIMESTAMP`：返回 `TIMESTAMP`，不受时区影响（始终 UTC）
+ Watermark 内部计算：仍用 UTC 毫秒

## 各类时间操作的时区行为
### 事件时间（Event Time） + Watermark
时区不影响 Watermark 底层计算，但影响窗口边界对齐

生产中推荐

+ 设置当前时区
+ 将事件时间明确转为`TIMESTAMP_LTZ`类型
+ Flink 自动将事件时间转为 UTC 毫秒（0 时区），Watermark 基于此 UTC 毫秒生成
+ 窗口将按照本地时区对齐

```plsql
SET 'table.local-time-zone' = 'Asia/Shanghai';

CREATE TABLE events (
  id STRING,
  -- 假设原始数据是 "2024-06-15 10:00:00"（北京时间）
  event_time_str STRING,
  -- 转为 TIMESTAMP_LTZ
  ts AS TO_TIMESTAMP_LTZ(event_time_str, 0),
  WATERMARK FOR ts AS ts - INTERVAL '5' SECOND
) WITH ( ... );
```

### 处理时间（Processing Time）
```plsql
SET 'table.local-time-zone' = 'Asia/Shanghai';

CREATE TABLE t (proc_time AS PROCTIME());
SELECT proc_time FROM t;
```

+ 设置当前时区
+ `PROCTIME()`返回 `TIMESTAMP_LTZ` 类型

## 窗口与时区
窗口（`TUMBLE` / `HOP` / `SESSION`）：窗口查询天然支持本地时区

+ 窗口划分边界：按` table.local-time-zone `对齐
    - 例如：`TUMBLE(..., INTERVAL '1' DAY)` 在 Shanghai 时区 = 每天 00:00 ~ 24:00（北京时间）
+ 窗口标签（`window_start`/`window_end`）：显示为本地时间
+ 底层触发逻辑：仍基于 UTC 毫秒（Watermark > `window_end`）



用户写“本地逻辑”，系统保“UTC 一致”。

## 使用建议
简而言之：用 `TIMESTAMP_LTZ` + `table.local-time-zone=Asia/Shanghai`

既能享受 UTC 的一致性，又能按业务时区划分窗口。



1. 统一时区标准，显式设置时区
    - 在所有作业开始时设置为 `SET 'table.local-time-zone' = 'Asia/Shanghai';`
2. 统一使用 `TIMESTAMP_LTZ`时间类型
    - 除非 100% 确定数据是 UTC，否则一律用 `TIMESTAMP_LTZ`
    - 显式转换：`CAST(ts AS TIMESTAMP_LTZ(3)) `
3. 关注源数据的时区：确保`table.local-time-zone`与源数据时区一致，或预处理转换
    - 源数据时间字符串 → `TO_TIMESTAMP_LTZ`
4. 不能依赖 JVM 时区，不要用 `-Duser.timezone` 控制 Flink SQL 时区，用 `table.local-time-zone`
5. 测试时验证时间值，确认时间是否复合预期：
    - `SELECT ts, DATE_FORMAT(ts, 'yyyy-MM-dd HH:mm:ss') FROM events;`
