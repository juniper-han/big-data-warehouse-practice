## Flink 都有什么状态后端，分别用于什么场景？
状态后端决定了：状态如何存储、访问、进行 Checkpoint 快照的。

回答要点：底层原理，适用场景

<font style="background-color:#E8F7CF;">自 Flink 1.13 版本起，社区对状态后端进行了重构，将“成员状态存储”与“检查点存储”解耦。</font>

目前主要有两大核心实现：

1. HashMapStateBackend (基于堆内存)
    1. 存储机制： 状态以 Java 对象的形式存储在 TaskManager 的 JVM 堆内存（Heap）中
    2. Checkpoint 过程： 在进行 Checkpoint 时，会将状态序列化后传输到<font style="background-color:#E8F7CF;">外部持久化存储（如 HDFS）</font>
    3. 读写性能： 极高。因为是直接的对象引用，没有序列化和反序列化的开销
    4. 限制： 受限于 JVM 堆内存大小，状态过大容易触发 GC 频繁或 OOM
2. EmbeddedRocksDBStateBackend (基于<font style="background-color:#E8F7CF;">内嵌本地磁盘</font>)
    1. 存储机制： 状态存储在 TaskManager <font style="background-color:#E8F7CF;">本地磁盘的 RocksDB 数据库</font>中（LSM 树结构）
    2. Checkpoint 过程： 支持增量快照，仅上传自上次快照以来变化的数据文件（SST文件）
    3. 读写性能： 较快。虽然涉及序列化和磁盘 IO，但 RocksDB 有自己的内存缓存（Block Cache）
    4. 限制： 状态大小仅受限于磁盘容量，而非内存，因此支持 TB 级别 的超大状态。



<font style="background-color:#E8F7CF;">实际生产选择策略，取决于：数据量级，延迟要求，资源预算</font>

+ 小型作业/毫秒级延迟：选 HashMapStateBackend。
+ 中大型作业/生产稳定性：首选 EmbeddedRocksDBStateBackend，并务必开启 增量快照

| 特性 | HashMapStateBackend | EmbeddedRocksDBStateBackend |
| --- | --- | --- |
| 状态规模 | 小状态（GB 级别以下） | 超大状态（TB 级别） |
| 读写延迟 | 极低（内存访问） | 较低（涉及磁盘 IO 和反序列化） |
| 可靠性 | 内存溢出风险较高 | 极高，非常稳定 |
| 增量快照 | 不支持（每次都是全量） | 支持（生产环境标配） |
| 典型场景 | 1. 简单的过滤、聚合<br/>2. 窗口时间极短的任务<br/>3. 对延迟要求在毫秒级的金融业务 | 1. 大状态 Join（双流 Join）<br/>2. 长窗口聚合（如 24 小时或更长）<br/>3. 海量去重（如统计 UV） |




进阶：根据指标选择状态后端，观察以下两个关键指标来做最终决定：

1. CPU 密集型 vs IO 密集型：
    1. 如果任务瓶颈在于序列化/反序列化导致的 CPU 高负载，优先考虑 HashMap
    2. 如果任务瓶颈在于状态太大频繁 OOM，必须切换到 RocksDB。
2. Checkpoint 耗时：
    1. 如果 Checkpoint 时间随着运行时间线性增长，说明全量快照压力太大，此时换用支持增量快照的 RocksDB 能显著提升系统稳定性。



未来趋势：Changelog State Backend（Flink 1.15+ 引入）

+ 核心思想是“存算分离”：将状态的变更（Log）实时异步写入持久化存储（如 Durable Log），而 Checkpoint 只是做一个轻量级的裁决（Cut-off）
+ 价值： 极大地降低了 Checkpoint 的延迟，让状态后端的选择不再在“高性能”和“快速 Checkpoint”之间痛苦权衡。

## RocksDB 为什么快于 HDFS
RocksDB 是本地存储（Local Storage），而 HDFS 是远程分布式存储（Remote Storage）。

它们的速度差异本质上是：本地磁盘 IO 与网络 IO/分布式协议之间的量级差别。

以下是 RocksDB 显著快于 HDFS 的四个核心原因：

存储层级与物理位置 (Locality)

1. RocksDB（内嵌式本地存储）： 它<font style="background-color:#E8F7CF;">直接运行在 TaskManager 节点的本地内存和磁盘（通常是 SSD）</font>上。数据的读写是进程内调用，走的是本地文件系统总线。
2. HDFS（远程分布式存储）： 它是<font style="background-color:#E8F7CF;">跨网络的</font>。每一次写入都需要经过 RPC 调用、数据打包、网络传输，并在 DataNode 之间进行三副本复制（Replication）。

底层数据结构：LSM-Tree vs. Block Storage

1. RocksDB 使用 LSM-Tree (Log-Structured Merge-Tree)：
    1. 写操作： 所有的写请求先进入内存中的 MemTable（跳表结构），然后顺序写入预写日志（WAL）。这种设计将随机写转变为顺序写，极大地发挥了磁盘（尤其是 SSD）的吞吐性能。
    2. 读操作： 结合了 Bloom Filter（布隆过滤器）和 Block Cache（内存缓存），高频访问的数据几乎都在内存中。
2. HDFS 是块存储： 它专门为高吞吐的顺序大文件读写设计。
    1. 对于 Flink 状态这种频繁、细碎的 Key-Value 增删改查，HDFS 的元数据管理（NameNode）和数据寻址开销会变得异常沉重。

数据一致性协议的成本

1. RocksDB： 它是单机引擎，不涉及分布式一致性。数据的可靠性通过 WAL 保证，速度极快。
2. HDFS： 遵循强一致性。写入一个 Block 时，必须等待 Pipeline 上的多个 DataNode 全部确认写入成功。这种网络往返时延（RTT）是 RocksDB 不存在的。

在 Flink 中的协同机制：在 Flink 的 Checkpoint 流程中，它们是这样分工的：

1. 算子计算时： 频繁读写状态（State Access），全部在 RocksDB 完成。利用 RocksDB 的高性能支撑每秒百万级的 KV 操作。
2. 触发 Checkpoint 时： RocksDB 将本地生成的增量 SST 文件异步备份到 HDFS

所以，快慢的真相是： Flink 利用 RocksDB 的低延迟支撑实时计算，利用 HDFS 的高可靠支撑故障恢复



总结

> 如果直接把 HDFS 当作状态后端进行频繁的点查和修改，NameNode 会因为处理海量的微小 IO 请求而崩溃
>
> RocksDB 的存在，本质上是在计算节点和分布式存储之间建立了一层高性能、可持久化的本地索引缓存。”
>

## Flink 的状态有哪些类型？
在 Flink 中，状态分类是从：托管方式、数据所有权两个维度来拆解

1. 托管状态：Flink 管理
    1. 键控状态
    2. 算子状态
2. 原始状态：用户管理



按“托管方式”分类（Managed vs. Raw）

概述：这是最顶层的区分，决定了状态是由 Flink 引擎管理，还是由开发者自己维护。

1. 托管状态 (Managed State)：
    1. 定义： 由 Flink 运行时托管，存储在配置的状态后端（HashMap 或 RocksDB）中。
    2. 特点： Flink 负责自动序列化、故障恢复（Checkpoint）以及在改变并行度时重新分配状态。
    3. 建议： 99% 的业务场景都应该使用托管状态。
2. 原始状态 (Raw State)：
    1. 定义： 算子自己管理的字节数组。Flink 只负责将其存入 Checkpoint，但不理解其内部数据结构。
    2. 缺点： 无法自动实现状态的重分发，序列化需手动完成，维护难度大。



按“数据所有权”分类（重点）

概述：在托管状态内部，根据<font style="background-color:#E8F7CF;">状态是否与特定的 Key 绑定</font>，分为以下两大类：

1. 键控状态 (Keyed State)：最常用
    1. 作用范围： 只能用在 KeyedStream（即执行了 keyBy 之后）的算子中。
    2. 逻辑结构： 可以理解为一个分布式 Map，Key 是 keyBy 指定的字段，Value 是该 Key 对应的状态。
    3. 具体子类型：
        1. `ValueState[T]`： 保存单个值（如计数、上一次的温度）。
        2. `ListState[T]`： 保存一个列表（如最近 10 条订单记录）。
        3. `MapState[K, V]`： 保存键值对（如统计每个城市的用户点击量）。
        4. `ReducingState[T] / AggregatingState[IN, OUT]`： 保存聚合后的结果，添加元素时会自动触发预定义的聚合逻辑。
2. 算子状态 (Operator State)
    1. 作用范围： <font style="background-color:#E8F7CF;">绑定到每一个并行算子实例（Subtask）上</font>，与具体的 Key 无关
    2. 常见场景： 主要用于 Source 或 Sink 的实现。例如 Kafka Source 需要记录每个 Partition 的 Offset
    3. 具体子类型：
        1. `ListState`： 状态以列表形式存放。在并行度改变时，列表会被平均分发或合并。
        2. `UnionListState`： 与 ListState 类似，但在恢复时，每个 Subtask 都会获得全量的状态列表（慎用，适合小规模元数据）
        3. `BroadcastState`： 广播状态。一条流的数据被发送到所有下游 Subtask，常用于配置动态更新或规则引擎



状态类型选型与注意事项：实战经验

1. 状态过期 (TTL)： 对于 KeyedState，一定要考虑过期清理（StateTtlConfig）。如果不设 TTL，状态会随时间无限增长，最终导致存储爆炸或 OOM。
2. 状态后端的影响： 
    1. 如果使用 HashMapStateBackend：状态类型（如 List 还是 Map）对性能影响不大。
    2. 如果使用 RocksDBStateBackend：<font style="background-color:#E8F7CF;">MapState 性能远优于 ListState 存 Map</font>。因为 RocksDB 的 MapState 支持单键查找，而 ListState 必须全量反序列化整个列表。
3. 状态的重分发 (Rescaling)：
    1.  KeyedState 是按 Key Group 重分发的
    2. Operator ListState 是按平均切分重分发的。



总结

> Keyed State 支撑了大部分业务逻辑（聚合、关联），
>
> Operator State 支撑了系统的基础连接（Offset 记录），
>
> 而 Broadcast State 则是处理动态规则的不二之选。
>

## Flink 的大状态是怎么解决的/大状态任务治理？
当状态达到 TB 级别或导致 Checkpoint 频繁超时、OOM 时，<font style="background-color:#E8F7CF;">不能只靠增加资源，必须从架构设计、参数调优、运维监控</font>三个层面进行深度治理。

从以下四个维度给出治理方案

1. 存储层治理：RocksDB + 增量 Checkpoint + 物理隔离本地磁盘
2. 应用层治理：状态 TTL + 选择合适的数据结构存储状态 + 稀疏场景下减少 Key 的粒度
3. 架构层治理：状态预聚合 + 外部存储替代状态存储 + 复杂任务拆分
4. 运行时治理：调参，调大写入 buffer，增大 Compaction 阈值，开启水位线对齐（让快流慢一些）



存储层治理：选型与增量化，解决大状态的物理基础。

1. 切换至 RocksDB 状态后端： 对于 GB 级别以上的状态，必须使用 EmbeddedRocksDBStateBackend。它将状态存储在磁盘上，突破了 JVM 堆内存的限制。
2. 开启增量检查点： 开启后，每次 Checkpoint 只上传变化的 SST 文件。
    1. 效果： 极大地减少了网络 IO 和 Checkpoint 的执行时长。
3. 使用物理隔离的本地磁盘： 确保 TaskManager 的本地磁盘使用 SSD，并与系统盘隔离，避免磁盘 IO 争抢导致 RocksDB 读写毛刺。

应用层治理：瘦身与策略优化，从业务逻辑上减少状态的存储压力。

1. 配置状态 TTL ： 大状态任务通常是因为“只增不减”导致的。必须设置合理的过期时间。
    - 优化点： 开启 `cleanupFullSnapshot` 或 `cleanupIncrementally`（RocksDB 支持在后台 Compaction 时清理过期状态），避免过期数据积压。
2. 选择合适的数据结构： 避免使用 ListState 存储 Map 逻辑
    1. 在 RocksDB 中，MapState 支持单个 Key 的读写，而 ListState 必须整条反序列化。大状态下，MapState 的性能远高于 ListState
    2. 减少 Key 的粒度： 检查 keyBy 的字段。如果 Key 过于稀疏（如包含随机数或毫秒级时间戳），会导致状态空间爆炸。

架构层治理：分而治之，当单个作业无法支撑时，需要从架构上进行拆分。

1. 状态预聚合 (Pre-aggregation)： 在进行 window 或 keyBy 聚合前，先在 map 端进行微批或简单的内存预处理，减少进入状态的数据条数。
2. 外部存储替代： 对于超大规模的维表关联（如百亿级的 ID 映射），不建议存入 Flink State，而是使用 Redis、HBase 或 ClickHouse 进行外部点查，并配合 Async I/O 提高吞吐。
3. 任务拆分： 将一个复杂的算子链路拆分为多个子任务，<font style="background-color:#E8F7CF;">通过消息队列（Kafka）连接</font>。这样每个子任务的状态规模更小，Checkpoint 更容易成功。

运行时治理：参数微调，针对 RocksDB 调整以下底层参数

1. 调整 Write Buffer 大小： 增加 `write_buffer_size` 和 `max_write_buffer_number`，提升写入吞吐。
2. 控制合并压力： 调整 `target_file_size_base`，防止频繁的 Compaction 导致 IO 尖峰。
3. 开启水位线对齐 (Watermark Alignment)： 针对多流 Join 任务，如果一个源太快导致状态积压，开启对齐可以让快的流等一下慢的流，从源头控制状态增长。



专家级治理清单总结：

| 治理维度 | 核心手段 | 预期目标 |
| --- | --- | --- |
| 物理层 | RocksDB + SSD + 增量 Checkpoint | 解决存储上限，降低 CP 延迟 |
| 策略层 | 强制 TTL + 优化数据结构 | 解决状态持续膨胀问题 |
| 架构层 | 异步 IO + 预聚合 | 解决单点状态压力过大 |
| 稳定性 | 非对齐 Checkpoint + 资源隔离 | 保证反压下的容错能力 |




业务场景

> 在治理一个 5TB 规模的状态任务时，我发现单纯调大内存没用。
>
> 通过分析 UI 发现，RocksDB 的后台合并（Compaction） 占用了大量 IO。
>
> 我通过自定义 RocksDBOptionsFactory 限制了后台线程数，并调大了 block_cache。
>
> 同时，配合 Changelog State Backend，我们将 Checkpoint 的 P99 耗时从 10 分钟压低到了 30 秒以内，极大地增强了系统的稳定性。
>
