## 数据质量保障
全流程的数据质量保障机制：事前预防、事中控制、事后发现、持续改进

1. 设计阶段，事前预防：设计阶段严格遵守开发规范，幂等性设计，数据探查
2. 开发阶段，事中控制：Code Review，重点关注逻辑正确性、SQL 性能、异常处理等；自测
3. 交付阶段，事后发现：DQC 监控
4. 运维阶段：持续改进：
    1. 问题复盘：产出RCA（根因分析），并转化为新的DQC规则或改进项
    2. 质量看板：展示各任务的质量评分、SLA达成率、监控覆盖度,让质量“可量化、可追踪”



闭环处理机制：发现了数据出错，然后

1. 熔断机制：停止下游，人工接管
2. 根因分析：判断是业务系统变更还是 ETL Bug
3. 修复重刷：确认修复后，利用调度系统重跑
4. 复盘沉淀

## DQC 设计/数据质量监控如何做
DQC（Data Quality Center）：一套体系化监控设计，分为 4 个模块

1. 规则引擎模块（Rule Engine）
    1. 内置通用规则模板：表行数、表大小、空值、重复值、枚举值合法性、唯一值、最小/最大/平均等
    2. 业务逻辑校验：指标校验，字段值校验，比如下单金额不能为负数
    3. 校验逻辑包括：与固定值比较，波动值比较，动态阈值
2. 执行与集成模块（Runtime）
    1. DQC 任务必须挂载再调度系统的算子节点上
    2. 强一致性校验：阻断下游任务
    3. 弱一致性校验：仅告警
3. 监控告警模块（Alerting）
    1. 多渠道，分级
4. 质量评分表（DashBoard）：每日产出质量看板



> 对数据质量监控的核心思路是：
>
> “分层设计精准覆盖，四类规则全面校验，动态阈值减少误报，分级告警避免疲劳，持续迭代不断进化。”
>
> + 分层上：ODS关注完整性,DWD关注准确性,DWS关注逻辑一致性,ADS关注业务可用性
> + 规则上：完整性、准确性、一致性、及时性四类规则,覆盖质量的所有维度
> + 阈值上：动态基线为主,固定阈值为辅,自适应业务变化
> + 告警上：P0电话、P1群消息、P2静默、P3记录,精准不扰民
> + 迭代上：问题驱动新增规则,定期优化阈值和规则健康度
>



规则类型：覆盖完整性、准确性、一致性、及时性维度

1. 完整性规则：保障数据不缺，任务结束后触发，对比源和目标
2. 准确性规则：保障数据不错，基于历史基线动态计算阈值，或业务方提供固定阈值
3. 一致性规则：保障数据对的上，跨表/跨任务对账，通常在 DWS 产出后执行
4. 及时性规则：保障数据不迟到，任务调度系统采集完成时间,与SLA阈值对比

## 数据质量出现问题的处理方案/SOP 流程
SOP：标准化工作流程

分为四个阶段：

1. 发现与响应：快速响应，评估等级，第一时间止血，暂停下游，透明通报，把影响控制在最小范围
2. 排查与定位：从日志、上游、代码、数据、资源五个维度逐层排查，找到根本原因
3. 修复与验证：制定稳妥方案，经过验证后上线，确保“修好一个,不带坏一片”
4. 复盘与沉淀：组织复盘、落地改进、知识沉淀、持续优化



1️⃣ 发现与响应：第一时间止血，暂停下游，透明通报，把影响控制在最小范围

1. 第一时间响应，初步判断影响范围
2. 快速止血（阻断或不阻断仅通知）：同时发布故障公告给下游（企业群），包括问题描述、影响范围、当前状态、预计恢复时间



2️⃣ 排查与定位：从日志、上游、代码、数据、资源五个维度逐层排查，找到根本原因

1. 任务日志：失败原因
2. 检查上游，看异常是否来自上游：上游是否正常产出、数据是否异常、源表数据突增突减、Schema变更等
3. 代码变更：是否有代码上线、口径变更、参数调整？
4. 资源环境：计算资源是否充足



3️⃣ 修复与验证：制定稳妥方案，经过验证后上线，确保“修好一个,不带坏一片”。

1. 制定数据修复方案
    1. 代码问题：修复代码，CR 后重新上线
    2. 上游问题：上游修复后，数据重跑
    3. 资源问题：申请扩容，或按优先级分别重跑，或优化 SQL
    4. 数据倾斜：优化代码，调整并行度
2. 数据修复策略
    1. 增量修复：异常范围可控，重跑对应分区的任务
    2. 全量重跑（口径变更、历史数据错误）：非高峰期执行、提前通知下游“将进行数据修正，期间数据可能波动”
3. 验证确认：验证后再上线修复版本，恢复下游任务
    1. 自验证：抽样核对，确认问题解决
    2. 业务验证
    3. 回归验证：检查其他相关指标是否正常



4️⃣ 复盘与沉淀：组织复盘会，产出改进措施并落地，把事故转化为团队能力的提升

1. 故障复盘会：故障时间线、根因分析、影响评估、改进措施（代码、监控、流程等方面）
2. 知识沉淀：将本次故障的排查过程、根因、解决方案，整理成故障报告，录入团队知识库



🛑 其他注意事项

1. 态度第一：不推诿，不甩锅，先解决问题，再界定责任
2. 透明沟通：让业务方“心里有底”：与业务高频沟通，同步进度，修复完成后明确告知
3. 区分“偶发问题”和“系统漏洞”
    1. 偶发问题：增加监控或自愈机制
    2. 系统漏洞：从根本解决问题
