围绕四大典型故障场景，结合监控指标、日志分析、工具命令、处理方案，整理出一套系统化、可操作的排查与处置手册，适用于 Kafka 2.8+（含 KRaft 模式）。

# 消费滞后（Consumer Lag 高）
现象：`kafka-consumer-groups.sh` 显示 Lag 持续增长，告警触发。

可能因素：下游能力不足 vs 数据倾斜？

## 排查步骤
### 步骤 1：确认 Lag 分布是否均匀
```bash
# 查看每个 Partition 的 Lag
bin/kafka-consumer-groups.sh --bootstrap-server broker:9092 \
  --group my-group --describe
```

输出示例：

```plain
TOPIC    PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
orders   0          10000           15000           5000
orders   1          10000           10010           10    ← 倾斜！
orders   2          10000           10020           20
```

+ <font style="background-color:#E8F7CF;">若 Lag 集中在少数 Partition → 数据倾斜；</font>
+ <font style="background-color:#E8F7CF;">若所有 Partition Lag 均匀增长 → 整体消费能力不足。</font>

### 步骤 2：检查 Consumer 分配情况
```bash
# 查看 Group 成员及分配的 Partition
bin/kafka-consumer-groups.sh --bootstrap-server ... \
  --group my-group --members --verbose
```

+ 确认 Consumer 实例数是否合理（如 4 个 Partition 配 4 个 Consumer）；
+ 若使用 `RangeAssignor`，可能导致前几个 Consumer 承载更多 Partition（倾斜根源）。

### 步骤 3：监控 Consumer 处理性能
+ 关键指标（通过 JMX）：
    - `records-consumed-rate`：消费速率（条/秒）；
    - `records-lag-max`：最大 Lag；
    - `poll-avg-time`：单次 poll 耗时。
+ 若 `records-consumed-rate` 远低于 Producer 发送速率 → 下游处理慢；
+ 若 `poll-avg-time > max.poll.interval.ms / 2` → 可能因处理耗时被踢出 Group。

## 处理方案
数据倾斜的处理方案

1. 改用 `StickyAssignor`或 `CooperativeStickyAssignor`
2. 重新设计消息 Key，使 Hash 更均匀
3. 增加 Partition 数（需重启 Producer）



消费能力不足的处理方案

1. 横向扩容 Consumer 实例
2. 优化业务逻辑（异步处理、批量提交）
3. 调大 `max.poll.records`，减少 `poll` 开销



💡 临时缓解：  
若无法立即扩容，可临时将非核心 Topic 的 Consumer 暂停，释放资源给关键业务。

# 消息积压：如何在不重启任务的情况下动态扩容 Consumer？
目标：快速增加消费能力，追赶 Lag，且不中断现有消费。

## 正确扩容流程（无需重启）
### 步骤 1：启动新的 Consumer 实例
+ 新实例使用相同的 `group.id`；
+ 配置与原实例一致（特别是 `partition.assignment.strategy`）；
+ 无需停止旧实例！

### 步骤 2：触发 Rebalance（自动）
+ 新 Consumer 加入后，GroupCoordinator 会自动触发 Rebalance；
+ 所有成员（包括旧实例）暂停消费（STW），重新分配 Partition；
+ 分配完成后，所有实例（含新）继续消费。

⚠️ 注意：

+ Rebalance 期间会有短暂消费停顿（通常 < 5s）；
+ 不要一次性扩容太多（如从 2 → 20），建议逐步增加（2 → 4 → 8），避免 Rebalance 时间过长。

### 步骤 3：验证扩容效果
```bash
# 观察 Lag 是否下降
watch -n 10 'kafka-consumer-groups.sh --group my-group --describe'

# 检查新实例是否分配到 Partition
kafka-consumer-groups.sh --group my-group --members --verbose
```

## 错误做法
+ 修改 `num.stream.threads`（Kafka Streams）而不重启 → 无效；
+ 仅增加线程数而不增加进程 → 受限于单机资源，效果有限；
+ 手动 `kill` 旧 Consumer 再启动新实例 → 导致双重 Rebalance，延长 STW。

## 高级技巧：配合 Cooperative Rebalance（Kafka ≥ 2.4）
+ 使用 `CooperativeStickyAssignor`：

```properties
partition.assignment.strategy=org.apache.kafka.clients.consumer.CooperativeStickyAssignor
```

+ 优势：Rebalance 时只迁移部分 Partition，其他 Partition 继续消费 → 大幅减少 STW 时间，适合高频扩容场景。

# 网络超时：`Request timed out` 异常排查
现象：Producer/Consumer 日志出现：`org.apache.kafka.common.errors.TimeoutException: Expiring 1 record(s) for topic-0: 30000 ms has passed since batch creation`

## 根因分析矩阵
| 可能原因 | 诊断方法 | 关键指标 |
| --- | --- | --- |
| Broker 负载过高 | 1. 查看 Broker CPU/IO   2. 检查 `RequestQueueSize` | `kafka.network:type=RequestChannel,name=RequestQueueSize`> 100 |
| Full GC 停顿 | 1. 查看 Broker GC 日志   2. 监控 JVM Pause Time | GC Pause > 1s |
| 网络问题 | 1. `ping`/ `telnet`测试连通性   2.  抓包分析 TCP 重传 | `netstat -s | grep retrans` |
| Producer 缓冲区满 | 查看 `buffer-available-bytes` | 接近 0 |
| Consumer 处理慢 | 查看 `max.poll.interval.ms`<br/> 是否超时 | `ConsumerCoordinator`<br/> 日志：`Member ... unsubscribed` |


## 排查流程
### Step 1：定位是 Producer 还是 Consumer 超时
+ Producer 超时：`record-error-rate > 0`；
+ Consumer 超时：频繁 Rebalance 或 `heartbeat-response-time-max` 高。

### Step 2：检查 Broker 端
```bash
# 查看请求队列
jconsole → kafka.network:type=RequestChannel,name=RequestQueueSize

# 查看磁盘 IO
iostat -x 1
# 若 %util > 90% 或 await > 20ms → 磁盘瓶颈
```

### Step 3：**检查 JVM**
+ Broker 启动参数是否合理：`-Xmx8g -Xms8g -XX:+UseG1GC`
+ Full GC 频率是否过高（> 1次/小时）。

### Step 4：**网络层测试**
```bash
# 测试 Broker 端口连通性
telnet broker-host 9092

# 测试带宽
iperf3 -c broker-host
```

## 处理方案
| 场景 | 方案 |
| --- | --- |
| Broker 负载高 | 1. 扩容 Broker   2. 将热点 Partition 迁移到新 Broker（`kafka-reassign-partitions.sh`） |
| GC 问题 | 1. 调优 JVM（增大堆、换 G1GC）   2. 降低单 Broker 承载 Partition 数 |
| 网络抖动 | 1. 联系网络团队排查   2. 增大 `request.timeout.ms`（临时） |
| Producer 缓冲区满 | 1. 增大 `buffer.memory`<br/>2. 优化批处理（调大 `batch.size`） |


# 磁盘写满：日志清理策略配置不当的恢复流程
现象：Broker 所在磁盘使用率 100%，Kafka 进程 crash 或拒绝写入。

## 根因分析
+ Delete 策略：未设置 `retention.ms` 或 `retention.bytes`，导致日志无限增长；
+ Compact 策略：Key 设计不合理（如 UUID），导致无法清理旧值；
+ 监控缺失：未对磁盘使用率设置告警。

## 安全恢复流程（避免数据丢失）
### 步骤 1：**紧急扩容磁盘（首选）**
+ 若使用云盘（如 AWS EBS），在线扩容磁盘；
+ 不停机，最安全。

### 步骤 2：若无法扩容，**临时清理日志（高危！）**
⚠️ 警告：直接删除 `.log` 文件会导致数据丢失和副本不一致！必须通过 Kafka 机制清理。



安全操作：

1. 停止所有 Producer 写入（避免新数据写入）；
2. 修改 Topic 清理策略：

```bash
# 缩短保留时间（例如设为 1 小时）
bin/kafka-configs.sh --bootstrap-server broker:9092 \
  --entity-type topics --entity-name my-topic \
  --alter --add-config retention.ms=3600000
```

3. 等待 Log Cleaner 自动清理：
    - Kafka 后台线程会定期扫描并删除过期 Segment；
    - 可通过日志观察：`[LogCleaner] Deleted log segment ...`
4. 监控磁盘使用率下降；
5. 恢复 Producer 写入；
6. 长期优化：
    - 设置合理的 `retention.ms`（如 7 天）；
    - 对 compact Topic，确保 Key 有重复（如 user_id）；
    - 部署磁盘使用率告警（> 80% 告警，> 85% 严重告警）。

### 步骤 3：极端情况（磁盘已满，Kafka 无法启动）
1. 挂载新磁盘；
2. 修改 `log.dirs` 配置指向新路径；
3. 启动 Kafka；
4. 使用 `kafka-reassign-partitions.sh` 将数据迁移到新磁盘；
5. 清理旧磁盘数据。

# 通用排查工具箱
| 工具 | 用途 |
| --- | --- |
| `kafka-consumer-groups.sh` | 查 Lag、成员、分配 |
| `kafka-topics.sh --describe` | 查 Partition 分布、ISR |
| `kafka-configs.sh` | 动态修改 Topic/Broker 配置 |
| `kafka-reassign-partitions.sh` | 迁移 Partition 到新 Broker |
| JConsole / Prometheus | 监控 JMX 指标 |
| `iostat`/ `vmstat` / `netstat` | OS 层资源诊断 |
| GC 日志分析（GCEasy） | JVM 性能分析 |


# 预防性最佳实践
1. 监控全覆盖：
    - 磁盘使用率、UnderReplicatedPartitions、Lag、Error Rate；
2. 配置标准化：
    - 所有 Topic 必须设置 `retention.ms`；
    - Consumer 必须开启 `enable.auto.commit=false` + 手动提交；
3. 容量规划：
    - 单 Broker Partition 数 < 2000；
    - 磁盘预留 20% 空间；
4. 演练常态化：
    - 模拟 Consumer 宕机、Broker 故障，验证恢复流程。

# 总结：Kafka 故障处理心法
“先止血，再治病；先保数据，再求性能”

+ 消费滞后 → 分清倾斜 vs 能力，精准扩容； 
+ 消息积压 → 动态加 Consumer，善用 Cooperative Rebalance； 
+ 网络超时 → 从 Broker → JVM → 网络逐层剥茧； 
+ 磁盘写满 → 优先扩容，严禁直接删文件！
