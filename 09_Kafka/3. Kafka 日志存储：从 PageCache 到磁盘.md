Kafka的存储设计是：

+ 分层（Topic-Partition-Segment）的物理隔离
+ 稀疏索引的空间换时间
+ PageCache的零拷贝思想

三者结合造就了"磁盘介质上的内存系统"。

# 存储结构：Topic → Partition → Segment 的层级关系
## 物理存储的层级架构
1. 逻辑层：Topic（主题），分成多个分区（Partition）
2. 逻辑层：Partition（分区），切分成<font style="background-color:#E8F7CF;">多个段</font>（Segment）
3. 物理层：Segment（段），由三种文件组成
    - `.log`：数据文件
    - `.index`：偏移量索引
    - `.timeindex`：时间戳索引

## 目录结构
假设我们有一个 Topic 叫"`orders`"，3 个分区，在磁盘上的样子：

+ Topic：根目录；
+ Partition：子目录
+ Segment：由 3 个文件组成的一组文件

```bash
/kafka-logs/
├── orders-0/                 # Partition 0
│   ├── 00000000000000000000.log      # 第一个segment，从offset 0开始
│   ├── 00000000000000000000.index    # 对应的偏移量索引
│   ├── 00000000000000000000.timeindex # 对应的时间戳索引
│   ├── 00000000000000000500.log      # 第二个segment，从offset 500开始
│   ├── 00000000000000000500.index
│   ├── 00000000000000000500.timeindex
│   └── leader-epoch-checkpoint        # Leader Epoch检查点
├── orders-1/                 # Partition 1
│   └── ...
└── orders-2/                 # Partition 2
    └── ...
```

## Segment
一个 Segment 包含三个同名、但后缀不同的文件，分别是

1. `.log`文件：数据文件
2. `.index`文件：索引文件，内容是：相对 offset → 物理位置
3. `.timeindex`文件：时间戳索引文件，内容是 时间戳 → 相对offset

### Segment 的命名规则
Segment 文件名是<font style="background-color:#E8F7CF;">该 segment 第一条消息的 offset</font>，用 20 位数字左补零：

+ `00000000000000000000.log`：offset 从0开始
+ `00000000000000000500.log`：offset 从500开始
+ `00000000000000001000.log`：offset 从1000开始

为什么是 20 位？

+ 因为Kafka的 offset 是 64 位整数，最大值是 9.2×10^18，20 位十进制数足够覆盖。

### Segment 的切分条件
一个 Segment 文件不会无限增长，由三个配置控制：

1. 按大小切分（默认 1GB）：`log.segment.bytes=1073741824`
2. 按时间切分（默认 7 天）：`log.roll.hours=168`
3. 按消息时间戳与当前时间的差值（默认 24 小时）：`log.roll.ms=86400000`

<font style="background-color:#E8F7CF;">切分策略：满足任一条件就滚动新 Segment</font>

### 为什么要设计 Segment？
1. 便于清理：删除过期数据时，直接删除整个旧 Segment 文件
2. 便于查找：通过二分查找先定位 Segment，再查索引
3. 便于恢复：单个 Segment 损坏不影响其他 Segment
4. 控制文件大小：避免单个文件过大，影响 IO 性能

# 索引机制：稀疏索引的巧妙设计
## 为什么需要索引？
如果没有索引，要查找 `offset=1234 `的消息：

+ 最坏情况：遍历整个 Partition 的所有日志文件
+ 即使有 Segment，也要在`.log`文件里从头扫描
+ 磁盘 IO 巨大，延迟不可接受

## 稀疏索引的原理
Kafka 的索引文件（`.index`）采用<font style="background-color:#E8F7CF;">稀疏索引：</font>

核心设计：

+ 不记录每条消息的位置，只记录<font style="background-color:#E8F7CF;">每隔若干条消息的</font>索引项
+ 默认每写入 4KB 消息数据，添加一条索引（由`log.index.interval.bytes`控制）
+ 索引项格式：`[相对offset] -> [物理位置]`

```plain
.index文件内容（二进制格式，这是示意）：
-------------------------------------
相对offset: 0    → 物理位置: 0
相对offset: 50   → 物理位置: 40960  # 第50条消息在文件的40960字节处
相对offset: 100  → 物理位置: 81920
相对offset: 150  → 物理位置: 122880
-------------------------------------
```

什么是"相对offset"？

+ 绝对 offset：全局唯一的 offset
+ 相对 offset：相对于 Segment 起始 offset 的差值
+ 例如 Segment 从 offset=500 开始，绝对 offset=550 的消息，相对 offset=50

## 通过 offset 查找消息的完整流程
假设要查找绝对 offset=1234 的消息：

Step 1：定位Segment（二分查找文件名）

```plain
Segment列表：
00000000000000000000.log (0-499)
00000000000000000500.log (500-999)
00000000000000001000.log (1000-1499) ← 目标在这里
00000000000000001500.log (1500-1999)
```

Step 2：计算相对 offset

```plain
绝对offset = 1234
Segment起始offset = 1000
相对offset = 1234 - 1000 = 234
```

Step 3：二分查找索引文件

加载`00000000000000001000.index`，内容示意：

```plain
相对offset: 0   → 物理位置: 0
相对offset: 50  → 物理位置: 40960
相对offset: 100 → 物理位置: 81920
相对offset: 150 → 物理位置: 122880
相对offset: 200 → 物理位置: 163840
相对offset: 250 → 物理位置: 204800  ← 这个比234大
```

二分查找找到小于等于234的最大索引项：

    - 索引项：相对offset=200，物理位置=163840
    - 目标offset=234，介于200和250之间

Step 4：顺序扫描`.log`文件

+ 从`.log`文件的 163840 字节处开始读取
+ 顺序扫描直到找到 `offset=234` 的消息
+ 因为消息是顺序存储的，扫描范围很小（最多4KB数据）



时间复杂度：

+ 定位 Segment：`O(logN)`，N 是 Segment数
+ 查索引：`O(logM)` ，M 是索引项数
+ 扫描日志：`O(1)` 因为范围有限

## 通过时间戳查找消息
`.timeindex`文件的格式：时间戳 → 相对 offset

```plain
时间戳: 相对offset
1612345678900: 100    # 2021-02-03 12:34:58 的消息在相对offset=100
1612345678950: 150
1612345679000: 200
```



查找流程（按时间戳`t=1612345678980`）：

1. 在所有 Segment 中，找到时间戳范围覆盖 t 的 Segment
2. 在`.timeindex`中二分查找，找到小于等于 t 的最大时间戳对应的 offset
3. 然后用这个 offset 去`.index`找物理位置
4. 扫描`.log`文件找到精确消息



为什么需要独立的.timeindex？

+ offset 是单调递增的，但时间戳不是（可能乱序）
+ 如果生产者发送的消息时间戳不是单调的，直接用时间戳索引文件可以处理

## 索引文件的维护
```java
// 简化的索引写入逻辑
class OffsetIndex {
    def append(offset: Long, position: Int): Unit = {
        // 检查是否需要写入索引
        if (position - lastIndexedPosition > indexIntervalBytes) {
            // 写入相对offset和物理位置
            mmap.putInt((offset - baseOffset).toInt)
            mmap.putInt(position)
            lastIndexedPosition = position
        }
    }
}
```

内存映射文件：

+ 索引文件使用`java.nio.MappedByteBuffer`（内存映射）
+ 读写索引不经过系统调用，直接在内存中操作
+ 由操作系统负责刷盘

# 刷盘机制：PageCache的巧妙利用
## Kafka不直接写磁盘
误区：Kafka 为了性能做了很多磁盘优化

实际：Kafka最大的性能秘诀是“几乎不写磁盘！”



核心设计：

1. Kafka写入数据时，只写入操作系统的 PageCache（页缓存）
2. 立即返回成功（如果配置了适当的 acks）
3. 由操作系统异步刷盘（fsync）

```plain
【写入路径】
Producer数据 → Socket接收缓冲区 → PageCache → （异步）磁盘
                ↑                    ↑
            立即返回成功         后台刷盘线程
```

## PageCache的工作原理
PageCache是：

+ 操作系统内核管理的【<font style="background-color:#E8F7CF;">内存缓存</font>】
+ 用于：缓存磁盘文件的内容
+ 读写文件时，先操作 PageCache，再异步同步到磁盘



Kafka 利用 PageCache 的方式：

1. 写数据：

```plain
应用程序（Kafka） → write() → PageCache（脏页） → pdflush/flush → 磁盘
                （立即返回）        （后台线程）        （物理写入）
```

2. 读数据：

```plain
应用程序 → read() → PageCache（命中） → 直接返回
              ↓（未命中）
            磁盘读取 → PageCache → 应用程序
```

## Kafka 的刷盘配置
Kafka 刷盘配置参数

1. 强制刷盘到磁盘的频率（毫秒）：`log.flush.interval.ms=null`
2. 消息累积多少条强制刷盘：`log.flush.interval.messages=9223372036854775807`

<font style="background-color:#E8F7CF;">默认配置：完全不主动刷盘，完全依赖操作系统</font>

## 为什么不主动刷盘？
性能对比：

| 操作 | 延迟 | 吞吐 |
| --- | --- | --- |
| 写 PageCache | 纳秒级 | 百万级 TPS |
| 写磁盘（fsync） | 毫秒级（10ms 左右） | 万级 TPS |
| 写磁盘（顺序 IO） | 微秒级 | 十万级 TPS |


Kafka的设计哲学：

+ 顺序 IO 已经很快，但fsync还是慢
+ 如果每条消息都 fsync，TPS 直接掉两个数量级
+ <font style="background-color:#E8F7CF;">依靠副本机制保证可靠性，而不是磁盘刷写</font>

## 刷盘的触发时机
虽然Kafka不主动刷盘，但以下情况会触发物理写入：

1. 操作系统自动触发：
    - PageCache空间不足
    - 脏页比例超过阈值（`vm.dirty_ratio`）
    - 定时刷盘（`vm.dirty_writeback_centisecs`）
2. Kafka主动触发：
    - Consumer 拉取消息时，如果 PageCache 未命中，从磁盘读
    - <font style="background-color:#E8F7CF;">日志段关闭/滚动时</font>
    - Broker 正常关闭时
3. 显式刷盘（如果配置了）：
    - 达到`log.flush.interval.messages`
    - 达到`log.flush.interval.ms`

## PageCache的灾难恢复
问题：如果只写PageCache就返回成功，机器断电怎么办？

答案：确实可能丢数据，所以：

1. `acks=all` + 多副本：数据在多个机器的PageCache里
2. 一台机器断电：其他机器还有数据
3. <font style="background-color:#E8F7CF;">全部断电：概率极低，且可以通过UPS（不间断电源）保障</font>

生产环境最佳实践：

```properties
# 依靠副本保证可靠性，而不是刷盘
replication.factor=3
min.insync.replicas=2
acks=all

# 不主动刷盘
log.flush.interval.messages=Long.MAX_VALUE
log.flush.interval.ms=null

# 让操作系统管理PageCache
vm.dirty_ratio=40
vm.dirty_background_ratio=10
```

## PageCache 的读优化
Kafka 不仅【写】利用 PageCache，读也利用：

读写结合的场景：

1. Producer 写入数据到 PageCache
2. Consumer 立即消费数据
3. 数据直接从 PageCache 读取，完全不需要磁盘 IO
4. 只有 Consumer 落后时，才需要从磁盘读

这就是Kafka能同时支持高吞吐写入和实时消费的秘密。
