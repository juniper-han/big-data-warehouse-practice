# ISR 机制
ISR 机制（In-Sync Replica）是 Kafka 在“<font style="background-color:#E8F7CF;">强一致性</font>”和“<font style="background-color:#E8F7CF;">高可用性</font>”之间做出的一个精妙权衡。

+ 它不像传统的同步复制那样要求所有副本确认后才返回成功（太慢）
+ 也不像异步复制那样完全不管副本是否跟上（数据丢失风险太高）

## 引入 ISR 的必要性与设计原理
Kafka 是一个分布式系统，数据以 Partition 为单位，为了容灾，每个 Partition 会有多个副本（Replica）分布在不同的 Broker上。



问题：当 Producer 发送一条消息时，它需要确认消息“安全”地存下来了。存在以下两种场景

1. 等所有副本都写完，才给 Producer 返回成功：很安全，但如果有慢节点或网络抖动，系统延迟+吞吐量下降
2. 只等 Leader 写完就返回成功，那么 Leader 一旦宕机，未同步到 Follower 的数据就会丢失。



ISR 的解法：Kafka引入了一个<font style="background-color:#E8F7CF;">动态的、可伸缩的“同步副本集合”</font>，也就是ISR。

+ ISR里的成员：是那些与 Leader“保持同步”的副本。



ISR 核心思想：Producer的`acks=all`配置，并不是等所有副本确认，而是<font style="background-color:#E8F7CF;">等待ISR里的所有副本确认</font>。

+ 由于ISR是动态变化的，它永远只包含那些“跟得上”的副本。

ISR 的权衡结果：ISR是一个保证“数据不丢”的最小可用副本集合

+ 正常情况下，ISR 包含了所有副本（比如 3 个），`acks=all`就需要等这 3 个副本都确认，保证了强一致性。
+ 如果某个 Follower 因为网络、GC 等问题“掉队”了，它会被踢出 ISR
    - 这时，Leader只需要等待剩下的ISR成员（比如 2 个）确认即可
    - 系统依然可以正常服务，只是冗余度暂时降低，但数据安全性依然得到了保证
+ 当那个“掉队”的 Follower 恢复并追上进度后，它会被重新加入ISR，系统恢复到完整冗余状态。

## 如何判定一个副本处于ISR中？
判定一个 Follower 副本是否在 ISR中，<font style="background-color:#E8F7CF;">核心依据是它是否“跟得上”Leader 的进度</font>。

如何判断：Follower 跟上上 Leader 的进度，在 Kafka 的设计演进中，有过两种判断标准：

1. 已废弃：基于 Follower 落后 Leader 的数据条数
2. 基于 Follower 与 Leader 通信的时间间隔，如果二者一直能通信，就认为 Follower 跟得上 Leader

### 阶段一：旧版本（0.9.x 及之前）——基于“消息滞后数”
判定条件：Follower 副本落后 Leader 的消息条数是否超过~~ ~~`~~replica.lag.max.messages~~`（已废弃）。

弊端：这个标准在高吞吐场景下非常不稳定。

+ 一旦出现短暂的流量高峰，Follower可能因为网络或磁盘I/O瞬间落后几条消息，就被踢出ISR
+ 导致ISR频繁震荡，影响系统稳定性。

### 阶段二：新版本（0.10.x 至今）——基于“时间滞后”
判定条件：

+ Follower 副本向 Leader 发起拉取请求（Fetch Request）的<font style="background-color:#E8F7CF;">最大时间间隔</font>是否超过 `replica.lag.time.max.ms`（默认值：10000ms，<font style="background-color:#E8F7CF;">即10秒</font>）。



工作机制：

1. <font style="background-color:#E8F7CF;">Follower 会持续向 Leader 发送 Fetch 请求</font>，这个请求有两个作用：
    - 同步 Leader 上的新消息。
    - 向 Leader“报到”，告诉 Leader“我还活着，并且我处理到了哪个 Offset”。
2. <font style="background-color:#E8F7CF;">Leader会为每个Follower维护一个时间戳</font>，记录它上一次收到该 Follower 的 Fetch 请求并成功响应的时间
3. 周期性检查：Leader 的后台线程定期检查：<font style="background-color:#E8F7CF;">当前时间与这个时间戳的差值</font>
4. 判定规则：
    - 如果 `当前时间 - 最后Fetch请求时间 <= replica.lag.time.max.ms`，则认为 Follower 是同步的，保留在ISR中。
    - 如果 `当前时间 - 最后Fetch请求时间 > replica.lag.time.max.ms`，则认为 Follower“失联”或处理能力严重落后，将其<font style="background-color:#E8F7CF;">移出 ISR</font>。



为什么这个设计更优？因为：它有效地将<font style="background-color:#E8F7CF;">短暂的性能波动</font>和<font style="background-color:#E8F7CF;">永久性故障/严重落后</font>区分开来

+ 只要 Follower 还在持续不断地发送 Fetch 请求，即使它每次都只能拉取到少量消息，表明它正在努力追赶
+ Leader 就认为它是活跃且同步的
+ 只有当它<font style="background-color:#E8F7CF;">超过10秒没有进行任何拉取</font>，才判定它出问题了。

## OSR（Out-of-Sync Replica）的处理
当 Follower 副本不满足 ISR 的条件时，它就变成了<font style="background-color:#E8F7CF;">OSR（Out-of-Sync Replica，非同步副本）</font>

### 成为 OSR 的后果
1. 失去成为 Leader 的资格：
    - 在 Leader 选举中，只有 ISR 中的副本才有资格被选为新的Leader
    - OSR 副本中的数据是不完整的，如果让它成为 Leader，会导致数据丢失和 inconsistent
2. 不再参与“写确认”
    - 当 Producer 设置`acks=all`时，Leader 只等待 ISR 集合的确认
    - OSR 副本的存在与否不会影响 Producer 的响应速度。
3. 继续追赶数据：
    - 虽然被踢出了 ISR，但 OSR 副本仍然会继续向 Leader 发送 Fetch 请求，尝试同步数据。

### 从 OSR 恢复到 ISR
当一个 Follower 副本落后不多，或者因为网络抖动短暂离线又恢复后，它会经历以下过程重新回到 ISR：

1. 持续拉取：OSR 副本不断地从 Leader 拉取消息，逐步缩小与 Leader 的差距。
2. 追上HW（High Watermark）：当 Follower 副本的 LEO（Log End Offset）追上 Leader的 HW（High Watermark，高水位）时，它就被认为重新进入了“同步”状态。
3. 重新入 ISR：Leader的检测线程在下一次检查时，发现<font style="background-color:#E8F7CF;">该Follower的“最后Fetch请求时间”已经恢复</font>，并且<font style="background-color:#E8F7CF;">数据差距也符合要求</font>，就会将它重新加入ISR。

### 一个危险的配置：`unclean.leader.election.enable`
`unclean.leader.election.enable`配置：

+ 关于 OSR 处理的重要配置，关系到集群的<font style="background-color:#E8F7CF;">可用性 vs 一致性</font>。
+ 针对的场景：当 Partition 的 Leader 宕机，且 ISR 集合为空（即所有副本都变成了 OSR）时
    - `false`：<font style="background-color:#E8F7CF;">不会选举 OSR 中的副本作为 Leader</font>，而是等待原来的 Leader（如果还有数据的话）恢复上线。这保证了数据一致性，但<font style="background-color:#E8F7CF;">牺牲了短暂的可用性</font>（服务不可写，直到Leader恢复）。
    - `true`：允许从 OSR 中选举出一个副本成为新的 Leader。这意味着集群可以快速恢复可用性，但会丢失数据
+ 默认值：`false`，在主流大厂的生产环境中，通常强制设置为`false`，因为数据一致性通常是第一位的

# HW 与 LEO：高水位与日志末端位移
## 基础概念与作用
LEO（Log End Offset）：日志末端位移，是每个副本的本地日志中，<font style="background-color:#E8F7CF;">下一条（待写入消息）的 Offset</font>

+ 每个副本（Leader和Follower）都维护自己的 LEO
+ Leader 会追踪所有 ISR 副本的 LEO
+ 如果副本已经有 10 条消息（offset 0-9），LEO = 10

HW（High Watermark）：高水位，指的是<font style="background-color:#E8F7CF;">所有 ISR 副本都成功同步到的位置（Offset）</font>

+ Partition 级别的概念，所有副本都维护 HW
+ 消费者只能消费到 HW 之前的消息，HW之后的消息被视为"未完全确认"
+ HW 的作用
    - <font style="color:rgb(15, 17, 21);">保证消息一致性：HW 确保了所有 ISR 副本中，HW 之前的消息都是一致的。</font>
    - 实现故障恢复：Leader 切换时，新 Leader 以 HW 为界截断日志

## ISR、LEO、HW 三者的关系：
1. Producer 发送消息到 Leader，Leader 的 LEO 增加
2. Follower 从 Leader 拉取消息，更新自己的 LEO
3. Leader 会追踪所有 ISR 副本的 LEO
    - 当 Leader 发现所有 ISR 副本的 LEO 都 >= 某个 Offset时，它就会更新这个 Partition 的 HW
    - Leader HW = min(所有ISR副本的LEO)
4. Follower 在下次拉取时，也会更新自己的 HW
5. 只有 LEO 追上（达到或超过）Leader的 HW 的副本，才有可能（并最终）被重新加入ISR
    - 因为这意味着它和 ISR 中的其他副本拥有了一致的数据截止点。

## Leader 切换时的数据丢失/不一致问题（经典场景）
这是Kafka老版本（0.11.0之前）的一个经典问题，新版已经修复。

### 场景：Producer发送消息后，Leader宕机
初始状态（R1是Leader，R2和R3是Follower）：

```plain
R1（Leader）: LEO=5, HW=3  （消息0-3已确认，消息4未确认）
R2（Follower）: LEO=4, HW=3  （同步到消息3，消息4还在同步中）
R3（Follower）: LEO=3, HW=3  （只同步到消息3）
```

发生故障：

+ R1（Leader）宕机
+ Controller从ISR中选新 Leader，假设选了 R2（因为 R2 的 LEO=4，比 R3 高）

问题出现：

+ R2成为新Leader后，将自己的HW=3作为新Partition的HW
+ R2向所有Follower发送指令：截断日志到HW=3
+ R2自己截断日志，删除offset=4的消息
+ R3本来就只有到3，没问题
+ 但关键是：消息4原本存在于R1（已宕机）和可能存在于R2的内存中，现在彻底丢失了

更糟的情况：

+ 如果后来 R1 恢复上线，它会作为 Follower 从新 Leader（R2）同步数据，发现自己的 LEO=5 > 新 Leader的 HW=3，于是它会主动截断自己的日志到 3，彻底删除消息 4 和 5。

结果：消息 4 永久丢失，尽管它曾经被写入过半的副本。

### 问题本质
根源在于 <font style="background-color:#E8F7CF;">HW更新是异步的，且依赖所有副本的LEO信息</font>

+ Leader 虽然收到了消息，但没来得及更新 HW 就宕机
+ Follower 可能落后，导致 Leader 的 HW 无法推进
+ 新 Leader 选举时，依赖 HW 来截断日志，可能导致不该删的数据被删了

这个问题的本质是：HW作为截断依据并不可靠，因为它反映的是“过去某个时刻的一致性状态”，而不是“当前所有副本的真实共识”。

### 新版本的解决方案：Leader Epoch机制

# <font style="color:rgb(15, 17, 21);">Leader Epoch 机制与数据一致性</font>
## 数据一致性
在分布式系统里，一致性指的是：<font style="background-color:#E8F7CF;">所有节点对数据的认知达成统一，而不是保证数据绝对不丢。</font>



一致性 ≠ 不丢数据

+ 一致性是"所有节点对数据状态达成共识"
+ 丢数据是"数据确实存在过但后来没了"



CAP理论中的C（Consistency，一致性）：

+ 强一致性：所有节点在同一时刻看到相同的数据
+ 最终一致性：经过一段时间后，所有节点达成一致



在Kafka的场景里：

+ 我们追求的是所有副本最终达成一致
+ 在达成一致的过程中，<font style="background-color:#E8F7CF;">宁可删掉不确定的数据</font>，也不能让数据出现分歧

## Leader Epoch 核心概念
Leader Epoch是一个版本号，由两部分组成：

1. Epoch：一个单调递增的版本号
    - 每次Leader变更时+1
    - 类似于 Git 的 commit ID或数据库的版本号
2. Start Offset：该 Epoch 开始时，Leader 的第一条消息的 Offset
    - 记录在这个 Epoch 期间，消息从哪个位置开始

## Leader Epoch 的数据结构
每个副本（Replica）都会在本地磁盘维护一个 Leader Epoch 序列文件，格式如下：

```plain
版本号: 起始Offset
0: 0      （第一任Leader，从offset 0开始）
1: 50     （第二任Leader，从offset 50开始）
2: 120    （第三任Leader，从offset 120开始）
3: 200    （第四任Leader，从offset 200开始）
```

这个文件持久化在磁盘上，Broker 重启也不会丢失

## 工作原理
Leader Epoch 机制彻底改变了 Leader 切换时的数据恢复流程：

1. 正常写入时的 Epoch 记录
    - 当R1作为Leader（Epoch=5）时：所有写入的消息都带有隐含的Epoch标记
    - R1记录：Epoch 5 从 offset 1000 开始
2. Leader 切换发生：R1（原 Leader，Epoch=5）宕机
    - Controller 选举 R2 作为新 Leader
    - R2 成为 Leader，Epoch 增加到 6
    - R2 记录：Epoch 6 从 offset 1200 开始
3. 关键恢复流程——Epoch 比对：当原 Leader R1恢复上线，作为 Follower 重新加入集群时：
    1. R1 向新 Leader R2 发送请求
        1. R1请求："我是旧Leader，我的Epoch=5，我最后的offset是1500"
        2. R2响应："你的Epoch太旧了，当前是Epoch=6，你需要在offset=1200 处截断"
    2. R1 截断日志：
        1. R1 检查自己的日志：发现自己有 offset 1200-1500 的数据，但这些数据是在 Epoch=5 时写入的
        2. 新 Leader 说 Epoch=6 从 1200 开始，说明 1200 之后的那些数据（Epoch=5）已经不被集群认可
        3. R1 删除 offset 1200 之后的所有数据
    3. 重新同步：R1 从 R2 拉取 Epoch=6 的新数据，R1 与 R2 保持一致

这样，Leader Epoch 解决了数据不一致问题

+ 集群达成一致：不一致的那部分数据（epoch5：offset 1200-1500），从未被真正提交过
+ <font style="background-color:#E8F7CF;">不会出现“R2 认为数据不存在，R1 认为数据存在的”不一致状态</font>

## Epoch 容错机制
Leader Epoch信息是持久化存储的，这意味着：

+ Broker重启：重启后从磁盘加载Epoch文件，知道自己在哪个Epoch、从哪个offset开始
+ 多副本恢复：所有Follower都保留自己的Epoch视图，方便比对
+ Controller切换：Controller本身也维护集群级别的Epoch信息

# ACKS参数：可靠性权衡的底层逻辑
`acks`是 <font style="background-color:#E8F7CF;">Producer 的一个参数</font>，<font style="background-color:#E8F7CF;">决定了 Leader 在什么情况下向 Producer 返回成功响应</font>。

## `acks=0`：发后即忘
底层逻辑：

+ Producer发送消息后，不等待任何Broker的确认
+ 网络层发出即认为成功

交互流程：

```plain
Producer -> Network（发送）-> 立即返回成功
                    ↓
                  Broker（异步接收，可能失败）
```

可靠性分析：

+ 优点：极致吞吐，延迟最低
+ 缺点：数据丢失风险最高（网络问题、Broker负载高、磁盘满都可能导致丢数据）
+ 适用场景：对数据完整性要求极低的场景；指标监控、日志采集（允许少量丢失）；性能压测

大厂实践：几乎不用，除非是测试环境。

## `acks=1`：Leader 确认
底层逻辑：

+ Leader 收到消息并写入自己的日志（`PageCache`）后，立即返回成功
+ 不等待 Follower 同步

交互流程：

```plain
Producer -> Leader Broker
            Leader写入PageCache
            Leader返回 ACK 给 Producer
            （异步）Follower 从 Leader 拉取数据
```

可靠性分析：

+ 优点：吞吐和延迟的较好平衡（大多数场景的默认选择）
+ 缺点：如果 Leader 在返回ACK后、Follower 拉取前宕机，这条消息就丢失了
+ 适用场景：配合下游有重放机制的消费端
    - 大部分业务日志
    - 允许秒级数据丢失的实时计算

## `acks=all`（或`-1`）：全部ISR确认
底层逻辑：

+ Leader 必须等待所有 ISR 副本都写入成功后，才返回 ACK
+ 配合`min.insync.replicas`控制最小同步副本数

交互流程（假设`min.insync.replicas=2`，ISR 有 3 个副本）：

```plain
Producer -> Leader
            Leader写入
            Follower1同步完成
            Follower2同步完成
            （Follower3慢，还在同步）
            Leader发现ISR中[Leader, F1, F2]都完成了，返回ACK
```

可靠性分析：

+ 优点：最强数据保证，只有ISR全部确认才算成功
+ 缺点：延迟受最慢的ISR副本影响，吞吐可能下降
+ 适用场景：任何"丢数据就是事故"的场景
    - 需要精确一次语义的实时数仓
    - 金融交易、订单数据
