**谓词下推（Predicate Pushdown）** 是数据库和大数据系统中一项极其重要但常被忽视的**查询优化技术**。

它能显著减少 I/O、网络传输和计算开销，是提升查询性能的“隐形加速器”。



一句话：**<font style="background-color:#FBF5CB;">用 Parquet/ORC 存数据，按时间/类别分区，写 SQL 时直接比较列值，谓词下推直接生效</font>**

# 核心思想和技术原理
一句话：把过滤条件（谓词）尽可能下推到数据源附近执行，避免读取和传输无用数据

## 什么是“谓词”（Predicate）？
在 SQL 中，`WHERE`、`JOIN ON`、`HAVING `中的**<font style="background-color:#FBF5CB;">过滤条件</font>**就是谓词。  
例如：

```sql
SELECT name FROM users 
WHERE age > 30 AND city = 'Beijing'; -- 这里有两个谓词
```

# 如何使用谓词下推
## 谓词下推通常不需要手动开启
现代计算引擎（Flink、Spark、Hive、Presto 等）**默认自动优化**，只要满足条件就会下推。

但是需要一些技巧，尽可能的让计算引擎使用谓词下推

## 使用技巧
### 写 SQL 时尽量让谓词“可下推”
1. **可下推的谓词**（推荐）：

```sql
WHERE id = 1001
WHERE event_time >= '2023-01-01'
WHERE status IN ('active', 'pending')
```

2. **不可下推的谓词**（避免）：对过滤字段加函数

```sql
WHERE YEAR(event_time) = 2023      -- 函数包裹列
WHERE CONCAT(first, last) = 'Tom'  -- 表达式操作列
```

💡 **优化技巧**：  
把 `YEAR(event_time) = 2023` 改为  
`event_time >= '2023-01-01' AND event_time < '2024-01-01'`

### 确保数据源支持下推
1. **支持下推的数据源**：
    - **<font style="background-color:#FBF5CB;">列式存储：Parquet、ORC（</font>****<font style="background-color:#FBF5CB;">✅</font>****<font style="background-color:#FBF5CB;"> 强烈推荐）</font>**
    - 数据库：MySQL、PostgreSQL、HBase
    - 消息队列：Kafka（通过元数据过滤，有限支持）
2. **不支持/弱支持**：
    - CSV、JSON（行式存储，需全读）
    - 自定义 Source（需手动实现）

### 验证生效
1. 查看 **执行计划（Explain Plan）**：
2. 如果看到类似：`Pushed Predicates`的词语

 说明谓词已下推

# 与其他优化技术的关系
谓词下推常与以下技术协同工作：

1. 分区裁剪（Partition Pruning）：  
先根据分区字段（如 `dt`）跳过无关文件。
2. 列裁剪（Column Pruning）：  
只读取 SELECT 和 WHERE 中涉及的列。
3. 布隆过滤器（Bloom Filter）：  
在 Parquet/ORC 中快速判断某值是否存在，避免读块。

✅ 三者结合 = 最大化减少 I/O
