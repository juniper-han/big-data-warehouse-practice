## Kafka 是怎么做监控的？
思考维度：基础设施 + Broker 核心指标 + 客户端性能 + 消费积压

组织逻辑：监控工具选择 + 核心指标 + 报警策略



**1️⃣**** 监控工具选择：根据企业的规模和技术栈选择不同的组合：**

1. 企业级全栈方案：Prometheus + Grafana。利用 JMX Exporter 将 Kafka 的 JMX 指标导出，这是目前工业界的标准做法。
2. 轻量化/可视化工具：
    - Kafka UI / CMAK (Kafka Manager)： 用于直观查看 Topic 分区分布、副本状态和简单的消息检索。
    - Kafka Eagle (EFAK)： 特别好用，能直观监控 Offset 积压趋势和集群拓扑。
3. 云原生方案： 如果在云上，会直接使用云厂商提供的监控，如 AWS CloudWatch 或阿里云 CMS。



**2️⃣**** 核心监控指标：延迟 (业务感知)  + 吞吐 (生产消费性能) + HBase 集群稳定性 + HBase 集群性能 + 基础设施**

阈值不能“一刀切”，必须综合考虑：硬件配置（网卡、磁盘类型）+ 业务 SLA + 集群规模  
需要给出一些经过实践检验的经验参考值（Best Practices）

1. 消费积压（业务感知）
    1. Consumer Lag (堆积量)：
        1. 阈值： 通常设置为 单分区 > 10万条 或 总积压 > 100万条（视 QPS 而定）
        2. 动态阈值： 即如果当前积压的数据需要 10 分钟才能处理完，必须介入。
    2. Kafka 处理时延或 End-to-End Latency (端到端时延)：
        1. 阈值： 实时数仓通常要求 < 3s。如果超过 10s，下游看板或分析任务就会感到明显卡顿
2. Broker 性能指标（吞吐量）
    1. BytesIn / BytesOut： 吞吐量带宽。
    2. Request Purgatory Size： 正在等待处理的请求数量。如果积压过多，说明 Broker 的处理能力跟不上请求的速度，且待处理的请求占用内存，可能导致 Broker 的 OOM，这通常由于 CPU 或磁盘瓶颈
    3. JVM GC Time： 频繁 Full GC 会导致 Broker 暂停（STW），直接引发客户端重连
    4. Network Processor Avg Idle (网络线程空闲率)：Broker 处理网络请求的线程有多“闲”
        1. 高空闲率（如 70%-100%）：健康，网络不是瓶颈
        2. 中等空闲率（30%-70%）：可接受，需关注趋势
        3. 低空闲率（0%-30%）：告警，存在网络瓶颈风险，需要调大`num.network.threads`或扩容
    5. Request Handler Avg Idle (I/O线程空闲率)：Broker 处理实际业务逻辑的线程有多"闲"
        1. 高空闲率（如 70%-100%）：健康，处理层不是瓶颈
        2. 中等空闲率（30%-70%）：可接受，需关注趋势和峰值
        3. 低空闲率（0%-30%）：告警，存在处理瓶颈，需要调大`num.io.threads`或扩容
    6. Request Local Time (平均请求处理时间)：
    - 阈值： Produce 请求 > 10ms，Fetch 请求 > 20ms。如果显著上升，通常意味着磁盘写入进入瓶颈
3. 集群稳定性
    1. Under-Replicated Partitions ：副本数不足的分区数量（有副本没有在 ISR 中）
        1. 报警阈值： > 0 (即刻报警)，此时部分副本同步失效，存在丢数风险
        2. 说明副本同步失效，需检查 Broker 宕机、网络故障或磁盘损坏说明
    2. Offline Partitions Count：当前没有 Leader 副本的分区数量，该分区完全不可用（不可读、不可写）
        1. 报警阈值： > 0
        2. 严重故障。部分数据已不可读写，通常伴随 Broker 宕机
    3. Active Controller Count： 整个集群必须且只能为 1，不为 1 时报警
        1. 0 意味着：集群无法从 Broker 中选举出 Controller
        2. 大于 1：多个 Broker 认为自己是 Controller，出现"脑裂"现象，元数据可能不一致
        3. 报警需检查 Controller 节点的 JVM 负载和 ZK/KRaft 连接
    4. IsrShrinks / IsrExpands： 副本被踢出/加入 ISR 的频率
        1. 报警阈值：> 5 次/分钟
        2. ISR 频繁收缩扩展，说明 Broker 压力过大导致心跳超时，需排查 I/O 、GC、磁盘情况
4. 资源与系统指标（基础设施）
    1. Disk Usage (磁盘使用率)：阈值： > 75% (预警)；> 85% (高危)
        1. 注意： Kafka 在磁盘满时会直接挂掉，建议开启 `log.retention.bytes` 配合监控
    2. JVM Heap GC (垃圾回收)：阈值： Full GC 次数 > 1次/小时 或 单次 G1 暂停时间 > 1s
        1. 经验： Kafka 依赖 PageCache，堆内存一般设置在 6GB - 12GB 足够，关键是控制对象晋升速度
    3. CPU Usage：阈值： > 70%
        1. Kafka 主要是 I/O 密集型，如果 CPU 高，通常是由于开启了 Producer 端压缩 (GZIP/Zstd) 或大量的 TLS 加密解密。

**3️⃣**** 告警分级策略**

1. P0 (紧急)： URP > 0、Controller 丢失、磁盘使用率 > 85%。
2. P1 (重要)： 单分区 Lag 超过阈值（如积压 100 万条）、Broker 频繁 Full GC。
3. P2 (提示)： 流量瞬时激增（如超过平时的 3 倍）。
