Kafka 基础架构，包括

+ 各个角色，以及其职责
+ 角色之间的交互流程

# 核心角色及职责
## Producer - 生产者
职责：数据的生产者和发送者。

+ 负责将消息（数据）发送到指定的 Topic；
+ 它不关心消息最终存储在哪个Broker上，但它可以通过策略影响存储位置。



行为：

1. 序列化：将对象转换成字节数组以便网络传输。
2. 分区：决定消息发往 Topic 下的哪个Partition
    - 默认策略：轮询（Round-robin）
    - 基于消息 Key 的哈希（保证相同Key的消息进入同一分区）。
3. 批量发送：为了提高吞吐量，Producer不会发送一条消息就立即发起网络请求，而是会在内存中积攒一批消息后批量发送。

## Consumer - 消费者
职责：数据的消费者

+ 负责从 Broker 拉取（Pull）消息进行处理。
+ 它维护着自己读取到了哪个位置（Offset，即偏移量）



行为：

+ <font style="background-color:#E8F7CF;">拉取模型</font>：Kafka使用 Consumer 主动向 Broker 拉取数据的模式，而不是 Broker 推送。
    - 这能让 Consumer 根据自己的处理能力来控制消费速度，防止自己被“压垮”。
+ Offset 管理：Consumer 需要记录自己消费到了每个 Partition 的哪个位置
    - 这个 Offset 可以保存在 Kafka 的内部 Topic（`__consumer_offsets`）中，由Kafka来管理（自动提交）或由开发者手动管理。

## Consumer Group - 消费者组
职责：

+ 这是Kafka实现<font style="background-color:#E8F7CF;">点对点（Queue）</font>和<font style="background-color:#E8F7CF;">发布订阅（Pub/Sub）</font>两种模型的关键。
+ 它是多个 Consumer 实例的逻辑分组。



关键行为：

+ <font style="background-color:#E8F7CF;">负载均衡</font>：Group 内的所有 Consumer 实例共同分担订阅 Topic 的所有 Partition
    - 一条消息只能被同一个 Group 内的一个 Consumer 实例消费。
    - 这是实现<font style="background-color:#E8F7CF;">水平扩展消费者处理能力的基础</font>。
+ <font style="background-color:#E8F7CF;">故障转移</font>：如果 Group 内一 个 Consumer 实例挂掉，它负责的 Partition 会重新分配给组内其他存活的Consumer
+ 广播模式：如果想让消息被所有消费者都消费一次，就让每个消费者使用不同的 `group.id`。

## Broker - 服务器节点
职责：一个独立的 Kafka 服务器进程

+ 它是 Kafka 集群的<font style="background-color:#E8F7CF;">存储和转发</font>节点
+ 负责<font style="background-color:#E8F7CF;">接收</font> Producer 的数据、<font style="background-color:#E8F7CF;">持久化</font>数据、<font style="background-color:#E8F7CF;">响应 </font>Consumer 的拉取请求。



关键行为：

+ <font style="background-color:#E8F7CF;">存储管理</font>：管理着其所在机器上磁盘的数据文件（Log Segment）。
+ <font style="background-color:#E8F7CF;">无状态（Stateless）</font>：从某种意义上说，Broker 在集群中的状态信息（如谁是 Leader）依赖于 Controller 或元数据，它本身主要处理数据读写。

## Partition - 分区
职责：Kafka <font style="background-color:#E8F7CF;">横向扩展和并行度</font>的核心

+ 一个 Topic 可以分成多个 Partition
+ 每个 Partition 是一个有序的、不可变的消息序列，可以理解为一个 Commit Log。



关键行为：

+ 并行单元：<font style="background-color:#E8F7CF;">Partition是 Kafka 并行处理的最小单位</font>。
    - 一个 Partition 只能被同一个 Consumer Group 内的一个 Consumer 消费，这保证了分区内消息的有序性
    - 如果有多个 Partition，就可以由多个 Consumer 并行消费，从而提高总体吞吐量。
+ 有序性：在单个 Partition 内，消息是有序的（按照 Offset 递增）
    - 但跨 Partition，Kafka 不保证全局有序。

## Replica - 副本
职责：保证数据的<font style="background-color:#E8F7CF;">高可用性</font>和<font style="background-color:#E8F7CF;">持久性</font>

+ 每个Partition可以有多个副本，分布在不同的Broker上

副本分为 Leader 和 Follower。

+ Leader Replica：处理所有读写请求，<font style="background-color:#E8F7CF;">Producer 和 Consumer 只与 Leader 交互</font>
+ Follower Replica：只负责同步。它们不断地从 Leader 拉取消息，保持与 Leader 数据尽可能同步
    - 当 Leader 所在的 Broker 宕机时，Follower 中的一员会被选举为新的 Leader。



关键行为：

+ <font style="background-color:#E8F7CF;">ISR（In-Sync Replicas，同步副本集合）</font>：这是 Kafka 保证数据一致性的核心机制。
    - 只有与 Leader“基本保持同步”（由参数`replica.lag.time.max.ms`控制）的副本才会被放入 ISR 集合中。
    - 只有 ISR 集合中的副本才有资格被选举为 Leader。

## Controller - 控制器
职责：Kafka集群的“大脑”。负责整个集群的管理工作。

+ 在集群启动时，第一个启动的 Broker 会通过 ZooKeeper 竞选成为 Controller。



关键行为：

+ 分区 Leader 选举：当有 Broker 宕机，导致其上的 Partition Leader 失效时，Controller 负责从该 Partition 的 ISR 集合中选出新的 Leader。
+ 元数据管理：监听 ZooKeeper 中的变化（如新增 Topic、Broker 加入或退出），并将最新的集群元数据广播给所有 Broker。
+ 集群成员管理：管理集群中所有 Broker 的存活状态。

##  ZooKeeper（传统架构） vs KRaft
### ZooKeeper 模式（Kafka < 3.3，默认）
ZooKeeper 职责：

+ 存储集群元数据：Broker 列表、Topic 配置、Partition 与 Replica 映射、ACL 等。
+ 选举 Controller（通过临时 znode /controller）。
+ 协助 Consumer Group 协调（旧版，现多用 Kafka 自身协调）。

缺点：

+ 运维复杂（需维护两套系统）
+ ZK 成为性能瓶颈和单点风险
+ 元数据一致性依赖 ZK，扩展性受限

### KRaft 模式（Kafka ≥ 3.3，推荐未来架构）
KRaft（Kafka Raft Metadata Mode）

+ 核心思想：移除 ZooKeeper 依赖，用 Kafka 自身实现元数据管理。



新角色：

+ Controller Quorum：一组专用 Broker（或混合部署），运行 KRaft Controller。
+ 使用 Raft 协议 在 Controller Quorum 内达成元数据一致性。

元数据存储：

+ 元数据以 Topic 形式 存储在内部 Topic`__cluster_metadata` 中。
+ 所有 Broker 从 Controller 订阅元数据变更。

优势：

+ 架构简化（单系统）
+ 更快的故障恢复（秒级 vs 分钟级）
+ 支持更大规模集群（百万级 Partition）

✅ 迁移路径：Kafka 3.x 支持 ZK → KRaft 的迁移工具（需停机）。

# 核心交互流程
## 流程一：生产者发送数据 （Producer -> Broker）
1. 连接元数据：Producer 向<font style="background-color:#E8F7CF;">任意一个 Broker 发送请求</font>，获取目标 Topic 的 Partition Leader 所在 Broker 的元数据信息
2. 数据分区：Producer 根据分区策略确定消息发往哪个 Partition。
3. 发送请求：Producer 直接向该 Partition 的 Leader Broker 发送消息。
4. Broker写入：Leader Broker 收到消息后，将其写入本地磁盘的<font style="background-color:#E8F7CF;">日志文件</font>，此时根据 Producer的 `acks` 配置决定 Leader 的行为
5. 返回响应：Leader 向 Producer 发送响应，包含消息的 Offset 等信息。

## 流程二：消费者消费数据 （Consumer <- Broker）
1. 加入组与分配分区：
    - Consumer启动时，向Group Coordinator（某个Broker）发送请求，表明自己属于哪个Group。
    - Group Coordinator 触发 Rebalance（重平衡），在 Group 内分配 Partition。最终，每个Consumer被分配到固定的Partition。
2. 拉取数据：
    - Consumer 持续向它负责的 Partition 的 Leader Broker 发送 <font style="background-color:#E8F7CF;">Fetch 请求</font>，请求<font style="background-color:#E8F7CF;">参数中包含了它上次处理的 Offset</font>。
    - 这是一个长轮询请求，Broker 如果有新数据就会立即返回；如果没数据，会保持连接一段时间等待数据到来，然后超时返回，避免频繁的轮询。
3. 提交偏移量：
    - Consumer处理完消息后，需要提交最新的 Offset，表明这些消息已经消费完毕。
    - Offset提交可以自动（定时）或手动。提交信息会发送给 Group Coordinator，并记录在一个内部Topic（`__consumer_offsets`）中。

## 流程三：故障转移 （Broker Down）
1. 检测故障：
    - Follower视角：Follower 定期向 Leader 发送 <font style="background-color:#E8F7CF;">Fetch 请求来同步数据</font>，同时这也是一种<font style="background-color:#E8F7CF;">心跳</font>。如果Follower 长时间拉取不到数据，它会认为 Leader 已宕机。
    - Controller视角：Controller 通过 ZooKeeper 的会话超时，或者通过 KRaft 的心跳机制，检测到某个Broker（例如Broker 1）不再响应。
2. 集群更新：Controller 立即感知到这个变化。
3. 重新选举 Leader：
    - Controller 找出 Broker 1 上所有担任 Leader 的 Partition。
    - 对于每一个 Partition，Controller 从其 ISR 集合中选择一个 Follower 作为新的 Leader（如果ISR为空且`unclean.leader.election.enable`开启，则可以从 OSR 中选择，但这有数据丢失风险）。
4. 广播新元数据：Controller 将新的 Leader 和 ISR 信息通过 RPC 广播给集群中所有存活的 Broker。
5. 恢复服务：
    - Producer 和 Consumer 会从元数据刷新中得知新的 Leader 地址，继续向新的 Leader 发送或拉取数据。
    - 原 Broker 1 恢复后，它会作为 Follower 重新加入集群，并同步它宕机期间遗漏的数据。
