## Kafka 如何保证数据不重复？
从 Kafka 内部幂等性、事务机制以及下游（如 Flink）状态一致性三个层面来回答

**1️⃣**** 生产者端：幂等性，针对单会话 + 单分区的情况**

1. 没有幂等性：如果 Producer 发送消息后没收到 ACK 而重试，Broker 就会记录两条相同的消息。
2. 采用幂等性：当 Broker 收到一条开启了幂等性的消息时，不会直接落盘，而是经过校验：重复数据直接丢弃，非重复数据落盘。
3. 局限性：只能保证单分区、单会话内的不重复。如果生产者重启（PID 变了）或者跨分区发送，幂等性就失效了。

**2️⃣**** 跨分区/全链路：事务机制，解决幂等性的局限**

1. Kafka 的事务机制（Atomic Writes）：保证一系列发送操作要么全部成功，要么全部回滚
2. Transactional ID：用户指定一个固定的事务ID，即使生产者重启，也能通过它找回之前的事务状态。
3. 控制消息 ：当事务提交时，Kafka 会在分区中写入一条特殊的控制消息
4. 消费者隔离级别：消费者需设置`isolation.level = read_committed`，这样消费者只会读取到已提交事务的消息，未提交或已回滚的消息对应用不可见。

消费端：端到端的 Exactly-Once，针对：消费端逻辑处理一半崩了，重启后可能会重复消费的情况

实时数仓通常采用以下两种方案：

1. 方案 A：两阶段提交 (2PC)，这是 Flink + Kafka 链路的标配。
    1. 预提交：Flink 开启 Checkpoint，将偏移量和计算状态写入自己的状态后端，并通知 Kafka 开启事务。
    2. 正式提交：当所有算子的 Checkpoint 都成功后，Flink 调用 Kafka 的事务提交接口，真正让数据对下游可见
2. 方案 B：下游幂等写入：下游是支持主键覆盖的数据库，这是最简单稳妥的方法
    1. 做法：在写入时，利用业务主键（或消息的唯一 ID）执行 `INSERT ON DUPLICATE KEY UPDATE` 或 `REPLACE INTO`
    2. 效果：即便消息因为故障被重发了多次，最终数据库里的结果依然是一致的

总结

> 保证不重复的核心逻辑是：‘上游带 ID，中游去重，下游幂等’：
>
> 1. Kafka 内部靠幂等性解决网络重试导致的重复
> 2. 跨算子靠事务 (Transaction) 保证原子性。
> 3. 最终端到端的准确性，往往依赖 Flink 的 Checkpoint 与 Kafka 事务的联动，或者下游存储端的主键覆盖。
>
