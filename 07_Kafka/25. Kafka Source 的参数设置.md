## Kafka Source 的参数设置？
配置原则：性能与可靠性的平衡

层面：连接配置 + 性能 + 数据可靠性 + 容错

  
以下是生产环境中常用的核心配置及其确定逻辑：

1. 核心连接与基础配置：最基础的连通性保证
    1. `connector`，`topic`，`properties.bootstrap.servers` (Broker 地址)
    2. `scan.startup.mode`：启动模式
        1. `group-offsets`：从消费组记录的位移开始，生产环境通标配
        2. `earliest-offset`：新上线的业务或需要回溯数据，
        3. timestamp 指定时间戳：补数任务
2. 性能与吞吐量优化（核心）：压榨 Flink 消费 Kafka 的性能
    1. properties.fetch.min.bytes：Flink 在一次拉取请求中，期望从 Broker 获得的最小数据量
        1. 参考值： 1 (默认，单位字节) -> 提升至 1024 * 1024 (1MB)
        2. 确定逻辑： 攒批，减少网络请求次数，提升高并发下的吞吐量
    2. properties.fetch.max.wait.ms：拉取的最长等待时间，参考值： 500ms
        1. 配合`fetch.min.bytes`使用
        2. 如果数据量稀疏，Broker 最多等 500ms 就会返回，防止数据延迟过大。
3. 数据质量与一致性
    1. `properties.isolation.level`：参考值： `read_committed`
        1. 如果上游开启了 Kafka 事务，必须配置此项。否则 Flink 会读取到上游未提交（可能回滚）的脏数据。
    2. `value.format: json / avro / debezium-json`
        1. 建议使用 avro 或 protobuf 以获得更高的序列化效率；
        2. 若是 CDC 场景则选 debezium-json。
4. 容错与异常处理
    1. `value.fields-error-handling.strategy`：建议值`IGNORE` 或 `DROP_AND_LOG`
    - 在大数据量下，难免会有脏数据或格式错误
    - 设为 `FAIL` 会导致整个 Flink 任务因为一条垃圾数据而频繁重启（Checkpoint 失败）

其他参数

1. Flink 的并行度：不是在 DDL 语句配置，但它是 Source 性能的关键
    1. 确定规则： Flink Source 并行度 ≤ Kafka Partition 数
    2. 逻辑：通常设置为 1:1，即并行度等于分区数，以达到最佳的性能平衡
        1. 如果 Parallelism > Partition，多出来的 Flink Subtask 会空转，浪费资源
        2. 如果 Parallelism < Partition，单个 Subtask 会处理多个分区，可能导致积压
2. 水位线（Watermark）策略：`scan.watermark.idle-timeout`，参考值： 60s。
    - 如果某个 Kafka 分区长时间没有新数据进入，Watermark 会停止推进，导致下游窗口无法触发
    - 配置闲置超时可以强制推进 Watermark



总结

> 在配置 Flink SQL 的 Kafka Source 时，遵循以下步骤：
>
> 1. 保底： 确认隔离级别和位移模式，确保不丢不重。
> 2. 压测： 通过调整 fetch.min.bytes 观察 吞吐量与 延迟曲线，找到甜点位。
> 3. 对齐： 严格对齐 并行度 与 分区数，防止单点瓶颈。”
>
