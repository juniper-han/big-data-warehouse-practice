## 如果有主键，用 Upsert-Kafka Sink
在实时数仓的中间层（DWD/DWS）处理“回撤流”或“更新流”时，upsert-kafka 是非常经典且高效的选择。

普通的 kafka connector 只能作为 Append-only（仅追加） 流，而 upsert-kafka 允许我们在 Kafka 中实现类似数据库的主键更新语义。

  
Upsert-Kafka Sink 的配置要点、工作原理以及在生产环境中的“坑”：

Upsert-Kafka 的必要性：如果上游逻辑包含 `GROUP BY`、`JOIN` 或 `Window` 等会产生更新或删除的操作，标准的 kafka connector 会因为无法处理回撤消息而报错  


Upsert-Kafka 的工作机制

1. Upsert-Kafka 端：
    1. `INSERT` / `UPDATE`：将数据作为普通的 Kafka 记录写入，Key 是 DDL 中定义的主键。
    2. `DELETE`：向 Kafka 写入一条 Tombstone（墓碑）消息。即：Key 是主键，Value 为 null
2. 下游：下游消费者（如另一个 Flink 任务）如果也使用 upsert-kafka 读入，它会自动根据 Key 覆盖旧值，从而在逻辑上实现“最终一致性”。



upsert-kafka Sink 的参数配置，配置逻辑与普通 Sink 有所不同：

原则：保证 Key 的唯一性、维持消息顺序、以及平衡写入延迟

| 参数名称 | 建议值 | 确定逻辑 |
| --- | --- | --- |
| key.format | json / avro | 必须配置。因为 upsert 语义强制要求 Key 序列化。 |
| value.format | json / avro | 存储除主键外的其他字段。建议与 Key 格式保持一致 |
| value.fields-include | 'all' (默认) | 建议保持默认。这意味着 Value 部分也会包含主键字段，方便下游消费者直接解析整行数据，无需手动拼接 Key |
| sink.partitioner | 不要配置，使用默认值 | upsert-kafka 默认使用内部的 Flink Fixed Partitioner。它会自动确保相同主键的数据进入同一个分区 |
| sink.parallelism | 等于 Partition 数 | 极其重要。为了保证同一主键的数据不乱序，必须确保相同的 Key 始终发往同一个分区。如果并行度与分区数不一致，Flink 在分发数据时可能引入额外的网络 Shuffle，增加延迟 |
| sink.buffer-flush.max-rows | 1000-5000 | 缓存多少条数据后刷盘。增大此值可提升吞吐，但会增加延迟。如果主键更新极其频繁（例如一个 ID 每秒更新 100 次），调大此值可以有效合并更新，大幅减轻 Kafka 的写入压力 |
| sink.buffer-flush.interval | 1s 或 2s | 配合 max-rows 使用，防止小流量时数据长时间滞留在内存 |


配合 Kafka Broker 端的配置：单靠 Flink 侧是不够的，Kafka Topic 本身的配置必须“接得住” Upsert 流：

1. cleanup.policy：值： compact
    1. 必须开启日志压缩。这样 Kafka 才会根据 Key 定期清理旧版本数据，只保留最新的 Value 或处理 Tombstone（删除标记）
2. delete.retention.ms：参考值： 86400000 (24小时)
    1. 当 Flink 发送删除消息（Value 为 null）后，Kafka 会保留这个“删除标记”多久
    2. 设置太短可能导致下游还没消费到，标记就消失了。



Upsert-Kafka 生产环境的“致命陷阱”：三个性能与数据质量问题

1. 分区一致性：upsert-kafka 强制使用 Flink 内部的 Hash 分算子
    1. 风险：如果你手动在 DDL 中改`sink.partitioner`，可能会导致同一主键的 Update 和Delete 消息发往不同分区
    2. 后果：下游消费时会出现数据乱序，导致主键残留，数据永远删不掉
    3. 结论：保持默认分区策略即可。
2. 稀疏更新导致的存储膨胀
    1. Kafka 并不是真正的数据库。虽然它支持 Log Compaction（日志压缩），但压缩是在 Segment 关闭后由 Broker 后台异步执行的
    2. 影响：如果在短时间内对同一个 Key 进行了百万次更新，Kafka 磁盘会瞬间飙升。
    3. 对策：必须确保 Kafka Topic 开启了 cleanup.policy=compact，并合理设置`min.cleanable.dirty.ratio`
3. “回撤流”的放大效应：如果 Flink 任务中间计算复杂，一个上游的更新可能会触发下游多次回撤
    1. 现象：写往 Kafka 的数据量远大于原始数据
    2. 优化：在 Sink 前使用 TableConfig 开启 MiniBatch 优化，减少中间状态的频繁变更，合并更新后再写入 Kafka

总结：何时不该使用 Upsert-Kafka？

> 如果下游是 ClickHouse 或 Doris 这种本身就支持 Unique Key 模型的数据库，其实没必要在中间层用 upsert-kafka 增加复杂度。
>
> 直接用普通 Kafka 传 Append 流，在最终端由数据库去重，性能通常会更好。
>
> 只有当你的下游依然是 Flink 任务，且需要基于主键进行聚合（如实时累加指标）时，upsert-kafka 才是不可替代的神器
>



配置 Upsert-Kafka 参数总结：

> 配置 upsert-kafka 的秘诀在于：Key 格式不能省，并行度要对齐，Broker 端要开启 Compact，任务开启 MiniBatch 去重。 这样既能保证数据的最终一致性，又能兼顾高吞吐
>

## Upsert-Kafka 处理回撤流调优
在复杂的 Flink SQL 任务中，upsert-kafka 往往会收到大量的 `UPDATE_BEFORE` 和 `UPDATE_AFTER`。

我的经验法则：如果发现 Kafka 写入流量是输入流量的 2 倍以上，说明回撤流太多。我会开启 Flink 的 微批处理优化 (MiniBatch)

```plsql
SET 'table.exec.mini-batch.enabled' = 'true';
SET 'table.exec.mini-batch.allow-latency' = '1s';
SET 'table.exec.mini-batch.size' = '1000';
```

这样 Flink 会在内存中预先合并同一个 Key 的多次修改，最后只发一条最新的 `UPDATE_AFTER` 给 upsert-kafka，极大地节省了 Kafka 的带宽和存储。
