## 幂等性是怎么保证的？
从【唯一标识】和【重复性的逻辑判断】两个维度来拆解幂等性的实现机制。

Kafka 的幂等性主要通过【PID】+【Sequence Number】+【Broker 端的状态缓存】来保证。

**1️⃣**** 唯一标识符：Kafka 引入三个元数据用于标识消息的唯一性，以识别重复数据：**

1. PID (Producer ID)：生产者的唯一ID，由 Broker 在生产者初始化时分配
2. Sequence Number (序列号)：消息的唯一ID，从 0 开始且单调递增，作用范围【PID】+【Topic的某一分区】
3. Epoch：用于区分生产者的不同“任期”，防止因网络延迟导致的旧请求干扰。

**2️⃣**** Broker 端的校验逻辑：Broker 收到消息后不会直接落盘，先判断是否重复**

1. 缓存检查：Broker 会在内存中为每个分区维护一份 ProducerState，记录该分区收到的 Last Sequence Number
2. 连续性校验：
    1. 如果 New Seq == Last Seq + 1：说明消息顺序正确且无重复，Broker 接收消息，落盘并更新内存中的 Last Seq
    2. 如果 New Seq <= Last Seq：说明这条消息已经处理过了，是一条重复请求。Broker 会直接丢弃该消息，但会向 Producer 返回成功（ACK），以防 Producer 陷入无限重试
    3. 如果 New Seq > Last Seq + 1：说明中间漏掉了消息（乱序或丢失），Broker 会返回 OutOfOrderSequenceException 异常。



以下两点局限性：

1. 单会话限制：PID 是动态分配的。如果 Producer 进程挂掉重启，PID 会变，此时 Broker 无法识别出它是原来的生产者，幂等性会失效
2. 单分区限制：幂等性无法跨分区保证。如果你发送消息到 Partition A 后重试发送到了 Partition B，Broker 是无法跨分区进行重复校验的。

提示： 这就是为什么如果需要跨会话、跨分区的强一致性，必须升级使用 Kafka Transaction（事务） 的原因
