# Broker 端参数（`server.properties`）
核心目标：高吞吐、低延迟、高可用、防雪崩

## **网络与请求处理**
| 序号 | 参数 | 推荐值 | 说明 |
| --- | --- | --- | --- |
| 1 | `num.network.threads` | `3~6` | 处理网络请求的线程数，建议CPU核数的 1~2 倍 |
| 2 | `num.io.threads` | `8~16` | 执行磁盘 I/O 的线程数，建议 ≥ 磁盘数量 × 2 |
| 3 | `socket.send.buffer.bytes` | `1048576`(1MB) | Socket 发送缓冲区，提升大消息吞吐 |
| 4 | `socket.receive.buffer.bytes` | `1048576`(1MB) | Socket 接收缓冲区 |
| 5 | `queued.max.requests` | `1000` | 请求队列最大长度，防 OOM |


## 日志存储与清理
| 序号 | 参数 | 推荐值 | 说明 |
| --- | --- | --- | --- |
| 1 | `log.dirs` | `/data1/kafka,/data2/kafka` | 多磁盘挂载，避免单盘瓶颈 |
| 2 | `log.segment.bytes` | `1073741824`(1GB) | 单个 Segment 大小，过大影响清理速度，过小增加文件数 |
| 3 | `log.retention.hours` | `168`(7天) | 必须设置！ 防止磁盘写满（优先级低于 `retention.ms`） |
| 4 | `log.retention.ms` | `604800000` | 更精确的保留时间（毫秒），建议 7 天 |
| 5 | `log.retention.check.interval.ms` | `300000` (5min) | 日志清理检查间隔 |
| 6 | `log.cleanup.policy` | `delete`或 `compact,delete` | 普通 Topic 用 `delete`，状态类用 `compact` |
| 7 | `log.cleaner.enable` | `true` | 启用 Log Cleaner（compact 必需） |
| 8 | `log.cleaner.threads` | `2~4` | Cleaner 线程数，根据 compact Topic 数量调整 |


## 副本同步与 ISR
| 序号 | 参数 | 推荐值 | 说明 |
| --- | --- | --- | --- |
| 1 | `replica.lag.time.max.ms` | `30000`(30s) | Follower 超时阈值，超过则踢出 ISR（KRaft 模式下仍有效） |
| 2 | `replica.fetch.wait.max.ms` | `500` | Follower 拉取时等待新数据的最大时间，提升吞吐 |
| 3 | `replica.fetch.max.bytes` | `1048576`(1MB) | 单次拉取最大字节数 |
| 4 | `num.replica.fetchers` | `2~4` | Follower 并行拉取线程数，加速同步 |
| 5 | `unclean.leader.election.enable` | `false` | 禁止非 ISR 副本当选 Leader，防止数据丢失（默认 false，勿改！） |


## **JVM 与 OS 优化（启动脚本）**
```bash
# kafka-server-start.sh 中 KAFKA_HEAP_OPTS
export KAFKA_HEAP_OPTS="-Xmx16g -Xms16g"

# GC 策略（G1GC 适合大堆）
export KAFKA_JVM_PERFORMANCE_OPTS="
  -server
  -XX:+UseG1GC
  -XX:MaxGCPauseMillis=20
  -XX:InitiatingHeapOccupancyPercent=35
  -XX:+ExplicitGCInvokesConcurrent
"
```

💡 堆内存建议： 

+ ≤ 32GB（避免指针压缩失效）； 
+ 不超过物理内存的 50%（留内存给 PageCache）。

## **KRaft 模式专属（kraft/server.properties）**
| 序号 | 参数 | 推荐值 |
| --- | --- | --- |
| 1 | `process.roles` | `broker,controller`（混合部署）或 `broker`（分离） |
| 2 | `controller.quorum.voters` | `1@host1:9093,2@host2:9093,...` |
| 3 | `controller.listener.names` | `CONTROLLER` |
| 4 | `sasl.enabled.mechanisms` | 按需启用（如 SCRAM-SHA-256） |


# Producer 端参数（客户端）
核心目标：高吞吐、低延迟、Exactly-once、防阻塞

| 序号 | 参数 | 推荐值 | 说明 |
| --- | --- | --- | --- |
| 1 | `bootstrap.servers` | `broker1:9092,broker2:,...` | 至少 2 个，防单点 |
| 2 | `acks` | `all` | 生产环境必须为 all，保证 ISR 同步 |
| 3 | `retries` | `Integer.MAX_VALUE` | 无限重试（配合幂等性） |
| 4 | `enable.idempotence` | `true` | 开启幂等性，防重复（自动设 acks=all） |
| 5 | `max.in.flight.requests.per.connection` | `5` | Kafka 2.1+ 支持 >1 且幂等，提升吞吐 |
| 6 | `batch.size` | `16384` (16KB) | 批大小，太小降低吞吐，太大增加延迟 |
| 7 | `linger.ms` | `20` | 批等待时间，平衡吞吐与延迟 |
| 8 | `buffer.memory` | `33554432`(32MB) | 缓冲区总大小，防 send() 阻塞 |
| 9 | `compression.type` | `lz4`或 `snappy` | 压缩节省带宽，CPU 换网络 |
| 10 | `delivery.timeout.ms` | `120000`(2min) | 总超时（含重试），避免永久阻塞 |
| 11 | `transactional.id` | `"service-name-${instId}"` | 事务 ID，用于 Exactly-once |


✅ 关键组合： 

```java
props.put("enable.idempotence", true);
props.put("acks", "all");
props.put("retries", Integer.MAX_VALUE);
props.put("max.in.flight.requests.per.connection", 5);
```

# Consumer 端参数（客户端）
核心目标：稳定消费、低 Lag、Exactly-once、防 Rebalance

| 序号 | 参数 | 推荐值 | 说明 |
| --- | --- | --- | --- |
| 1 | `bootstrap.servers` | 同 Producer | — |
| 2 | `group.id` | `"业务名-环境"` | 唯一标识 Consumer Group |
| 3 | `enable.auto.commit` | `false` | 必须关闭，手动提交保 Exactly-once |
| 4 | `auto.offset.reset` | `latest`或 `earliest` | 新 Group 初始位置 |
| 5 | `fetch.min.bytes` | `1024`(1KB) | 拉取最小字节数，减少空轮询 |
| 6 | `fetch.max.bytes` | `52428800` (50MB) | 单次拉取最大字节数 |
| 7 | `max.poll.records` | `500` | 单次 poll 返回记录数，防处理超时 |
| 8 | `max.poll.interval.ms` | `300000` (5min) | 根据业务处理时间调整，默认 5min |
| 9 | `session.timeout.ms` | `45000` (45s) | Coordinator 判定死亡超时 |
| 10 | `heartbeat.interval.ms` | `15000`(15s) | 心跳间隔，必须 ≤ session/3 |
| 11 | `partition.assignment.strategy` | `org.apache.kafka.clients.consumer.CooperativeStickyAssignor` | 推荐 CooperativeSticky，支持增量 Rebalance |
| 12 | `isolation.level` | `read_committed` | 事务消费时必须设置 |
| 13 | `group.instance.id` | `"consumer-app-01"` | 静态成员 ID，滚动升级不触发Rebalance |


✅ Rebalance 优化组合：

```java
props.put("session.timeout.ms", 45000);
props.put("heartbeat.interval.ms", 15000);
props.put("max.poll.interval.ms", 300000); // 根据实际处理时间
props.put("partition.assignment.strategy", 
    "org.apache.kafka.clients.consumer.CooperativeStickyAssignor");
props.put("group.instance.id", "my-consumer-" + HOSTNAME);
```

# Topic 级动态配置（通过 `kafka-configs.sh`）
按业务特性精细化调优

| 序号 | 场景 | 配置命令 | 说明 |
| --- | --- | --- | --- |
| 1 | 高吞吐日志 Topic | `--add-config retention.ms=86400000,segment.bytes=1073741824` | 保留 1 天，大 Segment |
| 2 | 状态存储 Topic（compact） | `--add-config cleanup.policy=compact,delete,segment.ms=604800000` | compact + delete 双策略 |
| 3 | 金融交易 Topic | `--add-config min.insync.replicas=2,unclean.leader.election.enable=false` | 强一致性 |
| 4 | 临时调试 Topic | `--add-config retention.ms=3600000` | 1 小时自动清理 |


# 操作系统级调优（`/etc/sysctl.conf`）
```plain
# 网络
net.core.somaxconn = 65535
net.ipv4.tcp_max_syn_backlog = 65535
net.ipv4.tcp_fin_timeout = 30

# 文件句柄
fs.file-max = 2097152
fs.nr_open = 2097152

# 虚拟内存（提升 PageCache 效率）
vm.dirty_ratio = 15
vm.dirty_background_ratio = 5
vm.swappiness = 1
```

执行：`sysctl -p`

# 总结：生产调优 Checklist
✅ Broker

+  多磁盘 `log.dirs`
+  设置 `retention.ms`
+ `unclean.leader.election.enable=false`
+  JVM 堆 ≤ 32GB + G1GC

✅ Producer

+ `enable.idempotence=true`
+ `acks=all`
+  合理 `batch.size` + `linger.ms`

✅ Consumer

+ `enable.auto.commit=false`
+  使用 `CooperativeStickyAssignor`
+  配置 `group.instance.id`

✅ 运维

+  部署磁盘/ISR/Lag 告警 
+  定期演练故障恢复

---

💡 建议

+ 没有“万能参数”，所有配置必须结合 业务流量模型、硬件规格、SLA 要求 进行压测验证。
+ 建议使用 Kafka Tutorials（Confluent） 或 kafka-producer-perf-test.sh / kafka-consumer-perf-test.sh 进行基准测试。
