## Kafka Sink 参数配置
场景：复杂 Flink 任务拆成 2 个，第一个任务的输出写入 Kafka，由第二个任务消费

+ 中间通过 Kafka 解耦，这是构建大规模实时数仓的典型逻辑
+ 第一个任务的 Kafka Sink 的配置直接决定了下游数据的是否可靠、有序

核心关注：一致性级别、写入吞吐量

1. 端到端一致性参数：`sink.delivery-guarantee`
    1. exactly-once (首选)： 依赖 Kafka 事务和 Flink Checkpoint 的协同
    2. at-least-once： 追求吞吐 +下游具备幂等去重能力，选这个可以显著降低写入延迟
    3. 确定依据： 看下游任务是否能容忍重复，由于你拆分了任务，为了保证整个链路的准确性，建议开启 exactly-once
2. 事务相关参数：如果启用了 exactly-once，以下参数必须配置：
    1. sink.transactional-id-prefix：设置为一个业务相关的唯一字符串（如 `order_process_sink_`）
        1. 这是 Kafka 事务的身份标识，防止不同任务或任务重启后事务 ID 冲突
    2. properties.transaction.timeout.ms：Kafka 事务超时时间，参考值： 900000 (15分钟)
        1. 必须大于 Flink 的 Checkpoint 间隔 + 预留的容错时间
        2. 如果 Checkpoint 设为 5 分钟，这个值至少要 10-15 分钟，否则 Checkpoint 还没完成，Kafka 事务就超时回滚了
3. 幂等性相关参数：配合 at-least-once 使用
    1. properties.max.in.flight.requests.per.connection：保证在重试时不乱序
    2. properties.retries：生产者发送消息失败时的最大重试次数，默认无限次
4. 分区与并行度：
    1. sink.partitioner
        1. default (Hash)： 如果 DDL 定义了 PRIMARY KEY，Flink 会按主键 Hash 写入。
        2. fixed： 每个 Flink Parallelism 对应一个 Kafka Partition，性能最高，但容易导致 Kafka 端负载不均
        3. round-robin： 轮询写入，负载最均衡
        4. 确定逻辑： 必须看下游的需求。如果下游任务需要按 user_id 做聚合（KeyBy），那么 Sink 端必须按 user_id 进行 Hash 分区，否则下游会产生大量的 Network Shuffle，性能剧降
    2. sink.parallelism：与 Kafka 分区一致，避免数据在 Sink 端的 Shuffle
5. 性能优化参数：吞吐量与批次控制，Kafka Sink 的性能瓶颈通常在于过多的网络请求
    1. properties.batch.size：参考值： 16384 (16KB) -> 提升至 65536 (64KB) 或更高
        1. 越大吞吐越高，但内存压力越大
        2. 根据单条消息大小决定，确保一个 Batch 能容纳多条消息
    2. properties.linger.ms：生产者发送批次前等待额外消息的时间，参考值： 50 (50毫秒)
        1. 强制让数据在 Producer 端攒一会再发。在非极端实时场景下，设置 50-100ms 能极大地减少 CPU 消耗并提升吞吐
    3. properties.compression.type：参考值： lz4 或 zstd
        1. 如果数据量极大，开启压缩可以显著节省网络带宽和磁盘 IO，lz4 是平衡压缩率和 CPU 消耗的最佳选择



实践踩坑

> 在拆分任务时，最坑的地方不是参数设置，而是【事务残留】
>
> 如果第一个任务崩溃且没有正确恢复，Kafka 可能会残留未提交的事务
>
> 特别关注 transaction.max.timeout.ms 这个 Broker 端的配置，确保它大于 Flink 的事务超时，否则 Flink 任务会因为无法获取事务权限而报错。”
