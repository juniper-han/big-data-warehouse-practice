Hive Metastore 是 Hive 的**核心元数据服务**，也是整个 Hadoop 生态中**事实上的元数据标准**。

它不仅被 Hive 使用，还被 Spark、Flink、Presto、Trino、Impala 等几乎所有大数据引擎所依赖。

# Hive Metastore 的理论基础
## 定义
Hive Metastore（HMS）

+ 是一个独立的元数据服务
+ 用于存储和管理 Hive 表、数据库、分区、列、SerDe、存储格式等元信息
+ 并提供 Thrift API 供客户端访问。

## 作用
| 功能 | 说明 |
| --- | --- |
| **元数据存储** | 表结构（Schema）、分区信息、存储路径、SerDe 类、表属性等 |
| **元数据服务** | 提供 Thrift 接口，供 Hive CLI、Beeline、Spark、Flink 等查询/修改元数据 |
| **统一元数据中心** | 成为 Hadoop 生态的“元数据枢纽”，实现多引擎元数据共享 |


## 为什么需要 Metastore？
+ 解耦计算与元数据：
    - Hive 执行引擎不直接读取 HDFS 上的表结构，而是通过 Metastore 获取；
+ 提升性能：
    - 避免每次查询都解析 HDFS 目录结构；
+ 支持复杂操作：如 `ALTER TABLE ADD PARTITION`、`MSCK REPAIR TABLE`；
+ 多引擎协同：
    - Spark 写入的表，Hive 能直接查询（前提是共用 Metastore）。

💡 **类比理解**： 

+ HDFS = 文件柜（存数据） 
+ Hive Metastore = 文件柜的目录索引（存“哪里有什么”） 
+ Hive/Spark = 查阅文件的人（通过索引找文件）

# Hive Metastore 架构设计
Hive Metastore 支持三种部署模式，生产环境只推荐**远程模式（Remote Mode）**。

## 三种部署模式对比
| 模式 | 架构 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- | --- |
| **内嵌模式（Embedded）** | Metastore 与 Hive Driver 同进程，使用 Derby 内存 DB | 无需配置，适合测试 | 单用户、无持久化、不能共享 | 本地开发、Demo |
| **本地模式（Local）** | Metastore 与 Hive Driver 同进程，连接 MySQL/PostgreSQL | 元数据持久化 | 无法被其他引擎共享；Hive Server 挂则 Metastore 不可用 | 小型单机环境 |
| **远程模式（Remote）****✅** | Metastore 独立进程（Hive Metastore Server），所有客户端通过 Thrift 连接 | 高可用、多引擎共享、解耦 | 需额外维护 HMS 服务 | **生产环境唯一选择** |


📌 **生产结论**：  
必须使用 Remote Mode + MySQL/PostgreSQL 作为后端数据库！

## Remote Mode 架构图
```plain
+----------------+     +---------------------+     +------------------+
|   Hive CLI     |     |                     |     |                  |
|   Beeline      |<--->| Hive Metastore      |<--->| MySQL /          |
|   Spark        | Thr | Server (独立进程)   | JDBC| PostgreSQL       |
|   Flink        | ift |                     |     | (元数据存储)     |
|   Trino        |     |                     |     |                  |
+----------------+     +---------------------+     +------------------+
                              ↑
                              | Thrift API
                              ↓
                       +------------------+
                       |    HDFS / S3     |
                       | (实际数据存储)   |
                       +------------------+
```

+ **Thrift Server**：Hive Metastore 的核心服务进程；
+ **RDBMS**：存储元数据的底层数据库（通常为 MySQL）；
+ **Clients**：任何支持 Hive Metastore 协议的客户端。

# Hive Metastore 核心组件与数据模型
## 元数据存储结构（MySQL 表）
Hive 在 RDBMS 中创建约 **40+ 张表**，关键表包括：

| 表名 | 作用 |
| --- | --- |
| `DBS` | 存储所有数据库（Database）信息 |
| `TBLS` | 存储所有表（Table）信息 |
| `PARTITIONS` | 存储所有分区（Partition）信息 |
| `SDS` | 存储 SerDe 和存储描述符（Storage Descriptor） |
| `COLUMNS_V2` | 存储列（Column）信息 |
| `PARTITION_KEY_VALS` | 存储分区键值（如 `dt='2024-01-01'`） |
| `TAB_COL_STATS` | 存储列统计信息（用于 CBO） |


🔍 示例：查询某表的所有分区

```sql
SELECT p.PART_NAME
FROM TBLS t
JOIN DBS d ON t.DB_ID = d.DB_ID
JOIN PARTITIONS p ON t.TBL_ID = p.TBL_ID
WHERE d.NAME = 'ods' AND t.TBL_NAME = 'user_log';
```

## 元数据对象层级
Hive Metastore 的元数据是**层级化**的：

```plain
Catalog (可选，Hive 3.0+)
└── Database (DBS)
    └── Table (TBLS)
        ├── Columns (COLUMNS_V2)
        ├── Storage Format (SDS)
        └── Partitions (PARTITIONS)
            └── Partition Values (PARTITION_KEY_VALS)
```

⚠️ 注意：Hive 默认没有 Catalog 层（兼容旧版），但 Flink/Hive 3.0+ 支持。

# Hive Metastore 的使用方式
## 客户端如何连接 Metastore？
所有客户端通过 `**hive-site.xml**` 配置连接信息：

```xml
<!-- 指定 Metastore 服务地址（Remote Mode） -->
<property>
  <name>hive.metastore.uris</name>
  <value>thrift://hms1:9083,thrift://hms2:9083</value>
</property>

<!-- 底层 RDBMS 配置（仅 HMS Server 需要） -->
<property>
  <name>javax.jdo.option.ConnectionURL</name>
  <value>jdbc:mysql://mysql:3306/hive_metastore?createDatabaseIfNotExist=true</value>
</property>
<property>
  <name>javax.jdo.option.ConnectionDriverName</name>
  <value>com.mysql.cj.jdbc.Driver</value>
</property>
```



✅ **关键**：所有引擎只要配置了相同的 `hive.metastore.uris`，就能**共享同一份元数据**！

## 常见客户端操作示例
### Hive CLI / Beeline
```plsql
SHOW DATABASES;
USE ods;
SHOW TABLES;
DESCRIBE user_log;
SHOW PARTITIONS user_log;
```

### Spark SQL
```scala
spark.sql("SELECT * FROM hive_table")
```

需在 `spark-defaults.conf` 中配置 `hive.metastore.uris`

### Flink SQL
```sql
CREATE CATALOG myhive WITH (
  'type' = 'hive',
  'hive-conf-dir' = '/etc/hive/conf'  -- 包含 hive-site.xml
);
```

# Hive Metastore 高可用（HA）部署
生产环境必须部署 HMS HA，避免单点故障。

## HA 架构
```plain
Clients → [Load Balancer (e.g., HAProxy)] → [HMS Server 1]
                                           [HMS Server 2]
                                           [HMS Server 3]
                                                 ↓
                                           [Shared MySQL Cluster]
```

## 配置要点
1. **多 HMS 实例**：启动多个 `hive --service metastore` 进程；
2. **负载均衡**：通过 HAProxy/Nginx 做 Thrift 流量分发；
3. **共享数据库**：所有 HMS 实例连接同一个 MySQL 主从集群；
4. **客户端配置**：

```xml
<property>
  <name>hive.metastore.uris</name>
  <!-- 可直接写多个地址，Hive Client 会自动 failover -->
  <value>thrift://hms1:9083,thrift://hms2:9083,thrift://hms3:9083</value>
</property>
```

💡 **注意**：Hive Client 内置重试机制，无需 LB 也可实现 HA（但 LB 更可控）。

# Hive Metastore 性能调优
Metastore 是高频访问服务，需重点优化。

## 数据库层优化
| 优化项 | 说明 |
| --- | --- |
| **使用 SSD** | 元数据查询对 I/O 延迟敏感 |
| **连接池** | 配置 `datanucleus.connectionPool.maxPoolSize` |
| **索引优化** | 确保 `TBLS.TBL_NAME`, `PARTITIONS.PART_NAME`等有索引 |
| **定期 ANALYZE** | `ANALYZE TABLE TBLS COMPUTE STATISTICS` |


## HMS Server 参数调优
```properties
# 增大线程池（默认 5，太小！）
hive.metastore.execute.setugi=true
hive.metastore.server.max.message.size=1073741824
hive.metastore.server.min.threads=200
hive.metastore.server.max.threads=1000

# 缓存（减少 DB 查询）
hive.metastore.cache.pinobjtypes="Table,StorageDescriptor,SerDeInfo,Partition"
```

# Hive Metastore 与现代数据湖（Lakehouse）
随着 Delta Lake、Iceberg、Hudi 的兴起，Metastore 的角色正在演进：

| 技术 | 与 Metastore 关系 |
| --- | --- |
| **Delta Lake** | 可选：通过 HMS Catalog 注册表（`spark.sql.catalog.hms`) |
| **Apache Iceberg** | 支持：Iceberg HMS Catalog 将元数据同步到 HMS |
| **Apache Hudi** | 支持：Hive Sync Tool 自动同步元数据到 HMS |


✅ 趋势：  
Metastore 正从“Hive 专属”变为“开放元数据服务”，继续作为 Lakehouse 的元数据枢纽。
