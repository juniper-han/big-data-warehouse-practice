## 建表参数
### 存储格式与压缩
1. 存储格式：ORC，`STORED AS ORC`
    1. ORC 优势：支持 ACID 的唯一格式，列式存储 + 内置索引（min/max）、谓词下推、高效压缩
    2. ❌ 避免使用 `TEXTFILE`（无压缩、无列裁剪），`SEQUENCEFILE` 已过时
2. 压缩：ORC 自带轻量级压缩（ZLIB），无需额外指定
    1. 也可选 SNAPPY（更快，压缩率略低），`TBLPROPERTIES ("orc.compress"="SNAPPY")`

### **注释与可维护性**
```sql
COMMENT '用户行为日志表，每日增量'
PARTITIONED BY (dt STRING COMMENT '日期分区 YYYY-MM-DD')
```

### 分区分桶排序
#### 分区
分区：Hive 表的逻辑/物理<font style="background-color:#E8F7CF;">目录</font>

+ 目的：分区裁剪，查询时直接跳过某些分区，减少 I/O 扫描量

适用场景：

1. 低基数字段：几十~几千
2. 频繁按该字段过滤



理想单分区大小：128MB ~ 10GB，兼顾 HDFS 效率和查询并行度

单表总分区数限制：1w 是安全线，几万需要考虑集群能力和调优 MetaStore 以及 HDFS

+ 需要关注参数：`<font style="color:rgb(6, 10, 38);background-color:rgba(6, 10, 38, 0.06);">hive.exec.max.dynamic.partitions = 1000</font>`



分区数不合适的影响

1. 分区数量过多：
    1. NameNode 内存压力大：目录也是元数据
    2. Hive MetaStore 性能下降：查询 `SHOW PARTITIONS` 或 `MSCK REPAIR TABLE` 会变慢
    3. 动态分区写入失败：超过 `hive.exec.max.dynamic.partitions.pernode`（默认 100）会报错
    4. 全表扫描资源调度开销大：分区数多意味着 InputSplit 多，需要启动的 MapTask 任务多。
2. 分区数量过少
    1. 无法利用分区裁剪的优势



分区参数选择总结

1. 分区键：低基数字段，或频繁用于过滤的字段
2. 单分区大小：128 MB ~ 10 GB
3. 分区数量：不超过 1w
4. 注意事项：不需要过滤 → 分区无用

#### 分桶
分桶：分区内按哈希值划分的<font style="background-color:#E8F7CF;">文件</font>

+ 目的：优化 `JOIN` 和 `GROUP BY` 性能
    - 相同 key 的数据总在同一个桶 → 支持 Map-side Join（避免 Shuffle）

适用场景：

1. 字段高基数、均匀分布
2. 表之间频繁按该字段 `JOIN`，或大量 `GROUP BY`
3. 需要高效采样



理想单桶大小：128 MB ~ 500 MB

分桶数量：表总大小 / 目标桶大小

+ 注意：常用桶数为 32、64、128、256（2 的幂次利于哈希分布）



分桶不合适的影响

1. 分桶数量大：每个桶都很小 → 小文件问题
2. 分桶键不合适：导致数据在各桶的分布不均匀，产生数据倾斜



分桶参数选择总结

1. 分桶键：高基数字段，表之间频繁按该字段 `JOIN`，或大量 `GROUP BY`
2. 单桶大小：128 MB ~ 500 MB
3. 分桶数量：表总大小 / 目标桶大小，取32、64、128、256 等 2 的幂次，利于哈希分布
4. 注意事项：
    1. 小表不分桶：除非用于 Bucket Map Join 且与大表对齐
    2. 大表按需分桶：仅当有高频 `JOIN` 或 `GROUP BY` 分桶键时才启用

#### 排序
排序：`SORTED BY`，是一种数据预排序机制，是物理存储层面的优化手段

必须与分桶连用

```plsql
CREATE TABLE tbl (...)
CLUSTERED BY (col1) 
SORTED BY (col2 [ASC|DESC]) 
INTO N BUCKETS;
```

排序的目的：

1. 如果过滤条件是排序键，可加速范围查询
2. 如果 Key 是排序键，可提升 `GROUP BY` 和 `ORDER BY` 性能：桶内已排序，MapTask 可做局部聚合，Shuffle 数量大幅减少
3. 支持高效采样：`TABLESAMPLE(BUCKET 1 OUT OF 10 ON col1)` + 桶内排序 → 采样结果有序，适合时间序列分析
4. 优化 Merge Join：若两张大表都按 join key 分桶 + 排序，可启用 Sort-Merge Bucket Map Join（SMB Map Join），完全避免 Reduce 阶段。

工作机制：写入时先分桶，每个桶内再排序后写入；

排序的坑

1. `SORTED BY`：仅保证桶内有序，不保证全局有序
2. 写入时必须显式启用分桶和排序：`INSERT` 必须包含 `DISTRIBUTE BY + SORT BY`
    1. 如果不显式写入：Hive 不会自动排序，桶内数据仍是乱序，`SORTED BY` 形同虚设

```plsql
INSERT INTO user_events PARTITION (dt='2020-01-01')
SELECT user_id, event_time, action
FROM staging
DISTRIBUTE BY user_id          -- 对应 CLUSTERED BY
SORT BY user_id, event_time;   -- 必须包含排序字段！
```

3. 增加写入开销：排序需要在 reducer 阶段做 in-memory sort 或 spill to disk



排序参数选择总结

1. 适用场景：高频 `RANGE SCAN`、`GROUP BY` 或 `SMB JOIN` 时
2. 排序字段：高过滤性，常用于范围/聚合字段
3. 排序字段包含分桶字段`SORT BY (bucket_col, sort_col)`，避免多`key`混合
4. 建议显式设置：`TBLPROPERTIES ("bucketing_version"="2")`
5. 写入时显式：`DISTRIBUTE BY + SORT BY`

### ACID 与事务支持
Hive 3.0 默认启用 ACID（原子性、一致性、隔离性、持久性），但需满足以下两个条件：

1. 表为 ORC 格式
2. 启用 Hive 事务（通常集群已配置）

建表时建议显式声明开启事务：

+ 优势：支持 `INSERT`, `UPDATE`, `DELETE`, `MERGE`，并自动处理小文件合并（Compaction）

```sql
TBLPROPERTIES (
  'transactional'='true',
  'transactional_properties'='default'
)
```

### 小文件自动合并（Compaction）
Hive 3.0 对 ACID 表自动触发 Minor/Major Compaction，但需确保配置以下参数（通常由管理员设置）

1. `hive.compactor.initiator.on = true`
2. `hive.compactor.worker.threads >= 1`
3. `hive.compactor.delta.num.threshold`（默认 10）→ 超过 10 个 delta 文件触发合并

小文件的自动合并（Compaction）不需要在 DDL（建表语句）中手工设置，但需监控 `sys.compaction_queue` 表。

### 外部表 vs 内部表选择
内部表（MANAGED_TABLE）：数据完全由 Hive 管理

1. `DROP TABLE`会删 HDFS 数据
2. 支持 ACID：必须用 ORC 格式，支持小文件自动合并（Compaction）
3. 需要 `UPDATE/DELETE` → 必须用内部表 + ORC + ACID

外部表（EXTERNAL_TABLE）：数据由用户管理

1. 不支持事务，不支持小文件自动合并
2. 适用于：仅用 Hive 查询的表，或避免误删数据

### 表属性（TBLPROPERTIES）参数
#### 生产环境必配参数
| 参数 | 推荐值 | 说明 |
| --- | --- | --- |
| `transactional` | `'true'` | 启用 ACID 事务（仅内部 ORC 表有效）。Hive 3.0 虽可能默认开启，但显式声明更安全。 |
| `bucketing_version` | `'2'` | 使用新版分桶哈希算法（兼容 Hive 3+，避免旧版 hash 不一致问题）。 |
| `orc.compress` | `'SNAPPY'`<br/>或 `'ZLIB'` | SNAPPY：压缩/解压快，适合 CPU 敏感场景；<br/>ZLIB：压缩率高，适合存储敏感场景。<br/>推荐 SNAPPY。 |


#### 分桶表专用参数
1. `hive.tez.bucket.pruning`：建表不设，但查询前可 `SET hive.tez.bucket.pruning=true;`启用桶裁剪

#### Compaction 调优（ACID 表）
Compaction 调优（ACID 表）

+ 一般无需在 DDL 中设置，除非某张表写入特别频繁（如每分钟 INSERT），可适当调低阈值

| 参数 | 推荐值 | 说明 |
| --- | --- | --- |
| `compactor.mapreduce.map.memory.mb` | `4096` | 合并任务内存（避免 OOM） |
| `hive.compactor.delta.num.threshold` | `10`（默认） | 超过 10 个 delta 文件触发 Minor Compaction |
| `hive.compactor.delta.threads` | `1`（默认） | 每表并发 compaction 线程数 |


## 会话级参数：`SET`中设置
在 ETL 脚本或查询前显式声明

### 分桶与排序写入
使用场景：写入 `CLUSTERED BY ... SORTED BY ... `表前必须设置

| 参数 | 推荐值 | 说明 |
| --- | --- | --- |
| `hive.enforce.bucketing` | `true` | 写入分桶表时强制 reducer 数 = 桶数 |
| `hive.enforce.sorting` | `true` | 写入 `SORTED BY`表时强制桶内排序 |
| `mapred.reduce.tasks` | 不建议手动设 | 若开启 `enforce.bucketing`，Hive 自动设为桶数；手动覆盖可能破坏分桶 |


### 动态分区（Dynamic Partitioning）
不设 `nonstrict` 会导致 INSERT INTO tbl PARTITION (dt) SELECT ..., dt FROM ... 失败

| 参数 | 推荐值 | 说明 |
| --- | --- | --- |
| `hive.exec.dynamic.partition` | `true` | 启用动态分区 |
| `hive.exec.dynamic.partition.mode` | `nonstrict` | 允许全动态分区（默认 `strict` 要求至少一个静态分区） |
| `hive.exec.max.dynamic.partitions` | `1000` ~ `5000` | 单次 INSERT 最大分区数（防 OOM） |
| `hive.exec.max.dynamic.partitions.pernode` | `300` | 单个 reducer 最多创建分区数 |


### 小文件与合并控制（Small File Handling）
适用于非 ACID 的外部表或旧表的小文件治理

| 参数 | 推荐值 | 说明 |
| --- | --- | --- |
| `hive.input.format` | `org.apache.hadoop.hive.ql.io.CombineHiveInputFormat` | Map 端自动合并小文件（非 ACID 表有效） |
| `mapreduce.input.fileinputformat.split.minsize` | `134217728` (128MB) | 控制 Split 最小大小，减少 MapTask 数 |
| `hive.merge.mapfiles` | `true` | 启用 Map-only 任务输出合并 |
| `hive.merge.size.per.task` | `268435456` (256MB) | 合并后每个文件目标大小 |


### 执行引擎与资源（Tez / MapReduce）
影响并发度和内存溢出风险

| 参数 | 推荐值 | 说明 |
| --- | --- | --- |
| `hive.execution.engine` | `tez` | 明确指定引擎（避免集群默认不一致） |
| `tez.grouping.min-size` | `134217728` | Tez 自动合并小 Split 的最小阈值 |
| `tez.grouping.max-size` | `1073741824` | 最大 Split 大小 |
| `hive.tez.container.size` | `4096` ~ `8192` | Tez 容器内存（MB），根据集群调整 |
| `hive.tez.java.opts` | `-Xmx3276m` | JVM 堆内存（约为 container size 的 80%） |


### 查询优化（Query Optimization）
| 参数 | 推荐值 | 说明 |
| --- | --- | --- |
| `hive.optimize.bucketmapjoin` | `true` | 启用 Bucket Map Join |
| `hive.optimize.sort.dynamic.partition` | `true` | 动态分区写入前按分区字段排序，减少 reducer 内存 |
| `hive.vectorized.execution.enabled` | `true` | 启用向量化执行（Hive 3.0 默认开） |
| `hive.cbo.enable` | `true` | 启用基于成本的优化（CBO） |

### 生产建表通用模板（含 SET 参数）
```sql
-- ==================================================
-- Hive 3.0 生产建表通用模板（内部 ACID 表）
-- 适用场景：需 UPDATE/DELETE、高频 JOIN/GROUP BY、自动小文件合并
-- ==================================================

-- 【1】建表 DDL
CREATE TABLE IF NOT EXISTS your_db.your_table_name (
  id BIGINT COMMENT '主键ID',
  user_id BIGINT COMMENT '用户ID',
  event_name STRING COMMENT '事件名称',
  event_time TIMESTAMP COMMENT '事件时间',
  properties MAP<STRING, STRING> COMMENT '扩展属性'
)
COMMENT '业务说明：例如 用户行为日志表，每日增量'
PARTITIONED BY (dt STRING COMMENT '日期分区，格式 YYYY-MM-DD')
CLUSTERED BY (user_id) 
SORTED BY (user_id, event_time) 
INTO 128 BUCKETS
STORED AS ORC
TBLPROPERTIES (
  "transactional" = "true",
  "bucketing_version" = "2",
  "orc.compress" = "SNAPPY"
);

-- ==================================================
-- 【2】ETL 写入脚本模板（.hql 文件开头）
-- ==================================================
-- 执行引擎
SET hive.execution.engine=tez;

-- 动态分区
SET hive.exec.dynamic.partition=true;
SET hive.exec.dynamic.partition.mode=nonstrict;
SET hive.exec.max.dynamic.partitions=2000;
SET hive.exec.max.dynamic.partitions.pernode=500;

-- 分桶 & 排序写入（若表定义了 CLUSTERED BY / SORTED BY）
SET hive.enforce.bucketing=true;
SET hive.enforce.sorting=true;

-- 资源调优（根据集群调整）
SET hive.tez.container.size=8192;
SET hive.tez.java.opts=-Xmx6553m;

-- 查询优化
SET hive.optimize.bucketmapjoin=true;
SET hive.optimize.sort.dynamic.partition=true;
SET hive.vectorized.execution.enabled=true;
SET hive.cbo.enable=true;

-- 【3】写入语句（关键：必须包含 DISTRIBUTE BY + SORT BY）
INSERT OVERWRITE TABLE your_db.your_table_name PARTITION (dt)
SELECT 
  id,
  user_id,
  event_name,
  event_time,
  properties,
  dt
FROM your_db.source_table
WHERE dt = '${BIZ_DATE}'  -- 调度系统传参
DISTRIBUTE BY user_id          -- 对应 CLUSTERED BY 字段
SORT BY user_id, event_time;   -- 对应 SORTED BY 字段
```

使用说明

+ 内部表 vs 外部表： 
    - 需要事务 → 用上述模板（内部表）； 
    - 仅查询/避免删数据 → 改为 `CREATE EXTERNAL TABLE`，移除 `transactional` 属性，不设 `enforce.bucketing`（除非明确管理写入）。
+ 不分桶场景： 
    - 移除 `CLUSTERED BY ... INTO N BUCKETS` 和 `SORTED BY`； 
    - 不需要 `SET hive.enforce.bucketing=true`。
+ 调度集成： 
    - 将 `${BIZ_DATE}` 替换为调度系统变量（如 Airflow 的 `{{ ds }}`）； 
    - 所有 `SET` 语句建议封装为公共头文件，供多个 ETL 复用。
