# Hive 中的`JOIN`
## Hive 支持的`JOIN`类型
| JOIN 类型 | 是否支持 | 说明 |
| --- | --- | --- |
| `INNER JOIN` | ✅ | 仅返回两表匹配的行 |
| `LEFT OUTER JOIN` | ✅ | 返回左表全部 + 右表匹配部分 |
| `RIGHT OUTER JOIN` | ✅ | 返回右表全部 + 左表匹配部分 |
| `FULL OUTER JOIN` | ✅ | 返回两表所有行，不匹配处补 NULL |
| `LEFT SEMI JOIN` | ✅ | Hive 特有优化型 JOIN，等价于 `IN` / `EXISTS` |
| `CROSS JOIN` | ✅（需开启） | 笛卡尔积（默认关闭，需设 `hive.strict.checks.cartesian.product=false`） |
| `MAP JOIN` / `BROADCAST JOIN` | ✅（自动或手动） | 小表广播到内存，避免 Reduce 阶段 |


Hive 不支持 `RIGHT SEMI JOIN` 或 `ANTI JOIN`（但可用 `LEFT JOIN + WHERE IS NULL` 模拟）

## `LEFT SEMI JOIN`半连接
+ 语义：返回左表中在右表存在匹配的记录，但不返回右表的任何字段。
+ 等价于：`WHERE left_key IN (SELECT right_key FROM right_table)`
+ 用途：高效实现“存在性检查”，避免数据膨胀

语法

```plsql
SELECT a.* 
FROM table_a a
LEFT SEMI JOIN table_b b
ON a.id = b.id;
```

❗ 注意：

1. `SELECT` 中不能引用右表字段（如` b.name` 会报错）。
2. `ON` 条件只能是等值连接（=），不支持 <, != 等。
3. 不能有 `WHERE` 子句对右表过滤（过滤应放在子查询中）



Semi Join 的优势：主要针对的是右表有重复的`Key`

| 场景 | 普通 `INNER JOIN` | `LEFT SEMI JOIN` |
| --- | --- | --- |
| 右表有重复 key | 结果会膨胀（1:N） | 结果不会膨胀（只保留左表一行） |
| 只关心“是否存在” | 需 `DISTINCT`去重 | 天然去重，更高效 |
| 性能 | 需传输右表数据 | 只需右表 `key` 列，减少 shuffle 数据量 |




示例：

```plsql
-- 找出所有购买过商品的用户（用户表 user，订单表 order）
-- 方式1：INNER JOIN（可能重复）
SELECT DISTINCT u.user_id, u.name
FROM user u
JOIN order o ON u.user_id = o.user_id;

-- 方式2：LEFT SEMI JOIN（推荐）
SELECT u.*
FROM user u
LEFT SEMI JOIN order o ON u.user_id = o.user_id;
```



注意：优先用 `LEFT SEMI JOIN` 替代 `IN` 子查询。

# 多表`JOIN`顺序
在 Hive（以及其他分布式 SQL 引擎如 Spark SQL）中，多表 `JOIN` 的顺序对查询性能、资源消耗甚至结果正确性（在某些边缘情况下）都有重要影响

## `JOIN`顺序的重要性
Hive 默认采用 “从左到右”依次 JOIN 的策略（除非启用了 CBO 优化器），这意味着：

```plsql
SELECT ...
FROM A
JOIN B ON A.id = B.id
JOIN C ON A.id = C.id;
```

会被 Hive 解释为：

1. 先`A JOIN B` → 得到中间结果 `AB`
2. 再`AB JOIN C` → 最终结果

📌 中间结果的大小直接决定了下一步 `JOIN` 的开销！

如果第一步就产生巨大中间表，后续 `JOIN` 会非常慢，甚至 OOM（内存溢出）

## Hive 的 `JOIN`模型（简化版）
Hive 的 JOIN 通常在 Reduce 阶段完成（除非是 Map Join）：

+ 每个 `JOIN` 操作 = 1 个 MapReduce Job（或 Tez DAG 中的一个 vertex）
+ 多表 `JOIN` 可能被合并为一个 Job（如果 `join key` 相同），但逻辑上仍是顺序执行

## 示例：3 表 `JOIN` 的两种顺序
❌ 不好的顺序（大表先 JOIN）

+ 第一步：`A JOIN B` → 可能输出 5亿行
+ 第二步：5亿行 `JOIN C`（1万行）→ 虽然 C 小，但要处理 5亿次匹配，很慢

```plsql
-- 假设：A（10亿行）、B（10亿行）、C（1万行，小表）
SELECT ...
FROM A
JOIN B ON A.id = B.id   -- ⚠️ 先 JOIN 两个大表！可能产生数亿行中间结果
JOIN C ON A.id = C.id;
```



✅ 好的顺序（小表尽早参与）

+ 第一步：`A`（10亿）`JOIN C`（1万）→ 假设只匹配 10万行
+ 第二步：10万行 `JOIN B`（10亿）→ 数据量小，速度快得多！

```plsql
SELECT ...
FROM A
JOIN C ON A.id = C.id   -- ✅ 先用小表 C 过滤 A，大幅减少数据量
JOIN B ON A.id = B.id;
```



💡 <font style="background-color:#E8F7CF;">核心思想：尽早用小表或高选择性条件过滤数据，减少中间结果集大小</font>

## Hive 如何决定 `JOIN` 顺序？
1. 默认行为（无 CBO）
    - Hive 严格按照 SQL 中 FROM 后的表顺序 + JOIN 书写顺序 执行。
    - 不会自动重排（即使你知道 C 是小表，写在最后也没用）
2. 启用 CBO（Cost-Based Optimization）后
    - Hive 会基于表统计信息（行数、列 NDV 等）自动重排 `JOIN` 顺序，目标是最小化中间数据
    - 需要

```plsql
SET hive.cbo.enable=true;
SET hive.compute.query.using.stats=true;
SET hive.stats.fetch.column.stats=true;
SET hive.stats.fetch.partition.stats=true;
ANALYZE TABLE xxx COMPUTE STATISTICS; -- 提前收集统计信息
```

    - ✅ 强烈建议在生产环境开启 CBO，但前提是定期更新表统计信息。

## 实践建议
✅ 原则 1：把大表放在最后（当未启用 CBO 时）

```plsql
-- 推荐写法（小表在前）
FROM small_table s
JOIN medium_table m ON s.id = m.id
JOIN large_table l ON m.id = l.id;
```

✅ 原则 2：优先让高过滤性的表先 `JOIN`

+ 即使不是物理小表，但如果 `ON` 条件能大幅减少结果行数，也应靠前。
+ 例如：订单表 `JOIN` VIP用户表（只占 1%）比 `JOIN` 所有用户表 更高效。

✅ 原则 3：显式使用子查询控制顺序（万能方法）

+ 如果不确定 Hive 是否优化，可以用子查询强制顺序：这样无论 CBO 是否开启，都先执行内层 JOIN。

```plsql
SELECT *
FROM (
    SELECT a.*, c.extra_info
    FROM large_table_a a
    JOIN small_lookup_table c ON a.code = c.code  -- 先过滤
) ac
JOIN huge_table_b b ON ac.id = b.id;
```

## 特殊优化：Map Join 与顺序的关系
+ Hive 的 自动 Map Join 只会对 JOIN 链中的小表生效。
+ 但 Map Join 通常只能作用于最后一个表（在旧版本中）。
+ 新版本（Hive 0.11+）支持 多个 Map Join，但仍建议：

<font style="background-color:#E8F7CF;">把最小的表放在最后，更容易触发 Map Join。</font>

🔔<font style="background-color:#E8F7CF;"> </font>注意：这与“大表放最后”的原则看似矛盾，其实不冲突

“大表放最后”是为了减少中间结果；“小表放最后”是为了触发 Map Join。

实际中需权衡，但通常减少中间结果更重要。



建议：

+ <font style="background-color:#E8F7CF;">小表应该放哪？尽早参与以过滤数据，或放最后以触发 Map Join（需权衡）</font>



# `IN/NOT IN`后跟子查询
## `IN`
HiveSQL 支持`IN` + 子查询

```plsql
SELECT * 
FROM table1 
WHERE col1 IN (SELECT col2 FROM table2);
```

子查询必须返回单列（否则报错）。

性能上，Hive 会将其转换为 MapJoin 或 Common Join（取决于数据量和配置）。



也可使用以下方式实现`IN`

+ `LEFT SEMI JOIN`
+ `LEFT OUTER JOIN` + `IS NOT NULL`
+ `INNER JOIN`

## `NOT IN` + 子查询：⚠️ 有陷阱
```plsql
SELECT * 
FROM table1 
WHERE col1 NOT IN (SELECT col2 FROM table2);
```

❗ 问题：

+ <font style="background-color:#E8F7CF;">如果子查询结果中包含 </font>`<font style="background-color:#E8F7CF;">NULL</font>`<font style="background-color:#E8F7CF;"> 值，整个 </font>`<font style="background-color:#E8F7CF;">NOT IN</font>`<font style="background-color:#E8F7CF;"> 条件会返回 </font>`<font style="background-color:#E8F7CF;">NULL</font>`<font style="background-color:#E8F7CF;">（即不匹配任何行）！</font>
+ <font style="background-color:#E8F7CF;">这是 SQL 标准行为，不仅限于 Hive，MySQL、PostgreSQL 等也一样。</font>

原因：以下二者等价

+ `col1 NOT IN (a, b, NULL)`
+ `col1 != a AND col1 != b AND col1 != NULL`

而 `col1 != NULL` 的结果永远是 `UNKNOWN`（即 `NULL`），导致整个 `WHERE` 条件为 `NULL` → 无结果返回



正确做法：排除`NULL`，

<font style="background-color:#E8F7CF;">使用</font>`<font style="background-color:#E8F7CF;">NOT IN</font>`<font style="background-color:#E8F7CF;">，务必确保子查询不含 </font>`<font style="background-color:#E8F7CF;">NULL</font>`<font style="background-color:#E8F7CF;">：</font>

```plsql
SELECT * 
FROM table1 
WHERE col1 NOT IN (
    SELECT col2 
    FROM table2 
    WHERE col2 IS NOT NULL  -- 关键！过滤掉 NULL
);
```



更安全的替代方案：`NOT EXISTS`，不受`NULL`影响

```plsql
SELECT * 
FROM table1 t1
WHERE NOT EXISTS (
    SELECT 1 
    FROM table2 t2 
    WHERE t2.col2 = t1.col1
);
```
