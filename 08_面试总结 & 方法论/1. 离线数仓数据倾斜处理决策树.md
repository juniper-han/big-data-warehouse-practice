数据倾斜优化的本质不是“打散”，而是“在稳定性和整体资源消耗之间做权衡”。

> 一般会先判断倾斜发生在哪个阶段，是聚合、Join 还是分区问题？
>
> 1. 如果是聚合类倾斜，会先分析 Key 分布，热点占比低时精准打散，占比高时全量打散。
> 2. 如果是 Join 倾斜，优先考虑广播 Join；
> 3. 若是双大表则拆热点或双边打散。
>
> 同时会权衡稳定性与资源成本，避免为少量热点引入全局额外 Shuffle。
>

## 数据倾斜处理决策树
前提：先判断，再处理

先判断倾斜发生的阶段：

+ 一般是 Reduce 端
+ Map 端也有，一般是分区不合理导致某些分区数据量大

### 第一步：判断倾斜类型
1. 聚合类倾斜（`GROUP BY`/ Reduce 阶段）
    - 表现：少数 Task 运行时间极长，Top Key 占比异常高
    - 最常见的“热点 Key 倾斜”。
2. Join 类倾斜
    - 表现：大表 Join 后 Reduce 端长尾，某些 Key 在两张表中同时高频
    - 双表叠加型倾斜
3. 数据分区倾斜（存储侧）
    - 表现：某些分区数据量异常大
    - 分区设计问题

---

### 第二步：根据类型不同，选择不同的处理路径
---

### ① 聚合类倾斜决策路径
<font style="background-color:#FBF5CB;">是否存在热点 Key？→ 占比多少？</font>

#### ✅ 情况 1：热点占比低（< 5%）
方案：精准打散

+ 识别热点 Key，只对热点加随机前缀
+ 两阶段聚合

优点：成本最低  
适用：离线批处理

#### ✅ 情况 2：热点占比高（> 30%）
方案：全量打散

+ 所有 Key 增加随机前缀
+ 两阶段聚合

因为分布本身已经严重不均。

#### ✅ 情况 3：Key 动态变化，难以预测
方案：全量打散 or 调整分区策略

适用于：

+ 实时流
+ 高频变动场景

---

### ② Join 类倾斜决策路径
<font style="background-color:#FBF5CB;">先判断：是否可以广播小表？</font>

#### ✅ 情况 1：一张表很小（< 几百 MB）
直接 Broadcast Join

+ 避免 Shuffle
+ 转 Map Join

这是最优方案。

#### ✅ 情况 2：两张都是大表
<font style="background-color:#FBF5CB;">继续判断：是否存在热点 Key？</font>

##### 🔹 有热点 Key
+ 方案 1：热点 Key 单独处理
    - 将热点 Key 单独拆出来，分开 Join
    - 再 Union
+ 方案 2：对热点 Key 打散
    - 双边打散，保证 Key 对齐

##### 🔹 无明显热点，但仍长尾
+ 检查：
    - 分区数量是否合理？
    - 是否存在数据 Skewed 分区？
+ 可调整：
    - increase shuffle partitions
    - 重分区

---

### ③ 分区倾斜决策路径
<font style="background-color:#FBF5CB;">判断：分区设计是否符合查询模式？</font>

#### ❌ 时间分区过细
比如按天分区，但查询常按月聚合。

改为月分区。

#### ❌ 某些分区数据异常大
采用二级分区（时间 + 业务维度）

---

## 什么时候不用打散？
1. 可以提前过滤：减少数据规模往往比打散更优
    - 比如：先筛选有效数据，再做聚合
2. 可以改模型：从“重复计算”变成“沉淀式计算”，比打散更高级
    - 例如：改为增量累加，改为预聚合

## 全量打散
### 全量打散是否可行？
技术上完全可行。

做法是：

1. `concat(rand_prefix, member_id)`
2. 然后两阶段聚合：先：带前缀聚合，然后：去前缀再聚合

### 但为什么通常不推荐“无脑全量打散”？
因为它引入了新的成本。

1. 会增加一次完整的二次聚合
    - 原本流程：`Map → Shuffle → Reduce`
    - 全量打散后：`Map → Shuffle → Reduce(第一次)→ Shuffle → Reduce(第二次)`
    - 人为增加了一轮 Shuffle
    - 如果原本只有少量热点 Key，现在为了 1% 的问题，让 100% 的数据承担额外成本。
    - 这在工程上是不优的。
2. 会增加网络传输与排序成本
    - 第二次聚合依然要：排序，Shuffle，网络传输
    - 如果数据规模很大，这部分成本不可忽略。
3. 会影响代码复杂度与可读性
    - 增加中间层、逻辑复杂度、维护成本
    - 如果热点比例很小，这是不值得的。

### 什么时候可以全量打散？
以下场景是合理的：

1. 热点比例极高
    - 例如：Top 5% Key 占 50% 数据量
    - 本质上是“整体分布失衡”，那就应该整体打散。
2. 无法提前识别热点 Key
    - 例如：数据实时流入、热点 Key 动态变化
    - 这时做精准识别成本高，全量打散反而更稳妥
3. 对稳定性要求极高
    - 有些场景宁愿牺牲一点性能，也不能出现长尾
    - 这时可以牺牲一点整体性能换稳定性。

### 工程上的实践
推荐策略是：精准识别热点 Key，只对热点打散。步骤：

1. 统计 Key 分布
2. 找出热点
3. 只对热点加前缀
4. 非热点正常走

这样：总体成本最低

+ 普通数据走一次 Shuffle
+ 热点数据走两次
