## 概述
基本原则：经验初选 + 压力测试微调 + 线上监控反馈



方法论：三步走策略

1. 静态预估（经验值）： 根据业务QPS、每条数据的大小（例如2KB）、算子复杂度，推算出初始的内存和并行度。
2. 动态压测（确定阈值）： 在预发环境进行 1.5 到 2 倍峰值流量的压测，观察延迟曲线和背压点。
3. 动态调优（消除瓶颈）： 针对瓶颈点（如 Kafka 磁盘 I/O 或 Flink GC）进行参数对冲。

## 各组件配置思路
### Kafka：吞吐量与可靠性的平衡：
+ 由于底层是HDFS，重点在于磁盘I/O和网络带宽。
+ Broker级别：
    - 网络线程数 / IO线程数（逻辑处理线程数）：通常按 1:2 比例配置，根据CPU核数调整。
    - 日志段文件大小：建议1G，配合 HDFS 存储特性。
+ Producer 级别：
    - 批次大小 & 等待时间（毫秒）：这是吞吐量的核心。
        * 经验：16KB 批次结合 10-50ms 延迟，能在吞吐和实时性间取得最优平衡。
    - 压缩类型：强烈建议开启 lz4 或 zstd，极大地减轻网络负载，但注意会增加 CPU 开销。

### Flink：内存模型与算子并发
+ 内存分配：
    - 托管内存（Flink 自身管理内存）： 状态后端+算子计算，必须分给它 40%-50%。
    - 网络内存： 任务间数据传输缓冲，太小会出现反压
+ 并发度（Parallelism）： 初始经验值 = Source 分区数
+ Checkpoint： 必须配置 minPauseBetweenCheckpoints，防止 Checkpoint 连续失败导致任务雪崩。

### HBase：作为维表或落地存储
+ 由于底层是HDFS，HBase 的性能取决于 RegionServer 的内存管理。
+ BlockCache 与 MemStore： 
    - 读多写少则调大 BlockCache；
    - 写多读少（实时落地）则调大 MemStore（如 0.4）。
+ HFile 合并： 触发 Minor Compaction 的 StoreFile 数量下限。
    - 实时链路最怕 Heavy Compaction 导致的 STW，通常会调大阈值并在凌晨手动触发 Major Compaction。

### Doris：实时数仓的写入性能
+ Stream Load 相关： 单批次最大数据量，控制批次大小（建议 100MB-500MB），避免过快产生过多小文件，过大 OOM
+ 内存控制：查询内存限制。BE 节点在高并发查询时容易 OOM，需要通过压测设定合理的百分比限制。
+ Compaction 调优： 针对实时高频写入，需调整【合并策略（增量）】为基于文件大小合并，加速小版本合并。
