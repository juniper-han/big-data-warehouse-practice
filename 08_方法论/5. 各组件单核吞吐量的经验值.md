由于物理机性能（磁盘 I/O、网络带宽）高度不一致，单核吞吐（Single Core Throughput）必须通过“基准测试（Benchmarking）+ 线上采样”来确定。

以下是针对各组件确定单核吞吐量的具体实操方法：

## Flink：单核吞吐量（TPS/Core）的确定
+ Flink 的单核吞吐：受业务逻辑复杂度和状态量（State）影响最大。
+ 基准压测法： 在预发环境开启一个 Parallelism = 1 的单算子/单任务，不限流地消费 Kafka 数据。
    - 监控指标： 观察 numRecordsOutPerSecond（输出速率）直到 CPU 达到 80%-90% 或出现 Backpressure。
    - 计算公式： 单核 TPS =最大吞吐量 / CPU 核心数
+ <font style="background-color:#E8F7CF;">经验参考值（通用）：</font>
    - <font style="background-color:#E8F7CF;">简单 ETL（清洗、转换）： 约 1w - 5w 条/秒/核。</font>
    - <font style="background-color:#E8F7CF;">复杂聚合（Window + RocksDB）： 约 1k - 5k 条/秒/核（受限于磁盘 I/O 随机读写）。</font>

## Kafka：单核/单分区吞吐量
+ 对于 Kafka，瓶颈通常不在 CPU，而在 网络带宽 或 磁盘 I/O。但我们仍可以按核数折算。
+ 测试工具： 使用自带的 `kafka-producer-perf-test.sh`。
+ 确定流程：
    - 固定单分区（Partition），单线程写入。
    - 逐步增加批处理大小，直到磁盘写入带宽（MB/s）达到物理极限（如 SATA 盘 150MB/s 或 SSD 500MB/s）。
    - 单核折算： 通常一个 `io.thread` 对应一个物理核，记录该核满载时的写入量。
+ 注意： 如果底层存储是 HDFS，需要考虑副本同步对带宽的消耗

## Doris：单核写入与查询吞吐
+ Doris 是计算密集型和 I/O 密集型并重，确定单核吞吐要分<font style="background-color:#E8F7CF;">“入”和“出”</font>。
+ 写入吞吐（Stream Load）：
    - 固定单 BE 节点，使用 curl 持续推送 100MB 左右的 CSV 数据。
    - 观察 be.mem_consumption 和 be.cpu_utilization。
    - <font style="background-color:#E8F7CF;">指标： 通常 1 个 Core 每秒能处理约 10MB - 30MB 的原始数据写入</font>。
+ 查询吞吐（QPS）：
    - 使用 Apache Benchmark (ab) 或 JMeter 模拟高并发简单查询。
    - 当 CPU 达到 80% 时，记录总 QPS，折算到单核。

## HBase：单核吞吐（Request/Core）
+ HBase 的单核性能主要看 RegionServer 的处理能力。
+ 测试方法： 使用 YCSB（Yahoo! Cloud Serving Benchmark）进行压测。
+ 确定瓶颈：
    - 读： 观测 BlockCache 命中率。若命中率高，单核吞吐极高；若走 HDFS 磁盘读，吞吐暴降。
    - 写： 观测 MemStore 刷新频率和 HLog 写入延迟。
+ 结论：<font style="background-color:#E8F7CF;"> 在物理机环境下，单核通常支撑 2,000 - 8,000 次随机读写（受限于 HDFS 延迟）</font>。

## 算力地图
| 组件 | 业务类型 | 测试环境规格 | 单核理论极限 (TPS) | 生产环境预留值 (Safety Margin) |
| --- | --- | --- | --- | --- |
| Flink | 简单维度关联 | 8C 16G | 1.2W | 8k (给突发预留 30%) |
| Doris | 实时明细写入 | 16C 64G | 25MB/s | 15MB/s |
| Kafka | 压缩写入 (LZ4) | 4C 8G | 80MB/s | 50MB/s |


下一步动作

+ 定期重测： 业务逻辑（如 JSON 变复杂）变化后重新进行单核压测。
+ 监控对比： 将“单核预估值”与“线上实时 CPU 负载”做对比，如果 CPU 占用远超预估，说明代码存在“热点”或逻辑漏洞。
