## 上游下游合作模式？
**上游：业务系统或数据源，以数据质量为核心**

1. 需求对齐：新业务上线前，参与 PRD 评审，明确埋点/日志/DB 表结构
2. 数据接口协议：明确数据格式、字段含义、更新频率、数据有效期
3. SLA 约束：产出时间 + 变更通知
    1. 对于关键源表，我们与上游约定产出时间SLA
    2. 上游提供：数据字典和元数据变更通知机制
4. 质量监控共担：
    - 共建数据质量监控看板：源表数据量突增突减、主键重复、空值率超标等
    - 告警：双方共同排查，上游负责源头修复，数仓负责在ETL中做兜底(如去重、默认值填充)



**下游是 BI、算法、运营、产品团队，以需求交付为中心**

1. 需求前置评审：
    1. 下游提供业务口径说明和期望的数据粒度、更新频率、时效性
    2. 数仓评估可行性、预估开发量
2. 数据交付方式：
    1. 提供：数据资产目录和数据 API，下游可以自助查询或订阅
    2. 对于核心报表，我们提供数据质量报告（如数据一致性校验结果）和变更通知（如字段废弃、口径调整提前通知）
3. 定期沟通机制：
    1. 每周同步进度、暴露风险、调整优先级
    2. 建立需求响应机制：紧急需求走“快车道”,但事后补齐文档和测试

## 指标谁负责？
业务主导，数仓落地，统一口径



**责任划分**

1. 指标定义：业务方、产品
2. 指标实现：数仓
3. 指标验收：业务方（抽样数据验证）、数据测试
4. 指标运维：
    1. 数仓：监控指标异常，修复
    2. 业务：负责业务口径变更的通知和确认



**建立指标管理体系：数据治理团队**

1. 指标字典：所有新指标上线前必须在此注册
2. 指标评审机制：新增/修改指标，须经数据治理委员会（由数仓、产品、业务、数据测试组成）评审
3. 指标血缘与影响分析：指标变动时，通知依赖方



**争议处理：以指标字典为准，以原始数据为证**

+ 数仓提供详细的 ETL 过程日志和中间表数据
+ 与业务方共同追溯差异根源
+ 最终修正口径或修复数据，并将案例沉淀为知识库

##  开发时间不足如何协调？
保核心、切版本、勤同步、有预案

1. 需求阶段：明确优先级，MoSCoW 法则分类：
    - Must have：监管报送、核心营收指标（优先保障）
    - Should have：重要运营看板（可延后 1~2 周）
    - Could have：探索性分析（转为自助查询）
    - Won’t have：模糊需求（打回澄清）
2. 推动“最小可行交付（MVP）
    1. 例：业务要“用户全生命周期画像”，我们先交付 关键阶段（注册→首购→复购），后续迭代补充；
3. 技术提效：寻求折中方案，减少重复劳动
    1. 复用已有模型：如 DWS 拼接
    2. 自动化工具：自动生成 DDL、调度脚本、数据质量校验脚本
    3. 标准化开发规范
    4. 提出技术降级方案：如实时链路改为准实时或 T+1
4. 资源调配：并行开发，资源借调
    1. 并行开发，负责不同模块
    2. 临时资源倾斜：增加人力、调低其他非核心需求的人员
5. 风险管理：提前暴露，主动沟通
    1. 预估可能延期：第一时间同步项目风险给项目PM和需求方,给出新的预估时间和可选方案
    2. 建立进度看板/每日站会
6. 事后复盘：分析不足，沉淀经验
    1. 需求预估不准？技术难度预估不足？

## 开发优先级？
**核心原则：业务价值驱动**



**优先级决策流程**

1. 业务方提交需求价值说明
    1. 业务目标、量化指标、影响范围
    2. 拒绝模糊需求：如“需要更精准的用户画像” → 要求拆解为具体指标
2. 价值-风险评估：双维度评估表，对每个需求打分（1-5分）
    - 优先级 = 业务价值 × 影响范围 / (技术风险 × 开发成本)
    - 得分越高，优先级越高。

| 维度 | 说明 | 评分标准 |
| --- | --- | --- |
| 业务价值 | 对核心业务目标的贡献 | 5=直接影响营收/留存，1=辅助分析 |
| 影响范围 | 覆盖用户/业务线数量 | 5=全站，1=单个频道 |
| 技术风险 | 实施难度/链路稳定性 | 5=高风险（需改核心表），1=低风险（用现成模型） |
| 开发成本 | 人日估算 | 5=2 周+，1=1 天 |


3. 定期评审机制
    1. 周会评审：各方同步新需求、确认优先级、调整排期
    2. 双周迭代：每个迭代开始前锁定需求清单，迭代期间原则上不接受新增需求，除非紧急需求
    3. 资源池透明：将团队的人力资源（每人当前任务、饱和度）在项目管理工具上可视化，让业务方理解“资源有限，有得必有舍”
4. 冲突处理机制：多个需求争夺有限资源时
    1. 向上升级：紧急需求冲突，同步给更高级协调
    2. 置换原则：如果插入新需求，置换旧需求，由业务方决定
    3. 时间换资源：延长交付或增加资源，由需求方协调选择



**其他评估视角也需要一定的优先级**

1. 技术债：数据倾斜优化、模型重构、资源治理
    1. 高风险：可能导致线上故障，优先级高
    2. 长期优化项：需求空窗期
2. 主动规划
    1. 主动梳理数仓能力的缺失（如缺少用户行为宽表），将共性能力建设作为高优先级推进
    2. 提前了解业务节奏，提前预留资源
3. 依赖与阻塞（权重约15%）
    1. 上游依赖：如果是多个项目的基表，优先级提高
    2. 阻塞影响：不做会不会阻塞其他团队进度？
    3. 合规与安全：涉及合规、审计，优先级最高



**资源紧张，拒绝需求**

+ 避免说做不了
+ 而是说“目前排期已满，如果要插入，需要从以下方案中选择：①推迟其他需求；②简化实现范围；③增加资源”。把选择题交给需求方。

## 接到一个需求，你们的流程是什么样的？
整个流程：每个阶段都有明确的输入、输出、参与方和质量门禁。

1. 需求评审输入与初步过滤
    1. 需求通过统一的平台提交
    2. 过滤机制：未明确指标、低价值需求
2. 需求评审：数据契约签署（业务+数仓共签）
    1. 目标：理解业务诉求，明确交付边界，避免“需求理解偏差”导致返工。
    2. 需求对齐：口径定义、技术实现、交付形式、SLA
    3. 输出物：需求单，包含明确的需求描述、口径说明、期望交付时间、优先级
3. 技术方案设计
    1. 目标：设计合理的数据模型和ETL链路，确保可维护性、性能和扩展性
    2. 输出物：技术方案文档，包含表结构、ETL流程图、依赖关系、预估资源消耗、开发工时评估
4. 开发与自测
    1. 目标：按方案实现，确保代码质量
    2. 代码开发（符合规范）、自测、代码审查
    3. 输出物：可部署的代码、自测报告、CR记录
5. 数据验证
    1. 目标：确保数据准确性、完整性、一致性
    2. 测试环境验证：数据质量、性能；
    3. 业务验收：需求方在需求单上确认签字(或系统确认)
    4. 输出物：数据验证报告、业务验收确认记录
6. 上线发布
    1. 目标：安全、可控地将数据交付到生产环境
    2. 上线前准备：回滚方案，走上线流程
    3. 上线执行：低峰期，按照上线checklist逐步执行：创建表结构→部署调度任务→首次运行→验证数据；灰度发布：先上线，验证1-2天再对业务方开放
    4. 上线验证：调度任务正常，数据质量验证，通知下游数据已就绪
    5. 输出物：上线checklist执行记录、上线后的数据验证截图。
7. 运维与迭代
    1. 目标：保障数据稳定产出,持续优化迭代。
    2. 监控配置，失败告警
    3. 知识沉淀：更新数据资产目录、更新指标字典
    4. 迭代支持：同样流程处理

## 数据开发的口径如何确认？沟通协作流程？
**“事前定义、事中确认、事后沉淀”三个环节**

核心原则：

+ 业务方：负责口径定义
+ 数仓：技术落地和一致性保障
+ 口径变更必须可追溯、可通知



**口径确认完整流程**

1. 需求提出阶段：明确指标定义，输出完整的口径说明的需求单
2. 需求评审阶段：多方达成共识
    1. 逐一口径澄清：比如飞机是全部飞机数还是可用飞机数
    2. 场景举例：构建业务场景，让业务方确认结果
    3. 与现有指标对齐：查询指标字段，看是否有相同/相近的指标，由业务方确认是否复用
    4. 评审结论记录：明确每条指标的最终口径、负责人、验收标准，签字确认
    5. 输出物：评审通过的指标口径文档，附带场景举例和验收标准
3. 技术设计阶段：口径可落地
    1. 数据源确认、逻辑拆解、异常处理
    2. 验证 SQL 预写：先写一段验证SQL，将结果发给业务方确认，避免“开发完才发现理解偏差”
    3. 输出物：技术方案文档，包含口径的技术实现说明、数据源依赖、异常处理逻辑
4. 开发验证阶段：确保实际产出与口径定义一致
    1. 自测，质量校验，业务验收，验收确认
    2. 输出物：数据验证报告、业务验收确认记录
5. 上线运维阶段：口径变更可追溯，下游依赖方及时感知
    1. 数据字典录入
    2. 变更管理：走流程
        1. 影响分析：通过数据血缘工具查询下游依赖，列出所有受影响的报表、API、任务。
        2. 变更评审：召集所有受影响方，确认变更方案、切换时间、数据补录策略。
        3. 双跑验证：新旧口径并行运行一段时间，验证一致性后再切换。
        4. 变更通知：提前通知所有下游依赖方，给出明确的切换时间点和新旧口径差异说明。



**沟通协作流程：建立“会议”机制**

1. 需求评审会：集中评审新需求，确认口径，排定优先级
2. 进度同步会（每周）
3. 上线验收会（每个迭代）：业务方集中验收已交付需求，反馈问题，避免零散沟通
4. 口径争议会（按需）：当业务方与数仓对口径有分歧时，邀请双方Leader参与，快速决策。我的原则是“不把争议带进开发”。
