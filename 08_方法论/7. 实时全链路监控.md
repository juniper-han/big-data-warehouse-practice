核心逻辑是：“不仅要看到系统活着，还要看到数据准时、准确地流动。”

监控体系方法论：分层治理
我会将全链路监控分为三层：
1. 基础设施层：CPU、内存、IO、网络（通常由运维侧 Prometheus + Exporter 完成）。
2. 组件性能层：Kafka 堆积、Flink 反压、HBase 延迟、Doris 导入频率。
3. 全链路指标监控：端到端延迟、数据倾斜、数据质量（丢包率/重复率）。

全链路核心指标：端到端延迟 (End-to-End Latency)，这是衡量实时系统好坏的“终极指标”。
1. 实现方案：在数据源头（如埋点日志）带上 event_time，在 Doris 最终表中记录 insert_time。
2. 监控方式：通过 insert_time - event_time 计算差值。
3. 预警机制：如果全链路延迟超过 5 秒，立即触发告警（通过钉钉/企业微信机器人）。
组件性能层
1. Kafka 监控（数据源头）：监控重点在于产销平衡。
  a. Consumer Lag（消费滞后）：这是最核心指标。我会监控每个 Partition 的滞后量。如果 Lag 持续上升，说明下游 Flink 处理不过来
  b. Under-Replicated Partitions：副本同步情况，预防 Broker 宕机导致的数据丢失。
  c. Producer/Consumer Throughput：字节流量和消息条数，用于判断上游是否有突发流量。
2. Flink 监控（计算大脑）
  a. Checkpoint 成功率与耗时：如果 CP 频繁失败或耗时突增，通常预示着状态过大或网络抖动，这是作业不稳定的先兆
  b. Backpressure（反压）：监控 isBackPressured 指标
  c. Watermark Delay：Current Emitter Watermark vs System Time。如果水位线不推进，窗口计算就不会触发，导致业务数据“假死”
  d. CPU/GC 频率：TaskManager 的堆内存使用率和频繁 Full GC 会直接导致任务吞吐骤降。
3. HBase 监控（维表/状态存储）
  a. Request Latency (P99/P999)：读写延迟。如果延迟超过 100ms，会直接拖垮 Flink 的异步 IO。
  b. RegionServer 状态：MemStore 占用率、BlockCache 命中率。
  c. Compaction 队列：如果大合并（Major Compaction）堆积，会产生严重的 IO 阻塞。
4. Doris 监控（OLAP 终点）
  a. Routine Load / Stream Load 状态：监控导入任务是否暂停（PAUSED），这是数据中断的最常见原因
  b. Query Latency：SQL 查询响应时间，确保报表端不卡顿。
  c. Tablet 状态：是否存在损坏或不一致的副本。
  d. BE/FE 节点的内存与磁盘占用：防止因磁盘满导致集群进入只读状态。
自动化告警与自愈 (Best Practices)：避免“告警风暴”：
1. 分级告警：P0（任务停滞）、P1（延迟超 10 分钟）、P2（小规模数据倾斜）
2. 动态阈值：针对高峰期和低峰期设置不同的 Lag 告警阈值
3. 链路压测：定期进行全链路压测，摸清 Kafka 到 Doris 的最大吞吐量，作为监控阈值的科学依据。
