## 概述
核心在于：“全链路压测的精细化治理”，而不是简单的“暴力测试”

涉及以下三个方面

1. 环境复用策略
2. 数据脱敏隔离
3. 流量染色技术



实际操作

1. 先在测试环境，测单机极限
2. 然后在生产环境的低谷期压测

## 压测环境：**不搞 1:1 还原（太浪费）**
原则：只有核心链路（如实时计算节点）做高配，存储和非核心节点用“逻辑等效”代替“物理等效”。

+ 混合云/弹性资源池： 压测时临时申请资源，结束后释放。
+ <font style="background-color:#E8F7CF;">计算节点</font>： <font style="background-color:#E8F7CF;">保持单机规格（CPU/内存）与生产一致，但副本数/节点数降级</font>
    - 例如： 生产是 100 台 Flink 节点，压测只需 5-10 台。
    - 通过单机吞吐极限 × 生产节点数，推算出整体水位。
+ <font style="background-color:#E8F7CF;">存储节点</font>（Kafka/HDFS）： 共享开发/预发集群，但通过磁盘 I/O 隔离和配额限制，确保不影响其他任务。

## 压测步骤：四阶段法
压测分为四个阶段：基准压测 → 压力递增 → 极限压测 → 稳定性压测。

1. 构造流量： 编写流量注入脚本（如单机版 Flink 读取 Mock 后的 Kafka 数据），模拟生产中的真实 QPS 比例。
2. 流量染色： 关键技术。
    - 在 Kafka Header 或 Flink Stream 中打上 `is_test=true` 标签，确保压测数据流入隔离的测试表（Shadow Table），防止污染生产 HDFS/Doris 报表。
3. 观测漏斗： 监控从 Source 到 Sink 的延迟曲线。

## 直接在生产环境压吗？（影子集群 vs 错峰执行）
真正的全链路压测，必须包含生产环境验证，但要加“安全锁”。

+ 影子库/影子表： 在 Doris 或 HBase 中建立同结构但不同名的表。压测数据最终流入影子表，不影响业务 BI。
+ 错峰/低谷执行： 选在凌晨 2:00 - 5:00。此时生产流量处于波谷，物理机有冗余的 CPU 和 I/O。
+ 自动熔断： 压测平台接入监控告警（如生产链路延迟超过 2s 或 HDFS 负载过高），自动触发脚本停止，保障生产安全。

## 成本平衡：服务器平时闲着？
+ 研发/测试环境复用： 压测环境在平时就是开发环境（DEV）和预发环境（STG）。
+ 潮汐分配： 在大数据平台中，白天的计算资源倾斜给“实时任务”做研发压测；晚上的计算资源（离线计算空闲时）倾斜给“存储组件”做高并发拉取压测。
+ 容器化削峰填谷： 通过 K8s 部署 Doris BE 或 Flink TM。压测时扩容，平时缩容，资源归还给公司公共池供其他部门（如算法训练）使用。
