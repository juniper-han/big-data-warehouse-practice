## State 大小估计
在 Apache Flink 中，State（状态）大小与输入数据量之间没有固定的倍数关系。State 的大小完全取决于业务逻辑、算子类型、Key 基数、窗口长度、是否启用 TTL 等因素，而不是原始数据量本身。

不过，我们可以根据典型场景给出经验性估算范围，做资源规划。

### 结论
Flink State 大小通常是<font style="background-color:#E8F7CF;">“活跃状态数据”的体积</font>，而非全量输入数据的简单倍数。

在大多数常规实时数仓场景中：

+ 无状态作业：State ≈ 0
+ 短窗口聚合（如 5 分钟）：State ≈ 输入数据量的 0.1～0.5 倍
+ 长窗口/双流 Join（如 24 小时）：State ≈ 输入数据量的 1～3 倍
+ 未设 TTL 的 KeyedState：State 持续增长，可能远超原始数据量

### 影响 State 大小的关键因素
| 因素 | 对 State 的影响 | 示例 |
| --- | --- | --- |
| 1. 是否使用状态 | 无状态算子（map/filter）→ State ≈ 0 | 简单清洗：State 几乎为 0 |
| 2. 状态类型 | KeyedState（按 key 存） vs OperatorState（全局） | 用户 ID 聚合 → 每个用户一个状态 |
| 3. Key 基数 | Key 越多，State 越大 | 百万级用户 vs 千级商品 |
| 4. 窗口/Join 时间窗口 | 窗口越长，缓存数据越多 | 1小时窗口 State ≈ 1小时数据量 |
| 5. 是否启用 TTL | 未设 TTL → State 无限累积；设 TTL → 自动清理 | TTL=1天 → State ≈ 1天活跃数据 |
| 6. 序列化效率 | POJO vs 字符串，体积差异大 | 优化后可减少 30%~50% |
| 7. 状态后端类型 | RocksDB 有索引/元数据开销（+10%~30%） | 1GB 逻辑 State → 1.2GB 物理磁盘占用 |


### 总结：State 与输入数据量的关系表
| 业务场景 | State / 日增数据量 | 绝对大小（日增1GB） | 是否推荐 |
| --- | --- | --- | --- |
| 无状态处理 | ≈ 0% | < 10 MB | ✅ |
| 短窗口聚合（<1h） | 0.1% ～ 10% | 1MB ～ 100MB | ✅ |
| 长窗口/Join（24h） | 100% ～ 300% | 1GB ～ 3GB | ⚠️ 需 SSD + TTL |
| 全量缓存维表 | 取决于维表大小 | 可能 > 10GB | ❌ 改用 Async I/O |
| 无 TTL KeyedState | 持续增长 | 不可控 | ❌ 禁止 |


### 建议
1. 不要用“输入数据量 × N”来估算 State，而要分析活跃状态的生命周期。
2. 对于 QPS=500、日增1GB 的场景，只要合理使用 TTL 和窗口，State 通常可控制在 1GB 以内，完全可以在单个 TaskManager（16GB 内存）上稳定运行。
3. 永远为 State 设置 TTL —— 这是生产环境的黄金法则。

## Checkpoint 估计
根据实际生产经验（基于 RocksDB State Backend，这是最常用的生产配置）

### 结论
Checkpoint Duration (秒) ≈ (增量 State Size in GB) × 10 / (HDFS Upload Speed in MB/s)

+ 增量 State Size：指本次 Checkpoint 与上次 Checkpoint 之间的状态增量（RocksDB 增量 Checkpoint 的核心优势）。
+ Upload Speed：上传到持久化存储（HDFS/S3）的速度（典型值：HDFS 100–200 MB/s）。

| 项目 | 推荐值 |
| --- | --- |
| 增量 State Size | 通常为总 State 的 5%–20% |
| HDFS Upload Speed | 150 MB/s（生产环境典型值） |
| 健康 Checkpoint 时长 | < 30 秒（避免超时） |




生产建议

1. 确保增量 State ≤ 5GB：Checkpoint 时长 < 30 秒（150 MB/s 时）。
2. 监控 `lastCheckpointDuration`：Flink Web UI 中查看。

### 详细估算模型（RocksDB 增量 Checkpoint）
#### **增量 State Size 与 Checkpoint 时长关系**
| 增量 State Size | 上传速度 (HDFS) | Checkpoint 时长 | 说明 |
| --- | --- | --- | --- |
| 1 GB | 150 MB/s | ≈ 7 秒 | 150 MB/s ÷ 1GB = 6.67 秒 |
| 5 GB | 150 MB/s | ≈ 33 秒 | 150 MB/s ÷ 5GB = 33.3 秒 |
| 10 GB | 150 MB/s | ≈ 67 秒 | 150 MB/s ÷ 10GB = 66.7 秒 |
| 20 GB | 150 MB/s | ≈ 133 秒 | > 2 分钟（可能超时） |


💡 关键点：

+ 增量 State Size 通常为总 State 的 5%–20%（取决于数据变化率）。
+ HDFS 上传速度是主要瓶颈（100–200 MB/s）。
+ RocksDB 增量 Checkpoint 是关键：避免全量上传（否则 10GB State → 100 秒）。

### 实测数据
| 项目 | 说明 | 实测值 |

| 增量 State Size | 日常变化率 10% | 1GB |
